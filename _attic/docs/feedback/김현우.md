## 2026-05-29
### 관리자(품목 관리) — 사용제품 변경 시 품목 코드 prefix 자동 갱신 안 됨
#관리자 #품목관리 #품목코드 #채번 #모델 #기타 #해결

사용 제품 선택을 바꿔도 품목 코드 변경이 안됨. 품목 코드는 그냥 하드코딩이었던건가?
카테고리와 사용제품을 골랐으면, 뒤에 번호는 알아서 매겨지는게 아니었나?

→ 해결 (2026-05-29)
- **품목 코드는 사람이 입력하지 않고 자동 부여 + 자동 갱신** 방향으로 전환:
  - 수정 폼 `ItemFormFields.tsx` 의 item_code 칸을 *읽기 전용 표시*로 변경. 재채번 버튼 + `handleRenumber()` 함수 제거. prefix 미리보기는 유지(변경 시 어떻게 바뀔지 확인용).
  - 백엔드 `update_item` (`backend/app/routers/items.py`) — model_slots/process_type/option_code 변경 시 `item_code` 자동 재계산.
    - **모델만 변경** → serial 유지, prefix 만 새로 (`8-AR-0307` → `3-AR-0307`).
    - **카테고리 변경** → 새 카테고리의 `next_serial_no` 부여 (`8-VA-0035` → `8-HF-{HF 다음번호}`).
    - **옵션만 변경** → suffix 만 갱신.
    - 안전망: 새 코드 중복 검사 (이론상 발생 X).
  - 직전에 잠깐 추가했던 `GET /api/items/next-code` endpoint 는 호출자 사라져서 제거.
- 사용 사례: TR 부품 일부가 모델 7로 잘못 설정된 케이스, VA → HF 이동한 부품 — 둘 다 모델·카테고리 토글만 바꾸면 코드가 따라옴.
- C+B 플랜(item_models 폐기 + prefix 기반 model_slots/필터)은 같은 세션에서 완료.

### DB — 분해 필드 진실 소스화 + Generated Column / 다중 모델 정규화 검토
#DB #구조 #정규화 #리팩토링 #generated_column #기타 #해결

품목 코드 자동 갱신 작업하다가 발견한 구조적 결함. items 테이블에 같은 정보가 5개 컬럼에 중복 저장:
- `model_symbol` (예: "8")
- `process_type_code` (예: "AR")
- `serial_no` (예: 307)
- `option_code` (예: "BG") — option_code 컬럼 폐기됨
- `item_code`/`mes_code` (예: "8-AR-0307") ← 위 세 개의 조합

→ 해결 (2026-06-02)
- **mes_code → STORED 생성열**: `backend/app/models/item.py:63-71` — `Computed("model_symbol || '-' || process_type_code || '-' || printf('%04d', serial_no)", persisted=True)`. DB 가 자동 계산, 코드·사람 모두 직접 쓰기 불가 → 어긋남 원천 차단.
- **진실 소스 단일화**: 분해 필드(model_symbol·process_type_code·serial_no) 3개가 진실, mes_code 는 파생 — 드리프트 불가 구조.
- **채번·파생 단일화**: symbol↔slot 하드코딩 제거 → `product_symbols` 테이블 런타임 파생 (`app/utils/mes_code.py` lazy cache).
- **option_code 폐기**: 피드백 당시 "option_code" 컬럼은 폐기 결정. mes_code 에 suffix 미포함 (3-part 코드 확정).
- **NOT NULL 제약**: nullable=True 유지 — 기존 불완전 품목(9-prefix 등) 허용 정책 유지. 강제 NOT NULL 은 별도 마이그레이션 필요 시 추후 결정.
- **다중 모델(B)**: `346789-VR-0004` 같은 케이스는 현재 그대로 유지(prefix LIKE/INSTR) — 피드백 "1번 권장" 대로.
- **도메인 rename**: `item_code` → `mes_code` 완료 (이전 세션 커밋).
- **참고**: DB 전반 정규화 계획은 [`_attic/docs/db-normalization-plan.md`](_attic/docs/db-normalization-plan.md) 에 보존.

### 입출고(불량) — 재작업 시 회수 외 항목이 사라짐 (격리로 가야 맞음)
#입출고불량 #재작업 #격리 #BOM #불편 #해결

재작업 시에 "회수"를 제외한 나머지 항목이 아예 사라져버림. 불량으로 격리시키는 게 맞음.

→ 해결 (2026-06-04, F4 — 커밋 `cd621bc3`)
- 원인 확인: 분해(`submit_defective_disassemble`)에서 회수(keep_qty)하지 않은 잔량을 `DEFECT_SCRAP` 로그만 남기고 재고를 안 움직여 격리 풀에 안 잡힘.
- 수정: 회수 외 잔량을 분해 부서의 격리(DEFECTIVE)로 신규 적재. `inv_defective.receive_defective`(차감 없이 DEFECTIVE 증가) 헬퍼 추가, `dept_adjustment` scrap_qty 분기를 `receive_defective` + `MARK_DEFECTIVE` 로그로 교체. 이후 폐기 여부는 불량 허브 격리→폐기(결재)에서 결정.
- 프론트: `DisassembleTree`/`PaPfDefectWizard` '폐기' 라벨 → '격리'(보류 톤)로 변경.

### 입출고 — 작업 중 저장 시 기존 내역에 덮어씀 (새 저장이 돼야 함)
#입출고 #저장 #audit #덮어쓰기 #불편 #해결

저장을 덮어씀. 입출고 작업 중에 저장하고 나가면 기존 작업 중 내역에 덮어씀. 새로운 저장이 돼야 함.

→ 해결 (2026-06-04, F5 — 커밋 `ff0c1f2e`)
- 원인 확인: `save_draft` 가 (work_type, sub_type) 기준으로 기존 draft를 삭제·재생성하는 덮어쓰기 동작.
- 수정: `batch_id` 분기 도입 — batch_id가 오면 해당 draft만 제자리 갱신, 없으면 새 슬롯 생성. 작업 중 탭에서 여러 임시저장을 이어서 누적 가능.
- 백엔드: `io_persist` 적재 루프를 `_add_bundles_and_lines` 로 추출(신규/갱신 공유), `io_draft.save_draft` batch_id 분기(소유자·상태 검증), `IoDraftUpsert.batch_id` 추가.
- 프론트: `saveDraft` 호출부에 batchId 전달, 새 작업 시작 시 슬롯 연결 해제.

### 관리자(모델 관리) — 모델 자리에 `9` 추가 안 됨
#관리자 #모델관리 #ProductSymbol #9코드 #불편 #해결

품목코드 모델 자리에 9가 들어가 있는 품목의 짝을 맞추기 위해서 9 품목을 추가하려 했는데 추가가 안 됨.

→ 해결 (2026-06-02)
- **진짜 원인**: `create_model` 이 "빈 slot" 을 탐색하는 로직 버그 → 시드가 slot 1~100 을 모두 채워 항상 400 에러. "9" 뿐 아니라 **모든 신규 모델 추가 불가** 상태였음.
- **수정**: `backend/app/routers/models.py:77-82` — 예약 슬롯 승격 방식으로 수정, 신규 모델 추가 재동작.
- **"9" 등록**: 사용자가 **신제품**으로 도메인 확정 → ProductSymbol `(slot=6, symbol="9", model_name="신제품")` 시드 등록 (`backend/bootstrap/seed.py`).
- **9-prefix 부품 15건**: model_symbol 에 `9` 포함 부품들의 model_slots 정합 확인 완료.
- `/api/models` 에 신제품 노출 + 관리자 모델 관리 화면에서 신규 모델 추가 정상 동작.
