#관리자 #품목관리 #품목코드 #채번 #모델 #기타 #해결

사용 제품 선택을 바꿔도 품목 코드 변경이 안됨. 품목 코드는 그냥 하드코딩이었던건가?
카테고리와 사용제품을 골랐으면, 뒤에 번호는 알아서 매겨지는게 아니었나?

→ 해결 (2026-05-29)
- **품목 코드는 사람이 입력하지 않고 자동 부여 + 자동 갱신** 방향으로 전환:
  - 수정 폼 `ItemFormFields.tsx` 의 item_code 칸을 *읽기 전용 표시*로 변경. 재채번 버튼 + `handleRenumber()` 함수 제거. prefix 미리보기는 유지(변경 시 어떻게 바뀔지 확인용).
  - 백엔드 `update_item` (`backend/app/routers/items.py`) — model_slots/process_type/option_code 변경 시 `item_code` 자동 재계산.
    - **모델만 변경** → serial 유지, prefix 만 새로 (`8-AR-0307` → `3-AR-0307`).
    - **카테고리 변경** → 새 카테고리의 `next_serial_no` 부여 (`8-VA-0035` → `8-HF-{HF 다음번호}`).
    - **옵션만 변경** → suffix 만 갱신.
    - 안전망: 새 코드 중복 검사 (이론상 발생 X).
  - 직전에 잠깐 추가했던 `GET /api/items/next-code` endpoint 는 호출자 사라져서 제거.
- 사용 사례: TR 부품 일부가 모델 7로 잘못 설정된 케이스, VA → HF 이동한 부품 — 둘 다 모델·카테고리 토글만 바꾸면 코드가 따라옴.
- C+B 플랜(item_models 폐기 + prefix 기반 model_slots/필터)은 같은 세션에서 완료.

### DB — 분해 필드 진실 소스화 + Generated Column / 다중 모델 정규화 검토
#DB #구조 #정규화 #리팩토링 #generated_column #기타 #미해결

품목 코드 자동 갱신 작업하다가 발견한 구조적 결함. items 테이블에 같은 정보가 5개 컬럼에 중복 저장:
- `model_symbol` (예: "8")
- `process_type_code` (예: "AR")
- `serial_no` (예: 307)
- `option_code` (예: "BG")
- `item_code` (예: "8-AR-0307-BG") ← 위 네 개의 조합

진실 소스 중복 → 어느 한 컬럼에 공란 생기면 어긋남 발생. 김민재 SOLO 필터 누락, 김건호 BOM 소수점, 김현우 prefix 갱신 안 됨, VA→HF 이동 시 코드 안 바뀜, model_symbol NULL 인데 item_code 있음(9-TR-0016 등) — 다 같은 패턴.

→ 미해결 (권동환 사원 휴직 중 사용자 본인이 처리 예정)

**원칙**: 분해 필드 4개가 진실 소스(각자 다른 의미를 가진 원자 정보), item_code 는 그 조합을 사람이 보기 좋게 표현한 *계산 결과*. 사람이 손으로 못 만짐.

**현재 상태** (이번 세션까지):
- 분해 필드 4개 살아있음 ✓
- 사용자 item_code 직접 입력 차단 (수정 폼 readonly) ✓
- 백엔드 자동 계산 + 자동 갱신 ✓
- DB 어긋남 0건 (34건 → 0건 일괄 정리 완료) ✓

남은 강화 옵션:

**A. Generated Column 도입** (가장 깔끔)
SQLite 의 `GENERATED ALWAYS AS (...)` 사용:
```sql
item_code TEXT GENERATED ALWAYS AS (
  model_symbol || '-' || process_type_code || '-' ||
  printf('%04d', serial_no) || COALESCE('-' || option_code, '')
) STORED
```
- DB 가 자동 계산, 사용자·코드 둘 다 직접 못 만짐 → 어긋남 *원천적 불가능*
- 백엔드의 `make_item_code` 호출 코드 다 제거 가능
- 인덱스 가능 → 단일 컬럼 조회 빠름
- 마이그레이션: 컬럼 재정의 (기존 데이터 보존 가능)

**B. 다중 모델 정규화** (선택 사항, 우선순위 낮음)
`346789-VR-0004` 같은 케이스 — 한 컬럼에 5개 값 (1NF 위반). 길:
1. **현재 그대로 유지** (model_symbol 컬럼에 다중 글자, prefix 검색 LIKE/INSTR) — 부품 수 1000~2000 정도면 충분히 빠름. 실용적 정답.
2. item_models 테이블 부활 + 자동 동기화 — 정규화 정통, 동기화 비용. 이번 세션 폐기 되돌림.
3. JSON 배열 컬럼 (`model_slots: [1,3,5,6,7]`) — SQLite JSON 함수 활용. SQL 복잡, 인덱스 약함.

→ **1번 권장**. 깔끔하지 않지만 운영 안정성 + 어긋남 0 + SQL 단순.

**큰 그림 — DB 전반 정규화 계획**:
이번에 발견한 "5컬럼 중복 + 다중 모델 1NF 위반" 외에도 DB 곳곳에 사용자 본인이 보기에도 더러운 부분이 누적되어 있음. 누적된 결과 — 코드 한 줄 고치려고 해도 DB 데이터 정리·스키마 손질이 같이 따라옴 → 운영 부담·실수 위험 증가.

목표: **앞으로 어떤 수정을 하든 DB 는 건드리지 않고 작업 가능한 상태로 정규화**.
- 진실 소스 단일화 (중복 컬럼 폐기 또는 Generated Column 화)
- 마스터 데이터 무결성 강제 (NOT NULL, CHECK, FK 제약)
- 1NF 위반 영역 정리 (다중 값 컬럼 → 별도 테이블 또는 JSON 배열)
- 데드 컬럼·테이블 제거
- 채번·자동 계산 규칙은 코드 한 곳에 집중 (분산되면 또 어긋남)

이게 잘 되면 — 사용자가 코드만 손봐도 데이터 정합성은 DB 가 알아서 보장. AI 활용한 작업도 *코드 변경 → 즉시 안전* 형태로 단순해짐.

**작업 시점**:
- 권동환 사원 휴직 (~6/15) 동안 사용자가 처리.
- 권동환 가이드 4번 (도메인명 `ItemCode → mes_code`) 변경과 묶어서 진행하면 자연스러움.
- 점진적 진행 — 한 영역씩 정규화 (item_code 부터 시작 → 다른 영역 차례로).
- 작업 전 매주 주간보고 사전 피드백 메일로 권동환 사원에게 진행 사항 공유 (5/22 메일의 "사전 협의" 원칙 준수).

**참고 사례** (구조 결함이 만든 누적 결과):
- item_models 411건 빈 채 방치 → C+B 작업으로 폐기
- DB 어긋남 34건 (코드 vs 분해 필드) → 이번 세션 정리
- 9 코드 부품 15건 (model_symbol NULL 2건 + ProductSymbol 미등록) → 별도 결정 영역
