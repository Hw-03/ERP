# 불량 처리 흐름 개선 설계

> 작성일: 2026-05-21
> 작성 배경: 입출고탭에서 불량 세부 작업(격리·폐기·공급처 반품·재작업)과 부서 결재 로직을 정비하기 위한 그릴(Grill) 인터뷰 결과를 정리한 문서.
> 상태: **설계 합의 완료 / 구현 대기**

## 0. 한눈에 보기

| 항목 | 결정 |
|---|---|
| 불량 상태 | `PRODUCTION` / `DEFECTIVE` 2개만 (재작업중 별도 상태 X) |
| 격리 단위 | 부서별로 따로 쌓임 (이미 코드 구조 그대로) |
| 격리 = 재작업 대기 | 같은 풀로 통합. 별도 상태 안 만듦 |
| 부서 조직 | 계층 도입. 생산부 산하에 6라인 |
| 결재 권한 | 생산부장(이필욱·김건호) OR 창고장(정/부) — 라인(튜브/고압/진공/튜닝/조립/출하) 자체에는 별도 부서장 없음 |
| 발의 | 누구나 |
| 격리/정상복귀 | 즉시 (결재 X) |
| 폐기/반품/분해 | 결재 필요 |
| R 처리 | 단순 4버튼 (격리 안 거쳐도 정상에서 바로 가능) |
| PA·PF 처리 | 무조건 격리 → BOM 트리 펼침 |
| 분해 살린 자식 | 분해한 부서의 정상 재고로 즉시 입고 |
| 사유 입력 | 모든 액션 필수 (카테고리 + 자유 메모) |
| UI 위치 | 입출고탭 → 요청 작성 → Step 1 [불량] → **불량 처리 허브** |
| 허브 디자인 | 대시보드 패턴 (KPI 카드 + 액션 + 부서별 목록) |
| 대시보드 표시 | 위치별 재고 카드/게이지에 **빨간색 불량 행/구간** |

## 2. 사용자 멘탈 모델 (조직)

사용자가 머릿속에 그리는 회사 조직도:

```
생산부 ─┬─ 튜브
        ├─ 고압
        ├─ 진공
        ├─ 튜닝
        ├─ 조립
        └─ 출하

영업부 (단독)
연구부 (단독)
A/S부  (단독)
기타   (단독)

이필욱 = 생산부 정 부서장
김건호 = 생산부 부 부서장
→ 위 6개 라인 모두에서 올라오는 결재를 두 사람이 처리
```

**현재 코드와의 차이**
- 코드: 부서 enum이 평면(flat). "생산부"라는 상위 개념 자체가 없음.
- 코드: `Employee.department` 가 단일 문자열이라 한 직원이 여러 부서에 속할 수 없음.
- 코드: 결재 권한 체크 [backend/app/services/stock_requests.py:1022-1023](../backend/app/services/stock_requests.py#L1022-L1023)에서 `approver.department != request.requester_department` 문자열 일치 비교 → 이필욱(조립) 이 진공/튜브/고압/튜닝/출하 결재 시 즉시 거부됨.

**해결책 (결정)**: 부서 마스터에 `parent_id` 추가, 결재 권한 체크 시 "자기 부서 OR 상위 부서의 책임자" 로 확장.

## 3. 데이터 모델 변경

### 3.1 `InventoryLocation.status`

```
유지: PRODUCTION / DEFECTIVE 2개만
추가: 없음 (재작업중 별도 상태는 만들지 않음)
```

**이유**:
- 사용자 회사에서 재작업 대기는 1년씩 쌓이지만, "작업 중인지"를 시스템이 알 필요 없음. 격리해두고 완료 시 결과만 입력.
- `DEFECTIVE = 재작업 대기 풀` 로 통합.
- 격리 일자(`InventoryLocation.created_at` 또는 별도 컬럼)를 화면에 표시해서 1년 묵은 거 추적.

### 3.2 `Department` 마스터에 `parent_id` 추가

```sql
ALTER TABLE departments ADD COLUMN parent_id INTEGER REFERENCES departments(id) NULL;
```

초기 데이터:
- `생산부` 신규 추가 (parent_id NULL)
- 튜브 / 고압 / 진공 / 튜닝 / 조립 / 출하 → parent_id = 생산부 id
- 영업 / 연구 / AS / 기타 → parent_id NULL (단독)

`DepartmentEnum` ([backend/app/models.py:78-88](../backend/app/models.py#L78-L88))도 같이 업데이트.

### 3.3 결재 권한 체크 로직

[backend/app/services/stock_requests.py:1022-1023](../backend/app/services/stock_requests.py#L1022-L1023)

**변경 전**:
```python
if approver.department != request.requester_department:
    raise PermissionError("다른 부서의 요청은 결재할 수 없습니다.")
```

**변경 후 (의사 코드)**:
```python
# 라인(튜브/고압/진공/튜닝/조립/출하) 자체에는 부서장이 없으므로
# (1)은 영업/연구/AS 같은 단독 부서에서만 실제로 매칭됨.
# 사실상 라인 결재의 정상 경로는 (2) 생산부장 또는 (3) 창고장.

# 1) 같은 부서 (단독 부서 한정)
if approver.department == request.requester_department:
    pass  # OK
# 2) 상위 부서의 책임자 = 생산부장이 라인 결재
elif is_ancestor_department(approver.department, request.requester_department):
    pass  # OK (예: 생산부장 → 진공부 결재)
# 3) 창고 정/부도 결재 가능 (이번 결정에 따라)
elif approver.warehouse_role in ("primary", "deputy"):
    pass  # OK
else:
    raise PermissionError(...)
```

`is_ancestor_department(a, b)` = a가 b의 parent 체인 어딘가에 있으면 True.

> 주의: **라인에는 `department_role=primary/deputy` 가 없음**. 즉 라인 결재자 풀은 비어 있고, 결재는 항상 (2) 생산부장 또는 (3) 창고장 경로로만 처리됨. 이필욱·김건호의 `department` 는 `조립` 이 아니라 **`생산부`** 로 시드 데이터를 변경해야 함.

### 3.4 `Employee` 시드 데이터

이필욱·김건호의 `department`를 `조립` → `생산부` 로 변경 (또는 `department_role` 의미를 "이 부서 + 산하 부서 모두"로 재정의).

## 4. 부서 조직 계층 도입 (옵션 A)

### 4.1 왜 옵션 A인가

대안:
- 옵션 B: Employee에 `managed_departments` JSON 컬럼 추가 (수동 관리)
- 옵션 C: 결재 코드만 핫픽스

옵션 A를 선택한 이유:
1. **사용자 멘탈 모델과 100% 일치** — 실제 회사 조직도가 시스템에 반영됨
2. **새 부서장 등록 시 자동** — "생산부장"으로만 등록하면 산하 모든 라인 결재 권한 자동
3. **다른 영역에도 확장 가능** — 향후 영업1팀/2팀 같은 산하 조직 생기면 자연 처리
4. **메모리 [project_dept_approval_routing_bug] 의 본질을 해결** — 증상 가림이 아니라 근본 해결

```

**현재**: 부서별 색상으로 누적 게이지

**변경 후**: 동일 패턴에 **빨간색 불량 구간** 추가

- **총 보유** = 정상 + 격리 (참고용)

```

### 6.3 디자인 원칙

- **대시보드 패턴 차용** — 사용자가 이미 익숙한 레이아웃
- 상단 **KPI 카드 4개** = 한눈에 상황 파악, 누르면 자동 필터링
- **액션 버튼 3개** = 자주 쓰는 신규 액션
- 목록 = **부서별 접고 펼치기** (부서 단위 사고에 맞춤)
- **⚠1년** 같은 경고 배지로 1년 묵은 거 자동 강조
- 각 항목 **[처리] 버튼** → R / PA·PF 에 따라 위자드 자동 분기

### 6.4 진입/권한

- 허브 진입: **누구나** (목록은 read-only로 모두에게 노출)
- 액션 버튼: 권한에 따라 활성/비활성
- 부서 필터 기본값: **내 부서** (사용자 본인 부서 우선), 토글로 생산부 전체 / 전체 확장

## 7. 불량 처리 흐름 상세

### 7.1 격리 (즉시)

**누가**: 누구나 (작업 중 불량 발견하면 즉시 누름)
**언제**: 4가지 모든 순간에 발생
- 창고 입고 검수 중
- 부서 작업 중
- 부서 자체 검사
- 출고 직전 검사

**결과**:
- `InventoryLocation` 의 `status` 가 `PRODUCTION` → `DEFECTIVE`
- `MARK_DEFECTIVE` 트랜잭션 기록 (입출고 내역에 자동 표시)
- **사유 필수** (카테고리 + 자유 메모)
- 결재 없음 → 즉시 반영

### 7.2 R (원자재) 처리

원자재는 BOM이 없으니 분해 개념 없음. 단순 4가지 액션.

**격리 안 거치고 정상에서 바로 가능한 액션**:
- 공급처 반품 (외부에서 사 온 거라 가능)
- 폐기

**격리에서 가능한 액션**:
- 정상 복귀 (재검사 통과)
- 폐기
- 공급처 반품

**처리 화면**:
```
[처리] 7-MAT-0001 텅스텐 와이어 × 5개
조립부 [불량] / 격리 2025-08-15 / 격리자: 김건호
사유: 외관 불량 (스크래치 다수)
─────────────────────────────────────────
[정상 복귀]  [폐기]  [공급처 반품]
       ↓ 누르면 사유 입력 + 결재 필요한 액션은 결재 흐름 시작
```

### 7.3 PA·PF (조립품·완제품) 처리

조립품/완제품은 분해 개념 있음. BOM 트리 펼쳐서 라인별 결정.

**격리만 가능** (공급처 반품은 의미 없음 — 내가 만든 거니까)

**처리 화면**:
```
[처리] 7-TR-0001 전극 어셈블리(70kV) × 5개
조립부 [불량] / 격리 2025-08-15 / 격리자: 김건호
사유: 기능 불량 (출력 부족)
─────────────────────────────────────────
이 격리 항목을 어떻게 할까요?

│ ○ 정상 복귀  (잘못 격리한 거였음)
│ ○ 전부 폐기  (BOM 통째로 버림)
│ ● 분해 + 자식 처리  ← 선택 시 아래 트리 펼침
─────────────────────────────────────────
[BOM 트리 (분해 결과 결정)]
  본인  전극 어셈블리 × 5  →  분해됨 (소멸)
  ├ 필라멘트          × 5  →  [정상 입고 ▼]   살림
  ├ 게터              × 5  →  [폐기 ▼]
  └ 애노드            × 5  →  [정상 입고 ▼]   살림

각 라인별 사유 입력:
  ├ 필라멘트: 검사 통과
  ├ 게터: 진공 불량으로 사용 불가
  └ 애노드: 검사 통과

[결재 요청 →]
```

**분해 결과**:
- 본인(PF) = 소멸 (DISASSEMBLE 트랜잭션)
- 살린 자식 = **분해한 부서의 정상 재고로 즉시 입고** (PRODUCTION 상태)
- 폐기 자식 = SCRAP 트랜잭션

### 7.4 액션별 결재 매트릭스

| 액션 | 대상 | 발의자 | 결재 필요 | 결재자 |
|---|---|---|---|---|
| 격리 | 정상 → 불량 | 누구나 | × | — |
| 정상 복귀 | 불량 → 정상 | 누구나 | × | — |
| R 정상 → 공급처 반품 | R 정상 재고 | 누구나 | ○ | 생산부장(이필욱·김건호) OR 창고장(정/부) |
| R 정상 → 폐기 | R 정상 재고 | 누구나 | ○ | 동일 |
| R 격리 → 폐기 | R 격리 | 누구나 | ○ | 동일 |
| R 격리 → 공급처 반품 | R 격리 | 누구나 | ○ | 동일 |
| PA·PF 격리 → 전부 폐기 | PA·PF 격리 | 누구나 | ○ | 동일 |
| PA·PF 격리 → 분해 처리 | PA·PF 격리 | 누구나 | ○ | 동일 |
| PA·PF 격리 → 정상 복귀 | PA·PF 격리 | 누구나 | × | — |

**핵심 원칙**:
1. 격리·정상복귀는 가벼움 → 즉시
2. 돈/외부 관계 걸린 결정(폐기·반품·분해)은 결재
3. **라인(튜브/고압/진공/튜닝/조립/출하) 자체에는 별도 부서장이 없음** — 생산부장(이필욱·김건호)이 6개 라인 결재를 통째로 처리. 코드 측면에서는 `department_role=primary/deputy` 가 "생산부" 단위로만 부여되고, 라인 단위 결재자 풀은 비어 있음.
4. 결재자는 **생산부장(이필욱·김건호) OR 창고장(정/부)** — 두 풀 중 누구 한 명이면 OK

## 8. 사유 입력

### 8.1 정책

**모든 액션에 사유 입력 필수**.

### 8.2 형식

- **카테고리 선택** (필수): 외관 불량 / 치수 불량 / 기능 불량 / 검사 통과 / 기타
- **자유 메모** (선택): 보충 설명

카테고리는 일단 5~6개로 시작, 실제 운영하며 추가.

### 8.3 표시

- 격리 시 입력한 사유 → 처리 화면에 자동 표시 (결재자가 판단 근거로 사용)
- 처리 시 입력한 사유 → 입출고 내역에 영구 기록

## 9. 입출고 내역 기록

모든 격리·처리·반품·폐기·분해 액션은 **자동으로 트랜잭션이 생성**되어 입출고 내역에 표시됨.

| UI 액션 | 생성되는 트랜잭션 |
|---|---|
| 격리 | `MARK_DEFECTIVE` |
| 정상 복귀 | `MARK_DEFECTIVE` 역방향 (또는 신규 `UNMARK_DEFECTIVE` 추가 검토) |
| 폐기 | `SCRAP` |
| 공급처 반품 | `SUPPLIER_RETURN` |
| 분해 + 자식 처리 | `DISASSEMBLE` + (자식별 PRODUCTION 또는 SCRAP) |

## 11. 마이그레이션 / 호환성 고려사항

### 11.1 기존 격리 데이터
- 현재 `InventoryLocation.status = DEFECTIVE` 행이 이미 존재할 수 있음
- 마이그레이션 시 격리 일자가 없으면 `created_at` 또는 `MARK_DEFECTIVE` 트랜잭션 시각을 채워 넣기

### 11.2 기존 부서 결재 흐름
- `Employee.department` 단일 컬럼 그대로 두고, parent_id만 사용해서 권한 체크 확장
- 이필욱·김건호 소속만 "생산부" 로 변경 (또는 마이그레이션 스크립트로 자동)

### 11.3 기존 [불량] 워크타입 코드
- 현재 워크타입 [불량] → 서브탭 [격리] / [공급처 반품] 코드는 **신규 허브로 로직 이동**
- 위자드 진입 시 Step 1에서 [불량] 누르면 라우터가 허브로 redirect
- 기존 위자드 컴포넌트 폐기 (또는 허브의 액션 위자드로 재활용)

### 11.4 OpenAPI 게이트
- 백엔드 스키마 변경 시 `_attic/docs/openapi.json` 재생성 필수
  - 메모리 [project_openapi_ci_gate] 참조

## 12. 미결 / 구현 시 결정할 사항

설계 단계에서 합의 안 된 디테일. 구현 들어가면서 자연스럽게 풀 수 있는 것들.

1. **사유 카테고리 최종 목록** — 일단 5~6개로 시작, 실제 운영하며 추가
2. **분해 시 BOM 비율 자동 계산 여부** — 현재 DISASSEMBLE 로직 확인 후 결정
3. **격리 일자 컬럼 이름/위치** — `InventoryLocation` 에 새 컬럼 vs 트랜잭션 시각 조회
4. **정상 복귀에도 트랜잭션 타입 추가할지** — 신규 `UNMARK_DEFECTIVE` vs `MARK_DEFECTIVE` 역방향
5. **공급처 반품 시 어떤 공급처로 보낼지** — 품목 마스터의 기본 공급처 자동 vs 사용자 선택
6. **결재 거절 시 흐름** — 거절 사유 입력, 발의자에게 알림
7. **부서 간 이동 결합** — 격리 항목을 다른 부서로 이동 후 처리 시나리오 (이번에는 보류)

## 14. 관련 메모리 인덱스

다음 세션에서 일관성 있게 풀려면 메모리에 다음 항목들이 저장되어야 함:

- 사용자 멘탈 모델: **생산부 = 상위, 튜브/고압/진공/튜닝/조립/출하 = 산하**
- 이필욱·김건호 = 생산부 정/부, **산하 6라인 모든 결재 처리**
- 불량 상태는 PRODUCTION/DEFECTIVE 2개만 (재작업중 별도 상태 만들지 마라)
- 격리 = 재작업 대기 풀 통합
- 격리/처리 화면은 입출고탭 안 "불량 처리 허브" (위자드 아닌 대시보드 패턴)
- 결재 권한: 자기 부서 OR 상위 부서장 OR 창고장
