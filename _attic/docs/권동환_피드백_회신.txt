권동환 사원님, 안녕하세요. 생산부 김현우입니다.

휴가 중에도 하나하나 피드백 주셔서 감사합니다.
말씀하신 5가지를 AI를 활용해 코드와 변경 이력을 하나하나 검토해봤고, 그 결과를 항목별로
공유드립니다.

1. 정상복귀 자동화 테스트 (1번·6번 항목)

1번과 6번이 사실 같은 정상복귀 테스트 정비를 두 군데로 나눠 적은 것이라 회귀가
반복되는 것처럼 보였는데, 실제로는 그렇지 않습니다.

2주간 테스트가 깨진 건 대부분 6/9~11 불량 처리 패널을 새로 만들면서 테스트를 새
화면에 맞춰준 것이었고(기능 자체는 정상), 실제 오류는 1건이었습니다. 그것도 화면
쪽은 사유 입력을 선택 항목으로 바꿨는데 백엔드를 같이 맞추지 않아 잠깐 둘이 안 맞았던
것으로, 같은 날 바로 수정했습니다. 직원 서버에는 나가지 않았고, 현재 관련 테스트는
모두 정상 통과합니다.


2. 저장 슬롯 분리 버그 / 데이터 손실 (7번 항목)

표현을 정확히 짚어주셨습니다. "데이터 손실 위험"이라고 적었지만, 검토 결과 영향
범위는 아직 제출하지 않은 임시저장 1건으로 한정됩니다. 확정된 재고나 입출고 기록은
손상될 수 없는 구조였습니다.

임시저장은 실제 재고가 반영되는 경로(제출·결재)를 전혀 거치지 않는 별도 보관함이고,
더구나 백엔드가 확정된 건은 삭제 자체를 거부하도록 막고 있었습니다. 현재 DB를
확인해도 잘못 남은 임시저장은 0건입니다.

그래서 말씀하신 DB 정밀 검증(Data auditing)은 다행히 필요하지 않은 상황입니다.
대신 커밋 메시지를 "데이터 손실 위험"이 아니라 "미제출 임시저장 오삭제 방지"로
정정해 오해 소지를 없애겠습니다.


3. 직원별 품목 순서 (4번 항목)

세 가지 모두 검토해봤는데, 다행히 우려하신 방향으로 이미 설계돼 있었습니다.

- 격리: 순서 데이터는 직원 계정 단위로 완전히 분리 저장됩니다. 한 사람이 순서를
  바꿔도 다른 직원이나 공유 화면엔 영향이 없습니다. 말씀하신 "계정별 Sequence ID
  격리"가 사실상 지금 구조(직원ID+품목ID 키 + 순서값)와 같았습니다.
- 부하: 정렬은 서버가 매번 쿼리하지 않고, 개인 순서를 한 번만 불러와 캐싱한 뒤
  화면에서 처리합니다. 품목 목록 쿼리는 이 순서 테이블을 건드리지 않아 사용자가
  늘어도 조회 비용은 그대로입니다.

현재 규모(품목 약 900여 개·직원 30명 규모)에선 부하 이슈가 없었습니다. 다만 품목이
수천 단위로 커지면 그때 화면 페이징을 손봐야 하므로, 그건 별개 과제로 메모해두겠습니다.


4. 모바일 카트 일괄 전송 (6번 항목)

이 부분은 짚어주신 게 정확했고, 보강하겠습니다.

단건 처리 자체는 트랜잭션이 안전한데(한 건이 실패하면 그 건만 롤백), 말씀하신
불안정한 와이파이에서 타임아웃 후 재시도 상황에 대한 방어가 비어 있었습니다. 서버는
처리했는데 응답만 못 받은 경우 사용자가 다시 누르면 같은 폐기가 한 번 더 실행될 수
있습니다(폐기는 되돌릴 수 없어 특히 주의가 필요합니다). 또 쓰기 요청에 타임아웃이
없어 끊기면 버튼이 오래 멈출 수 있습니다.

다행히 중복을 막는 장치가 백엔드 일부 경로엔 이미 있어서 프론트에서 식별값만 채우면
되는 부분이 있고, 없는 경로는 백엔드를 조금 보강하면 됩니다. ①중복 방지(멱등성)
②쓰기 타임아웃 두 가지를 우선 작업하겠습니다.


5. 모바일 미지원 관리 화면 (7번 항목)

관리 화면은 한 화면에 담기는 정보가 많아 모바일 크기에 그대로 옮기기가 까다롭고,
다 옮기려면 시간이 꽤 걸립니다. 그래서 관리 기능은 PC에서만 쓰고, 모바일에서는
입출고·불량 처리 같은 일반 현장 작업만 하도록 나눴습니다.

다만 말씀하신 대로 사용자가 헷갈리지 않도록, 모바일에 "관리 기능은 PC에서
이용해 주세요" 정도의 안내를 넣고 현장에도 이 구분을 알리겠습니다.

확인할 수 있어서, 바로 보강하겠습니다. 복귀하셔서 코드를 직접 보시면 더 명확하게
논의될 것 같습니다.

가정 일과 병행하시느라 바쁘실 텐데 건강 잘 챙기시기 바랍니다.
