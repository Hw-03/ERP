# DEXCOWIN MES — 작성 표준 (CODING_STYLE)

> 7눈 리뷰([code-review-2026-06-02.md](code-review-2026-06-02.md) §7)를 **"짤 때 따르는 규칙"으로 뒤집은 문서**.
> 권동환 사원 복귀 리뷰의 기준점이며, 새 규칙을 만들지 않고 7눈에서만 파생한다.

## 7가지 원칙

| # | 규칙 | 기준 |
|---|---|---|
| 1 | 파일 맨 위 1줄 설명 | 모듈 docstring 필수 — 이 파일이 뭘 하는지 |
| 2 | 함수는 ≤50줄 | 넘으면 단계·분기를 `_`-prefix 헬퍼로 추출 |
| 3 | 한 함수 = 한 가지 일 | `1) 2) 3)` 번호주석이 필요하면 쪼개라는 신호 |
| 4 | 이름이 의도를 말한다 | `tmp`·`data`·`x`·`val` 같은 모호한 이름 금지 |
| 5 | 인자 ≤4, 중첩 ≤3 | **`db` 세션은 표준 의존성이라 인자 수에서 제외** |
| 6 | 매직넘버는 상수, 주석은 "왜" | 숫자에 이름을, 주석은 "무엇"이 아니라 "왜" |
| 7 | 같은 일은 같은 방식 | 에러는 `http_error`, 반환은 타입 객체(생짜 dict 지양) |

## 좋은 예 / 나쁜 예

**좋은 예** — `services/inv_calc.py` (73줄, 함수당 한 일, 명확한 이름)
`production_total` / `defective_total` / `available` / `_sync_total` 가 각각 단일 책임이고, 위에서 아래로 술술 읽힌다.

**개선한 예** — `routers/production.py` `production_receipt` (168줄 → 2026-06-02 분리)
한 함수가 BOM전개·사전검증·차감·생산입고·로깅 5단계를 처리하던 것을 `_load_and_merge_requirements` / `_preload_components` / `_assert_no_shortage` / `_backflush_components` / `_record_production` 헬퍼로 분리하고 본체는 흐름 조율만 남겼다(동작 보존).

## 적용 범위

- **새 코드**는 처음부터 이 표준으로.
- **기존 코드**는 만질 때 점진 적용 — 대규모 일괄 변경은 회귀 위험이 크므로 **테스트 그물(단위/회귀)을 먼저 깐 뒤** 분리한다.
- **제외:** 동결 영역(`_weekly_sections/`·`weekly_report.py`)·`_archive/`·`_attic/`.

## 시그니처/구조 변경은 신중히

함수 **내부 분리**(헬퍼 추출, 시그니처 무변경)는 안전하다. 그러나 **인자 dataclass화·import 구조 변경** 같은 시그니처/의존성 변경은 호출자 전반에 영향을 주고 회귀 위험이 크므로, 실익이 분명할 때만 별도 작업으로 진행한다. (예: `inv_defective` 인자 정리는 호출자 광범위 영향으로 보류, 분기 중복만 헬퍼로 추출.)
