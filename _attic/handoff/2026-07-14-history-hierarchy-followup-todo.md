## 입출고 내역 목록

### 1. 요청자·승인자·메모 셀의 세 줄 정보를 잘리지 않게 표시
- 화면/영역: PC 입출고 내역 탭 > 목록 > 요청자/승인자/메모가 함께 있는 우측 메타 셀
- 증상: 요청자, 승인자, 메모가 모두 있는 행도 현재 고정 높이와 `overflow-hidden` 때문에 마지막 정보가 잘린다. 해당 열 위아래에는 활용 가능한 세로 여백이 있는데도 한두 줄만 읽힌다.
- 사용자 불편: 누가 요청했고 누가 승인했는지, 사용자 메모가 있는지를 목록에서 한 번에 확인할 수 없다. 메모가 존재해도 잘려서 데이터가 없는 것처럼 보일 수 있다.
- 확정 정책: 메타 셀은 값이 있는 정보만 최대 세 줄로 모두 표시한다. 행의 다른 열과 수직 정렬을 해치지 않는 범위에서 행 높이를 그 정보량에 맞추며, `title`만으로 핵심 정보를 숨기지 않는다.
- 구현 방향: `historyTableHelpers.tsx`의 사람/상태 메타 렌더러에서 고정 `h-10`/`h-11`과 잘림 규칙을 분리한다. 요청자, 승인자, 메모 상태 chip의 줄 수에 맞는 최소 높이와 line-height를 정하고, 메모 전문은 현재 tooltip 보조도 유지한다.
- 수용 기준: 요청자·승인자·메모가 모두 있는 목록 행에서 세 정보가 잘리지 않고 보인다. 요청자만 있는 행은 불필요하게 커지지 않으며, 같은 화면의 다른 열 텍스트와 수직 시작선이 어긋나지 않는다.
- 브라우저 확인 시나리오: 요청자와 승인자가 모두 있고 메모 chip도 있는 행, 요청자만 있는 행, 메모가 긴 행을 나란히 확인한다. 1920px와 일반 데스크톱 폭에서 셀 정보가 잘리지 않는지 확인한다.
- 우선순위: 높음

### 2. 모든 묶음 작업에 대표 행과 실제 영향 하위 행의 공통 계층 문법 적용
- 화면/영역: PC 입출고 내역 탭 > BOM 생산, 품목 전환, 재작업, 출하 준비 묶음 행의 펼침 상태
- 증상: BOM은 대표 행을 누르면 BOM이라는 중간 부모와 구성품 영향이 펼쳐져 작업 구조를 이해하기 쉽다. 품목 전환도 유사한 구조를 쓰지만 하위 역할과 시각 규칙이 다르고, 재작업과 출하 준비는 서로 다른 전용 행 구조를 사용해 같은 종류의 작업 묶음처럼 읽히지 않는다.
- 사용자 불편: 사용자는 작업 유형마다 어떤 행이 부모이고 어떤 행이 실제 재고 변화인지 다시 해석해야 한다. 펼친 뒤 대표 품목, 실행 결과, 구성품 영향의 관계를 빠르게 비교하기 어렵다.
- 확정 정책: 묶음 작업은 모두 `대표 작업 행 -> 실제 재고 영향 행`의 두 단계 구조를 사용한다. 대표 행은 한 번만 보이고, 펼친 행은 해당 작업의 실재 입고·차감만 보여 준다. 중간 묶음이 필요한 BOM만 추가 접힘을 허용하되, 그 외에는 불필요한 부모 행을 만들지 않는다.
- 구현 방향: `HistoryTable.tsx`의 BOM, 참조 묶음, 재작업 분기를 공통 hierarchy contract로 정리한다. 대표 chevron, 하위 행 들여쓰기, 작업/흐름 역할, 품목코드, 부호 수량, 선택·focus 상태를 하나의 row contract로 맞춘다. 기존 BOM 다단계 펼침과 키보드 접근성은 유지한다.
- 수용 기준: BOM, 품목 전환, 재작업, 출하 준비를 각각 펼쳤을 때 첫 행은 대표 작업, 그 아래는 실제 영향이라는 같은 시선 흐름으로 읽힌다. 대표 행의 정보가 하위 행에 중복되지 않고, 하위 행 수량은 모두 실제 `+/- 단위`로 표시된다.
- 브라우저 확인 시나리오: 각 작업 유형을 하나씩 펼쳐 대표 chevron 위치, 하위 행의 대상명 시작선, 역할 라벨, 품목코드, 부호 수량 pill, 선택 강조를 나란히 비교한다.
- 우선순위: 높음

### 3. 품목 전환·출하 준비 하위 행을 실제 재고 흐름 단계로 명확히 그룹화
- 화면/영역: PC 입출고 내역 탭 > 품목 전환 및 출하 준비 묶음 펼침 > 하위 영향 행
- 증상: 품목 전환은 기존품과 변경품의 관계가 펼친 행에서 분산되어 보이고, 출하 준비는 최종 PF/PA 관련 역할명이 현장 작업 관점보다 내부 처리 단계처럼 느껴진다. 두 작업 모두 무엇을 차감하고 무엇을 입고했는지보다 시스템 용어가 먼저 읽힌다.
- 사용자 불편: 작업 결과를 검산하려면 여러 행의 역할 문구를 머릿속에서 조합해야 한다. 기존품에서 변경품으로 바뀌는 방향과 출하 준비로 실제 빠진 품목을 즉시 확인하기 어렵다.
- 확정 정책: 품목 전환은 `기존 품목 차감`, `구성품 회수 입고`, `추가 구성품 차감`, `변경 품목 입고`처럼 실제 재고 흐름을 역할명으로 쓴다. 출하 준비는 `출하품 차감`, `준비품 입고` 등 실제 변화 중심 역할명을 쓰며, PF/PA는 품목코드와 보조 정보로만 남긴다.
- 구현 방향: `historyPresentation.ts`, `historyTableHelpers.tsx`, `historyBatchInterpreter.ts`의 역할 라벨과 정렬 규칙을 목록·우측 상세에 함께 적용한다. 모든 하위 행은 품목명, 품목코드, 역할, 실제 증감 순서를 유지하고, 값이 없는 중간 단계나 대표 행 복제는 렌더하지 않는다.
- 수용 기준: 품목 전환과 출하 준비 묶음을 펼친 뒤 각 행만 읽어도 차감과 입고의 대상·방향을 알 수 있다. `final PA`, `final PF` 같은 내부 용어가 하위 역할의 주 문구로 노출되지 않으며, 우측 상세의 실제 영향도 같은 순서와 역할명으로 보인다.
- 브라우저 확인 시나리오: 구성품 회수와 추가 차감이 모두 있는 품목 전환, PA/PF가 함께 생성되는 출하 준비를 각각 펼친다. 목록과 우측 상세에서 역할명·품목코드·부호 수량 순서가 일치하는지 확인한다.
- 우선순위: 높음

### 4. 재작업을 별도 처리결과 목록이 아닌 대표 작업의 실제 영향으로 정리
- 화면/영역: PC 입출고 내역 탭 > 재작업 묶음 펼침 > 현재 `처리결과` 하위 행 및 우측 상세
- 증상: 재작업은 BOM·품목 전환과 달리 `처리결과`라는 별도 표현을 사용해 대표 작업과 하위 품목 영향의 관계가 약하다. 펼친 행이 재작업의 결과인지, 별도 후속 작업인지 목록만으로 직관적으로 구분하기 어렵다.
- 사용자 불편: 재작업에서 어떤 품목이 회수되고, 어떤 품목이 폐기되거나 차감됐는지를 다른 묶음과 같은 기준으로 읽을 수 없다. 재작업만 별도 UI처럼 보여 작업 이력 비교 속도가 느려진다.
- 확정 정책: 재작업도 대표 행 아래에 실제 영향만 표시한다. 하위 역할은 `회수 입고`, `재사용 입고`, `폐기 차감`, `부품 차감`처럼 결과의 재고 방향을 설명하며, `처리결과`는 독립 부모 행이나 반복 badge로 두지 않는다.
- 구현 방향: `ReworkBatchHeader.tsx`와 `ReworkBatchDetail.tsx`의 전용 header/detail 구조를 공통 하위 영향 row contract에 맞춘다. 재작업의 사유, 요청자, 승인자, 취소 상태는 대표 행과 우측 상세의 메타 정보로 유지하고, 하위 행에는 실제 영향만 둔다.
- 수용 기준: 재작업 묶음을 펼치면 BOM·품목 전환과 같은 깊이와 수량 표현으로 회수·폐기·차감 결과를 비교할 수 있다. 재작업 사유와 취소 동선, 우측 상세의 작업 정보는 사라지지 않는다.
- 브라우저 확인 시나리오: 회수와 폐기가 함께 있는 재작업, 취소 가능한 재작업을 각각 펼친다. 대표 행의 상태·사유와 하위 영향 행의 역할·부호 수량이 중복 없이 분리되는지, 우측 상세와 목록이 같은 결과를 말하는지 확인한다.
- 우선순위: 높음

### 5. 펼친 묶음의 실제 재고 영향 행은 선택 대상으로 만들지 않음
- 화면/영역: PC 입출고 내역 > BOM·품목 전환·출하 준비 묶음 > 펼친 하위 영향 행
- 증상: BOM/참조 묶음의 하위 영향 행은 현재 클릭·키보드 선택이 가능한 행으로 렌더되어 우측 상세를 다시 바꾼다. 반면 재작업의 하위 결과 행은 읽기 전용으로 보인다.
- 사용자 불편: 대표 작업 행을 눌러 묶음을 펼친 뒤에도 각 구성품을 다시 클릭해야 할 것처럼 보여, 같은 재고 영향 목록인데 작업 유형별 조작 방식이 달라 보인다.
- 확정 정책: 대표 작업 행만 선택 및 우측 상세 전환을 담당한다. 펼친 실제 영향 행은 품명·품목 코드·역할·증감만 보여 주는 읽기 전용 행으로 통일한다. BOM 내부의 별도 구성 묶음 접기/펼치기가 필요한 경우에만 그 묶음 헤더에 조작을 남긴다.
- 구현 방향: `HistoryTable.tsx`가 `ReferenceBatchDetail`로 전달하는 하위 로그 선택 콜백을 제거하고, `historyTableHelpers.tsx`의 영향 행에서 `role=button`, `tabIndex`, 클릭/키보드 이벤트, 포인터 커서를 제거한다. 우측 상세는 대표 로그 또는 대표 batch 선택만 사용한다.
- 수용 기준: BOM·품목 전환·출하 준비를 펼쳐도 구성품 영향 행에는 클릭 가능 문법이 없고, 재작업 결과 행과 같은 읽기 전용 행 문법으로 보인다. 대표 행을 클릭했을 때만 우측 패널이 변경된다.
- 브라우저 확인 시나리오: BOM 생산 1건, 품목 전환 1건, 출하 준비 1건을 각각 펼쳐 하위 행의 포인터/키보드 포커스가 없고 우측 상세가 대표 작업 기준으로 유지되는지 확인한다.
- 우선순위: 중간

### 6. 다음 100건 불러오기를 대표 작업 100건 기준으로 변경
- 화면/영역: PC 입출고 내역 > 목록 하단 `다음 100건 불러오기`
- 증상: 현재 페이지 수가 원시 거래 로그 또는 펼친 하위 행까지 포함하는 방식처럼 보여, 한 작업의 구성품이 많으면 사용자가 기대하는 대표 작업 100건보다 적은 수의 작업만 보인다.
- 사용자 불편: `100건`이라는 문구를 보고 대표 작업 100건을 기대하지만, BOM·품목 전환·출하 준비의 하위 영향 행이 페이지 용량을 소모하면 실제로 비교 가능한 작업 수가 크게 줄어든다.
- 확정 정책: 목록의 100건은 대표 작업(부모 행) 100건을 의미한다. 펼친 하위 영향 행과 BOM 구성 행은 페이지 수에 포함하지 않는다. 하나의 묶음은 페이지 경계에서 분리하지 않는다.
- 구현 방향: 원시 로그 응답을 바로 화면 제한에 쓰지 않고, `buildGroups` 결과의 대표 작업 수를 기준으로 누적한다. 부족하면 다음 원시 페이지를 추가로 가져와 묶음을 완성한 뒤 대표 작업 100개를 화면에 내보내고, 이어지는 묶음 시작점은 다음 페이지로 보존한다.
- 수용 기준: 구성품 9개가 있는 BOM 생산 1건도 목록 수에서는 1건으로만 계산된다. `다음 100건 불러오기` 뒤에는 대표 작업이 최대 100개 추가되며, 어느 묶음도 부모와 하위 행이 서로 다른 페이지로 갈라지지 않는다.
- 브라우저 확인 시나리오: 하위 영향 행이 많은 BOM/품목 전환/출하 준비가 섞인 상태에서 추가 로드를 실행하고, 추가된 대표 행 수와 묶음 완결성을 확인한다.
- 우선순위: 높음

### 7. 시스템 자동 처리 표시의 사용자 의미와 위치 정리
- 화면/영역: PC 입출고 내역 > 요청자·승인자·메모 메타 셀 및 상태 chip
- 확인한 현재 동작: `historyPresentation.ts`는 시스템 행위자(`구성품 변경`, `자동 처리`, `시스템`)를 `시스템 처리 · …`으로 요청자 칸에 표시하고, 자동 파생 로그에는 별도로 `자동 처리` chip을 붙인다.
- 사용자 불편: 사람이 요청한 작업인지, 일반적인 시스템 후속 처리인지, 사용자가 추가로 주의하거나 조작해야 하는 예외인지 구분되지 않은 채 같은 셀에 겹쳐 보여 정보가 과하다.
- 정책 확인 필요: 일반적인 구성품 자동 파생 처리는 목록에서 `자동 처리` chip을 숨기고 우측 상세의 처리 정보로만 남길지, 또는 사용자의 추가 조치가 필요한 예외에만 상태 chip으로 남길지 결정한다. 실제 요청자/승인자가 있으면 그 사람 정보는 시스템 표기보다 항상 우선한다.
- 구현 방향: 정책 확정 후 `historyPresentation.ts`의 `isAutomaticLog` chip 생성과 시스템 요청자 대체 규칙을 분리한다. 목록과 우측 상세에 같은 정책을 적용하고, 기존 legacy 데이터의 `구성품 변경` 값은 데이터 수정 없이 표시만 정리한다.
- 수용 기준: 목록 메타 셀은 실제 요청자·승인자·메모만 우선 보여 주며, 시스템 파생 정보는 선택한 정책에 따라 예외 상태 또는 우측 상세에서만 일관되게 확인된다.
- 브라우저 확인 시나리오: 자동 구성품 변경, 실제 요청자가 있는 품목 전환, legacy 시스템 행을 각각 확인해 요청자 표시와 상태 chip이 정책대로 구분되는지 확인한다.
- 우선순위: 중간

### 8. 취소된 품목 전환·묶음과 하위 영향 행에 취소 상태 문법 적용
- 화면/영역: PC 입출고 내역 > 취소 또는 무효화된 품목 전환·묶음 대표 행 및 펼친 실제 재고 영향 행
- 증상: 취소된 작업도 일반 작업과 같은 강도로 읽혀, 해당 이력이 현재 유효하지 않다는 점을 목록에서 즉시 구분하기 어렵다.
- 사용자 불편: 취소 이력을 현재 유효한 재고 작업으로 오해할 수 있고, 대표 작업과 그 하위 영향의 무효 상태를 함께 파악하기 어렵다.
- 확정 정책: 취소 또는 무효화된 대표 행과 하위 영향 행은 일반 작업보다 흐리게 표시하고, 대표 품명과 수량 요약에는 취소선을 적용한다. 선택·호버·포커스는 취소 상태보다 우선하되 취소선은 유지한다. 이는 재고를 되돌리는 표시가 아니라 해당 이력이 현재 유효하지 않음을 나타내는 시각 문법이다.
- 구현 방향: 공통 대표 행과 실제 영향 행의 취소 상태 입력을 통일하고, 묶음의 취소 상태가 하위 영향 행에도 전달되도록 한다. 기존 선택·호버·포커스 색상 계약은 유지한다.
- 수용 기준: 취소된 품목 전환·묶음을 펼치면 대표 행과 모든 하위 영향 행이 흐리게 보이고, 대표 품명과 수량 요약의 취소선은 선택·호버·포커스 중에도 유지된다. 유효한 작업의 표현은 바뀌지 않는다.
- 브라우저 확인 시나리오: 취소된 품목 전환 또는 묶음을 펼친 뒤 일반 작업과 나란히 비교하고, 대표 행과 하위 행의 흐림·취소선 및 선택·호버·키보드 포커스 상태를 확인한다.
- 우선순위: 높음

### 9. 흐름 열에서 실제 재고 증감 설명을 제거
- 화면/영역: PC 입출고 내역 > 묶음 작업 대표·하위 행의 흐름 열
- 증상: `여러 위치` 아래의 `부품 차감 + 완제품 입고` 같은 문구가 수량 열의 실제 증감 정보와 중복된다.
- 사용자 불편: 위치·공정 흐름과 재고 변화가 한 열에 섞여, 각 열을 빠르게 훑을 때 정보가 반복되고 읽기 밀도가 떨어진다.
- 확정 정책: 흐름 열은 `조립`, `여러 위치`, `창고 → 조립` 같은 위치·공정 흐름만 표시한다. 실제 재고 증감 설명은 수량 열에만 둔다.
- 구현 방향: 흐름 라벨 생성 시 재고 증감 문구를 제외하고, 공통 수량 표시가 실제 `+/- 단위`와 역할을 단독으로 전달하도록 목록과 우측 상세의 표시를 정리한다.
- 수용 기준: 흐름 열에는 위치 또는 공정 정보만 남고, `부품 차감 + 완제품 입고`처럼 재고 증감을 반복하는 문구는 표시되지 않는다. 실제 증감은 수량 열만 읽어도 확인할 수 있다.
- 브라우저 확인 시나리오: 조립·여러 위치·창고 간 이동이 있는 묶음 작업을 열어 흐름 열과 수량 열의 정보가 중복되지 않는지 확인한다.
- 우선순위: 중간

### 10. 재작업 내부 메모 태그를 표시에서만 숨김
- 화면/영역: PC 입출고 내역 > 재작업 우측 상세의 메모 영역
- 증상: 우측 상세 메모에 `[rework:scrap_child]`가 사용자 메모처럼 노출된다.
- 확인한 원인: `backend/app/services/dept_adjustment.py:499`가 재작업 하위 폐기 트랜잭션을 구분하기 위해 `notes="[rework:scrap_child]"`를 저장한다. 이는 사용자 메모가 아닌 내부 태그다.
- 사용자 불편: 내부 처리 표식이 사용자 작성 메모처럼 보여 작업의 의미를 오해하게 한다.
- 확정 정책: 프런트 공통 메모 정규화 helper에서 알려진 내부 `[rework:*]` 태그만 숨기고, 사용자 메모와 저장 데이터는 보존한다. 태그만 남은 메모라면 상세 패널의 메모 영역 자체를 표시하지 않는다.
- 구현 방향: 목록과 상세가 공용으로 쓰는 메모 표시 경로에 내부 태그 정규화를 추가한다. API, DB 스키마, 저장된 `notes` 값은 변경하지 않고, 알려지지 않은 태그나 사용자 메모는 그대로 표시한다.
- 수용 기준: `[rework:scrap_child]`만 저장된 재작업 하위 폐기 기록은 메모 영역을 표시하지 않으며, 태그 뒤에 실제 사용자 메모가 있으면 사용자 메모만 표시된다. 저장된 데이터는 그대로 유지된다.
- 브라우저 확인 시나리오: 태그만 있는 재작업 하위 폐기 기록과 태그·사용자 메모가 함께 있는 기록을 우측 상세에서 각각 확인한다.
- 우선순위: 높음

### 11. 수량 열 용어를 실제 재고 영향 중심으로 확정
- 화면/영역: PC 입출고 내역 > 생산·분해·재작업 묶음의 수량 열
- 증상: 현재 생산 묶음의 `완제품 +N EA`, `부품 -N EA`는 품목 마스터 분류가 아니라 대표 생산품과 BOM 구성품을 나눈 내부 요약이어서, 현장 사용자가 실제 재고 동작보다 분류 용어를 먼저 읽게 된다.
- 사용자 불편: 수량 열만 보았을 때 어떤 재고 동작이 일어났는지 즉시 해석하기 어렵고, 분해·재작업과 생산의 표현 기준도 통일되지 않는다.
- 정책 확인 필요: 생산은 `생산 입고 +N EA`, `구성품 차감 -N EA`, 분해·재작업은 `회수 입고 +N EA`, `폐기 차감 -N EA`처럼 실제 재고 영향을 먼저 보여 주는 용어로 바꿀지 사용자와 확정한다.
- 구현 방향: 정책 확정 후 공통 영향 표현의 역할 라벨과 수량 열 요약을 함께 바꾸고, 목록과 우측 상세에 같은 용어를 적용한다. 품목 마스터 분류나 저장 데이터는 변경하지 않는다.
- 수용 기준: 확정된 용어는 생산·분해·재작업의 실제 재고 방향을 일관되게 설명하고, 목록과 우측 상세에서 동일하게 표시된다.
- 브라우저 확인 시나리오: 생산, 분해, 회수·폐기가 섞인 재작업을 각각 열어 수량 열의 용어와 실제 `+/-` 방향이 일치하는지 확인한다.
- 우선순위: 중간
