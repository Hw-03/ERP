# 출하 탭 성능 개선 투두

작성일: 2026-06-29
상태: 구현 및 검증 완료

## 배경

출하 탭 첫 진입 시 화면이 오래 `출하 데이터를 불러오는 중입니다.` 상태에 머무는 문제가 있다. 화면 상단에는 `DB 연결 일시 오류`가 같이 보일 수 있다.

현재 확인한 핵심 원인은 출하 탭 초기 렌더에서 여러 API를 동시에 호출하고, SQLite 개발 DB 설정이 읽기 요청도 `BEGIN IMMEDIATE`로 시작해 요청들이 서로 락 대기하는 구조라는 점이다.

## 확인 근거

- `frontend/app/mes/_components/DesktopShippingView.tsx:188` 부근의 `loadData()`가 초기 로딩에서 아래 API를 `Promise.all`로 동시에 호출한다.
  - `api.getItems({ limit: 2000 })`
  - `api.getShippingRequests()`
  - `api.getShippingHistory()`
- `backend/app/database.py:62` 부근에서 SQLite 트랜잭션 시작 시 `BEGIN IMMEDIATE`를 사용한다.
- `backend/logs/mes.log`에서 `/api/items`, `/api/shipping/requests`, `/api/shipping/history`가 `BEGIN IMMEDIATE`에서 0.5~2초 정도 대기한 `slow_query`, `slow_req` 기록이 확인됐다.

## 개선 투두

1. 출하 첫 화면은 품목 목록 로딩을 기다리지 않고 바로 렌더링한다.
   - 허브, 요청 목록, 준비 중 목록, 이력 목록은 먼저 표시한다.
   - 품목 데이터가 없어도 볼 수 있는 화면은 막지 않는다.

2. `items` 2000개 로딩은 새 요청 작성 진입 시점으로 늦춘다.
   - `출하 요청 -> 새 요청 만들기`를 눌렀을 때만 품목 목록을 불러온다.
   - 이미 불러온 품목 목록은 컴포넌트 상태나 캐시로 재사용한다.

3. 초기 `requests/history` 중복 호출을 정리한다.
   - 개발 모드 effect 재실행, refresh key, 상태 변경으로 같은 API가 두 번 호출되는지 확인한다.
   - 필요하면 abort controller나 stale guard를 둔다.

4. 출하 목록 API 호출 수를 줄인다.
   - 단기안: `getShippingRequests()`와 `getShippingHistory()`를 순차 또는 필요한 화면별 지연 로딩으로 바꾼다.
   - 중기안: 목록 요약 API 또는 상태별 lazy load를 검토한다.

5. DB 락 대기 개선은 별도 백엔드 과제로 분리한다.
   - 읽기 요청까지 `BEGIN IMMEDIATE`를 잡는 현재 설정은 동시 조회에 불리하다.
   - 단, 이 설정은 기존 동시성 보장 의도가 있으므로 출하 탭만 보고 즉시 바꾸지 않는다.
   - 읽기 전용 세션/엔진 분리 또는 특정 조회 라우터의 트랜잭션 전략 변경을 별도 계획으로 검토한다.

## 우선순위

1. `items` 지연 로딩
2. 출하 탭 초기 로딩 화면 즉시 표시
3. 중복 API 호출 제거
4. 출하 목록/API 구조 정리
5. DB 세션 전략 개선 검토


## 디자인 정리 투두

1. 출하 첫 화면의 `작업 선택` 텍스트를 제거한다.
   - 상단 타이틀과 3개 진입 버튼만으로 화면 목적이 충분히 드러나게 한다.
   - 제거된 텍스트 영역만큼 버튼 영역 높이를 키워 본문을 더 꽉 채운다.

2. 출하 첫 화면 3개 진입 버튼을 입출고/불량 탭의 작업 버튼 디자인 문법과 통일한다.
   - `$design` 스킬 기준으로 먼저 기존 디자인 언어와 주변 컴포넌트를 확인한다.
   - 특히 입출고 탭과 불량 탭의 큰 작업 버튼 구조, 여백, 라운드, 그림자, 아이콘/텍스트 위치를 비교한다.
   - 현재 출하 버튼 하단의 짧은 `ㅡ` 모양 강조선은 나쁘지는 않지만, 입출고/불량 탭 버튼과 문법이 달라 보이므로 유지 여부를 재검토한다.
   - 필요하면 강조선 대신 기존 탭들이 쓰는 hover/active/선택/카운트 표현 방식으로 맞춘다.

3. 새 디자인은 토큰 기반으로만 구현한다.
   - `LEGACY_COLORS`, `tint`, CSS 변수, 기존 lucide 아이콘 흐름을 사용한다.
   - inline hex는 추가하지 않는다.
   - 카드 안 카드처럼 보이는 중첩 장식은 피하고, 실제 업무 진입 버튼처럼 보이게 한다.

4. 요청 목록 화면의 상단 중복 헤더를 통합한다.
   - 현재 `뒤로가기 버튼 + 요청 목록 설명`이 들어간 위쪽 칸과, 그 아래 `출하 요청 1건 / 새 요청을 만들거나 기존 요청의 상태를 확인합니다.` 영역이 역할상 겹친다.
   - 위쪽 칸은 제거하는 방향을 우선 검토한다.
   - 뒤로가기 버튼, `요청 목록`, 건수, 설명, `새 요청 만들기`를 하나의 전폭 목록 툴바/헤더 영역으로 합친다.
   - 이렇게 해서 목록 컬럼 영역이 더 위로 올라오고, 화면 세로 공간을 더 넓게 쓰도록 한다.
   - 같은 정리 원칙을 `준비 중 목록`, `출하 이력 목록`에도 적용할 수 있는지 함께 확인한다.

5. 상세/이전 화면도 내용이 짧아도 화면 높이를 채우게 정리한다.
   - 요청 상세처럼 실제 내용이 적은 화면에서도 바깥 작업 패널이 남은 세로 공간을 끝까지 차지하게 한다.
   - 내부 카드나 정보 그리드는 내용 크기만큼만 자연스럽게 보이되, 전체 작업 영역은 `flex-1`, `min-h-0`, `h-full` 계열로 화면 하단까지 이어지게 한다.
   - 아래쪽이 큰 빈 배경으로 끊겨 보이지 않게 하고, 입출고/불량 탭처럼 한 화면 전체가 하나의 작업 영역처럼 느껴지게 한다.
   - 이 원칙은 `요청 상세`, `준비 체크 상세`, `출하 이력 상세`, 이전/뒤로가기 진입 화면에 공통 적용한다.
   - 단, 억지로 모든 내부 박스를 키워 정보 밀도를 망치지 말고, 바깥 작업 패널과 주요 리스트/상세 영역이 남는 높이를 흡수하는 방식으로 처리한다.

6. 요청 상세 화면의 중복 헤더와 내부 ID 노출을 정리한다.
   - 현재 요청 상세 화면의 위쪽 `뒤로가기 + 요청 상세 + UUID · 제품명` 헤더와, 아래쪽 `제품명 + 요청자/생성일 + 상태 배지` 헤더는 역할이 겹친다.
   - 두 영역을 하나의 상세 헤더로 합친다.
   - 상세 헤더에는 뒤로가기, `요청 상세`, 기준 PF/제품명, 요청 상태, 생성일 정도만 노출한다.
   - `4151711f-566a-4ab9-b449-151ec1e04797` 같은 UUID/내부 request id는 사용자 화면에 표시하지 않는다.
   - 필요하면 내부 ID는 개발자용 로그나 data-testid/tooltip이 아닌 곳에만 남기고, 일반 업무 화면에서는 숨긴다.
   - 같은 원칙을 준비 상세/이력 상세에도 적용해 내부 UUID가 사용자에게 드러나는지 점검한다.

7. 요청됨 상태의 요청 상세에서 요청 취소 기능을 추가한다.
   - `요청됨` 박스에서 요청 건을 눌러 상세로 들어갔을 때 `요청 취소` 액션을 제공한다.
   - 요청 취소는 `REQUESTED` 상태에서만 가능하게 한다.
   - `PREPARING`, `PREPARED`, `PICKED_UP` 상태에서는 요청 취소를 노출하지 않거나 비활성화한다.
   - 취소 전 확인 모달을 띄워 실수로 요청이 사라지지 않게 한다.
   - 취소된 요청을 완전 삭제할지, `CANCELLED` 상태로 이력에 남길지는 구현 전에 결정한다.
   - 실무 추적을 생각하면 완전 삭제보다는 취소 상태/이벤트 이력으로 남기는 방향을 우선 검토한다.

## 상태 전환 버그 조사 메모

작성일: 2026-06-29

### 현재 확인된 DB 상태

- `shipping_requests`에는 현재 `REQUESTED` 1건, `PICKED_UP` 1건이 있다.
- 화면에 보이는 요청 `4151711f566a4ab9b449151ec1e04797`은 실제 DB에서도 `REQUESTED` 상태다.
- 이 요청은 `shipping_request_bom_lines` 17건, `shipping_request_checklist_lines` 16건이 생성되어 있다.
- 이벤트는 `REQUEST_CREATED`만 있고 `SENT_TO_PREP`가 없다.
- 같은 날 검증에 사용한 요청 `d38799fa423342c099a9580e6f231988`은 `SENT_TO_PREP -> PREPARED -> PICKED_UP` 이벤트가 있고 정상 진행됐다.

### 코드상 전환 흐름

- 프론트 `DesktopShippingView.tsx`의 `handleSendToPrep()`는 `saveRequest()` 후 `api.sendShippingToPrep(saved.request_id)`를 호출한다.
- 백엔드 `shipping_svc.send_to_prep()`는 `REQUESTED` 상태만 `PREPARING`으로 바꾸고 `SENT_TO_PREP` 이벤트를 남긴다.
- 현재 코드 경로만 보면 전환 성공 시 `PREPARING`으로 이동하는 로직은 존재한다.

### 현재 가설

- `4151711f...` 요청은 예전 `shipping_request_companion_lines.included` ORM 오류가 있던 시점에 만들어진 잔여 요청일 가능성이 높다.
- 당시 `POST /api/shipping/requests`가 DB commit 이후 응답 직렬화 단계에서 실패하면서, 프론트는 저장 실패로 인식했고 `send-to-prep` 호출까지 가지 못했을 수 있다.
- 즉 현재 화면에 남은 요청은 “지금 전환 로직이 항상 실패한다”기보다 “이전 오류 중 생성만 완료되고 전환되지 않은 요청”일 가능성이 높다.

### 버그 수정 준비 투두

1. 재현 테스트를 준비한다.
   - `POST /api/shipping/requests` 성공 직후 `POST /send-to-prep`가 호출되면 상태가 `PREPARING`이 되는지 API 테스트로 고정한다.
   - 이미 `REQUESTED`로 남아 있는 기존 요청을 상세에서 다시 준비 중으로 보낼 수 있는지 프론트 테스트를 추가한다.

2. 요청 상세에서 복구 액션을 제공한다.
   - `REQUESTED` 상세 화면에 `준비 중으로 보내기` 버튼을 추가한다.
   - 지금처럼 요청 생성 후 전환이 끊긴 요청도 상세에서 바로 다시 전환할 수 있어야 한다.
   - `PREPARING` 이상 상태에서는 이 버튼을 숨기거나 비활성화한다.

3. 요청 취소와 함께 잔여 요청 정리 흐름을 만든다.
   - 전환하지 않을 요청은 `요청 취소`로 정리할 수 있게 한다.
   - 취소는 완전 삭제보다 `CANCELLED` 상태 또는 이벤트 이력으로 남기는 방향을 우선 검토한다.

4. 응답 직렬화 실패 방어를 추가 검토한다.
   - DB commit 이후 `_to_response()`에서 실패하면 사용자는 실패로 보지만 DB에는 요청이 남을 수 있다.
   - 출하 요청 생성/수정/전환 API는 commit 이후 응답 변환에서 실패하지 않도록 response mapper 테스트를 강화한다.
   - 특히 companion/checklist/bom relationship 컬럼 불일치 같은 ORM 스키마 오류가 다시 생기지 않게 테스트를 둔다.

5. 현재 잔여 데이터 처리 방침을 정한다.
   - `4151711f...` 요청은 실제 업무 요청이 아니라 테스트/오류 잔여 요청인지 확인 후 취소 또는 준비 중 전환한다.
   - 데이터 직접 삭제는 하지 말고, 화면 기능으로 처리하는 방향을 우선한다.
## 구현 시 주의

- 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.
- 주간보고 frozen 파일은 건드리지 않는다.
- 불량 탭 관련 남은 unstaged 변경과 섞어 커밋하지 않는다.
- 출하 UI는 PC 실사용 흐름을 우선한다.
- 성능 개선 후 실제 브라우저에서 출하 탭 첫 진입 시간을 확인한다.

## 구현 결과

- 출하 첫 진입은 요청 목록만 먼저 불러오고, 품목 2000건 로딩은 새 요청 작성/편집 진입 시점으로 늦췄다.
- 출하 이력은 별도 초기 API 호출 없이 요청 목록의 PICKED_UP 상태에서 파생하도록 정리했다.
- 허브, 요청 목록, 준비 중 목록, 이력 목록, 요청 상세를 전폭 작업 패널 문법으로 정리했다.
- 사용자 화면에서 request UUID 노출을 제거했다.
- 기존 REQUESTED 요청 상세에서 준비 중으로 보내기를 다시 실행할 수 있게 했다.
- REQUESTED 요청 상세에서 확인 모달을 거쳐 요청 취소(삭제)할 수 있게 했다.
- DELETE /api/shipping/requests/{request_id} API와 OpenAPI baseline을 추가/갱신했다.
- 실제 Chrome에서 4151711f... 요청을 PREPARING으로 전환했고, DB 상태도 PREPARING으로 확인했다.
## 추가 디자인 투두 - 허브 버튼 배경 통일

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

### 관찰

- 출하 허브의 3개 큰 버튼은 현재 버튼 내부에 별도 배경/강조 면이 거의 없어, 불량 탭 허브 버튼과 비교했을 때 통일성이 약해 보인다.
- 불량 탭은 버튼 내부에 아이콘·제목·설명 묶음이 하나의 작업 카드처럼 보이도록 배경감과 구조가 잡혀 있는데, 출하 탭은 넓은 빈 카드 위에 텍스트만 놓인 느낌이 강하다.
- 특히 출하 탭의 `바로 열기` 텍스트는 하단에 단독으로 떠 있어, 불량 탭의 하단 설명 영역과 문법이 다르게 느껴진다.

### 개선 방향

1. 출하 허브 버튼 내부에도 불량 탭과 같은 작업 버튼 배경/면 구조를 만든다.
   - 단순히 카드 외곽선만 유지하지 말고, 아이콘·제목·설명·카운트·하단 안내가 한 묶음으로 읽히게 한다.
   - 버튼 내부 정보 영역과 넓은 빈 공간의 관계를 정리해서 “빈 박스”가 아니라 “큰 작업 진입 버튼”처럼 보이게 한다.

3. `바로 열기` 표현은 유지 여부를 재검토한다.

4. `$design` 스킬을 확인한 뒤 구현한다.
   - 기존 디자인 언어와 입출고/불량 탭 허브 버튼을 먼저 비교한다.

## 추가 UX 투두 - 요청 목록 진입/컬럼 높이 정리

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

### 관찰

- 출하 허브에서 `출하 요청`을 클릭해 요청 목록으로 들어간 뒤 브라우저 뒤로가기를 누르면, 출하 허브로 돌아가지 않고 그 이전 탭/이전 화면으로 빠진다.
- 현재 출하 내부 화면 전환이 URL/history 상태와 충분히 동기화되지 않아, 사용자가 기대하는 브라우저 뒤로가기 흐름과 실제 동작이 다르다.
- 요청 목록의 상태 컬럼에서 `준비 중` 컬럼은 실제 요청 row가 짧게 들어가 있고, `요청됨 없음`, `준비 완료 없음` 빈 상태 박스는 더 큰 높이로 보여서 세 컬럼의 시각 균형이 어긋난다.

1. 출하 내부 화면 전환을 브라우저 history와 동기화한다.
   - `hub -> requestList -> requestDetail/prepDetail/historyDetail` 같은 내부 view 전환을 URL query 또는 history state에 반영한다.
   - 브라우저 뒤로가기를 누르면 앱 밖 이전 화면으로 바로 빠지지 않고, 출하 탭 내부 이전 단계로 돌아가게 한다.
   - 예: `?tab=shipping&shippingView=requestList`, `?tab=shipping&shippingView=prepDetail&id=...` 같은 방식 검토.
   - 기존 상단/사이드바 탭 전환 URL 정책과 충돌하지 않게 한다.

2. 요청 목록 상태 컬럼의 row/empty 높이 문법을 통일한다.
   - 빈 상태 박스와 실제 요청 row의 높이 차이가 지나치게 벌어지지 않게 조정한다.
   - `준비 중`처럼 요청이 1건만 있는 컬럼도 너무 작게 떠 보이지 않도록 row 또는 row container 높이를 키운다.
   - 단순히 모든 row를 과하게 키우기보다, 컬럼 안에서 `요청 카드 영역`이 남는 높이를 자연스럽게 채우는 구조를 검토한다.
   - 요청이 여러 건일 때는 스크롤/간격이 업무 목록답게 유지되어야 한다.

3. 같은 원칙을 준비 중 목록과 이력 목록에도 점검한다.
   - 빈 상태와 실제 row가 섞일 때 카드 높이와 여백이 어색하지 않은지 본다.
   - 요청 목록만 고쳐서 다른 목록 화면과 다시 달라지지 않게 한다.

4. 구현 전 확인 포인트
   - 브라우저 뒤로가기/앞으로가기 테스트를 실제 Chrome에서 한다.
   - `출하 허브 -> 출하 요청 -> 요청 상세 -> 뒤로가기` 흐름과 `출하 허브 -> 출하 준비 중 -> 준비 상세 -> 뒤로가기` 흐름을 모두 확인한다.
   - 새 URL query가 새로고침 후에도 적절한 출하 내부 화면을 복원하는지 검토한다.
   - 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.
## 추가 디자인 투두 - 요청 상세 정보 구조 정돈

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

### 관찰

- 요청 상세 화면은 필요한 정보 자체는 대부분 들어가 있지만, 한눈에 읽히는 순서와 그룹 위계가 약해 난잡하게 느껴진다.

- PA 구성품과 PF 구성품 목록은 텍스트 줄이 촘촘하게 쌓여 있어 실제 작업자가 빠르게 훑기 어렵다.
- `카톤·동반 출하품`처럼 내용이 비어 있는 영역도 PA/PF 구성품과 같은 카드 무게를 가져서 화면이 더 복잡해 보인다.
- 하단 수정 버튼이 화면 끝에 떨어져 있어 상세 정보와 액션의 관계가 약하게 느껴진다.

### 개선 방향

1. 상세 화면을 `요약 -> 구성 -> 액션` 흐름으로 다시 정리한다.
   - 상단에는 기준 PF, 상태, 생성일, 요청자 정도만 강하게 보여준다.
   - 최종 PA/PF가 아직 `준비 전`인 경우에는 별도 큰 카드로 과하게 보이기보다, 상태성 정보로 묶어 표시한다.
   - 준비 완료 이후에는 최종 PA/PF가 더 중요한 정보가 되므로 상태별로 강조 수준을 다르게 둔다.

2. PA/PF 구성품 목록의 가독성을 높인다.
   - 단순 텍스트 리스트 대신 품목명, 코드, 수량이 구분되어 보이게 한다.
   - 긴 품목명은 줄바꿈/간격을 정리해서 행 단위로 스캔 가능하게 한다.
   - PA 구성품과 PF 구성품은 구분하되, 두 카드가 같은 구조와 밀도를 갖게 한다.

3. 빈 정보 영역의 시각 무게를 낮춘다.
   - `카톤·동반 출하품`이 비어 있을 때는 큰 카드 하나를 차지하기보다 작고 조용한 안내 상태로 표현하는 방향을 검토한다.
   - 준비 완료/픽업 단계에서 실제 데이터가 생기면 그때 주요 정보 영역으로 승격한다.

4. 액션 위치를 상세 흐름 안으로 끌어올린다.
   - `수정`, `준비 중으로 보내기`, `요청 취소` 같은 액션은 화면 하단 구석에 고립시키지 말고 상세 헤더 또는 하단 고정 액션바 중 하나로 일관되게 배치한다.
   - 상태별로 가능한 액션만 보이게 하되, 사용자가 다음에 뭘 해야 하는지 자연스럽게 보이게 한다.

5. 디자인 기준
   - `$design` 스킬 기준으로 기존 불량/입출고 상세 화면의 정보 밀도와 카드 문법을 비교한다.
   - 새 색상이나 장식으로 해결하지 말고, 간격, 위계, 행 구조, 카드 무게 조절로 정리한다.
   - `LEGACY_COLORS`, CSS 변수, lucide 아이콘만 사용하고 inline hex는 추가하지 않는다.
   - 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.

  - 아래쪽: `출하 요청 수정 / 한 화면에 하나의 작업만 처리합니다.`
- 두 문구는 역할이 거의 같고, 화면 세로 공간을 중복으로 사용한다.
- 작성/수정 화면은 BOM 조정 영역처럼 세로 공간을 많이 쓰는 작업이 많기 때문에, 중복 헤더를 줄이고 작업 영역을 더 넓게 쓰는 편이 맞다.

### 개선 방향

1. 작성/수정 화면의 상단 헤더와 내부 작업 헤더를 하나로 합친다.
   - 뒤로가기 버튼, 화면 제목, 짧은 설명, 상태 배지를 하나의 헤더 영역에 정리한다.
   - 같은 제목을 두 번 보여주지 않는다.
   - `PF 선택 후 PF/PA 구성품을 조정합니다.`와 `한 화면에 하나의 작업만 처리합니다.` 중 하나만 남기거나, 더 짧은 하나의 문장으로 합친다.

2. 단계 네비게이션은 통합 헤더 바로 아래에 붙인다.
   - 헤더를 줄인 만큼 `기준 PF 선택 / BOM 구성 조정 / BOM 매칭 / 요청 정보 / 저장 및 전환` 단계 영역과 실제 작업 영역이 위로 올라오게 한다.
   - 현재 단계 설명은 각 단계 본문 제목에서만 보이게 해서 상단 설명 중복을 줄인다.

3. 화면 높이 확보를 우선한다.
   - BOM 구성 조정 단계처럼 목록이 긴 화면에서 불필요한 위쪽 여백과 중복 설명을 제거한다.
   - 작업 영역 내부 스크롤이 너무 빨리 생기지 않도록 세로 공간을 최대한 확보한다.

4. 같은 원칙을 새 요청 작성과 기존 요청 수정 모두에 적용한다.
   - `출하 요청 작성`, `출하 요청 수정` 두 모드가 같은 레이아웃 문법을 쓰게 한다.
   - 상태 배지는 수정 모드에서만 필요한 경우 자연스럽게 표시한다.

### 결정

- 출하 요청의 `요청자`는 요청 작성자가 직접 입력하지 않는다.

   - 요청 정보 단계에는 메모 등 실제로 바꿀 수 있는 정보만 남긴다.

3. 기존 요청 중 `요청자 없음`인 데이터 표시 방식을 정한다.
   - 새 요청부터는 자동 입력되게 한다.
   - 기존 데이터는 일단 `요청자 없음` 대신 더 조용한 표시로 바꾸거나, 필요하면 현재 로그인 사용자로 보정할지 별도 판단한다.

4. 테스트/검증 항목
   - 로그인 사용자 상태에서 새 요청을 만들면 `requested_by_name`이 자동 저장되는지 확인한다.
   - 요청 목록 row에 `요청자 없음` 대신 로그인 사용자 이름이 표시되는지 확인한다.
   - 요청 수정 시 요청자가 임의로 바뀌지 않는지 확인한다.

- 출하 요청 작성/수정 화면의 `PF 선택`, BOM 라인 품목 선택, 동반 출하품 선택이 현재 네이티브 `<select>`로 보인다.

- 특히 PF 목록처럼 항목이 긴 선택 UI는 기본 select로 쓰면 탐색성이 떨어지고, 화면을 덮는 느낌이 강하다.

### 기존 컴포넌트 후보

- `frontend/app/mes/_components/common/AppSelect.tsx`
  - 관리자/입출고/불량 쪽에서 쓰는 공용 select 컴포넌트.
  - role=combobox 기반의 커스텀 드롭다운이라 네이티브 select보다 제품 톤과 맞는다.
- `frontend/app/mes/_components/_warehouse_v2/_atoms.tsx`의 `LabeledSelect`
  - `AppSelect`를 감싼 라벨형 선택 컴포넌트.
  - 입출고/불량 필터에서 이미 쓰는 문법이라 출하 화면에도 맞출 수 있다.
- 긴 품목 선택은 `IoTargetPicker` / `DefectItemPicker`의 검색·필터·정렬 문법도 참고한다.

### 개선 방향

1. 출하 화면의 네이티브 `<select>`를 기존 공용 선택 컴포넌트로 교체한다.
   - 단순한 작은 선택은 `AppSelect` 또는 `LabeledSelect`를 우선 사용한다.
   - 새 bespoke dropdown을 만들지 않는다.

2. PF 선택은 긴 목록에 맞는 검색형 선택 UX로 정리한다.
   - PF 목록은 항목 수가 많으므로 단순 드롭다운만으로 충분한지 확인한다.
   - 필요하면 `AppSelect`를 그대로 쓰기보다 기존 품목 선택기 검색 패턴을 가져와 `PF 검색 + 선택 결과 리스트` 형태로 다듬는다.
   - 현재 별도 `PF 검색` input과 `PF 선택` select가 중복되어 보이므로 하나의 검색형 선택 흐름으로 합치는 방향을 우선 검토한다.

3. BOM 구성품 선택도 같은 문법으로 통일한다.
   - PA/PF 구성품 행의 품목 선택 dropdown도 공용 컴포넌트를 사용한다.
   - 추가 품목, 제외/다시 포함, 수량 입력과 시각적으로 어긋나지 않게 행 높이와 포커스 스타일을 맞춘다.

4. 디자인/검증 기준
   - `$design` 기준으로 기존 입출고/불량/관리 탭의 선택 컴포넌트 톤을 먼저 확인한다.
   - token 기반 색상만 사용하고 inline hex는 추가하지 않는다.
   - 키보드 조작, focus ring, 빈 값 placeholder, 긴 텍스트 overflow를 확인한다.
   - 실제 Chrome에서 PF 선택 dropdown이 화면을 과하게 덮지 않는지 확인한다.
   - 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.

## 추가 UX 투두 - 출하 요청 목록 헤더/컬럼 밀도 정리

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

### 관찰

- 요청 목록 화면에서 현재 첫 번째 상태 컬럼 제목이 `요청됨`으로 표시되어 있는데, 업무자가 보는 단계명으로는 `출하 요청`이 더 자연스럽다.
- `새 요청 만들기` 버튼이 요청 목록 헤더 오른쪽 끝에 따로 떨어져 있어, “출하 요청을 보고 새 요청을 만든다”는 흐름이 한 상자 안에서 묶여 보이지 않는다.
- `요청됨 없음 / 표시할 출하 요청이 없습니다.` 빈 상태 박스는 높이가 충분한데, 실제 요청이 들어 있는 `준비 중` row 박스는 상대적으로 낮아서 3개 컬럼의 시각 리듬이 어긋난다.

### 개선 방향

1. 요청 목록의 첫 번째 상태 컬럼 라벨을 `요청됨`에서 `출하 요청`으로 변경한다.
   - 상태 값 자체(`REQUESTED`)나 API 상태명은 바꾸지 않는다.
   - 화면 표시 라벨만 업무 흐름에 맞게 바꾼다.

2. `새 요청 만들기` 버튼을 `출하 요청` 텍스트 옆으로 이동한다.
   - 버튼은 요청 목록 바깥 우측 끝에 따로 두지 않는다.
   - `출하 요청` 컬럼/상자 안에서 제목과 같은 맥락으로 보이게 한다.
   - 새 요청 생성이 “출하 요청 목록에서 시작하는 주 액션”처럼 보여야 한다.

3. 요청 목록의 실제 요청 row 높이를 빈 상태 박스 리듬에 맞게 키운다.
   - `준비 중`에 요청이 1건 있을 때 row가 너무 얇아 보이지 않게 한다.
   - 빈 상태 박스와 동일 높이로 강제할지, row container만 자연스럽게 키울지는 구현 시 화면을 보며 판단한다.
   - 목표는 3개 컬럼이 같은 업무 보드처럼 안정적으로 보이는 것이다.

4. 같은 패턴을 다른 상태 컬럼에도 일관되게 적용한다.
   - `출하 요청`, `준비 중`, `준비 완료` 모두 row/empty 높이 문법이 어긋나지 않게 한다.
   - 실제 요청이 여러 건일 때는 과도하게 큰 row 때문에 목록성이 떨어지지 않도록 내부 간격과 스크롤을 함께 확인한다.

### 확인 기준

- `출하 요청` 컬럼 제목 옆에 `새 요청 만들기`가 같은 상자 안에서 보인다.
- 화면 상단의 별도 `새 요청 만들기` 버튼은 사라진다.
- `요청됨 없음` 빈 상태 박스와 `준비 중` 실제 요청 row가 시각적으로 비슷한 무게와 높이 리듬을 가진다.
- 실제 Chrome에서 요청 0건/1건/여러 건 상태를 각각 확인한다.
- 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.

## 추가 UX 투두 - 요청 상세 취소 범위와 상단 요약 재배치

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

### 관찰

- 현재 `요청 취소` 버튼은 구현되어 있지만, 사용자가 기대한 위치와 범위가 아니다.
- 사용자의 의도는 `요청됨` 상태뿐 아니라, 이미 `준비 중`으로 넘어간 요청도 상황에 따라 취소하거나 되돌릴 수 있어야 한다는 것이다.
- 요청 상세 화면 상단에 `기준 PF / 최종 PA / 최종 PF` 3개 요약 박스가 별도 줄로 크게 차지하고 있어, 실제로 중요한 구성품 목록 영역이 아래로 밀린다.
- 현재 화면에서는 `요청 상세 / 제품명 · 생성일` 헤더와 3개 요약 박스가 정보상 겹치는 느낌이 있다.

### 개선 방향

1. 출하 요청 취소/되돌리기 범위를 다시 정의한다.
   - `REQUESTED`: 요청 취소 가능.
   - `PREPARING`: 준비 중인 요청도 취소 또는 요청 단계로 되돌릴 수 있어야 한다.
   - `PREPARED`: 이미 준비 완료된 요청은 재고 반영이 들어갔을 수 있으므로, 기존 `준비 완료 취소` 흐름과 연결해서 처리 범위를 판단한다.
   - `PICKED_UP`: v1에서는 픽업 완료 취소를 만들지 않는 기존 결정 유지 여부를 다시 확인한다.

2. 요청 상세 화면에 상태별 액션을 명확히 배치한다.
   - 요청 상세에서 `수정`과 함께 `요청 취소` 또는 `준비 취소/요청으로 되돌리기` 계열 액션이 보이도록 한다.
   - 버튼은 화면 하단 구석에 떨어뜨리지 말고 상세 헤더 또는 액션 영역에 모은다.
   - 각 상태에서 가능한 액션만 보여준다.
   - 재고에 영향을 주는 상태 취소는 확인 모달에서 영향 범위를 명확히 알려준다.

3. `기준 PF / 최종 PA / 최종 PF` 요약을 요청 상세 헤더 줄로 끌어올린다.
   - 현재 별도 3박스 구조를 줄이고, `요청 상세 / 제품명 · 생성일`과 같은 상단 라인 또는 바로 옆 요약 영역에 배치한다.
   - `기준 PF`는 제품명과 중복되므로 과하게 큰 박스로 반복하지 않는다.
   - `최종 PA`, `최종 PF`가 아직 `준비 전`이면 작은 상태 정보로 표현한다.
   - 준비 완료 이후 최종 PA/PF가 확정되면 그때 더 강조해도 된다.

4. 구성품 영역을 더 크게 쓴다.
   - 상단 요약 박스가 줄어든 만큼 `PA 구성품`, `PF 구성품`, `카톤·동반 출하품` 영역을 더 넓고 읽기 좋게 만든다.
   - 구성품 목록은 품목명/코드/수량이 행 단위로 잘 스캔되게 정리한다.
   - 카톤·동반 출하품이 비어 있을 때는 큰 박스보다 작은 안내 상태로 충분하다.

### 확인 기준

- `요청됨` 상세에서 요청 취소가 보인다.
- `준비 중` 상세에서도 사용자가 준비를 취소하거나 요청을 정리할 수 있는 액션이 보인다.
- 상태별로 취소/되돌리기 액션의 의미가 혼동되지 않는다.
- 상단 `기준 PF / 최종 PA / 최종 PF` 3박스가 별도 큰 줄을 차지하지 않는다.
- 구성품 목록 영역이 현재보다 더 넓고 크게 보인다.
- 재고 영향이 있는 취소는 확인 모달에서 충분히 설명된다.
- 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.

## 추가 UX 투두 - 출하 요청 작성 단계바/스크롤 정리

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

### 관찰

- 출하 요청 작성 화면에서 `1~5단계` 단계 버튼들이 `출하 요청 작성 / 한 화면에 하나의 작업만 처리합니다.` 헤더 아래에 별도 줄로 내려와 있다.
- 이 때문에 상단 공간을 많이 차지하고, 실제 작업 영역이 아래로 밀린다.
- 현재 작성 화면에서는 페이지 전체가 세로 스크롤되고 있어, 단계형 작업 화면인데도 화면이 고정된 작업대처럼 느껴지지 않는다.
- 특히 PF 선택 단계에서 리스트와 하단 안내/액션 영역이 동시에 보이지 않거나, 불필요하게 전체 페이지가 스크롤되는 느낌이 있다.

### 개선 방향

1. `1~5단계` 단계 버튼들을 작성 헤더와 같은 줄에 배치한다.
   - 왼쪽에는 뒤로가기, 아이콘, `출하 요청 작성/수정`, 짧은 설명을 둔다.
   - 오른쪽 또는 같은 행의 남는 공간에 `1 기준 PF 선택 / 2 BOM 구성 조정 / 3 BOM 매칭 / 4 요청 정보 / 5 저장 및 전환`을 가로로 배치한다.
   - 단계 버튼은 너무 커지지 않게 하되, 현재 단계와 완료 단계는 명확히 구분한다.
   - 좁은 PC 폭에서는 줄바꿈되더라도 화면을 과하게 밀지 않도록 한다.

2. 출하 요청 작성 화면은 페이지 전체 스크롤이 생기지 않게 정리한다.
   - 화면 전체는 상단바 아래 남은 높이를 채우는 고정 작업 영역처럼 동작해야 한다.
   - 스크롤이 필요하면 페이지가 아니라 현재 단계의 내부 리스트 영역에서만 발생하게 한다.
   - PF 선택 결과 리스트, BOM 구성품 목록처럼 긴 데이터 영역은 내부 스크롤을 사용한다.
   - 하단 `이전/다음` 또는 `저장/전환` 액션 영역은 가능하면 항상 보이게 한다.

3. PF 선택 단계의 높이 배분을 다시 잡는다.
   - 검색창, PF 리스트, 안내 문구, 하단 액션이 한 화면 안에서 자연스럽게 보이게 한다.
   - PF 리스트가 길면 리스트 영역만 스크롤된다.
   - `기준 PF를 먼저 선택하세요` 안내 박스가 화면을 불필요하게 크게 차지하지 않도록 한다.

4. BOM 구성 조정 단계도 같은 원칙으로 처리한다.
   - PA/PF 구성품 목록은 내부 스크롤로 제한한다.
   - 전체 페이지가 아래로 길게 늘어나지 않게 한다.
   - 구성품 추가/제외/수량 수정 작업이 PC 화면에서 바로 보이고 조작 가능해야 한다.

### 확인 기준

- 작성/수정 화면에서 단계 버튼들이 헤더와 같은 줄에 배치되어 상단 높이가 줄어든다.
- 출하 요청 작성 화면 진입 시 브라우저 페이지 전체 스크롤이 생기지 않는다.
- PF 리스트가 길어도 리스트 영역만 스크롤된다.
- 하단 이동/저장 액션이 화면 밖으로 밀리지 않는다.
- BOM 구성 단계에서도 페이지 전체가 아니라 구성품 목록 영역만 스크롤된다.
- 실제 Chrome PC 화면에서 1920px 폭 기준으로 검증한다.
- 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.

### 보강 메모 - 스크롤 제거의 실제 의도

- 사용자의 의도는 단순히 CSS로 스크롤을 막는 것이 아니다.
- `1~5단계` 각각에서 작업자가 판단해야 하는 핵심 정보, 입력 필드, 다음 액션이 한 화면 안에 들어와야 한다.
- 즉, 화면을 억지로 잘라서 스크롤만 없애는 것이 아니라, 각 단계의 정보량과 배치를 다시 설계해야 한다.
- PF 선택, BOM 구성 조정, BOM 매칭, 요청 정보, 저장 및 전환 모두 같은 기준을 적용한다.
- 긴 목록은 필요한 만큼 요약/검색/필터/내부 리스트 높이 조정으로 다루되, 작업자가 다음 행동을 하기 위해 페이지 아래로 내려가야 하는 구조는 피한다.
- 하단 액션 버튼이 항상 보여야 하고, 현재 단계에서 필요한 판단 정보도 같은 화면 안에서 확인 가능해야 한다.
- 구현 시 각 단계별로 “이 단계에서 꼭 봐야 하는 정보는 무엇인가?”를 먼저 정하고, 부가 정보는 접기/요약/내부 스크롤/검색 결과 제한 등으로 줄인다.

### 보강 메모 - 내부 스크롤 기준

- 목록 자체가 길어지는 경우까지 전부 한 화면에 펼치라는 뜻은 아니다.

- 중요한 기준은 브라우저/페이지 전체가 스크롤되는 것이 아니라, 현재 단계의 작업 박스 안에서 필요한 목록만 스크롤되는 구조다.
- 검색창, 단계 제목, 핵심 요약, 하단 액션은 고정적으로 보이고, 긴 리스트 영역만 내부에서 움직이는 것이 목표다.
- 구현 시 각 단계의 컨테이너는 남은 높이를 계산해 고정하고, 리스트 영역에는 `min-h-0`, `overflow-y-auto`, 적절한 max height를 적용한다.

## 추가 UX 투두 - BOM 매칭 단계 버튼 제거/자동화 정리

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

### 결정

- `동일 BOM 확인`은 BOM 변경 시 자동으로 실행되는 흐름이므로 별도 버튼으로 둘 필요가 없다.
- `기본 BOM 다시 불러오기` 버튼은 v1 출하 요청 작성 흐름에서 제거한다.

### 관찰

- 현재 3단계 `BOM 매칭` 화면에 `동일 BOM 확인` 버튼과 `기본 BOM 다시 불러오기` 버튼이 보인다.
- 그런데 동일 BOM 후보 확인은 사용자가 버튼을 눌러서 하는 작업이 아니라, BOM이 바뀌는 순간 자동으로 알려주는 것이 기존 결정이었다.
- 기본 BOM은 1단계 PF 선택 시 이미 불러오고, 이후 사용자는 제외/추가/수량 수정으로 커스텀한다.
- 중간에 `기본 BOM 다시 불러오기` 버튼이 있으면 사용자가 작성한 커스텀 변경을 날릴 수 있고, 작업 흐름도 헷갈린다.

### 개선 방향

1. `BOM 매칭` 단계에서 `동일 BOM 확인` 버튼을 제거한다.
   - BOM 변경 후 자동 매칭 결과만 보여준다.
   - 자동 매칭 중이면 작은 상태 표시나 로딩 문구만 보여준다.
   - 매칭 실패/오류가 있을 때만 재시도 안내를 검토한다.

2. `기본 BOM 다시 불러오기` 버튼을 제거한다.
   - PF를 바꾸면 기본 BOM이 다시 불러와지는 흐름만 유지한다.
   - 이미 수정한 BOM을 초기화하는 기능이 꼭 필요하면 별도 `변경사항 초기화` 같은 명확한 위험 액션으로 다시 설계한다.
   - v1에서는 혼동을 줄이기 위해 제거 우선으로 둔다.

3. BOM 매칭 단계는 결과 확인과 필요한 이름 입력에 집중한다.
   - 동일 BOM이면 “기존 PA/PF 재사용” 결과만 보여준다.
   - PA/PF 이름 입력이 필요한 경우에만 입력 필드를 보여준다.
   - 사용자가 할 일은 `다음`으로 넘어가거나 필요한 새 이름을 입력하는 것뿐이어야 한다.

4. 자동 매칭 트리거를 명확히 검증한다.
   - PF 선택 후 기본 BOM 로드 완료 시 매칭 결과가 표시되는지 확인한다.
   - BOM 라인 제외/다시 포함/추가/수량 변경 시 자동으로 매칭 결과가 갱신되는지 확인한다.
   - 요청 수정 화면에서도 기존 BOM 기준으로 자동 매칭이 다시 동작하는지 확인한다.

### 확인 기준

- 3단계 `BOM 매칭` 화면에 `동일 BOM 확인` 버튼이 보이지 않는다.
- 3단계 `BOM 매칭` 화면에 `기본 BOM 다시 불러오기` 버튼이 보이지 않는다.
- BOM 매칭 결과는 자동으로 표시된다.
- 이름 입력이 필요한 경우에만 새 PA/PF 이름 필드가 보인다.
- 사용자가 커스텀 BOM을 수정해둔 상태에서 실수로 기본 BOM을 다시 불러와 변경사항을 날릴 수 없다.

## 추가 UX 투두 - BOM 구성품 row 드롭다운 제거

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

### 결정

- 이미 BOM 목록에 들어와 있는 구성품 row에는 품목 선택 드롭다운이 필요 없다.
- 구성품 추가는 각 섹션 상단의 `PA 구성품 추가`, `PF 구성품 추가` 같은 추가 액션에서만 처리한다.
- 기존 row에서는 품목을 다른 품목으로 바꾸는 기능을 제공하지 않는다.

### 관찰

- 현재 BOM 구성 조정 단계에서 각 구성품 row의 품목명이 드롭다운처럼 보인다.
- 하지만 업무 흐름상 이미 들어온 구성품은 “다른 품목으로 교체”하는 것이 아니라, 필요 없으면 제외하고 필요한 품목은 새로 추가하면 된다.
- row마다 드롭다운이 있으면 사용자가 기존 구성품을 직접 바꿔도 되는 것처럼 보여 작업 의미가 흐려진다.
- 또한 드롭다운 UI가 row 높이와 시각 복잡도를 키워 BOM 목록을 빠르게 훑기 어렵게 만든다.

### 개선 방향

1. BOM 구성품 row에서 품목 선택 드롭다운을 제거한다.
   - row에는 품목명, 품목 코드, origin 배지(`기본`, `추가됨`, `제외됨`), 수량, 제외/다시 포함 액션만 둔다.
   - 품목명은 읽기 전용 텍스트로 표시한다.
   - 긴 품목명은 줄바꿈/말줄임 기준을 정해 스캔 가능하게 만든다.

2. 품목 추가는 상단 추가 버튼에서만 처리한다.
   - `PA 구성품 추가`, `PF 구성품 추가` 버튼을 누르면 새 품목을 검색/선택하는 흐름을 제공한다.
   - 새로 추가된 품목도 row에서는 품목 교체가 아니라 제외/삭제 또는 다시 추가 흐름으로만 다룬다.
   - 새로 추가한 직후 품목 선택이 필요한 임시 row가 있다면, 선택 완료 전까지만 선택 UI를 노출하고 완료 후에는 읽기 전용 row로 전환한다.

3. 기존 품목 변경은 “제외 + 추가” 흐름으로 명확히 한다.
   - 기본 BOM 품목을 빼고 다른 품목을 넣고 싶으면 기존 품목은 `제외`, 새 품목은 `추가`한다.
   - 이렇게 해야 출하 요청의 변경 이력이 기본 BOM 대비 무엇이 빠지고 무엇이 추가됐는지 명확해진다.

4. BOM row의 작업성을 개선한다.
   - 수량 입력은 유지한다.
   - 제외/다시 포함 버튼은 명확히 보이게 한다.
   - row는 드롭다운 박스처럼 보이지 않고 읽기 쉬운 업무 행처럼 보여야 한다.
   - PA/PF 구성품 목록이 길어도 내부 스크롤에서 빠르게 스캔 가능해야 한다.

### 확인 기준

- 기본 BOM으로 불러온 구성품 row에는 품목 드롭다운이 보이지 않는다.
- 이미 추가된 커스텀 구성품 row에도 품목 드롭다운이 보이지 않는다.
- 구성품을 새로 넣을 때만 검색/선택 UI가 나타난다.
- 기존 구성품을 다른 품목으로 바꾸는 직접 교체 UX는 없다.
- 기본 BOM 대비 제외/추가 내역이 화면에서 명확히 구분된다.
- 자동 BOM 매칭은 included=true 라인 기준으로 계속 정상 동작한다.

## 추가 UX 투두 - 출하 허브 카드 하단 CTA 제거 및 설명 문구 통일

### 관찰

   - 하단에 별도 CTA 버튼처럼 보이는 요소는 두지 않는다.
   - 키보드 접근성과 focus ring은 카드 버튼 자체에서 유지한다.

   - `출하 이력`: 픽업 완료와 재고 로그를 확인합니다.

3. 카드 내부 정보 위계를 정리한다.
   - 상단: 아이콘, 제목, 건수 배지
   - 중단: 짧은 기능 요약
   - 하단: 상세 설명 문구
   - 하단 설명은 버튼처럼 보이지 않게 하고, 불량 탭과 비슷한 텍스트 무게/위치로 맞춘다.

- 카드 하단에는 불량 탭과 비슷한 무게의 설명 문구가 보인다.
- 카드 전체 클릭으로 기존처럼 `출하 요청`, `출하 준비 중`, `출하 이력`으로 진입한다.

- 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.

확인 기준:

### 추가 UX 투두 - 작성 단계 이동 가능 상태 명확화

작성일: 2026-06-30

- 출하 요청 작성 1단계에서 기준 PF를 선택하지 않은 상태라면 `다음` 버튼은 눌릴 수 없어야 한다.

  - 5단계: 저장/준비 중 전환 조건이 충족되지 않으면 해당 액션 비활성

확인 기준:
- 기준 PF 미선택 상태에서 `다음` 버튼이 흐리게 보이고 클릭되지 않는다.
- 기준 PF를 선택하면 `다음` 버튼이 활성화된다.
- 비활성 버튼이 활성 버튼처럼 보이지 않는다.
- 키보드 조작으로도 비활성 단계 이동이 실행되지 않는다.

### 추가 UX 투두 - 5단계 저장 전환 화면에 새 PA/PF 생성 결과 명확히 표시

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

- 3단계 `BOM 매칭`에서 새 PA 이름 또는 새 PF 이름을 입력했다면, 5단계 `저장 및 전환` 화면에도 그 이름이 자연스럽게 반영되어야 한다.
- 현재 5단계는 기준 PF와 PA/PF 구성 개수만 보여줘서, 사용자가 3단계에서 입력한 `REMOTE 제외` 같은 새 이름이 실제로 어떤 결과를 만들지 확인하기 어렵다.
- 5단계에는 “이 요청 저장 시 생성/재사용될 품목”을 명확히 보여준다.
  - 동일 PA 재사용이면: `기존 PA 재사용: {PA 이름}`
  - 새 PA 생성이면: `새 PA 생성 예정: {입력한 새 PA 이름}`
  - 동일 PF 재사용이면: `기존 PF 재사용: {PF 이름}`
  - 새 PF 생성이면: `새 PF 생성 예정: {입력한 새 PF 이름}`
- 3단계에서 새 이름을 수정하면 5단계 요약에도 즉시 같은 이름이 반영되어야 한다.
- “준비 중으로 보내기”를 누르면 지금 보이는 새 PA/PF 이름과 BOM 구성이 저장되고, 실제 준비 완료 시점에 그 기준으로 PA/PF 생성 또는 재사용이 진행된다는 맥락을 안내한다.
- 안내 문구는 너무 개발자식으로 쓰지 말고, 작업자가 이해하기 쉽게 “새 구성으로 저장됩니다”, “준비 완료 때 이 이름으로 품목이 만들어집니다”처럼 표현한다.

확인 기준:
- 3단계에서 새 PA/PF 이름을 입력하면 5단계에 같은 이름이 표시된다.
- 5단계에서 새로 생성될 PA/PF와 재사용될 PA/PF가 구분되어 보인다.
- 5단계의 `마지막 확인` 문구가 실제 저장/전환 결과를 설명한다.
- 이름 입력이 필요 없는 동일 BOM 재사용 케이스에서는 불필요한 새 품목 생성 문구가 보이지 않는다.
- 사용자가 5단계만 보고도 “저장하면 무엇이 새로 만들어지는지” 알 수 있다.

### 추가 UX 투두 - 4단계 요청 정보 레이아웃 정리

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

- 4단계 `요청 정보`에서 요청자는 로그인 사용자로 고정되는 값이므로 입력 필드처럼 보이면 안 된다.
- 요청자 영역은 수정 가능한 input이 아니라 읽기 전용 정보로 표시한다.
  - 예: `요청자 김현우` 또는 작은 정보 칩/요약 행 형태
  - disabled input처럼 보이게 두기보다, 애초에 입력 대상이 아닌 정보처럼 디자인한다.
- 요청 메모는 이 단계에서 실제로 작성하는 핵심 입력이므로 화면의 주 영역을 차지하게 한다.
- 현재처럼 요청자 박스와 요청 메모 textarea가 어색한 2열로 나뉘어 큰 빈 공간이 생기지 않게 한다.
- 추천 배치:
  - 상단: 요청자/작성 시각 같은 고정 정보를 낮은 무게의 요약 줄로 배치
  - 본문: 요청 메모 textarea를 넓게 배치
  - 필요하면 메모 예시/도움말은 textarea placeholder 또는 하단 보조 문구로 처리
- 4단계도 전체 페이지 스크롤 없이 한 화면에 자연스럽게 들어와야 한다.
- 요청자가 비어 있는 기존 요청은 `요청자 없음`을 입력 필드처럼 보여주지 말고, 낮은 무게의 상태 텍스트로 표시한다.

확인 기준:
- 요청자 `김현우`가 입력 가능한 필드처럼 보이지 않는다.
- 요청 메모가 4단계의 주 작업 영역으로 넓고 자연스럽게 배치된다.
- 요청자와 요청 메모 사이에 어색한 빈 공간이 줄어든다.
- 새 요청 생성 시 로그인 사용자 이름이 요청자로 저장되는 기존 흐름은 유지된다.
- 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.

### 추가 UX 투두 - 5단계 저장 전환 화면에 변경 품목코드 정보 표시

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

- 5단계 `저장 및 전환` 화면에는 변경된 이름뿐 아니라 관련 품목코드 정보도 함께 보여줘야 한다.
- 기준 PF는 품명만 보여주지 말고 품목코드도 같이 표시한다.
  - 예: `기준 PF: 3-PF-0001 · DX3000_60kV, 2mA_러시아_IP Tsizina S. A`
- 3단계에서 새 PA/PF 이름을 입력한 경우, 5단계에는 다음 정보를 구분해서 보여준다.
  - 기준 PF 코드/품명
  - 기존 PA를 재사용하는지, 새 PA가 생성될 예정인지
  - 기존 PF를 재사용하는지, 새 PF가 생성될 예정인지
  - 새 PA/PF는 아직 시스템 코드가 확정 전이라면 `품목코드는 저장/준비 완료 시 자동 생성`처럼 안내한다.
- BOM 변경으로 인해 새 PA/PF가 생성될 예정이라면, 5단계에서 “새 이름 + 코드 생성 예정” 맥락이 자연스럽게 보여야 한다.
- 기존 품목을 재사용하는 경우에는 기존 품목코드와 품명을 함께 보여준다.
- 사용자가 5단계만 보고도 “어느 기준 PF에서 출발했고, 어떤 PA/PF가 새로 만들어지거나 재사용되는지” 확인할 수 있어야 한다.

확인 기준:
- 5단계 기준 PF 요약에 품목코드와 품명이 함께 표시된다.
- 새 PA/PF 생성 예정이면 입력한 이름과 `품목코드 자동 생성 예정` 안내가 보인다.
- 기존 PA/PF 재사용이면 기존 품목코드와 품명이 보인다.
- 품목코드가 아직 없는 새 항목을 빈 값처럼 보이지 않게 처리한다.
- 3단계 이름 변경 후 5단계의 이름/코드 안내가 함께 갱신된다.

### 추가 UX 투두 - 5단계 하단 액션 바 버튼 배치 정리

작성일: 2026-06-30
상태: 다음 일괄 개선 때 처리

- 5단계 `저장 및 전환` 화면 하단의 `마지막 단계입니다.` 라벨은 제거한다.
- 5단계에서는 사용자가 해야 할 실제 행동이 `요청 저장` 또는 `준비 중으로 보내기`이므로, 이 두 버튼을 하단 액션 바의 오른쪽 영역에 배치한다.
- 현재 본문 안쪽에 있는 `요청 저장`, `준비 중으로 보내기` 버튼은 하단 액션 바로 이동한다.
- 하단 액션 바 구조는 다음처럼 정리한다.
  - 왼쪽: `이전`
  - 가운데: 필요 시 짧은 상태 안내 또는 비워둠
  - 오른쪽: `요청 저장`, `준비 중으로 보내기`
- `준비 중으로 보내기`를 primary 액션으로, `요청 저장`을 secondary 액션으로 보이게 한다.
- 마지막 단계에서 본문은 최종 확인 정보에 집중하고, 실행 버튼은 하단 고정 액션 바에 모은다.
- 버튼 비활성 조건이 있으면 하단 액션 바에서도 같은 규칙을 유지한다.

확인 기준:
- 5단계 하단에 `마지막 단계입니다.` 텍스트가 보이지 않는다.
- `요청 저장`, `준비 중으로 보내기` 버튼이 하단 액션 바 오른쪽에 보인다.
- 본문 안쪽에는 중복 실행 버튼이 남아 있지 않다.
- `준비 중으로 보내기`가 가장 중요한 실행 버튼처럼 보인다.
- 버튼 클릭 동작은 기존 저장/전환 로직과 동일하게 유지된다.
