- 구현 방향: 전체/사용 중/비활성 필터, 검색, 선택 강조, 색상 의미, 부서 추가·상태 변경 기능을 유지하면서 행별 카드 외곽선은 제거한다.
- 허용 기준: 1877×852 데스크톱에서 부서명, 코드·소속 직원 수, 상태가 고정 열로 정렬되고 색상 표식은 부서 식별 보조 수단으로만 보인다. 행 높이와 구분선은 품목·모델·직원 목록과 일관된다.
- 브라우저 확인 시나리오: 전체/사용 중/비활성 필터와 검색을 전환하고, 직원 수가 0명·여러 명인 부서를 각각 선택해 목록·상세 연동과 세로 스크롤을 확인한다.
- 우선순위: 중간

### 21. BOM 세 목록을 입출고 품목 선택표의 행 표면·툴팁 기준으로 통일

- 화면/영역: 데스크톱 관리자 > BOM 관리 > 좌측 부모 품목 목록, 가운데 하위 품목 추가 목록, 우측 현재 구성 목록.
- 증상: 좌측·가운데 목록은 새 `BomTablePrimitives`의 개별 `title` 속성으로 이름 셀 위에서만 브라우저 기본 툴팁을 표시한다. 우측 현재 구성 행은 기존 `BomRow`의 `TruncatedText`를 사용해 툴팁 표면·동작이 다르고, 세 목록 모두 입출고 품목 선택표의 평평한 행 밀도·hover 표면과 다르다.
- 사용자 불편: 긴 품목명을 확인하려면 각 목록마다 다른 위치와 다른 모양의 툴팁을 찾아야 하며, 행을 선택하려고 움직일 때 이름 툴팁의 기준점이 달라져 조작 감각이 어색하다.
- 확정 정책: BOM 목록의 직접 시각·상호작용 기준은 입출고 3단계의 `IoTargetPicker`로 통일한다. 평평한 전폭 행, 고정 머리글, 정보 열 정렬, 선택·hover 표면을 따르되, 부모 목록은 상태, 후보 목록은 추가, 현재 구성 목록은 수량·삭제를 마지막 열에 둔다. 이름이 잘린 경우 전체 품목명은 이름 셀이 아니라 행 전체 hover에서 표시한다.
- 구현 방향: `IoTargetPicker`의 행 높이·패딩·헤더·hover/선택 색·구분선 규칙을 BOM 공통 목록 primitive에 반영한다. 좌측·가운데의 개별 이름/코드 `title`을 제거하고 세 목록의 각 행을 공통 행 툴팁 트리거로 감싼다. 툴팁은 기존 `@/lib/ui/TruncatedText`와 동일한 접근성 규칙을 사용하며, 행의 선택·추가·수량 편집·삭제 버튼은 기존대로 동작한다.
- 허용 기준: 1877×852 데스크톱에서 BOM 세 목록은 입출고 품목 선택표와 같은 행 밀도·고정 머리글·hover/선택 표면을 사용한다. 긴 품목명이 잘린 모든 BOM 행은 이름 셀이 아닌 행 어느 지점에 pointer를 올려도 같은 모양의 전체 품목명 툴팁을 표시한다. 짧은 이름에는 불필요한 툴팁이 나타나지 않는다.
- 브라우저 확인 시나리오: 좌측 부모·가운데 후보·우측 현재 구성에서 긴 이름과 짧은 이름을 각각 hover하고, 행 선택·후보 추가·수량 편집·삭제 버튼 hover를 확인한다. 툴팁이 버튼 조작을 가리지 않고 키보드 포커스에서도 전체 이름을 제공하는지 확인한다.
- 우선순위: 중간

## 관리자 모델 관리

### 1. 모델 상세 카드를 가용 높이에 맞춰 배분하고 삭제 버튼을 하단 고정

- 화면/영역: 데스크톱 관리자 > 모델 관리 > 선택 모델 상세의 모델명·기호, 연결 품목 수, 연결 BOM 수, 연결 품목 미리보기, 모델 삭제 영역.
- 증상: 1919×908 화면에서 연결 품목 수·연결 BOM 수가 가로로 나란히 놓여 있고, 연결 품목 미리보기와 삭제 버튼은 각 내용 높이만 사용해 상세 패널 하단에 큰 빈 영역이 남는다. 삭제 버튼은 카드 묶음 바로 아래에 있어 상세 패널의 하단 동작이라는 위치 문법이 약하다.
- 사용자 불편: 모델의 연결 현황과 목록이 상세 영역의 가용 높이를 하나의 작업 단위로 쓰지 못해 카드들이 작고 분산돼 보이며, 위험 작업인 삭제 버튼도 패널 하단에서 찾기 어렵다.
- 확정 정책: 1919×908 데스크톱에서 모델명·기호 입력 아래의 가용 높이는 연결 현황 영역이 채운다. 연결 품목 수와 연결 BOM 수는 좌측 열에 위아래로 쌓고, 연결 품목 미리보기는 우측 열에서 같은 가용 높이를 채운다. 두 요약 카드와 미리보기는 내용의 최소 높이를 보장하면서 남은 높이를 자연스럽게 사용한다. `이 모델 삭제` 버튼은 상세 패널의 최하단에 전폭으로 고정한다. 데이터는 생략하지 않으며, 연결 항목이 많은 경우 미리보기의 기존 요약 범위와 좁은 화면의 반응형 동작은 유지한다.
- 구현 방향: `AdminModelsSection.tsx`의 `ModelDetailView`에서 모델명·기호 편집 카드, 연결 현황/미리보기 영역, 삭제 버튼을 세로 가용 높이를 공유하는 레이아웃으로 연결한다. `data-model-linked-layout`의 좌측 요약 카드는 데스크톱에서 한 열의 세로 스택으로 바꾸고, 우측 미리보기 카드와 함께 남은 높이를 사용하게 한다. 삭제 버튼에는 하단으로 밀어내는 배치를 적용하되, 삭제 확인 모달과 PIN 인증 흐름·버튼의 전폭 의미는 유지한다.
- 허용 기준: 1919×908 데스크톱에서 모델명/기호, 위아래로 배치된 두 요약 카드, 연결 품목 미리보기, 모델 삭제 동작이 모두 보이고 상세 본문 세로 스크롤바가 나타나지 않는다. 연결 현황 영역은 상세 패널의 가용 높이를 자연스럽게 채우며, 좌측 두 요약 카드와 우측 미리보기의 하단 기준선이 정렬된다. 삭제 버튼은 다른 콘텐츠 아래의 상세 패널 최하단에 전폭으로 위치한다. 연결 품목 수와 BOM 수, 미리보기의 실제 데이터 의미는 바뀌지 않는다.
- 브라우저 확인 시나리오: 모델 관리에서 연결 품목이 많은 모델과 없는 모델을 각각 선택해 1919×908 및 더 좁은 데스크톱 폭에서 요약 카드 세로 배치, 요약/미리보기의 높이 배분, 상세 전체 표시와 내부 스크롤 유무를 확인한다. 연결 목록의 항목명·코드·`외 N건` 표시가 유지되고, 삭제 버튼이 항상 최하단에 보이는지 확인한 뒤 키보드와 마우스로 삭제 확인·PIN 인증 취소/확정을 조작한다.
- 우선순위: 중간

### 17. 모델 관리 목록·상세 영역을 데스크톱 2:8 비율로 조정

- 화면/영역: 데스크톱 관리자 > 모델 관리 > 좌측 `모델 목록` 패널과 우측 선택 모델 상세 패널.
- 증상: 현재 좌측 목록 폭이 너무 좁아 모델명, 코드, 사용 상태가 압축되어 보이고, 우측 상세 영역에 비해 균형이 맞지 않는다.
- 사용자 불편: 모델을 빠르게 훑거나 서로 비슷한 모델명을 구분할 때 목록의 정보가 답답하게 보이며, 상세를 열어 둔 상태에서 목록 전환이 불편하다.
- 확정 정책: 1917×906 데스크톱 기준으로 좌측 목록과 우측 상세를 2:8 비율로 배치한다.
- 구현 방향: 모델 관리의 데스크톱 분할 그리드를 `20% 80%` 비율로 조정하되, 기존 최소 폭과 좁은 화면의 세로/가로 스크롤 동작은 보존한다. 목록 행의 구조·선택·추가·상태 표시는 바꾸지 않는다.
- 수용 기준: 1917×906에서 좌측 목록이 전체 작업 영역의 약 20%, 우측 상세가 약 80%를 차지하며, 모델명·코드·상태가 겹치거나 불필요하게 줄지 않는다. 상세 카드의 기존 높이 배분과 삭제 동작은 유지된다.
- 브라우저 확인 시나리오: 모델 관리에서 짧은 이름·긴 이름 모델을 각각 선택해 좌측 목록과 우측 상세의 실제 폭 비율을 확인한다. 추가 버튼, 목록 선택, 상세 저장/삭제 확인, 좁은 화면의 기존 스크롤을 확인한다.
- 우선순위: 중간

## 관리자 품목 관리

### 2. 상세 폼 상단의 중복 선택 품목 제목 처리 결정

- 화면/영역: 데스크톱 관리자 > 품목 관리 > 선택 품목 상세 > 기본 정보 탭 상단의 `히팅 싱크 + 방열팬 (구형)` 제목.
- 증상: 좌측 선택 목록에 이미 보이는 품목명이 상세 폼 상단에서 다시 제목으로 렌더링되고, 바로 아래의 품목명 입력과도 내용이 겹친다.
- 사용자 불편: 편집에 필요한 입력보다 선택된 품목명 표시가 먼저 차지하여 세로 공간을 낭비한다.
- 정책 확인 필요: 상세 폼 상단의 선택 품목명과 상태 표시를 모두 제거할지, 스크롤 상태에서도 선택 맥락을 보존할 최소 표시만 남길지 결정한다. 현재 사용자의 제안은 제거다.
- 구현 방향: 정책 확정 후 `EditItemForm.tsx`의 상단 제목과 상위 상세 카드의 상태 배지 표시를 함께 검토해, 같은 품목명이 중복되지 않도록 한 위치만 유지한다. 품목명 입력, 좌측 목록의 선택 표시, 저장/삭제 동작은 유지한다.
- 허용 기준: 확정된 한 위치에서만 선택 품목의 식별 정보가 보이며, 상세 폼을 열었을 때 입력 시작점이 불필요하게 아래로 밀리지 않는다.
- 브라우저 확인 시나리오: 정상·부족·품절 품목을 각각 선택해 좌측 목록 선택 표시와 상세 폼의 제목/상태가 중복되지 않는지, 기본 정보·재고 정보·BOM 탭 전환 시에도 현재 품목을 구별할 수 있는지 확인한다.
- 우선순위: 낮음

### 3. 품목 기본 정보 필드 순서를 입력 흐름에 맞게 재배치

- 화면/영역: 데스크톱 관리자 > 품목 관리 > 기본 정보 탭의 입력 필드와 `품목 코드` 미리보기.
- 증상: 현재 기본 정보는 품목명 뒤에 자재분류·공급사·안전재고가 먼저 오고, 카테고리·단위·사용 제품·품목 코드가 뒤에 표시된다. 품목 코드가 어떤 선택 결과인지 파악하기 전에 보조 속성부터 입력하게 되어 업무 순서와 화면 순서가 어긋난다.
- 사용자 불편: 품목을 식별하고 분류·적용 제품을 결정하는 핵심 입력보다 보조 정보가 먼저 보여, 위에서 아래로 자연스럽게 입력하거나 기존 품목 정보를 훑기 어렵다.
- 확정 정책: 기본 정보의 표시 순서는 `품목명 → 카테고리 → 사용 제품 → 자재분류 → 안전재고 → 공급사 → 단위 → 품목 코드`로 고정한다. 사용 제품의 선택 기호는 기존 결정대로 레이블 옆에 인라인으로 표시한다. 품목 코드는 사용 제품·카테고리·옵션으로 계산된 결과이므로 마지막에 계산 안내와 함께 표시한다.
- 구현 방향: `ItemFormFields.tsx`에서 각 필드 블록의 렌더 순서만 위 확정 순서로 재배치한다. `MesCodeSection`은 사용 제품 블록 뒤의 최종 블록으로 유지하고, 사용 제품 기호의 인라인 표시(항목 4)는 함께 적용한다. 자동 계산·저장 시 번호 부여·카테고리 변경 규칙, 필수/선택 상태, 각 입력의 값과 검증은 변경하지 않는다.
- 허용 기준: 새 품목과 기존 품목 모두에서 위 여덟 필드가 동일한 순서로 보인다. 품목 코드는 항상 마지막에 있고, 사용 제품을 선택하면 레이블 옆 기호와 코드 미리보기가 기존 규칙대로 갱신된다. 안전재고는 공급사보다 먼저, 단위는 공급사보다 뒤에 표시되며 모든 필수/선택 배지가 유지된다.

- 우선순위: 높음

### 4. ✅ 선택된 제품 기호를 `사용 제품` 레이블 옆에 인라인 표시

- 상태: 구현 완료 확인. 위 `구현 완료 코드 검증`의 품목 관리 4번 근거를 따른다.

- 허용 기준: 제품을 하나 이상 선택하면 `사용 제품` 레이블 옆에 기호가 보이고, 기호만을 위한 추가 줄은 없다. 제품을 해제하면 기호도 즉시 사라지고 사용 제품 미지정 안내가 정상 표시된다.

- 우선순위: 중간

### 5. ✅ 목적 없는 `변경 이력 (준비 중)` 탭의 제거 또는 실제 이력 구현 결정

### 16. 품목 관리 좌측 목록을 입출고 품목 선택표와 같은 목록 언어로 정비

- 증상: 품목 목록이 입출고 화면의 표형 품목 선택표와 달리, 큰 둥근 테두리와 넓은 행 간격을 가진 카드형 행으로 표현된다. 품목명·품목 코드·상태 배지가 각각 다른 수직 기준을 사용해 긴 목록을 훑을 때 정보 밀도와 정렬감이 떨어진다.
- 사용자 불편: 작업자가 입출고에서 익힌 목록 읽기 방식으로 품목을 찾기 어렵고, 관리 화면으로 전환할 때마다 다른 디자인 규칙을 다시 해석해야 한다. 선택 행의 강조도 행 전체의 표형 선택보다 카드 단위 강조로 보여 목록보다 별도 컨트롤처럼 느껴진다.
- 정책 확인 필요: 품목 관리 좌측 목록도 입출고의 데스크톱 표형 목록(고정 열 머리글, 행 구분선, 일관된 밀도)을 기준으로 통일할지 최종 확인한다. 권장안은 `품목명 | 품목 코드 | 상태`를 공통 열로 두고, 현재 카드형 외곽선·행별 둥근 박스는 제거하는 것이다.
- 구현 방향: `AdminMasterItemsSection`의 좌측 목록을 입출고 `IoTargetPicker`의 표 구조와 같은 행 리듬으로 재구성한다. 상단 검색/추가 동작은 유지하고, 목록 본문에는 고정 열 머리글과 단일 구분선을 둔다. 선택 행은 카드 테두리 대신 행 배경·좌측 강조선으로 표현하며, 상태는 마지막 열에 정렬한다. 검색, 선택, 추가, 가상화/스크롤, 정상·부족·품절 필터와 상세 폼 연동은 바꾸지 않는다.
- 수용 기준: 1917×906에서 품목명, 품목 코드, 상태가 모든 행에서 같은 열에 정렬되고, 긴 이름은 상태 열을 밀지 않는다. 선택 행은 목록의 일부로 읽히며 현재 선택·검색·필터·추가와 독립 세로 스크롤은 그대로 동작한다. 입출고 품목 선택표와 나란히 보았을 때 행 높이, 구분선, 열 기반 정보 위계가 일관된다.
- 브라우저 확인 시나리오: 품목 관리에서 정상·부족·품절과 긴 품목명 항목을 각각 조회하고, 검색·선택·추가를 확인한다. 입출고 > 창고 입출고 > 품목 선택 단계의 표형 목록과 열 정렬·선택 강조·스크롤을 비교하고, 좁은 화면에서 기존 목록 스크롤과 상세 폼 연동이 유지되는지 확인한다.
- 우선순위: 중간

### 7. 편집 중 부서·직급 요약을 즉시 갱신하고 표준 직급으로 통일 표시

- 화면/영역: 데스크톱 관리자 > 직원 관리 > 선택 직원의 좌측 목록 요약과 상세 제목·부제목, 기본 정보의 `직급`·`부서` 선택기.
- 증상: 선택 직원 `심이라`의 저장된 직급은 `영업/과장`이지만 편집 폼은 이를 `과장`으로 정규화해 표시한다. 좌측 목록과 상세 부제목은 저장된 원문을 계속 사용하므로 `영업 · 영업/과장`으로 남고, 부서·직급을 고쳐도 저장을 눌러 목록 데이터가 갱신될 때까지 바뀌지 않는다.
- 사용자 불편: 같은 직원에 대해 목록과 폼의 직급이 다르게 보이고, 수정한 결과를 저장 전에는 확인할 수 없어 입력이 반영되지 않은 것처럼 느껴진다.
- 확정 정책: 선택 직원의 좌측 요약과 상세 제목·부제목은 편집 중인 `부서`·표준 `직급`을 즉시 미리보기로 표시한다. 저장 전 변경은 DB에 자동 저장하지 않으며, 저장 시에만 영구 반영한다. 선택하지 않은 직원도 화면 표시는 표준 직급으로 정규화한다.

- 브라우저 확인 시나리오: 복합 직급(`영업/과장`) 직원과 표준 직급 직원을 각각 선택한다. 선택 직원의 부서·직급을 변경해 목록/상세 즉시 갱신을 확인하고, 저장·새로고침 후 영구 반영을 확인한다. 저장하지 않은 변경에서 다른 직원을 선택할 때 기존 변경 확인 절차와 검색 결과가 정상 동작하는지 확인한다.

- 사용자 불편: 화면에 보이지 않는 부모 품목을 선택할 수 없고, 필터를 좁히지 않으면 긴 목록의 하단으로 이동할 수 없다.
- 확정 정책: 좌측 부모 품목 목록은 검색·필터 영역을 고정한 채 할당된 BOM 작업영역 안에서 독립적으로 세로 스크롤한다. 미배치 원자재 패널을 열어도 목록은 남은 높이 안에서 스크롤하며 내용이 잘리거나 패널과 겹치지 않는다.
- 구현 방향: `BomWorkbench.tsx`의 작업영역과 첫 번째 열 래퍼에 `flex-1`·`min-h-0` 높이 제약을 연결하고, `BomParentList.tsx`의 기존 목록 `overflow-y-auto`를 실제 스크롤 영역으로 유지한다. 가운데 하위 품목 및 우측 현재 구성 패널의 독립 스크롤, BOM 편집/사용처 모드 전환은 변경하지 않는다.
- 허용 기준: 1919×908 화면에서 125건 이상의 부모 품목 목록을 마우스 휠·트랙패드·스크롤바로 끝까지 탐색할 수 있고, 검색·필터 헤더는 목록 위에 계속 보인다. 하단 미배치 원자재 패널을 닫거나 열어도 좌측 목록의 스크롤·선택·상태 표시가 정상 동작한다.
- 브라우저 확인 시나리오: 전체·공정 필터를 각각 선택해 긴 목록을 위아래로 스크롤하고, 하단 미배치 원자재 패널을 열고 닫은 상태를 비교한다. 화면 하단의 부모 품목을 선택해 가운데 추가 패널과 우측 BOM 구성이 갱신되는지, 가운데·우측 패널의 스크롤과 사용처 모드 전환에 회귀가 없는지 확인한다.
- 우선순위: 높음

### 15. BOM 세 목록을 입출고 품목 선택표와 같은 정보 밀도로 통일

- 화면/영역: 데스크톱 관리자 > BOM 관리 > 편집 모드의 좌측 부모 품목 목록, 가운데 하위 품목 추가 목록, 우측 현재 구성 목록.
- 증상: 세 목록이 각각 카드형 행과 서로 다른 필터·상태·조작 표현을 사용한다. 같은 품목을 다루는데도 입출고 화면의 품목 선택표와 달리 열 기준과 행 밀도가 일관되지 않아, 목록 세 개가 별도 화면을 이어 붙인 것처럼 보인다.
- 사용자 불편: 부모 선택, 후보 추가, 현재 구성 확인을 오갈 때 품목명·코드·상태·수량을 훑는 위치가 계속 달라진다. 특히 가운데 후보 목록과 우측 현재 구성은 작업 대상 목록인데도 입출고의 표형 목록보다 정보 위계와 조작 위치를 빠르게 파악하기 어렵다.
- 정책 확인 필요: 세 목록을 입출고 품목 선택표처럼 공통 열 머리글을 가진 데스크톱 표형 목록으로 통일할지 최종 확인한다. 권장안은 통일하되, 세 목록의 역할 차이(부모 상태·후보 추가·현재 구성 수량/삭제)는 마지막 열의 상태 또는 조작으로만 구분하는 방식이다.
- 구현 방향: `IoTargetPicker`의 표형 목록을 시각 기준으로 삼아, 세 패널에 고정 열 머리글·동일한 행 높이·구분선을 적용한다. 공정 배지, 품목명, 품목 코드, 역할별 마지막 열을 공통 순서로 두고, 좌측은 BOM 상태, 가운데는 추가 상태/버튼, 우측은 수량 편집과 삭제를 마지막 열에 둔다. 기존 검색·부서/공정 필터·후보 중복 비활성화·수량 Enter/Escape·확인 모달은 유지한다.
- 수용 기준: 1917×910에서 세 목록 모두 같은 행 리듬과 열 정렬을 사용하며, 긴 품목명은 코드·상태·조작을 밀지 않는다. 목록별 차이는 마지막 열의 역할로만 구분되고, 각 패널의 독립 스크롤·선택 강조·기존 BOM 추가/수정/삭제 동작은 유지된다.
- 브라우저 확인 시나리오: BOM 편집 모드에서 조립 필터와 부모 품목을 선택한 뒤, 좌측 부모 목록·가운데 후보 목록·우측 현재 구성 목록의 열 머리글/행 정렬을 입출고 > 창고 입출고 > 품목 선택 단계의 표형 목록과 비교한다. 후보 추가, 수량 수정, 삭제 확인, 긴 품목명·빈 목록·좁은 화면 스크롤을 각각 확인한다.
- 우선순위: 중간

### 13. 섹션 요약 바의 위아래 여백을 16px로 정렬

- 화면/영역: 데스크톱 관리자 > 모델 관리에서 확인한 상단 관리자 섹션 탭 바와 `모델 관리 · 전체 모델 · 사용 중 · 비활성` 요약 바, 그리고 그 아래 작업 영역 사이의 세로 여백. 이 레이아웃은 관리자 공통 작업 영역에 적용된다.
- 증상: 1917×910 CSS 뷰포트 측정에서 상단 관리자 섹션 탭 바의 하단은 y=174px, 작업 영역은 y=186px부터 시작하고, 요약 바는 작업 영역의 12px 상단 패딩 때문에 y=198px에 시작한다. 따라서 요약 바 위 여백은 24px(상위 `gap-3` 12px + `pt-3` 12px)인 반면, 요약 바 하단 y=240px부터 작업 영역 시작 y=256px까지의 아래 여백은 `mb-4` 16px이다. 위아래 여백이 8px 달라 요약 바가 세로 중앙에 놓이지 않는다.
- 사용자 불편: 상단 내비게이션과 섹션 콘텐츠 사이에서 요약 바가 아래쪽으로 치우쳐 보이고, 탭마다 같은 리듬이 반복되어 화면 정렬이 어색하게 느껴진다.
- 확정 정책: 1917×910 데스크톱 기준 섹션 요약 바의 위아래 여백을 모두 16px로 맞춘다. 상위 레이아웃의 `gap-3` 12px과 요약 바의 `mb-4` 16px은 유지하고, 공통 작업 영역 상단 패딩만 4px으로 조정해 `12px + 4px = 16px`을 만든다.
- 구현 방향: `DesktopAdminView.tsx`의 공통 작업 영역 클래스에서 `pt-3`을 `pt-1`로 바꾼다. `AdminPageHeader.tsx`의 `mb-4`와 상위 레이아웃의 `gap-3`, 관리자 섹션 탭·상세 패널의 크기 및 반응형 스크롤 동작은 변경하지 않는다.
- 허용 기준: 1917×910 CSS 뷰포트에서 상단 관리자 섹션 탭 바 하단부터 섹션 요약 바 상단까지, 그리고 요약 바 하단부터 다음 작업 영역 상단까지의 거리가 각각 16px이다. 모델·품목·직원·부서·BOM 관리 전환 후에도 요약 바가 같은 수직 리듬을 유지한다.
- 브라우저 확인 시나리오: 1917×910 뷰포트에서 모델, 품목, 직원, 부서, BOM 관리 탭을 차례로 연다. 각 탭에서 상단 섹션 탭 바·요약 바·첫 작업 영역의 `getBoundingClientRect()`를 확인해 위·아래 간격이 각각 16px인지 측정하고, 좁은 데스크톱 폭에서 탭 줄바꿈·가로 스크롤·콘텐츠 시작 위치에 회귀가 없는지 확인한다.
- 우선순위: 중간

### 14. 관리자 섹션 탭 바 하단 그림자 제거

- 화면/영역: 데스크톱 관리자 상단의 `기준 정보`·`구성 관리`·`시스템` 그룹을 담는 `관리자 섹션` 탭 바.
- 증상: 탭 바 하단에 카드 그림자처럼 보이는 음영이 렌더링된다. 실제로 `AdminSectionTabs.tsx`의 `boxShadow: var(--c-card-shadow)`가 적용돼 있다.
- 사용자 불편: 탭 바 바로 아래에 불필요한 경계와 깊이감이 생겨, 탭 바와 본문이 겹쳐 있거나 떠 있는 것처럼 보인다.
- 확정 정책: 관리자 섹션 탭 바의 외곽 그림자를 제거한다. 배경색·테두리·모서리 반경·탭 그룹 구분·선택 상태·가로 스크롤 동작은 유지한다.
- 구현 방향: `AdminSectionTabs.tsx`의 외곽 `<nav>` 인라인 스타일에서 `boxShadow` 속성만 제거한다. 공용 `--c-card-shadow` 토큰이나 다른 카드·모달·사이드바의 그림자는 변경하지 않는다.
- 허용 기준: 관리자 섹션 탭 바 하단에 그림자/음영이 보이지 않는다. 탭 바의 1px 테두리와 배경은 유지되며, 모델·품목·직원·부서·BOM·시스템 탭의 선택·키보드 탐색·좁은 화면 가로 스크롤은 기존대로 동작한다.
- 브라우저 확인 시나리오: 라이트·다크 테마에서 관리자 화면을 열어 탭 바 하단의 그림자 유무와 테두리 가독성을 확인한다. 각 탭 그룹과 탭을 클릭·키보드로 전환하고, 좁은 데스크톱 폭에서 가로 스크롤로 마지막 `보안` 탭까지 이동해 그림자 제거가 레이아웃·선택 표현에 영향을 주지 않는지 확인한다.
- 우선순위: 낮음
