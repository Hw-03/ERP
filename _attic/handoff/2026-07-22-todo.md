### 2. 내 요청에서도 상단 기본 섹션 탭 유지

- 화면/영역: 데스크톱 입출고 > `내 요청` 섹션.
- 증상: 사용자가 `내 요청`으로 들어가면 상단의 `요청 작성`·`작성 중`·`내 요청` 기본 섹션 탭이 표시되지 않는 경로가 있다. 현재 소스에는 내 요청 URL 직접 진입 시 세 탭을 표시하는 단위 테스트가 있지만, 실제 브라우저 동선에서 누락이 관찰됐다.
- 사용자 불편: 내 요청을 확인한 뒤 요청 작성이나 작성 중 목록으로 돌아가려면 입출고의 기본 탐색 맥락을 잃고, 다른 경로를 다시 찾아야 한다.
- 확정 정책: `요청 작성`·`작성 중`·`내 요청`은 입출고의 공통 상단 섹션 탭으로 유지한다. 세 섹션 중 무엇을 보고 있든 항상 표시하며, 요청 작성 안의 품목 전환 집중 단계에서만 기존처럼 일시적으로 숨길 수 있다.
- 구현 방향: 실제 브라우저에서 `내 요청` 진입 동선과 `section=mine` URL 직접 진입을 각각 추적해, 상단 `WarehouseSectionTabs`가 빠지는 렌더 조건·상태 동기화·클라이언트 갱신 경로를 바로잡는다. 단위 테스트만으로 끝내지 않고 실제 렌더 트리에서 탭의 가시성을 검증한다.
- 수용 기준: `내 요청`을 클릭해 진입하거나 `?tab=warehouse&section=mine`으로 직접 열어도 상단에 `요청 작성`·`작성 중`·`내 요청`이 모두 보이고, `내 요청`만 활성 상태로 표시된다. 세 탭 간 이동 시 URL·콘텐츠·활성 상태가 일치하며, 품목 전환 집중 단계 외에는 탭이 사라지지 않는다.
- 브라우저 확인 시나리오: 데스크톱에서 요청 작성 → 내 요청 → 작성 중 → 내 요청 순서로 이동하고, 새로고침과 `?tab=warehouse&section=mine` 직접 진입을 각각 수행한다. 각 상태에서 탭 세 개의 가시성·활성 표시·URL·콘텐츠가 일치하는지 확인한다.
- 우선순위: 높음

- 확정 정책: 데스크톱에서는 관리자 섹션 탭이 가용 너비를 모두 사용한다. `기준 정보`·`구성 관리`·`시스템` 그룹은 각 그룹의 탭 수에 비례해 폭을 배분하고, 같은 그룹 안의 각 탭은 균등한 넓은 버튼 폭을 갖는다. 그룹 캡션·가는 구분선·탭 순서·활성 색상·이동 동작은 유지한다.
- 구현 방향: `AdminSectionTabs`의 내부 그룹 래퍼를 콘텐츠 폭 고정이 아닌 가용 폭 분배 레이아웃으로 바꾸고, 그룹별 탭 행은 탭 수 기준의 균등 그리드를 사용한다. 좁은 데스크톱 폭에서는 현재처럼 최소 너비와 가로 스크롤로 접근성을 보장해 버튼 글자나 그룹 구분이 과도하게 압축되지 않게 한다.
- 수용 기준: 1916×908에서 마지막 `보안` 탭 뒤에 큰 빈 공간이 남지 않고, 모든 관리자 탭이 넓고 일관된 클릭 영역으로 상단 폭을 채운다. 그룹 캡션은 버튼처럼 보이지 않고, `기준 정보`·`구성 관리`·`시스템`의 경계와 활성 탭은 계속 즉시 구분된다. 좁은 데스크톱 폭에서는 마지막 시스템 탭까지 가로 스크롤로 접근할 수 있다.
- 브라우저 확인 시나리오: 모델 관리·직원 관리·BOM 관리·보안을 차례로 열어 1916px 폭에서 그룹별 폭 배분, 탭 버튼 너비, 활성 상태와 우측 여백 소진을 확인한다. 좁은 데스크톱 폭에서 가로 스크롤, 그룹 캡션·divider·탭 라벨의 겹침 여부와 모든 탭 이동 동작을 확인한다.
- 우선순위: 중간

## 관리자 모델 관리

### 4. 모델명과 기호 입력을 한 행으로 정렬

- 화면/영역: 데스크톱 관리자 > 모델 관리 > 선택한 모델의 기본 정보 편집 영역.
- 증상: 모델명 입력은 한 행 전체를 사용하고, 짧은 `기호 (1-5자)` 입력은 다음 행 왼쪽에만 배치되어 오른쪽에 큰 빈 공간이 생긴다.
- 사용자 불편: 서로 한 묶음인 모델명·기호를 위아래로 읽어야 하고, 짧은 입력 하나가 넓은 편집 카드의 세로 공간을 불필요하게 차지한다.
- 확정 정책: 모델명과 기호는 같은 행에서 나란히 편집한다. 모델명은 남은 가용 폭을 사용하고, 기호는 1~5자 입력 목적에 맞는 짧고 고정적인 폭을 사용한다. 현재 5자 제한과 저장 동작은 유지한다.
- 구현 방향: `ModelDetailView`의 모델명 전용 전체 행을 제거하고, 모델명·기호 `EditFieldRow`를 같은 데스크톱 그리드 행에 둔다. 좁은 폭에서는 기호 입력이 모델명과 겹치지 않도록 기존 한 열 배치로 자연스럽게 전환한다.
- 수용 기준: 1918px 폭에서 모델명과 기호가 한 줄에 보이고, 모델명은 충분히 길게 편집할 수 있으며 기호 입력은 과도하게 넓지 않다. 기호는 5자를 넘겨 입력되지 않고, 모델명·기호 변경 후 기존 저장과 연결 품목·BOM 요약이 정상 동작한다.
- 브라우저 확인 시나리오: 모델 관리에서 짧은 이름과 긴 이름의 모델을 각각 선택해 두 입력의 한 줄 정렬·폭·포커스 테두리를 확인한다. 창을 좁혀 한 열 전환 시 라벨·입력값이 겹치지 않는지, 기호를 6자 이상 입력해도 5자로 제한되는지 확인한다.
- 우선순위: 중간

## 관리자 품목 관리

### 5. 품목 상세 탭을 MES 공통 선택 문법으로 정리

- 화면/영역: 데스크톱 관리자 > 품목 관리 > 품목 상세의 `기본 정보`·`재고 정보`·`BOM / 사용처`·`변경 이력 (준비 중)` 탭 바.
- 증상: 품목 상세 탭은 단순한 밑줄 중심의 좁은 텍스트 버튼으로 표시되어, 둥근 표면·토큰 tint·명확한 활성 버튼을 사용하는 다른 DEXCOWIN MES 탭과 시각 문법이 다르게 보인다.
- 사용자 불편: 어떤 탭이 선택됐는지와 탭이 실제로 눌리는 조작 요소인지가 약하게 느껴지고, 한 화면 안에서도 상세 탭만 별도 UI처럼 보인다.
- 확정 정책: 품목 상세 탭은 DEXCOWIN MES의 기존 선택 버튼 문법에 맞춘다. 각 탭은 충분한 안쪽 여백과 둥근 버튼 표면을 갖고, 활성 탭만 토큰 기반 tint·테두리·텍스트 강조로 가장 강하게 보인다. 탭 순서와 `변경 이력 (준비 중)`의 준비 상태 표시는 유지한다.
- 구현 방향: `AdminDetailCard`의 공용 탭 바를 관리자·입출고의 기존 탭 버튼과 같은 라운드·여백·hover·active 문법으로 조정한다. 품목 상세에서 쓰는 탭의 선택 상태·키보드 접근성·내용 전환·준비 중 탭 동작은 유지한다.
- 수용 기준: 기본 정보·재고 정보·BOM / 사용처·변경 이력이 같은 상세 카드 안에서 자연스럽게 보이고, 활성 탭은 즉시 구분되며 비활성 탭은 보조 위계를 가진다. 탭 전환·키보드 조작·준비 중 표시는 기존 동작과 같고, 다른 `AdminDetailCard` 사용 화면의 탭도 동일 문법을 따른다.
- 브라우저 확인 시나리오: 품목 관리에서 네 탭을 차례로 선택해 활성·비활성 상태, hover·focus-visible·내용 전환을 확인한다. 직원·부서 등 같은 상세 카드 탭이 있는 화면과 비교해 라운드·여백·색상·선택 위계가 일관되는지, 다크 모드에서도 대비가 유지되는지 확인한다.
- 우선순위: 중간

### 6. 품목 코드 prefix 보조 배지 제거

- 화면/영역: 데스크톱 관리자 > 품목 관리 > 기본 정보 > 품목 코드 영역의 `현재 코드 prefix: 46-AA-` 배지.
- 증상: 사용 제품·카테고리에서 자동 계산한 코드 앞부분을 별도 배지로 표시하지만, 바로 아래 전체 품목 코드 미리보기와 설명에도 같은 자동 계산 정보가 있다.
- 사용자 불편: 사용자가 수정할 수 없는 내부 계산 과정을 별도 문구로 읽어야 하고, 전체 코드 미리보기를 이해하는 데 필요한 정보와 중복되어 화면이 복잡해 보인다.
- 확정 정책: `현재 코드 prefix` 배지는 사용자 화면에서 제거한다. 품목 코드는 전체 코드 미리보기와 그 아래의 자동 계산 설명만으로 안내한다.
- 구현 방향: `MesCodeSection`에서 prefix 전용 배지 렌더링만 제거하고, 전체 코드 미리보기·카테고리 변경 시 다음 번호 부여 안내·실제 저장 시 코드 생성 규칙은 변경하지 않는다.
- 수용 기준: 품목 기본 정보에서 `현재 코드 prefix:` 문구와 보라색 배지가 보이지 않는다. 전체 품목 코드 미리보기는 사용 제품·카테고리·옵션 변경에 따라 기존처럼 갱신되고, 저장된 품목 코드와 일치한다.
- 브라우저 확인 시나리오: 기존 품목을 열고 사용 제품·카테고리·옵션을 각각 바꿔 전체 코드 미리보기와 안내 문구를 확인한다. prefix 배지가 없는 상태에서도 코드가 자동 계산됨을 이해할 수 있고, 저장 후 실제 품목 코드가 미리보기와 일치하는지 확인한다.
- 우선순위: 낮음

## 관리자 부서 관리

### 9. 색상 팔레트 확장 디자인과 상세 본문 폭을 안정화

- 화면/영역: 데스크톱 관리자 > 부서 관리 > 선택한 부서의 `색상` 카드 > `전체 색상 보기`.
- 증상: 기본 HTML `details`·`summary`의 텍스트 링크로 색상 팔레트를 접고 편다. 팔레트를 펼쳐 상세 본문이 넘치면 `AdminDetailCard`의 세로 스크롤바가 새로 생기면서 가용 너비가 줄어들고, 메타 정보·색상 카드·소속 직원 등 본문 전체가 왼쪽으로 밀린다.
- 사용자 불편: 같은 부서 상세에서 펼침 전후 요소의 가로 기준선이 바뀌어 화면이 흔들리는 것처럼 보이며, `전체 색상 보기`만 프로젝트의 둥근 선택 버튼·아이콘 기반 확장 문법과 동떨어져 보인다.
- 확정 정책: 상세 본문은 펼침 전부터 세로 스크롤바 영역을 안정적으로 확보해, 팔레트 확장·축소로 스크롤 가능 여부가 바뀌어도 콘텐츠의 x축 위치와 폭이 변하지 않게 한다. `전체 색상 보기`는 기본 `details` 링크가 아닌 DEXCOWIN MES의 버튼형 확장 컨트롤로 바꾸며, 라운드 표면·Chevron 방향·활성 상태·hover·focus-visible을 프로젝트의 기존 선택 문법에 맞춘다.
- 구현 방향: `AdminDetailCard`의 스크롤 본문에 안정적인 스크롤바 gutter를 적용해 한 상세 카드 안의 확장 콘텐츠가 가로 재배치를 만들지 않게 한다. `DeptDetailView`는 React 확장 상태와 `aria-expanded`·`aria-controls`를 쓰는 버튼형 트리거로 전환하고, 색상 견본 그리드·선택 색상·키보드 조작·기본 접힘 상태는 유지한다. 기존 `details` 요소를 전제로 한 테스트도 새 접근성 계약으로 갱신한다.
- 수용 기준: 1918px 폭에서 팔레트를 여러 번 열고 닫아도 부서명·메타 셀·색상 카드·소속 직원의 왼쪽 위치와 입력 폭이 변하지 않는다. 색상 트리거는 텍스트 링크가 아닌 충분한 클릭 영역의 둥근 버튼으로 보이고, 열림·닫힘 상태가 Chevron·`aria-expanded`에 일치한다. 견본 선택, 색상 코드 직접 입력, 저장 전 변경 표시와 저장 동작은 기존과 같다.
- 브라우저 확인 시나리오: 스크롤이 생기는 부서 상세에서 팔레트를 열기 전·후의 메타 셀과 색상 코드 입력 왼쪽 좌표를 비교하고, 반복 열기·닫기와 창 폭 축소에서도 가로 밀림·레이아웃 점프가 없는지 확인한다. 마우스·Tab·Enter·Space로 트리거를 조작하고, 견본을 선택한 뒤 저장·새로고침해 선택 색상이 유지되는지 확인한다.
- 우선순위: 중간

## 관리자 직원 관리

### 7. PIN·위험 작업 카드를 내용 높이로 압축하고 메타 정보를 제목 옆으로 이동

- 화면/영역: 데스크톱 관리자 > 직원 관리 > 선택한 직원의 하단 `PIN`, `계정 상태 및 위험 작업` 카드.
- 증상: `기본 PIN (0000)`과 `마지막 변경` 정보가 PIN 카드 본문에 세로로 쌓이고, PIN 카드와 같은 그리드 행의 위험 작업 카드도 그 높이에 맞춰 늘어난다. 위험 작업 버튼 아래에는 기능 없는 넓은 빈 공간이 남는다.
- 사용자 불편: PIN의 핵심 상태와 변경 이력을 카드 제목과 떨어진 곳에서 읽어야 하고, 하단 두 카드가 실제 내용보다 큰 박스로 보여 직원 상세의 세로 밀도가 낮아진다.
- 확정 정책: PIN 상태와 마지막 변경 정보는 `PIN` 제목 옆의 한 줄 메타 영역에 둔다. PIN 카드는 초기화 버튼에 필요한 높이만 사용하며, 위험 작업 카드는 PIN 카드의 높이에 강제로 맞추지 않고 버튼 내용 높이만큼 자연스럽게 줄어든다.
- 구현 방향: `DetailCardSlot` 또는 PIN 카드 전용 헤더에 제목 보조 영역을 추가해 기본 PIN 상태 배지와 마지막 변경을 같은 기준선에 배치한다. 하단 두 카드의 그리드는 항목별 높이를 유지하도록 정렬 방식을 조정하고, 버튼의 최소 44px 터치 영역·위험 색상·확인 모달 동작은 유지한다.
- 수용 기준: 1918px 폭에서 `PIN` 제목, 기본 PIN 상태, 마지막 변경이 한 줄에서 자연스럽게 읽히고 카드 본문에는 PIN 초기화 버튼만 남는다. PIN·위험 작업 카드는 각각의 내용 높이만 사용하며, 위험 작업 버튼 아래에 큰 빈 공간이 남지 않는다. 카드의 상단 기준선·간격·버튼 폭은 서로 정돈되어 보인다.
- 브라우저 확인 시나리오: 기본 PIN 직원과 PIN을 변경한 직원을 각각 선택해 제목 옆 메타 정보·날짜 형식·PIN 초기화 버튼을 확인한다. 활성·비활성 직원에서 위험 작업 버튼 문구와 색상을 확인하고, 창 폭을 줄여 메타 정보·버튼·카드가 겹치지 않는지, PIN 초기화·직원 비활성화·삭제 확인 모달이 기존처럼 동작하는지 확인한다.
- 우선순위: 중간

### 8. 레거시 부서/직급 값을 표준 직급으로 자동 선택

- 화면/영역: 데스크톱 관리자 > 직원 관리 > 선택한 직원의 `직급` 선택기.
- 증상: 기존 직원의 `role`에는 `튜브/주임`처럼 부서와 직급이 합쳐진 레거시 값이 남아 있다. 현재 선택기는 표준 직급 목록에 없는 전체 문자열을 `기존 직급: 튜브/주임`이라는 비활성 임시 옵션으로 추가한다.
- 사용자 불편: 직원마다 현재 직급을 다시 판단해 드롭다운에서 직접 골라야 하며, 선택기에 수정할 수 없는 기존 값이 계속 남아 표준 직급 체계로 정리되지 않는다.
- 확정 정책: 직원 선택 시 레거시 `부서/직급`의 마지막 값을 표준 직급으로 자동 선택하고, `기존 직급:` 임시 옵션은 제거한다. 일반 값은 `주임`·`대리`·`과장`·`부장`·`대표`·`사원`처럼 마지막 직급을 그대로 사용한다. `연구소/책임`은 `책임연구원`으로, 비활성 직원의 `진공/퇴사`는 퇴사 여부를 직급으로 쓰지 않도록 `사원`으로 정한다. 이미 표준 직급만 저장된 직원은 그대로 둔다.
- 구현 방향: 편집 폼을 만들 때 원본 `role`을 정규화하는 순수 헬퍼를 두고, 표준 직급 또는 `부서/직급`의 매핑 결과만 직급 선택기에 전달한다. 저장은 사용자가 기존 저장 버튼을 누를 때 선택된 표준 직급으로만 갱신하며, 화면 진입만으로 전체 직원 데이터를 일괄 변경하지 않는다. 부서는 이미 별도 필드로 관리하므로 레거시 접두어로 덮어쓰지 않는다.
- 수용 기준: `튜브/주임` 직원 선택 시 직급 선택기에 `주임`이 선택되어 있고 `기존 직급:` 옵션은 보이지 않는다. `연구소/책임`은 `책임연구원`, 비활성 `진공/퇴사`는 `사원`으로 표시된다. 표준 직급 직원의 선택값·부서·저장 동작은 바뀌지 않으며, 저장한 직원은 이후 다시 열어도 표준 직급만 표시된다.
- 브라우저 확인 시나리오: `튜브/주임`, `연구소/책임`, 비활성 `진공/퇴사`, 이미 표준 직급인 직원을 차례로 선택해 자동 선택값과 부서 보존을 확인한다. 변경 없이 화면을 전환했을 때 DB가 갱신되지 않는지, 저장 후 새로고침해 표준 직급이 유지되는지, 직급 선택과 직원 상태 변경·PIN 초기화·삭제 동작이 서로 영향을 주지 않는지 확인한다.
- 우선순위: 중간

## 출하

### 10. 요청 상세에서 준비 중 요청을 준비 완료로 전환하는 동선 복구

- 화면/영역: 데스크톱 출하 > 출하 요청 > 요청 상세. URL `?tab=shipping&shippingView=requestDetail&shippingRequestId=<준비 중 요청 ID>`로 연 요청의 하단 작업 영역.
- 증상: 요청 상세는 `준비 중` 요청을 수정 가능으로 표시하지만, 하단에는 `수정`·`요청 취소`만 있고 `준비 완료` 버튼이 없다. 같은 준비 완료 처리와 확인 모달은 별도 `준비 작업` 화면에만 연결되어 있어, 요청 상세 경로에서는 상태를 `준비 완료`로 전환할 수 없다.
- 사용자 불편: 출하 준비를 마친 사용자가 현재 열어 둔 요청 상세 화면에서 작업을 완료하지 못하고, 상태 전환 기능이 사라진 것으로 인식한다. 출하 준비·재고 반영·다음 픽업 단계가 막힌다.
- 확정 정책: `준비 중(PREPARING)` 출하 요청의 요청 상세에도 `준비 완료` 작업을 제공한다. 기존 준비 작업 화면과 같은 확인 절차·재고 검증·상태 전환을 재사용하며, `요청` 상태의 출하 요청 전송, `준비 완료` 상태의 취소·픽업 동선은 바꾸지 않는다.
- 구현 방향: `RequestDetailEntry`에 준비 완료 콜백을 전달하고, `PREPARING` 상태일 때 기존 `completePrepare` 확인 흐름을 여는 `준비 완료` 액션을 렌더한다. 준비 작업 화면과 요청 상세가 동일 API·확인 모달·pending 상태를 쓰도록 연결해 두 경로의 상태 전환 규칙이 갈라지지 않게 한다.
- 수용 기준: 준비 중 요청의 요청 상세에서 `준비 완료` 버튼이 보이고, 클릭하면 기존 준비 완료 확인이 열린다. 확인 후 성공하면 상태 배지와 버튼이 `준비 완료`·`준비 완료 취소`·`픽업 완료`로 즉시 갱신된다. 처리 중에는 중복 실행할 수 없고, 재고 부족 등 기존 실패 시에는 상태가 바뀌지 않으며 오류를 확인할 수 있다.
- 브라우저 확인 시나리오: 준비 중 요청을 요청 목록에서 상세로 열고, 새 버튼 → 확인 → 준비 완료 처리까지 실행한다. 처리 후 동일 상세에서 취소·픽업 버튼과 수정 잠금 표시를 확인한다. 별도 준비 작업 화면에서도 준비 완료가 계속 가능하고, 부족 재고 요청에서는 실패 안내와 상태 유지가 같은지 확인한다.
- 우선순위: 높음

### 11. 출하 요청으로 생성한 새 PA/PF를 대시보드에 즉시 반영

- 화면/영역: 데스크톱 출하 > 출하 요청 작성 5단계 > `출하 요청` 실행 직후, 데스크톱 대시보드 재고 품목 목록.
- 증상: 동일 BOM이 없어 새 PA/PF를 생성하는 출하 요청을 완료하면 서버에는 두 품목이 생성되지만, 프런트엔드는 출하 요청 캐시만 갱신한다. 대시보드의 품목 쿼리는 이전 캐시를 계속 써서 브라우저를 새로고침하기 전까지 새 PA/PF가 표시되지 않는다.
- 사용자 불편: 출하 요청 성공 직후 대시보드로 이동해도 방금 만든 품목이 없어, 저장 또는 생성이 실패한 것으로 오해하고 불필요하게 새로고침해야 한다.
- 확정 정책: 새 PA/PF를 만들 수 있는 출하 요청이 성공하면 품목 데이터 캐시도 동기화한다. 사용자가 바로 대시보드로 이동했을 때 새로고침 없이 최신 품목 목록에서 새 PA와 PF를 확인할 수 있어야 하며, 기존 BOM과 매칭되어 새 품목이 생기지 않는 요청의 흐름은 유지한다.
- 구현 방향: 출하 요청 생성·수정 성공 뒤 품목 관련 React Query 캐시(`queryKeys.items.all`)를 무효화하거나 최신 데이터로 갱신해, 대시보드가 다음 진입 또는 활성 상태에서 새 목록을 다시 읽게 한다. 요청 실패·취소 경로에는 실제로 생성되지 않은 품목을 캐시에 추가하지 않고, 출하 요청 목록의 즉시 갱신 동작도 유지한다.
- 수용 기준: 새 PA/PF 이름을 입력해 출하 요청을 성공한 뒤 브라우저 새로고침 없이 즉시 대시보드로 이동하면 두 품목이 목록·검색 결과에 표시된다. 기존 BOM 일치 요청, 요청 실패, 대시보드 재방문에서도 오래된 목록이나 유령 품목이 남지 않는다.
- 브라우저 확인 시나리오: 새 PA/PF가 필요한 BOM 조합으로 출하 요청을 완료하고 대시보드로 바로 이동해 PA·PF 이름과 코드가 표시되는지 확인한다. 같은 동선을 새로고침 없이 반복하고, 기존 BOM 일치 요청과 이름·BOM 검증 실패 요청에서도 품목 목록이 정확한지 비교한다.
- 우선순위: 높음

### 12. 출하 요청 수량에 입출고 공용 수량 조절기 재사용

- 화면/영역: 데스크톱 출하 > 출하 요청 작성 1단계의 하단 `출하 수량` 입력. 수량 변경을 누른 뒤 되돌아오는 동일 입력 영역.
- 증상: 입출고 품목 확인에는 `-10`·`-1`·직접 입력·`+1`·`+10`으로 빠르게 수량을 조절하는 공용 조절기가 있지만, 출하 요청은 숫자 입력 칸만 별도로 사용한다.
- 사용자 불편: 같은 DEXCOWIN MES 안에서 수량을 바꾸는 방법이 화면마다 달라지고, 반복 출하 수량을 조절할 때 직접 숫자를 지우고 입력해야 한다.
- 확정 정책: 출하 요청 수량은 입출고의 공용 수량 조절 문법을 사용한다. `-10`·`-1`·직접 입력·`+1`·`+10`을 제공하고, 출하 수량의 업무 제약에 맞춰 최소값은 1대로 유지한다. 수량 변경은 현재처럼 요청 작성 1단계의 한 곳에서만 한다.
- 구현 방향: 출하 요청의 독립 숫자 입력을 기존 `QuantityStepper` 기반으로 교체하거나 공통 컴포넌트를 최소값 1을 지원하도록 확장한다. 기존 직접 입력, 수량 변경 버튼의 1단계 복귀·포커스, pending/수정 잠금, BOM 수량·최종 요약 연동은 유지한다.
- 수용 기준: 출하 요청 1단계에 `출하 수량` 라벨과 `-10`·`-1`·수량 입력·`+1`·`+10`이 보인다. 직접 입력과 각 버튼으로 수량이 즉시 바뀌며 1 미만으로 내려가지 않는다. 변경한 수량은 BOM 매칭·필요 구성품·최종 확인의 계산에 동일하게 반영되고, 처리 중 또는 수정 잠금 상태에서는 기존처럼 수정할 수 없다.
- 브라우저 확인 시나리오: 새 출하 요청에서 기본값 1로 `-1`·`-10`을 눌러 하한을 확인하고, `+1`·`+10`·직접 입력으로 수량을 바꾼다. 3단계의 `수량 변경`으로 1단계에 돌아온 뒤 다시 바꿔 BOM 필요 수량과 5단계 최종 수량이 일치하는지 확인한다. 준비 중 요청 수정·처리 중 상태에서도 기존 잠금 동작이 유지되는지 확인한다.
- 우선순위: 중간
