### 1. 흐름/수량 pill 높이를 얇은 형태로 통일

- 수용 기준: 입출고 내역 목록에서 모든 흐름/수량 pill이 같은 높이로 보인다. `조정 -1 EA`, `이동 2품목 · 22 EA`, `사양 전환 2품목 · 2 EA` 같은 문구가 섞여도 행마다 pill 높이가 달라지지 않는다.
- 브라우저 확인 시나리오: `http://192.168.0.63:3001/mes?tab=history`에 접속해 입출고 내역 목록을 확인한다. 사양 전환, 수량 조정, 창고 이동 row를 함께 비교해 흐름/수량 pill 높이가 모두 얇은 형태로 통일됐는지 본다.

### 2. 펼침 하위 row 표시 문법을 작업 유형 간 통일
- 화면/영역: PC 입출고 내역 탭 > 목록 > 묶음 row 펼침 하위 row
- 증상: `창고 -> 부서` 묶음을 펼쳤을 때는 하위 row가 `단품` 중심의 단순한 표현으로 보이고, `품목 전환`이나 `출하 준비 완료` 묶음을 펼쳤을 때는 `소스 차감`, `대상 입고`, `final PF 준비`, `final PA 차감`처럼 역할 중심의 작업 라벨과 수량 pill이 함께 보인다. 품목 전환은 모든 하위 거래가 한 번에 평평하게 펼쳐져 BOM의 접기·펼치기 구조보다 복잡하며, 대표 row에는 변경품만 보여 기존품에서 변경품으로 바뀐 방향을 알 수 없다. BOM 하위 row는 흐름과 실제 증감만 간결하게 보이지만, 품목 전환·출하·출하 준비 하위 row는 상위와 같은 요청자·처리 상태·기준 재고까지 반복해 오른쪽 열이 과밀하다.
- 사용자 불편: 같은 "상위 묶음 아래의 세부 처리 내역"인데 작업 유형마다 하위 row 문법이 달라 보여, 사용자가 어떤 수준의 정보를 기준으로 읽어야 하는지 흔들린다. 품목 전환은 어느 품목을 어느 품목으로 바꿨는지, 관련 구성품이 어느 단계에 속하는지 목록만으로 바로 판단할 수 없다. 상위 row에 이미 나온 요청자와 상태를 하위에서도 반복해, 실제로 확인하려는 구성품의 흐름·수량을 빠르게 스캔하기 어렵다.
- 확정 정책: 펼침 하위 row는 재고 변화 단계 중심으로 통일한다. BOM의 접기·펼치기 상호작용을 유지하고, 품목 전환도 `기존품 차감`과 `변경품 입고` 하위 묶음을 각각 접고 펼칠 수 있게 한다. 모든 하위 row의 오른쪽 열은 BOM처럼 `흐름`과 `실제 증감`을 기본으로 하고, `수량·재고` 열도 `부품 차감 -5 EA`처럼 실제 증감 수량만 한 줄로 표시한다. 상위 묶음에서 이미 보이는 요청자·처리 상태·기준 재고는 반복하지 않는다. 대표 row의 대상 열은 `기존품명 → 변경품명`을, 품목코드 열은 첫 줄 기존품 코드와 다음 줄 `↓ 변경품 코드`를 표시한다.
- 구현 방향: 하위 row가 공통으로 가져야 할 정보 단위를 정한다. `창고 -> 부서`, `품목 전환`, `출하 준비 완료` 펼침 row가 같은 컬럼 밀도, 같은 pill 높이, 같은 라벨 위치, 같은 수량/재고 보조 텍스트 규칙을 쓰도록 presentation helper 또는 row 컴포넌트의 분기 규칙을 정리한다. 하위 row는 상위의 requester/status/stock presentation을 상속해 다시 렌더하지 않고, 각 line의 역할과 실제 수량 증감만 표현한다. 품목 전환은 소스·추가 차감 라인을 `기존품 차감` 하위 묶음에, 변경품·회수 입고 라인을 `변경품 입고` 하위 묶음에 넣고 각 묶음은 BOM과 같은 chevron·접기 상태·ARIA 규칙을 사용한다. 대표 대상명은 길면 말줄임과 전체명 tooltip을 유지한다.
- 수용 기준: 세 유형의 묶음 row를 각각 펼쳤을 때 하위 row가 같은 종류의 정보 구조로 보인다. 품목 전환·출하·출하 준비 하위 row의 오른쪽 열은 BOM 하위 row처럼 흐름·실제 증감만 간결하게 보여 중복 요청자·상태·기준 재고가 없고, `수량·재고` 열에는 기준 재고의 두 번째 줄이 붙지 않는다. 품목 전환 대표 row만 보고 기존품과 변경품의 품명·품목코드 방향을 이해할 수 있고, 펼치면 `기존품 차감`·`변경품 입고` 아래에서 필요한 구성품을 단계별로 검산할 수 있다.
- 브라우저 확인 시나리오: `http://192.168.0.63:3001/mes?tab=history`에서 BOM 생산, 품목 전환, 출하, `출하 준비 완료`, `창고 -> 부서` 묶음을 각각 펼친다. 각 하위 row의 오른쪽 열에 흐름·실제 증감만 남는지 비교하고, BOM과 품목 전환의 하위 묶음을 클릭·키보드로 접고 펼치며 품목 전환 대표 row의 품명·코드 화살표와 하위 단계 관계를 확인한다.
- 우선순위: 높음
- 구현 상태: 완료. 품목 전환의 기존품→변경품 대표 표현과 기존품 차감·변경품 입고 접이식 하위 묶음을 BOM 문법으로 통일하고, 하위 row는 실제 흐름·증감만 표시한다.

### 3. 펼침 하위 row의 시각 위계를 상위 row보다 낮게 정리
- 화면/영역: PC 입출고 내역 탭 > 목록 > 묶음 row 펼침 하위 row
- 증상: `품목 전환`과 `출하 준비 완료`를 펼치면 하위 row가 상위 row와 거의 같은 높이와 밀도로 보인다. 상위 row 하나를 설명하는 보조 row라기보다, 같은 레벨의 새 거래 row가 이어지는 것처럼 보인다.
- 사용자 불편: 사용자가 어디까지가 묶음의 대표 row이고 어디부터가 내부 처리 내역인지 빠르게 구분하기 어렵다. 펼침이 많아지면 목록 전체가 무거워지고, 상위 거래 간 경계가 흐려진다.
- 확정 정책: 펼침 하위 row는 상위 row보다 한 단계 낮은 시각 위계를 가져야 한다. 단, 정보 구조는 축약하지 말고 `역할 라벨 / 대상 / 흐름 / 수량·재고 / 처리자`의 같은 틀을 유지한다.
- 구현 방향: 하위 row의 padding, 배경 tint, 글자 굵기, 아이콘 크기, row 높이를 상위 row보다 낮게 조정한다. 상위 row는 대표 거래로 강조하고, 하위 row는 같은 컬럼 정렬 안에서 세부 라인처럼 읽히게 한다.
- 수용 기준: 펼침 상태에서 상위 row가 묶음 대표로 먼저 보이고, 하위 row는 그 아래에 딸린 세부 단계로 보인다. 하위 row가 상위 row와 같은 무게로 경쟁하지 않는다.
- 브라우저 확인 시나리오: `품목 전환`, `출하 준비 완료`, `창고 -> 부서` 묶음을 펼친 뒤 상위 row와 하위 row의 높이, 배경, 글자 굵기를 비교한다.
- 우선순위: 높음

### 4. 하위 row 역할 라벨을 내부 용어가 아닌 현장 동작 언어로 정리
- 화면/영역: PC 입출고 내역 탭 > 목록 > 펼침 하위 row의 작업 라벨
- 증상: 출하 준비 하위 row에 `final PF 준비`, `final PA 차감` 같은 라벨이 보인다. 개발/데이터 모델 관점에서는 정확할 수 있지만, 현장 사용자가 즉시 이해하기에는 추상적이다.
- 사용자 불편: 사용자가 `final`이 무엇인지, PF/PA가 실제로 어떤 작업 단계인지 한 번 더 해석해야 한다. 내역을 검토할 때 거래의 의미보다 시스템 용어가 먼저 눈에 들어온다.
- 정책 확인 필요: `final PF 준비`, `final PA 차감`을 그대로 둘지, `출하품 생산`, `구성품 차감`, `출하 준비 생산`, `준비품 차감`처럼 현장 동작 중심 라벨로 바꿀지 결정해야 한다.
- 구현 방향: 하위 row 라벨을 작업자가 확인해야 하는 재고 변화 기준으로 재정의한다. 필요하면 보조 텍스트나 tooltip에 PF/PA 코드를 남기고, 기본 라벨은 한국어 동작 언어로 표시한다.
- 수용 기준: 출하 준비 하위 row를 처음 보는 사용자가 각 줄이 `무엇을 차감했고 무엇을 생산/입고했는지`를 라벨만 보고 이해할 수 있다.
- 브라우저 확인 시나리오: `출하 준비 완료` 묶음을 펼쳐 하위 row 라벨을 읽는다. 라벨이 내부 품목 단계명보다 실제 작업 동작을 먼저 설명하는지 확인한다.
- 우선순위: 중간

### 5. 수량·재고 칸의 보조 재고 표기 규칙 통일
- 화면/영역: PC 입출고 내역 탭 > 목록 > 수량·재고 컬럼
- 증상: 상위 row는 `이동 2품목 · 22 EA` 아래 `창고 386 EA`처럼 요약 수량과 기준 재고가 함께 보인다. 하위 row는 어떤 경우에는 `11 EA`만 보이고, 어떤 경우에는 `생산 4 EA` 아래 `출하 404 EA`처럼 보조 재고가 붙는다.
- 사용자 불편: 같은 수량·재고 컬럼에서 어떤 숫자가 이번 거래 수량이고, 어떤 숫자가 처리 후 재고인지 유형마다 해석해야 한다. 특히 펼침 하위 row에서 `11 EA`만 있으면 처리 전/후 또는 위치 기준이 불명확하다.
- 확정 정책: 수량·재고 칸은 `이번 변화량`과 `처리 후 기준 재고`를 분리해서 표시한다. 보조 재고를 표시할 수 없는 경우에도 `-`만 두기보다 의도적으로 숨김인지, 데이터 없음인지 규칙을 정한다.
- 구현 방향: `QuantityStockCell`과 하위 row presentation에서 변화량 label과 stock label의 출력 조건을 정리한다. 단품 이동 하위 row에도 가능하면 `이동/차감 수량`과 `처리 후 위치 재고`를 같은 위치에 보여준다.
- 수용 기준: 사용자는 모든 row에서 첫 pill을 이번 변화량으로, 아래 보조 텍스트를 처리 후 재고로 읽을 수 있다. 유형별로 숫자의 의미가 달라지지 않는다.
- 브라우저 확인 시나리오: `창고 -> 부서`, `품목 전환`, `출하 준비 완료`, `수량 조정` row를 비교한다. 수량 pill과 아래 보조 재고 텍스트의 의미가 모두 같은 규칙으로 읽히는지 확인한다.
- 우선순위: 높음

### 6. 상위 row 요약과 펼침 하위 row 합계가 자연스럽게 대응되게 만들기
- 화면/영역: PC 입출고 내역 탭 > 목록 > 묶음 상위 row와 펼침 하위 row
- 증상: 상위 row에는 `준비 2품목 · 8 EA`, `사양 전환 2품목 · 10 EA`, `이동 2품목 · 22 EA`처럼 묶음 요약이 표시된다. 펼침 하위 row를 보면 어떤 하위 row들이 그 합계를 구성하는지 바로 연결되지 않는 경우가 있다.
- 사용자 불편: 사용자는 상위 요약이 하위 row의 어떤 수량을 합친 값인지 확인하려면 각 하위 row를 다시 계산해야 한다. 요약이 정확해도 검산 가능한 구조로 보이지 않으면 신뢰가 떨어진다.
- 확정 정책: 상위 row의 `N품목 · N EA` 요약은 펼침 하위 row의 수량 표현과 계산 기준이 맞아야 한다. 합계 대상에서 제외되는 라인이 있으면 하위 row에서 시각적으로 구분한다.
- 구현 방향: 상위 movement summary와 하위 line presentation의 집계 기준을 같은 helper 기준으로 맞춘다. 필요하면 상위 row에 `요약 기준`을 더 짧게 바꾸거나, 하위 row에 `포함/제외` 상태를 드러내 합계 근거가 보이게 한다.
- 수용 기준: 묶음을 펼쳤을 때 상위 row의 품목 수와 수량이 하위 row에서 자연스럽게 검산된다. 사용자가 “이 8 EA가 어디서 나왔지?”라고 되묻지 않아도 된다.
- 브라우저 확인 시나리오: `출하 준비 완료`, `품목 전환`, `창고 -> 부서` 묶음을 펼쳐 상위 row의 요약 수량과 하위 row 수량을 비교한다.
- 우선순위: 중간

### 7. 목록과 우측 상세 패널의 설명 언어를 같은 규칙으로 맞추기
- 화면/영역: PC 입출고 내역 탭 > 목록과 우측 상세 패널
- 증상: 같은 거래를 선택했을 때 목록에서는 `생산 4 EA / 출하 404 EA`처럼 보이고, 우측 상세 패널에서는 `처리 전 400 / 처리 후 404`, `재고 영향`, `출하 생산 +4`처럼 더 자세한 언어로 다시 설명된다. 정보는 유용하지만 목록과 상세가 서로 다른 문법으로 말한다.
- 사용자 불편: 목록에서 본 표현과 상세에서 본 표현을 사용자가 머릿속에서 매핑해야 한다. 특히 검토 중에는 “목록의 수량 pill이 상세의 어느 항목과 같은 값인지”가 즉시 보이지 않는다.
- 확정 정책: 목록, 펼침 row, 우측 상세 패널은 같은 기본 용어를 공유한다. 상세 패널은 더 자세하게 설명하되, 목록의 핵심 라벨과 수량 표현을 그대로 다시 확인할 수 있어야 한다.
- 구현 방향: 목록 presentation helper와 상세 패널 summary helper의 라벨/수량 용어를 맞춘다. 상세 패널 상단에는 목록에서 본 `흐름`과 `수량·재고` 요약을 같은 문구로 재표시하고, 아래에 처리 전/후와 재고 영향 같은 상세 정보를 덧붙인다.
- 수용 기준: 사용자가 row를 클릭했을 때 목록에서 본 핵심 문구가 우측 상세 패널 상단에서도 같은 말로 보인다. 상세 패널의 추가 정보는 목록 표현을 대체하지 않고 보강한다.
- 브라우저 확인 시나리오: `출하 준비 완료` 하위 row나 `수량 조정` row를 선택한다. 목록의 흐름/수량 표현과 우측 상세 패널 상단 표현이 같은 용어로 연결되는지 확인한다.
- 우선순위: 중간

### 8. `흐름` 열의 의미를 한 가지 기준으로 정리
- 화면/영역: PC 입출고 내역 탭 > 목록 > `흐름` 컬럼
- 증상: `고압`, `조립`은 재고 위치나 공정처럼 보이고, `사양 전환`, `구성 전환`, `출하`, `출하 준비`는 작업 의도처럼 보인다. 같은 `흐름` 열 안에서 위치, 작업 종류, 처리 단계가 섞여 있다.
- 사용자 불편: 사용자는 `흐름` 열을 볼 때 어떤 기준으로 읽어야 하는지 매번 해석해야 한다. 한 row에서는 위치를 보고, 다른 row에서는 작업 의도를 보게 되어 열 이름과 정보 역할이 흔들린다.
- 정책 확인 필요: `흐름` 열을 `재고 위치/이동 방향` 중심으로 둘지, `작업 단계/의도` 중심으로 둘지 결정해야 한다. 권장 방향은 입출고 내역의 목적에 맞춰 `어디 재고가 어떻게 움직였는가`를 보여주는 위치/이동 방향 중심이다.
- 구현 방향: 목록 presentation에서 `흐름` 열의 책임을 재정의한다. 작업 종류는 `작업` 배지에 두고, `흐름` 열에는 `창고 -> 진공`, `고압`, `조립`, `출하`처럼 재고 위치나 이동 방향 기준으로 통일한다. 작업 의도 설명이 필요하면 상세 패널이나 보조 hint로 내린다.
- 수용 기준: 사용자가 `흐름` 열만 봐도 해당 거래가 어느 재고 위치 또는 이동 방향과 관련 있는지 같은 기준으로 읽을 수 있다. 작업 의도와 위치가 같은 열 안에서 뒤섞이지 않는다.
- 브라우저 확인 시나리오: 생산, 분해, 품목 전환, 출하 준비, 창고 이동 row의 `흐름` 열을 비교한다. 모든 row가 같은 기준으로 읽히는지 확인한다.
- 우선순위: 높음

### 9. 작업/흐름/수량 열의 중복 라벨 줄이기
- 화면/영역: PC 입출고 내역 탭 > 목록 > 작업, 흐름, 수량·재고 컬럼
- 증상: `품목 전환 / 사양 전환 / 사양 전환 2품목 · 10 EA`처럼 같은 성격의 라벨이 작업, 흐름, 수량 칸에 반복된다. `출하 준비 완료 / 출하 준비 / 준비 2품목 · 8 EA`도 비슷하게 같은 의미가 여러 열에 나뉘어 반복된다.
- 사용자 불편: 여러 열이 서로 다른 정보를 주기보다 같은 말을 조금씩 바꿔 반복하면, 목록의 정보 밀도만 높아지고 실제 판단 속도는 떨어진다.
- 확정 정책: 작업 배지, 흐름 열, 수량·재고 열은 각각 다른 역할을 가져야 한다. 작업 배지는 거래 종류, 흐름 열은 위치/방향, 수량·재고 열은 변화량과 처리 후 재고를 맡는다.
- 구현 방향: presentation helper에서 같은 라벨이 여러 열에 반복되는 케이스를 줄인다. 예를 들어 작업 배지가 `품목 전환`이면 흐름은 `창고/조립/출하` 같은 재고 위치 기준으로, 수량은 `2품목 · 10 EA`처럼 변화량 중심으로 압축한다.
- 수용 기준: 같은 row 안에서 같은 단어가 불필요하게 2~3번 반복되지 않는다. 각 열을 읽었을 때 새 정보가 하나씩 추가된다.
- 브라우저 확인 시나리오: 품목 전환, 출하 준비 완료, 출하 픽업 완료 row를 비교한다. 작업/흐름/수량 열이 서로 다른 정보를 제공하는지 확인한다.
- 우선순위: 중간

### 10. `상위/하위` 표현을 현장 동작 언어로 바꾸기
- 화면/영역: PC 입출고 내역 탭 > 목록 > 생산/분해/BOM 계열 row의 수량·재고 컬럼
- 증상: 생산/분해 row에서 `상위 +31 EA`, `하위 -434 EA`처럼 BOM 구조 기반 문구가 표시된다. BOM 구조를 아는 사람에게는 의미가 있지만, 현장 작업 관점에서는 무엇이 입고되고 무엇이 차감됐는지 한 번 더 해석해야 한다.
- 사용자 불편: 사용자는 재고 변화 내역을 보고 싶은데, `상위/하위`라는 구조 용어가 먼저 보여 실제 동작인 완제품 입고, 부품 차감, 부품 회수의 의미가 늦게 읽힌다.
- 확정 정책: 기본 목록 표현은 현장 동작 언어를 우선한다. 생산은 `완제품 +N / 부품 -N`, 분해는 `완제품 -N / 부품 +N`처럼 재고 변화의 방향이 바로 보이게 한다.
- 구현 방향: 생산/분해 movement summary에서 `상위/하위`를 기본 라벨로 쓰지 않고, 작업 유형에 따라 `완제품`, `부품`, `회수`, `차감`, `입고` 같은 현장 동작 라벨로 변환한다. BOM 구조 용어가 필요하면 상세 패널의 구성 라인에서 보조적으로 제공한다.
- 수용 기준: 생산/분해 row를 처음 보는 사용자가 무엇이 늘고 줄었는지 수량 pill만 보고 이해할 수 있다.
- 브라우저 확인 시나리오: 생산 row와 분해 row를 비교한다. `상위/하위` 대신 실제 재고 변화 동작이 먼저 보이는지 확인한다.
- 우선순위: 높음

### 11. 요청자 칸의 사람/시스템 주체 구분
- 화면/영역: PC 입출고 내역 탭 > 목록 > 상태·처리 컬럼
- 증상: 품목 전환 row에 `요청 구성품 변경`처럼 사람 이름이 아닌 시스템성 문구가 요청자 위치에 보인다. 다른 row의 `요청 김건호`, `요청 이계숙`과 같은 문법으로 보여 실제 요청자인지 시스템 처리명인지 구분이 약하다.
- 사용자 불편: 사용자는 누가 작업했는지 추적하려고 상태·처리 칸을 보는데, 시스템 생성 주체가 사람 요청자처럼 보이면 감사/확인 맥락이 흐려진다.
- 정책 확인 필요: 시스템 생성 거래의 주체를 `시스템`, `자동 처리`, `구성품 변경` 중 어떤 라벨로 표시할지 결정해야 한다. 권장 방향은 사람 이름이 없으면 `시스템 처리`처럼 사람 요청자와 다른 라벨을 쓰는 것이다.
- 구현 방향: people/status presentation에서 요청자가 실제 직원 이름인지 시스템성 문자열인지 구분한다. 시스템성 주체는 `요청` 접두를 쓰지 않거나, 별도 chip/tone으로 표시한다.
- 수용 기준: 목록에서 실제 사람 요청자와 시스템/자동 처리 주체가 같은 모양으로 혼동되지 않는다.
- 브라우저 확인 시나리오: 품목 전환 row와 일반 생산/출하 row의 상태·처리 칸을 비교한다. 사람 이름과 시스템성 주체가 구분되는지 확인한다.
- 우선순위: 중간

### 12. 자동 처리/승인 없음 상태의 표시 규칙 고정
- 화면/영역: PC 입출고 내역 탭 > 목록과 우측 상세 패널 > 상태·처리 영역
- 증상: 어떤 거래는 승인자가 표시되고, 어떤 자동 처리 거래는 승인 줄이 숨겨진다. 현재 규칙이 명확히 드러나지 않으면 승인 누락인지 자동 처리인지 사용자가 판단해야 한다.
- 사용자 불편: 승인 절차가 없는 거래인데 승인자가 빠진 것처럼 보이거나, 반대로 자동 승인처럼 오해하면 실제 처리 흐름을 잘못 이해할 수 있다.
- 확정 정책: 승인 절차가 없는 자동 처리 거래는 승인자 줄을 억지로 표시하지 않는다. 다만 자동 처리라는 사실이 업무상 중요한 거래는 `자동 처리` 또는 `승인 없음`의 짧은 상태 라벨로 구분한다.
- 구현 방향: 거래 유형별 승인 필요 여부를 presentation 레벨에서 분리한다. 승인자가 없는 값, 승인 절차가 없는 값, 승인 대기 값을 같은 placeholder로 처리하지 않는다.
- 수용 기준: 사용자는 승인자가 없는 row를 보고 데이터 누락인지, 승인 절차가 없는 자동 처리인지 구분할 수 있다.
- 브라우저 확인 시나리오: 생산, 품목 전환, 창고 이동, 수량 조정 row의 상태·처리 칸과 상세 패널을 비교한다. 승인자 표시 규칙이 거래 유형별로 일관적인지 확인한다.
- 우선순위: 중간

### 13. 메모/사유/제외 같은 보조 chip 위치 통일
- 화면/영역: PC 입출고 내역 탭 > 목록 > 상태·처리 컬럼과 보조 상태 chip
- 증상: `메모`, `사유`, `제외` 같은 보조 상태가 row마다 다른 맥락으로 붙어 보일 수 있다. 요청자/승인자 정보와 섞이면 보조 정보인지 처리 상태인지 즉시 구분하기 어렵다.
- 사용자 불편: 사용자는 메모나 사유가 있는 row를 빠르게 찾아야 하지만, 보조 chip 위치가 흔들리면 눈으로 스캔하기 어렵다. 반대로 요청자/승인자보다 보조 chip이 더 튀면 처리 정보의 위계가 깨진다.
- 확정 정책: 보조 chip은 같은 위치, 같은 크기, 같은 색 문법으로 통일한다. 메모/사유는 처리자 정보와 섞지 않고 보조 정보로만 읽히게 한다.
- 구현 방향: 상태·처리 셀 안에서 `요청자/승인자`, `상태 chip`, `메모/사유 chip`의 순서와 간격을 고정한다. 메모가 없으면 placeholder를 보이지 않게 하고, 메모가 있으면 같은 위치에 작은 chip으로만 표시한다.
- 수용 기준: 사용자가 목록을 스캔할 때 메모/사유/제외가 있는 row를 같은 위치에서 찾을 수 있다. 보조 chip 때문에 요청자/승인자 정보가 밀리거나 섞이지 않는다.
- 브라우저 확인 시나리오: 메모가 있는 창고 이동 row, 사유가 있는 불량 row, 제외가 있는 row를 비교한다. 보조 chip 위치와 크기가 같은지 확인한다.
- 우선순위: 중간

### 14. 대표 row와 비슷한 시간대 중복 row의 묶음 기준 명확화
- 화면/영역: PC 입출고 내역 탭 > 목록 > 같은 시간대/같은 대상의 연속 row
- 증상: 같은 시간대와 대상에서 `여러 위치 부품 차감 + 완제품 입고` row와 `조립` row가 별도 대표 row처럼 이어져 보일 수 있다. 실제로 같은 작업 묶음의 일부인지, 서로 다른 작업인지 화면만 보고 판단하기 어렵다.
- 사용자 불편: 사용자는 같은 작업이 두 번 발생한 것인지, 하나의 작업이 여러 거래로 쪼개진 것인지 구분해야 한다. 묶음 기준이 불명확하면 감사/추적 상황에서 혼란이 생긴다.
- 정책 확인 필요: 같은 operation batch나 reference로 묶을 수 있는 row를 대표 row 하나와 펼침 하위 row로 볼지, 현재처럼 일부 row를 별도 대표 row로 유지할지 결정해야 한다. 권장 방향은 같은 작업 묶음이면 대표 row 하나로 묶고 펼침에서 세부 거래를 보여주는 것이다.
- 구현 방향: grouping 기준을 점검한다. 같은 작업을 구성하는 transaction들이 별도 대표 row로 노출되는 케이스가 있으면 묶음 기준을 보정하거나, 별도 row로 유지해야 하는 이유를 라벨로 드러낸다.
- 수용 기준: 사용자가 비슷한 시간대의 연속 row를 보고 같은 작업이 중복 처리된 것으로 오해하지 않는다.
- 브라우저 확인 시나리오: 생산/분해처럼 같은 시간대에 여러 row가 이어지는 구간을 확인한다. 대표 row와 세부 row의 관계가 명확한지 본다.
- 우선순위: 중간

### 15. 상단 KPI/필터와 목록 결과의 관계 명확화
- 화면/영역: PC 입출고 내역 탭 > 상단 KPI/필터 영역과 목록 헤더
- 증상: 상단에 `이번달 437건`, `창고 67건`, `부서 336건`, `수량조정 12건` 같은 숫자가 보이지만, 현재 목록에 적용된 필터 결과인지 표시 전용 KPI인지 즉시 명확하지 않다. 코드상 KPI 박스는 표시 전용이고 실제 필터는 별도 패널인데, 화면에서는 이 차이가 강하게 드러나지 않는다.
- 사용자 불편: 사용자는 KPI를 눌렀을 때 필터가 바뀌는지, 단순 요약인지 예측하기 어렵다. 목록 헤더의 `100 / 437건`과 상단 KPI 숫자의 관계도 한 번 더 해석해야 한다.
- 확정 정책: KPI가 표시 전용이면 표시 전용으로 보이게 하고, 클릭 가능한 필터처럼 오해되지 않게 한다. 목록 헤더의 건수와 상단 KPI의 기준이 다르면 문구로 구분한다.
- 구현 방향: KPI 카드 또는 filter chip에 `현재 기간 요약`, `표시 전용`, `전체 조건` 같은 기준 문구를 보강한다. 클릭 가능한 UI가 아니라면 hover/active 스타일도 필터 버튼과 다르게 둔다.
- 수용 기준: 사용자는 KPI 숫자와 목록 건수가 어떤 관계인지 바로 이해할 수 있다. KPI를 눌렀을 때 동작을 예측할 수 있다.
- 브라우저 확인 시나리오: 입출고 내역 탭 상단 KPI와 목록 헤더 건수를 함께 본다. 기간 필터를 바꿨을 때 숫자 기준이 자연스럽게 이해되는지 확인한다.
- 우선순위: 중간

### 16. 목록 row 높이와 밀도 기준 통일
- 화면/영역: PC 입출고 내역 탭 > 목록 row 전체
- 증상: 거래 유형에 따라 row 높이가 미묘하게 다르게 보인다. 긴 품명, 두 줄 수량 pill, 보조 chip 유무에 따라 특정 row만 높거나 무거워 보인다.
- 사용자 불편: row 높이와 밀도가 흔들리면 목록을 빠르게 스캔하기 어렵다. 특정 거래가 실제로 중요한 것이 아니라 단순히 내용이 길어서 더 강조된 것처럼 보일 수 있다.
- 확정 정책: 대표 row와 펼침 하위 row 각각의 표준 높이/밀도 기준을 둔다. 내용이 길어도 정해진 줄 수와 축약 규칙 안에서 처리한다.
- 구현 방향: row padding, line-height, title clamp, pill height, 보조 chip 위치를 유형별이 아니라 row 역할별로 통일한다. 대표 row는 대표 row끼리, 하위 row는 하위 row끼리 같은 리듬을 갖게 한다.
- 수용 기준: 생산, 불량, 품목 전환, 출하 준비, 창고 이동 row를 나란히 봤을 때 row 높이와 밀도가 안정적으로 보인다.
- 브라우저 확인 시나리오: 다양한 거래 유형이 섞인 구간에서 row 높이와 내부 요소 정렬을 비교한다. 긴 품명이 있는 row만 과하게 커지지 않는지 확인한다.
- 우선순위: 높음

### 17. 우측 상세 패널 첫 화면을 `결론 먼저`로 강화
- 화면/영역: PC 입출고 내역 탭 > 우측 상세 패널
- 증상: 상세 패널은 처리 전/후, 재고 영향, 구성 라인 등 많은 정보를 제공하지만, 사용자가 row를 클릭한 직후 “그래서 이 거래의 결론이 무엇인가”를 가장 먼저 읽기 어렵다.
- 사용자 불편: 목록에서 궁금해서 상세를 열었는데 결론보다 근거 정보가 먼저 보이면, 사용자가 다시 목록의 수량 pill과 상세 내용을 매핑해야 한다.
- 확정 정책: 상세 패널 첫 화면은 결론 먼저 보여준다. 상단에는 `무슨 작업으로 어느 재고가 얼마나 변했는가`를 목록과 같은 용어로 요약하고, 아래에 처리 전/후와 재고 영향 같은 근거를 배치한다.
- 구현 방향: 상세 패널 상단 hero/summary 영역에 목록의 `작업`, `흐름`, `수량·재고` 핵심 문구를 같은 말로 재표시한다. 재고 영향과 구성 라인은 결론을 검증하는 하위 정보로 둔다.
- 수용 기준: 상세 패널을 열자마자 거래 결론을 1~2줄 안에서 이해할 수 있다. 아래 상세 정보는 결론을 보강하는 구조로 읽힌다.
- 브라우저 확인 시나리오: 생산, 품목 전환, 출하 준비, 수량 조정 row를 각각 선택한다. 상세 패널 상단에서 거래 결론이 먼저 보이는지 확인한다.
- 우선순위: 높음

### 18. 목록-펼침-상세 간 용어 사전 고정
- 화면/영역: PC 입출고 내역 탭 > 필터, 달력, 목록, 펼침 하위 row, 우측 상세 패널 전체
- 증상: 같은 거래가 목록에서는 `입고 1 EA`, 펼침에서는 `회수 입고`, 상세 결론에서는 `원자재 입고`, 상세 본문에서는 `흐름 품목 전환`, 재고 영향에서는 `조립 생산 +1`처럼 서로 다른 문법으로 설명된다. 필터의 `거래 종류`도 `생산 | 입고`, `분해 | 출고`, `자동 차감`처럼 목록 배지와 다른 언어를 쓴다.
- 사용자 불편: 사용자는 필터, 목록, 펼침, 상세의 문구를 머릿속에서 번역해야 한다. 특히 검토/감사 상황에서는 같은 수량과 같은 동작이 같은 말로 반복되어야 신뢰가 생긴다.
- 확정 정책: 핵심 용어는 하나의 presentation 규칙으로 공유한다. 필터, 달력, 목록, 펼침 row, 상세 패널은 자세함의 정도만 다르고 기본 라벨과 수량 의미는 같아야 한다.
- 구현 방향: 표시 helper의 용어 사전을 정리한다. 작업명, 흐름, 변화량, 처리 후 재고, 요청/승인 상태, 필터 거래 종류를 공통 vocabulary로 정의하고 각 영역이 같은 vocabulary를 재사용하게 한다.
- 수용 기준: 사용자가 필터에서 고른 문구를 목록, 펼침 row, 우측 상세 패널에서도 같은 말로 다시 확인할 수 있다. 상세 패널의 추가 정보가 목록 표현을 대체하지 않고 보강한다.
- 브라우저 확인 시나리오: 품목 전환 거래를 펼친 뒤 하위 row 하나를 선택한다. 필터 `거래 종류`, 목록 상위 row, 펼침 하위 row, 우측 상세 패널의 결론/재고 영향 문구가 같은 용어 체계로 연결되는지 본다.
- 우선순위: 높음

## 필터·달력·상세·상태 동선

### 19. 필터 `거래 종류`와 목록 작업 배지 용어 통일
- 화면/영역: PC 입출고 내역 탭 > 필터 패널 > 거래 종류, 목록 > 작업/흐름 컬럼
- 증상: 필터의 `거래 종류`에는 `원자재 입고`, `생산 | 입고`, `분해 | 출고`, `자동 차감`, `창고 반출`, `창고 반입` 같은 transaction type 중심 라벨이 보인다. 목록에는 `생산`, `분해`, `품목 전환`, `출하 준비 완료`, `출하 픽업 완료`처럼 batch/sub_type 기반 업무 라벨이 보인다.
- 사용자 불편: 사용자는 필터에서 어떤 값을 골라야 화면에 보이는 `품목 전환`이나 `출하 준비 완료`를 찾을 수 있는지 추측해야 한다. 필터는 고르는 말이고 목록은 확인하는 말인데, 두 말이 다르면 필터가 믿기 어렵다.
- 확정 정책: 필터의 `거래 종류`는 목록에서 사용자가 보는 작업 배지와 같은 분류 언어를 쓴다. 내부 transaction type은 필요하면 보조 설명이나 상세 검색 조건으로 내린다.
- 구현 방향: `OPERATION_OPTIONS`의 라벨을 목록 presentation vocabulary와 맞춘다. 백엔드 필터 값은 유지하더라도 화면 라벨은 `품목 전환`, `출하 준비`, `출하`, `생산`, `분해`, `수량 조정`, `불량 처리`처럼 사용자가 목록에서 보는 말로 정리한다.
- 수용 기준: 사용자가 필터에서 `품목 전환`을 고르면 목록에서도 `품목 전환` 배지가 보인다. 필터 라벨과 목록 라벨을 따로 번역하지 않아도 된다.
- 브라우저 확인 시나리오: `http://192.168.0.63:3001/mes?tab=history`에서 필터 패널을 열고 `거래 종류` 라벨을 확인한다. 같은 화면의 목록 작업 배지와 필터 라벨이 같은 단어 체계인지 비교한다.
- 우선순위: 높음

### 20. 필터 chip의 건수·기준·`미상` 표현 정리
- 화면/영역: PC 입출고 내역 탭 > 필터 패널 > 부서 구분, 모델 구분, 거래 종류
- 증상: `부서 구분` chip은 `조립 176`, `미상 92`, `창고 67`처럼 건수가 붙지만, `모델 구분`과 `거래 종류` chip은 같은 형태의 건수가 없다. `미상`은 실제 부서가 없는 값인지 데이터 정제가 안 된 값인지 화면만으로 알기 어렵다.
- 사용자 불편: 사용자는 어떤 필터가 결과 수를 예측하게 해주는지, 어떤 필터는 단순 선택지만 제공하는지 매번 다르게 읽어야 한다. `미상 92`는 꽤 큰 수인데도 원인이나 처리 기준이 보이지 않아 데이터 문제처럼 느껴진다.
- 정책 확인 필요: 모든 필터 chip에 현재 조건 기준 건수를 표시할지, 아니면 건수 없는 chip으로 통일할지 결정해야 한다. `미상`은 `부서 없음`, `분류 없음`, `기록 없음` 중 어떤 의미인지 정책명을 정해야 한다.
- 권장 초안: 필터 chip은 가능하면 같은 기준의 건수를 표시한다. `미상`은 원인에 따라 `부서 기록 없음`처럼 사용자가 조치 가능성을 이해하는 말로 바꾼다.
- 구현 방향: 필터 summary API나 프론트 derivation에서 모델/거래 종류별 count를 제공할 수 있는지 확인한다. 제공이 어렵다면 부서 count만 강조되지 않게 시각 위계를 낮추고, `미상`에는 tooltip 또는 보조 문구를 붙인다.
- 수용 기준: 세 필터 그룹의 chip 문법이 서로 크게 다르지 않다. `미상`을 보고 사용자가 실제 부서명인지 시스템 fallback인지 혼동하지 않는다.
- 브라우저 확인 시나리오: 필터 패널을 열어 `부서 구분`, `모델 구분`, `거래 종류` chip을 나란히 비교한다. 건수 표시와 `미상` 라벨의 의미가 자연스럽게 읽히는지 확인한다.
- 우선순위: 중간

### 21. 달력 날짜 칸의 총건수와 분류 합계 검산 가능하게 만들기
- 화면/영역: PC 입출고 내역 탭 > 달력 패널
- 증상: 날짜 칸에는 `1`, `73`, `창고 2건`, `부서 68건`처럼 날짜, 총건수, 분류 건수가 함께 보인다. 큰 숫자 `73`이 총 거래 수로 보이지만 `총 73건`처럼 라벨이 없어 의미를 한 번 더 해석해야 한다. 분류는 `창고/부서/조정`만 보여서 총건수와 합계가 맞는지도 바로 검산되지 않는다.
- 사용자 불편: 사용자는 달력에서 특정 날짜를 고르기 전에 이 숫자가 전체 거래인지, 현재 필터 결과인지, 로드된 목록 수인지 헷갈릴 수 있다. 분류 합계와 총건수의 관계가 모호하면 날짜 선택 기준도 약해진다.
- 확정 정책: 달력 날짜 칸의 큰 숫자는 `총 N건`처럼 의미를 명시한다. 분류 합계와 총건수의 관계가 다르면 빠진 분류를 `기타 N건` 또는 보조 설명으로 드러낸다.
- 구현 방향: 날짜 셀의 count label을 `총 N건`으로 바꾸고, legend에 총건수와 분류 기준을 함께 설명한다. `창고/부서/조정` 외 거래가 있으면 숨기지 말고 `기타` 또는 더 정확한 거래 분류로 표시한다.
- 수용 기준: 사용자가 달력 날짜 칸만 보고 날짜, 총 거래 수, 분류별 거래 수를 구분할 수 있다. 총건수와 분류 합계가 왜 다를 수 있는지 화면 안에서 이해된다.
- 브라우저 확인 시나리오: 달력 패널을 열고 2026년 7월 1일, 2일, 7일처럼 거래가 많은 날짜를 본다. `73`, `75`, `137` 같은 숫자가 무엇인지 명시되어 있고, 아래 분류 수와 자연스럽게 연결되는지 확인한다.
- 우선순위: 중간

### 22. `펼침 닫기` 버튼의 범위와 위치 명확화
- 화면/영역: PC 입출고 내역 탭 > 목록 sticky 헤더 > 펼침 닫기 버튼
- 증상: 묶음 row를 펼치면 목록 헤더 오른쪽에 `펼침 닫기` 버튼이 나타난다. 버튼이 현재 펼친 묶음 하나를 닫는지, 모든 펼침을 닫는지, 선택된 상세 패널도 닫는지 문구만으로 분명하지 않다.
- 사용자 불편: 사용자는 펼쳐진 row 안에서 닫기 버튼을 찾다가 헤더의 전역 버튼을 보게 된다. 특히 필터/달력/상세 패널이 함께 열린 상태에서는 `펼침 닫기`가 어떤 화면 상태를 정리하는지 애매하다.
- 확정 정책: `펼침 닫기`는 현재 펼쳐진 묶음만 닫는지, 전체 펼침을 닫는지 범위를 문구와 위치로 드러낸다.
- 구현 방향: 단일 묶음만 펼칠 수 있는 정책이면 `현재 묶음 접기`처럼 문구를 바꾸거나, 펼친 row 가까이에 접기 affordance를 둔다. 여러 묶음 지원 정책으로 바꿀 경우 `모두 접기` 문구와 카운트를 표시한다.
- 수용 기준: 사용자가 버튼을 누르기 전에도 닫히는 범위를 알 수 있다. 펼침 하위 row, 필터 패널, 우측 상세 패널의 닫기와 혼동되지 않는다.
- 브라우저 확인 시나리오: 품목 전환 row를 펼치고 목록 헤더의 `펼침 닫기`를 확인한다. 버튼 문구와 위치만 보고 어떤 범위가 접히는지 예측 가능한지 본다.
- 우선순위: 중간

### 23. 하위 row 선택 상태와 우측 상세 패널 대상 연결 표시
- 화면/영역: PC 입출고 내역 탭 > 펼침 하위 row, 우측 상세 패널
- 증상: 펼침 하위 row를 클릭하면 우측 상세 패널은 해당 하위 line의 품목을 보여주지만, 목록에서는 상위 묶음 row의 강조가 더 크게 보인다. 사용자가 지금 상세 패널이 상위 묶음 전체인지, 특정 하위 row 한 줄인지 구분하기 어렵다.
- 사용자 불편: 상세 패널의 `원자재 입고`, `입고 1 EA`가 펼침의 어느 `회수 입고` line과 연결되는지 눈으로 다시 찾아야 한다. 선택 상태가 정확히 표시되지 않으면 상세 검토 중 다른 row를 보고 있다고 오해할 수 있다.
- 확정 정책: 우측 상세 패널이 하위 row를 설명할 때는 해당 하위 row 자체에도 선택 상태가 보여야 한다. 상위 row는 묶음 컨텍스트로만 강조하고, 실제 상세 대상은 별도 highlight로 구분한다.
- 구현 방향: `selection.kind === "log"`이고 해당 log가 펼침 하위 line이면 하위 row에 selected style을 적용한다. 상위 row highlight와 하위 row highlight의 색/테두리/배경 강도를 다르게 둔다.
- 수용 기준: 우측 상세 패널을 보고 있는 동안 사용자는 목록에서 어떤 row가 상세 대상인지 한눈에 알 수 있다. 상위 묶음과 하위 line의 선택 상태가 혼동되지 않는다.
- 브라우저 확인 시나리오: 품목 전환 묶음을 펼치고 `회수 입고` 하위 row를 선택한다. 우측 상세 패널 제목과 목록의 선택 highlight가 같은 하위 row를 가리키는지 확인한다.
- 우선순위: 높음

### 24. 우측 상세 패널 `취소` 액션의 범위와 조건 명확화
- 화면/영역: PC 입출고 내역 탭 > 우측 상세 패널 > 하단 메타/취소 영역
- 증상: 상세 패널 하단에 `취소` 버튼이 표시된다. 누르면 사유와 PIN을 요구하지만, 버튼을 누르기 전에는 이 취소가 단일 로그만 되돌리는지, 펼침 묶음 전체를 취소하는지, 어떤 재고 영향이 반대로 생성되는지 알기 어렵다.
- 사용자 불편: 입출고 내역은 이미 반영된 재고 기록이라 `취소`가 위험 액션이다. 범위가 불명확하면 사용자는 버튼을 눌러 확인 모달을 보기 전까지 실제 영향도를 판단할 수 없다.
- 확정 정책: 주 버튼은 단건·묶음 모두 `이 내역 취소`로 짧게 통일한다. 취소 범위, 대상 수량, 영향 위치, 권한 조건은 버튼 주변의 보조 문구와 취소 확인 화면에서 명확히 보여준다.
- 구현 방향: `이 작업 묶음 전체 취소`, `배치 취소`처럼 내부 구조를 전면에 내세우는 CTA를 `이 내역 취소`로 통일한다. confirm 영역에는 `이 내역에 포함된 N건의 재고 변동을 함께 취소합니다`처럼 실제 범위와 되돌릴 재고 영향 요약, 취소 사유/PIN을 함께 배치한다.
- 수용 기준: 사용자는 버튼 문구를 읽을 때 내부 묶음 개념을 해석할 필요가 없고, 버튼을 누르기 전후에는 취소되는 실제 범위와 영향을 알 수 있다.
- 브라우저 확인 시나리오: 품목 전환 하위 row와 생산 묶음을 각각 선택한다. 주 버튼은 모두 `이 내역 취소`로 읽히며, 취소 확인 단계에서만 단건/여러 건 범위가 구체적으로 설명되는지 확인한다.
- 우선순위: 높음
- 구현 상태: 완료. 단건·묶음 CTA를 `이 내역 취소`로 통일하고, 확인 단계에서만 실제 대상 건수와 되돌릴 재고 영향을 표시한다.

### 25. 상세 패널 결론 카드와 재고 영향 카드의 용어 연결 강화
- 화면/영역: PC 입출고 내역 탭 > 우측 상세 패널 > 결론 카드, 재고 영향 카드
- 증상: 같은 품목 전환 하위 row가 상세 상단에서는 `원자재 입고`, 결론 카드에서는 `흐름 품목 전환 / 입고 1 EA`, 재고 영향에서는 `조립 생산 +1`로 표시된다. 각 정보는 맞을 수 있지만, 사용자가 같은 거래의 다른 관점이라는 것을 바로 연결하기 어렵다.
- 사용자 불편: 상세 패널은 검산을 도와야 하는데, 결론과 근거가 서로 다른 용어를 쓰면 사용자가 “입고 1 EA가 왜 조립 생산 +1이지?”라고 다시 해석해야 한다.
- 확정 정책: 상세 패널의 첫 결론, 본문 요약, 재고 영향은 같은 동작명과 같은 위치명을 공유한다. 재고 영향은 더 기술적인 표현을 쓰더라도 목록/결론 용어와 연결 라벨을 함께 둔다.
- 구현 방향: `HistoryDetailHero`, `HistoryDetailDecisionSummary`, `HistoryInventoryEffectPanel`이 같은 presentation vocabulary를 참조하게 한다. 재고 영향 row에는 `결론의 입고 1 EA`와 대응되는 source label을 표시하거나 용어를 통일한다.
- 수용 기준: 상세 패널 안에서 상단 결론, 수량·재고 카드, 재고 영향 카드가 같은 거래를 같은 말로 설명한다. 사용자는 수량 검산을 카드 간에 자연스럽게 할 수 있다.
- 브라우저 확인 시나리오: 품목 전환 하위 row를 선택해 상세 패널을 확인한다. `원자재 입고`, `입고 1 EA`, `조립 생산 +1`이 같은 의미 체계로 연결되는지 본다.
- 우선순위: 높음

### 26. 우측 상세 패널 열린 상태의 목록 압축 기준 검증
- 화면/영역: PC 입출고 내역 탭 > 목록 + 우측 상세 패널 2열 레이아웃
- 증상: 우측 상세 패널이 열리면 목록 컬럼 폭이 줄어들고, 흐름/수량/상태 영역이 압축된다. 긴 품명과 펼침 하위 row가 함께 있을 때 목록에서 핵심 정보가 잘리거나, 어느 열이 우선인지 흐려질 수 있다.
- 사용자 불편: 사용자는 상세 패널을 보면서 목록의 다른 row와 비교해야 한다. 패널이 열렸다는 이유로 목록의 판단 정보가 줄어들면, 상세 검토와 목록 탐색을 오가기가 어렵다.
- 정책 확인 필요: 상세 패널이 열렸을 때도 목록의 1순위 정보인 흐름/수량 변화는 유지할지, 아니면 상세 패널이 열린 동안 목록을 탐색 보조 수준으로 낮출지 결정해야 한다.
- 권장 초안: 상세 패널이 열려도 `작업`, `대상`, `흐름`, `수량·재고`는 한눈에 남겨 둔다. 덜 중요한 `품목코드`나 처리자 정보부터 압축한다.
- 구현 방향: compact column spec을 실제 사용 순서 기준으로 재조정한다. 패널 열린 상태에서 긴 품명, 펼침 하위 row, 2개 pill 수량, 요청자 표시가 겹치는 케이스를 기준으로 폭/ellipsis/tooltip 정책을 잡는다.
- 수용 기준: 상세 패널이 열린 상태에서도 목록의 핵심 재고 변화가 사라지거나 과도하게 잘리지 않는다. 사용자는 목록과 상세를 동시에 비교할 수 있다.
- 브라우저 확인 시나리오: 우측 상세 패널을 연 뒤 품목 전환 묶음과 긴 품명 생산 row를 비교한다. 목록의 흐름/수량/상태가 패널 닫힘 상태와 같은 우선순위로 읽히는지 확인한다.
- 우선순위: 중간

### 27. 상세/펼침/필터/달력 상태와 브라우저 뒤로가기 동기화
- 화면/영역: PC 입출고 내역 탭 > 브라우저 history, 상세 패널, 펼침 row, 필터 패널, 달력 패널
- 증상: 코드상 우측 상세 패널의 내부 드릴 이동은 browser history에 쌓지만, 일반 row 선택, 묶음 펼침, 필터 패널 열기, 달력 패널 열기는 별도 history state로 쌓이지 않는다. 사용자는 화면 단계처럼 느끼는 상태에서 브라우저 뒤로가기를 눌렀을 때 무엇이 닫힐지 예측하기 어렵다.
- 사용자 불편: 상세 패널이나 필터/달력이 열린 상태에서 뒤로가기를 눌렀을 때 패널이 닫히길 기대할 수 있지만, 실제로는 이전 탭/페이지로 이동할 수 있다. 반대로 내부 드릴만 history에 쌓이면 어떤 상태는 뒤로가기가 되고 어떤 상태는 안 되는 일관성 문제가 생긴다.
- 정책 확인 필요: 입출고 내역 탭에서 browser history에 쌓을 UI 상태 범위를 정해야 한다. 후보는 `상세 패널 열림`, `상세 내부 드릴`, `필터/달력 패널`, `선택 날짜`, `펼침 묶음`이다.
- 권장 초안: 사용자가 화면 단계로 인식하는 `상세 패널 열림/내부 드릴/필터·달력 열림`은 뒤로가기로 닫히게 한다. 단순 row 펼침은 목록 내부 상태로 보고 URL/history에 쌓지 않아도 된다.
- 구현 방향: `DesktopHistoryView`의 selection, filterPanelOpen, calendarOpen, selectedDay 상태와 browser history 정책을 명확히 분리한다. 뒤로가기 시 닫히는 순서를 문서화하고 테스트로 고정한다.
- 수용 기준: 사용자가 브라우저 뒤로가기를 눌렀을 때 현재 화면에서 가장 최근에 연 단계가 먼저 닫힌다. 뒤로가기 후 선택 row, 펼침 상태, 목록 스크롤이 어긋나지 않는다.
- 브라우저 확인 시나리오: row 선택으로 상세 패널을 열고, 필터 패널과 달력 패널을 각각 연 뒤 브라우저 뒤로가기를 눌러본다. 사용자가 기대하는 순서대로 닫히거나 이동하는지 확인한다.
- 우선순위: 높음

### 28. `100 / 437건`과 `100건 더보기`의 로드 범위 명확화
- 화면/영역: PC 입출고 내역 탭 > 목록 sticky 헤더, 더보기 버튼
- 증상: 목록 헤더에는 `100 / 437건`과 `07/07 13:56 ~ 07/08 11:42` 같은 날짜 범위가 표시되고, 하단에는 `100건 더보기` 버튼이 있다. 날짜 범위가 현재 로드된 100건의 범위인지, 전체 437건의 기간 범위인지 문구만으로 분명하지 않다.
- 사용자 불편: 사용자는 현재 화면에 보이는 기간이 전체 조건의 범위인지, 아직 더 불러오지 않은 데이터 때문에 잘린 범위인지 판단해야 한다. 더보기 후 날짜 범위가 어떻게 바뀌는지도 예측하기 어렵다.
- 확정 정책: 목록 헤더의 건수와 날짜 범위는 `현재 로드됨`과 `전체 조건`을 분리해 표시한다. 더보기 버튼은 다음에 몇 건을 추가로 불러오는지, 전체 중 어디까지 왔는지 명확히 보여준다.
- 구현 방향: 헤더 문구를 `현재 100건 표시 / 전체 437건`, `표시 범위 07/07 13:56 ~ 07/08 11:42`처럼 구체화한다. 더보기 버튼은 `다음 100건 더보기` 또는 `100건 추가로 불러오기`처럼 동작을 드러낸다.
- 수용 기준: 사용자는 현재 보이는 목록이 전체가 아니라 일부라는 점과, 날짜 범위가 표시된 row 기준이라는 점을 바로 이해한다.
- 브라우저 확인 시나리오: 입출고 내역 목록 하단의 `100건 더보기`를 보기 전후로 헤더 건수와 날짜 범위를 비교한다. 더보기 전후 의미가 같은 문법으로 유지되는지 확인한다.
- 우선순위: 중간

### 29. 품목 전환 요청자 기록 누락을 없애고 `자동 처리` 상태와 분리
- 화면/영역: PC 입출고 > 품목 전환 실행, 입출고 내역 탭 > 목록 상태·처리 컬럼과 우측 상세 패널
- 증상: 품목 전환 row에 `시스템 처리 · 구성품 변경`이 요청자처럼 표시된다. 코드 확인 결과 독립 품목 전환의 프론트 `ItemConversionPayload`와 백엔드 `ItemConversionExecuteRequest`가 요청자 ID를 받지 않고, `shipping.py`가 생성 로그의 `produced_by`를 `구성품 변경`으로 고정한다. 입출고 내역 presentation은 이 값을 다시 `시스템 처리 · 구성품 변경`으로 바꾼다.
- 사용자 불편: 실제로 로그인한 사람이 실행한 품목 전환인데도 이력에서는 작업자가 사라진다. 사용자는 누가 실행했는지 검산할 수 없고, `자동 처리` 상태와 요청 주체가 같은 의미처럼 보인다. 취소 권한도 `producer_employee_id`를 우선 사용하므로 단순 표시를 넘어 데이터 책임 주체가 누락된 문제다.
- 확정 정책: 사용자가 실행한 품목 전환에는 실제 요청자를 저장하고 목록과 상세에서 `요청 {이름}`으로 표시한다. 승인 없이 완료되는 성격은 별도 `자동 처리` chip으로만 표현한다. 사람을 대신하는 `시스템 처리` 요청자 표시는 사용하지 않는다. 실제 요청자 근거가 없는 과거 기록에 임의 이름을 만들어 넣지 않는다.
- 정책 확인 필요: 기존 독립 품목 전환 이력처럼 요청자 근거가 저장되지 않은 기록은 `요청자 미기록`으로 명시할지, 별도의 신뢰 가능한 원천을 확보한 뒤 데이터 보정할지 구현 전에 결정한다. 서버 로그의 시간대 추정만으로 이름을 채우지는 않는다.
- 구현 방향: `IoComposeView`가 보유한 로그인 `operator.employee_id`를 `ItemConversionWorkView`와 실행 payload로 전달한다. 백엔드는 해당 직원을 검증한 뒤 품목 전환 로그에 `producer_employee_id`와 실제 이름 snapshot을 저장한다. `historyPresentation.ts`와 상세 메타도 같은 requester 필드를 사용하고, `자동 처리`는 status chip 계산에만 남긴다.
- 수용 기준: 새 품목 전환을 실행한 뒤 모든 상위·하위 row와 우측 상세 패널에 `요청 {실제 이름}`이 일관되게 표시된다. 같은 위치에 `자동 처리` chip이 함께 있어도 주체와 상태가 구분된다. `시스템 처리 · 구성품 변경`은 새 기록에서 생성되지 않는다.
- 브라우저 확인 시나리오: 로그인 사용자로 품목 전환 1건을 실행하고 `http://192.168.0.63:3001/mes?tab=history`에서 해당 묶음을 연다. 상위 row, 추가 차감/회수 입고 하위 row, 우측 상세 패널이 모두 같은 요청자 이름을 표시하는지 확인한다. 생산·분해·창고 이동의 자동 완료 row도 `요청 {이름}`과 `자동 처리`가 분리되는지 비교한다.
- 우선순위: 높음
- 구현 상태: 완료. 독립·출하 요청 품목 전환의 실제 실행자를 검증해 모든 로그에 이름과 직원 ID를 기록하고, 과거 누락 기록과 자동 완료 상태를 분리했다.

### 30. 스크롤/더보기 후 나타나는 모든 목록 row 높이 전수 통일
- 화면/영역: PC 입출고 내역 탭 > 목록 > 스크롤 후 하단 row, 더보기로 추가 로드된 row
- 증상: 화면 첫 구간에서는 row 높이가 맞아 보이더라도, 스크롤을 더 내리면 `창고 → 부서`, `출고`, `생산`, `새 불량`, `수량 조정` 같은 다른 유형 row 중 일부가 다시 다른 높이로 보인다. 즉, 현재 보이는 첫 묶음만 맞추면 전체 목록 리듬이 보장되지 않는다.
- 사용자 불편: 사용자는 목록을 훑다가 중간 이후 행 높이가 다시 튀는 순간 같은 정보 구조가 다른 카드처럼 보인다고 느낀다. 이미 “행 높이 통일”을 기대한 상태라, 스크롤 후 다시 깨지면 수정이 덜 끝난 것처럼 보인다.
- 확정 정책: 입출고 내역의 메인 목록 row는 로드 위치와 거래 유형에 상관없이 같은 높이를 가져야 한다. 긴 품명, 보조 meta, 요청/승인 2줄, 상태 chip, 수량·재고 2줄, `더보기`로 추가된 row도 같은 row height 슬롯 안에서 처리한다.
- 구현 방향: 단건 row, reference 묶음 row, operation batch row, 재작업 row, 스크롤 후 등장하는 모든 거래 유형이 같은 row height 상수와 같은 셀 padding/line-clamp 규칙을 쓰는지 전수 확인한다. 높이를 키우는 예외 원인인 wrap, 빈 상태 placeholder, 승인자 2줄, 긴 chip, 긴 target meta를 공통 helper에서 막는다. 필요하면 브라우저 측정 스크립트로 현재 로드된 `tbody > tr.cursor-pointer` 전체의 `getBoundingClientRect().height`를 수집해 예외 row를 찾는다.
- 수용 기준: 현재 로드된 모든 메인 row의 높이가 동일하다. `100건`, `200건`, `300건` 표시 상태와 `다음 100건 불러오기` 이후에도 높이가 다른 메인 row가 남지 않는다. 펼침 하위 row는 메인 row보다 낮은 별도 규칙을 쓰되, 같은 하위 row끼리는 높이가 통일된다.
- 브라우저 확인 시나리오: `http://192.168.0.63:3001/mes?tab=history`에서 PC 폭으로 확인한다. 첫 화면, 중간 스크롤 구간, 하단 구간, `다음 100건 불러오기` 이후를 각각 확인하고, 생산/분해/품목 전환/출하 준비/창고 이동/출고/새 불량/수량 조정 row의 높이가 모두 같은지 본다. 가능하면 브라우저 콘솔/자동화로 로드된 모든 `tr.cursor-pointer` 높이의 unique 값이 1개인지 확인한다.
- 우선순위: 높음
- 구현 상태: 완료. 선택 가능한 메인 row는 64px, 펼침 하위 row는 40px 공통 규칙으로 통일했다.

### 31. 우측 상세 패널 열림 시 표가 압축되지 않고 밀리는 레이아웃 회귀 정리
- 화면/영역: PC 입출고 내역 탭 > 목록 테이블 + 우측 상세 패널 2열 레이아웃
- 증상: row를 선택해 우측 상세 패널이 열리면, 원래는 목록 영역이 자연스럽게 줄어들고 compact 컬럼 기준으로 표가 안쪽에서 재배치되어야 한다. 현재는 상세 패널이 열리면서 표가 줄어드는 느낌보다 오른쪽 패널이 본문을 밀어내고, 목록의 오른쪽 컬럼 일부가 잘리거나 가로 위치가 어긋난 것처럼 보인다.
- 사용자 불편: 사용자는 상세 패널을 열어도 목록과 상세를 함께 비교하고 싶다. 그런데 표가 밀려 보이면 목록의 품목코드, 흐름, 수량·재고, 상태·처리 컬럼이 어디까지 보이는지 예측하기 어렵고, 상세 패널을 열 때마다 화면 구조가 흔들린다고 느낀다.
- 확정 정책: 우측 상세 패널은 목록을 화면 밖으로 밀어내는 방식이 아니라, 정해진 2열 컨테이너 안에서 목록 영역이 줄어들고 compact column spec이 적용되는 방식으로 열린다. 패널이 열린 상태에서도 목록 컨테이너의 오른쪽 경계, 테이블 헤더, row 컬럼 축이 상세 패널과 겹치거나 밀리지 않아야 한다.
- 구현 방향: `DesktopHistoryView`/입출고 내역 레이아웃에서 목록 영역과 우측 패널의 grid/flex width 계산을 확인한다. 상세 패널이 열린 상태에서는 목록 wrapper에 `min-width: 0`, 테이블 overflow container, compact column spec, panel width가 함께 적용되는지 점검한다. 최근 row 높이/셀 padding/수량 pill 폭 변경이 table min-width 또는 overflow 계산을 깨뜨렸는지도 같이 확인한다.
- 수용 기준: 우측 상세 패널을 열면 목록 영역은 같은 화면 안에서 안정적으로 줄어든다. 표가 상세 패널 아래로 밀리거나 화면 밖으로 과도하게 넘치지 않는다. 패널 열림/닫힘 전환 중에도 헤더와 row 컬럼 축이 일관되게 유지된다.
- 브라우저 확인 시나리오: `http://192.168.0.63:3001/mes?tab=history` PC 폭에서 품목 전환 row, 생산 row, 창고 이동 row를 각각 선택한다. 상세 패널이 열린 상태에서 목록 오른쪽 경계가 패널과 분리되어 보이는지, 품목코드/흐름/수량·재고/상태 컬럼이 compact 규칙대로 줄어드는지 확인한다. 패널을 닫았다 다시 열어도 표가 좌우로 밀리거나 잘리지 않는지 본다.
- 우선순위: 높음
- 구현 상태: 완료. `HistoryTable` 카드와 table scroller에 shrink 가능한 폭 규칙을 추가하고, 상세 패널 열린 compact 상태에서 `table-fixed`를 적용했다.

- 브라우저 확인 시나리오: `DX3000 데코가이드` 품목 전환과 `ADX6000s...` 생산 묶음을 차례로 선택한다. 품목 전환에서 목록 대표 row와 우측 카드가 같은 `기존품 → 변경품`을 표시하는지, 생산 묶음의 완제품·부품 정보가 하나의 재고 영향 목록에 결합되어 있는지 확인한다.
- 우선순위: 높음
- 구현 상태: 완료. 품목 전환 결론과 생산·분해의 실제 영향·BOM 정보를 하나의 키 포인트 재고 영향 목록으로 통합했다.

### 35. 변동 없는 재고 표기를 숨기고 실제 영향 위치를 정본으로 사용
- 화면/영역: PC 입출고 내역 탭 > 우측 상세 패널 > 처리 전·후, 수량·재고, 재고 영향
- 증상: 품목 전환 상세에서 실제 변화는 `조립 생산 +1`인데 `입고 1 EA`, `처리 전 309 → 처리 후 401`, `창고 401 EA`가 함께 보인다. presentation은 `quantity_before/after` 값 존재 여부와 transaction type만으로 `창고`를 붙이고, 해당 scope가 실제로 변했는지는 확인하지 않는다. 실제 변화 위치는 이미 `inventory_effect`에 기록되어 있다.
- 사용자 불편: 변하지 않은 창고 재고가 강조되고 서로 기준이 다른 전후 수량이 한 흐름처럼 보인다. 사용자는 실제로 1개가 늘었는지 92개가 늘었는지, 창고가 변했는지 조립 재고가 변했는지 다시 해석해야 한다.
- 확정 정책: 우측 요약의 재고 변화는 0이 아닌 `inventory_effect` 행을 정본으로 사용한다. 변하지 않은 scope의 재고, 비교 기준이 다른 전후 수량, 거래와 직접 관계없는 총재고는 핵심 요약에서 숨긴다. 현재 재고가 필요하면 `처리 후 참고값`으로 한 번만 분리한다.
- 구현 방향: `historyInventoryEffect.ts`의 정규화 결과를 상세 summary에 연결한다. `getStockPresentation`은 scope 변화 여부를 검증하거나 상세 핵심 요약에서는 사용하지 않는다. 단일 영향이면 summary에 흡수하고, 여러 영향일 때만 접을 수 있는 세부 목록을 보여준다.
- 수용 기준: 조립 위치만 +1 된 품목 전환 상세에는 `조립 생산 +1 EA`가 한 번 표시되고 `창고 401 EA`는 표시되지 않는다. 창고와 부서가 실제로 함께 변한 이동 거래는 두 영향이 각각 정확히 표시된다.
- 브라우저 확인 시나리오: 품목 전환 하위 입고 row와 창고→부서 이동 row를 비교한다. 실제 변한 위치만 노출되는지, 영향 합계가 거래 변화량과 맞는지 확인한다.
- 우선순위: 높음
- 구현 상태: 완료. 0이 아닌 `inventory_effect`의 품목·단위·위치·box 기준만 상세 재고 영향 정본으로 사용한다.

### 36. 단건·묶음 취소 액션의 위치와 문법을 통일
- 화면/영역: PC 입출고 내역 탭 > 우측 상세 패널 > 취소 액션 및 확인 폼
- 증상: 단건 취소는 사유·메모·재고 영향 뒤의 하단 meta strip에 `이 이력 1건 취소`로 표시되지만, 묶음 취소는 Hero 안에 `배치 취소`로 표시된다. 확인 CTA도 `범위 확인 후 취소`와 `취소 확정`으로 다르다.
- 사용자 불편: 거래 유형을 바꿀 때마다 위험 액션의 위치와 범위를 다시 찾아야 한다. 같은 위치에 있을 것으로 기대한 취소가 생산 묶음에서는 위쪽에 있어 일관성이 깨진다.
- 확정 정책: 모든 취소 액션은 핵심·근거 정보 뒤의 패널 하단 액션 영역에 둔다. 단건은 `이 이력 1건 취소`, 묶음은 `이 작업 묶음 전체 취소`로 범위를 명시하며, 확인 폼의 제목·설명·CTA 순서를 공통으로 사용한다.
- 구현 방향: 단건과 묶음이 공유하는 취소 action/confirm 컴포넌트를 만들거나 동일한 구조를 적용한다. 묶음 Hero 안의 취소 버튼은 제거하고, 확인 단계에는 되돌릴 실제 재고 영향 요약을 먼저 보여준다.
- 수용 기준: 어떤 거래를 선택해도 취소 버튼이 같은 하단 위치에 있고, 누르기 전에 단건인지 묶음 전체인지 알 수 있다. 확인 CTA 문법도 동일하다.
- 브라우저 확인 시나리오: 품목 전환 하위 row와 생산 묶음을 각각 선택해 취소 버튼 위치와 문구를 비교한다. 확인 폼을 열어 범위·영향·사유·PIN·CTA 순서가 같은지 본다.
- 우선순위: 높음
- 구현 상태: 완료. 공통 취소 컴포넌트와 패널 하단 액션 영역으로 위치·범위·확인 문법을 통일했다.

### 37. 조회 실패를 정상 빈 목록과 분리하고 재시도 제공
- 화면/영역: PC 입출고 내역 탭 > 첫 조회, KPI, 목록 빈 상태
- 증상: `useHistoryData`의 첫 페이지 요청 실패가 오류를 버리고 `loading=false`, `logs=[]`로 끝나며 `HistoryTable`은 이를 `거래 이력이 없습니다`로 렌더링한다. summary 요청만 실패하면 이전 summary와 새 빈 목록이 함께 남을 가능성도 있다.
- 사용자 불편: 감사·추적 성격의 화면에서 서버 장애를 실제 이력 부재로 오판할 수 있다. 사용자는 데이터가 없는지 연결이 실패했는지 구분할 수 없다.
- 확정 정책: `로딩`, `정상 빈 결과`, `조회 실패`를 별도 상태로 관리한다. 조회 실패 시 오류 요약과 재시도 명령을 보여주며, 실패한 조건의 이전 KPI를 현재 값처럼 남기지 않는다.
- 구현 방향: `useHistoryData`가 error 상태와 retry 함수를 노출하고 `DesktopHistoryView`/`HistoryTable`이 전용 오류 상태를 렌더링한다. 첫 조회 실패 시 summary도 같은 조건 기준으로 초기화하거나 오류 표시한다. 실패·재시도 테스트를 추가한다.
- 수용 기준: API 실패 시 `거래 이력이 없습니다`가 나타나지 않고, 재시도 후 성공하면 목록과 KPI가 같은 조건의 값으로 복구된다.
- 브라우저 확인 시나리오: 거래 API를 실패하도록 한 테스트 환경에서 탭을 연다. 오류 상태와 재시도 버튼을 확인하고, API 복구 후 재시도로 정상 목록이 나오는지 본다.
- 우선순위: 높음
- 구현 상태: 완료. PC·모바일 최초 조회와 더보기 오류를 빈 결과와 분리하고 기존 행 보존 및 재시도를 제공한다.

### 38. 배치 취소 성공 직후 목록과 우측 상세 상태를 함께 갱신
- 화면/영역: PC 입출고 내역 탭 > 생산/작업 묶음 우측 상세 > 취소 완료 직후
- 증상: 배치 패널은 전달받은 `selection.logs`로 취소 여부를 계산하지만, 취소 성공 콜백은 목록의 logs만 갱신하고 현재 selection과 displaySelection은 갱신하지 않는다. 패널을 닫기 전까지 취소 전 상태와 취소 버튼이 남을 수 있다.
- 사용자 불편: 취소가 성공했는데도 같은 패널에서 다시 취소할 수 있는 것처럼 보여 결과를 신뢰하기 어렵다. 중복 취소 요청을 시도할 위험도 있다.
- 확정 정책: 취소 성공은 목록 row, 현재 selection, 우측 상세, 상태 chip을 하나의 상태 변경으로 반영한다. 성공 직후 취소 버튼은 제거하고 `취소된 거래`와 사유를 표시한다.
- 구현 방향: `onBatchCancelled`가 갱신된 로그 묶음을 selection/displaySelection에도 반영하거나 취소 후 해당 배치를 재조회한다. 제출 중·성공 후 중복 호출을 막고 상태 동기화 테스트를 추가한다.
- 수용 기준: 묶음 취소 성공 직후 패널을 닫지 않아도 목록과 상세가 모두 취소 상태로 바뀌며, 취소 버튼이 다시 노출되지 않는다.
- 브라우저 확인 시나리오: 취소 가능한 테스트 묶음을 취소하고 토스트·목록 row·우측 패널을 순서대로 확인한다. 새로고침 없이 세 위치의 상태가 같은지 본다.
- 우선순위: 높음
- 구현 상태: 완료. 취소 결과를 목록·선택·묶음 캐시·React Query 캐시에 반영하고 최신 전체 범위 상태를 상세 패널에 즉시 동기화한다.

### 39. 패널 닫기·대상 변경 시 취소 사유와 PIN 즉시 폐기
- 화면/영역: PC 입출고 내역 탭 > 우측 상세 패널 > 취소 확인 입력 상태
- 증상: 패널은 닫힘 애니메이션 동안 마지막 `displaySelection`을 유지하고 상세 컴포넌트도 마운트된 상태로 남는다. 단건 상세의 취소 사유와 PIN은 로그 ID 변경이나 패널 닫힘을 기준으로 초기화되지 않아 같은 거래를 다시 열면 입력값이 남을 수 있다.
- 사용자 불편: 민감한 PIN이 보이지 않는 패널 상태에 남고, 이전 거래의 취소 사유가 다음 확인에 재사용될 수 있다. 사용자는 닫기 동작이 입력을 폐기했다고 기대한다.
- 확정 정책: 취소 사유와 PIN은 패널 닫기, 선택 대상 변경, 취소 성공 시 즉시 폐기한다. 닫힘 애니메이션을 위해 콘텐츠를 보존하더라도 민감 입력 state는 보존하지 않는다.
- 구현 방향: 단건·묶음 취소 state에 `open` 또는 selection identity 변화를 전달해 reset effect를 적용한다. 가능하면 공통 취소 컴포넌트가 생명주기와 초기화를 소유한다. PIN 값이 로그·오류 메시지·DOM 속성에 남지 않는지 테스트한다.
- 수용 기준: 취소 폼에 사유와 PIN을 입력한 뒤 패널을 닫거나 다른 row를 선택하면 입력값이 즉시 사라진다. 같은 거래를 다시 열어도 빈 폼이다.
- 브라우저 확인 시나리오: 취소 폼에 테스트 값을 입력하고 X, Escape, 다른 row 선택을 각각 수행한다. 다시 폼을 열어 모든 경우 입력이 초기화됐는지 확인한다.
- 우선순위: 높음
- 구현 상태: 완료. 패널 닫기, 대상 변경, 취소 성공 시 취소 사유와 PIN을 즉시 초기화한다.

### 40. 검색·필터·기간 변경 후 범위 밖 상세 선택 정리
- 화면/영역: PC 입출고 내역 탭 > 검색, 거래 필터, 날짜 범위, 우측 상세 패널
- 증상: 목록 조회 조건과 `selection`이 독립되어 있어 조건을 변경한 뒤 선택한 거래가 새 결과에 없어도 이전 우측 상세가 계속 열린다. 목록은 새 조건 또는 빈 결과인데 table은 selection 때문에 compact 상태를 유지한다.
- 사용자 불편: 왼쪽 목록과 오른쪽 상세가 서로 다른 조건의 데이터를 보여준다. 사용자는 현재 필터 결과에 포함된 거래라고 오해할 수 있다.
- 확정 정책: 새 조회 결과에도 선택 거래가 존재하면 상세를 유지하고, 결과 범위에서 사라지면 선택과 상세를 닫는다. 필터 패널을 여는 순간이 아니라 새 조건의 조회가 확정되는 시점에 판정한다.
- 구현 방향: 조회 성공 후 현재 selection의 log ID 또는 batch ID가 새 그룹에 포함되는지 검사한다. 포함되지 않으면 selection·displaySelection·관련 drill history를 정리하고 table compact 상태를 해제한다. 검색 debounce와 빠른 조건 변경을 포함한 테스트를 추가한다.
- 수용 기준: 필터로 선택 거래를 제외하면 우측 패널이 닫히고 목록 폭이 원래대로 돌아온다. 선택 거래가 새 결과에도 있으면 패널과 highlight가 유지된다.
- 브라우저 확인 시나리오: row를 선택한 뒤 검색어, 오늘/이번주, 거래 종류 필터로 해당 row를 포함·제외해 본다. 목록과 상세가 항상 같은 조건을 가리키는지 확인한다.
- 우선순위: 중간
- 구현 상태: 완료. 성공한 새 조회 결과와 선택을 재조정해 대상이 사라지면 패널과 펼침 상태를 닫는다.

### 41. 목록 비교용 우측 패널을 비모달 보조 영역으로 동작하게 정리
- 화면/영역: PC 입출고 내역 탭 > 목록 + 우측 상세 패널 > 키보드·스크린리더
- 증상: 패널은 화면 안에서 목록과 나란히 비교하도록 설계됐지만 공통 `SlidePanel`이 `aria-modal=true`와 focus trap을 적용한다. 키보드 사용자는 패널을 닫기 전 다른 목록 row로 이동할 수 없고, `aria-labelledby="slide-panel-title"`이 실제 제목 ID와 연결되지 않는다.
- 사용자 불편: 마우스로는 목록 row를 바꾸며 상세를 비교할 수 있지만 키보드로는 같은 흐름이 막힌다. 스크린리더에서는 패널 제목도 정확히 전달되지 않을 수 있다.
- 확정 정책: PC 입출고 내역의 나란히 비교 패널은 비모달 complementary 영역으로 동작한다. 목록과 패널 사이 포커스 이동을 허용하고, 제목·닫기 버튼·선택 상태의 접근성 이름을 연결한다. 다른 화면의 진짜 모달 SlidePanel 동작은 바꾸지 않는다.

- 우선순위: 중간
- 구현 상태: 완료. 입출고 내역 패널만 비모달 `complementary` 영역으로 사용하고 다른 SlidePanel의 모달 기본 동작은 유지했다.

### 42. 묶음 펼침 제어의 마우스·키보드 동작 일치
- 화면/영역: PC 입출고 내역 탭 > 묶음 row chevron, BOM 내부 묶음 row
- 증상: chevron은 click에서만 이벤트 전파를 막아 Enter/Space 키 입력이 상위 `<tr>`의 선택 handler까지 전달될 수 있다. 일부 BOM 내부 묶음 row는 click만 있고 `tabIndex`와 키보드 handler가 없어 키보드로 펼칠 수 없다.
- 사용자 불편: 마우스로는 펼침만 되지만 키보드로는 상세 패널까지 열리거나, 아예 펼칠 수 없는 row가 생긴다. 같은 아이콘의 동작을 예측하기 어렵다.
- 확정 정책: chevron의 Enter/Space는 묶음 펼침·접기만 수행하고 상세 선택을 발생시키지 않는다. row 본문 Enter/Space는 상세 선택, chevron은 펼침이라는 역할을 분리한다.
- 구현 방향: chevron keydown 전파를 차단하고 실제 button 기본 동작을 사용한다. click-only BOM row는 명시적 button 또는 동일한 키보드 handler와 focus style을 제공한다. 중첩 interactive 구조가 접근성 규칙을 위반하지 않는지도 함께 정리한다.
- 수용 기준: 모든 묶음에서 마우스 클릭과 키보드 Enter/Space가 같은 결과를 내며, 펼침 제어로 우측 상세가 의도치 않게 열리지 않는다.
- 브라우저 확인 시나리오: reference 묶음, operation batch, BOM 내부 묶음의 chevron을 Tab으로 이동해 Enter와 Space로 각각 조작한다. 펼침 상태와 selection 상태를 확인한다.
- 우선순위: 중간
- 구현 상태: 완료. 펼침 제어를 실제 버튼과 ARIA 상태로 통일하고 Enter·Space의 행 선택 전파를 차단했다.

### 43. 잘린 긴 품목명의 전체 내용 확인 수단 제공
- 화면/영역: PC 입출고 내역 탭 > 목록 대상 셀, 우측 패널 제목, 구성 라인
- 증상: 목록 대상명은 한 줄 `truncate`, 패널 제목은 두 줄 `line-clamp`를 사용하지만 tooltip이나 전체명 확인 수단이 없다. `ADX6000s_80kV, 5mA_USA_Dexcowin Global...` 같은 이름은 목록과 상세 모두에서 끝까지 읽을 수 없다. 대상 셀을 한 줄 말줄임으로 고정하면 좁은 폭에서 표의 최소 폭이 커져 가로 스크롤이 생긴다.
- 사용자 불편: 비슷한 모델명이나 국가·포장 사양이 이름 뒤쪽에 있으면 어떤 품목인지 확정할 수 없다. 상세 패널을 열어도 전체명이 나오지 않아 다른 화면을 찾아야 한다. 긴 품명을 읽으려고 목록 자체를 좌우로 스크롤하는 것은 입출고 내역의 빠른 스캔 목적에 맞지 않는다.
- 확정 정책: 목록 대상명은 최대 두 줄까지 자연스럽게 줄바꿈하고, 두 줄을 넘어가는 경우에만 말줄임과 전체 이름 tooltip·접근성 이름을 제공한다. 대표 row는 64px 안에서 두 줄을 수용하고, 하위 row는 필요 시 대상명 줄바꿈 대신 상위 summary나 tooltip으로 보조한다. 긴 품명 때문에 PC 목록에 가로 스크롤을 만들지 않는다.
- 구현 방향: `TargetSummaryBlock`, `HistoryLogRow`, reference/BOM 하위 row의 대상 셀에 `min-w-0`, 두 줄 clamp, 단어 단위 줄바꿈 규칙을 적용한다. `HistoryTable`의 min-width와 colgroup을 48번의 가로 폭 정책에 맞춰 조정하고, overflow 감지 tooltip은 두 줄을 넘어 잘린 경우에만 보인다. 우측 패널 제목·구성 라인의 tooltip 지원은 유지한다.
- 수용 기준: 긴 품목명은 목록에서 두 줄까지 읽히고, 표의 가로 스크롤 없이 흐름·수량·상태 열을 함께 볼 수 있다. 두 줄을 넘어가는 이름은 hover/focus로 전체 문자열을 확인할 수 있으며, 짧은 이름에는 불필요한 tooltip이 나타나지 않는다.
- 브라우저 확인 시나리오: 가장 긴 생산 품목명과 품목 전환 대상명을 패널 열림·닫힘 상태에서 확인한다. 대상명 두 줄 표시, 가로 스크롤 미발생, 우측 세 열 가독성, tooltip 전체명 제공을 함께 확인한다.
- 우선순위: 중간
- 구현 상태: 완료. 대표 대상명은 두 줄까지 표시하고 넘치는 경우에만 전체명 tooltip을 제공하며, 긴 이름 때문에 가로 스크롤이 생기지 않도록 정리했다.

### 44. 전체 건수의 정확한 배수에서 불필요한 더보기 요청 제거
- 화면/영역: PC 입출고 내역 탭 > `다음 100건 불러오기` 버튼
- 증상: `canLoadMore`가 조건 전체 건수가 아니라 마지막 응답 길이가 100건인지로만 결정된다. 전체가 정확히 100건 또는 200건이면 빈 페이지를 한 번 더 요청해야 버튼이 사라진다.
- 사용자 불편: 더 가져올 데이터가 없는데도 버튼이 보여 불필요한 클릭과 네트워크 요청이 발생한다. 클릭 후 아무 변화가 없어 동작 실패처럼 느껴진다.
- 확정 정책: 서버 summary의 전체 건수와 현재 로드 건수를 비교해 `현재 로드 < 조건 전체`일 때만 더보기를 노출한다. summary를 사용할 수 없는 예외에서만 페이지 길이를 fallback으로 사용한다.
- 구현 방향: `useHistoryData`의 `canLoadMore` 계산에 total count를 연결하고, 조건 변경·추가 로드·summary 실패 상태를 테스트한다.
- 수용 기준: 전체 100건·200건처럼 페이지 크기의 정확한 배수인 조건에서도 마지막 데이터가 로드되면 더보기 버튼이 즉시 사라진다.
- 브라우저 확인 시나리오: 전체 건수가 100, 101, 200인 테스트 조건에서 더보기 버튼 노출과 추가 로드 후 상태를 비교한다.
- 우선순위: 낮음
- 구현 상태: 완료. PC·모바일 모두 summary 전체 건수를 우선 사용해 100·200건에서는 종료하고 101·201건에서는 마지막 건까지 접근하게 했다.

### 46. `여러 위치`를 흐름명이 아닌 실제 작업 결과로 표시

- 사용자 불편: 생산·분해·이동 중 무엇이 일어났는지보다 위치 분산 여부가 먼저 강조되어, 유사한 묶음의 흐름이 서로 다르게 보인다.
- 확정 정책: 흐름 열의 첫 문구는 항상 업무상 결과를 표현한다. 여러 위치 여부는 필요한 경우에만 보조 정보 또는 상세에서 제공한다. compact 상태에서도 결과 문구는 축약된 한 줄로 끝까지 읽히게 하며, 말줄임으로 핵심 방향을 숨기지 않는다.
- 구현 방향: 생산·분해 묶음 presentation에서 `여러 위치` fallback을 제거하고 `완제품 입고·부품 차감`, `완제품 차감·부품 입고`처럼 방향을 즉시 읽을 수 있는 짧은 결과 문구를 대표로 만든다. 위치별 분산은 펼침/상세의 검산 정보로 이동하고, compact 셀에는 결과 문구가 들어갈 고정 최소 폭을 둔다.
- 수용 기준: 생산·분해 묶음을 훑을 때 첫 문구만으로 완제품과 구성품의 방향을 파악할 수 있고, 위치 개수는 그 흐름을 방해하지 않는다. 패널이 열린 상태에서도 `완제품 입고·부품 차감` 같은 결과 문구가 잘리지 않는다.
- 브라우저 확인 시나리오: 조립·여러 위치 생산·분해 묶음을 나란히 비교하고 우측 패널을 연다. 흐름 열의 첫 줄이 같은 문법으로 읽히며 결과 문구가 잘리지 않는지 확인한다.
- 우선순위: 높음
- 구현 상태: 완료. 여러 위치라는 내부 합산 조건을 대표 흐름에서 제거하고 실제 입고·차감 결과를 표시한다.

### 47. 묶음 수량과 기준 재고의 의미를 분리해 과밀도를 낮춤
- 화면/영역: PC 입출고 내역 탭 > 생산·분해 묶음의 `수량 · 재고` 열
- 증상: `완제품 +5 EA`, `부품 -100 EA`, `출하 400 EA`처럼 서로 다른 의미의 값이 작은 pill과 보조 텍스트에 함께 표시된다. 마지막 재고가 어느 품목·위치의 처리 후 수량인지 한눈에 연결되지 않는다. 패널이 열린 상태에서는 `완제품 +3 EA` 같은 pill의 텍스트가 폭보다 길어져 밖으로 튀어나온다. 수량이 세 자리인 `완제품 +100 EA`가 나타날 때도 pill 폭이 매 row마다 달라져 우측 열의 리듬이 흔들린다. 또한 품목 전환의 `자동 차감`은 실제로 해당 품목의 부서 생산 재고에서 차감되지만, 목록은 `BACKFLUSH`의 기준 재고를 일괄 `창고`로 표기한다.
- 사용자 불편: 사용자는 실제 변동량과 기준 재고를 구분하려고 세 문구를 다시 해석해야 하며, 수량이 큰 묶음일수록 우측 열이 가장 복잡해진다. 실제로 창고 물건이 차감된 것처럼 오해할 수 있다.
- 확정 정책: 목록은 실제 재고 영향만 우선 표시하고, 기준 재고는 대표 품목·대표 위치에 대한 값일 때만 보조로 표시한다. 서로 다른 품목의 재고를 하나의 값처럼 보이게 하지 않는다. 부서 생산 위치는 `조립 재고`, `진공 재고`처럼 재고임을 명시하고, 자동 차감은 실제 위치와 다른 `창고`로 표시하지 않는다. 수량 pill은 세 자리 수량까지 같은 기본 폭으로 고정하고, 네 자리 이상은 `1.2k` 같은 축약과 전체 수량 tooltip을 사용한다. 텍스트가 외부로 넘치거나 수량 자릿수마다 pill 폭이 바뀌면 안 된다.
- 구현 방향: 묶음 수량 presentation을 `결과 수량`과 `검산용 기준 재고`로 분리한다. 대표 재고가 의미 없거나 여러 품목에 걸치면 목록에서는 숨기고 우측 상세의 영향 목록으로 보낸다. `BACKFLUSH`의 실제 `inventory_effect` 위치를 목록 scope label에 연결하고, 부서+`PRODUCTION` 표기를 `부서 재고` 문법으로 정리한다. 수량 pill은 `+999 EA`까지 수용하는 고정 폭을 기준으로 잡고, 그 이상은 축약 표기와 보조 tooltip으로 처리한다.
- 수용 기준: 한 row에서 무엇이 늘고 줄었는지는 첫 줄만 보고 알 수 있으며, 기준 재고를 표시하는 경우 어떤 품목·위치의 값인지 명확하다. 품목 전환 추가 차감은 창고가 아닌 실제 부서 재고로 표시되고, `+3 EA`·`+100 EA` pill은 같은 폭으로 정렬되며 모든 수량 pill의 텍스트가 경계 밖으로 넘치지 않는다.
- 브라우저 확인 시나리오: 완제품·부품이 함께 변하는 생산, 분해, 여러 위치 작업과 품목 전환의 추가 차감 row를 비교한다. 우측 패널을 연 뒤 pill 넘침이 없고, 한 자리·세 자리·네 자리 수량의 pill 폭과 tooltip, `조립 재고` 등 실제 재고 위치가 표시되는지 확인한다.
- 우선순위: 높음
- 구현 상태: 완료. 실제 증감을 우선 표시하고 `BACKFLUSH`는 실제 부서 재고로 표기하며, 수량 pill의 세 자리·축약 규칙을 통일했다.

### 48. 우측 패널 상태별 대상·품목코드 열 배분을 다시 조율
- 화면/영역: PC 입출고 내역 탭 > 목록의 대상·품목코드 열, 우측 상세 패널 열림/닫힘 상태
- 증상: 패널이 닫힌 넓은 화면에서는 대상명과 품목코드 사이 공백이 크게 늘고, 상태·처리 열 뒤에 상세 패널을 닫고도 남은 듯한 큰 여백이 보인다. 패널을 열면 대상 열이 급격히 좁아진다. 같은 표가 패널 상태에 따라 읽는 리듬이 크게 달라진다. 현재 압축 과정에서 흐름·수량·재고·상태·처리 열까지 함께 줄어들어 수량 pill이 넘치고 우측 정보가 혼잡해진다.
- 사용자 불편: 품목코드까지 눈을 옮기는 거리가 불필요하게 길거나, 패널을 열었을 때 긴 대상명을 비교하기 어려워진다.
- 확정 정책: 패널이 닫힌 상태에서는 표가 메인 pane의 가용 폭을 모두 사용하고, 상태·처리 열 뒤에 의미 없는 빈 영역을 남기지 않는다. 대상·품목코드는 자연스러운 스캔 간격을 유지하고, 열린 상태에서는 대상·품목코드 사이의 여유 폭만 우선 압축한다. 대상명은 43번의 두 줄 규칙 안에서 폭을 흡수하며, 긴 품명 때문에 어느 상태에서도 가로 스크롤을 만들지 않는다. 흐름·수량·재고·상태·처리 열의 최소 폭과 상호 간격은 가능한 유지한다.
- 구현 방향: closed/compact 각각의 `colgroup` 비율과 대상 셀 내부 최대 폭을 별도 점검한다. 현재 패널 열림 회귀 수정의 `min-w-0` 계약은 유지하고, 닫힌 상태에서는 table/card가 main pane 폭을 채우도록 잔여 폭 배분을 정리한다. 열린 상태에서는 대상 열의 가변 폭과 코드 열의 압축 폭을 우선 조정한다. `overflow-x-auto`는 긴 품명 보완 수단으로 사용하지 않고, 흐름·수량·상태 열은 읽기 가능한 고정 최소 폭을 두며 43번의 대상 줄바꿈·46·47번의 짧은 문구와 pill 축약 규칙을 그 폭 안에서 적용한다.
- 수용 기준: 1920px과 1440px에서 패널 열기 전후에도 대상명, 품목코드, 우측 세 열을 한 번의 시선 이동으로 비교할 수 있다. 긴 품명으로 인한 가로 스크롤이 없고, 닫힌 상태의 표는 상태·처리 뒤에 상세 패널 자리처럼 보이는 빈 영역을 남기지 않는다. 패널을 열 때 주로 대상·품목코드 구간만 줄고, 수량·재고와 상태·처리의 간격·가독성은 유지된다. 표와 패널은 겹치지 않는다.
- 브라우저 확인 시나리오: 긴 생산 품목과 품목 전환 묶음에서 패널을 열고 닫아 표의 우측 여백, 대상·품목코드의 간격, 두 줄 대상명, 가로 스크롤 부재, 우측 세 열의 폭, 수량 pill 넘침과 흐름 문구 말줄임을 함께 비교한다.
- 우선순위: 중간
- 구현 상태: 완료. 닫힌 표는 메인 pane 폭을 사용하고, compact에서는 대상·품목코드만 우선 압축해 우측 열의 판단 폭을 보존한다.

### 49. 같은 작업의 연속 대표 row가 분리된 이유를 목록에서 설명
- 화면/영역: PC 입출고 내역 탭 > 같은 대상·근접 시각의 생산·분해 대표 row
- 증상: 같은 대상이 16:18, 16:14, 16:01처럼 연속 노출될 때 모두 독립된 `생산` 대표 row처럼 보인다. 실제로 서로 다른 작업인지, 하나의 작업이 위치·취소·재처리로 나뉜 것인지 목록만으로 구분하기 어렵다.
- 사용자 불편: 사용자는 중복 생산이나 반복 처리로 오해할 수 있고, 원하는 이력을 찾으려면 각 row를 모두 열어 봐야 한다.
- 확정 정책: 서로 다른 operation/reference는 독립 row로 유지하되, 동일 작업 계열이 연속될 때는 차이를 설명하는 보조 문구를 제공한다. 동일한 operation/reference라면 하나의 묶음으로 수렴한다.
- 구현 방향: grouping key와 대표 row metadata를 점검하고, 분리 유지되는 경우에는 `취소된 처리`, `다른 요청`, `다른 완료 시각`처럼 분리 근거를 상태·처리 열 또는 펼침 요약에 노출한다.
- 수용 기준: 근접 시각의 같은 대상 row를 보고 중복인지 별도 작업인지 판별할 수 있으며, 단순한 위치 분산만으로 대표 row가 늘어나지 않는다.
- 브라우저 확인 시나리오: 07/09의 동일 생산 품목 연속 row와 생산 취소 row를 비교해 분리 근거와 묶음 관계가 목록에서 드러나는지 확인한다.
- 우선순위: 중간
- 구현 상태: 완료. 연속 대표 row는 취소·다른 요청·다른 완료 시각 등 분리 근거를 표시하고, 같은 operation/reference만 묶는다.

### 50. 취소된 이력 row의 시각적 위계를 낮추되 조회·선택은 유지
- 화면/영역: PC 입출고 내역 탭 > 목록의 취소된 단건·묶음·하위 row
- 증상: 취소된 row는 상태·처리 열의 작은 `취소` chip만 다르고, 나머지 대상·흐름·수량·재고가 완료 row와 같은 대비로 보인다. 목록을 스캔할 때 이미 무효가 된 거래라는 사실의 시각적 영향이 약하다.
- 사용자 불편: 같은 품목의 완료·취소 이력이 섞이면 취소된 수량을 실제 유효 재고 변화처럼 읽기 쉽다. 반면 row 자체를 비활성화하면 취소 사유·실제 영향·수정 이력을 확인할 수 없어 감사 흐름도 끊긴다.
- 확정 정책: 취소된 row 전체는 완료 row보다 낮은 대비의 텍스트·배경·pill 톤을 사용해 한눈에 구분한다. 단, 실제 비활성화는 하지 않으며 클릭·키보드 선택·펼침·우측 상세 조회는 기존처럼 가능해야 한다.
- 구현 방향: `cancelled` 상태를 대표/하위/BOM row의 공통 row presentation에 연결한다. 취소 chip만 바꾸지 말고 대상·코드·흐름·수량의 foreground와 background tint를 일관되게 낮춘다. 포커스·선택 highlight는 취소 상태에서도 충분한 대비를 유지하고, `disabled` 또는 `aria-disabled` 속성은 사용하지 않는다.
- 수용 기준: 취소된 row는 목록에서 즉시 낮은 대비로 인식되지만, 같은 row를 클릭·Enter·Space로 선택해 우측 상세와 취소 사유를 정상적으로 확인할 수 있다.
- 브라우저 확인 시나리오: 완료·취소된 생산 묶음과 품목 전환 하위 row를 나란히 비교한다. 취소 row의 시각적 위계, 선택 highlight, 펼침과 상세 패널 진입을 각각 확인한다.
- 우선순위: 중간
- 구현 상태: 완료. 취소 row는 낮은 대비로 표시하되 클릭·키보드 선택·상세 조회는 그대로 유지한다.

### 51. 불량 즉시 처리와 격리 후 처리 이력의 생성 경로를 분리
- 화면/영역: 불량 처리 탭 > 정상 재고 바로 처리, 새 불량 등록 후 처리 / 입출고 내역 탭 > 새 불량·불량 처리 row
- 증상: 같은 작업자·같은 품목에서 `새 불량`과 `불량 처리`가 근접 시각에 각각 독립 row로 남는다. `새 불량`은 격리 기록(`MARK_DEFECTIVE`)이고 `불량 처리`는 격리 재고를 폐기·반품·재작업하는 별도 거래지만, 바로 처리한 작업처럼 보일 때 두 행이 중복으로 읽힌다.
- 사용자 불편: 사용자는 한 번의 불량 처리에서 거래가 두 번 실행됐는지, 등록만 한 뒤 나중에 별도로 처리한 것인지 목록만 보고 구분하기 어렵다. 즉시 처리 의도인데 중간 `새 불량` 행까지 남으면 목록이 불필요하게 길고 감사 해석도 흔들린다.
- 확정 정책: 정상 재고를 바로 폐기·반품·재작업하는 즉시 처리에서는 격리 기록을 만들지 않고 최종 처리 이력 한 건만 남긴다. 먼저 새 불량으로 격리하고 이후 별도 시점에 처리한 경우에는 두 감사 이력을 유지하되, 동일 작업 묶음으로 연결할 근거가 있을 때 목록에서 관계를 드러낸다. 근거 없는 과거 row를 시간·이름만으로 임의 묶지 않는다.
- 구현 방향: 불량 탭의 정상 재고 바로 처리 동선이 `SCRAP_NORMAL`·`RETURN_NORMAL`·`REWORK_NORMAL` 같은 직접 처리 요청으로만 실행되는지 점검하고, 새 불량 등록 뒤 후속 처리를 자동으로 연쇄 호출하는 경로가 있으면 제거하거나 명확히 분리한다. 연결 가능한 격리→처리 이력에는 공통 operation/reference를 저장·표현하고, 입출고 목록은 하나의 대표 row와 하위 단계로 보여 준다.
- 수용 기준: 정상 재고 즉시 처리 1건은 입출고 내역에 최종 처리 row 하나만 남는다. 격리 후 별도 처리한 이력은 두 사건이 사라지지 않되, 연결된 경우 등록→처리 관계를 목록에서 이해할 수 있다.
- 브라우저 확인 시나리오: 불량 탭에서 정상 재고 바로 폐기·반품·재작업을 각각 1건씩 수행하고, 새 불량 등록 후 나중에 처리하는 흐름도 1건 수행한다. 입출고 내역에서 생성 row 수, 거래명, 묶음 관계를 비교한다.
- 우선순위: 높음
- 구현 상태: 완료. 정상 재고 직접 처리의 최종 로그 한 건과 격리 후 감사 로그 두 건을 분리하고, 같은 품목·작업자·부서·수량·사유가 60초 이내인 과거 이력만 표시상 묶는다.

- 확정 정책: 대상 열은 두 줄 품명 규칙을 만족하는 범위까지만 사용하고 남는 폭을 독점하지 않는다. 작업 열은 현재보다 한 단계 넓혀 52번의 고정 작업 배지를 여유 있게 수용한다. 품목코드·흐름·수량·재고 열은 대상 축소분만큼 좌측으로 이동하되, 기존 읽기 폭은 줄이지 않는다. 상태·처리 열은 세 개의 짧은 상태 chip을 한 줄에 모두 표시할 수 있는 고정 최소 폭을 확보하며, 패널 열림 상태에서도 이 폭을 다시 줄이지 않는다.

- 사용자 불편: 사용자는 동일 수량이 중복 차감됐는지 다시 해석해야 하며, 사람이 읽을 수 없는 UUID가 우측 카드의 핵심 정보를 밀어낸다. 보관 위치를 확인하려는 목적도 아닌데 창고 지도 내부 식별자가 기본 화면에 노출된다.
- 확인 근거: 백엔드의 재고 영향은 창고 총재고와 박스별 수량을 함께 snapshot/diff로 기록한다(`backend/app/services/inv_effect.py:32-77`). 프런트는 `warehouse_box`를 `박스 <box_id>`로 그대로 라벨링한다(`historyInventoryEffect.ts:45-59`), 이후 실제 영향 요약이 창고/박스 scope를 구분 없이 모두 렌더링한다(`historyDetailSummary.ts:77-84`, `HistoryKeyPointSummary.tsx:62-108`). 따라서 같은 물리적 감소가 총량과 위치 추적 두 관점으로 중복 노출된다.
- 확정 정책: 우측 상세의 기본 `실제 영향`은 업무 결과 기준으로 한 번만 보여 준다. 같은 품목·같은 증감의 창고 총재고와 박스 재고가 함께 있으면 창고 총재고만 표시하고, 박스 UUID는 기본 카드에서 숨긴다. 향후 위치 추적이 필요하면 별도 보관 위치 상세에서 사람이 읽을 수 있는 박스명/지도 위치로 제공한다.
- 구현 방향: 영향 요약 모델에서 `warehouse`와 동일 item·unit·delta를 가진 `warehouse_box`를 위치 추적용 보조 effect로 분류하고 기본 impactGroups에서 제외한다. 박스 effect만 존재하거나 창고 총재고와 delta가 달라 재고 검산에 필요한 경우에는 숨기지 않고, UUID 대신 표시 가능한 박스명/위치를 조회할 수 있는 경로를 검토한다. 취소 확인에 쓰는 실제 영향도 같은 요약 규칙을 사용하되, 되돌릴 수량 자체는 변경하지 않는다.
- 수용 기준: 창고 수량과 같은 수량으로 변한 박스 effect는 기본 우측 카드에서 중복 표시되지 않는다. 창고 `-1 EA` 한 줄만 보고 결과를 이해할 수 있으며, 박스별 수량만 변한 예외나 불일치 상황은 감사 정보가 사라지지 않는다. 취소 확인의 영향 수량과 실제 재고 반전 범위는 기존과 동일하다.
- 브라우저 확인 시나리오: 박스 추적 재고를 사용하는 창고 불량 처리·출고·반품 이력을 각각 연다. 동일 delta의 창고/박스 pair가 한 결과로 보이는지, 박스만 변한 예외가 숨겨지지 않는지, 취소 확인에서 되돌릴 수량이 바뀌지 않는지 확인한다.
- 우선순위: 높음

### 57. 품목 전환 대표 행의 품목코드를 기존품·화살표·변경품 세 줄로 분리
- 화면/영역: PC 입출고 내역 탭 > 품목 전환 대표 row > `품목코드` 열
- 증상: 품목 전환 대표 row가 기존품 코드와 `↓ 변경품 코드`를 두 줄로 붙여 표시한다. 화살표가 변경품 코드와 같은 줄에 있어, `기존품 코드 / 화살표 / 변경품 코드`라는 전환 순서가 한눈에 읽히지 않는다.
- 사용자 불편: 품목 전환은 무엇이 무엇으로 바뀌었는지가 핵심인데, 코드 셀을 다시 읽어야 이전·이후 코드와 방향을 구분할 수 있다. 품목명은 전환 결론을 보여 주지만 코드도 같은 문법으로 따라오지 않아 검산 시 시선이 끊긴다.
- 확인 근거: 전환 대표 row가 `ItemCodeCell`에 `sourceCode`와 target code를 전달한다(`historyTableHelpers.tsx:670`). 이 컴포넌트는 source code 첫 줄, `↓ ${code}` 두 번째 줄을 렌더링한다(`historyTableHelpers.tsx:241-246`). 대표 row 높이는 64px로 고정돼 있다(`historyTableHelpers.tsx:41-42`).
- 확정 정책: 품목 전환 대표 row의 코드 셀은 `기존품 코드`, 가운데 아래 방향 화살표, `변경품 코드`를 각각 한 줄씩 세로로 표시한다. 화살표는 코드와 분리해 방향만 읽히게 하며, 일반 단건·BOM·하위 row의 한 줄 코드 표시는 바꾸지 않는다.
- 구현 방향: `ItemCodeCell`의 source/target 분기만 세 줄 구조로 바꾸고, 화살표 줄은 작은 고정 line-height와 중앙 정렬을 사용한다. 64px 대표 row 규칙 안에서 세 줄이 넘치지 않도록 상·하단 여백과 글자 크기를 조정하며, code 열 폭보다 긴 경우에는 각 코드만 독립적으로 말줄임하고 `title` 또는 접근성 이름으로 전체 두 코드를 제공한다. 패널 열림 compact 상태도 같은 세 줄 구조를 유지한다.
- 수용 기준: `4-AA-0077`, `↓`, `46-AR-0093`이 서로 다른 세 줄에 표시된다. 두 코드 모두 가운데 정렬되고 화살표가 특정 코드에 붙어 보이지 않는다. 대표 row 높이는 64px을 유지하며, 일반 이력과 하위 row의 코드 표시·열 정렬은 회귀하지 않는다.
- 브라우저 확인 시나리오: 짧은 코드와 긴 코드의 품목 전환 대표 row를 패널 닫힘/열림 PC 화면에서 각각 확인한다. 기존 코드·화살표·변경 코드의 세 줄 순서, 64px row 높이, 코드 말줄임 시 hover/focus 전체 텍스트, 인접 흐름·수량·상태 열 정렬을 확인한다.
- 우선순위: 높음

### 56. 입출고 내역 쇼케이스의 불량 직접 처리 출처를 업무 맥락에 맞게 선택
- 화면/영역: 개발 DB 검증 데이터 > `[HISTORY-DEMO]` 불량 직접 폐기·반품 이력 / 입출고 내역 우측 상세
- 증상: 현재 쇼케이스 생성기는 일반 창고 재고 품목 하나를 골라 격리·직접 폐기·직접 반품을 모두 실행한다. 그래서 목록과 상세에 `창고 -1 EA` 불량 처리가 연속으로 생겨, 조립·생산 중 발견한 불량을 검증하려는 화면에서는 다소 어색하게 보인다.
- 사용자 불편: 테스트 데이터임을 알아도 실제 업무 예시처럼 읽히지 않아, 불량 처리 흐름 UI 자체가 창고 전용 처리처럼 오해될 수 있다.
- 확인 근거: 쇼케이스는 `warehouse_qty >= 5`인 일반 품목을 선택한다(`scripts/dev/seed_history_showcase.py:101-111`). 불량 시나리오도 그 품목을 사용하며 직접 폐기·반품의 출발 bucket을 `WAREHOUSE`로 고정한다(`scripts/dev/seed_history_showcase.py:387-397`). 반면 서비스는 창고와 부서 생산 재고의 불량 격리를 모두 허용한다(`backend/app/services/io_preview.py:203-208`, `backend/app/models/stock_request.py:48-62`).
- 정책 확인 필요: 쇼케이스에서 직접 불량 폐기·반품은 가능한 경우 조립 등 부서 생산 재고를 우선 사용하고, 재고가 없을 때만 창고를 fallback으로 할지 결정한다. 창고 불량 처리는 입고 검사·보관 중 발견 같은 실제 업무 경로로 계속 허용하며, 제품 기능에서 금지하는 대상은 아니다.
- 구현 방향: 쇼케이스 계획 단계에 2 EA 이상인 생산 위치 후보를 추가로 찾고, 직접 폐기·반품은 그 후보에서 실행한다. 후보가 없을 때만 현재 일반 창고 품목을 사용하고, 콘솔 결과에 실제 출처를 명시한다. 격리 후 처리와 정상 재고 재작업의 기존 검증 범위·원자성은 유지한다.
- 수용 기준: 생산 위치 후보가 있는 개발 DB에서는 쇼케이스의 직접 불량 폐기·반품이 부서 생산 재고에서 생성된다. 후보가 없을 때는 전체 실행을 깨지 않고 창고 fallback을 사용하며, 생성 결과에서 출처를 확인할 수 있다. 창고 불량 처리 API와 기존 업무 동작은 변경되지 않는다.
- 브라우저 확인 시나리오: 쇼케이스 실행 후 `[HISTORY-DEMO]` 검색 결과에서 직접 불량 폐기·반품의 실제 영향 위치를 확인한다. 생산 후보가 있을 때는 `조립 생산` 등 부서 재고가, 후보가 없을 때는 `창고`가 명확히 표시되는지 확인한다.
- 우선순위: 낮음
