### 3. 관리자 공통 타이포그래피와 보조 정보 밀도 낮추기

- 화면/영역: 관리자 메뉴의 그룹 레이블·설명·활성 배지, 목록 행의 코드·상태, 표 헤더와 보조 설명.

- 사용자 불편: 긴 품목명·직원 정보·로그 파일명을 읽을 때 시선이 작은 상태 배지와 메타에 분산된다. 고해상도에서도 관리 화면이 축소된 표처럼 느껴진다.
- 확정 정책: 본문·입력값·핵심 목록명은 14px 이상, 보조 코드·메타는 12px 이상을 기본으로 한다. 10~11px은 열 제목이나 매우 짧은 비반복 보조 정보에만 제한한다.
- 구현 방향: 공통 관리자 프리미티브와 반복 목록 행부터 서체 크기·행간·배지 패딩을 정리한다. 데이터값의 우선순위를 바꾸지 않고, 상태는 텍스트 크기를 줄이는 대신 색·아이콘·여백으로 보조한다.
- 허용 기준: 모델명·품목명·직원명·부서명·파일명이 상태·코드보다 먼저 읽힌다. 10~11px의 반복 본문 사용이 사라지고, 버튼·입력 필드·표 행의 클릭 가능 영역은 줄지 않는다.
- 브라우저 확인 시나리오: 긴 품목명, 긴 직원 직급, 파일명, BOM 자식 목록을 확인한다. 100% 확대에서 본문과 메타가 구분되고, 좁은 폭에서 불필요한 줄바꿈·잘림·배지 겹침이 없는지 확인한다.
- 우선순위: 중간

- 증상: 공통 KPI 바는 전달된 항목 수만큼 항상 같은 폭의 카드를 렌더링한다(`AdminKpiBar.tsx:22-35`). 모델·품목·직원·부서는 의미 있는 요약이지만, 내보내기·외부 제출용 로그·보안에서도 헤더·KPI·대형 카드가 세로로 반복돼 실제 작업보다 화면 크롬과 빈 영역이 두드러진다.
- 사용자 불편: 사용자는 어떤 화면이 즉시 작업하는 워크벤치인지, 어떤 화면이 단순 조회·설정인지 첫눈에 구분하기 어렵다. 짧은 화면은 과하게 넓고, 긴 화면은 카드 안의 카드 때문에 좁아진다.
- 정책 확인 필요: KPI 바를 각 화면의 고정 문법으로 유지할지, 필터·우선순위 판단에 직접 쓰이는 경우만 남길지 결정한다.
- 구현 방향: 정책 결정 후 화면을 `편집 워크벤치`, `조회/내보내기`, `단일 설정`으로 구분한다. 워크벤치는 작업 폭과 스크롤을 우선하고, 가벼운 화면은 제한된 읽기 폭의 하나의 주요 표면으로 단순화한다. 데이터와 기존 버튼 기능은 바꾸지 않는다.
- 허용 기준: 사용자는 첫 화면에서 화면 성격과 주된 작업을 이해할 수 있다. 같은 정보가 KPI·카드 제목·본문에 반복되지 않고, 빈 화면은 의도적인 읽기 폭으로 정리된다.
- 브라우저 확인 시나리오: 모델·BOM·내보내기·보안을 나란히 비교한다. 워크벤치에는 충분한 작업 폭이 남고, 내보내기·보안에는 불필요한 빈 카드나 중첩 표면이 없는지 확인한다.
- 우선순위: 중간

- 사용자 불편: 적은 모델을 빠르게 비교·선택하는 단순 화면인데도 화면 크롬이 많고, 모델명·사용 상태·연결 수의 관계를 한눈에 읽기 어렵다.

- 브라우저 확인 시나리오: 모델을 각각 선택하고 추가 폼·삭제 확인 상태도 연다. 모델 수가 적은 현재 데이터와 더 많은 데이터가 있는 환경에서 목록 스크롤, 저장 버튼, 연결 품목 미리보기의 가독성을 확인한다.
- 우선순위: 중간

- 허용 기준: 직원 선택 뒤 기본 정보·권한·PIN·위험 동작의 순서가 분명하다. 긴 부서/직급도 목록에서 구분 가능하고, 비활성화·삭제는 의도치 않게 일반 저장 행동과 혼동되지 않는다.
- 브라우저 확인 시나리오: 일반 직원, 관리자/창고 권한 직원, 비활성 직원을 각각 선택한다. 탭 권한을 변경하고 스크롤한 뒤 저장·비활성화·삭제 위치와 확인 흐름을 점검한다. 좁은 데스크톱 폭에서 두 열이 겹치지 않는지도 확인한다.
- 우선순위: 중간

- 브라우저 확인 시나리오: 색상이 다른 부서를 여러 개 선택하고 색상을 바꾼다. 현재 선택 표시·색상 코드·목록의 점·직원 관리의 부서 색상이 일치하는지 확인한다. 좁은 폭과 다크 모드에서 색상 대비도 확인한다.
- 우선순위: 중간

## BOM 관리

### 9. 3패널 BOM 워크벤치의 작업 폭과 필터 계층 재배치

- 화면/영역: 관리자 > BOM 관리의 KPI, 부서 탭·부모 헤더, 부모 목록·하위 품목 추가·현재 구성 3패널.

- 사용자 불편: 부모 품목, 후보 자재, 현재 구성을 동시에 비교해야 하지만 긴 이름·수량·삭제 행동이 좁은 열에 들어간다. 필터와 실제 편집 영역의 위계가 약해 작업 시작점이 늦다.
- 확정 정책: BOM은 관리자 화면 중 작업 폭을 가장 우선하는 워크벤치로 취급한다. 부모 선택·후보 검색·현재 구성의 3가지 핵심 작업은 유지하되, 고정 탐색과 반복 요약이 본문보다 더 많은 폭·높이를 차지하지 않게 한다.
- 구현 방향: 공통 관리자 탐색 개선을 선행하고, 그 결과의 실제 폭에서 3패널 최소 폭·패널 간격·상단 필터 계층을 재측정한다. 부모 목록 폭, 후보 추가, 수량 저장, 삭제 확인, 사용처 모드, 미배치 원자재 접힘 동작은 바꾸지 않는다.
- 허용 기준: 긴 부모·자식 품목명, 코드, 수량, 추가/삭제 행동이 잘리지 않고 읽힌다. 편집 모드와 사용처 모드의 전환, 부서·상태 필터, 미배치 원자재는 기존처럼 동작하며 본문 작업 영역이 첫 화면에서 우선 보인다.
- 브라우저 확인 시나리오: 긴 이름의 부모 품목을 선택해 자식 후보 검색·추가·수량 변경·삭제 확인을 수행한다. 각 부서·상태 필터, 사용처 모드, 좁은 데스크톱 폭, 낮은 높이에서 수평/수직 스크롤과 버튼 겹침을 확인한다.
- 우선순위: 높음

## 내보내기

### 10. 내보내기 화면을 가벼운 작업 흐름에 맞게 압축

- 화면/영역: 관리자 > 내보내기의 KPI, 전체 Excel·선택 CSV 카드, 범위 설정, 최근 기록·마지막 내보내기 영역.
- 증상: 기본 상태에서 내보내기 기록이 없어도 4개 KPI와 두 개의 큰 작업 카드, 대형 기록 영역이 수직으로 쌓이고 하단에 넓은 빈 영역이 남는다. 실제 작업은 내보내기 종류와 범위 선택인데, 화면의 카드 수가 작업보다 많다.
- 사용자 불편: 사용자는 전체 내보내기와 선택 내보내기의 차이, 범위 설정, 최근 결과를 빠르게 연결하기 어렵다. 빈 상태가 데이터 없음이 아니라 화면이 비어 보이는 느낌을 준다.
- 확정 정책: 내보내기 종류와 범위 설정을 하나의 짧은 흐름으로 읽게 하고, 최근 기록이 없을 때는 빈 상태를 의도적인 안내로 표현한다. 데이터 범위와 다운로드 형식의 의미는 유지한다.
- 구현 방향: 화면 성격에 맞춰 KPI 필요성을 재검토하고, Excel/CSV 선택·범위 설정·최근 결과의 순서를 단순화한다. 실제 다운로드, 헤더 포함, 비활성 데이터 포함, 최근 기록 local storage 동작은 변경하지 않는다.
- 허용 기준: 사용자는 Excel 전체 내보내기와 CSV 선택 내보내기의 차이, 현재 기간·데이터 유형·범위를 한 화면에서 이해하고 실행할 수 있다. 기록이 없을 때도 공백이 아니라 명확한 빈 상태로 보인다.
- 브라우저 확인 시나리오: 전체 Excel과 선택 CSV의 범위·옵션을 각각 바꾸고, 빈 기록·기록이 있는 상태·다운로드 실패 상태를 확인한다. 긴 날짜 범위와 좁은 폭에서 버튼·옵션·최근 기록이 겹치지 않는지 확인한다.
- 우선순위: 중간

## 외부 제출용 로그

### 11. 외부 제출용 로그의 설명·표·다운로드 행동을 한 작업 단위로 정리

- 화면/영역: 관리자 > 외부 제출용 입출고 로그의 KPI, 자동 누적 설명, 백필 재실행, 월별 파일 표, 월 컬럼 안내.
- 증상: 월 파일이 3개인 기본 상태에서 KPI·설명 카드·표·하단 안내 카드가 따로 놓이고, 표 헤더·파일명·행동 버튼은 작은 글자와 여러 테두리로 표시된다. 하단에는 넓은 빈 공간이 남아 간단한 월별 다운로드 작업보다 화면 구조가 더 복잡해 보인다.
- 사용자 불편: 심사 대비 파일을 찾고 Excel/CSV를 내려받는 행동보다 자동 누적 설명·백필 재실행·표 보조 정보가 시선을 나눠, 필요한 월의 파일을 빠르게 선택하기 어렵다.
- 확정 정책: 월별 파일 확인과 다운로드를 주 작업으로, 자동 누적·백필은 보조 운영 정보로 둔다. 백필은 위험성이 있는 운영 동작이므로 월별 다운로드와 시각적으로 구분한다.
- 구현 방향: 월별 파일 표의 읽기 폭·행 높이·다운로드 버튼을 우선하고, 자동 누적 설명·백필·컬럼 안내는 한 보조 블록으로 정리한다. 파일 생성, Excel/CSV 다운로드 URL, 백필 재실행 동작과 권한은 변경하지 않는다.
- 허용 기준: 사용자는 월, 파일명, 거래 행, 용량, 다운로드 행동을 한 행에서 빠르게 읽을 수 있다. 백필 재실행은 일반 다운로드와 구분되며, 표가 작아져도 파일명·버튼이 잘리지 않는다.
- 브라우저 확인 시나리오: 여러 월 파일과 파일이 없는 월을 확인하고 Excel·CSV 버튼의 hover·disabled·다운로드 흐름을 점검한다. 백필 재실행 전후의 설명·확인·오류 상태, 긴 파일명·좁은 폭·다크 모드도 확인한다.
- 우선순위: 중간

## 보안

### 12. 관리자 PIN 변경 화면의 읽기 폭을 제한하고 단일 작업에 집중

- 화면/영역: 관리자 > 보안의 `관리자 PIN 변경` 카드와 현재 PIN·새 PIN·새 PIN 확인 입력.
- 증상: 세 PIN 입력이 매우 넓은 단일 카드에 가로로 배치되고(`AdminDangerZone.tsx:31-65`), 입력 아래에는 큰 빈 영역이 남는다. 관리자 메뉴와 공통 헤더까지 남아 있어 단일 설정 작업이 화면 전체에 흩어진 느낌을 준다.
- 사용자 불편: 사용자는 PIN 변경이라는 하나의 민감한 작업보다 넓은 빈 카드와 주변 탐색을 먼저 보게 된다. 입력 순서·오류·변경 행동의 집중도가 낮다.
- 확정 정책: PIN 변경은 제한된 읽기 폭의 단일 보안 작업으로 표현한다. 현재 PIN, 새 PIN, 확인, 변경 버튼, 유효성 오류의 순서를 분명히 하고, 위험도는 유지하되 불필요한 여백은 줄인다.
- 구현 방향: 공통 탐색 개선 이후 카드의 최대 폭·입력 배치·설명 길이를 단일 작업에 맞게 조정한다. PIN 검증 규칙, 변경 API, 성공/실패 토스트, 관리자 잠금·인증 흐름은 변경하지 않는다.
- 허용 기준: 넓은 화면에서도 PIN 변경 폼이 읽기 좋은 폭으로 정렬되고, 현재 PIN·새 PIN·확인·오류·변경 버튼의 관계가 명확하다. 키보드만으로 모든 입력과 변경을 수행할 수 있다.
- 브라우저 확인 시나리오: 빈 값, 불일치 PIN, 짧은 PIN, 정상 PIN 변경의 유효성 상태를 확인한다. 비밀번호 마스킹·오류 문구·성공 토스트·포커스 순서가 유지되고, 좁은 폭·다크 모드에서도 입력이 겹치지 않는지 점검한다.
- 우선순위: 낮음
