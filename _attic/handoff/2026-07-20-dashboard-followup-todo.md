## 대시보드

### 1. 빠른 작업 입고·출고 색상 체계 정리
- 화면/영역: 데스크톱 대시보드 품목 상세 패널의 `빠른 작업` 영역. 입고/출고 상위 버튼과 펼쳐지는 세부 작업 카드.
- 증상: 입고·출고 세부 작업군의 바깥 상자와 내부 작업 버튼이 모두 계열색으로 채워져, 버튼이 상자의 구분색과 경쟁하며 화면이 과하게 강조된다.
- 사용자 불편: 입고와 출고의 작업군 구분은 필요하지만, 내부 버튼까지 색으로 채워져 있어 빠른 작업 선택지가 무겁고 산만하게 보인다.
- 확정 정책: 입고 작업군의 바깥 상자는 파스텔 푸른색, 출고 작업군의 바깥 상자는 파스텔 붉은색 계열을 유지한다. 그 안의 세부 작업 버튼은 계열색 배경을 사용하지 않고 중립 표면으로 표시한다. 버튼의 텍스트·아이콘·테두리도 불필요하게 계열색으로 채우지 않으며, 선택·hover·focus 상태만 기존 상호작용 문법으로 구분한다.
- 구현 방향: 공통 토큰(`LEGACY_COLORS.blue`/`red`) 자체를 변경하지 않고, `InventoryDetailPanel`의 데스크톱 빠른 작업 분기에서 작업군 컨테이너에만 저채도 계열색 배경·테두리를 적용한다. 내부 세부 작업 버튼은 공통 중립 표면·중립 테두리로 되돌리되, 텍스트 대비와 기존 클릭·펼침 동작은 유지한다.
- 허용 기준: 대시보드 빠른 작업에서 입고·출고 작업군은 각각 파스텔 블루·레드의 바깥 상자로 구분되며, 그 안의 모든 세부 작업 버튼은 색으로 채워지지 않는다. 다른 화면과 모바일 빠른 작업의 색상은 변경되지 않는다.
- 브라우저 확인 시나리오: 품목을 선택한 뒤 입고와 출고를 각각 펼친다. 각 작업군의 바깥 상자만 해당 계열색으로 보이고 내부 세부 작업 버튼은 중립 표면인지, 선택·해제 및 입출고 화면 이동이 기존과 동일하게 동작하는지 확인한다.
- 우선순위: 중간

### 2. BOM 상세 팝업으로 하위 구성 보기
- 화면/영역: 데스크톱 대시보드 품목 상세 우측 패널의 `하위 구성` 영역.
- 증상: 현재 하위 구성이 좁은 우측 패널 안에 즉시 펼쳐져, 품목명·부서·코드·소요수량이 압축된 작은 트리로 보인다. 구성 단계를 파악하려면 패널을 계속 스크롤해야 하고, 빠른 작업 영역과 시선 경쟁이 생긴다.
- 사용자 불편: BOM을 독립적으로 읽고 펼쳐 보기 어렵고, 품목 이미지 확대처럼 집중해서 보는 화면이 없다.
- 확정 정책: 기존 인라인 `하위 구성 보기/접기`와 패널 내 트리 표시는 제거하고, BOM 부모 품목에만 `BOM 상세 보기` 버튼을 제공한다. 버튼을 누르면 이미지 확대 팝업과 동일하게 배경을 차단하고 ESC·배경 클릭·닫기 버튼으로 닫을 수 있는 읽기 전용 BOM 상세 팝업을 연다.
- 구현 방향:
  - 기존 `GET /api/bom/{itemId}/tree`와 `BOMTreeNode` 데이터를 재사용한다. BOM 조회용 새 API나 관리자 BOM 편집 기능은 추가하지 않는다.
  - 범용 `ImageLightbox`는 이미지 전용으로 유지한다. 팝업의 배경 차단·닫기·포커스 처리 문법은 `ImageLightbox`와 `ConfirmModal`을 따르되, BOM 본문을 담는 별도 `BomDetailModal`을 만든다.
  - `InventoryDetailPanel`에서는 기존 `showBom` 인라인 분기를 새 모달 열기 상태와 `BOM 상세 보기` 버튼으로 교체한다. 품목을 바꾸면 팝업과 펼침 상태를 초기화한다.
  - 트리 표현은 기존 `BomSubExpander`의 선형 계층·펼침 동작을 재사용하거나, 이를 BOM 상세 전용 트리 뷰로 최소 추출한다. 같은 트리 렌더러를 이중 구현하지 않는다.
- 디자인 가이드라인:
  - 팝업은 화면 중앙의 넓은 읽기 패널(`max-width` 약 960px, 최대 높이 약 84vh)로 두고, 기존 모달처럼 `rounded-[24px]`, `LEGACY_COLORS.s1`, `LEGACY_COLORS.border`, `var(--c-card-shadow)`를 사용한다. 직접 HEX 색상을 추가하지 않는다.
  - 어두운 배경막·ESC·배경 클릭·우상단 원형 닫기 버튼은 이미지 팝업의 친숙한 상호작용을 유지한다. 다만 다이얼로그 표면은 이미지가 아닌 정보 화면이므로 닫기 버튼과 제목에 명확한 포커스 순서를 제공한다.
  - 헤더에는 `Network` 아이콘, `BOM 상세`, 부모 품목명, MES 코드, 읽기 전용 배지를 한 줄의 정보 계층으로 배치한다. 긴 품목명은 두 줄까지만 표시하고 전체 이름은 접근 가능한 레이블로 보존한다.
  - 본문은 고정 헤더와 스크롤 가능한 트리로 나눈다. 첫 단계 자식은 기본으로 보이고, 더 깊은 단계는 각 행의 chevron으로 펼친다. 트리 선은 중립색으로 유지해 부서 색상과 경쟁하지 않게 한다.
  - 각 BOM 행은 `품목명`을 가장 강하게, 부서 배지·MES 코드·`소요 ×수량`·`현재고`를 보조 정보로 정렬한다. 현재고는 API가 제공하는 `current_stock`을 사용하고, 수량이 부족한지를 새로 판정하거나 경고 상태를 만들지는 않는다.
  - 행 hover와 펼침 버튼은 `LEGACY_COLORS.s4` 및 150ms 전환을 사용한다. 아이콘 버튼의 실제 터치 영역은 44px 이상으로 하며, 아이콘은 `lucide-react` 16~20px로 통일한다.
  - 로딩은 본문 스켈레톤 또는 간결한 로딩 문구로, 실패는 같은 본문 안의 오류 설명과 `다시 시도` 버튼으로 표시한다. 자식이 없으면 빈 상태를 보여 주고, 빈 목록을 오류처럼 보이게 하지 않는다.
  - 이번 범위는 데스크톱 대시보드다. 모바일 품목 상세 및 이미지 확대 팝업의 화면 구성은 변경하지 않는다.
- 허용 기준: BOM 부모 품목에서만 상세 버튼이 보이고, 인라인 하위 구성은 더 이상 우측 패널의 높이를 차지하지 않는다. 팝업에서 다단 BOM을 펼쳐 읽을 수 있으며, 닫기·오류·빈 상태·긴 품목명·다크 모드에서도 정보와 대비가 유지된다.
- 브라우저 확인 시나리오:
  1. BOM 부모 품목을 선택해 `BOM 상세 보기`를 연다. 부모 식별 정보와 1단계 구성품이 보이는지 확인한다.
  2. 자식이 있는 구성품을 여러 단계로 펼치고 접는다. 계층선·수량·현재고·부서·코드가 다른 행과 겹치지 않는지 확인한다.
  3. ESC, 배경 클릭, 닫기 버튼으로 각각 닫은 뒤 우측 패널 선택과 빠른 작업 상태가 그대로인지 확인한다.

### 3. 빠른 작업을 우측 패널 하단에 고정
- 화면/영역: 데스크톱 대시보드 품목 상세 우측 패널의 `빠른 작업` 전체. 제목, 입고·출고 상위 버튼, 펼쳐진 세부 작업 선택지를 모두 포함한다.
- 증상: 빠른 작업이 수량 현황·위치별 재고·하위 구성과 같은 본문 스크롤 영역의 마지막에 있다. 상세 내용이 길어지면 빠른 작업을 보려면 패널을 끝까지 스크롤해야 하고, 버튼 자체도 본문과 함께 이동한다.
- 사용자 불편: 품목을 확인한 뒤 입고·출고를 바로 실행하려면 반복 스크롤이 필요하며, 스크롤 중 버튼 위치를 잃는다.
- 확정 정책: 빠른 작업 전체를 품목 상세 패널의 최하단 고정 영역으로 옮긴다. 본문에 스크롤이 생겨도 빠른 작업은 스크롤되지 않고 계속 보인다. 열린 입고·출고 세부 선택지도 고정 영역에 함께 표시한다.
- 구현 방향:
  - `InventoryDetailPanel`의 데스크톱 빠른 작업만 `DesktopRightPanelFooter`로 포털 렌더링한다. `DesktopRightPanelFooter`는 이미 스크롤 본문 밖의 `shrink-0` 푸터를 제공하므로, 새로운 sticky CSS나 별도 스크롤 컨테이너를 만들지 않는다.
  - 본문에는 수량 현황·위치별 재고·BOM 상세 진입 버튼만 남기고 `빠른 작업` 블록을 제거한다. 모바일은 `DesktopRightPanelFooter` 컨텍스트가 없으므로 현재 인라인 배치와 동작을 유지한다.
  - 현재 데스크톱 빠른 작업의 넘침 감지 후 본문을 강제로 스크롤하는 `firstQuickChoiceRef`/`useDesktopRightPanelBody` 처리도 함께 제거한다. 고정 푸터 안에서 상위 버튼과 세부 선택지를 모두 표시하므로 본문 스크롤 보정이 필요 없다.
  - 푸터 내부에는 본문과 구분되는 상단 경계와 적절한 상단 여백만 추가하고, 입고·출고 버튼의 현재 색상/파스텔 색상 개선(항목 1), 권한, 선택·해제, 입출고 화면 이동은 변경하지 않는다.
- 허용 기준: 긴 재고·예약·BOM 정보로 우측 패널 본문을 스크롤해도 `빠른 작업` 제목과 입고·출고 버튼이 하단에 계속 보인다. 입고 또는 출고를 펼치면 세부 선택지도 본문 스크롤과 무관하게 모두 보이고, 버튼을 눌러 이동한 뒤에는 기존처럼 선택 상태가 닫힌다.
- 브라우저 확인 시나리오:
  1. 내용이 긴 품목을 선택하고 우측 패널 본문을 최상단·중간·최하단까지 스크롤한다. 빠른 작업의 위치가 변하지 않는지 확인한다.
  2. 입고와 출고를 각각 펼친다. 세부 선택지가 잘리지 않고, 본문을 스크롤해도 같이 이동하지 않는지 확인한다.
  3. 세부 작업을 선택해 입출고 화면으로 이동한 뒤 품목을 다시 선택한다. 빠른 작업이 닫힌 초기 상태이며 권한별 입고 선택지가 기존과 같은지 확인한다.
  4. 모바일 대시보드와 낮은 데스크톱 높이에서 본문만 줄어들고 빠른 작업이 중복·겹침·별도 스크롤 없이 유지되는지 확인한다.
- 우선순위: 중간

### 8. 수량 현황에서 사용 가능 재고와 승인 대기 수량의 위치 교체
- 화면/영역: 데스크톱 대시보드 품목 상세 우측 패널의 `수량 현황` 카드 첫 줄.
- 증상: 현재 2열 카드에서 `승인 대기 수량`이 왼쪽, `사용 가능 재고`가 오른쪽에 표시된다.
- 사용자 불편: 품목을 선택한 뒤 가장 먼저 확인할 수 있는 사용 가능 재고가 오른쪽에 있어, 좌에서 우로 읽는 정보 흐름에서 핵심 수량을 바로 파악하기 어렵다.
- 확정 정책: 첫 번째 카드(왼쪽)는 `사용 가능 재고`, 두 번째 카드(오른쪽)는 `승인 대기 수량`으로 배치한다. 각 수량의 값·색상·경고 테두리 규칙은 유지한다.
- 구현 방향: `InventoryDetailPanel` 수량 현황의 2열 카드 렌더링 순서만 교체한다. 계산식(`availableQty`/`pendingQty`), 카드 디자인, 공급처 등 아래 정보, 모바일·다른 화면의 수량 표시에는 손대지 않는다.
- 허용 기준: 선택한 품목의 수량 현황에서 왼쪽에 사용 가능 재고와 기존 상태색이, 오른쪽에 승인 대기 수량과 승인 대기 시 기존 노란 경고가 보인다. 두 값과 다른 정보는 바뀌지 않는다.
- 브라우저 확인 시나리오:
  1. 승인 대기 수량이 0인 품목을 선택한다. 왼쪽에 `사용 가능 재고`, 오른쪽에 `승인 대기 수량 0`이 보이는지 확인한다.
  2. 승인 대기 수량이 있는 품목을 선택한다. 오른쪽 카드만 기존처럼 노란 강조 테두리·값 색상을 유지하는지 확인한다.
  3. 품목을 여러 개 바꾸고, 위치별 재고·BOM 보기·빠른 작업과 모바일 품목 상세를 확인한다. 값 갱신과 다른 레이아웃이 바뀌지 않는지 확인한다.
- 우선순위: 중간

## 입출고

### 4. 내 요청에서도 기본 탐색 탭 유지
- 화면/영역: 데스크톱 입출고 상단의 `요청 작성`·`작성 중`·`내 요청` 기본 탐색 탭과 `내 요청` 목록 화면.
- 증상: 요청 작성 화면에서는 세 탭이 보이지만, `내 요청`으로 들어가면 목록만 표시되고 같은 탭이 보이지 않는다.
- 사용자 불편: 다른 입출고 작업으로 이동하려면 현재 위치를 벗어나거나 별도 탐색을 해야 하며, 사용자가 자신이 `내 요청` 섹션에 있다는 맥락과 세 화면의 관계를 즉시 파악하기 어렵다.
- 확정 정책: `내 요청`에서도 요청 작성·작성 중·내 요청의 세 탭을 상단에 항상 표시한다. 현재 선택된 `내 요청` 탭은 기존 선택 상태로 구분한다.
- 구현 방향:
  - `DesktopWarehouseView`가 모든 일반 섹션에서 `WarehouseSectionTabs`를 렌더링하도록 보장한다. 작성 화면의 품목 전환 집중 상태에서만 탭을 숨기는 기존 예외는 유지하되, `mine` 진입·직접 URL 진입·새로고침에서 그 숨김 상태가 남아 있지 않게 한다.
  - `WarehouseSectionTabs`의 기존 세 기본 탭 정의와 클릭·URL 동기화 방식을 그대로 사용한다. 탭을 `MyRequestsPanel` 내부에 중복 렌더링하거나 새로운 탭 상태를 만들지 않는다.
  - 탭의 크기·색상·알림 배지·권한별 추가 탭(승인함 등)은 변경하지 않는다. 이번 범위는 데스크톱 입출고의 기본 탐색 노출 회복이다.
- 허용 기준: `?tab=warehouse&section=mine` 직접 진입, 요청 작성에서 내 요청으로 전환, 새로고침 후 재진입 모두에서 세 기본 탭이 목록 위에 보인다. 내 요청 탭만 선택 상태이고, 요청 작성·작성 중 탭을 누르면 기존 URL과 화면 전환이 동일하게 동작한다.
- 브라우저 확인 시나리오:
  1. 요청 작성 화면에서 내 요청을 누른다. 목록 상단에 세 탭이 남아 있고 내 요청만 선택 표시인지 확인한다.
  2. `?tab=warehouse&section=mine` URL로 직접 진입하고 새로고침한다. 탭이 숨지 않는지 확인한다.
  3. 내 요청에서 요청 작성·작성 중으로 각각 이동한 뒤 뒤로 가기와 새로고침을 수행한다. 탭 선택 상태와 URL이 일치하는지 확인한다.
  4. 품목 전환 집중 상태의 작성 화면에서는 기존처럼 탭 숨김 동작이 유지되고, 모바일 입출고 화면은 바뀌지 않는지 확인한다.
- 우선순위: 중간

### 5. 펼칠 수 있는 하위 행 전체를 펼침 조작으로 사용
- 화면/영역: 데스크톱 입출고 내역에서 상위 이력을 펼친 뒤 나타나는 BOM 묶음, 불량 처리 결과, 참조 묶음 등 자식을 가진 하위 행.
- 증상: 현재 하위 행의 자식 구성은 행 안의 작은 chevron 버튼을 정확히 눌러야만 펼쳐진다. 행의 다른 부분을 눌러도 펼침이 일어나지 않는다.
- 사용자 불편: 행 전체가 하나의 묶음처럼 보이지만 실제 조작 영역이 작아, 자식 구성을 빠르게 확인하기 어렵다.
- 확정 정책: 자식이 있어 chevron이 표시되는 모든 하위 묶음 행은 행 전체를 눌러 펼치고 접을 수 있게 한다. BOM 구성, 단품 수량보정 구성, 폐기·불량 처리 결과, 참조 묶음에 같은 규칙을 적용한다.
- 구현 방향:
  - `BomBatchDetail`의 `BundleRows`, `ReworkBatchDetail`의 펼칠 수 있는 `ReworkSummaryRow`, `historyTableHelpers`의 펼칠 수 있는 `ReferenceBatchLineRow`에 공통 행 토글 계약을 적용한다.
  - chevron 버튼은 유지하되 행 클릭과 같은 토글을 수행하게 하고, 버튼 클릭은 이벤트 전파를 막아 한 번의 클릭으로 두 번 토글되지 않게 한다.
  - 행 전체에는 클릭 가능 커서·hover·키보드 포커스를 제공하고, Enter/Space도 click과 동일하게 토글한다. `aria-expanded`와 `aria-controls`는 행과 chevron이 같은 상세 영역을 가리키도록 유지한다.
  - 자식이 없는 하위 상세 행의 기존 로그 선택/우측 상세 패널 동작은 바꾸지 않는다. 펼칠 수 있는 행에서 선택과 펼침이 함께 발생하던 경우에는 펼침만 한 번 수행해, 한 클릭으로 상세 선택과 펼침이 경쟁하지 않게 한다.
  - 최상위 이력 행의 현재 선택·상세 표시 동작과 행 높이·열 정렬·수량 표시는 변경하지 않는다.
- 허용 기준: chevron이 보이는 하위 묶음 행은 날짜·작업·대상·수량 어느 열을 눌러도 같은 자식 목록을 한 번씩 펼치고 접는다. chevron 버튼과 키보드도 동일하게 동작하며, 자식 없는 행을 클릭하면 기존 상세 선택만 수행한다.
- 브라우저 확인 시나리오:
  1. BOM 또는 단품 수량보정 묶음을 연 뒤, 그 하위 묶음 행의 대상·수량·빈 셀을 각각 클릭한다. 모든 위치에서 자식 행이 한 번만 열리고 닫히는지 확인한다.
  2. 불량 격리·폐기 결과와 참조 묶음도 같은 방식으로 확인한다. 각 유형의 결과 행이 일관되게 동작하는지 비교한다.
  3. 행과 chevron에 키보드로 포커스를 두고 Enter/Space를 누른다. `aria-expanded` 상태와 표시된 자식 목록이 일치하는지 확인한다.
  4. 자식 없는 하위 행과 최상위 이력 행을 클릭한다. 기존 우측 상세 선택만 유지되고 의도하지 않은 펼침이 생기지 않는지 확인한다.
- 우선순위: 중간

### 6. 불량 이력의 우측 상세 작업 분류를 실제 처리 유형으로 표시
- 화면/영역: 데스크톱 입출고 내역의 불량 이력 행과 우측 상세 패널의 `작업`, `위치/이동 경로`, 재고 변화 요약.
- 증상: 입출고 내역의 상위 작업 열은 의도대로 `불량`으로 보이지만, 재작업 묶음을 선택했을 때 우측 상세의 작업이 `폐기`로 표시될 수 있다. 재작업에 포함된 부품별 결과 로그의 `DEFECT_SCRAP`을 대표 작업으로 해석해, 실제로는 재작업인 묶음이 폐기처럼 보인다.
- 사용자 불편: 재작업·폐기·반품·정상 복귀 등 서로 다른 불량 처리 작업을 우측 상세에서 구분할 수 없다. 특히 `폐기 N EA · 회수 N EA`는 재작업 중 발생한 부품 처리 결과인데, 상위 작업까지 폐기로 오해하게 만든다.
- 확정 정책:
  - 목록의 `작업` 열은 모든 불량 탭 처리 흐름에서 계속 `불량`으로 표시한다.
  - 우측 상세의 `작업`은 불량 탭에서 사용자가 선택한 실제 처리 유형으로 표시한다. 표기 기준은 `불량 격리`, `정상 복귀`, `폐기`, `재작업`, `반품`이다.
  - `defect-disassemble`/`DISASSEMBLE` 재작업 묶음은 내부에 `DEFECT_SCRAP` 및 `RECEIVE` 로그가 있어도 작업을 반드시 `재작업`으로 표시한다. 같은 맥락에서 위치·이동 경로가 종료 지점을 `폐기`로 반복 표기하지 않게 실제 작업과 일치시킨다.
  - 재작업 하위 결과의 `폐기 N EA`, `회수 N EA`, `폐기 N EA · 회수 N EA` 표시는 부품별 처리 결과이므로 유지한다. 이는 상위 작업명과 다른 정보 계층이다.
- 구현 방향:
  - 우측 상세용 작업 분류 해석기를 단일화해 `IoBatch.sub_type`, 재작업 참조 번호/`DISASSEMBLE`, 배치 전체 로그를 우선 판단한다. 첫 번째 비차감 로그 또는 개별 `DEFECT_SCRAP`만으로 대표 작업을 고르지 않는다.
  - 이 분류기를 `buildHistoryDetailSummary`, 배치 상세 헤더 및 우측 패널의 작업·흐름 라벨에 함께 적용한다. 목록 작업 열의 `getHistoryListOperationLabel` 정책은 분리하여 변경하지 않는다.
  - 불량 탭의 모든 실제 진입 동작을 매핑 테스트로 고정한다: 격리, 정상 복귀, 전체 폐기, 공급처 반품, BOM 재작업, 그리고 격리 없이 처리하는 동일 유형. 권한에 따라 노출되는 반품은 창고 흐름에서만 검증한다.
  - 재작업 테스트에는 `DEFECT_SCRAP`과 `RECEIVE`가 함께 있는 배치를 포함하고, 부모 행·하위 결과 행을 각각 선택해도 우측 상세의 작업이 `재작업`으로 유지되는지 확인한다.
- 허용 기준: 불량 이력의 목록 작업 열은 계속 `불량`이다. 우측 상세는 사용자가 수행한 실제 유형을 표시하며, 재작업은 어떤 부품 결과를 선택해도 `폐기`로 바뀌지 않는다. 반면 재작업 결과 목록에서는 폐기·회수 수량이 정확히 함께 보인다.
- 브라우저 확인 시나리오:
  1. 불량 격리, 정상 복귀, 전체 폐기, 공급처 반품, BOM 재작업 이력을 각각 연다. 목록은 모두 `불량`이고 우측 상세만 실제 처리 유형으로 구분되는지 확인한다.
  2. `폐기 1 EA · 회수 1 EA`가 포함된 재작업 묶음에서 상위 행과 각 하위 결과 행을 번갈아 선택한다. 우측 상세 작업은 계속 `재작업`, 하위 수량 결과는 계속 폐기·회수로 보이는지 확인한다.
  3. 재작업 우측 상세의 위치·이동 경로와 재고 변화가 `폐기` 단일 작업처럼 모순되지 않는지, 부모품과 회수 부품의 재고 증감이 모두 보이는지 확인한다.
  4. 격리 없이 바로 처리한 폐기·반품·정상 복귀 이력과 창고 권한의 반품 이력을 확인한다. 기존 취소·상세 선택·자식 펼침 동작은 바뀌지 않는지 확인한다.
- 우선순위: 중간

### 7. 직원 관리의 직접 부서 색상 폴백을 제거하고 부서 마스터를 재사용
- 화면/영역: 데스크톱 관리자 > 부서 관리의 부서 목록 색상과 관리자 > 직원 관리의 직원별 부서 점 표시.
- 증상: 부서 관리에서는 `기타` 부서의 저장된 `color_hex`가 목록과 상세 미리보기에 보인다. 그러나 직원 관리의 `권동환 · 기타` 같은 직원 행은 이름 기반 `getDepartmentFallbackColor()`를 직접 사용해, 기타의 기본 폴백색으로 계속 표시된다.
- 사용자 불편: 같은 `기타` 부서가 관리자 화면 안에서 서로 다른 색으로 보이며, 관리자가 저장한 부서 색상 설정이 직원 관리에는 적용되지 않은 것처럼 느껴진다.
- 확정 정책: 직원 관리의 부서 색상 표시는 부서 관리와 동일한 `DepartmentMaster` 색상 해석을 재사용한다. 직원 관리의 목록 행에서 이름만으로 색을 다시 추론하는 `getDepartmentFallbackColor()` 직접 호출은 제거한다.
- 구현 방향:
  - `AdminEmployeesSection`에 이미 전달되는 `departments`에서 직원 부서와 일치하는 `DepartmentMaster`를 찾고, 부서 관리가 사용하는 `deptColor(department)` 해석을 그대로 사용한다. 이 화면의 `getDepartmentFallbackColor` import와 직접 호출을 삭제한다.
  - 부서 색상 저장 뒤 부서 관리와 직원 관리가 같은 부서 마스터 상태를 갱신·공유하도록 보장한다. 별도의 색상 캐시, 이름별 색상 맵, 직원 전용 폴백을 만들지 않는다.
  - 부서 마스터의 색상값이 비어 있는 예외는 부서 관리의 기존 단일 해석기에서만 처리한다. 직원 관리가 별도의 폴백 규칙을 갖지 않게 하며, 전체 팔레트나 기존 부서 색상값은 변경하지 않는다.
- 허용 기준: `기타`를 포함해 색상을 변경·저장한 모든 부서는 부서 관리 목록, 직원 관리 목록의 부서 점, 직원 상세의 해당 부서 표기에서 같은 색으로 보인다. 색상값이 없는 부서는 기존 폴백색을 유지한다.
- 브라우저 확인 시나리오:
  1. 부서 관리에서 `기타`의 색상을 변경하고 저장한다. 좌측 부서 목록의 점과 상세 미리보기 색상을 기록한다.
  2. 직원 관리로 이동해 `기타` 소속 직원 행을 확인한다. 부서 점 색상이 방금 저장한 색상과 정확히 같은지 확인한다.
  3. 다른 부서의 색상도 변경한 뒤 직원 관리에서 바로 확인하고, 새로고침·다른 직원 선택·관리자 탭 왕복 후에도 최신 색상이 유지되는지 확인한다.
  4. `color_hex`가 없는 부서와 비활성 부서도 확인한다. 기존 폴백색·불투명도·직원 선택 상태가 달라지지 않는지 확인한다.
- 우선순위: 중간
