# 출하 PC UI 다음 개선 투두

작성일: 2026-06-30
상태: 수집 중

## 이전 투두 정리

기준 파일: `_attic/handoff/2026-06-29-shipping-performance-todo.md`
정리일: 2026-06-30

### 구현 완료로 정리

- 출하 첫 진입 성능 개선
  - 품목 2000건 로딩을 초기 진입에서 분리하고, 요청 작성/수정 진입 시점으로 늦김.
  - 출하 이력은 별도 초기 호출 대신 요청 목록의 `PICKED_UP` 상태에서 파생.
  - `verify_local.ps1` 기준 프론트 검증 통과.

- 상태 전환/잔여 요청 복구
  - `REQUESTED` 상세에서 다시 `준비 중으로 보내기` 가능.
  - `REQUESTED/PREPARING` 요청 취소 흐름 추가.
  - `PREPARED`는 준비 완료 취소 흐름으로 분리.
  - 기존 잔여 요청은 화면 기능으로 `PREPARING` 전환 확인.

- 출하 PC 화면 구조 개선
  - 허브, 요청 목록, 준비 중 목록, 이력 목록, 요청 상세를 전폭 작업 패널 문법으로 정리.
  - 내부 UUID 노출 제거.
  - 브라우저 URL/history 기반 출하 내부 화면 이동 반영.
  - 요청 상세의 액션 영역과 상태별 버튼 노출 정리.

- 출하 요청 작성/수정 마법사 개선
  - `PF 선택 -> BOM 구성 -> BOM 매칭 -> 요청 정보 -> 저장 및 전환` 단계형 흐름 적용.
  - 단계 탭과 헤더 중복 정리.
  - 페이지 전체 스크롤을 줄이고 긴 목록은 내부 스크롤로 처리하는 방향 적용.
  - 기준 PF 미선택 등 필수 조건 미충족 시 다음 단계 이동 비활성화.

- BOM 편집/매칭 UX 개선
  - 기본 BOM 다시 불러오기 버튼 제거.
  - 동일 BOM 확인 버튼 제거, 자동 매칭 결과 중심으로 정리.
  - 기존 BOM row의 품목 드롭다운 제거.
  - 품목 변경은 `제외 + 추가` 흐름으로 정리.
  - 새 PA/PF 이름과 품목코드 생성/재사용 안내를 최종 단계에 표시.

- 요청자/저장 액션 정리
  - 새 요청 생성 시 로그인 사용자 이름을 요청자로 사용.
  - 요청자 입력란은 읽기 전용 정보로 정리.
  - 5단계의 `요청 저장`, `준비 중으로 보내기`를 하단 액션바로 이동.

- 허브 카드 4차 정리
  - `바로 열기` CTA 제거.
  - 카드 하단 설명 문구 추가.
  - 건수 배지 가독성 개선.

### 보류 또는 후속 검토

- DB 락/세션 전략 개선
  - SQLite 읽기 요청까지 `BEGIN IMMEDIATE`를 잡는 구조는 출하 성능 문제의 근본 원인 후보지만, 전체 DB 동시성 정책과 관련되어 별도 백엔드 과제로 보류.
  - 읽기 전용 세션/엔진 분리 또는 조회 라우터 트랜잭션 전략 변경은 별도 계획 필요.

- 취소 이력 정책
  - 현재 `REQUESTED/PREPARING` 취소는 요청 삭제 중심으로 처리.
  - `CANCELLED` 상태를 만들어 이력으로 남기는 안은 보류.
  - `PICKED_UP` 취소는 v1 범위 밖으로 유지.

- 실사용 화면 미세 조정
  - 기존 큰 항목들은 구현됐지만, 실제 작업자가 화면을 보며 느끼는 여백, 밀도, 문구, 액션 위치는 계속 추가 투두로 수집한다.
  - 이 파일의 `수집 항목`에 앞으로 새 개선사항을 누적한다.

## 원칙

- 이 파일은 다음 출하 PC UI 개선사항을 구현 전까지 모아두는 투두 리스트다.
- 사용자가 명시적으로 구현을 지시하기 전까지 여기 적힌 항목은 구현하지 않는다.
- PC 실사용 흐름과 디자인 완성도를 우선한다.
- 모바일 출하 화면과 모바일 하단 탭바 frozen 스타일은 건드리지 않는다.
- 주간보고 frozen 파일은 건드리지 않는다.
- 구현 시 `$design` 스킬을 먼저 확인하고, 불량/입출고 탭의 업무형 화면 문법과 맞춘다.
- 색상은 `LEGACY_COLORS`, CSS 변수, 기존 token 흐름만 사용하고 inline hex는 추가하지 않는다.

## 수집 항목

### 1. 출하 요청 수량 지정 흐름 추가 검토

- 현재 문제:
  - 출하 요청이 최종 PF 1대 기준으로 설계되어 있다.
  - 실제 출하에서는 같은 PF가 여러 대 나갈 수 있으므로 요청 수량을 지정할 수 있어야 한다.
  - 수량이 들어가면 BOM 구성품 수량, 준비 체크, 준비 완료 재고 반영, 픽업 완료 수량이 모두 요청 수량 기준으로 연결되어야 한다.
- 사용자 메모:
  - 별도 `수량 지정` 단계를 만들 수도 있지만, 단계가 너무 많아질 수 있다.
  - 기존 단계 어딘가에 자연스럽게 합치는 방향이 좋아 보인다.
- 검토 방향:
  - 1단계 `기준 PF 선택`에 수량을 함께 넣는 안:
    - 장점: PF를 고르는 순간 “몇 대 출하할지”까지 기준이 잡혀 이후 BOM 수량을 바로 계산할 수 있다.
    - 단점: PF 선택 화면이 검색/선택과 수량 입력을 같이 품어야 하므로 첫 단계가 조금 무거워질 수 있다.
  - 4단계 `요청 정보`에 수량을 넣는 안:
    - 장점: 요청자/메모와 함께 요청 조건을 입력하는 느낌이라 자연스럽다.
    - 단점: 2~3단계 BOM 조정 화면에서는 수량 기준 총 필요량을 미리 보기 어렵다.
  - 5단계 `저장 및 전환`에서 최종 수량 확인만 하는 안:
    - 장점: 마지막 확인 흐름은 깔끔하다.
    - 단점: 실제 수량 입력 위치로는 너무 늦고, 앞 단계 계산과 맞지 않는다.
- 현재 추천:
  - 별도 단계를 늘리지 말고 1단계 `기준 PF 선택`에 `출하 수량`을 함께 배치한다.
  - 2단계 BOM 구성 조정에서는 `1대 기준 수량`과 `출하 수량 기준 총 필요 수량`을 같이 보여줄지 검토한다.
  - 5단계에서는 입력한 출하 수량과 그 기준으로 생성/재사용될 PA/PF, 차감될 구성품 요약을 최종 확인한다.
- 구현 전 확인 질문:
  - 요청 수량은 기본값 1대, 정수만 허용하는 것으로 충분한가?
  - 여러 대 출하 시 준비 체크리스트는 `제품 1대씩 반복 체크`가 필요한가, 아니면 `전체 수량 준비 완료 체크`가 필요한가?
  - 재고 반영은 BOM 수량에 요청 수량을 곱해서 한 번에 처리하면 되는가?


### 2. 출하 요청 작성 마지막 액션 문구/흐름 정리

- 현재 문제:
  - 5단계 `저장 및 전환`에서 `요청 저장`과 `준비 중으로 보내기`가 같이 보이는데 업무 흐름상 어색하다.
  - 사용자는 출하 요청을 작성하는 것이 목적이지, 임시로 저장한 뒤 다시 전환하는 흐름을 기대하지 않는다.
  - `준비 중으로 보내기`라는 표현도 사용자 입장에서는 시스템 상태 전환처럼 느껴지고, 실제 업무 명령인 `출하 요청`보다 덜 자연스럽다.
- 사용자 메모:
  - `요청 저장`은 없애는 방향이 좋다.
  - `준비 중으로 보내기`는 `출하 요청`으로 바꾸는 것이 자연스럽다.
- 검토 방향:
  - 작성 화면의 최종 primary 액션은 `출하 요청` 하나로 단순화한다.
  - 클릭 시 기존처럼 요청 저장 후 바로 `PREPARING` 상태로 전환하는 흐름을 유지할지 확인한다.
  - 별도 임시 저장 기능이 필요하지 않다면 `REQUESTED` 상태로만 남기는 저장 버튼은 제거한다.
  - 수정 화면에서도 같은 버튼명을 쓸지, 상태에 따라 `수정 완료`/`출하 요청 반영`처럼 나눌지 검토한다.
- 현재 추천:
  - 새 요청 작성의 마지막 버튼은 `출하 요청` 하나로 둔다.
  - 내부 동작은 기존 `저장 + 준비 중 전환`을 한 번에 처리한다.
  - `REQUESTED` 상태로만 저장하는 기능은 v1에서 제거하거나 숨긴다.
  - 단, 기존 요청 수정 화면에서는 이미 준비 중인 요청을 수정할 수 있으므로, 수정 후 액션 문구는 별도 검토가 필요하다.
- 구현 전 확인 질문:
  - 새 요청은 항상 작성 완료와 동시에 `준비 중`으로 넘어가도 되는가?
  - `REQUESTED` 상태는 앞으로도 목록에 남겨둘 필요가 있는가, 아니면 사실상 작성 중/임시 저장 용도가 없어지면 제거해도 되는가?
  - 준비 중 요청을 수정한 뒤 버튼명은 `출하 요청` 그대로가 좋은가, 아니면 `변경사항 반영`이 더 자연스러운가?


### 3. 출하 준비 체크 화면에서 요청/BOM 수정 버튼 숨김

- 현재 문제:
  - `출하 준비 중` 카드에서 들어온 준비 체크 화면에 `요청/BOM 수정` 버튼이 노출된다.
  - 준비 체크 화면은 현장에서 구성품을 체크하고 준비 완료를 처리하는 실행 화면인데, 수정 버튼이 같이 있으면 작업 목적이 섞여 보인다.
  - 사용자가 준비 화면에서 실수로 수정 흐름에 들어갈 가능성이 있고, 체크 작업 집중도가 떨어진다.
- 사용자 메모:
  - 출하 준비를 선택해서 들어온 화면에서는 `요청/BOM 수정` 버튼이 뜨지 않았으면 좋겠다.
- 검토 방향:
  - 같은 요청 상세라도 진입 경로에 따라 액션 노출을 다르게 처리한다.
  - `출하 요청` 목록/상세에서 들어온 경우에는 수정 가능 상태라면 `요청/BOM 수정`을 보여준다.
  - `출하 준비 중` 목록에서 준비 체크로 들어온 경우에는 `전체 해제`, `준비 완료` 같은 준비 작업 액션만 보여준다.
  - BOM 변경이 필요한 경우는 `출하 요청` 쪽 상세 화면으로 돌아가 수정하도록 흐름을 분리한다.
- 현재 추천:
  - `prepDetail` 화면에서는 `요청/BOM 수정` 버튼을 숨긴다.
  - 대신 필요하면 상단/하단에 작은 안내로 “구성 변경은 출하 요청 상세에서 수정” 정도만 검토한다. 화면이 복잡해지면 안내도 생략한다.
- 구현 전 확인 질문:
  - 준비 체크 화면에서 구성 변경 진입을 완전히 막아도 되는가?
  - 준비 중 요청을 수정해야 하는 실무 흐름은 반드시 `출하 요청` 목록으로 돌아가서 처리하는 것으로 통일해도 되는가?


### 4. 준비 체크 화면의 PC/모바일 역할 분리

- 현재 문제:
  - `출하 준비 중` 상세 화면이 PC에서도 체크리스트 중심으로 보인다.
  - 하지만 체크리스트는 실제 현장에서 제품을 준비하며 모바일로 체크하는 용도로 구상된 기능이다.
  - PC에서는 체크박스를 하나씩 누르는 작업보다, 어떤 제품이 몇 대 나가고 어떤 구성품/카톤/동반 출하품이 필요한지 확인하는 것이 더 중요하다.
- 사용자 메모:
  - 준비 체크 단계는 모바일과 데스크탑이 명확하게 달라야 한다.
  - 체크리스트 자체는 모바일에서 사용할 기능이다.
  - 데스크탑에는 체크리스트가 없어도 된다.
  - 데스크탑에서는 첨부 화면처럼 PA 구성품, PF 구성품, 카톤·동반 출하품 내용을 보고, 몇 개의 제품이 나가야 하는지 알 수 있으면 된다.
- 검토 방향:
  - 모바일 `준비 중` 상세는 체크박스 중심 유지:
    - 구성품 체크/해제
    - 전체 해제
    - 현장 준비 보조
  - PC `준비 중` 상세는 구성 요약 중심으로 재구성:
    - 기준/최종 PF 품명과 출하 수량
    - PA 구성품 목록
    - PF 구성품 목록
    - 카톤·동반 출하품 목록
    - 준비 완료/픽업 관련 PC 액션
  - PC에서는 체크박스를 숨기거나 제거하고, 행 단위 구성품 목록만 보여준다.
  - PC 화면에서는 “체크 상태”보다 “출하해야 할 제품 수량과 구성 내용”이 한눈에 보이도록 한다.
- 현재 추천:
  - 데스크탑 `prepDetail`은 요청 상세 화면과 비슷한 읽기/처리 화면으로 바꾼다.
  - 체크리스트 UI는 모바일 준비 화면에만 남긴다.
  - PC에는 출하 수량이 도입되면 `출하 수량 N대`, `구성품 총 필요 수량`을 강조해서 보여준다.
  - `요청/BOM 수정` 버튼은 데스크탑 준비 화면에서 숨기고, 준비 완료 같은 처리 액션만 남긴다.
- 구현 전 확인 질문:
  - PC 준비 화면에서 체크박스는 완전히 제거해도 되는가?
  - PC 준비 화면에 모바일 체크 진행률을 표시할 필요가 있는가, 아니면 구성 요약만 있으면 충분한가?
  - 출하 수량이 여러 대일 때 PC 구성품 수량은 `1대 기준 x 요청 수량 = 총 필요 수량` 형태로 보여주는 것이 맞는가?


### 5. 준비 완료 모달의 카톤·동반 출하품 추가 UX 개선

- 현재 문제:
  - 준비 완료 모달에서 검색어를 입력하면 아래 드롭다운 후보가 바뀌는 구조인데, 검색과 선택의 관계가 어색하다.
  - `동반 출하품 추가`를 누른 뒤 다시 검색하고 드롭다운에서 선택하는 흐름도 직관적이지 않다.
  - 사용자는 품목을 검색해서 찾은 뒤 바로 추가하기를 기대하는데, 현재는 검색 필드와 드롭다운 선택 필드가 분리되어 같은 일을 두 번 하는 느낌이다.
- 사용자 메모:
  - 검색하면 드롭다운 메뉴가 바뀌는 것 자체가 어색하다.
  - 동반 출하품 추가를 클릭하고, 검색해서 드롭다운 선택을 하는 것도 이상하다.
  - 그냥 검색해서 추가하는 방향이 좋아 보인다.
- 검토 방향:
  - 카톤·동반 출하품 추가는 `검색 -> 결과 row 선택/추가 -> 수량 지정` 흐름으로 단순화한다.
  - 검색 결과는 드롭다운이 아니라 모달 안의 후보 리스트 또는 command-list처럼 표시한다.
  - 후보 row에는 품목명, 품목코드, 현재 위치/재고 정보가 필요하면 함께 보여준다.
  - `추가` 버튼을 누르면 선택된 품목이 아래 추가 목록에 들어가고, 그 row에서 수량을 조정한다.
  - 이미 추가된 품목은 검색 결과에서 `추가됨`으로 표시하거나 비활성화한다.
- 현재 추천:
  - 드롭다운 선택 필드를 제거한다.
  - 검색창 아래에 검색 결과 리스트를 바로 표시하고, 각 row의 `추가` 버튼으로 동반 출하품 목록에 넣는다.
  - 추가된 목록은 별도 영역에서 `품목명/품목코드/수량/삭제`만 관리한다.
  - 준비 완료 버튼은 최소 0개 동반 출하품도 허용하되, 추가된 row 중 품목 미선택/수량 오류가 없을 때만 활성화한다.
- 구현 전 확인 질문:
  - 카톤·동반 출하품은 품목 전체에서 검색하면 되는가, 아니면 PR/PF 등 특정 공정 타입만 검색해야 하는가?
  - 검색 결과에 현재 재고/부서 위치 수량도 같이 보여줘야 하는가?
  - 동일 품목을 여러 줄로 추가할 수 없게 막고 수량만 합치는 것이 맞는가?


### 6. 카톤·동반 출하품 입력 시점을 요청 단계로 이동 검토

- 현재 문제:
  - 카톤·동반 출하품을 준비 완료 모달에서 입력하도록 되어 있다.
  - 하지만 카톤이나 동반 출하품은 준비 완료 순간에 갑자기 정하는 값이라기보다, 출하 요청 구성에 포함되어야 하는 항목에 가깝다.
  - 준비 완료 단계는 이미 준비된 내용을 확정하는 처리 단계인데, 그 시점에 출하 구성 자체를 추가하는 것은 업무 흐름상 어색하다.
- 사용자 메모:
  - 카톤이나 다른 항목을 추가하는 일은 필요하다.
  - 다만 준비 완료 단계에서 하는 것은 어색하다.
  - 요청 단계에서 하는 것이 맞는 것 같다.
- 검토 방향:
  - 카톤·동반 출하품 입력을 `출하 요청 작성/수정` 단계로 옮긴다.
  - 요청 작성 중 BOM 구성과는 분리해서 `카톤·동반 출하품` 섹션으로 다룬다.
  - 카톤·동반 출하품은 여전히 BOM 매칭/PA/PF 생성 대상에서는 제외한다.
  - 준비 중/준비 완료 화면에서는 요청 단계에서 지정된 카톤·동반 출하품을 확인만 한다.
  - 준비 완료 처리에서는 동반 출하품을 새로 입력하지 않고, 요청에 저장된 항목을 기준으로 다음 상태를 처리한다.
- 현재 추천:
  - 요청 작성 마법사에 카톤·동반 출하품 입력 영역을 추가한다.
  - 별도 단계를 늘리기보다는 2단계 `BOM 구성 조정` 또는 4단계 `요청 정보` 중 하나에 포함한다.
  - 의미상으로는 “출하 구성”이므로 2단계에 `BOM 구성품`과 구분된 `카톤·동반 출하품` 영역으로 두는 쪽이 가장 자연스럽다.
  - 준비 완료 모달에서는 카톤·동반 출하품 입력 UI를 제거하고, 요청된 동반 출하품 요약과 최종 확인만 보여준다.
- 구현 전 확인 질문:
  - 카톤·동반 출하품은 2단계 `BOM 구성 조정`에 같이 두는 것이 좋은가, 아니면 4단계 `요청 정보`에 두는 것이 좋은가?
  - 요청 단계에서 지정한 카톤·동반 출하품도 요청 수정 시 언제든 변경 가능해야 하는가?
  - 준비 완료 직전에도 예외적으로 동반 출하품을 수정할 수 있어야 하는가, 아니면 반드시 요청 수정 화면에서만 바꾸게 할 것인가?
