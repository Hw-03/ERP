## TODO 후보

### 1. 우측 상세 패널의 첫 화면 결론성 강화

- 상태: 후보
- 문제:
  - 우측 상세는 정보량은 충분하지만, 선택한 거래의 결론이 한 문장으로 먼저 잡히지 않는다.
  - 목록에서 이미 본 작업/수량/재고 영향이 상세 패널에서 다시 카드 조각으로 분산된다.
- 개선 방향:
  - 상세 패널 상단에 `결론 영역`을 둔다.
  - 예: `출하 구성 18건 · 총 18 EA 출고 · 처리 후 박스 재고 399 EA`
  - 단건, 출하/출고 묶음, 생산 BOM, 재작업이 같은 문장 문법을 쓰게 한다.
- 검증 기준:
  - 패널을 열자마자 스크롤 없이 작업 결과와 재고 영향을 읽을 수 있어야 한다.

### 2. 목록 row의 밀도와 행 높이 재조정

- 상태: 후보
- 문제:
  - 현재는 각 row가 정돈되었지만 PC 화면에서 한 번에 보이는 거래 수가 아직 적다.
  - 배지, 보조 텍스트, 여백이 많아 실제 내역 스캔 속도가 떨어질 수 있다.
- 개선 방향:
  - 대표 row와 하위 row의 높이 기준을 분리한다.
  - 대표 row는 결론을 유지하되, 하위 row는 더 촘촘한 작업 흐름 표로 만든다.
  - 보조 설명은 항상 보이지 않아도 되는 정보와 즉시 보여야 하는 정보를 다시 나눈다.
- 검증 기준:
  - 1920px 기준으로 목록 영역에서 주요 row가 충분히 보이고, 하위 전개 시 과도하게 화면을 밀어내지 않아야 한다.

### 3. 색/배지 체계 단순화

- 상태: 후보
- 문제:
  - `출하`, `출고`, `생산`, `자동차감`, `불량`, `재작업`, `메모`, `부족`, `제외`가 모두 비슷한 pill 형태로 섞여 보인다.
  - 색이 많아졌지만 의미 그룹은 아직 한눈에 분리되지 않는다.
- 개선 방향:
  - 배지를 3계층으로 정리한다.
  - 작업 종류: 출하/출고/생산/이동/조정
  - 이상 신호: 불량/부족/제외/취소/수정됨
  - 보조 정보: 메모/승인/요청
  - 작업 종류는 낮은 채도, 이상 신호는 높은 주목도, 보조 정보는 더 약하게 둔다.
- 검증 기준:
  - 사용자가 빨간색 계열만 봐도 이상 신호를 찾을 수 있고, 파란색/청록색 계열은 정상 작업 흐름으로 읽혀야 한다.

### 4. 출하/출고 구성 row의 대표 수량 의미 명확화

- 상태: 후보
- 문제:
  - `출하 구성 18건`과 `하위 18건`은 묶음 규모는 말하지만 총 수량, 품목 수, 라인 수의 차이가 즉시 분명하지 않을 수 있다.
  - 현장에서는 `몇 품목`, `몇 EA`, `어느 재고에서 빠졌는지`가 더 중요할 수 있다.
- 개선 방향:
  - 대표 row의 묶음 meta를 `18라인`보다 업무적으로 읽히는 문구로 조정한다.
  - 예: `18품목 · 총 18 EA` 또는 `18라인 · 박스 재고 -18`
  - 수량 합계가 단위 혼합 때문에 위험하면 `18라인`만 유지하고 우측 상세에서 합계를 보강한다.
- 정책 확인:
  - 서로 다른 단위가 섞일 수 있는 출하/출고 묶음에서 총량을 합산해도 되는지 확인이 필요하다.

### 5. 생산 BOM의 상위/하위 역할 문구 재검토

- 상태: 후보
- 문제:
  - 품목코드 위치는 통일됐지만, `BOM`, `자동차감`, `하위 차감`, `하위 7라인` 같은 용어가 아직 내부 시스템 말투에 가깝다.
  - 사용자는 BOM 구조보다 `완제품 입고`와 `부품 차감`을 먼저 보고 싶을 수 있다.
- 개선 방향:
  - 대표 row는 `생산 입고`, 하위 row는 `부품 차감` 중심으로 문구를 재정리한다.
  - `BOM`은 작업 종류보다 구조 힌트로 낮춰 표시한다.
- 검증 기준:
  - 생산 row를 펼쳤을 때 `무엇이 늘고 무엇이 줄었는지`가 용어 해석 없이 읽혀야 한다.

### 6. 재작업 상세 흐름의 원인/결과 구분 강화

- 상태: 후보
- 문제:
  - 대표 row는 통일됐지만, 재작업의 핵심인 `원래 불량`, `회수`, `폐기`, `새 불량`, `처리 결과`의 흐름이 더 강하게 필요할 수 있다.
  - 하위 row가 많아지면 재작업도 다시 로그 목록처럼 보일 위험이 있다.
- 개선 방향:
  - 펼침 내부를 단순 table row만이 아니라 `원인 -> 처리 -> 결과` 그룹으로 나누는 안을 검토한다.
  - 재작업만 별도 흐름 UI를 쓸지, 전체 batch detail 문법 안에서 해결할지 비교한다.
- 검증 기준:
  - 같은 품목이 반복되어도 사용자가 `불량 처리 전후`를 혼동하지 않아야 한다.

### 7. 메모 노출 방식 재정리

- 상태: 후보
- 문제:
  - `메모` chip은 존재 여부는 알려주지만, 현장 판단에 필요한 메모 내용은 우측 패널을 열어야 한다.
  - 반대로 메모 전문을 목록에 항상 노출하면 행 높이가 커진다.
- 개선 방향:
  - 목록에는 한 줄 preview를 제한적으로 보여주는 안을 검토한다.
  - 이상 거래나 불량/재작업 거래는 메모 preview 우선순위를 높인다.
  - 일반 정상 거래는 메모 존재 표시만 유지할 수 있다.
- 검증 기준:
  - 메모 때문에 목록 스캔성이 떨어지지 않으면서, 중요한 사유 메모는 클릭 전에 힌트가 보여야 한다.

### 8. 상단 KPI와 필터의 관계 정리

- 상태: 후보
- 문제:
  - 상단 KPI가 현재 목록을 설명하는 요약인지, 클릭 가능한 필터인지 역할이 애매할 수 있다.
  - `이번달 N건`과 실제 목록의 기간/검색/필터 조건이 사용자의 머릿속에서 분리될 수 있다.
- 개선 방향:
  - 현재 조건 요약 문장을 KPI 아래 또는 목록 헤더에 추가한다.
  - KPI 카드를 필터로 만들지 않을 거라면 카드처럼 과하게 보이지 않게 낮춘다.
  - 필터로 만들 거라면 선택 상태와 해제 동작을 명확히 한다.
- 검증 기준:
  - 사용자가 현재 목록이 어떤 조건의 결과인지 별도 추론 없이 알 수 있어야 한다.

### 9. 목록 헤더와 컬럼 폭 재조정

- 상태: 후보
- 문제:
  - `흐름`, `수량 · 재고`, `상태 · 처리`가 좋아졌지만 실제 긴 품명/긴 메모/다중 상태에서 균형이 무너질 수 있다.
  - 대상 열이 너무 좁아지면 품명과 코드의 통일 효과가 약해진다.
- 개선 방향:
  - PC 기준 컬럼 폭을 다시 측정한다.
  - 대상 열은 더 넓히고, 상태/처리는 핵심만 남기는 안을 검토한다.
  - 우측 상세 패널이 열린 상태와 닫힌 상태의 컬럼 폭을 별도 기준으로 본다.
- 검증 기준:
  - 긴 품명도 첫 줄에서 제품 정체가 어느 정도 보이고, 수량/상태 열이 밀리지 않아야 한다.

### 10. 펼침 상태와 선택 상태의 시각 구분

- 상태: 후보
- 문제:
  - 선택된 row, 펼쳐진 group, 하위 row 배경이 모두 연한 파란 계열로 겹치면 사용자가 현재 초점과 구조를 헷갈릴 수 있다.
- 개선 방향:
  - 선택 상태는 좌측 accent line이나 굵은 border로, 펼침 하위 영역은 배경 band로 분리한다.
  - 하위 row마다 진한 border를 반복하기보다 group 단위 구획을 강화한다.
- 검증 기준:
  - 한눈에 `현재 선택`, `펼쳐진 묶음`, `하위 구성`이 구분되어야 한다.

### 11. 출하 묶음 펼침/닫힘 인터랙션과 BOM 표시 문법 통일

- 상태: 후보
- 사용자 관찰:
  - 생산 row는 행 전체를 클릭하면 BOM이 열리고, 같은 행을 다시 클릭하면 닫힌다.
  - 출하 row는 행 전체 클릭으로 열리지만, 다시 클릭했을 때 닫히지 않는다.
  - 출하 묶음의 하위 row 표시 방식이 생산 BOM 하위 row와 다르게 보여서 같은 `작업 묶음 펼침`으로 읽히지 않는다.
- 문제:
  - 같은 화면 안에서 `펼칠 수 있는 대표 row`의 클릭 규칙이 생산과 출하에서 다르게 동작한다.
  - 사용자는 생산에서 학습한 `다시 클릭하면 닫힘` 규칙을 출하에도 기대하지만, 출하에서는 해제가 되지 않아 화면 제어감이 떨어진다.
  - 출하 하위 row가 BOM row와 다른 문법을 쓰면 `대표 작업 + 구성 라인`이라는 공통 구조가 약해진다.
- 개선 방향:
  - 생산 BOM, 출하 구성, 출고 구성, 재작업 대표 row의 펼침 토글 규칙을 하나로 통일한다.
  - 대표 row 전체 클릭 시:
    - 닫힌 상태면 펼침
    - 열린 상태면 닫힘
    - 우측 상세 선택 상태와 펼침 상태가 충돌하지 않게 기준을 정한다.
  - 출하 하위 row의 행 문법을 BOM 하위 row와 최대한 맞춘다.
    - 들여쓰기 깊이
    - 펼침/하위 아이콘 위치
    - 품명 아래 품목코드
    - 역할 badge 위치
    - 수량/재고 열 표현
    - row 배경과 구분선
  - 단, 출하는 실제 BOM이 아니므로 라벨은 `BOM`처럼 보이지 않게 `출하품` 또는 `구성품`으로 유지한다.
- 구현 전 확인:
  - `HistoryTable`에서 생산 `operation_batch_id` 그룹과 출하 `reference_no` 그룹의 row click handler가 같은 경로를 타는지 확인한다.
  - 펼침 상태 key가 생산/출하에서 같은 방식으로 생성되는지 확인한다.
  - 우측 상세 패널 선택 state와 펼침 state가 같은 클릭 이벤트에서 서로 덮어쓰는지 확인한다.
- 검증 기준:
  - 생산 대표 row를 클릭하면 열리고, 같은 row 재클릭 시 닫힌다.
  - 출하 대표 row도 같은 방식으로 열리고 닫힌다.
  - 출하 하위 row와 생산 BOM 하위 row가 같은 목록 문법으로 보인다.
  - 출하 하위 row는 BOM으로 오해되지 않고, `출하 구성`의 하위 품목으로 읽힌다.
### 12. 출하 대표 row 대상명을 `출하 구성 N건`이 아니라 실제 출하 대상명으로 표시

- 상태: 후보
- 사용자 관찰:
  - 출하 묶음 대표 row의 대상 열에 `출하 구성 18건`처럼 구조 설명만 표시된다.
  - 출하 탭에는 실제 출하 대상이 있고, 사용자는 입출고 내역에서도 그 이름을 먼저 보고 싶다.
- 확인한 출하 도메인 정보:
  - 출하 요청 응답에는 `base_pf_item_name`, `final_pf_item_name`, `custom_pf_name`, `companion_lines`가 있다.
  - 픽업 완료 로그 notes에는 `출하 픽업 완료: {final_pf_item.item_name}` 형식의 최종 출하품 힌트가 남는다.
  - 동반 품목 로그 notes에는 `출하 동반 품목: {line.item.item_name}` 형식의 보조 출하품 힌트가 남는다.
  - 출하 묶음은 `reference_no = SHIP-{request_id 앞 8자리}`로 묶이고, 거래 로그에는 `shipping_request_id`/`shipping_phase`가 백엔드 모델에 존재한다.
- 문제:
  - `출하 구성 N건`은 작업 구조를 설명하지만, 사용자가 가장 먼저 찾는 `무엇을 출하했는지`를 말하지 않는다.
  - 출하 row를 스캔할 때 매번 펼치거나 우측 상세를 열어야 실제 대상명을 확인하게 된다.
  - 생산 BOM 대표 row는 생산 대상 품명이 바로 보이는데, 출하 대표 row만 구조명으로 보이면 화면 문법이 어긋난다.
- 개선 방향:
  - 출하 대표 row의 대상명 우선순위를 정한다.
    1. `final_pf_item_name` 또는 `custom_pf_name`
    2. `base_pf_item_name`
    3. 픽업 완료 로그 notes의 `출하 픽업 완료: ...` 파싱 결과
    4. 그래도 없을 때만 `출하 구성 N건` fallback
  - 대상명 아래 meta로 구조 정보를 낮춰 표시한다.
    - 예: `COGR + COCB + FET BD 70kV, 2mA`
    - 아래: `출하 구성 18라인` 또는 `동반 17품목`
  - 동반 품목이 많을 때 대표 row 제목은 최종 출하품 1개를 중심으로 두고, 동반 품목 수는 meta로 둔다.
  - 출하 탭의 표현과 용어를 맞춘다.
    - 출하 탭에서 사용자가 보는 최종 대상명이 `custom_pf_name`인지 `final_pf_item_name`인지 확인한다.
    - 입출고 내역 대표 row도 같은 이름을 사용한다.
- 구현 전 확인:
  - 현재 입출고 내역 API 응답이 `shipping_request_id`, `shipping_phase`, 출하 요청명 필드를 프론트에 내려주는지 확인한다.
  - 내려주지 않는다면 notes 파싱만으로 해결할지, 백엔드에서 출하 요청 요약 필드를 추가할지 판단한다.
  - `reference_no`는 화면에 다시 노출하지 않고 내부 연결 key로만 사용한다.
- 검증 기준:
  - 출하 대표 row 대상 열에서 실제 출하 대상명이 먼저 보인다.
  - `출하 구성 N건`은 제목이 아니라 보조 meta로 내려간다.
  - 출하 탭의 대상명과 입출고 내역의 대표 대상명이 일치한다.
  - 동반 품목은 대표 대상을 가리지 않고, 펼침 row에서 하위 구성으로 확인된다.
### 13. 출하 펼침 내부를 `출하 대상 -> 동반 출하품 -> PA/PF 준비 흐름 -> 하위 차감`으로 읽히게 정리

- 상태: 후보
- 사용자 관찰:
  - 출하 row를 펼쳤을 때 단순히 출하 로그 여러 줄이 열리는 것처럼 보이면 어색하다.
  - 출하는 PA와 PF가 같이 얽히고, 그 아래에 다시 하위 품목/자동 차감이 붙기 때문에 생산 BOM보다 구조 판단이 더 어렵다.
  - 따라서 열림/닫힘만 맞추는 것이 아니라, 펼쳤을 때 내용의 흐름도 출하 업무 문법에 맞아야 한다.
- 핵심 판단:
  - 출하는 `상위 품목 하나 + BOM 하위품`만 있는 구조가 아니다.
  - 하나의 출하 요청 안에 다음 정보가 섞인다.
    - 실제 밖으로 나가는 최종 출하 대상: PF 또는 사용자 지정 PF명
    - 같이 나가는 품목: 동반 품목, PA/PF가 함께 출하되는 정책이 있으면 PA도 같은 출하 대상 묶음
    - 출하 준비를 위해 만들어지거나 이동한 품목: PA 생산, PF 생산, 출하 부서 이동
    - 준비 과정에서 빠지는 품목: PA/PF 하위 부품 자동 차감
  - 그래서 하위 row를 모두 같은 깊이의 `출하품`으로 보여주면, 무엇이 실제 출하품이고 무엇이 준비 과정/하위 차감인지 흐려진다.
- 개선 방향:
  - 출하 펼침 내부는 가능하면 2단계 구조로 읽히게 한다.
    1. `출하 대상` 그룹
       - 최종 PF 또는 사용자 지정 출하품
       - 실제 함께 출하되는 PA/PF/동반 품목
       - 수량은 `출하 -N EA` 중심
    2. `출하 준비` 그룹
       - PA 생산, PF 생산, 출하 부서 이동, 자동 차감
       - 수량은 `생산 +N`, `하위 차감 -N`, `이동` 중심
  - PA/PF가 함께 나가는 경우에는 둘을 서로 독립 로그처럼 흩뜨리지 않는다.
    - 대표 row 제목은 실제 출하 대상 중심으로 잡는다.
    - 펼침에서는 PA/PF를 같은 `출하 대상` 그룹 안에서 나란히 보여주거나, PF 아래 PA/하위품으로 들여쓴다.
    - 어떤 방식이 맞는지는 출하 탭의 업무 표현을 먼저 따른다.
  - 하위 품목의 하위 품목이 필요한 경우에는 단일 깊이 row로 억지 표시하지 않는다.
    - 예: `PF 출하품` 아래 `PA 구성`, 그 아래 `부품 차감`처럼 깊이를 다르게 둔다.
    - 다만 PC 목록이 너무 복잡해지면 목록에서는 2단계까지만 보이고, 우측 상세에서 3단계를 풀어도 된다.
  - 라벨 문법은 생산 BOM과 맞추되, 의미는 출하에 맞춘다.
    - 생산: `BOM`, `부품 차감`, `완제품 입고`
    - 출하: `출하품`, `동반품`, `출하 준비`, `하위 차감`
  - 펼침 순서는 시간순보다 사용자가 이해하는 업무순을 우선 검토한다.
    - 권장 순서: `실제 출하품` -> `동반 출하품` -> `PA/PF 준비` -> `하위 차감/제외/부족`
    - 단, 취소/수정/부족 같은 이상 신호는 해당 줄에서 눈에 띄게 유지한다.
- 구현 전 확인:
  - 출하 탭에서 PA/PF가 실제로 어떻게 표시되는지 확인한다.
  - `ShippingRequest`의 `final_pa_item_name`, `final_pf_item_name`, `custom_pa_name`, `custom_pf_name`, `companion_lines`, `bom_lines`, `transactions.shipping_phase`를 어떤 조합으로 받을 수 있는지 확인한다.
  - 입출고 내역 API가 출하 요청 요약을 내려주지 않는다면, notes 파싱만으로 충분한지 또는 `shipping_request_id` 기반 요약 조인이 필요한지 판단한다.
  - `PREPARE` phase와 `PICKUP` phase를 화면에서 섞어 보여줄지, 그룹으로 나눠 보여줄지 결정한다.
- 검증 기준:
  - 출하 row를 펼쳤을 때 사용자가 먼저 실제 출하 대상명을 본다.
  - PA/PF/동반품/하위 차감이 같은 레벨의 로그 나열처럼 보이지 않는다.
  - 생산 BOM과 같은 인터랙션 문법을 쓰지만, 출하 고유의 업무 흐름이 유지된다.
  - 하위 품목의 하위 품목이 있을 때도 들여쓰기와 라벨만 보고 부모/자식 관계를 알 수 있다.
  - 펼침 내부가 너무 복잡해져 목록 스캔성을 해치면, 목록은 요약하고 우측 상세에서 전체 구조를 풀어준다.

### 14. 오른쪽 3열(`흐름 / 수량·재고 / 상태·처리`) 내부 여백과 정보 덩어리 간격 조정

- 상태: 후보
- 사용자 관찰:
  - 오른쪽 3열은 정보가 많은데 너무 빽빽하게 붙어 있는 느낌이다.
  - 공간을 조금 더 둬도 될 것 같다.
- 문제:
  - `흐름`, `수량·재고`, `상태·처리`는 현장 판단에 필요한 핵심 열인데, 각 열 내부의 badge와 보조 텍스트가 세로로 가까워 한 덩어리처럼 뭉쳐 보인다.
  - 특히 수량 pill 아래 `처리 후 N EA`, 상태 열의 `요청/승인/메모`가 서로 가까워 시선이 쉬지 못한다.
  - 오른쪽 3열이 모두 좁고 빽빽하면 사용자는 행을 읽을 때 대상명보다 상태 조각에 먼저 시선을 빼앗긴다.
- 개선 방향:
  - 오른쪽 3열은 각 열 내부를 `주 정보`와 `보조 정보`로 분리해 간격을 둔다.
    - 흐름: badge 또는 흐름명 / 아래 설명 문장
    - 수량·재고: 수량 pill / 아래 처리 후 재고
    - 상태·처리: 요청자 / 승인자 / 메모·이상 상태
  - 열 사이의 가로 간격도 조금 늘려서 세 열이 하나의 덩어리처럼 붙어 보이지 않게 한다.
  - row 높이를 무작정 키우기보다, 오른쪽 3열 내부의 `gap`, `line-height`, `min-width`를 먼저 조정한다.
  - 상태 열은 정보가 많으므로 요청/승인 텍스트와 chip 사이에 명확한 세로 리듬을 둔다.
  - 하위 row에서는 대표 row보다 조금 더 촘촘하게 하되, 오른쪽 3열의 문법은 유지한다.
- 구현 전 확인:
  - `FlowSummaryCell`, `QuantityStockCell`, `PeopleStatusCell`의 내부 spacing이 공통 helper에서 조정 가능한지 확인한다.
  - 대표 row와 BOM/출하/재작업 하위 row가 같은 helper를 쓰는지 확인한다.
  - 우측 상세 패널이 열려 목록 폭이 줄어든 상태와, 상세 패널이 닫힌 넓은 상태를 둘 다 확인한다.
- 검증 기준:
  - 오른쪽 3열을 봤을 때 `흐름`, `수량`, `상태`가 서로 다른 정보 덩어리로 읽힌다.
  - 수량 pill 아래 처리 후 재고가 답답하게 붙어 보이지 않는다.
  - 요청자/승인자/메모 chip이 한 줄에 몰려 보이지 않는다.
  - 행 높이가 과하게 커져 목록 스캔성이 떨어지지 않는다.
  - 1920px PC 화면에서 오른쪽 3열이 여백은 생기지만 비어 보이지 않는다.

### 15. 품목코드를 품명 아래가 아니라 별도 `품목코드` 열로 분리 검토

- 상태: 후보
- 사용자 제안:
  - 품목코드를 품목명 밑에 두는 것보다, 따로 열을 하나 만드는 것이 나을 수 있다.
- 판단:
  - PC 화면에서는 찬성 쪽이다.
  - 품목명 아래 코드 방식은 한 셀 안에서 이름과 식별자를 같이 묶어 단순해 보이지만, 실제 현장에서는 품목코드를 기준으로 빠르게 대조하는 일이 많다.
  - 코드가 별도 열로 분리되면 세로 스캔이 쉬워지고, BOM/출하/재작업 하위 row에서도 코드 위치가 더 예측 가능해진다.
- 반대/주의점:
  - 품목코드는 보조 식별자이므로 품명보다 더 강하게 보이면 안 된다.
  - 별도 열을 만들면 대상 열 폭이 줄어 긴 품명이 잘릴 수 있다.
  - 우측 상세 패널이 열린 PC 화면에서는 목록 폭이 좁아지므로, 코드 열 폭을 과하게 잡으면 오른쪽 3열이 더 답답해질 수 있다.
  - 모바일은 이번 범위에서 제외하되, 나중에 모바일에서는 별도 열보다 품명 아래 코드가 더 적합할 가능성이 높다.
- 개선 방향:
  - PC 목록에 `품목코드` 열을 작게 추가한다.
  - 권장 컬럼 흐름:
    - `일시 / 작업 / 대상 / 품목코드 / 흐름 / 수량·재고 / 상태·처리`
  - 대상 열에는 품명과 업무 meta만 둔다.
    - 예: 품명, `BOM 1묶음`, `2라인`, `18라인`, `동반 17품목`
  - 품목코드 열에는 `7-NF-0009`, `8-VA-0018` 같은 MES 코드만 둔다.
  - 공정 코드 badge(`VA`, `PF`, `PR`)는 다시 노출하지 않는다.
  - BOM/출하/재작업 하위 row도 같은 코드 열을 사용한다.
  - 하위 row에서 실제 품목코드가 없는 구조 row라면 `-` 또는 빈 칸으로 두고, 억지로 구조 라벨을 넣지 않는다.
- 구현 전 확인:
  - `HistoryTable` header와 row grid template을 함께 바꿔야 한다.
  - `TargetSummaryBlock`, `ReferenceBatchLineRow`, `BomBatchDetail`, `ReworkBatchDetail`이 모두 같은 코드 열 문법을 쓰도록 정리한다.
  - 우측 상세 패널이 열린 상태에서 대상 열이 너무 좁아지지 않는지 확인한다.
  - 오른쪽 3열 spacing 조정과 같이 보면서 전체 컬럼 균형을 잡는다.
- 검증 기준:
  - 품목명 세로 스캔과 품목코드 세로 스캔이 각각 쉬워진다.
  - 품목코드가 품명보다 과하게 튀지 않고 보조 정보로 읽힌다.
  - BOM/출하/재작업 하위 row에서도 코드 위치가 일반 row와 같다.
  - 긴 품명이 대상 열에서 지나치게 잘리지 않는다.
  - 오른쪽 3열의 여백 개선과 충돌하지 않는다.

### 16. `BOM 1묶음`, `N라인` chip의 기본 목록 노출 필요성 재검토

- 상태: 후보
- 사용자 질문:
  - 대상 열 아래에 붙는 `BOM 1묶음`, `2라인`, `7라인`, `18라인` 표시가 꼭 있어야 하는가?
- 판단:
  - 기본 목록에서는 줄이는 쪽이 좋아 보인다.
  - 이 정보는 `무엇을 처리했는가`보다 `데이터 구조가 몇 줄인가`에 가까워서, 대상명/품목코드/수량보다 우선순위가 낮다.
  - 특히 chip 형태로 보이면 실제 상태나 중요한 badge처럼 보여 시선을 빼앗는다.
- 유지할 가치:
  - 펼침 가능한 묶음인지 알려주는 힌트로는 의미가 있다.
  - 생산 BOM이나 출하 구성의 규모를 빠르게 알 수 있게 해준다.
  - `전체 펼치기` 전에 대략 몇 줄이 열릴지 예측할 수 있다.
- 문제:
  - `BOM 1묶음`은 현장 사용자가 바로 판단해야 하는 정보가 아니다.
  - `N라인`은 로그/구조 개수이지 출하 수량이나 품목 수와 혼동될 수 있다.
  - 대상 열에 품명, 품목코드, 구조 chip이 함께 있으면 대상 식별이 흐려진다.
  - 품목코드를 별도 열로 빼면 대상 열의 보조 chip은 더 엄격하게 줄여야 한다.
- 개선 방향:
  - 기본 목록 대상 열에서는 `BOM 1묶음`, `N라인` chip을 숨기거나 더 약한 텍스트로 낮춘다.
  - 펼침 affordance는 좌측 caret 또는 작업 badge/row 상태로 해결한다.
  - 구조 개수는 필요하면 다음 위치 중 하나로 이동한다.
    - row 펼침 버튼 tooltip 또는 접근성 label
    - 우측 상세 패널의 구성 요약
    - 펼친 직후 첫 하위 header row
    - 목록 헤더/대표 row meta의 아주 약한 텍스트
  - 출하 대표 row에서는 `18라인`보다 실제 출하 대상명과 동반 품목 수가 더 중요하다.
  - 생산 대표 row에서도 `BOM 1묶음`보다 `완제품 입고 + 부품 차감` 흐름이 더 중요하다.
- 구현 전 확인:
  - 현재 `target.meta`에 들어가는 값 중 업무적으로 필요한 것과 구조 개수만 나타내는 것을 분리한다.
  - `BOM 1묶음`, `N라인` 제거가 펼침 가능 여부를 알아보기 어렵게 만들지 확인한다.
  - 품목코드 별도 열 설계와 함께 대상 열을 얼마나 단순화할지 결정한다.
- 검증 기준:
  - 대상 열에서 실제 품명/출하 대상명이 더 먼저 보인다.
  - `BOM 1묶음`, `N라인`이 사라져도 사용자가 펼칠 수 있는 row를 알아볼 수 있다.
  - 구조 개수는 필요한 곳에서만 보조적으로 확인된다.
  - 목록이 chip 과밀 상태에서 벗어나 더 차분하게 읽힌다.

### 17. PC 목록 컬럼 배치를 더 넓은 호흡의 `대상 / 품목코드 / 흐름 / 수량 / 상태` 구조로 재정렬

- 상태: 후보
- 사용자 제안:
  - 현재처럼 품목코드를 별도 열로 뺀 방향은 맞지만, 전체 배치는 더 넓게 숨 쉬는 느낌이 필요하다.
  - 대상명과 품목코드는 붙어 있되, 오른쪽 `흐름 / 수량·재고 / 상태·처리`와는 시각적으로 더 분리되어야 한다.
  - 사용자가 그린 기준 화면처럼 중간에 여백성 구간이 생기더라도, 각 열의 역할이 더 또렷하게 보이는 쪽을 원한다.
- 문제:
  - 현재 테이블은 모든 열이 균등하게 채워진 표처럼 보여서, 정보 우선순위가 강하게 드러나지 않는다.
  - `대상`은 가장 중요한 식별 정보인데, `품목코드`, `흐름`, `수량`, `상태`가 같은 강도로 이어져 보이면 시선이 오른쪽 조각 정보로 흩어진다.
  - 품목코드 열은 생겼지만 출하 대표 row에서는 `-`가 떠서, 별도 열을 만든 이유가 약해 보인다.
- 개선 방향:
  - PC 목록 컬럼을 두 덩어리로 읽히게 한다.
    - 왼쪽 작업 식별 영역: `일시 / 작업 / 대상 / 품목코드`
    - 오른쪽 판단 영역: `흐름 / 수량·재고 / 상태·처리`
  - `대상` 열은 충분히 넓게 유지하고, 품목코드 열은 작고 안정적인 고정 폭으로 둔다.
  - `대상 + 품목코드` 이후에 약간의 여백성 간격을 두어, 오른쪽 3열이 별도 판단 영역처럼 보이게 한다.
  - 헤더도 같은 리듬을 따라가게 하며, 세로 구분선을 과하게 쓰기보다 column width, padding, 빈 공간으로 분리감을 만든다.
  - 우측 상세 패널이 열린 compact 상태와 닫힌 wide 상태의 컬럼 폭을 별도로 잡는다.
- 구현 전 확인:
  - `HistoryTable`의 `gridTemplateColumns` 또는 `colgroup` 기준이 실제 `<td>` 개수와 맞는지 확인한다.
  - `ItemCodeCell`이 대표 row, 단건 row, BOM 하위 row, 출하 하위 row, 재작업 row에서 모두 같은 위치에 들어가는지 확인한다.
  - 대상 열을 넓혔을 때 오른쪽 3열이 너무 눌리지 않는지, 반대로 오른쪽 3열을 넓혔을 때 대상명이 지나치게 잘리지 않는지 확인한다.
- 검증 기준:
  - 1920px PC 화면에서 `대상 + 품목코드`가 하나의 식별 영역처럼 읽힌다.
  - `흐름 / 수량·재고 / 상태·처리`는 오른쪽 판단 영역으로 분리되어 보인다.
  - 빈 공간이 어색한 여백이 아니라 정보 그룹을 나누는 호흡으로 보인다.
  - 긴 품명, 긴 출하 대상명, 우측 상세 패널 열린 상태에서도 표가 깨지지 않는다.

### 18. 출하 대표 row의 품목코드 누락 수정

- 상태: 즉시 수정 후보
- 사용자 관찰:
  - 출하 대표 row의 대상명은 실제 출하 대상명으로 보이는데, 품목코드 열에는 `-`가 표시된다.
  - 대상명에 맞는 품목코드가 있으면 그대로 표시되는 것이 자연스럽다.
- 확인한 원인:
  - 출하 대표 row는 `getReferenceBatchPresentation`에서 notes 또는 대표 `SHIP` log를 이용해 `targetTitle`을 만든다.
  - 하지만 `targetCode`는 현재 `homogeneous` 그룹일 때만 `first.mes_code`를 쓰고, 여러 품목이 섞인 출하 묶음이면 `null`이 된다.
  - 즉 대표 대상명을 찾은 로그와 대표 품목코드를 찾는 기준이 서로 다르다.
- 개선 방향:
  - 출하 대표 대상명을 찾을 때, title뿐 아니라 해당 로그의 `mes_code`도 함께 반환한다.
  - 우선순위:
    1. `출하 픽업 완료: ...` notes를 가진 대표 `SHIP` log의 `mes_code`
    2. 동반 품목이 아닌 첫 `SHIP` log의 `mes_code`
    3. 그래도 없고 동일 품목 묶음이면 `first.mes_code`
    4. 정말 알 수 없을 때만 `-`
  - 대상명과 품목코드가 서로 다른 로그에서 엇갈리지 않게, `title`과 `code`를 같은 source log 기준으로 묶는다.
  - 동반 품목이 대표 row의 품목코드를 덮어쓰지 않게 한다.
- 구현 전 확인:
  - 현재 출하 notes 파싱 함수가 title만 반환하는지 확인하고, `{ title, code }` 형태의 helper로 바꿀지 판단한다.
  - `getReferenceBatchPresentation` 테스트에 `SHIP` 묶음에서 `targetCode`가 대표 출하품 코드로 나오는 케이스를 추가한다.
  - 동반품만 있는 비정상/예외 묶음에서는 fallback이 안전하게 동작하는지 확인한다.
- 검증 기준:
  - 출하 대표 row의 `품목코드` 열에 실제 대표 출하품 코드가 표시된다.
  - 출하 동반 품목의 코드가 대표 코드로 잘못 표시되지 않는다.
  - 출하 하위 row의 품목코드와 대표 row의 품목코드가 서로 의미 있게 대응된다.
  - 품목코드를 알 수 없는 경우에만 `-`가 표시된다.

### 19. 재작업 펼침 row를 `부품별 폐기/회수 한 쌍`으로 묶어 난잡함 줄이기

- 상태: 즉시 수정 후보
- 사용자 관찰:
  - 재작업 row를 펼쳤을 때 화면이 매우 난잡해 보인다.
  - 같은 품목이 `폐기 -199 EA`와 `회수 +1 EA`로 반복되는데, 줄이 너무 많고 빨강/초록 배경이 번갈아 보여 시선이 어지럽다.
- 문제:
  - 재작업의 실제 의미는 `불량품 200 EA 중 199 EA 폐기, 1 EA 회수` 같은 처리 결과인데, 현재 펼침은 이를 한 처리 단위로 묶지 못하고 로그 2줄로 나열한다.
  - `폐기`와 `회수`가 각각 독립 사건처럼 보여서 사용자가 품목별 결과를 머릿속에서 다시 계산해야 한다.
  - 하위 row마다 강한 배경색이 반복되어, 위험 신호가 아니라 장식 패턴처럼 보인다.
  - 일부 하위 row 안에 또 작은 펼침 버튼이 보여 계층 깊이가 불필요하게 복잡해 보인다.
- 개선 방향:
  - 재작업 펼침은 시간순 로그 나열보다 품목별 처리 결과 요약을 우선한다.
  - 같은 품목의 `폐기`와 `회수` 로그를 한 줄 또는 한 묶음으로 합친다.
    - 예: `DX3000 에어라이트 플레이트 ASS'Y` / `3-AA-0018` / `폐기 199 EA · 회수 1 EA`
  - 하위 row의 작업 badge는 `폐기`, `회수`를 각각 큰 pill로 반복하지 말고, 결과 열 안의 작은 텍스트나 compact chip으로 낮춘다.
  - 배경색은 품목별 row마다 번갈아 칠하지 말고, 재작업 group 전체에 아주 약한 band를 주고 이상 수량만 빨강/초록 텍스트로 강조한다.
  - 대표 row에는 `재작업 200 EA`, 아래에는 `22종 처리`처럼 결론을 두고, 펼침 내부는 `품목 / 코드 / 처리 결과 / 비고` 중심의 더 조용한 표로 만든다.
  - `회수 +1`, `폐기 -199`처럼 수량 부호가 중요한 정보는 유지하되, 색과 위치를 통일해 스캔 가능하게 한다.
- 구현 전 확인:
  - `ReworkBatchDetail`에서 같은 `item_id` 또는 `mes_code` 기준으로 폐기/회수 log를 묶을 수 있는지 확인한다.
  - 폐기/회수가 항상 한 쌍인지, 폐기만 있거나 회수만 있는 예외가 있는지 확인한다.
  - 현재 `defect-disassemble:*` reference 묶음에서 원본 불량 row, 폐기 row, 회수 row, 제외 row가 어떤 transaction_type/notes로 구분되는지 확인한다.
  - 품목별 묶음으로 바꿔도 재고 영향 합계와 우측 상세 패널의 거래 추적성이 깨지지 않는지 확인한다.
- 검증 기준:
  - 재작업을 펼쳤을 때 같은 품목이 `폐기`와 `회수` 두 줄로 반복되지 않고 한 처리 결과로 읽힌다.
  - 빨강/초록 배경 줄무늬가 사라지고, 실제 이상 수량만 눈에 띈다.
  - 사용자가 품목별로 `얼마나 폐기했고 얼마나 회수했는지`를 한 줄에서 판단할 수 있다.
  - 대표 row, 하위 row, 우측 상세 패널이 같은 재작업 결과를 서로 다르게 말하지 않는다.
### 20. 흐름 열의 보조 설명 제거 또는 조건부 축소

- 상태: 즉시 수정 후보
- 사용자 관찰:
  - `흐름` 열의 badge 아래 설명 문장까지 항상 보일 필요는 없어 보인다.
  - 예: `창고 -> 외부` 아래 `회사 밖으로 나감`, `불량 처리` 아래 `불량 재고 폐기`가 반복되어 정보 밀도를 높인다.
- 문제:
  - 흐름 badge 자체가 이미 핵심 경로를 말하고 있어, 아래 설명이 대부분 같은 의미를 반복한다.
  - 오른쪽 3열이 빽빽해 보이는 주요 원인 중 하나가 각 열의 보조 텍스트 상시 노출이다.
  - 설명 문장이 항상 보이면 중요한 예외 정보와 단순 해설이 같은 무게로 보인다.
- 개선 방향:
  - 기본 목록에서는 `흐름` 열의 아래 설명을 숨긴다.
  - 설명이 꼭 필요한 경우에만 조건부 노출한다.
    - 예: 혼합 흐름, 취소/수정, 부족/제외, 재작업처럼 해석이 필요한 케이스
  - 일반 출하/이동/불량 해제처럼 badge만으로 의미가 분명한 경우에는 badge 하나만 보여준다.
  - 특히 출하로 외부에 나가는 거래는 `창고 -> 외부`보다 `출하`만 표시한다.
    - 출하 row의 핵심은 물류 경로가 아니라 업무 결과이므로, 흐름 열에서는 `출하`로 충분하다.
    - 실제 출하 대상명과 품목코드는 대상/품목코드 열에서 보여주고, 재고 기준은 수량·재고 열에서 보강한다.
  - 자세한 설명은 우측 상세 패널 또는 tooltip/접근성 label로 이동한다.
- 구현 전 확인:
  - `FlowSummaryCell`의 `presentation.flow.hint` 상시 렌더링을 어디까지 제거할지 확인한다.
  - `getHistoryDisplaySubLabel`이 다른 영역에서도 쓰이는지 확인해 목록에서만 숨기고 상세 패널은 유지할지 판단한다.
  - E2E나 golden test가 흐름 설명 텍스트를 직접 기대하는지 확인한다.
- 검증 기준:
  - 목록의 `흐름` 열은 badge 중심으로 조용하게 읽힌다.
  - 일반 거래에서 반복 설명이 사라져 오른쪽 3열이 덜 답답해 보인다.
  - 출하 거래의 흐름 열은 `창고 -> 외부`가 아니라 `출하`로 읽힌다.
  - 해석이 필요한 예외 거래에서는 필요한 설명이 완전히 사라지지 않는다.

### 21. `처리 후 N EA`가 어느 재고 위치의 수량인지 명확히 표시

- 상태: 즉시 수정 후보
- 사용자 질문:
  - `수량·재고` 열의 `처리 후 N EA`가 어느 위치의 수량을 말하는지 알기 어렵다.
- 확인한 현재 의미:
  - 현재 `처리 후 N EA`는 프론트의 `getStockPresentation`에서 `transaction_log.quantity_after`를 그대로 표시한다.
  - 즉 단건 row에서는 해당 거래 로그가 영향을 준 재고 위치의 처리 후 수량이다.
  - 하지만 화면에는 그 위치명이 함께 표시되지 않아, `창고 재고`, `불량 재고`, `부서 재고`, `외부 출하 후 남은 창고 재고` 중 무엇인지 사용자가 추론해야 한다.
  - 묶음 row에서는 대표/첫 로그 기준의 `quantity_after`처럼 읽힐 수 있어 더 위험하다.
- 문제:
  - `처리 후 410 EA`만 보면 사용자가 `어디에 410개가 남았는지` 알 수 없다.
  - 흐름 열에 `창고 -> 외부`가 있더라도, 처리 후 수량이 출발지 재고인지 도착지 재고인지 명확하지 않다.
  - 불량 해제처럼 `불량 재고 -> 정상 재고` 흐름에서는 처리 후 수량이 어느 쪽 재고인지 특히 헷갈린다.
- 개선 방향:
  - 처리 후 재고 label에 기준 위치를 붙인다.
    - 예: `창고 처리 후 410 EA`
    - 예: `불량 재고 처리 후 398 EA`
    - 예: `부서 처리 후 12 EA`
  - 더 짧게 가려면 `창고 410 EA`, `불량 398 EA`처럼 label을 줄이는 안도 검토한다.
  - 출하/출고처럼 출발지 재고가 핵심인 거래는 출발 위치 기준임을 명확히 한다.
  - 불량 해제/이동처럼 출발지와 도착지가 모두 의미 있는 거래는 어느 위치의 after인지 표시 정책을 정한다.
  - 묶음 row에서는 단일 `처리 후` 값을 억지로 대표 표시하지 말고, 기준이 불명확하면 우측 상세 또는 펼침 row에서만 보여준다.
- 구현 전 확인:
  - API의 `TransactionLog`에 `warehouse_id`, `warehouse_name`, `department_id`, `department_name`, `stock_status` 등 위치/재고상태 판별 필드가 충분히 내려오는지 확인한다.
  - 현재 `getHistoryFlowLabel`, `getHistoryMovementSummary`, `getSingleLogMovement`가 이미 위치명을 해석하고 있으므로 그 helper를 재사용할 수 있는지 확인한다.
  - `quantity_after`가 항상 source 재고 기준인지, transaction_type별로 target 재고 기준이 섞이는지 백엔드 기록 규칙을 확인한다.
- 검증 기준:
  - `처리 후` 수량을 봤을 때 사용자가 기준 위치를 바로 알 수 있다.
  - 출하, 불량 처리, 불량 해제, 창고↔부서 이동에서 기준 위치가 서로 헷갈리지 않는다.
  - 기준을 알 수 없는 묶음 row에서는 오해를 부르는 대표 재고값을 숨기거나 더 명확한 요약으로 대체한다.
## 우선순위 제안

### 즉시 볼 항목

1. 흐름 열의 보조 설명 제거 또는 조건부 축소
2. `처리 후 N EA`가 어느 재고 위치의 수량인지 명확히 표시
3. 재작업 펼침 row를 `부품별 폐기/회수 한 쌍`으로 묶어 난잡함 줄이기
4. 출하 대표 row의 품목코드 누락 수정
5. PC 목록 컬럼 배치를 더 넓은 호흡의 `대상 / 품목코드 / 흐름 / 수량 / 상태` 구조로 재정렬
6. 출하 묶음 대표 대상명 표시
7. 출하 펼침 내부 흐름 정리
8. 출하 묶음 펼침/닫힘 인터랙션과 BOM 표시 문법 통일
9. 품목코드 별도 열 분리 검토
10. `BOM 1묶음`, `N라인` chip 기본 노출 축소
11. 오른쪽 3열 내부 여백과 정보 덩어리 간격 조정
12. 우측 상세 패널 첫 화면 결론성 강화
13. 색/배지 체계 단순화
14. 펼침 상태와 선택 상태의 시각 구분
15. 목록 row 밀도와 행 높이 재조정

### 구조 검토 후 볼 항목

1. 재작업 상세 흐름의 원인/결과 구분 강화
2. 생산 BOM의 상위/하위 역할 문구 재검토
3. 출하/출고 구성 row의 대표 수량 의미 명확화
4. 목록 헤더와 컬럼 폭 재조정

### 정책 확인 필요

1. 출하/출고 묶음에서 단위가 다른 수량을 합산 표시해도 되는지
2. KPI 카드를 클릭 필터로 만들지, 순수 요약으로 둘지
3. 메모 preview를 어떤 거래 유형에서 기본 노출할지

## 다음 진행 방식

다음 개선은 한 번에 모두 구현하지 말고, PC 화면 기준으로 한 묶음씩 진행한다.

추천 순서:

1. 출하 대표 대상명과 펼침 내부 흐름 설계
2. 품목코드 별도 열 분리, 구조 chip 축소, 오른쪽 3열 간격을 함께 설계
3. 우측 상세 패널 결론 영역 설계
4. 목록 선택/펼침/하위 row 시각 구분 정리
5. 색/배지 의미 체계 정리
6. row 밀도와 컬럼 폭 조정
7. 재작업/BOM 상세 흐름 고도화

각 단계는 구현 전에 현재 화면 스크린샷 기준으로 `문제 -> 개선안 -> 검증 기준`을 먼저 확정한다.
