### 5. `N종 처리`를 품명 뒤 보조 정보로 한 줄 배치

- 확정 정책: `N종 처리`는 품명 뒤에 탭 한 칸 정도의 고정 간격을 두고 같은 줄의 보조 정보로 표시한다. 실제 탭 문자는 사용하지 않으며, 품명이 길면 품명을 우선해 줄임표 처리하고 보조 정보는 의미가 유지되게 한다.

- 수용 기준: `4종 처리` 같은 처리 건수는 대상 품명 바로 뒤에 한 줄로 보이며, 아래 메타 줄에는 중복 표시되지 않는다. 긴 품명은 기존처럼 읽을 수 있고, 처리 건수·품목코드·흐름·수량 열이 겹치지 않는다.
- 브라우저 확인 시나리오: `4종 처리`가 있는 일반 행과 긴 품명의 여러 처리 행을 확인한다. 1920px 및 일반 데스크톱 폭에서 품명 뒤 간격, 줄임표, 메타 줄 중복 여부, 행 높이를 비교한다.
- 우선순위: 중간

### 6. 생산·분해 수량 배지를 작업 의미로 구분
- 상태: 구현 대기
- 화면/영역: PC 입출고 내역 > 목록 > 수량 열의 생산·분해 수량 배지
- 증상: 생산 완료 행은 `완제품 +3 EA`, 분해 행은 `완제품 -2 EA`로 표시된다. AF가 완제품을 뜻하는 상황에서 이 배지는 재고 품목 분류만 반복하고, 실제 수행한 작업 의미를 충분히 드러내지 못한다.
- 사용자 불편: 같은 `완제품` 표현만으로는 해당 증가가 생산 완료인지, 해당 감소가 분해인지 목록을 훑을 때 바로 구분하기 어렵다.
- 확정 정책: 생산 완료의 최상위 품목 증가는 `생산 +N 단위`로, 분해 대상 완제품 감소는 `분해 -N 단위`로 표시한다. 함께 표시되는 `부품` 증감 배지와 수량·색상 규칙은 유지한다.
- 구현 방향: `historyBatchInterpreter.ts`의 생산/분해 최상위 품목 요약 라벨을 작업 유형별로 분기한다. `getHistoryMovementSummary`의 생산·분해 golden test와 목록 수량 배지 테스트의 기대 문구를 함께 갱신한다. 상세 실제 영향의 역할명 `완제품` 등 별도 정보 구조는 이 변경 범위에 포함하지 않는다.
- 수용 기준: 생산 행은 `생산 +3 EA`처럼, 분해 행은 `분해 -2 EA`처럼 표시된다. 부품 증감 배지, 양수/음수 색상, 단위 생략·혼합 단위 규칙과 다른 입출고 유형의 문구는 기존대로 유지된다.
- 브라우저 확인 시나리오: 생산 완료 행과 분해 행을 각각 확인해 최상위 수량 배지가 `생산`·`분해`로 구분되는지, 같은 행의 부품 수량 및 일반 재고 이동·품목 전환 수량 배지가 바뀌지 않는지 확인한다.
- 우선순위: 높음

### 7. 실제 영향 클릭 행의 전역 버튼 테두리 제거

### 8. 다음 페이지를 대표 행 100개 단위로 불러오기
- 상태: 구현 대기
- 화면/영역: PC 입출고 내역 > 목록 하단 > `다음 100건 불러오기`
- 증상: 현재 페이지 크기는 원본 거래 로그 100건이다. 화면은 원본 로그를 단건·작업 묶음·요청 묶음·불량 lifecycle 묶음의 대표 행으로 다시 구성하므로, 한 대표 행에 속한 하위 로그가 많으면 버튼을 눌러도 대표 행은 100개보다 적게 추가된다.
- 사용자 불편: 버튼의 `100건`을 클릭했을 때 펼쳐지는 하위 행을 포함한 100줄이 아니라, 목록에서 바로 보이는 부모(대표) 행 100개가 추가될 것으로 기대한다. 현재는 묶음 크기에 따라 추가되는 대표 행 수가 달라져 여러 번 눌러야 한다.
- 확정 정책: 이력 페이지의 `100건`은 화면의 최상위 대표 행 100개를 의미한다. 단건, `operation_batch_id` 묶음, 요청번호 묶음, 불량 lifecycle 묶음은 각각 한 개의 대표 행으로 센다. 대표 행을 펼쳐 보여 주는 하위 로그는 페이지 건수에 포함하지 않는다.
- 구현 방향: `useHistoryData`의 원본 로그 페이지네이션과 `buildGroups` 결과를 분리해, 최초 조회와 `다음 100건` 모두 대표 행 기준으로 100개를 채울 때까지 필요한 원본 로그 페이지를 내부적으로 추가 조회·버퍼링한다. 화면에는 대표 행 경계에서만 다음 페이지를 공개하고, 한 묶음이 원본 페이지 경계를 넘더라도 중복·누락·부분 묶음 없이 하나의 대표 행으로 유지한다. 버튼 문구는 대표 행 기준을 오해하지 않도록 필요 시 `다음 대표 100건 불러오기`로 함께 조정한다.
- 수용 기준: 첫 진입과 버튼 클릭 뒤 각각 최대 100개의 새 대표 행이 보인다(마지막 페이지는 남은 수만 표시). 대표 행을 모두 펼쳐도 하위 로그 수는 100개 산정에 포함되지 않으며, 원본 로그가 많은 묶음·요청·불량 lifecycle도 중복 또는 분리되지 않는다. 기존 필터, 검색, 날짜 조건, 추가 조회 실패 재시도와 정렬 순서는 유지된다.
- 브라우저 확인 시나리오: 하위 로그가 많은 생산/분해 묶음과 단건이 섞인 조건에서 초기 목록 및 `다음 100건` 클릭 후의 최상위 대표 행 수를 각각 센다. 대표 행을 여러 개 펼쳐도 100개 산정이 바뀌지 않는지, 원본 페이지 경계의 묶음이 한 대표 행으로 유지되는지, 마지막 페이지·필터 변경·추가 조회 실패 후 재시도를 확인한다.
- 우선순위: 중간

### 9. 목록의 흐름 열을 우측 상세 카드로 이동
- 상태: 구현 대기
- 화면/영역: PC 입출고 내역 > 목록의 `흐름` 열 및 단건·묶음 우측 상세 카드
- 증상: 목록의 `흐름` 열은 작업 종류와 별도로 재고 위치 또는 이동 경로를 표시하지만, 생산·수량 조정처럼 `조립` 등 반복되는 값이 많아 넓은 목록 열을 계속 차지한다. 위치·경로의 세부 맥락은 행을 선택한 뒤 읽는 편이 적합하다.
- 사용자 불편: 목록을 빠르게 훑을 때 작업·대상·수량보다 우선순위가 낮은 정보가 한 열을 차지해 대상명과 수량 영역의 가용 폭을 줄인다. `흐름`이라는 제목도 위치인지 이동 경로인지 즉시 알기 어렵다.
- 확정 정책: 목록에서는 `흐름` 열을 제거한다. 재고 위치와 이동 경로는 선택한 거래의 우측 상세 카드에서만 보여 주며, 단건·작업 묶음·요청 묶음 모두 같은 의미의 위치/이동 경로 정보를 확인할 수 있어야 한다.
- 구현 방향: `historyTableHelpers.tsx`의 대표·하위 행에서 흐름 헤더와 셀을 제거하고 남은 열의 폭을 대상·수량 중심으로 재배분한다. 현재 단건 상세의 흐름 표시를 기준으로, 작업 묶음과 요청 묶음 우측 상세에도 출발 위치·도착 위치 또는 단일 위치를 명확한 `위치/이동 경로` 레이블로 추가·통일한다. 거래 유형상 위치가 없거나 추론 불가능하면 상세에서도 빈 placeholder를 만들지 않는다.
- 수용 기준: 목록 헤더와 대표·하위 행 어디에도 `흐름` 열이 없고, 대상명·수량 열은 늘어난 가용 폭을 사용한다. 목록에서 행을 선택하면 우측 상세에서 해당 거래의 단일 위치 또는 `출발 → 도착` 경로를 확인할 수 있다. 생산·수량 조정처럼 위치가 반복되는 거래, 창고↔부서 이동, 불량 전환, 출하·품목 전환 묶음의 경로 표시는 모두 일관된다.
- 브라우저 확인 시나리오: 단건 생산, 창고↔부서 이동, 불량 전환, 작업 묶음, 요청 묶음을 각각 선택한다. 목록 열 제거 뒤 대상명·수량·상태 열의 정렬과 가용 폭을 확인하고, 각 우측 상세 카드에서 위치/이동 경로가 누락·중복·오해 없이 표시되는지 확인한다.
- 우선순위: 중간

### 10. 우측 상세의 요청자·요청 시각을 최상단 메타로 이동

- 수용 기준: 상세 카드를 열면 제목 다음에 요청자와 요청 시각이 즉시 보인다. 실제 영향 품목 수와 무관하게 요청자 정보의 세로 위치는 변하지 않으며, 요청자 없는 시스템 거래의 기존 대체 표현·승인자·취소 기능은 회귀하지 않는다.
- 브라우저 확인 시나리오: 실제 영향이 많은 생산/수량조정 묶음, 단건 거래, 승인자가 있는 거래, 취소 가능한 거래를 각각 연다. 제목 다음 메타의 위치와 한 번만 표시되는지, 실제 영향·승인·취소 UI가 기존대로 동작하는지 확인한다.
- 우선순위: 중간
