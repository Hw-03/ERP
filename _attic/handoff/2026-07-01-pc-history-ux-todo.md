# PC 입출고 내역 UX 투두리스트

작성일: 2026-07-01
상태: 수집 중

## 원칙

- 이 파일은 `입출고 내역` PC 화면의 후속 UX 개선사항을 구현 전까지 모아두는 투두리스트다.
- 사용자가 명시적으로 구현을 지시하기 전까지 여기 적힌 항목은 구현하지 않는다.
- 모바일은 이번 목록의 기본 범위에서 제외한다.
- 핵심 기준은 예쁜 화면보다 `거래 단위`, `원인`, `수량 영향`, `이상 여부`가 한눈에 보이는지다.
- 구현 시 실제 렌더 경로인 `frontend/app/mes/_components/DesktopHistoryView.tsx`와 `_history_sections/`를 먼저 확인한다.
- 생산/BOM, 출고 구성, 불량 처리, 이동/조정이 같은 목록 문법 안에서 서로 헷갈리지 않게 한다.

## 수집 항목

### 1. 출고 작업 자세히 보기를 생산 BOM처럼 구성 전개로 표시

- 화면: PC `입출고 내역` 목록에서 출고 묶음 row를 펼쳤을 때의 하위 상세 영역
- 사용자 메모:
  - 생산은 자세히 보기 하면 BOM처럼 하위 라인이 전개된다.
  - 출고 작업도 그렇게 보였으면 좋겠다.
- 현재 문제:
  - 생산은 상위 제품과 하위 BOM 라인의 관계가 한 작업 단위로 읽힌다.
  - 출고 묶음은 여러 품목이 같이 나간 작업이어도 `묶음 N건`, `단품`, `적용`처럼 보여 구성 관계가 약하다.
  - 현장 사용자는 “무엇이 같이 출고됐는지”, “어떤 품목이 본체이고 어떤 품목이 딸려 나갔는지”를 빠르게 보기 어렵다.
- 개선 방향:
  - 출고 배치가 여러 라인을 가진 묶음이면 생산 BOM과 같은 접기/펼치기 문법을 적용한다.
  - 실제 BOM 기준 출고인지, 여러 품목 동시 출고인지가 섞일 수 있으므로 라벨은 무조건 `BOM`보다 `출고 구성`을 우선 검토한다.
  - 하위 라인은 생산 BOM처럼 들여쓰기하여 품명, 품목코드, 역할, 차감 수량, 부족/제외 상태를 보여준다.
  - 단일 품목 출고는 과하게 전개하지 않고 기존 단건 표현을 유지한다.
  - 생산/재작업의 `상위 + / 하위 -` 의미는 유지하고, 출고는 `출고 -N` 중심으로 별도 의미를 부여한다.
- 구현 전 확인:
  - 출고 배치에서 상위/하위 관계를 판별할 수 있는 필드가 충분한지 확인한다.
  - `source_kind`, `origin`, `has_children`, `operation_batch_id`, `io_bundles/io_lines` 구조를 실제 API 응답으로 확인한다.
  - `BomBatchDetail.tsx`를 그대로 확장할지, `BatchStructureDetail` 같은 더 일반적인 이름으로 분리할지 판단한다.
- 검증 기준:
  - 생산 BOM 전개는 기존처럼 보인다.
  - 다중 출고 묶음은 `출고 구성`으로 읽히고, 하위 출고 품목들이 한 작업 단위로 보인다.
  - 단일 출고 row는 불필요하게 복잡해지지 않는다.
  - 부족/제외/메모/참조 상태가 하위 라인에서도 유지된다.

### 2. BOM 전개 행의 품목코드 위치와 공정 코드 노출 정리

- 화면: PC `입출고 내역` 목록에서 생산/BOM row를 펼쳤을 때의 상위/하위 라인
- 사용자 메모:
  - 일반 상위 row는 품목코드가 품명 아래에 있는데, BOM 전개 row는 품목코드가 별도 열에 표시되어 어색하다.
  - BOM 상위/하위도 품목코드 위치를 일반 row와 통일하는 것이 좋아 보인다.
  - 이동건 등 일반 row에서 품목코드 옆에 공정 코드 badge가 붙는데, 실제 내역 확인에는 필요 없어 보인다.
- 현재 문제:
  - 일반 거래 row와 BOM 전개 row가 같은 품목 정보를 서로 다른 위치에 보여준다.
  - 사용자는 품명 아래에서 코드를 찾다가 BOM 행에서는 오른쪽 별도 열을 봐야 해서 시선 흐름이 끊긴다.
  - 공정 코드 badge(`VA`, `PF` 등)는 내역 판단에 비해 정보 가치가 낮고, 품목코드 옆에 붙어 목록을 더 복잡하게 만든다.
- 개선 방향:
  - BOM 전개 행도 일반 row처럼 품명 아래에 품목코드를 배치한다.
  - 기존 품목코드 전용 열은 제거하거나, 해당 공간을 역할/흐름/수량 정보가 더 잘 읽히도록 재배치한다.
  - 일반 이동/출고/입고 row의 품목코드 옆 공정 코드 badge는 숨기는 방향을 우선 검토한다.
  - 공정 코드가 정말 필요한 경우에는 상세 패널이나 필터/고급 정보에 두고, 기본 목록에서는 품명과 품목코드만 남긴다.
- 구현 전 확인:
  - `TargetSummaryBlock`의 일반 row 메타 구성과 `BomBatchDetail.tsx`의 BOM row 칼럼 구조를 비교한다.
  - 공정 코드 badge가 어디서 생성되는지 확인하고, 숨겼을 때 테스트/필터/식별 흐름에 영향이 없는지 본다.
  - BOM row에서 품목코드를 품명 아래로 내렸을 때 열 너비와 줄바꿈이 PC 화면에서 안정적인지 확인한다.
- 검증 기준:
  - 일반 row와 BOM 전개 row 모두 품명 아래에서 품목코드를 찾을 수 있다.
  - 기본 목록에서 불필요한 공정 코드 badge가 사라져도 품목 식별이 가능하다.
  - 생산, 출고, 이동, 조정 row의 열 정렬이 더 일관되게 보인다.
### 3. `참조` 배지 의미와 노출 방식 정리

- 화면: PC `입출고 내역` 목록의 상태/처리 칸과 우측 상세 패널
- 사용자 메모:
  - `메모`는 의미를 알겠는데 `참조`는 무엇인지 바로 이해되지 않는다.
- 현재 의미:
  - `참조`는 `batch.reference_no` 또는 `log.reference_no` 값이 있을 때 표시되는 배지다.
  - 이 값은 사용자가 쓴 메모가 아니라, 요청번호/배치번호/출하 연계번호/레거시 묶음 기준 등 거래를 다른 업무 객체와 연결하기 위한 참조번호다.
- 현재 문제:
  - `참조`라는 한 단어만으로는 사용자가 무엇을 참조한다는 뜻인지 알기 어렵다.
  - hover/title을 보지 않으면 실제 참조번호 값도 바로 보이지 않는다.
  - 모든 목록 row에 배지로 노출할 만큼 중요한 정보인지 불명확하다.
- 개선 방향:
  - 작업자용 프론트 화면에서는 `참조` 배지와 참조번호 값을 표시하지 않는다.
  - `reference_no`는 그룹핑/연계/검색 같은 내부 처리에는 유지하되, 기본 UI 문구와 상세 패널에서는 숨긴다.
  - 여러 row를 하나의 작업으로 묶는 데 필요한 내부 reference는 화면 문법에는 쓰되, 사용자에게 내부 키처럼 보이지 않게 한다.
  - 출하/요청/재작업처럼 별도 업무번호가 정말 필요한 화면이 생기면, `reference_no`가 아니라 해당 도메인 전용 필드와 라벨로 새로 설계한다.
- 구현 전 확인:
  - `reference_no`가 생성되는 경로를 유형별로 확인한다: stock request, io batch, shipping, defect disassemble, legacy group.
  - 목록 그룹핑에서 `reference_no`가 필요한 케이스와 단순 표시용 케이스를 분리한다.
  - 검색 placeholder의 `참조번호` 문구와 실제 UI 노출 정책을 맞춘다.
- 검증 기준:
  - 사용자가 목록에서 `참조`라는 모호한 배지를 보고 멈칫하지 않는다.
  - 작업자 화면에는 참조번호 값이 노출되지 않는다.
  - 그룹핑/검색 기능은 기존처럼 동작한다.
### 4. 재작업 행의 표현 통일성 정리

- 화면: PC `입출고 내역` 목록에서 재작업/새 불량/불량 처리 관련 row와 우측 상세 패널
- 사용자 메모:
  - 재작업은 뭔가 통일성이 없다.
- 현재 문제:
  - 같은 사건 흐름 안에서 `재작업`, `새 불량`, `불량 처리`가 각각 다른 행 문법으로 보인다.
  - 재작업 row는 작업명처럼 보이고, 새 불량 row는 결과 상태처럼 보이며, 두 행의 관계가 한눈에 연결되지 않는다.
  - 목록에서 같은 품목이 재작업 처리 전후로 반복되는데, 원인/결과/재고 영향의 위계가 약하다.
  - 우측 상세 패널도 선택한 단건 기준으로는 정보를 보여주지만, 재작업 묶음 전체 흐름을 설명하는 힘이 부족하다.
- 개선 방향:
  - 재작업을 `원본 불량 -> 분해/회수 -> 처리 결과` 흐름으로 읽히게 정리한다.
  - 재작업 관련 row는 생산/BOM 전개와 같은 구조 문법을 재사용하되, 라벨은 `재작업 구성` 또는 `분해/회수`처럼 업무 의미에 맞춘다.
  - 단건 `새 불량` row가 재작업 묶음의 일부라면 목록에서 독립 사건처럼 튀지 않게 묶음 표현을 우선 검토한다.
  - 재작업 상세 패널은 `대상`, `원인`, `처리`, `재고 영향` 순서로 재배치하는 방향을 검토한다.
- 구현 전 확인:
  - 재작업이 `operation_batch_id`로 묶이는 케이스와 `reference_no=defect-disassemble:*`로 묶이는 레거시 케이스를 모두 확인한다.
  - `DISASSEMBLE`, `MARK_DEFECTIVE`, `DEFECT_SCRAP`이 같은 화면 문법에서 어떻게 구분되어야 하는지 실제 데이터로 본다.
  - 재작업 row와 새 불량 row가 동시에 보이는 경우, 어느 행을 대표 row로 삼을지 정한다.
- 검증 기준:
  - 사용자가 재작업 목록을 보고 “무엇을 재작업했고, 무엇이 나왔고, 재고가 어떻게 바뀌었는지”를 한 흐름으로 읽을 수 있다.
  - 재작업/새 불량/불량 처리의 색, 라벨, 수량 표현이 서로 충돌하지 않는다.
  - 기존 불량 처리 단건은 재작업 묶음이 아닐 때 독립 거래로 명확히 보인다.
### 5. 메인 워크트리 날짜 필터 변경 사항 확인 및 반영

- 화면: PC `입출고 내역` 상단 기간 필터, 달력 필터, 선택 날짜 칩, 목록/통계 조회 범위
- 사용자 메모:
  - 메인 워크트리 쪽에서 입출고 내역의 날짜 필터 관련 수정이 있었다.
  - 이 분리 worktree에서 다음 구현을 진행하기 전에 그 수정도 확인해서 반영해야 한다.
- 현재 위험:
  - `history-ux-redesign` worktree가 메인 워크트리와 분리되어 있어, 메인에 반영된 날짜 필터 수정이 누락될 수 있다.
  - 날짜 필터는 목록 조회, KPI/요약 조회, 달력 선택, 선택 날짜 해제, 필터/달력 패널 토글과 동시에 얽혀 있어 누락 시 사용자가 보는 건수와 실제 목록이 어긋날 수 있다.
  - UI 개선 작업 중 `DesktopHistoryView.tsx`, `HistoryFilterBar.tsx`, `HistoryCalendarPanel.tsx`, `HistoryCalendarStrip.tsx`, `historyQuery.ts`, `historyFormat.ts`를 건드리면 메인 수정과 충돌하거나 퇴행할 수 있다.
- 개선 방향:
  - 구현 전 메인 워크트리(`C:\ERP`)와 현재 worktree(`C:\ERP\.worktrees\history-ux-redesign`)의 날짜 필터 관련 파일을 비교한다.
  - 메인 쪽 변경이 사용자 승인/운영 반영된 수정이면 이 worktree에도 먼저 이식한다.
  - 기간 칩(`오늘/이번주/이번달/전체`), 달력 날짜 선택, 선택 날짜 해제, 목록 조회 `dateFrom/dateTo`, 요약 조회 범위가 같은 기준으로 움직이게 맞춘다.
  - 필터 패널과 달력 패널의 상호 열림/닫힘 동작도 메인 기준 변경과 충돌하지 않게 확인한다.
- 구현 전 확인 파일:
  - `frontend/app/mes/_components/DesktopHistoryView.tsx`
  - `frontend/app/mes/_components/_history_sections/HistoryFilterBar.tsx`
  - `frontend/app/mes/_components/_history_sections/HistoryCalendarPanel.tsx`
  - `frontend/app/mes/_components/_history_sections/HistoryCalendarStrip.tsx`
  - `frontend/app/mes/_components/_history_sections/historyQuery.ts`
  - `frontend/app/mes/_components/_history_sections/historyFormat.ts`
- 검증 기준:
  - 메인 워크트리에 있는 날짜 필터 수정이 누락되지 않는다.
  - 기간 필터와 달력 선택이 동시에 살아서 목록 범위를 혼란스럽게 만들지 않는다.
  - 목록 건수, 상단 요약, 달력 선택 상태가 같은 날짜 범위를 설명한다.
  - `io-history-labels.spec.ts` 또는 별도 e2e로 날짜 필터/달력 기본 동작을 확인한다.
## 구현 전 확인

- 실제 PC 화면에서 문제 row를 다시 선택해 스크린샷 기준으로 확인한다.
- 관련 테스트는 우선 `historyPresentation.test.ts`, `io-history-labels.spec.ts`에 최소 회귀 케이스를 추가한다.
- 기존 미커밋 변경과 섞어 커밋하지 않는다.
