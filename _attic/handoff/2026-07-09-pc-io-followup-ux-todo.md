## 자율 점검 기준
- 같은 정보는 같은 모양이어야 한다. 품목, 수량, 부족, 생성 예정, 재사용, 수정 가능 범위가 화면마다 다른 문법으로 보이면 TODO로 잡는다.
- 선택된 것은 빈 검색 결과처럼 보이지 않아야 한다. 선택 이후에는 검색 패널이 아니라 선택 요약과 변경 액션이 보여야 한다.
- 사용자가 고르지 않아도 되는 판단은 시스템이 해야 한다. 사양/구성 전환처럼 데이터로 알 수 있는 판단을 사용자가 고르게 하지 않는다.
- 핵심 입력은 보조 필드처럼 숨기지 않는다. 전환 수량, 출하 수량, 부족 수량, 수정 대상은 화면 위계 안에서 크게 보여야 한다.
- 요약은 펼쳤을 때 검산 가능해야 한다. `N개 변경`, `N개 부족`, `PA/PF 재사용` 같은 결론은 실제 품목 목록과 바로 연결되어야 한다.
- 출하, 입출고, 품목 전환의 선택 문법을 맞춘다. 검색 결과 카드, 선택 요약, 변경 버튼, 빈 상태 문구가 화면마다 다르면 TODO로 잡는다.
- 빈 큰 영역, 중복 카드, 막힌 이유 없는 비활성 버튼을 줄인다. 아직 선택 불가한 영역은 큰 빈 결과판보다 다음 조건을 알려주는 보조 영역으로 낮춘다.
- 부족, 제외, 경고, 준비 중 같은 상태는 목록 안에서도 보여야 한다. 별도 경고 박스만 있고 실제 라인에서 상태가 안 보이면 TODO로 잡는다.
- 수정 동선은 한 군데로 정한다. 수량처럼 연쇄 계산을 일으키는 값은 어디서 수정하는지 화면마다 같은 규칙으로 보여야 한다.
- 입출고 내역과 작업 화면은 같은 거래를 같은 말로 설명해야 한다. 요청/준비/전환에서 본 용어가 내역 탭에서 다른 말로 바뀌면 TODO로 잡는다.

## 확정 결정 요약
- 품목 전환은 이미 자동 판정 흐름으로 바뀌었지만, 선택 전후 패널 문법은 추가 정리가 필요하다.
- 출하 후속 단계에서 출하 수량은 직접 수정하지 않고 `수량 변경`으로 1단계에 돌아가는 정책을 유지한다.
- 준비 작업 부족 문구는 `N개 부족`처럼 짧게 가되, 부족 대상이 출하품인지 구성품인지 동반 출하품인지는 같이 드러내야 한다.
- 기존 12개 완료 TODO와 이번 후속 TODO는 섞지 않는다. 이 문서는 후속 개선 후보만 다룬다.

## 공통

### 1. 형제 화면의 검색/선택 문법 통일
- 화면/영역: PC 입출고 탭 전체 > 품목 검색, 선택 요약, 변경 액션
- 증상: 출하 요청, 불량 격리, 품목 전환, 입출고 작업에서 품목을 검색하고 선택하는 방식이 조금씩 다르다. 어떤 화면은 선택 후 요약 카드와 변경 버튼이 보이고, 어떤 화면은 선택된 뒤에도 검색 결과 영역처럼 남는다.
- 사용자 불편: 같은 제품 안에서 품목을 고르는 방법이 매번 새로워 보이면, 사용자는 화면의 목적보다 조작 규칙을 먼저 배워야 한다.
- 확정 정책: 품목 검색/선택 UI는 `검색 입력 -> 결과 카드 선택 -> 선택 요약 -> 변경/해제 액션` 문법을 기본으로 통일한다.
- 구현 방향: 각 화면의 품목 선택 영역을 비교해 선택 전, 선택 후, 검색 결과 없음, 변경 액션의 공통 규칙을 만든다. 화면별 특수 조건은 보조 설명으로만 둔다.
- 수용 기준: 사용자가 품목을 선택한 뒤에는 선택된 품목이 결과 없음 상태처럼 보이지 않는다. 변경이 필요한 경우 같은 위치에서 `변경` 또는 `다시 선택` 액션을 찾을 수 있다.
- 브라우저 확인 시나리오: 품목 전환, 출하 요청 1단계, 불량 격리 품목 선택 흐름을 각각 열고 선택 전/후 문법이 같은지 비교한다.
- 우선순위: 높음

### 2. 비활성 버튼과 하단 상태 문구가 막힌 조건을 직접 말하게 하기
- 화면/영역: PC 입출고 탭 전체 > 단계형 작업 하단 action bar, 다음 버튼, 제출 버튼
- 증상: 일부 단계에서 하단 문구가 `품목·수량 선택`처럼 넓은 범주만 말하거나, 버튼은 비활성인데 정확히 무엇이 부족한지 가까이에 보이지 않는다.
- 사용자 불편: 사용자는 다음으로 못 가는 이유를 찾기 위해 위쪽 패널을 다시 훑어야 한다. 특히 단계형 화면에서는 하단이 결론이어야 하는데, 현재 막힌 조건이 명확하지 않으면 흐름이 끊긴다.
- 확정 정책: 비활성 버튼 근처에는 현재 부족한 조건을 한 문장으로 표시한다. 조건이 여러 개면 가장 먼저 해결해야 할 조건을 우선 보여준다.
- 구현 방향: 각 wizard/action bar의 status text를 `소스 품목을 선택하세요`, `대상 품목을 선택하세요`, `출하 정보를 입력하세요`, `부족 라인을 해결하세요`처럼 구체화한다.
- 수용 기준: 사용자가 화면 하단만 봐도 다음 행동을 알 수 있다. 비활성 버튼만 있고 이유가 없는 상태가 남지 않는다.
- 브라우저 확인 시나리오: 품목 전환 미선택 상태, 출하 요청 단계별 미입력 상태, 입출고 제출 불가 상태에서 하단 문구와 버튼 상태를 확인한다.
- 우선순위: 높음

## 품목 전환

### 3. 소스 선택 후 왼쪽 패널이 `검색 결과가 없습니다`로 남는 문제
- 화면/영역: PC 입출고 > 품목 전환 > 소스·대상 선택 단계
- 증상: 소스 품목을 선택하면 상단 요약에는 선택 품목이 보이지만, 왼쪽 소스 패널 본문은 큰 빈 영역과 `검색 결과가 없습니다.` 상태로 남을 수 있다.
- 사용자 불편: 사용자는 품목을 이미 선택했는데도 검색 실패처럼 보이는 화면을 보게 된다. 선택이 완료된 것인지, 검색 조건이 꼬인 것인지 헷갈린다.
- 확정 정책: 선택된 품목은 빈 검색 결과가 아니라 선택 요약 카드로 보여준다.
- 구현 방향: 소스 선택 후 왼쪽 패널을 `선택된 소스 품목` 카드로 전환하고, 품명/코드/재고/전환 가능 여부와 `변경` 액션을 보여준다. 검색 결과 목록은 변경 액션을 눌렀을 때 다시 열리게 한다.
- 수용 기준: 소스 품목 선택 후 화면 어디에도 `검색 결과가 없습니다.`가 선택 완료 상태를 대신하지 않는다.
- 브라우저 확인 시나리오: `/mes?tab=warehouse`에서 품목 전환에 진입해 소스 품목을 하나 선택한다. 왼쪽 패널이 선택 요약으로 바뀌고 대상 후보 선택에 집중되는지 확인한다.
- 우선순위: 높음

### 4. 소스 선택 전 대상 품목 영역이 큰 빈 패널로 화면 절반을 차지하는 문제
- 화면/영역: PC 입출고 > 품목 전환 > 소스·대상 선택 단계
- 증상: 소스 품목을 고르기 전에도 대상 품목 영역이 큰 패널로 보이고, 내부에는 `소스 품목을 먼저 선택하세요` 같은 빈 상태만 놓인다.
- 사용자 불편: 아직 조작할 수 없는 영역이 화면의 절반을 차지해, 사용자의 시선이 첫 행동인 소스 선택에 집중되지 않는다.
- 확정 정책: 아직 선택 불가한 영역은 큰 결과판으로 두지 않고, 다음 조건을 알려주는 보조 영역으로 낮춘다.
- 구현 방향: 소스 선택 전에는 대상 패널을 축소하거나 안내형 보조 카드로 바꾼다. 소스 선택 후에만 대상 검색 결과 영역을 크게 펼친다.
- 수용 기준: 최초 진입 화면에서 사용자의 1순위 행동이 소스 품목 선택이라는 점이 명확하다. 조작 불가한 대상 영역이 빈 리스트처럼 보이지 않는다.
- 브라우저 확인 시나리오: 품목 전환에 처음 진입해 소스 미선택 상태를 확인한다. 대상 영역이 빈 결과판이 아니라 단계 조건 안내로 보이는지 확인한다.
- 우선순위: 높음

### 5. 품목 전환 하단 상태 문구가 현재 막힌 조건을 구체적으로 말하지 않는 문제
- 화면/영역: PC 입출고 > 품목 전환 > 하단 action bar
- 증상: 소스, 대상, 수량 중 일부가 빠진 상태에서도 하단 문구가 `품목·수량 선택`처럼 포괄적으로 표시된다.
- 사용자 불편: 사용자는 다음 버튼이 왜 비활성인지 즉시 알기 어렵다. 특히 소스는 선택했고 대상만 남은 상태에서도 같은 문구가 나오면 진행 상태가 흐려진다.
- 확정 정책: 하단 상태 문구는 현재 남은 필수 조건을 직접 말한다.
- 구현 방향: 상태 계산을 `소스 품목 미선택`, `대상 품목 미선택`, `전환 수량 확인 필요`, `차이 확인 가능`처럼 단계별로 분기한다.
- 수용 기준: 각 미완료 상태에서 하단 문구가 하나의 구체적인 행동을 안내한다.
- 브라우저 확인 시나리오: 소스 미선택, 소스만 선택, 소스/대상 선택, 수량 오류 상태를 각각 만들고 하단 문구가 바뀌는지 확인한다.
- 우선순위: 높음

### 6. 선택 요약 카드에서 `변경`/`해제` 액션 위계가 부족한 문제
- 화면/영역: PC 입출고 > 품목 전환 > 선택 요약 영역
- 증상: 상단 요약에는 소스/대상/수량이 보이지만, 이미 고른 품목을 바꾸거나 해제하는 액션이 항상 같은 위계로 보이지 않는다.
- 사용자 불편: 잘못 선택했을 때 어디를 눌러야 다시 고를 수 있는지 즉시 보이지 않으면 `선택 초기화` 같은 큰 범위 액션에 의존하게 된다.
- 정책 확인 필요: 소스와 대상 각각에 `변경`만 둘지, `해제`까지 둘지 결정해야 한다. 권장 방향은 개별 카드에는 `변경`을 기본으로 두고, 전체 리셋은 하단 `선택 초기화`로 유지하는 것이다.
- 구현 방향: 선택 요약 카드에 소스/대상별 `변경` 액션을 붙이고, 전체 초기화와 구분되는 시각 위계를 둔다.
- 수용 기준: 사용자가 선택을 잘못했을 때 전체 흐름을 초기화하지 않고 해당 품목만 다시 선택할 수 있다.
- 브라우저 확인 시나리오: 소스와 대상을 모두 선택한 뒤 각 카드에서 개별 변경 액션이 보이는지 확인한다.
- 우선순위: 중간

## 입출고

### 7. 입출고 부족 라인 `창고에서 가져오기`의 선택 범위가 약한 문제
- 화면/영역: PC 입출고 > 일반 입출고 작업 > 부족 라인 처리
- 증상: 부족 라인을 창고 반출 작업으로 가져오는 흐름에서, 선택이 없으면 부족 라인 전체가 대상이고 선택이 있으면 선택된 부족 라인만 대상이 되는 규칙이 코드상 존재한다.
- 사용자 불편: 선택 0개가 "아무것도 안 함"인지 "전체 가져오기"인지 화면에서 충분히 드러나지 않으면, 사용자는 의도보다 큰 범위의 작업을 만들 수 있다고 느낀다.
- 정책 확인 필요: 선택 0개를 전체 대상으로 유지할지, 반드시 명시 선택을 요구할지 결정해야 한다. 권장 방향은 현재 동작을 유지하되 버튼 문구를 `부족 3개 전체 가져오기`처럼 범위가 보이게 하는 것이다.
- 구현 방향: 버튼 라벨과 확인 문구에 대상 범위를 표시한다. 선택이 있으면 `선택한 2개 가져오기`, 선택이 없으면 `부족 3개 전체 가져오기`처럼 구분한다.
- 수용 기준: 사용자가 버튼을 누르기 전에 생성될 창고 반출 작업의 대상 범위를 알 수 있다.
- 브라우저 확인 시나리오: 부족 라인이 있는 입출고 작업에서 체크박스 선택 없음/일부 선택 상태를 만들고 버튼 문구가 바뀌는지 확인한다.
- 우선순위: 높음

### 8. 입출고 작업의 부족 해결 동선과 출하 준비 부족 표시 문법이 다른 문제
- 화면/영역: PC 입출고 > 일반 입출고 부족 처리, 출하 준비 부족 표시
- 증상: 일반 입출고는 부족 라인을 `창고에서 가져오기` 같은 해결 동선으로 연결하고, 출하 준비는 `N개 부족` 표시 중심으로 보여준다. 두 화면 모두 부족을 다루지만 사용자가 읽는 문법이 다를 수 있다.
- 사용자 불편: 같은 `부족` 상태인데 어떤 화면에서는 해결 액션이 중심이고, 어떤 화면에서는 경고 표시만 중심이면 다음 행동을 예측하기 어렵다.
- 정책 확인 필요: 출하 준비에서도 부족 라인에 즉시 해결 동선을 둘지, 표시만 하고 별도 준비 프로세스로 남길지 결정해야 한다. 권장 방향은 이번 후속에서는 표시 문법만 맞추고, 재고 차감/해결 동선 확대는 별도 업무로 둔다.
- 구현 방향: 부족 라벨, 빨간 tint, 부족 수량 배지 문법을 일반 입출고와 출하 준비에서 통일한다. 해결 액션 유무는 화면 목적에 따라 다르다는 보조 문구를 둔다.
- 수용 기준: 사용자는 두 화면의 부족 라인을 같은 상태로 인식한다. 해결 액션이 없는 경우에도 왜 없는지 화면 문맥에서 이해된다.
- 브라우저 확인 시나리오: 일반 입출고 부족 라인과 출하 준비 부족 라인을 나란히 비교해 색, 라벨, 수량 문법이 같은 체계인지 확인한다.
- 우선순위: 중간

## 출하 요청

### 9. `준비 중` 요청 수정 화면의 수정 가능 범위가 불명확한 문제
- 화면/영역: PC 출하 > 출하 요청 수정 > 기존 요청 편집
- 증상: 기존 요청 화면은 `출하 요청 수정` 제목과 `준비 중` 상태를 동시에 보여준다. 사용자는 이미 준비 중인 요청에서 무엇을 바꿀 수 있고, 바꾼 내용이 준비 작업에 어떻게 반영되는지 화면만으로 알기 어렵다.
- 사용자 불편: 작업자가 수정해도 되는 범위를 확신하지 못하면, 준비 중인 출하 요청을 잘못 바꾸거나 아예 수정하지 못하고 멈출 수 있다.
- 확정 정책: 기존 요청 수정 화면은 현재 상태에서 가능한 수정 범위와 반영 결과를 상단에 먼저 보여준다.
- 구현 방향: 헤더나 결론 카드에 `준비 중 요청`, `수정 가능 항목`, `수정 후 준비 목록 재계산 여부`를 짧게 표시한다. 상태에 따라 불가능한 필드는 비활성 사유를 붙인다.
- 수용 기준: `준비 중` 요청을 열었을 때 사용자는 수정 가능한 범위와 다음 영향 범위를 한 화면에서 알 수 있다.
- 브라우저 확인 시나리오: `shippingView=requestWork&shippingRequestId=...`로 기존 요청 수정 화면을 열고, 상태/수정 가능 범위/다음 영향 설명이 보이는지 확인한다.
- 우선순위: 높음

### 10. PA/PF 생성·재사용 라벨과 실제 최종 이름이 헷갈릴 수 있는 문제
- 화면/영역: PC 출하 > 출하 요청 > 3단계 BOM 매칭, 5단계 최종 확인
- 증상: 확인한 화면에서 `PA 기존 PA 재사용 · UI TEST`, `PF 기존 PF 재사용 · UI TEST`처럼 표시된다. 데이터 의미에 따라 `UI TEST`가 새 이름인지 재사용 이름인지 모호하게 읽힐 수 있다.
- 사용자 불편: 사용자는 새 PA/PF가 생성되는지, 기존 PA/PF를 재사용하는지, 단순히 이름만 같은 것인지 헷갈릴 수 있다. 최종 출하 품목 이름 확인도 약해진다.
- 확정 정책: PA/PF 상태 라벨은 `새 PA 생성 예정`, `기존 PA 재사용`, `변경 없음`을 명확히 구분한다. 이름은 그 다음에 보조 값으로 붙인다.
- 구현 방향: 매칭 결과의 상태와 표시 이름을 분리한다. 결론 카드에는 상태 라벨을 먼저, 품명/예정 이름/코드는 다음 줄에 둔다.
- 수용 기준: 사용자는 라벨만 보고도 생성/재사용/변경 없음 중 어느 상태인지 알 수 있다.
- 브라우저 확인 시나리오: 새 PA/PF 생성 예정 케이스와 기존 재사용 케이스를 각각 열고 라벨이 서로 헷갈리지 않는지 확인한다.
- 우선순위: 높음

### 11. 출하 수량이 상단과 하단에 반복되지만 수정 동선 연결이 약한 문제
- 화면/영역: PC 출하 > 출하 요청 > 3단계 이후 단계의 수량 요약과 하단 action bar
- 증상: 3단계 화면에서 상단에는 `출하 수량 10대`와 `수량 변경`이 보이고, 하단 action bar에도 `출하 수량 10대`가 반복된다.
- 사용자 불편: 같은 수량이 두 군데 보이는데 한쪽에는 수정 액션이 있고 다른 쪽에는 없다. 사용자는 하단 수량이 단순 요약인지 눌러야 하는 대상인지 헷갈릴 수 있다.
- 확정 정책: 후속 단계에서 출하 수량 수정은 `수량 변경` 액션 하나로 통일한다. 하단 반복 표시는 현재 단계 결론 또는 고정 요약임을 분명히 한다.
- 구현 방향: 하단 action bar의 수량 표시는 `현재 수량 10대`처럼 읽기 전용임을 드러내거나, 상단과 같은 `수량 변경` 액션을 같은 문법으로 연결한다. 단, 직접 입력 필드로 만들지 않는다.
- 수용 기준: 사용자는 출하 수량을 바꾸려면 어디를 눌러야 하는지 한 군데로 이해한다.
- 브라우저 확인 시나리오: 3단계와 5단계에서 상단 수량 카드와 하단 action bar 수량을 비교하고, 수정 동선이 충돌하지 않는지 확인한다.
- 우선순위: 중간

## 출하 준비

### 12. 부족 대상이 출하품/구성품/동반 출하품 중 무엇인지 약한 문제
- 화면/영역: PC 출하 > 준비 작업 > 재고 부족 표시
- 증상: 부족 라인에는 `10개 부족`처럼 부족 수량이 짧게 표시되지만, 그 부족이 최종 출하품인지 PA/PF 구성품인지 동반 출하품인지 즉시 구분되지 않을 수 있다.
- 사용자 불편: 사용자는 부족 원인을 파악하려고 목록을 다시 대조해야 한다. 특히 출하품 자체 부족과 구성품 부족은 다음 행동이 다를 수 있다.
- 확정 정책: 부족 수량은 짧게 유지하되, 부족 대상의 성격 라벨을 함께 표시한다.
- 구현 방향: 부족 라인 또는 부족 요약에 `출하품`, `PA 구성품`, `PF 구성품`, `동반 출하품` 라벨을 붙인다. 색은 기존 red tint/border 체계를 따른다.
- 수용 기준: 사용자는 부족 배지만 보고도 부족 대상의 위치와 성격을 알 수 있다.
- 브라우저 확인 시나리오: 출하품 부족 케이스와 구성품 부족 케이스를 열고 각각 다른 라벨로 구분되는지 확인한다.
- 우선순위: 높음

### 13. 부족 요약과 실제 목록 라인의 검산 연결이 약한 문제
- 화면/영역: PC 출하 > 준비 작업 > 재고 부족 요약, PA/PF/동반 출하품 목록
- 증상: 상단 부족 요약 또는 경고가 있더라도, 사용자가 실제 목록에서 어느 라인이 그 부족에 해당하는지 바로 추적하기 어려울 수 있다.
- 사용자 불편: 경고를 보고도 실제 준비 목록에서 확인해야 할 항목을 찾는 데 시간이 든다.
- 확정 정책: 부족 요약은 실제 부족 라인과 검산 가능하게 연결되어야 한다.
- 구현 방향: 부족 요약 항목과 목록 라인의 라벨/품명/코드/수량 표현을 맞춘다. 가능하면 부족 요약 클릭 또는 포커스 이동으로 해당 라인을 찾게 한다.
- 수용 기준: 사용자는 부족 요약에서 본 품목을 목록 안에서 같은 이름과 같은 부족 수량으로 찾을 수 있다.
- 브라우저 확인 시나리오: 부족이 있는 준비 작업 화면에서 상단 부족 요약과 PA/PF/동반 출하품 목록의 라인을 대조한다.
- 우선순위: 높음

## 입출고 내역 연결성

### 14. 요청·준비·전환 화면의 결론이 입출고 내역 탭에서 같은 말로 이어지는지 검증 필요
- 화면/영역: PC 입출고 탭 전체 > 출하 요청, 출하 준비, 품목 전환, 입출고 내역
- 증상: 작업 화면에서는 `구성 전환`, `새 PF 생성 예정`, `N개 부족`, `출하 준비`처럼 결론을 보여주고, 입출고 내역 탭에서는 거래 row와 펼침 row가 별도 presentation 규칙으로 표시된다.
- 사용자 불편: 작업을 완료한 뒤 내역에서 같은 거래를 찾았을 때 용어와 수량 의미가 달라 보이면, 사용자는 방금 한 작업이 제대로 기록됐는지 검산하기 어렵다.
- 정책 확인 필요: 입출고 내역 탭의 용어를 작업 화면 용어에 맞출지, 내역 탭의 재고 변화 중심 용어를 기준으로 작업 화면을 보조 정리할지 결정해야 한다. 권장 방향은 내역 탭은 재고 변화 중심으로 두되, 작업 화면의 결론 라벨을 내역의 작업 배지와 연결되게 맞추는 것이다.
- 구현 방향: 품목 전환 실행, 출하 준비 완료, 일반 입출고 제출 후 내역 row에 나타나는 작업명/수량/부족 또는 제외 라벨을 end-to-end로 비교하는 검산 체크리스트를 만든다.
- 수용 기준: 사용자는 작업 화면에서 본 결론을 내역 탭에서 같은 거래로 자연스럽게 찾고, 수량 의미를 다시 해석하지 않아도 된다.
- 브라우저 확인 시나리오: 품목 전환 또는 출하 준비 완료 후 입출고 내역 탭으로 이동한다. 목록 row, 펼침 row, 상세 패널에서 작업 화면의 결론과 같은 의미의 라벨이 보이는지 확인한다.
- 우선순위: 높음
