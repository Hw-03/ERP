# 2026-07-06 입출고 요청 작성 UX 사이드 세션 핸드오프

## 문서 목적

이 문서는 메인 세션에서 진행 중인 `구성품 변경 -> 품목 전환` 이동 작업과 별개로, 사이드 세션에서 논의한 `입출고` 탭의 요청 작성 UX 개선 방향을 빠짐없이 넘기기 위한 핸드오프다.

중요한 점은 이번 사이드 세션에서 결론난 내용이 곧바로 코드 구현 지시였던 것은 아니라는 점이다. 다만 사용자는 이 내용을 바탕으로 메인 세션에서 다음 구현 계획을 이어가길 원한다.

## 사이드 세션의 출발점

사용자는 사이드 세션이 무엇을 위한 것인지 물었다. 답변 과정에서 다음처럼 정리했다.

- 사이드 세션은 메인 세션의 작업을 멈추거나 오염시키지 않고 옆에서 질문, 정리, 판단 보조를 하는 공간이다.
- 사이드 세션은 boundary 이전 메인 세션 맥락을 참고할 수 있지만, 이전 지시를 이어서 실행하거나 파일 수정/커밋/서버 조작을 하면 안 된다.
- 단, 사용자가 boundary 이후 명시적으로 문서 작성처럼 별도 mutation을 요청하면 최소 범위로 수행할 수 있다.
- 이 사이드 세션은 `C:\ERP` DEXCOWIN MES 작업에서 갈라진 사이드 대화로 이해했다.

## 메인 세션에서 하던 일에 대한 공유 맥락

메인 세션에서는 출하 탭에 임시로 들어갔던 `구성품 변경` 기능을 입출고 탭으로 옮겨 `품목 전환`으로 표시하는 작업을 진행 중이었다.

메인 작업의 핵심은 다음과 같았다.

- 기존 출하 탭 `구성품 변경`은 소스 PA 재고를 대상 PA 재고로 전환하고, BOM 차이에 따라 구성품 차감/회수를 처리하는 기능이었다.
- 논의 결과 이 기능은 출하 전용이라기보다 입출고/재고 반영 계열의 공식 재고 작업에 가깝다고 판단했다.
- 따라서 사용자 표시명은 `구성품 변경`보다 더 넓은 `품목 전환`이 적합하다고 봤다.
- 1단계 구현은 backend API/DB를 바꾸지 않고, 프론트 화면 소속과 문구만 옮기는 방향이었다.
- 기존 API `/api/shipping/component-change-preview`, `/api/shipping/component-change`는 유지하고, 화면에서는 `품목 전환`으로 감싸는 방향이었다.

## 품목 전환 후속 논의의 출발점

처음에는 다음 후속 쟁점들을 정리했다.

- PA뿐 아니라 AF/AA까지 확장할지
- BOM 비교 범위를 직계만 볼지, 전체 전개로 볼지
- BOM이 같으면 단순 전환, 다르면 구성 전환으로 나눌지
- 권한/승인/메모/이력 표현을 어떻게 할지
- 기존 재작업/생산/분해 기능과 경계를 어떻게 둘지

사용자의 답은 다음과 같았다.

- PA뿐 아니라 AF/AA까지 확장할 것이다.
- BOM 비교 범위는 아직 잘 모르겠다.
- 큰 작업명은 모두 `전환`으로 본다.
- 권한/승인/메모/이력 정책은 아직 잘 모르겠다.
- 기존 기능과의 경계에서는 특히 `분해`가 문제다.

## 품목 전환과 분해의 경계

사이드 세션에서 정리한 개념 구분은 다음과 같다.

- 실제로 완전히 해체해서 자재 단위로 돌아가면 `분해`.
- 완제품/반제품 상태를 유지한 채 일부 구성이나 사양만 바꾸면 `품목 전환`.
- 불량 때문에 수리하거나 폐기/회수하면 `재작업` 또는 `불량 처리`.
- 처음부터 새로 만드는 작업이면 `생산`.

사용자는 `분해`라는 라벨이 입출고 작업 화면에 꼭 필요하냐는 문제의식을 이어갔다.
이에 대해, `분해/출고`를 입출고의 기본 작업명으로 노출하는 것은 약하고, `출고` 자체도 현재 업무 구조에서는 정상 프로세스의 우회 경로가 될 수 있다고 정리했다.

정상적인 재고 감소는 다음처럼 이미 다른 탭/업무에 흡수되어 있다.

- 고객에게 나감: `출하`
- 생산/조립에 투입됨: 생산/BOM 자동 차감
- 불량으로 빠짐: `불량 처리`
- 위치 변경: `이동`
- 품목 정체성 변경: `품목 전환`
- 그 외 실제 재고와 수치가 다름: `수량 조정`

따라서 입출고 탭의 역할 자체를 다시 봐야 한다는 결론으로 이어졌다.

## 입출고 탭 이름에 대한 논의

처음에는 입출고 탭 자체가 이름과 역할이 안 맞는다는 이야기가 나왔다.

후보로는 다음이 논의됐다.

- `재고 작업`
- `재고 처리`
- `재고 변동`
- `재고 반영`
- `재고 업무`

사용자는 `재고 작업`은 별로라고 했다.

`재고 처리`는 출고/처분 느낌이 강하고, 생산도 하는 탭이므로 맞지 않는다고 했다.

그 뒤 `재고 반영`이 후보로 떠올랐다.
이 이름은 "현장에서 일어난 재고 변화를 시스템에 반영한다"는 의미라서 좋다고 판단했고, 사용자도 `재고 반영`이 좋은 것 같다고 했다.

하지만 이후 실제 화면과 현장 적응 문제를 보면서 결론이 바뀌었다.

최종적으로 이번 단계에서는 탭 이름을 `재고 반영`으로 바꾸지 않는다.
이유는 현장 작업자들이 이미 `입출고` 탭에 적응해 있기 때문이다.

최종 결정:

- `입출고` 탭 이름은 유지한다.
- `재고 반영`은 장기 방향성/개념어로는 유효하지만, 이번 구현 범위에는 넣지 않는다.

## 1단계 카드명 변경 여부

초기에는 `창고 입출고`를 `창고 반입·반출`로, `부서 입출고`를 `부서 재고 반영`으로 바꾸자는 아이디어가 있었다.

사용자는 현장에서는 `반입/반출`이라는 말을 사용한다고 설명했다.

- 창고에 들어가면 `반입`
- 창고에서 나가면 `반출`

그래서 한때 `창고 반입·반출`이 적절해 보였다.

하지만 곧 사용자는 다음과 같이 결정했다.

- 지금 와서 1단계 카드명을 바꾸면 작업자가 헷갈릴 수 있다.
- 따라서 `창고 입출고`, `부서 입출고`는 수정하지 않는다.
- `원자재 입고`도 유지한다.

최종 결정:

- `원자재 입고` 유지
- `창고 입출고` 유지
- `부서 입출고` 유지
- 새로 추가된 `품목 전환` 카드 유지

중요한 점:

- 하지만 이번 UX 개선 범위에서는 기존 세부 작업명을 건드리지 않는다.
- 사용자 적응을 우선한다.

## 불량/출하 탭 문법을 참고해야 한다는 지적

사용자는 현재 입출고 화면이 불량/출하 탭과 화면 문법이 다르다고 했다.

문제의식:

- 불량/출하 탭은 들어가자마자 업무 보드/허브가 보인다.
- 그 안에서 특정 작업을 선택하면 전용 작업 화면으로 들어가는 느낌이 있다.
- 반면 입출고 탭은 탭에 들어오자마자 바로 위저드 1단계가 펼쳐져 있다.
- `요청 작성`, `작성 중`, `내 요청`이라는 상단 흐름이 있음에도, 실제 본문은 곧바로 위저드여서 전용 작업 화면 전환감이 약하다.

사용자는 "불량 탭을 참고하면 되지"라고 했고, 이후 질문은 불량 탭 문법을 기준으로 정리했다.

## 상단 흐름 유지 여부

상단의 `요청 작성 / 작성 중 / 내 요청` 흐름은 유지하기로 했다.

이유:

- 사용자가 이미 이 흐름에 적응했다.
- 이 탭에 들어오자마자 기본이 `요청 작성`이다.
- 불량/출하처럼 완전한 허브로 갈 수도 있지만, 이번 단계에서 상단 흐름까지 뜯어고치면 변화가 너무 크다.

최종 결정:

- `요청 작성` 유지
- `작성 중` 유지
- `내 요청` 유지

## 카드 클릭 후 화면 전환 문법

가장 중요한 결론은 다음이다.

현재 코드도 넓게 보면 전용 작업 화면 성격은 있다.
하지만 시각적으로는 `작업 유형 선택` 큰 카드 아래에 2단계, 3단계가 계속 누적되는 위저드처럼 보여서 전용 작업에 들어왔다는 느낌이 약하다.

따라서 작업 유형 카드를 클릭한 뒤에는 불량/출하 탭처럼 전용 작업 화면에 들어온 느낌을 강화한다.

확정된 방향:

- 작업 유형 선택 후 1단계 큰 카드 영역은 숨긴다.
- 같은 큰 컨테이너 안 상단에 전용 작업 헤더를 둔다.
- 기존 기능 흐름과 상태 모델은 유지한다.
- 시각 문법만 전용 작업 화면처럼 바꾼다.

## 전용 작업 헤더

전용 작업 헤더는 기존 큰 카드 컨테이너 안 상단에 둔다.

헤더 구성:

- `← 작업 유형 선택`
- 선택한 작업명
- 기존 work type description

중요:

- description은 새로 쓰지 않는다.
- 코드의 기존 description을 그대로 사용한다.

예시:

- `원자재 입고` / `발주 품목 입고`
- `창고 입출고` / `창고↔부서`
- `부서 입출고` / `부서 내 작업`

헤더의 위치:

- 카드 컨테이너 바깥이 아니라 기존 큰 컨테이너 안 상단
- 이유는 현재 입출고 화면의 큰 파란 테두리/카드 구조에 현장 사용자가 익숙하기 때문이다.

## 작업 유형 선택으로 돌아가기

전용 헤더의 `← 작업 유형 선택`은 기존 dirty/unsaved 보호 로직을 재사용해야 한다.

새로 별도 상태 초기화/확인 로직을 만들면 안 된다.

작성 중인 내용이 있을 때의 보호, 임시저장 여부, 뒤 단계 데이터 유지/초기화는 현재 코드에 이미 있는 기능을 따라야 한다.

progress 단계명은 새로 만들지 않고, 실제 코드의 `WizardStepCard` title을 단일 소스로 사용한다.

중요:

- 코드와 UI 문구가 어긋나지 않게 실제 step title을 사용한다.

## 이전 단계 요약 row

전용 화면에서 이전 단계는 기존 `WizardStepCard` 접힘 상태를 그대로 쓰지 않는다.

대신 새 얇은 요약 row를 만든다.

요약 row의 성격:

- 어울리게 새로 디자인한다.
- 단계명과 선택 요약을 보여준다.
- 클릭 가능하다.
- 클릭하면 기존 단계 이동 로직으로 해당 단계에 돌아간다.

사용자는 처음에 `수정` 버튼이라는 표현을 이해하지 못했다.
따라서 `수정 버튼`을 별도로 만든다는 식으로 구현하지 말고, 요약 row 자체가 클릭 가능한 것으로 이해하는 편이 맞다.

결정:

- 요약 row 클릭 가능
- progress bar 완료 단계도 클릭 가능
- 둘 다 기존 `state.goTo(n)` 계열의 단계 이동 로직을 재사용

## 현재 단계와 다음 단계 표시

현재 단계는 기존처럼 큰 카드로 유지한다.

이유:

- 현재 UI는 작업자가 큰 카드에서 터치/클릭하기 좋게 설계되어 있다.
- 입력 면적, 단계 내부 폼 구조, 카드 높이는 크게 바꾸지 않는다.

다음 단계는 미리 세로로 누적해서 보여주지 않는다.

표시 원칙:

- 완료된 이전 단계: 얇은 요약 row
- 현재 단계: 큰 카드
- 미래 단계: 숨김 또는 progress에서 비활성으로만 표시

## 완료 후 위치

요청 제출/완료 후에는 `작성 중`이나 `내 요청`으로 이동하지 않는다.

사용자는 "애초에 이 탭에 들어오자마자도 요청 작성이잖아"라고 했다.

따라서 완료 후에는 `요청 작성` 첫 화면, 즉 작업 유형 선택 화면으로 돌아간다.

결정:

- 완료 후 `요청 작성` 첫 화면으로 복귀
- `작성 중` 자동 이동 없음
- `내 요청` 자동 이동 없음

## 품목 전환 상세 화면

`품목 전환` 상세 화면은 이번 사이드 세션에서 별도로 설계하지 않기로 했다.

처음에는 품목 전환의 progress를 다음처럼 논의하려 했다.

- `품목 선택 -> 차이 확인 -> 실행 확인 -> 완료`
- `소스/대상 선택 -> 변경 확인 -> 실행 -> 완료`
- `전환 정보 -> 재고 영향 -> 확인 -> 완료`

하지만 사용자는 `품목 전환`은 나중에 설계하자고 했다.

따라서 이번 범위에서 할 일:

- `품목 전환` 카드는 1단계 카드 배열 안에서 기존 3개 카드와 시각적으로 어색하지 않게 배치/톤만 맞춘다.

이번 범위에서 하지 말 것:

- `품목 전환` 상세 화면 설계
- `품목 전환` 내부 progress 설계
- AF/AA 확장 구현
- BOM 비교 정책 변경
- 권한/승인/메모/이력 정책 변경

## 이번 구현에서 하지 말아야 할 것

사용자 결정을 기준으로 이번 단계에서 하지 말아야 할 작업은 다음과 같다.

- `입출고` 탭명을 바꾸지 않는다.
- `재고 반영`으로 즉시 리네이밍하지 않는다.
- `원자재 입고` 카드명을 바꾸지 않는다.
- `창고 입출고` 카드명을 바꾸지 않는다.
- `부서 입출고` 카드명을 바꾸지 않는다.
- 기존 세부 작업명 `창고 → 부서`, `부서 → 창고`, `생산`, `분해`, `수량보정 입고`, `출고`를 바꾸지 않는다.
- backend API를 바꾸지 않는다.
- DB schema를 바꾸지 않는다.
- 입출고 상태 모델을 바꾸지 않는다.
- 기존 dirty/unsaved/임시저장 보호 로직을 새로 만들지 않는다.
- `품목 전환` 상세 화면을 이번에 설계하지 않는다.

## 이번 구현의 핵심 목표

이번 구현의 목표는 기존 업무명/상태/세부 작업을 바꾸는 것이 아니다.

목표는 다음 한 문장으로 정리된다.

> 입출고 탭의 기존 요청 작성 흐름은 유지하되, 작업 유형 선택 후 화면이 불량/출하 탭처럼 전용 작업 화면에 들어온 것처럼 보이도록 렌더링 문법을 바꾼다.

구체적으로는 다음을 구현해야 한다.

1. `요청 작성 / 작성 중 / 내 요청` 흐름 유지
2. 1단계 작업 유형 카드명 유지
3. 작업 유형 선택 후 1단계 큰 카드 숨김
4. 기존 큰 컨테이너 안 상단에 전용 작업 헤더 추가
5. 헤더에는 `← 작업 유형 선택`, 작업명, 기존 description 표시
6. progress bar 표시
7. progress 단계명은 실제 `WizardStepCard` title 사용
8. 완료된 progress 단계 클릭 가능
9. 현재 progress 단계는 클릭해도 동작 없음
10. 미래 progress 단계는 비활성
11. 이전 단계는 새 얇은 요약 row로 표시
12. 요약 row 클릭 시 기존 단계 이동 로직으로 이동
13. 현재 단계는 기존 큰 카드 유지
14. 미래 단계는 화면에 누적 표시하지 않음
15. 완료 후 `요청 작성` 첫 화면으로 복귀
16. `품목 전환` 카드는 1단계 배열에서 어색하지 않게만 맞춤

## 메인 세션에 전달할 짧은 요약

입출고 탭 이름과 기존 카드명은 바꾸지 않는다. `원자재 입고 / 창고 입출고 / 부서 입출고` 및 세부 작업명은 그대로 유지한다. 이번 목표는 작업 유형 선택 후 1단계 큰 카드가 계속 남아 위저드처럼 보이는 문제를 줄이고, 불량/출하 탭처럼 전용 작업 화면에 들어온 느낌을 주는 것이다. 카드 선택 후 1단계 큰 카드는 숨기고, 같은 컨테이너 안 상단에 `← 작업 유형 선택 + 작업명 + 기존 description` 헤더를 둔다. progress는 실제 `WizardStepCard` title을 쓰고 완료된 단계는 클릭 가능하게 하며, 이전 단계는 새 얇은 요약 row로 표시하고 클릭 가능하게 한다. 현재 단계는 기존 큰 카드로 유지하고 미래 단계는 숨긴다. 완료 후에는 `요청 작성` 첫 화면으로 돌아간다. `품목 전환` 상세 화면 설계는 후속으로 미루고, 이번에는 카드 배열에서 어색하지 않게만 맞춘다.
