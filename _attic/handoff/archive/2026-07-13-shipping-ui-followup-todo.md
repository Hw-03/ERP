### 4. BOM 편집 행의 수량·제외 동작 높이와 정렬 통일
- 화면/영역: PC 출하 > 출하 요청 작성 > 2단계 `BOM 구성 조정`의 PA 구성품·PF 구성품 목록 행
- 증상: 품명이 두 줄로 늘어난 행에서는 수량 입력과 `제외` 버튼이 짧은 한 줄 높이를 유지해, 행 중앙에 맞지 않고 숫자도 좌우로 치우쳐 보인다.
- 사용자 불편: 품명 길이에 따라 같은 목록의 수량 확인·수정 위치가 흔들리고, 제외 동작을 읽거나 누르기 어렵다.
- 확정 정책: 모든 BOM 편집 행에서 수량은 가운데 정렬하고, 수량 입력과 제외 버튼은 행 콘텐츠 높이에 맞춰 늘어난다.
- 구현 방향: PA·PF 공통 행 grid의 수량 입력과 제외 버튼에 행 높이 추종 규칙을 적용한다. 품명·코드 영역은 두 줄까지 자연스럽게 늘어나며, 수량 입력과 제외 버튼은 같은 행의 전체 높이를 사용한다.
- 수용 기준:
  - 한 줄 품명과 두 줄 품명 모두에서 수량 숫자가 가로·세로 중앙에 있다.
  - 두 줄 품명 행에서 수량 입력과 제외 버튼의 높이가 항목 행 높이와 맞는다.
  - PA와 PF의 동일 동작 행은 같은 크기·정렬 문법을 사용한다.
- 브라우저 확인 시나리오:
  1. 긴 품명이 있는 PA·PF 구성품을 각각 표시한다.
  2. 행의 수량 숫자와 `제외` 버튼이 행 중앙에 정렬되는지 확인한다.
  3. 수량을 바꾸고 제외한 뒤에도 행 높이와 정렬이 유지되는지 확인한다.
- 우선순위: 높음

- 증상: 동반 출하품 검색 결과와 선택 목록이 PA·PF 구성품과 다른 형태로 보이며, 긴 품명에서 가로 스크롤이 생긴다.
- 사용자 불편: 같은 단계 안에서만 목록 읽기 방식이 달라지고, 가로 스크롤 때문에 후보 품명과 추가 동작을 빠르게 비교하기 어렵다.
- 확정 정책: 동반 출하품은 PA·PF 구성품과 같은 행 밀도와 정보 순서를 사용하며, 어떤 화면 폭에서도 가로 스크롤을 만들지 않는다.
- 구현 방향: 후보와 선택 행을 `품명 / 품목코드 / 추가 또는 제거`의 세로 정보 문법으로 바꾸고, 긴 품명은 줄바꿈 또는 말줄임으로 처리한다. 컨테이너와 행 모두 `overflow-x`가 발생하지 않도록 grid 폭을 제한한다.
- 수용 기준:
  - 동반 출하품 후보·선택 목록이 PA·PF 목록과 유사한 카드·행 계층으로 보인다.
  - 긴 품명, 긴 품목코드, 다수 후보에서도 수평 스크롤바가 없다.
  - 추가·제거 후에도 선택 목록의 행 폭과 버튼 위치가 유지된다.
- 브라우저 확인 시나리오:
  1. 긴 이름의 동반 출하품을 검색한다.
  2. 후보 목록과 선택 목록의 가로 스크롤바 유무를 확인한다.
  3. 여러 항목을 추가·제거하며 PA·PF 목록과 정보 밀도가 일관적인지 확인한다.
- 우선순위: 높음

### 6. BOM 매칭 결과 3칸의 본문 높이 활용
- 화면/영역: PC 출하 > 출하 요청 작성 > 3단계 `BOM 매칭`의 기본 BOM·PA·PF 결과 카드
- 증상: 외곽 프레임을 제거한 뒤에도 결과 카드 아래에 큰 빈 공간이 남아, 매칭 결과가 작업 영역을 충분히 사용하지 못한다.
- 사용자 불편: 현재 매칭 결과가 한눈에 정리되어도 화면이 비어 보여 다음 단계로 넘어가기 전 확인 밀도가 낮다.
- 확정 정책: 기본 BOM·PA·PF 결과 3칸이 3단계 본문 높이를 자연스럽게 나누어 사용한다.
- 구현 방향: 3단계 본문을 결과 카드와 BOM 변경 영역의 높이 분배가 가능한 flex/grid 구조로 바꾸고, 결과 카드에는 최소 높이와 확장 규칙을 둔다. 세 카드의 높이는 동일하게 유지한다.
- 수용 기준:
  - 기본 BOM·PA·PF 카드가 남는 본문 높이를 사용해 균형 있게 확장된다.
  - 세 카드의 높이와 상단 정렬이 동일하다.
  - 품명과 품목코드는 기존처럼 서로 다른 줄에서 읽힌다.
- 브라우저 확인 시나리오:
  1. BOM 변경 없음과 변경 있음 요청을 각각 3단계까지 진행한다.
  2. 결과 3칸이 본문에 남는 큰 공백 없이 자연스럽게 확장되는지 확인한다.
  3. 긴 PA/PF 품명에서도 품명과 품목코드 줄이 겹치지 않는지 확인한다.
- 우선순위: 중간

### 7. BOM 변경 없음 안내를 3단계 하단 action bar로 이동
- 화면/영역: PC 출하 > 출하 요청 작성 > 3단계 `BOM 매칭`의 본문 `BOM 변경 없음` 안내와 하단 action bar 중앙
- 증상: BOM 변경이 없을 때 본문에는 `BOM 변경 없음` 안내가 있고, 하단에는 `현재 출하 수량 N대 · 변경은 상단 수량 변경` 안내가 중복으로 남는다.
- 사용자 불편: 매칭 결과와 직접 관계없는 수량 안내가 하단을 차지하고, 중요한 BOM 변경 여부가 본문과 분리된다.
- 확정 정책: BOM 변경이 없으면 해당 안내를 하단 action bar 중앙에 표시하고, 3단계 본문의 안내와 수량 안내는 모두 표시하지 않는다.
- 구현 방향: 3단계 action bar 중앙 슬롯을 BOM 변경 상태에 따라 분기한다. 변경 없음일 때는 compact `BOM 변경 없음` 안내를, 변경 있음일 때는 하단 안내를 비워 두고 변경 구성 목록은 본문에서만 표시한다.
- 수용 기준:
  - BOM 변경 없음 요청에서 하단 action bar 중앙에 `BOM 변경 없음` 안내가 표시된다.
  - 같은 안내가 3단계 본문에 중복 표시되지 않는다.
  - 3단계에서 `현재 출하 수량 N대 · 변경은 상단 수량 변경` 문구는 표시되지 않는다.
- 브라우저 확인 시나리오:
  1. BOM 변경 없는 요청을 3단계까지 진행한다.
  2. 본문에는 결과 카드와 수량 행만 남고, 하단 중앙에 BOM 변경 없음 안내가 있는지 확인한다.
  3. BOM을 변경한 요청에서는 변경 구성 목록이 본문에만 남고 하단 중앙은 비어 있는지 확인한다.
- 우선순위: 중간

### 8. 요청 정보 단계 하단 중앙 영역 비우기
- 화면/영역: PC 출하 > 출하 요청 작성 > 4단계 `요청 정보`의 하단 action bar 중앙
- 증상: 요청자와 메모를 입력하는 화면에서도 `현재 출하 수량 N대 · 변경은 상단 수량 변경` 안내가 하단 중앙에 표시된다.
- 사용자 불편: 현재 단계 작업과 관계없는 안내가 입력 화면의 하단을 차지해 시선이 분산된다.
- 확정 정책: 4단계 하단 action bar 중앙은 공란으로 둔다.
- 구현 방향: 4단계의 footer hint 렌더링을 생략하고, 이전·다음 버튼의 위치와 동작은 유지한다.
- 수용 기준:
  - 4단계 하단 중앙에 수량 안내나 대체 문구가 없다.
  - 이전·다음 버튼은 기존 위치와 활성화 조건을 유지한다.
  - 요청자·메모 입력 영역의 높이와 스크롤 동작이 변하지 않는다.
- 브라우저 확인 시나리오:
  1. 4단계로 이동한다.
  2. 하단 action bar 중앙이 비어 있는지 확인한다.
  3. 메모를 입력하고 다음 단계로 이동할 수 있는지 확인한다.
- 우선순위: 중간

### 9. 최종 확인에서 변경 구성만 자체 스크롤
- 화면/영역: PC 출하 > 출하 요청 작성 > 5단계 `저장 및 전환`의 `BOM·동반 출하품`과 `변경된 구성품` 영역
- 증상: 변경된 구성품이 여러 줄이면 최종 본문 전체가 길어져 BOM·동반 출하품 영역의 표시 높이가 함께 줄어들 수 있다.
- 사용자 불편: 출하 대상의 기본 BOM·동반 출하품을 확인하는 핵심 영역이 변경 목록 때문에 작아져, 최종 확인 시 비교가 어려워진다.
- 확정 정책: `BOM·동반 출하품` 영역은 정해진 최소 표시 높이를 유지하고, 변경된 구성품이 많을 때는 해당 목록 안에서만 세로 스크롤한다.
- 구현 방향: 5단계 본문을 고정 높이 카드 배치로 나누고, 변경 구성 요약 카드의 목록 영역에만 `overflow-y-auto`와 최소·최대 높이를 둔다. 화면 전체와 BOM·동반 출하품 영역에는 추가 스크롤을 만들지 않는다.
- 수용 기준:
  - 변경 구성품이 두 개 이상이어도 BOM·동반 출하품 영역의 높이가 줄지 않는다.
  - 변경 구성 목록이 카드 내부에서만 세로 스크롤된다.
  - 출하 품목 요약, BOM·동반 출하품, 변경 구성 요약의 카드 폭과 하단 요청 정보 action bar가 겹치지 않는다.
- 브라우저 확인 시나리오:
  1. 변경 구성품이 없는 요청과 여러 개인 요청을 각각 5단계까지 진행한다.
  2. 여러 변경 구성품 요청에서 BOM·동반 출하품 영역의 높이가 유지되는지 확인한다.
  3. 변경 구성 카드 안에서만 목록을 스크롤하고, 출하 요청 버튼과 하단 요약 정렬이 유지되는지 확인한다.
- 우선순위: 높음

### 10. 매칭·최종 요약 카드의 제목과 행 간격 축소
- 화면/영역: PC 출하 > 출하 요청 작성 > 3단계 `BOM 매칭`의 기본 BOM·PA·PF 결과 카드, 5단계 `저장 및 전환`의 출하 품목 요약·`BOM·동반 출하품` 행·변경 구성 요약 카드
- 증상: 새 PA/PF 생성 예정처럼 제목 아래에 품명과 품목코드가 표시되는 카드와 `BOM·동반 출하품` 행에서 제목 크기와 제목·품명·품목코드 사이 간격이 커, 카드와 행 높이가 필요 이상으로 늘어난다.
- 사용자 불편: 3단계와 5단계 모두 요약 정보가 성기게 보여 한 화면에서 비교할 수 있는 정보량이 줄어든다.
- 확정 정책: 매칭 결과와 최종 요약 카드는 제목·품명·품목코드의 위계를 유지하되, 데스크톱 목록 카드와 같은 조밀한 제목 크기와 줄 간격을 사용한다.
- 구현 방향: 3단계 `ShippingConclusionCard`, 5단계 `ShippingShipmentHero`, `FinalRequirementReview` 행, 변경 구성 요약 카드에서 제목 폰트 크기, 상단 여백, 품명·품목코드 간 margin 및 line-height를 함께 낮춘다. 품명은 최대 두 줄, 품목코드는 다음 줄 고정 규칙을 유지한다.
- 수용 기준:
  - 새 PA/PF 생성 예정과 기존 PA/PF 재사용 카드에서 제목·품명·품목코드가 과도한 빈 간격 없이 읽힌다.
  - 5단계 `BOM·동반 출하품` 행과 출하 품목 요약도 품명·품목코드 간격이 같은 조밀한 문법을 사용한다.
  - 품명과 품목코드는 기존처럼 서로 다른 줄에 표시된다.
  - 3단계와 5단계의 대응 요약 카드가 같은 텍스트 밀도를 사용한다.
- 브라우저 확인 시나리오:
  1. 새 PA/PF 이름을 입력한 요청과 기존 PA/PF 재사용 요청을 각각 3단계까지 진행한다.
  2. 제목·품명·품목코드의 줄 간격과 카드 높이를 비교한다.
  3. 5단계에서도 출하 품목·BOM·변경 구성 요약의 제목 및 행 간격이 같은 밀도로 보이는지 확인한다.
- 우선순위: 중간
