범위: PC 목록, 펼침 row, 우측 상세 패널, 승인자/요청자 표시 검증

## 목적

이 문서는 2026-07-01~2026-07-02에 반영한 PC 입출고 내역 목록 개선 이후에도 남은 어색함과 검증해야 할 지점을 계속 쌓는 TODO다.

판단 기준은 화면이 예쁜가가 아니라, 현장 사용자가 거래의 대상, 수량 영향, 처리 상태, 승인 흐름을 오해 없이 빠르게 읽을 수 있는가다.

## 현재 기준

- 출하/출고/BOM/재작업 목록 문법은 한 차례 정리됐다.
- 품목코드 별도 열, 흐름 설명 축소, 처리 후 재고 기준 위치 표시, 재작업 요약, 출하 대표 코드 표시가 반영된 상태를 기준으로 후속 문제를 본다.
- 수량 조정 승인자 표시 오류는 별도 수정이 들어갔지만, 화면과 API가 같은 결론을 말하는지 추가 확인이 필요하다.
- 현재 워킹트리에는 다른 세션 변경이 섞일 수 있으므로, 구현 시에는 history 관련 파일만 선별해서 봐야 한다.

### 1. 수량 조정 승인자 표시의 화면/API 재검증

- 상태: 검증 후보
- 사용자 관찰:
  - 김현우는 수량 조정 요청을 승인할 권한이 없는데, 입출고 내역에 `승인 김현우`가 표시됐다.
- 이미 확인한 원인:
  - 수량 조정 거래에서 실제 승인자 정보가 누락되면 프론트가 요청자를 승인자로 fallback 표시할 수 있었다.
  - 수량 조정 배치와 StockRequest 연결 정보가 부족하면 백엔드가 실제 부서 승인자를 transaction history 응답에 채우지 못할 수 있었다.
- 후속 확인:
  - 개발 서버 재시작 후 같은 거래가 `승인 김건호` 또는 승인자 없음 상태로 올바르게 보이는지 확인한다.
  - 목록 row, 우측 단건 상세, 우측 배치 상세가 같은 승인자 값을 표시하는지 확인한다.
  - CSV/XLSX export가 history list와 같은 승인자 규칙을 쓰는지 확인한다.
  - 과거에 생성된 로그 중 `reference_no` 또는 `stock_request_id`가 비어 있는 수량 조정 건이 여전히 잘못 표시될 수 있는지 확인한다.
- 개선 방향:
  - 승인자 데이터가 없으면 요청자를 대신 표시하지 않고 `승인 -` 또는 `승인자 없음`으로 둔다.
  - 자동/자가 처리와 별도 승인 처리를 UI 문구에서 구분할지 검토한다.
  - 권한이 없는 사용자가 승인자로 보이는 케이스는 regression test와 브라우저 확인 케이스로 남긴다.
- 검증 기준:
  - 김현우 요청, 김건호 승인 수량 조정 건에서 목록과 상세 모두 `요청 김현우 / 승인 김건호`로 보인다.
  - 승인 데이터가 없는 거래에서 요청자 이름이 승인자로 복제되지 않는다.
  - 권한상 승인자가 될 수 없는 사용자가 승인자로 표시되지 않는다.

### 2. 우측 상세 패널의 승인/요청 영역 문법 정리

- 상태: 후보
- 문제:
  - 단건 상세와 배치 상세에서 요청자/승인자 표시 문법이 서로 다를 수 있다.
  - 승인자가 없는 거래, 자가 처리 거래, 별도 승인 거래가 모두 `승인 -`처럼만 보이면 현장 사용자가 처리 상태를 헷갈릴 수 있다.
- 개선 방향:
  - 목록의 상태·처리 열과 우측 상세의 요청/승인 영역이 같은 용어를 쓰게 한다.
  - `승인자 없음`, `자동 처리`, `별도 승인`, `승인 대기`를 어떤 조건에서 표시할지 정책을 정한다.
  - 권한이 있는 승인자 이름은 강조하고, 승인 없음 상태는 약하게 표시한다.
- 검증 기준:
  - 수량 조정, 불량 처리, 출하, 생산, 이동 건에서 요청/승인 표시가 서로 다른 의미로 읽히지 않는다.

### 3. 우측 상세 패널의 첫 화면 결론성 강화

- 상태: 후보
- 문제:

  - 출하/생산/재작업/불량 처리도 같은 문장 구조를 쓰게 한다.
- 검증 기준:

### 4. 승인 권한과 화면 표시의 정책 연결

- 상태: 정책 확인 필요

- 검증 기준:

### 6. `출하 구성 N건` fallback 재발 원인: 출하 준비 묶음과 픽업 완료 묶음 구분

- 상태: 즉시 진단/수정 후보
- 사용자 관찰:
  - 2026-07-01 16:26 row가 다시 `출하 구성 18건`으로 표시된다.
  - 이전에 출하 대표명을 실제 대상명으로 보이게 잡았다고 생각했는데 fallback이 다시 보인다.
- 확인한 실제 데이터:
  - 화면의 2026-07-01 16:26은 DB 기준 `2026-07-01 07:26` 로그 묶음이다.
  - 해당 `SHIP-...` reference 묶음에는 `shipping_phase=PICKUP`인 `SHIP` 로그가 없고, `shipping_phase=PREPARE`인 `PRODUCE/BACKFLUSH` 로그만 있다.
  - 즉 실제 외부 출하 완료 묶음이 아니라, 출하 준비 과정에서 만든/차감한 품목 묶음이다.
- 현재 코드상 원인:
  - 프론트는 `reference_no`가 `SHIP-`로 시작하면 출하 묶음으로 분류한다.
  - 대표 대상명은 `출하 픽업 완료:` notes 또는 동반품이 아닌 `SHIP` 로그에서 찾는다.
  - 준비 단계 묶음에는 `SHIP` 로그가 없으므로 대표 출하 대상명을 찾지 못하고 `출하 구성 N건` fallback으로 떨어진다.
- 문제:
  - 준비 단계 묶음을 완성 출하처럼 `출하`라고만 표시하면 사용자가 실제 픽업 완료 출하와 혼동한다.
  - 반대로 `출하 구성 N건`은 구조명이라, 사용자가 무엇을 준비했는지 알기 어렵다.
  - 출하 탭의 업무 상태인 `준비 완료`와 입출고 내역의 `출하` 표현이 어긋난다.
- 개선 방향:
  - `SHIP-` reference 묶음을 `픽업 완료 출하`와 `출하 준비`로 나눈다.
    - `PICKUP` phase 또는 `SHIP` transaction이 있으면 `출하`
    - `PREPARE` phase만 있으면 `출하 준비`
  - 준비 묶음 대표명은 픽업 notes가 아니라 준비 결과 log에서 찾는다.
    - 우선순위: `shipping_phase=PREPARE` 중 최종 PF `PRODUCE` log -> 최종 PA `PRODUCE` log -> 첫 `PRODUCE` log -> fallback `출하 준비 N건`
  - 준비 묶음의 흐름 열은 `출하`가 아니라 `출하 준비` 또는 `준비`로 표시한다.
  - 수량·재고 열도 `출하 17품목 · 24 EA`처럼 외부 출하처럼 보이게 하지 말고, `준비 2품목 · 차감 16라인`처럼 준비/차감 의미를 분리한다.
  - 실제 픽업 완료 묶음만 `출하`와 `출하 -N EA` 문법을 쓴다.
- 구현 전 확인:
  - `TransactionLog.shipping_phase`가 history API 응답 타입에 포함되어 프론트까지 내려오는지 확인한다.
  - `getReferenceBatchPresentation`에서 `SHIP-` 여부만 보지 말고 logs 안의 `shipping_phase`와 `transaction_type` 조합을 함께 본다.
  - 준비 완료 후 픽업 완료까지 완료된 요청은 PREPARE/PICKUP 로그가 같은 reference에 함께 묶일 수 있으므로, 대표 row를 하나로 둘지 단계별로 나눌지 확인한다.
- 검증 기준:
  - PICKUP 로그가 없는 출하 준비 묶음은 `출하 구성 N건`이 아니라 실제 준비 대상명 또는 `출하 준비`로 보인다.
  - 실제 픽업 완료 출하 묶음은 기존처럼 실제 출하 대상명과 품목코드가 보인다.
  - 준비 묶음과 완료 출하 묶음이 같은 `출하` 용어로 뭉개지지 않는다.

### 7. `흐름`과 `수량·재고` 열의 의미 재설계

- 상태: 깊은 UX 검토 후보
- 사용자 관찰:
  - 많이 다듬어졌지만 아직 `흐름` 열과 `수량·재고` 열이 직관적이지 않다.
  - 줘야 할 정보는 많은 열인데, 디자인이 난잡한 문제인지, 한 단어로 설명을 잘 못하는 문제인지 고민이 필요하다.
- 현재 문제 진단:
  - `흐름` 열은 업무 의미와 위치 이동을 동시에 말하려 한다.
    - 예: `창고 -> 부서`, `수량 조정`, `출하`
  - `수량·재고` 열은 한 칸에서 변화량과 처리 후 잔량을 동시에 말한다.
    - 예: 첫 줄 `이동 2품목 · 2 EA`, 둘째 줄 `창고 400 EA`
  - 두 열을 나란히 읽으면 문장으로 연결되지 않고, 각각 맞는 조각 정보가 흩어져 보인다.
  - 특히 출하/출하 준비/이동/수량 조정처럼 업무 의미와 재고 위치가 다른 축일 때 어색함이 커진다.
- 핵심 질문:
  - `흐름` 열은 "업무 결과"를 말해야 하는가, "재고 위치 이동"을 말해야 하는가?
  - `수량·재고` 열은 "이번 거래 변화량"이 주 정보인가, "처리 후 잔량"이 주 정보인가?
  - 처리 후 잔량은 목록에서 항상 필요한가, 아니면 우측 상세/펼침 row에서만 필요한가?
- 개선안 A: 업무 중심 문법
  - 흐름 열은 업무 결과만 표시한다: `출하`, `출하 준비`, `생산`, `이동`, `수량 조정`, `불량 처리`.
  - 수량·재고 열은 이번 거래의 변화량을 크게 표시하고, 잔량은 작게 둔다.
  - 예: `출하 1 EA` / `창고 410 EA`
  - 장점: 한 단어로 빠르게 읽힌다.
  - 단점: 어디서 어디로 움직였는지 약해질 수 있다.
- 개선안 B: 재고 영향 중심 문법
  - 흐름 열은 위치 기준으로 표시한다: `창고 -> 외부`, `불량 -> 정상`, `창고 -> 조립`.
  - 수량·재고 열은 `-1 EA`, `+200 EA`, `창고 410 EA`처럼 재고 영향 중심으로 표시한다.
  - 예: `창고 -> 외부` / `-1 EA · 창고 410 EA`
  - 장점: 재고 장부 관점이 명확하다.
  - 단점: 출하 같은 업무 결과가 물류 경로처럼만 보인다.
- 개선안 C: 두 열을 한 문장처럼 맞추는 하이브리드
  - 흐름 열은 업무명, 수량·재고 열은 `변화량 + 기준 위치`를 붙여 문장처럼 읽히게 한다.
  - 예:
    - 흐름 `출하` / 수량 `-1 EA · 창고 410 EA`
    - 흐름 `출하 준비` / 수량 `완제품 +1 · 부품 -17`
    - 흐름 `수량 조정` / 수량 `+200 EA · 창고 400 EA`
    - 흐름 `불량 해제` / 수량 `정상 +1 · 불량 -1`
  - 장점: 업무 의미와 재고 영향을 모두 살린다.
  - 단점: 거래 유형별 요약 규칙을 더 세밀하게 만들어야 한다.
- 현재 추천:
  - 개선안 C를 우선 검토한다.
  - 흐름 열은 "업무명"으로 단순하게 만들고, 수량·재고 열은 "이번 변화량 + 기준 잔량"을 같은 문법으로 정리한다.
  - 처리 후 잔량은 항상 `처리 후`라고 쓰지 말고 `창고 410 EA`, `불량 398 EA`처럼 짧게 유지하되, 대표 묶음에서 기준이 모호하면 숨긴다.
  - 출하 준비 묶음처럼 여러 영향이 섞인 경우에는 단일 잔량보다 `완제품 +1 · 부품 -17` 같은 묶음 요약이 더 낫다.
- 구현 전 확인:
  - 현재 `FlowSummaryCell`, `QuantityStockCell`, `MovementSummaryCell`, `getStockPresentation`, `summarizeReferenceLogs`가 어떤 값을 만들어 내는지 유형별로 샘플 표를 만든다.
  - 출하, 출하 준비, 생산 BOM, 재작업, 수량 조정, 불량 처리, 불량 해제, 창고↔부서 이동의 표시 문법을 한 표로 먼저 확정한다.
  - 우측 상세 패널이 같은 정보를 더 자세히 풀어주는 구조인지 확인한다.
- 검증 기준:
  - 사용자가 `흐름 + 수량·재고` 두 열을 왼쪽에서 오른쪽으로 읽으면 하나의 문장처럼 이해된다.
  - 출하와 출하 준비가 서로 다른 의미로 보인다.
  - 변화량과 처리 후 잔량이 서로 다른 정보라는 점이 시각적으로 구분된다.
  - 일반 거래는 짧고, 복합 거래만 필요한 만큼 자세해진다.

### 8. 출하 준비 로그가 픽업 후 같은 묶음으로 재해석되는 문제

- 상태: 설계 결정 필요
- 사용자 질문:
  - 출하는 요청, 준비, 픽업 3단계다.
  - 요청 단계는 입출고 내역이 남지 않는 것이 자연스럽다.
  - 준비 단계에서는 입출고 내역이 남아야 하고, 픽업 단계에서도 입출고 내역이 남아야 한다.
  - 현재 화면은 준비 때 있던 내역을 픽업하면 수정하는 느낌인데 실제로 그런지 확인이 필요하다.
- 확인한 코드 기준 사실:
  - `prepare_complete`는 `reference_no = SHIP-{request_id}`로 `PREPARE` phase의 `PRODUCE/BACKFLUSH` 거래 로그를 새로 만든다.
  - `pickup_complete`는 기존 `PREPARE` 로그를 수정하지 않고, 같은 `reference_no = SHIP-{request_id}`로 `PICKUP` phase의 `SHIP` 거래 로그를 새로 만든다.
  - `buildGroups`는 `reference_no`가 같으면 phase와 관계없이 한 batch row로 묶는다.
  - 따라서 픽업 후에는 같은 row 묶음 안에 준비 로그와 픽업 로그가 함께 들어오고, 프론트 대표 row가 다시 계산되면서 준비 row가 출하 row로 바뀐 것처럼 보인다.
- 문제:
  - 데이터는 수정되지 않았지만 화면에서는 준비 내역이 픽업 내역으로 덮인 것처럼 느껴진다.
  - 준비 완료 시점의 재고 영향과 픽업 완료 시점의 재고 영향은 둘 다 입출고 내역에 남아야 하는 별도 사건이다.
  - 그런데 같은 `reference_no` 단위로 묶으면 두 사건의 시점과 의미가 섞인다.
- 설계 선택지:
  1. 같은 대표 row 안에서 `출하 준비` 섹션과 `픽업 출하` 섹션으로 나눈다.
     - 장점: 하나의 출하 요청/BOM 작업으로 읽힌다.
     - 단점: 준비 시점과 픽업 시점이 한 row에 합쳐져 시간순 내역 감각이 약해질 수 있다.
  2. 목록에서는 `출하 준비` row와 `출하` row를 별도 사건으로 분리하되, 펼치면 같은 출하 요청임을 보여준다.
     - 장점: 입출고 내역의 시간성과 재고 사건성이 명확하다.
     - 단점: 하나의 출하가 두 줄로 보여 목록이 길어진다.
  3. 대표 row는 하나로 유지하되, 준비만 된 상태와 픽업 완료 상태의 제목/흐름/수량 문법을 단계별로 다르게 보여준다.
     - 장점: 현재 구조 변경이 적다.
     - 단점: 사용자가 여전히 준비 내역이 픽업 후 바뀐 것처럼 느낄 수 있다.
- 현재 추천:
  - 출하가 BOM으로 한 번에 출하되는 업무 감각을 살리려면 1번이 가장 자연스럽다.
  - 다만 `입출고 내역`은 시간순 거래 로그이기도 하므로, 준비 단계와 픽업 단계의 시각적 섹션 분리는 반드시 필요하다.
  - 최소한 `reference_no`만으로 묶지 말고 `shipping_phase`를 대표 row/하위 row 문법에 반영해야 한다.
- 검증 기준:
  - 준비 완료 직후에는 `출하 준비`로 보이고, 픽업 완료 후에도 준비 내역이 사라지거나 수정된 것처럼 보이지 않는다.
  - 픽업 완료 후 펼침 안에서 `준비: 생산/차감`과 `픽업: 실제 출하`가 구분된다.
  - 사용자는 하나의 출하 요청이라는 연결감과 두 번의 재고 사건이라는 시간성을 동시에 이해할 수 있다.

### 9. 출하 준비/픽업 펼침에서 동반품 차감 시점 결정

- 상태: 정책 결정 필요
- 사용자 제안:
  - 대표 row 1: `출하 준비 완료` / 대상 `tset` / 펼침: PA 생산, PF 생산, 하위 차감, 동반품 차감
  - 대표 row 2: `출하 픽업 완료` / 대상 `tset` / 펼침: 최종 PF 출하
- 확인한 현재 코드 기준:
  - `prepare_complete`는 PA/PF 생산과 BOM 하위 차감 로그만 만든다.
  - `pickup_complete`는 최종 PF 출하와 `companion_lines` 동반품 출하 로그를 만든다.
  - 출하 탭 확인 문구도 준비 완료는 `하위 자재를 반영하고 최종 PA/PF 생산 로그`, 픽업 완료는 `최종 PF와 동반 출하품을 출하 처리`로 설명한다.
- 판단:
  - 동반품을 실제로 준비 단계에서 창고/부서 재고에서 빼는 업무라면 사용자 제안이 맞다.
  - 반대로 동반품은 픽업 시 최종 PF와 함께 외부로 나가는 재고라면 현재 코드 정책이 맞고, 준비 펼침에는 동반품을 재고 차감으로 표시하면 안 된다.
  - 입출고 내역은 실제 재고 이동 내역이므로, 재고가 움직이지 않은 동반품을 `차감`으로 보여주면 장부와 화면이 어긋난다.
- 설계 후보:
  1. 동반품 차감 시점을 준비 완료로 앞당긴다.
     - 준비 row 펼침: PA 생산, PF 생산, 하위 차감, 동반품 차감
     - 픽업 row 펼침: 최종 PF 출하
     - 필요 변경: backend `prepare_complete`/`pickup_complete` 재고 처리 정책 변경, 취소/되돌림 정책 재검토
  2. 현재 재고 정책은 유지하고, 동반품은 픽업 완료 row에 둔다.
     - 준비 row 펼침: PA 생산, PF 생산, 하위 차감
     - 픽업 row 펼침: 최종 PF 출하, 동반품 출하
     - 필요 변경: frontend history 표시 문법 정리 중심
  3. 준비 row에는 동반품을 `출하 예정/준비 품목`으로만 보여주고 재고 차감은 픽업 row에 둔다.
     - 장점: 작업자는 준비 단계에서 동반품 존재를 볼 수 있다.
     - 단점: 입출고 내역에 재고 영향 없는 정보를 섞게 되어 원장성이 약해진다.
- 현재 추천:
  - 재고 원장 관점만 보면 2번이 안전하다.
  - 현장 실제 업무가 `준비 완료` 시점에 동반품을 물리적으로 빼놓는 것이라면 1번으로 정책 변경하는 것이 더 맞다.
  - 이 결정은 UI 문제가 아니라 출하 탭 기능/재고 차감 정책 문제이므로 구현 전 사용자 확정이 필요하다.
- 검증 기준:
  - 준비 row와 픽업 row가 실제 재고 차감 시점과 일치한다.
  - 동반품이 두 번 차감되거나, 준비에는 보이는데 재고 영향은 없는 것처럼 혼동되지 않는다.
  - 준비 취소 시 동반품 차감도 함께 되돌릴지 정책이 명확하다.

### 10. 출하 준비 완료 후 `가용 재고`에서 제외되어야 하는 출하 예약 상태 설계

- 상태: 구조/정책 설계 필요
- 사용자 혼란:
  - 출하 준비 완료 후 PF와 동반품은 회사 안에 있으므로 총재고에는 잡혀야 한다.
  - 하지만 이미 특정 출하 요청에 포장/예약된 품목이므로 다른 작업의 가용 수량으로 잡히면 안 된다.
  - 픽업 완료 시 그 예약된 재고가 실제 차감되고, 준비 취소 시 하위 자재까지 원복되어야 한다.
- 확인한 현재 코드 기준:
  - `Inventory.pending_quantity`는 실제로 사용 중이다. StockRequest 승인 대기/예약에서 `reserve`, `release`, `consume_pending`이 호출된다.
  - 다만 이 pending은 주석과 제약상 `warehouse_qty` 대비 예약이다. `warehouse_qty >= pending_quantity` 제약이 있고, `consume_pending`도 warehouse를 차감한다.
  - 출하 준비/픽업 플로우는 현재 이 pending을 사용하지 않는다.
  - 출하 준비 완료 시 PA/PF 생산과 하위 차감은 실제 재고 이동으로 기록된다.
  - PA는 보통 PF 생산에 바로 투입되어 최종 가용으로 남지 않고, 최종 PF가 부서 PRODUCTION 위치에 남는다.
  - 동반품은 준비 단계에서 차감/예약되지 않고, 픽업 완료 때 `consume_from_item_department`로 실제 출하 차감된다.
- 현재 문제:
  - 준비 완료 후 최종 PF는 실제 재고로 생기지만 특정 출하 요청용이므로 일반 가용으로 잡히면 위험하다.
  - 동반품도 포장/예약 상태라면 총재고에는 남되 일반 가용에서는 빠져야 한다.
  - 기존 `pending_quantity`는 창고 예약 전용이라, 부서 PRODUCTION 위치의 최종 PF나 동반품 예약을 그대로 표현하기 어렵다.
- 설계 방향 후보:
  1. 기존 `pending_quantity`를 확장해서 출하 예약에도 쓴다.
     - 장점: available 계산에 이미 반영되어 있다.
     - 단점: 현재 제약이 warehouse 기준이라 부서 위치 재고 예약과 맞지 않는다.
  2. 출하 전용 `reservation/allocation` 모델을 만든다.
     - 예: `ShippingInventoryReservation` 또는 범용 `InventoryReservation`
     - 필드: `item_id`, `quantity`, `bucket`, `department`, `status`, `shipping_request_id`, `phase`, `created_at`, `released_at/consumed_at`
     - available 계산에서 총 예약 수량을 제외한다.
     - 픽업 완료 시 예약을 실제 차감으로 전환하고, 준비 취소 시 예약을 해제한다.
  3. UI에서만 `출하 대기`로 표시하고 가용 계산은 바꾸지 않는다.
     - 장점: 구현이 작다.
     - 단점: 실제 다른 작업에서 그 수량을 사용할 수 있어 재고 충돌을 막지 못한다.
- 현재 추천:
  - 장기적으로는 2번이 맞다.
  - `pending_quantity`는 이미 쓰이고 있지만 창고 예약 전용이므로, 출하 준비품처럼 부서 위치/목적이 있는 예약까지 넣기에는 모델이 좁다.
  - 단기 구현을 한다면 출하 준비 완료 화면에는 `출하 예약 필요`를 먼저 표시하고, 실제 차감 정책 변경은 별도 구조 설계 후 진행한다.
- 기대 생명주기:
  - 요청/준비 중: 재고 변화 없음.
  - 준비 완료: 하위 자재 차감, 최종 PF 생산, 최종 PF와 동반품을 출하 예약으로 묶어 가용에서 제외.
  - 픽업 완료: 예약된 최종 PF와 동반품을 실제 출하 차감 처리.
  - 준비 취소: 준비 로그 reverse, 출하 예약 해제, 최종 PF/하위 자재 상태 원복.
- 검증 기준:
  - 총재고에는 출하 준비품이 남아 있지만 available에는 잡히지 않는다.
  - 다른 출고/생산/불량 처리에서 출하 예약 수량을 사용할 수 없다.
  - 픽업 완료와 준비 취소가 예약 수량을 남기지 않는다.

### 11. 출하 준비 완료 시 PA/PF 재고 흐름과 기존 PA 재고 사용 여부 재검토

- 상태: 정책 검토 필요
- 사용자 관찰:
  - 이미 PA까지 만들어둔 것을 사용할 수도 있는데, 현재 준비 완료 로직은 매번 PA 생산처럼 보인다.
  - 준비 완료 때 PF와 동반품이 출하 예약/가용 제외 상태가 되어야 한다는 방향은 맞아 보인다.
- 확인한 현재 코드 기준:
  - `_resolve_or_create_pa`는 동일 BOM의 PA 품목 정의를 찾거나 새 품목을 만든다. 기존 PA 재고를 사용할지 결정하는 로직은 아니다.
  - `prepare_complete`는 항상 최종 PA를 `PRODUCE +request_qty`로 입고한 뒤, PF BOM 구성에서 최종 PA를 `BACKFLUSH -request_qty`로 소비한다.
  - 일반적인 PF BOM에서 PA 수량이 1이면 PA 재고는 준비 완료 후 순증하지 않고, 최종 PF만 남는다.
  - 이미 만들어둔 PA 재고가 있어도 현재 기본 흐름은 그 PA 재고를 우선 사용하는 것이 아니라, 생산 로그와 소비 로그를 한 쌍으로 남긴다.
- 검토 질문:
  - 출하 준비는 항상 `PA 생산 -> PF 생산`이어야 하는가?
  - 이미 가용 PA가 있을 때는 PA 생산을 생략하고 기존 PA를 PF에 투입해야 하는가?
  - 이 경우 출하 준비 row의 펼침에는 `PA 사용`과 `PA 생산`을 구분해야 하는가?
- 검증 기준:
  - 준비 완료 후 PA/PF/동반품의 총재고와 가용재고가 업무 의도와 일치한다.
  - 이미 만들어둔 PA를 쓰는 정책을 도입할 경우, history row가 `PA 생산`이라고 잘못 말하지 않는다.

### 12. BOM 변경 출하에서 기존 PA를 재구성하는 전용 흐름 필요

- 상태: 구조 설계 필요
- 사용자 질문:
  - 출하 요청에서 기존 BOM을 가져오되, 출하 건에 맞게 BOM이 바뀌면 새 PA/PF 품목이 생성된다.
  - 현장에서는 이미 만들어둔 PA에서 부속품을 추가하거나 제외하고 다시 포장하는 경우가 있다.
  - 현재 프로그램 기능으로는 분해 출고로 기존 PA를 분해하고, 새 PA로 생산 입고를 잡아야 하는지 의문이다.
- 확인한 현재 기능:
  - 출하 탭의 BOM 매칭은 동일 BOM의 PA/PF 품목 정의를 찾거나 새 품목 정의를 만든다.
  - 출하 준비 완료는 기존 PA 재고를 재구성하지 않고, PA 하위자재 차감 -> PA 생산 -> PA 투입 -> PF 생산 흐름으로 기록한다.
  - 창고/부서 입출고에는 `분해(disassemble)`와 `생산(produce)` BOM 작업이 있다.
- 기존 기능으로 가능한 우회:
  - 기존 PA를 `분해`해서 하위 구성품을 회수한다.
  - 변경된 BOM 기준 새 PA를 `생산`한다.
  - 이후 출하 준비/픽업을 진행한다.
- 우회 방식의 문제:
  - 실제 현장은 PA를 완전히 분해하는 것이 아니라 일부 부속품을 추가/제외하는 재구성에 가깝다.
  - 완전 분해 -> 재생산은 장부상 정합성은 맞출 수 있지만, 현장 작업 기록이 과장된다.
  - 부분 변경분만 차감/회수해야 하는데 현재 기능은 delta 재구성 문법이 없다.
- 개선 방향:
  - 출하 준비에 `기존 PA 사용/재구성` 선택지를 추가한다.
  - 후보 흐름:
    1. 새 PA 생산: 기존처럼 하위자재 차감 -> PA 생산
    2. 기존 PA 사용: PA 재고 차감 -> PF 생산
    3. 기존 PA 재구성: 기존 PA 차감 + 변경분 부품 추가/회수 -> 새 PA 생산 또는 바로 PF 투입
  - 변경 전 PA BOM과 변경 후 PA BOM의 delta를 계산한다.
    - 추가된 부품: 차감
    - 제외된 부품: 회수
    - 수량 변경: 차이만 차감/회수
  - 입출고 내역에는 `PA 재구성`으로 표시하고, 펼침에는 `기존 PA 투입`, `추가 부품 차감`, `제외 부품 회수`, `최종 PA/PF 준비`를 보여준다.
- 검증 기준:
  - 현장 작업이 완전 분해가 아닌 부분 재구성일 때도 장부와 화면이 거짓말하지 않는다.
  - 기존 PA 재고를 사용하면 PA 수량이 실제로 줄어든다.
  - 변경분 부품만 추가 차감/회수된다.
  - 새 PA 생산, 기존 PA 사용, 기존 PA 재구성이 입출고 내역에서 서로 구분된다.

### 13. 출하 BOM 변경 작업명은 `기존 PA 재구성`보다 `구성품 변경`으로 표현

- 상태: 용어/흐름 확정 후보
- 사용자 판단:
  - `기존 PA 재구성`이라는 이름은 개발자/시스템 말투에 가깝다.
  - 현장에서는 `구성품 변경` 정도가 더 자연스럽다.
  - 기존 PA에서 구성품을 변경해서 다른 PA가 되는 작업으로 보는 것이 맞다.
- 용어 정의:
  - `구성품 변경` = 기존 PA를 기준으로 일부 구성품을 추가, 제외, 수량 변경해서 변경 후 PA 상태로 전환하는 작업.
  - 완전 분해 후 재생산이 아니라, 기존 PA를 바탕으로 변경분만 반영하는 현장 작업이다.
- 장부 흐름:
  - 기존 PA `-1`
  - 추가된 구성품 `-N`
  - 제외된 구성품 `+N`
  - 변경 후 PA `+1`
  - 변경 후 PA가 PF에 투입되면 변경 후 PA `-1`, PF `+1`
- 입출고 내역 표현 후보:
  - 작업 열: `구성품 변경`
  - 대상: 변경 후 PA 이름
  - 펼침:
    - 기존 PA 사용 `-1`
    - 추가 구성품 차감
    - 제외 구성품 회수
    - 변경 후 PA 입고 `+1`
- 출하 준비 완료와의 연결:
  - `구성품 변경`은 출하 준비 완료 이전 또는 준비 과정의 하위 작업으로 발생한다.
  - 출하 준비 완료 row에서는 변경 후 PA 투입, PF 생산, 동반품 출하 예약을 보여준다.
  - 하나의 출하 요청 안에서 `구성품 변경`과 `출하 준비 완료`가 어떻게 묶여 보여야 하는지 추가 설계가 필요하다.
- 구현 전 확인:
  - 출하 탭에서 BOM 변경 시 변경 전 PA와 변경 후 PA를 모두 식별할 수 있어야 한다.
  - 기존 PA 재고가 충분하지 않을 때는 `구성품 변경`을 할 수 없거나, 부족분만 새 PA 생산으로 보완하는 정책이 필요하다.
  - 변경 전/후 BOM delta 계산이 필요하다.
- 검증 기준:
  - 작업자가 실제로 한 일이 완전 분해가 아닌 구성품 변경이라면 history에도 `분해`가 아니라 `구성품 변경`으로 남는다.
  - 기존 PA 재고, 추가/제외 구성품, 변경 후 PA 재고가 모두 장부와 일치한다.
  - 출하 준비/픽업 흐름과 연결되어도 같은 재고가 중복 차감되지 않는다.

### 14. 출하 요청 단계에서 새 PA/PF 품목이 생성되어야 함

- 상태: 도메인 결정 정정
- 사용자 정정:
  - 출하 요청 단계에서 BOM 변경 결과에 맞는 새 PA/PF 품목이 생성되는 것이 맞다.
  - 준비 완료 단계는 이미 정해진 새 PA/PF 품목을 실제 재고로 준비하는 단계다.
- 현재 코드 확인:
  - `create_request`는 요청, BOM 라인, 동반품을 저장한다.
  - `prepare_complete`에서 `_resolve_or_create_pa`, `_resolve_or_create_pf`를 호출하고 `final_pa_item_id`, `final_pf_item_id`를 채운다.
- 문제:
  - 업무 의미상 "출하 요청 대상"은 요청 생성 시점부터 변경 후 PF여야 한다.
  - 현재 구현처럼 준비 완료 시점에 final PA/PF가 정해지면, 요청 목록/준비 목록/입출고 내역에서 대상 품목명이 뒤늦게 바뀌는 것처럼 보일 수 있다.
- 개선 방향:
  - 요청 생성 또는 요청 확정 시 BOM signature 기준으로 final PA/PF를 resolve/create 한다.
  - `ShippingRequest.final_pa_item_id`, `final_pf_item_id`는 요청 단계부터 보유한다.
  - `prepare_complete`는 품목 생성이 아니라 재고 작업만 수행한다.
  - 같은 BOM 품목이 이미 있으면 기존 PA/PF 품목 정의를 재사용하고, 없으면 사용자 입력 이름으로 새 품목 정의를 생성한다.
- 입출고 내역 기대:
  - 요청 단계 자체는 재고 로그 없음.
  - 준비 완료 row 대상은 요청 단계에서 이미 정해진 final PF.
  - 펼침에는 final PA/PF 준비 과정, 하위 차감, 동반품 예약/준비가 표시된다.

### 15. 출하 요청 단계 PA/PF 생성 시 재고 수량 정책

- 상태: 도메인 결정 확정
- 결정:
  - 출하 요청 단계에서는 변경 BOM 기준의 새 PA/PF `품목 정의`만 생성한다.
  - 재고 수량은 0으로 유지한다.
  - 실제 PA/PF 재고 증감은 `출하 준비 완료` 단계에서만 발생한다.
- 이유:
  - 요청은 아직 현장 작업이 끝난 상태가 아니므로 재고가 생기면 안 된다.
  - 준비 완료가 되어야 하위 자재 차감, PA/PF 생산, 동반품 예약/준비가 장부에 반영된다.
  - 이렇게 해야 입출고 내역에서 `요청`, `준비 완료`, `픽업 완료`의 의미가 분리된다.
- 기대 흐름:
  - 요청 생성: final PA/PF 품목 정의 생성 또는 기존 품목 정의 매칭, 재고 로그 없음.
  - 준비 완료: final PA/PF 준비 재고 로그 생성.
  - 픽업 완료: final PF 및 동반품 출하 차감 로그 생성.
- 구현 시 주의:
  - `prepare_complete`는 더 이상 PA/PF 품목을 새로 결정하는 주 책임을 갖지 않아야 한다.
  - `prepare_complete`는 요청 단계에서 이미 확정된 `final_pa_item_id`, `final_pf_item_id`를 사용해 재고 작업만 수행한다.
  - 요청 수정으로 BOM이 바뀌면 final PA/PF 품목 정의도 다시 resolve/create 되어야 한다. 단, 이미 준비 중/준비 완료 이후에는 수정 정책을 별도로 잠가야 한다.

### 16. 출하 준비 완료 취소 정책

- 상태: 기존 기능 재정의 필요
- 현재 코드 확인:
  - `prepare_cancel`은 `PREPARED` 상태에서만 실행된다.
  - `shipping_phase=PREPARE`인 입출고 로그를 찾아 `inventory_effect`를 역분개하고 로그를 취소 처리한다.
  - 이후 요청 상태를 `PREPARING`으로 되돌리고 `prepared_at`을 비운다.
  - 현재 구현은 `final_pa_item_id`, `final_pf_item_id`도 `None`으로 비운다.
- 새 도메인 정책과의 충돌:
  - 출하 요청 단계에서 final PA/PF 품목 정의가 생성되는 정책으로 가면, 준비 취소는 품목 정의를 취소하는 작업이 아니다.
  - 준비 취소는 실제 재고 작업만 되돌리는 작업이다.
  - 따라서 준비 취소 시 `final_pa_item_id`, `final_pf_item_id`는 유지되어야 한다.
- 준비 취소가 되돌려야 할 것:
  - PA/PF 생산 로그 역분개
  - 하위 자재 차감 로그 역분개
  - 동반품 출하 예약/가용 제외가 생기면 예약 해제
  - 요청 상태 `PREPARING` 복귀
- 준비 취소가 되돌리면 안 되는 것:
  - 요청 자체
  - 요청 단계에서 확정된 BOM 라인
  - 요청 단계에서 생성 또는 매칭된 final PA/PF 품목 정의
- 입출고 내역 표현:
  - 원 준비 완료 로그는 취소됨으로 표시한다.
  - 취소 사유가 있으면 상세에서 보여준다.
  - 같은 요청을 다시 준비 완료하면 새 준비 완료 묶음이 생겨야 한다.
- 추가 설계 필요:
  - 픽업 완료 후 취소는 `준비 취소`로 볼 수 없다. 이미 외부 출하가 끝난 거래이므로 반품/출하 취소/역출하 같은 별도 정책이 필요하다.

### 17. 준비 취소 범위 정정: 픽업 취소가 아니라 준비물 취소

- 상태: 이전 논의 정정
- 사용자 의도:
  - 픽업 완료 후 외부 출하를 되돌리는 이야기가 아니다.
  - 준비를 해둔 상태(`PREPARED`)에서, 아직 픽업되지 않은 준비 작업을 취소하는 기능이다.
- 정확한 업무 의미:
  - 출하 요청은 유지한다.
  - 요청 단계에서 확정된 final PA/PF 품목 정의도 유지한다.
  - 준비 완료로 인해 생긴 재고 변동만 되돌린다.
  - 상태는 다시 `PREPARING`으로 돌아간다.
- 되돌릴 대상:
  - PA/PF 준비 생산 로그
  - 하위 자재 차감 로그
  - 동반품 출하 예약/가용 제외가 도입되면 그 예약 해제
- 되돌리지 않을 대상:
  - 출하 요청 row
  - 변경 BOM
  - final PA/PF 품목 정의
  - 요청 수량/요청자/메모
- 구현 체크:
  - 현재 `prepare_cancel`의 큰 방향은 맞다.
  - 단, 새 정책에서는 `final_pa_item_id`, `final_pf_item_id`를 비우면 안 된다.
  - 입출고 내역에서는 준비 완료 묶음이 취소됨으로 보이고, 재준비 시 새 준비 완료 묶음이 생겨야 한다.

### 18. 출하 준비 완료 시 PA 공급 방식 결정 필요

- 상태: 정책 논의 중
- 현재 코드 확인:
  - `prepare_complete`는 PA 하위자재를 차감한다.
  - 그 다음 final PA를 생산 입고한다.
  - PF 생산 단계에서 final PA를 다시 투입 차감한다.
  - 결과적으로 final PA 수량은 `+Q -Q`라서 순증감 0이다.
- 문제:
  - 현장에서 이미 만들어진 PA를 실제로 사용한 경우, 장부상 기존 PA 재고가 줄지 않는다.
  - 동시에 PA 하위자재가 다시 차감되어 이중 차감처럼 보일 수 있다.
  - 따라서 `준비 완료 시 PA는 무조건 만들어져 있는 PA에서 차감`도 정답은 아니다.
- 권장 정책:
  - 준비 완료에는 PA 공급 방식을 구분해야 한다.
  - 새 PA 생산: 하위자재 차감 -> final PA 생산 -> final PA 투입 -> final PF 생산.
  - 기존 final PA 사용: final PA 재고 차감 -> final PF 생산. 하위자재는 차감하지 않음.
  - 구성품 변경: 기존 PA 차감 + 변경분 구성품 추가/회수 -> final PA 준비 -> final PF 생산.
- 판단 기준 후보:
  - final PA 재고가 충분하고 작업자가 기존 PA 사용을 선택하면 `기존 final PA 사용`.
  - base PA를 가져와 BOM 변경을 반영하면 `구성품 변경`.
  - 사용할 PA 재고가 없으면 `새 PA 생산`.
- 입출고 내역 기대:
  - 세 방식은 모두 `출하 준비 완료` 아래 하위 흐름으로 묶이되, 하위 row 라벨은 `새 PA 생산`, `기존 PA 사용`, `구성품 변경`으로 구분되어야 한다.

### 19. 출하 준비 완료는 만들어져 있는 PA를 소비하는 방식으로 단순화

- 상태: 정책 방향 강하게 유력
- 사용자 판단:
  - 준비 완료 시 PA를 새로 생산하는 것보다, 이미 만들어져 있는 PA에서 차감하는 방식이 가장 확실하다.
- 정책 의미:
  - `출하 준비 완료`는 PA를 만드는 작업이 아니다.
  - `출하 준비 완료`는 이미 존재하는 final PA 재고를 PF에 투입해 final PF를 준비하는 작업이다.
  - 따라서 준비 완료 시 final PA 재고가 부족하면 준비 완료를 막는다.
- 기대 장부 흐름:
  - 요청 단계: 변경 BOM 기준 final PA/PF 품목 정의 생성, 재고 0.
  - 준비 전 작업: 필요한 final PA를 일반 생산 또는 `구성품 변경`으로 미리 만들어 둔다.
  - 준비 완료: final PA `-Q`, final PF `+Q`, 동반품 예약/가용 제외.
  - 픽업 완료: final PF `-Q`, 동반품 실제 출하 차감.
- 장점:
  - PA 실재고와 장부가 일치한다.
  - 준비 완료 로그가 `PA 생산까지 포함한 복합 작업`이 아니라 `출하용 PF 준비`로 단순해진다.
  - BOM 변경이 필요한 경우 `구성품 변경`이라는 별도 작업으로 분리되어 추적이 쉬워진다.
- 현재 코드와의 차이:
  - 현재 `prepare_complete`는 PA 하위자재 차감 -> PA 생산 -> PA 투입 -> PF 생산을 한 번에 수행한다.
  - 새 정책에서는 `prepare_complete`에서 PA 하위자재 차감과 PA 생산을 제거하고, final PA 차감만 수행해야 한다.
- 입출고 내역 기대:
  - `구성품 변경` row가 필요한 경우 출하 준비 완료 이전에 별도 기록된다.
  - `출하 준비 완료` row 펼침에는 final PA 투입, final PF 준비, 동반품 예약만 표시된다.
  - 하위자재 차감은 출하 준비 완료 밑에 직접 섞이지 않는다. 하위자재 차감은 PA 생산 또는 구성품 변경의 하위 내역으로 남는다.
- 남은 확인:
  - final PA 재고가 없을 때 자동 생산으로 이어갈지, 아니면 준비 완료를 막고 별도 PA 생산/구성품 변경을 먼저 하게 할지 최종 확정이 필요하다.

### 20. `구성품 변경` 화면 신규 설계 필요

- 상태: 신규 기능 후보 확정
- 배경:
  - 출하 요청 단계에서 변경 BOM 기준 final PA/PF 품목 정의는 이미 생성된다.
  - 출하 준비 완료는 만들어져 있는 final PA를 차감해서 final PF를 준비하는 방식으로 가는 것이 유력하다.
  - 따라서 final PA 재고가 부족하면 준비 완료를 막고, 먼저 final PA를 확보해야 한다.
- 기능 의미:
  - `구성품 변경`은 기존 PA 재고를 가져와 변경 BOM 기준 final PA로 전환하는 작업이다.
  - 기존 PA 전체를 분해하는 작업이 아니라, BOM 차이만 추가/회수하는 작업이다.
- 화면 위치 후보:
  - 출하 탭의 `준비 중` 상세 화면 안에 `구성품 변경` 진입 버튼을 둔다.
  - 별도 메인 탭보다는 출하 요청 맥락 안에서 처리하는 것이 자연스럽다.
  - final PA 재고가 부족한 요청에서는 준비 완료 버튼 근처에 `구성품 변경 필요` 상태를 보여준다.
- 화면 입력:
  - 대상 출하 요청
  - 변경 후 final PA 품목
  - 사용할 기존 PA 품목
  - 변경 수량
- 화면이 보여줘야 할 비교:
  - 기존 PA BOM
  - 변경 후 final PA BOM
  - 추가로 차감할 구성품
  - 제외되어 회수될 구성품
  - 그대로 유지되는 구성품
- 장부 흐름:
  - 기존 PA `-Q`
  - 추가 구성품 `-N`
  - 제외 구성품 `+N`
  - 변경 후 final PA `+Q`
- 출하 준비 완료와의 관계:
  - 구성품 변경 완료 후 final PA 재고가 생긴다.
  - 이후 출하 준비 완료에서 final PA `-Q`, final PF `+Q`가 발생한다.
  - final PA 재고가 부족하면 출하 준비 완료는 막는다.
- 입출고 내역 표현:
  - 작업 열: `구성품 변경`
  - 대상 열: 변경 후 final PA 품목
  - 펼침: 기존 PA 사용, 추가 구성품 차감, 제외 구성품 회수, 변경 후 PA 입고
  - 출하 준비 완료 묶음 안에 숨기지 않고 별도 작업으로 남긴다.
- 구현 메모:
  - 새 transaction type 또는 기존 재작업/생산 타입으로 표현할지 정책 결정 필요.
  - source PA와 target final PA의 BOM delta 계산 helper가 필요하다.
  - 준비 완료 API는 final PA 재고 부족 시 422로 막아야 한다.

### 21. 구성품 변경 화면의 기존 PA 자동 추천 기준

- 상태: 추천 기준 설계
- 핵심 원칙:
  - 구성품 변경에서 추천할 기존 PA는 `이미 준비되어 있는 아무 PA`가 아니다.
  - 출하 요청의 기준 PF에 원래 들어 있던 PA를 1순위 source PA로 추천한다.
- 코드 근거:
  - 출하 요청은 `base_pf_item_id`를 가진다.
  - 기존 PF BOM 안의 PA 라인은 `_pf_lines_with_final_pa`에서 final PA로 치환되는 대상이다.
  - 즉 기준 PF의 PF-stage BOM 안에 있는 PA가 "원래 이 PF를 구성하던 PA"다.
- 추천 순서:
  - 1순위: 기준 PF의 직계 BOM 중 `process_type_code = PA`인 품목이며, 가용 재고가 요청 수량 이상인 PA.
  - 2순위: 같은 기준 PF의 PA지만 가용 재고가 부족한 경우, 부족 수량을 표시하고 선택은 막거나 경고한다.
  - 3순위: 같은 model_symbol의 PA 중 final PA와 BOM delta가 가장 작은 품목이며, 가용 재고가 있는 PA.
  - 추천 후보가 없으면 자동 추천 없음. 작업자가 PA 생산 또는 다른 PA 선택을 먼저 해야 한다.
- 화면 표시:
  - 기본 선택값: `기준 PF의 기존 PA`.
  - 함께 표시할 정보: 현재 가용 수량, 요청 수량, 변경 후 final PA, 추가/회수될 구성품 delta.
  - 작업자는 필요하면 source PA를 바꿀 수 있어야 한다.
- 주의:
  - 자동 추천은 편의 기능이지 장부 확정이 아니다.
  - 실제 `구성품 변경 실행` 시점에는 source PA 가용 수량을 다시 검증해야 한다.

### 22. 구성품 변경 source PA는 작업자가 직접 선택

- 상태: 도메인 결정 확정
- 결정:
  - 구성품 변경 화면에서 기존 PA는 자동 추천하지 않는다.
  - 작업자가 실제로 사용할 기존 PA 품목을 직접 선택한다.
- 이유:
  - 현장에서 어떤 PA를 실제로 꺼내 수정하는지는 시스템이 확정적으로 알 수 없다.
  - 기준 PF의 기존 PA가 항상 실제 사용 PA라는 보장이 없다.
  - 자동 추천은 편해 보이지만, 잘못 추천되면 장부가 현장과 어긋난다.
- 화면 정책:
  - source PA 검색/선택은 필수 입력이다.
  - 선택한 source PA의 가용 재고와 요청 수량을 바로 보여준다.
  - source PA 재고가 부족하면 구성품 변경 실행을 막는다.
  - target final PA는 출하 요청에서 이미 확정된 품목으로 고정한다.
  - 화면은 source PA BOM과 target final PA BOM의 차이를 계산해 추가/회수 구성품을 보여준다.
- 입출고 내역 기대:
  - `구성품 변경` row에는 작업자가 선택한 source PA와 target final PA가 모두 추적 가능해야 한다.
  - 대표 대상은 target final PA로 표시한다.
  - 펼침에는 source PA 차감, 추가 구성품 차감, 제외 구성품 회수, target final PA 입고가 표시된다.
- 이전 후보 폐기:
  - 기준 PF의 기존 PA 자동 추천은 폐기한다.
  - 같은 모델/BOM 유사도 기반 추천도 이번 범위에서는 제외한다.

### 23. 구성품 변경은 source PA 재고를 target PA 재고로 전환하는 작업

- 상태: 도메인 의미 확정
- 사용자 설명:
  - 작업자가 source PA를 직접 선택해야, 그 source PA의 재고를 변경할 PA 재고로 옮길 수 있다.
- 용어 정리:
  - 내부 장부 의미는 `source PA -> target final PA 전환`이다.
  - 화면 문구는 `이동`보다 `구성품 변경` 또는 `전환`이 맞다.
  - `이동`이라고 쓰면 창고/부서 이동처럼 오해될 수 있다.
- 장부 흐름:
  - source PA `-Q`
  - BOM 차이 중 추가된 구성품 `-N`
  - BOM 차이 중 제외된 구성품 `+N`
  - target final PA `+Q`
- 중요한 해석:
  - source PA와 target PA는 서로 다른 품목일 수 있다.
  - 따라서 단순 위치 이동이 아니라 품목 변환이다.
  - 변경 후 target final PA 재고가 생겨야 출하 준비 완료에서 target final PA를 소비할 수 있다.
- 화면 표현:
  - 대표 작업명: `구성품 변경`
  - 대상: target final PA
  - 보조 정보: source PA -> target final PA
  - 펼침: source PA 사용, 추가 구성품 차감, 제외 구성품 회수, target PA 입고
- 검증 기준:
  - source PA 가용 재고가 부족하면 실행 불가.
  - 추가 구성품 재고가 부족하면 실행 불가.
  - 실행 후 source PA는 줄고 target final PA는 늘어야 한다.

### 24. 구성품 변경 화면 상세 방향: source PA와 target PA BOM 비교 후 일부 수량 전환

- 상태: 사용자 요구 구체화
- 화면 목표:
  - 작업자가 source PA를 선택한다.
  - 좌측에는 source PA와 source PA BOM을 표시한다.
  - 우측에는 변경 후 target final PA와 target PA BOM을 표시한다.
  - 두 BOM 차이를 화면에서 바로 확인한다.
  - source PA 재고 중 몇 개를 target PA로 구성품 변경할지 수량을 입력한다.
  - 완료하면 source PA 재고가 target PA 재고로 전환되고, BOM 차이에 맞게 하위 품목 재고도 변동된다.
- BOM 비교 규칙:
  - source에는 없고 target에 있는 구성품: 추가 구성품, 실행 시 차감.
  - source에는 있고 target에는 없는 구성품: 제외 구성품, 실행 시 회수 입고.
  - 양쪽에 있지만 수량이 증가한 구성품: 증가분만 추가 차감.
  - 양쪽에 있지만 수량이 감소한 구성품: 감소분만 회수 입고.
  - 양쪽 수량이 같은 구성품: 유지, 재고 변동 없음.
- 실행 수량 Q:
  - 작업자는 source PA 재고 중 몇 개를 target PA로 바꿀지 입력한다.
  - 기본값은 출하 요청 수량 또는 source PA 가용 수량 중 작은 값으로 둘 수 있다. 최종 정책 확인 필요.
  - 모든 delta는 `BOM 차이 * Q`로 계산한다.
- 재고 변동 예시:
  - source PA `-Q`
  - target final PA `+Q`
  - 추가 구성품 `-(추가 수량 * Q)`
  - 회수 구성품 `+(회수 수량 * Q)`
- 검증:
  - source PA 가용 재고가 Q보다 작으면 실행 불가.
  - 추가 차감 구성품의 가용 재고가 필요 수량보다 작으면 실행 불가.
  - 회수 구성품은 입고이므로 부족 검증 대상이 아니다.
  - target final PA는 출하 요청에서 확정된 품목으로 고정한다.
- 입출고 내역 신규 표현:
  - 대표 row 작업: `구성품 변경`
  - 대표 대상: target final PA
  - 보조 흐름: source PA -> target final PA
  - 하위 row 역할: source PA 사용, 추가 구성품 차감, 제외/감소 구성품 회수, target PA 입고
  - 기존 생산 BOM/출하 준비와 별도 묶음으로 보여준다.
- 구현 영향:
  - source PA와 target PA의 BOM diff를 계산하는 backend helper가 필요하다.
  - 구성품 변경 실행 API가 필요하다.
  - TransactionLog에 구성품 변경 묶음을 구분할 reference/grouping 값이 필요하다.
  - 입출고 내역에는 `구성품 변경` batch 표현이 새로 필요하다.

### 25. 구성품 변경 BOM 비교 범위는 PA 직계 BOM만 사용

- 상태: 도메인 결정 확정
- 결정:
  - source PA와 target final PA의 BOM 비교는 직계 BOM만 대상으로 한다.
  - 하위 BOM 재귀 전개는 하지 않는다.
- 이유:
  - 구성품 변경은 PA 안에 직접 들어간 구성품을 바꾸는 현장 작업이다.
  - 하위 BOM까지 전개하면 작업자가 실제로 손대지 않은 자재가 장부에서 움직일 수 있다.
  - 직계 BOM 비교가 화면 이해와 입출고 내역 추적에 가장 명확하다.
- 계산 규칙:
  - source PA 직계 BOM과 target PA 직계 BOM을 item_id 기준으로 비교한다.
  - 차이 수량에 실행 수량 Q를 곱해 재고 변동을 만든다.
  - 동일 item_id, 동일 수량은 유지 항목으로 보고 재고 변동 없음.
