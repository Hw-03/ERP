### A. 바로 재작업 품목 선택 화면 레이아웃 재정리

- 화면: 불량 > 바로 처리 > 재작업 > 품목 선택
- 상태: 구현/검증 완료 (2026-06-30). 사용자 예시처럼 오른쪽 제목 줄과 왼쪽 툴바가 같은 작업줄로 읽히게 정리했다.
- 문제:
  - `순서 편집`과 `재작업 품목`은 같은 높이로 맞아야 하지만, 오른쪽 제목이 카드/표 구조와 따로 노는 방식이면 어색하다.
  - 사용자가 의도한 정렬은 `순서 편집` 버튼 오른쪽, 오른쪽 패널 시작부 위에 `재작업 품목`이 자연스럽게 놓이는 느낌이다.
  - 단순히 좌표상 centerY를 맞추는 방식은 충분하지 않다. 표 헤더, 오른쪽 카드 경계, 버튼 위치가 함께 읽혀야 한다.
- 해결 방향 메모:
  - 예시 이미지처럼 왼쪽 필터/검색/순서편집 라인의 오른쪽 끝과 오른쪽 패널 제목 라인이 하나의 상단 작업줄처럼 보이게 정리한다.
  - `재작업 품목`은 오른쪽 패널 위의 제목으로 두되, 카드 안에 갇힌 헤더도 아니고 페이지 밖에 떠 있는 라벨도 아닌 중간 톤을 찾는다.
  - 왼쪽 표와 오른쪽 카드의 top만 맞추는 것이 아니라, 제목-컨트롤-표/카드의 세로 리듬을 같이 맞춘다.
  - 구현 전 브라우저 스크린샷으로 1차 확인하고, 수치 계측은 보조 기준으로만 사용한다.

### D. 품목 리스트 박스 내부 스크롤 복구

- [x] 완료: Step 2 grid stretch와 `h-full min-h-0` 구조를 복구해 페이지 전체 스크롤 없이 품목 리스트 내부에서만 스크롤되게 했다.
- 화면: 불량 > 바로 처리 > 재작업 > 품목 선택
- 문제:
  - 원래 품목 리스트는 표 박스 안에서만 스크롤되어야 하는데, 현재는 화면 전체가 스크롤된다.
  - 작업 하단 버튼과 오른쪽 패널이 화면 기준으로 안정적으로 남아 있어야 하는데, 리스트 내용이 페이지 전체 높이를 밀어내고 있다.
- 원인 메모:
  - 직전 UX 보정에서 Step 2 grid에 `items-start`를 넣으면서 grid 자식이 남은 높이까지 stretch 되지 않게 됐다.
  - 그 결과 `DefectItemPicker`의 `h-full / min-h-0 / overflow-y-auto`가 고정 부모 높이를 잃고, 표 내용 전체가 페이지 높이를 늘리는 구조가 됐다.
- 해결 방향 메모:
  - Step 2 grid의 `items-start`를 제거하거나, 왼쪽/오른쪽 grid item에 `h-full min-h-0`를 명시해서 내부 스크롤 소유권을 복구한다.
  - 품목 표 컨테이너가 `overflow-y-auto`를 유지하고, 페이지 전체 스크롤이 생기지 않는지 브라우저에서 직접 확인한다.
  - 다음 UX 정리 때 `순서 편집`/`재작업 품목` 정렬과 함께 처리하되, 스크롤 구조를 먼저 깨지 않도록 고정 높이 체계를 우선 확인한다.

### E. 재작업 품목 패널 높이를 품목 리스트 박스와 맞추기

- [x] 완료: 오른쪽 `재작업 품목` 패널을 왼쪽 품목 리스트 박스와 같은 top/height로 맞추고 내부 스크롤 구조를 유지했다.
- 화면: 불량 > 바로 처리 > 재작업 > 품목 선택
- 문제:
  - 오른쪽 `재작업 품목` 영역이 담긴 품목 카드 내용 높이만큼만 작게 보이면, 왼쪽 품목 리스트 박스와 한 쌍의 패널처럼 읽히지 않는다.
  - 사유 카테고리 드롭다운을 열었을 때도 오른쪽 영역이 중간에서 끊긴 느낌이 강하고, 화면 오른쪽 하단 공간이 비어 보여 균형이 나쁘다.
- 요구:
  - 오른쪽 `재작업 품목` 박스 높이를 왼쪽 품목 리스트 박스 높이에 맞춘다.
  - 내부에 선택 품목이 1개만 있어도 패널 자체는 왼쪽 리스트와 같은 높이로 유지한다.
  - 패널 내부 콘텐츠가 많아지면 오른쪽 패널 내부에서 스크롤되게 하고, 화면 전체 스크롤로 번지지 않게 한다.
- 해결 방향 메모:
  - Step 2 좌우 grid item을 같은 높이로 stretch시키고, 왼쪽 `defect-picker-table`과 오른쪽 `defect-cart-panel`이 같은 세로 크기를 갖도록 구조를 잡는다.
  - `재작업 품목` 제목 줄은 사용자가 예시로 준 위치 감각을 유지하되, 패널 높이 계산에서 제외할지 포함할지 브라우저 캡처로 판단한다.
  - AppSelect 드롭다운이 열려도 패널 경계가 어색하게 끊기지 않는지 확인한다.

  - `dept_adjustment.py` 공통 재작업 엔진 추가.
  - child item `process_type_code` 기준으로 부서를 계산.
  - 정상/격리/폐기 각각 로그와 재고 처리를 분리.
- 검증:
  - `python -m pytest backend/tests/services/test_sr_execution.py backend/tests/test_defect_flow.py -q`

### 8. 브라우저 뒤로가기/앞으로가기

- 화면: 불량 > 바로 처리 > 출처 선택/품목 선택
- 증상:

- 해결:

  - popstate에서 Step 1/Step 2를 복원한다.
- 검증:

### 9. 바로 재작업 BOM 확인 단계/수량 입력 UX

- 화면: 불량 > 바로 처리 > 재작업
- 요구:
  - 재작업 BOM 트리를 오른쪽 장바구니 안에서 작게 보지 않고, 기존 불량 처리처럼 큰 화면 단계에서 확인한다.
  - 기본값은 전량 정상이며, 사용자가 격리/폐기 수량을 입력하면 정상 수량이 자동으로 줄어든다.
  - 정상+격리+폐기 합계는 항상 처리 수량과 같아야 하며, 적거나 많아지지 않아야 한다.
- 해결:
  - 데스크톱/모바일 바로 재작업을 `작업 선택 → 출처·부서 선택 → 품목 선택 → BOM 확인` 단계로 분리.
  - `DisassembleTree`에서 격리/폐기 입력 시 정상 수량을 `총수량 - 격리 - 폐기`로 자동 보정.
  - 수량 입력/분해 방식 버튼의 터치 영역을 키워 작업 현장에서 누르기 쉽게 보정.
- 검증:
  - `DefectCartFlow.test.tsx`에서 BOM 확인 단계 진입, 격리 입력 후 정상 수량 자동 감소, `rework_normal` payload를 검증.
