- 브라우저 확인 시나리오: 생산 완료 행과 분해 행을 각각 확인해 최상위 수량 배지가 `생산`·`분해`로 구분되는지, 같은 행의 부품 수량 및 일반 재고 이동·품목 전환 수량 배지가 바뀌지 않는지 확인한다.

### 8. 다음 페이지를 대표 행 100개 단위로 불러오기
- 상태: 구현 완료 (2026-07-15)
- 화면/영역: PC 입출고 내역 > 목록 하단 > `다음 100건 불러오기`
- 증상: 현재 페이지 크기는 원본 거래 로그 100건이다. 화면은 원본 로그를 단건·작업 묶음·요청 묶음·불량 lifecycle 묶음의 대표 행으로 다시 구성하므로, 한 대표 행에 속한 하위 로그가 많으면 버튼을 눌러도 대표 행은 100개보다 적게 추가된다.
- 사용자 불편: 버튼의 `100건`을 클릭했을 때 펼쳐지는 하위 행을 포함한 100줄이 아니라, 목록에서 바로 보이는 부모(대표) 행 100개가 추가될 것으로 기대한다. 현재는 묶음 크기에 따라 추가되는 대표 행 수가 달라져 여러 번 눌러야 한다.
- 확정 정책: 이력 페이지의 `100건`은 화면의 최상위 대표 행 100개를 의미한다. 단건, `operation_batch_id` 묶음, 요청번호 묶음, 불량 lifecycle 묶음은 각각 한 개의 대표 행으로 센다. 대표 행을 펼쳐 보여 주는 하위 로그는 페이지 건수에 포함하지 않는다.
- 구현 방향: `useHistoryData`의 원본 로그 페이지네이션과 `buildGroups` 결과를 분리해, 최초 조회와 `다음 100건` 모두 대표 행 기준으로 100개를 채울 때까지 필요한 원본 로그 페이지를 내부적으로 추가 조회·버퍼링한다. 화면에는 대표 행 경계에서만 다음 페이지를 공개하고, 한 묶음이 원본 페이지 경계를 넘더라도 중복·누락·부분 묶음 없이 하나의 대표 행으로 유지한다. 버튼 문구는 대표 행 기준을 오해하지 않도록 필요 시 `다음 대표 100건 불러오기`로 함께 조정한다.
- 수용 기준: 첫 진입과 버튼 클릭 뒤 각각 최대 100개의 새 대표 행이 보인다(마지막 페이지는 남은 수만 표시). 대표 행을 모두 펼쳐도 하위 로그 수는 100개 산정에 포함되지 않으며, 원본 로그가 많은 묶음·요청·불량 lifecycle도 중복 또는 분리되지 않는다. 기존 필터, 검색, 날짜 조건, 추가 조회 실패 재시도와 정렬 순서는 유지된다.
- 브라우저 확인 시나리오: 하위 로그가 많은 생산/분해 묶음과 단건이 섞인 조건에서 초기 목록 및 `다음 100건` 클릭 후의 최상위 대표 행 수를 각각 센다. 대표 행을 여러 개 펼쳐도 100개 산정이 바뀌지 않는지, 원본 페이지 경계의 묶음이 한 대표 행으로 유지되는지, 마지막 페이지·필터 변경·추가 조회 실패 후 재시도를 확인한다.
- 우선순위: 중간

### 13. BOM `제외` 배지를 대표 수량 요약 옆에 한 번만 표시
- 상태: 구현 완료 (2026-07-15)
- 화면/영역: PC 입출고 내역 > 확장된 작업 묶음 > 대표 행·BOM 요약행의 `제외` 상태
- 증상: 제외된 BOM 구성품이 있는 묶음은 `제외` 배지가 대표 행의 `상태 · 처리` 열과 BOM 요약행에 중복 표시된다. 넓은 상태 열 안에 배지만 단독으로 남고, 확장된 하위 행에는 빈 상태 영역이 반복되어 목록의 정보 간격이 과도하게 느껴진다.
- 사용자 불편: `제외`가 이미 해당 묶음의 수량 요약과 함께 이해할 수 있는 보조 정보인데 분리된 상태 열에서 중복되어, 대상·수량을 읽는 흐름이 끊기고 표가 불필요하게 넓어 보인다.
- 확정 정책: 제외된 구성품 수는 대표 행의 수량 요약(예: `이동 4종목 · 75 EA`) 바로 뒤에 `제외 N`으로 한 번만 표시한다. BOM 요약행과 각 하위 행에는 같은 제외 배지를 다시 표시하지 않는다. 이로써 줄어드는 `상태 · 처리` 열의 가로 폭은 같은 만큼 `작업` 열에 재배분해 작업 배지와 명칭의 좌우 여백을 넓힌다. 부족·취소·메모·승인자 등 다른 상태 정보의 기존 표현은 유지한다.
- 구현 방향: `historyPresentation.ts`의 대표 행 상태 chip과 `BomBatchDetail.tsx`의 BOM 요약 상태 렌더링을 조정해 제외 수를 수량 요약의 보조 정보로 전달한다. 이어 `HistoryTable.tsx`의 `COLUMNS`에서 `상태 · 처리` 열 폭을 제외 배지에 필요했던 여백만큼 줄이고, 정확히 같은 폭을 `작업` 열에 더한다. 상태 열은 사람·메모·다른 상태 정보를 계속 읽을 수 있는 최소 폭을 보장하며, 열 구조를 제거하지 않는다.
- 수용 기준: 제외 구성품이 있는 묶음에서 대표 행 수량 요약 바로 뒤에 `제외 N`이 한 번만 보이고, BOM 요약행·하위 구성품 행에는 중복 배지가 없다. 줄어든 상태 열 폭과 같은 폭만큼 작업 열이 넓어져 작업 배지·명칭의 여백이 자연스럽게 늘어난다. 제외가 없는 묶음, 부족이 있는 묶음, 취소·메모가 있는 거래의 기존 상태 표시는 바뀌지 않는다.
- 브라우저 확인 시나리오: 제외 구성품만 있는 BOM 묶음, 제외와 부족이 함께 있는 묶음, 제외 없는 일반 작업 묶음, 즉시 재작업 묶음, 단건 거래를 각각 펼쳐 비교한다. 대표 행의 수량 요약 인접 표시와 상태 열 중복 제거를 확인하고, 이전 상태 열의 빈 폭만큼 작업 열이 넓어졌는지, 즉시 재작업 하위 행의 빈 상태 영역도 같은 열 폭 재배분으로 줄어드는지, 대상명·품목코드·수량 열 간격과 우측 상세의 제외 집계가 모두 유지되는지 확인한다.
- 우선순위: 중간

### 14. 직접 불량 폐기 거래를 `불량 폐기`로 표시
- 상태: 구현 완료 (2026-07-15)
- 화면/영역: PC 입출고 내역 > `DEFECT_SCRAP` 단건 행 > 작업 배지·수량 배지·우측 상세 작업명
- 증상: 불량 재고를 별도 후속 처리 없이 바로 폐기해 생성된 `DEFECT_SCRAP` 행이 작업 배지에서 `불량 처리`, 수량 배지에서 `불량 처리 1 EA`로 표시된다. 화면의 실제 동작은 폐기인데 포괄적인 `처리`라는 표현이라 결과를 즉시 알기 어렵다.
- 사용자 불편: 목록을 훑을 때 불량 격리·해제·반품·분해 같은 다른 불량 흐름과 직접 폐기 거래를 작업명만으로 구분하기 어렵다.
- 확정 정책: `DEFECT_SCRAP` 거래는 목록 작업 배지, 수량 배지, 우측 상세의 작업명 등 공통 거래 라벨을 모두 `불량 폐기`로 표시한다. 불량 처리라는 상위 업무 범주·필터명과 `MARK_DEFECTIVE`(새 불량), `UNMARK_DEFECTIVE`(불량 해제), 반품·분해 등 다른 거래 유형의 라벨은 바꾸지 않는다.
- 구현 방향: `frontend/lib/io/glossary.ts`의 `DEFECT_SCRAP` 및 이에 대응하는 직접 폐기 sub-type 표시 라벨을 `불량 폐기`로 맞추고, 입출고 내역의 공통 작업명·수량 요약 경로가 그 라벨을 사용하도록 확인한다. 이 정본 라벨을 기대하는 glossary·이력 golden 테스트를 함께 갱신한다.
- 수용 기준: 직접 불량 폐기 행은 작업 열에 `불량 폐기`, 수량 열에 `불량 폐기 N EA`, 우측 상세 작업명에 `불량 폐기`를 일관되게 표시한다. 불량 필터의 범주명과 새 불량·불량 해제·원자재 반품·불량 분해의 기존 명칭 및 색상은 변하지 않는다.
- 브라우저 확인 시나리오: 바로 폐기한 단건 `DEFECT_SCRAP` 거래를 목록에서 선택해 작업 배지·수량 배지·우측 상세를 확인한다. 새 불량, 불량 해제, 원자재 반품, 불량 분해 거래도 나란히 확인해 직접 폐기 거래만 새 명칭으로 바뀌고 불량 필터 결과와 색상·아이콘이 유지되는지 확인한다.
- 우선순위: 중간

### 17. 재작업 결과의 폐기·회수 수량 색상을 의미별로 분리
- 상태: 구현 완료 (2026-07-15)
- 화면/영역: PC 입출고 내역 > 즉시 재작업 묶음 > 하위 `처리결과` 행의 수량 열
- 증상: 한 품목에서 폐기와 회수가 함께 일어난 결과가 `폐기 1 EA · 회수 1 EA`처럼 한 줄로 표시될 때, 행 전체에 단일 위험 색상이 적용되어 회수 수량도 빨간색으로 보인다.
- 사용자 불편: 재고를 줄이는 폐기와 재고를 되돌리는 회수는 의미가 반대인데 같은 색으로 보이므로, 수량 결과를 빠르게 훑을 때 회수라는 긍정 결과를 구분하기 어렵다.
- 확정 정책: 재작업 결과 수량에서 `폐기 N EA`는 빨간색, `회수 N EA`는 초록색으로 각각 표시한다. 폐기만·회수만 있는 경우에도 같은 의미 색상을 사용하며, 제외된 결과 행과 취소선의 기존 표현은 유지한다.
- 구현 방향: `reworkSummary.ts`의 결과 요약은 의미 단위 데이터를 유지하고, `ReworkBatchDetail.tsx`가 폐기·회수 텍스트를 각각 독립 span으로 렌더링해 색상을 적용한다. 단일 `resultTone`으로 수량 셀 전체를 칠하는 방식은 혼합 결과에서 사용하지 않으며, 수량 형식·단위·구분점은 유지한다.
- 수용 기준: `폐기 1 EA · 회수 1 EA`에서는 폐기만 빨간색, 회수만 초록색으로 보인다. 폐기만 있는 행은 빨간색, 회수만 있는 행은 초록색이며, 제외·취소 행의 흐림과 취소선은 기존대로 동작한다.
- 브라우저 확인 시나리오: 폐기만, 회수만, 폐기와 회수가 함께 있는 즉시 재작업 하위 행을 각각 확인한다. 취소된 재작업 묶음과 제외된 결과 행도 펼쳐 색상 의미·텍스트 형식·취소선이 서로 충돌하지 않는지 확인한다.

### 19. 작업 열 알약을 최장 작업 라벨 기준의 동일 폭으로 축소

- 수용 기준: `수량보정 입고`와 `추가 구성품 차감`이 아이콘·텍스트·좌우 여백을 유지한 채 한 줄로 표시되고, 부모 행의 작업 알약과 펼친 `BOM`·`단품`·`자동차감`·`수동`·`처리결과` 알약이 모두 같은 폭으로 중앙 정렬된다. 작업 열에 남는 여백은 좌우 균형을 이루며, 긴 하위 라벨은 기존 truncation·접근성 라벨 규칙을 유지한다.

### 21. 다품목 작업 펼침을 BOM 계층형 접기·펼치기 문법으로 통일

- 증상: 현재 `op_batch`만 `BomBatchDetail`의 부모 → 구성 묶음 → 하위 품목 계층을 사용하지만, 참조 묶음·불량 lifecycle·재작업은 각각 `ReferenceBatchDetail` 또는 `ReworkBatchDetail`의 평면 행 구조로 전개된다. 같은 다품목 작업인데도 어떤 것은 BOM처럼 다시 접고 펼 수 있고, 어떤 것은 모든 하위 행이 한 번에 길게 노출된다.

### 27. 여러 재고 위치의 실제 영향을 위치별 아코디언으로 분리
- 상태: 구현 완료 (2026-07-16)
- 화면/영역: PC 입출고 내역 > 우측 상세 카드 > `실제 영향`
- 증상: 창고와 출하 부서처럼 실제 영향이 둘 이상의 재고 위치에 생긴 거래에서 모든 품목 변화가 하나의 긴 목록으로 이어진다. 각 행의 보조 문구를 읽기 전에는 어느 장소의 재고 변화인지 구분하기 어렵다.
- 사용자 불편: 여러 위치와 품목이 섞인 긴 영향 목록에서 필요한 장소의 변화만 확인하려 해도 관련 없는 행을 함께 훑어야 하므로 우측 상세가 난잡하게 보인다.
- 확정 정책: 재고 위치가 2곳 이상이면 모든 위치 아코디언을 접힌 상태로 시작한다. 사용자는 필요한 위치만 열어 보며, 한 위치를 열거나 닫아도 다른 위치의 상태는 바뀌지 않는다. 위치가 한 곳뿐인 거래는 기존처럼 목록을 바로 보이게 한다. 우측 상세는 정보량보다 깔끔함을 우선한다.
- 구현 방향: `historyDetailSummary.ts`의 단일 `actual` 영향 묶음을 실제 재고 위치(창고, 부서 재고, 불량 재고, 박스 재고)별로 분류한다. `HistoryKeyPointSummary.tsx`에서 위치별 헤더에 위치명·영향 품목 수·순증감 요약을 표시하고, 헤더 클릭 및 Enter/Space로 각 목록을 독립적으로 접고 펼치게 한다. 행 클릭으로 우측 상세를 이동하는 기존 동작, 취소/불량/BOM 효과 집계, 영향 데이터 자체는 변경하지 않는다.
- 수용 기준: 창고와 출하처럼 두 곳 이상에 영향이 있으면 위치별로 분리된 접이식 섹션으로 보이고, 한 위치를 열고 닫아도 다른 위치의 상태는 유지된다. 각 섹션 안에는 해당 위치의 품목만 표시되며, 한 곳만 영향받은 거래에는 불필요한 아코디언 계층이 추가되지 않는다.
- 브라우저 확인 시나리오: 2026-07-09 16:05 `창고 → 부서` 다품목 거래를 열어 창고·출하 섹션을 각각 열고 닫는다. 생산, 창고↔부서 이동, 불량 재고, 박스 재고가 섞인 거래도 확인해 위치별 품목·증감 수량·요약 건수, 키보드 Enter/Space, 행 클릭, 긴 목록 스크롤이 정확한지 검증한다.
- 우선순위: 중간

### 28. 목록 작업 열을 업무 메뉴 분류로 간소화하고 수량 열과 역할을 분리
- 상태: 구현 완료 (2026-07-16)
- 화면/영역: PC 입출고 내역 > 목록 > `작업`·`수량` 열
- 증상: 현재 작업 열은 `재작업`처럼 세부 행위를 표시하고, 수량 열도 `재작업 1 EA`처럼 같은 행위를 다시 표시한다. 대표 행 라벨은 `getHistoryDisplayLabel`의 세부 sub_type/거래 유형을 사용하고, 수량 열은 별도의 변동 요약을 사용해 중복이 생긴다.
- 사용자 불편: 목록을 훑을 때 열마다 같은 단어가 반복되고, 거래의 큰 업무 맥락보다 세부 용어가 먼저 보여 정보 밀도에 비해 화면이 복잡해 보인다.
- 확정 정책: 작업 열은 업무 메뉴 분류만 표시한다: `원자재 입고`, `창고 입출고`, `부서 입출고`, `품목 전환`, `출하`, `불량`, `AS·연구 사용출고`. 세부 행위와 수량은 수량 열에만 표시한다(예: 작업 `불량`, 수량 `재작업 1 EA`). 재작업은 불량 탭에서 수행하는 업무이므로 항상 `불량`으로 분류한다.
- 정책 결정: 수량 조정은 `부서 입출고`, 레거시 단건 거래는 거래 유형에 따른 가장 가까운 업무 메뉴 분류를 사용한다.
- 구현 방향: 배치가 있는 행은 `IoBatch.work_type`과 출하 phase를 기준으로 작업 열 카테고리를 반환하는 단일 presentation 함수를 둔다. 단건·레거시 로그는 거래 유형으로 가장 가까운 카테고리로 안전하게 fallback한다. `HistoryLogRow.tsx`, 대표 묶음 헤더, 하위 행에서 열의 역할을 분리하되, 우측 상세의 실제 작업명·이동 경로·메모·수량 및 목록 수량 열의 세부 행위는 유지한다.
- 수용 기준: `재작업` 대표 행의 작업 열과 수량 열에 같은 단어가 반복되지 않는다. 목록 작업 열은 정해진 업무 메뉴 분류만 사용하고, 수량 열은 해당 거래의 실제 행위·수량을 계속 구분해 보인다. 품목 전환·출하·불량·부서 입출고·창고 입출고의 색상·아이콘·접이식 묶음·필터 결과·우측 상세는 정확히 유지된다.
- 브라우저 확인 시나리오: 재작업, 불량 폐기, 창고↔부서 이동, 부서 생산/분해, 수량보정, 품목 전환, 출하, AS·연구 반출, 원자재 입고를 각각 확인한다. 작업 열의 짧은 분류와 수량 열의 세부 행위가 중복되지 않는지, 단건·묶음·하위 행·취소 이력·좁은 PC 폭에서도 빠르게 구분되는지 검증한다.
- 우선순위: 높음
