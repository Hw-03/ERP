## 공통 레이아웃

### 1. 모든 관리자 탭에서 안내 문구를 제거하고 KPI를 헤더 행으로 이동

- 화면/영역: 관리자 > 모델 관리, 품목 관리, 직원 관리, 부서 관리, BOM 관리, 내보내기, 외부 제출용 로그, 보안 등 각 탭의 페이지 헤더와 KPI 영역.
- 증상: 제목 아래의 탭별 설명 문구가 반복되고, KPI가 별도 줄을 차지해 실제 목록·상세·작업 영역이 아래로 밀린다.
- 사용자 불편: 화면 성격은 제목과 탭만으로 충분히 알 수 있는데 보조 문구와 큰 KPI가 상단 높이를 과도하게 사용한다.
- 확정 정책: 모든 관리자 탭에서 페이지 설명 문구를 표시하지 않는다. KPI는 카드 크기를 줄여 제목 우측의 헤더 행에 배치하고, 별도 KPI 행은 없앤다. KPI가 없는 탭은 제목·행동 버튼만 표시한다.
- 구현 방향: `AdminPageHeader`와 `AdminKpiBar`의 조합을 공통화해 탭별 반복 여백을 없애고, 각 섹션이 전달하던 `description`과 별도 KPI 렌더링을 새 공통 배치로 전환한다. 수치·상태 색상·탭별 행동 버튼의 의미와 동작은 바꾸지 않는다.
- 허용 기준:
  - 어느 관리자 탭에서도 기존 설명 문구가 렌더링되지 않는다.
  - KPI가 있는 탭에서는 제목 우측 헤더 영역 안에만 축소된 KPI가 보인다.
  - KPI가 없는 내보내기·외부 제출용 로그·보안 탭의 제목/행동 버튼 정렬이 깨지지 않는다.
  - 목록·상세·BOM 작업 영역의 시작 위치가 기존보다 위로 올라가며 가로 스크롤이 생기지 않는다.
- 브라우저 확인 시나리오: 1919×908에서 모든 관리자 탭을 순서대로 열어 헤더 높이, KPI 정렬, 제목 말줄임, 우측 행동 버튼, 목록/상세 영역 시작 위치를 확인한다. 좁은 데스크톱 폭에서도 KPI가 제목 또는 버튼과 겹치지 않는지 확인한다.
- 우선순위: 높음

## 관리자 탭 바

### 4. 탭 그룹의 경계를 분명하게 표시

- 화면/영역: 관리자 탭 바의 `기준 정보`, `구성 관리`, `시스템` 그룹과 각 그룹의 탭 버튼.
- 증상: 그룹 제목이 작은 보조 텍스트로만 보이고 탭들과 같은 한 줄에 밀집해 있어, 세 그룹의 시작과 끝이 즉시 구분되지 않는다.
- 사용자 불편: 모델/품목/직원처럼 자주 쓰는 기준 정보와 BOM, 내보내기·보안을 빠르게 구획해 찾기 어렵다.
- 확정 정책: `기준 정보`, `구성 관리`, `시스템`의 시각적 그룹 경계를 강화한다. 그룹명은 탭보다 한 단계 낮은 정보로 유지하되, 구분선·여백·그룹 컨테이너를 조합해 각 그룹이 분리되어 보이게 한다.
- 구현 방향: `AdminSectionTabs`의 `SECTION_GROUPS` 렌더링에서 그룹 제목의 대비와 그룹 간 경계를 조정한다. 탭의 순서, 선택 상태 색상, 탭별 이동 동작은 유지하며 탭 바의 세로 높이를 불필요하게 늘리지 않는다.
- 허용 기준:
  - 1919×908에서 세 그룹의 시작/끝이 첫눈에 구분된다.
  - 활성 탭은 계속 가장 강하게 보이며, 그룹명과 구분 장식이 탭 선택 상태와 경쟁하지 않는다.
  - 가로폭이 줄어도 기존 가로 스크롤/탭 접근 방식이 유지된다.
- 브라우저 확인 시나리오: 모델 관리, BOM 관리, 보안 탭을 차례로 선택해 각 그룹의 활성·비활성 상태와 그룹 경계를 확인한다. 가로폭을 줄여 마지막 시스템 탭까지 도달 가능한지 확인한다.

## 부서 관리

### 10. 품목 코드의 공정 부서 기준으로 관련 품목 수 집계

- 화면/영역: 관리자 > 부서 관리의 선택 부서 상세 메타 정보 `관련 품목 N개`.
- 증상: 현재 `관련 품목`은 품목 마스터의 별도 `department` 값만 세므로, MES 품목 코드가 나타내는 공정 부서와 집계 기준이 어긋날 수 있다.
- 사용자 불편: 부서 상세에서 품목 코드로 알 수 있는 공정 소속이 아닌 별도 값으로 계산되어, 관련 품목 수가 사용자의 업무 판단 기준과 일치하지 않는다.
- 확인한 현재 동작: 화면은 `items`의 `item.department`가 선택 부서명과 일치하는 품목만 계산한다. 반면 품목 코드는 두 번째 구간의 `process_type_code`로 부서를 정하며, 예를 들어 `46-AA-0080`의 `AA`는 조립이다.
- 확정 정책: `관련 품목`은 MES 품목 코드의 공정 구간(`process_type_code`)이 가리키는 부서 기준으로 집계한다. 튜브·고압·진공·튜닝·조립·출하의 6개 공정 부서만 코드에서 매핑하며, `기타` 등 매핑되지 않는 조직 부서는 0개가 정상임을 화면에서 알 수 있게 한다.
- 구현 방향: 표시용 `mes_code` 문자열을 직접 분해하지 않고, 그 진실 소스인 `item.process_type_code`를 공통 `PROCESS_TO_DEPT` 매핑으로 변환해 집계한다. 관련 품목 메타 정보에 `품목 코드 공정 기준`이라는 보조 설명을 추가해 직원 수·재고 위치·BOM 사용처와 다른 수치임을 분명히 한다.
- 허용 기준:
  - 선택한 6개 공정 부서의 수가 해당 `process_type_code` 품목 수와 일치한다.
  - `기타`처럼 코드 공정에 매핑되지 않는 부서는 0개와 그 이유가 함께 표시된다.
  - 직원 수·재고 위치·BOM 사용처 수가 관련 품목 수에 섞이지 않는다.
- 브라우저 확인 시나리오: 조립·고압 등 공정 부서와 `기타`를 각각 선택해, 품목 코드의 두 번째 구간 기준 집계와 보조 설명을 확인한다. `AA`·`AR`·`AF` 품목이 조립에 함께 포함되는지 확인한다.
- 우선순위: 중간

## 직원 관리

### 8. 부서 필터를 목록 헤더 우측으로 이동하고 드롭다운 폭 확보

- 화면/영역: 관리자 > 직원 관리의 왼쪽 직원 목록 헤더, 검색창 아래의 `전체 부서` 필터, 펼친 부서 목록.
- 증상: 부서 필터가 검색창 아래에 별도 줄로 배치되어 목록 시작 위치를 밀어낸다. 좁은 trigger 폭을 그대로 따라 펼친 목록도 매우 좁아져 `경영지원`, `대표이사` 같은 부서명이 줄바꿈된다.
- 사용자 불편: 직원 수가 많은 목록에서 한 줄의 공간을 낭비하고, 필터를 펼치면 부서명을 읽기 어렵다.
- 확정 정책: 부서 필터는 `직원 목록·N명` 헤더의 우측 빈 공간으로 이동한다. trigger와 목록은 모든 부서명이 한 줄로 보이는 폭을 확보하며, 옵션 줄바꿈을 허용하지 않는다.
- 구현 방향: 직원 관리에서 `AdminListPanel`의 헤더 우측 슬롯을 사용해 `AppSelect`를 배치하고, 현재 필터 줄은 제거한다. 직원 목록 전용 trigger/목록의 최소 폭과 no-wrap 처리를 지정하되 공통 드롭다운의 다른 사용처 레이아웃은 바꾸지 않는다. 부서 필터링·검색·키보드 조작은 유지한다.
- 허용 기준:
  - `전체 부서` 필터가 목록 헤더 우측에 표시되고 검색창 아래에 빈 필터 줄이 남지 않는다.
  - 펼친 목록의 모든 부서명이 한 줄로 표시되며 긴 이름도 의도한 폭 안에서 읽힌다.
  - 전체 부서와 각 부서 선택 시 직원 목록 수·행이 즉시 기존 조건대로 갱신된다.
- 브라우저 확인 시나리오: 전체 부서와 가장 긴 부서명을 선택하고, 드롭다운 열린 상태에서 줄바꿈·잘림·목록 패널 경계 넘침을 확인한다. 키보드 방향키/Enter/Escape로 필터 선택과 포커스 복귀를 확인한다.
- 우선순위: 중간

### 9. 직원 직급을 지정 목록 드롭다운으로 선택

- 화면/영역: 관리자 > 직원 관리의 직원 추가 폼과 선택 직원 상세의 `직급` 입력.
- 증상: 직급이 자유 입력 텍스트 필드라 표기 방식이 제각각 저장될 수 있고, 직원 목록·검색에서 직급 분류의 일관성이 떨어진다.
- 사용자 불편: 같은 직급을 다른 문자열로 입력할 수 있어 검색과 목록 비교가 어려워진다.
- 확정 정책: 직원 추가와 수정 모두 직급을 다음 지정 목록에서 선택한다: `사원`, `주임`, `대리`, `과장`, `부장`, `이사`, `대표`, `선임연구원`, `책임연구원`. 이번 범위에서는 이 목록만 제공한다.
- 정책 확인 필요: 현재 저장된 직급 중 지정 목록에 없는 값(예: 복합 표기나 기존 직책)을 어떻게 표시·정리할지 결정해야 한다. 자동 일괄 변경은 하지 않는다.
- 구현 방향: 직원 추가/상세의 `TextInput` 직급 필드를 기존 `AppSelect` 기반 선택 컴포넌트로 교체하고, 목록·상세 헤더·검색이 저장된 직급 문자열을 계속 사용하게 한다. 기존 API와 DB의 직급 문자열 저장 구조는 변경하지 않는다.
- 허용 기준:
  - 신규 직원과 기존 직원의 직급 입력에서 지정한 9개 옵션만 선택할 수 있다.
  - 선택·저장 후 직원 목록 보조 정보와 상세 헤더에 동일한 직급이 표시되고 검색된다.
  - 부서, 연락처, 결재 역할, PIN 등 다른 직원 정보의 입력·저장 동작은 유지된다.
- 브라우저 확인 시나리오: 직원 추가와 기존 직원 편집에서 9개 옵션을 열어 선택·저장하고, 선택한 직급으로 검색해 목록에 표시되는지 확인한다. 지정 목록 밖의 기존 직급 데이터가 있는 경우에는 결정된 표시/정리 정책에 따라 확인한다.
- 우선순위: 중간

### 6. 품목 목록의 MES 코드를 보조 식별자 문법으로 통일

- 화면/영역: 관리자 > 품목 관리의 왼쪽 품목 목록에서 품목명 우측에 표시하는 MES 코드.
- 증상: 현재 품목 코드는 선택된 행에서 강한 파란색 사각 배지로 바뀌고, 미선택 행에서도 별도 배지로 표시된다. 재고·BOM 등 인접 화면은 코드를 일반 보조 텍스트 또는 절제된 모노스페이스 텍스트로 표시한다.
- 사용자 불편: 품목 코드가 상태나 주요 행동처럼 강조되어, 품목명·재고 상태보다 먼저 시선을 끈다. 화면마다 동일한 MES 코드의 표시 문법도 일관되지 않다.
- 확정 정책: MES 코드는 품목명의 보조 식별자로 표시한다. 선택 여부는 행 배경·테두리만으로 표현하며, 코드 자체를 선택 색상의 강한 배지로 바꾸지 않는다.
- 구현 방향: 품목 목록 코드의 별도 강조 배지를 제거하고, 인접 재고/BOM 화면의 보조 코드 규칙처럼 muted 계열의 작은 텍스트(필요 시 모노스페이스)로 표시한다. 품목명, 재고 상태 pill, 선택/드래그/검색 동작은 유지한다.
- 허용 기준:
  - 선택·미선택 행 모두에서 MES 코드가 보조 정보로 읽히며 상태 pill과 경쟁하지 않는다.
  - 선택된 행은 코드 색상이 아니라 행의 배경·테두리로 구분된다.
  - 긴 코드에서도 품목명, 코드, 상태 pill이 겹치거나 가로 스크롤을 만들지 않는다.
- 브라우저 확인 시나리오: 정상·부족·품절 품목과 긴 품목명/코드가 있는 행을 각각 선택해 코드의 강조 수준, 말줄임, 상태 pill, 선택 표시를 확인한다. 재고/BOM 화면의 코드 표현과 나란히 비교한다.
- 우선순위: 중간

### 7. 품목 상세 탭을 공통 탭 문법에 맞게 정리

- 화면/영역: 관리자 > 품목 관리 상세 카드의 `기본 정보`, `재고 정보`, `BOM / 사용처`, `변경 이력 (준비 중)` 전환 영역.
- 증상: 탭들이 작은 사각 버튼처럼 연속되어 공통 화면의 탭 선택 문법과 다른 인상을 주고, 활성·비활성·준비 중 상태의 위계가 약하다.
- 사용자 불편: 상세 정보의 범주를 전환하는 탭보다 독립적인 보조 버튼 집합처럼 보여 현재 위치와 전환 가능 상태를 빠르게 파악하기 어렵다.
- 확정 정책: 품목 상세 탭은 공통 상세 카드의 단일 탭 패턴으로 보이게 한다. 활성 탭만 색상과 하단 표시로 강조하고, 비활성 탭은 보조 텍스트로 유지한다. `변경 이력 (준비 중)`은 현재 기능 상태를 분명하게 구분한다.
- 구현 방향: `AdminDetailCard`의 탭 primitive와 품목 탭 데이터의 레이블/상태 표현을 디자인 언어에 맞춰 조정한다. 탭 클릭 시의 기존 콘텐츠 전환, 변경 이력 안내, 삭제·저장 버튼과의 관계는 바꾸지 않는다.
- 허용 기준:
  - 탭 전환 영역이 버튼 묶음이 아니라 하나의 탐색 컨트롤로 읽힌다.
  - 활성 탭은 파란 강조와 하단 indicator로 명확하고, 비활성 탭은 과도한 테두리·배경 없이 보조적으로 표시된다.
  - 준비 중 탭의 제한 상태가 인지되며, 탭 전환과 하단 저장/삭제 동작이 유지된다.
- 브라우저 확인 시나리오: 네 탭을 차례로 전환해 활성 표시와 콘텐츠 전환을 확인하고, `변경 이력 (준비 중)`의 안내가 현재 기능 범위를 오해하게 하지 않는지 확인한다. 좁은 데스크톱 폭에서도 탭명이 잘리지 않거나 의도대로 스크롤되는지 확인한다.
- 우선순위: 중간

## 공통 동작 제거

### 2. 관리자 잠금 버튼과 전용 잠금 진입 코드 완전 제거

- 화면/영역: 관리자 탭 바 우측의 `관리자 잠금` 버튼 및 그 버튼이 호출하는 전용 콜백 경로.
- 증상: 모든 관리자 화면 상단에 잠금 버튼이 고정 표시되지만, 현재 화면 흐름에서는 필요하지 않은 제어다.
- 사용자 불편: 탭 이동과 무관한 버튼이 상단 공간과 시선을 차지하고, 관리자 화면의 주요 작업과 관계없는 선택지를 제공한다.
- 확정 정책: 관리자 탭 바에서 `관리자 잠금` 버튼을 제거한다. 해당 버튼만을 위해 존재하는 import, prop, 콜백 전달, 테스트는 함께 제거한다.
- 구현 방향: `AdminSectionTabs`의 `Lock` 아이콘 import, `onLock` prop, 버튼 마크업을 삭제하고 호출부에서 더 이상 전달하지 않는다. 버튼 전용 테스트는 제거하거나 탭 전환 검증만 남기도록 재구성한다. 다른 인증·PIN 변경·보안 탭 기능은 유지한다.
- 허용 기준:
  - 어떤 관리자 탭에서도 `관리자 잠금` 버튼 또는 관련 구분선/빈 여백이 남지 않는다.
  - 버튼 전용 `onLock` 경로와 테스트가 남지 않는다.
  - 일반 탭 전환, 보안 탭 진입, PIN 변경 기능은 기존대로 동작한다.
- 브라우저 확인 시나리오: 관리자 각 탭에서 탭 바 우측 끝과 탭 전환을 확인하고, 보안 탭에서 PIN 변경을 실행 가능한 상태로 유지하는지 확인한다.
- 우선순위: 높음

## 모델 관리

### 5. 선택 모델의 상태와 모델 코드 중복 표시 제거

- 화면/영역: 관리자 > 모델 관리의 선택 모델 목록 행, 상세 카드 헤더, 상세 기본 정보 블록.
- 증상: 선택 모델의 상태가 목록 행과 상세 카드 제목 옆, 상세 기본 정보 블록에 반복 표시된다. 모델 코드도 상세 카드 부제목과 읽기 전용 `모델 코드` 필드에 두 번 표시된다.
- 사용자 불편: 편집할 수 없는 중복 정보가 상세 상단을 차지해 실제 편집 필드와 연결 정보의 밀도가 낮아진다.
- 확정 정책: 선택 모델의 상태와 모델 코드는 상세 카드 헤더에서만 표시한다. 상세 기본 정보 블록의 읽기 전용 `상태`, `모델 코드` 필드는 제거한다. 목록 행의 상태/코드는 선택과 비교를 위한 목록 정보로 유지한다.
- 구현 방향: `ModelDetailView`에서 읽기 전용 코드·상태 `EditFieldRow`를 제거하고, 상세 카드 헤더의 상태 pill과 `M-xxxx` 부제목을 식별 정보의 단일 출처로 유지한다. 모델명·기호 편집, 저장, 연결 품목/BOM 요약, 삭제 기능은 바꾸지 않는다.
- 허용 기준:
  - 선택 모델의 상세 카드 안에서 `모델 코드`와 `상태` 읽기 전용 필드가 더 이상 렌더링되지 않는다.
  - 상세 카드 헤더에는 해당 모델의 `M-xxxx`와 상태가 각각 한 번만 보인다.
  - 모델명·기호 입력 및 저장, 연결 품목/BOM 요약, 삭제 동작은 유지된다.
- 브라우저 확인 시나리오: 사용 중 모델과 비활성 모델을 각각 선택해 목록 행·상세 헤더·기본 정보 블록의 코드/상태 노출 횟수를 확인하고, 모델명·기호 편집 후 저장 상태를 확인한다.
- 우선순위: 중간

## 테스트 정리

### 3. 관리자 공통 헤더·탭 바 변경에 맞춰 회귀 테스트 갱신

- 화면/영역: 관리자 공통 헤더와 `AdminSectionTabs` 단위 테스트.
- 증상: 현재 탭 바 테스트가 관리자 잠금 버튼과 그 클릭 콜백을 전제로 한다.
- 사용자 불편: 삭제된 화면 동작을 테스트가 계속 강제하면 이후 유지보수 시 잘못된 UI가 되살아날 수 있다.
- 확정 정책: 제거한 잠금 버튼의 테스트는 삭제하고, 유지되는 탭 선택 동작과 헤더의 새 KPI 배치를 검증한다.
- 구현 방향: 기존 `AdminSectionTabs` 테스트에서 잠금 버튼 전용 기대값을 없애고, 공통 헤더/KPI 배치의 렌더링 계약을 필요한 범위에서 추가한다. 화면 스타일의 픽셀값이 아니라 사용자에게 보이는 구성과 접근 가능한 이름을 검증한다.
- 허용 기준:
  - 테스트 코드에 `관리자 잠금` 또는 삭제된 `onLock` 계약이 남지 않는다.
  - 탭 클릭 후 선택 상태 전환이 검증된다.
  - 설명 문구 비표시와 KPI의 헤더 내 렌더링이 최소 한 개의 KPI 탭에서 검증된다.
- 브라우저 확인 시나리오: 자동 테스트 통과 후 모델 관리와 품목 관리에서 헤더/KPI 레이아웃을 실제 브라우저로 재확인한다.
- 우선순위: 중간
