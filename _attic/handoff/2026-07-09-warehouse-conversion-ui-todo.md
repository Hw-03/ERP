## 확정 결정 요약
- 품목 전환 화면은 “선택 후 요약 카드로 대체”보다 “항상 목록을 유지하고 선택된 행만 체크/강조”하는 문법을 따른다.
- 상단의 소스/대상 선택 요약 카드는 본문 목록과 하단 상태/액션 영역과 중복되면 제거한다.
- 전환 수량은 별도 상단 카드가 아니라 하단 상태/액션 영역의 주요 입력으로 노출한다.
- 검색창 폭을 줄이는 보조 라벨이나 중복 카드보다, 품목 검색과 선택 목록의 가시성을 우선한다.
- 하단 `선택 초기화`처럼 화면을 복잡하게 만드는 보조 버튼은 제거하고, 선택 변경은 목록에서 다른 행을 고르는 방식으로 해결한다.
- 하단 주요 CTA 버튼은 품목 전환 화면 전용 색감이 아니라 기존 DEXCOWIN MES에서 쓰던 기본 primary/secondary 버튼 문법과 맞추고, 단계가 바뀌어도 크기와 높이가 흔들리지 않게 한다.
- 상단 단계 네비게이션으로 돌아갈 수 있는 동작은 하단에 중복 버튼으로 두지 않는다.
- 최종 실행 단계는 요약 한 줄만 두지 않고, 직전 차이 확인 단계의 핵심 차이를 함께 보여준다.
- 전환 실행은 확인 팝업의 승인 뒤에만 수행하고, 성공 후에는 별도 완료 화면을 거치지 않고 일반 입출고 작성 첫 화면으로 돌아간다.
- 출하 요청 작성 1단계는 PF 검색·후보 목록을 넓게 보여 주고, 출하 수량은 하단 action bar의 핵심 입력으로 노출한다.
- 출하 요청 작성의 단계 버튼은 진행 바 전체 폭을 균등 분할하며, 바깥 장식 상자와 중복 안내 문구는 제거한다.

## 입출고 > 품목 전환

### 1. 상단 소스/대상 선택 요약 카드 제거
- 화면/영역: PC 입출고 > 품목 전환 > 상단 단계 진행 영역 아래의 소스/대상 선택 요약 카드
- 증상: 상단에 `소스 품목 미선택`, `대상 품목 미선택` 같은 요약 카드가 있고, 같은 정보가 아래 소스/대상 목록과 하단 상태 문구에도 반복된다.
- 사용자 불편: 화면 위쪽 공간을 중복 정보가 차지해 실제로 골라야 하는 품목 목록과 검색 영역이 좁아진다.
- 확정 정책: 상단 소스/대상 선택 요약 카드는 제거한다. 선택 상태는 각 목록의 선택 체크/강조와 하단 상태/액션 영역에서만 확인하게 한다.
- 구현 방향: `SelectedSummary` 계열 상단 카드 렌더링을 제거하거나 최소화하고, 단계 진행 nav와 목록 영역 사이의 불필요한 높이를 줄인다.
- 수용 기준: 최초 진입, 소스 선택 후, 대상 선택 후 모두 상단에 별도 `소스 품목/대상 품목 미선택` 요약 카드가 반복 노출되지 않는다.
- 브라우저 확인 시나리오: `/mes?tab=warehouse`에서 품목 전환으로 진입해 상단 영역을 확인한다. 소스와 대상을 차례로 선택해도 상단 요약 카드가 다시 나타나지 않는지 본다.
- 우선순위: 높음

### 2. 대상 품목 미선택 상단 카드 제거
- 화면/영역: PC 입출고 > 품목 전환 > 상단 `대상 품목 미선택` 카드
- 증상: 대상 품목을 아직 선택하지 않았을 때 상단에 별도 `대상 품목 미선택` 카드가 표시된다.
- 사용자 불편: 오른쪽 대상 목록 본문에도 아직 소스 품목을 선택하라는 안내가 있어, 같은 상태를 중복으로 읽게 된다.
- 확정 정책: `대상 품목 미선택`은 상단 카드로 따로 보여주지 않는다. 대상 선택 가능 여부와 빈 상태 안내는 대상 목록 패널 안에서 처리한다.
- 구현 방향: 대상 요약 카드를 제거하고, 대상 목록 패널의 empty/guide text만 남긴다. 소스 선택 전에는 대상 목록 영역 안에서만 `소스 품목을 선택하세요` 안내를 보여준다.
- 수용 기준: 소스 미선택 상태에서 대상 선택 상태를 알려주는 상단 카드가 사라지고, 대상 패널 안의 안내만 남는다.
- 브라우저 확인 시나리오: `/mes?tab=warehouse` 최초 진입 상태에서 상단 카드와 오른쪽 대상 패널을 확인한다. 대상 미선택 정보가 두 군데 반복되지 않는지 본다.
- 우선순위: 높음

### 3. 전환 수량 입력을 하단 상태/액션 영역으로 이동
- 화면/영역: PC 입출고 > 품목 전환 > 현재 상단 오른쪽 `전환 수량` 입력, 하단 action bar의 `소스 품목을 선택하세요` 상태 문구 자리
- 증상: `전환 수량` 입력이 상단 오른쪽 별도 카드로 배치되어 있고, 하단에는 큰 상태 문구와 `선택 초기화`/`다음` 버튼이 있다.
- 사용자 불편: 수량 입력 카드 때문에 소스/대상 목록과 검색창이 좁아진다. 실제 작업 흐름에서는 품목을 고른 뒤 바로 하단에서 수량을 확인하고 다음으로 가는 편이 자연스럽다.
- 확정 정책: 전환 수량 입력은 하단 상태/액션 영역에 잘 보이게 배치한다. `소스 품목 선택` 같은 별도 라벨은 제거하고, 그 자리에서 수량 선택이 핵심 조작으로 보이게 한다.
- 구현 방향: 상단 `전환 수량` 카드 렌더링을 하단 bar 내부로 이동한다. 하단에는 현재 부족 조건 문구와 수량 입력, 다음 CTA가 함께 보이도록 재배치한다. 검색창과 목록 컬럼은 가능한 넓게 유지한다.
- 수용 기준: 상단 오른쪽에 별도 수량 카드가 남지 않는다. 하단 action bar에서 수량을 바로 조절할 수 있고, 소스/대상 검색창 폭이 이전보다 좁아지지 않는다.
- 브라우저 확인 시나리오: 소스 품목과 대상 품목을 선택한 뒤 하단에서 수량을 조절한다. 수량 변경이 차이 확인/다음 버튼 활성 조건에 정상 반영되는지 본다.
- 우선순위: 높음

### 4. 하단 선택 초기화 버튼 제거
- 화면/영역: PC 입출고 > 품목 전환 > 하단 action bar의 `선택 초기화` 버튼
- 증상: 하단 왼쪽에 `선택 초기화` 버튼이 항상 보인다.
- 사용자 불편: 사용자가 초기화보다 품목 선택과 다음 진행에 집중해야 하는 화면에서 보조 버튼이 시선을 빼앗는다. 선택을 바꾸려면 목록에서 다른 품목을 고르면 된다.
- 확정 정책: 하단 `선택 초기화` 버튼은 제거한다. 전체 초기화가 꼭 필요한 경우에도 기본 화면에서는 드러내지 않는다.
- 구현 방향: `resetSelection` 계열 버튼 렌더링을 제거한다. 소스/대상 변경은 목록의 다른 행 선택이나 기존 변경 버튼 문법으로 처리한다.
- 수용 기준: 하단 action bar에 `선택 초기화` 버튼이 보이지 않는다. 선택된 소스/대상은 목록에서 다른 항목을 누르면 바뀐다.
- 브라우저 확인 시나리오: 소스와 대상을 선택한 상태에서 하단 action bar를 확인한다. 초기화 버튼이 없고, 다른 품목 선택으로 상태 변경이 가능한지 본다.
- 우선순위: 중간

### 5. 소스 품목 선택 후에도 소스 목록 유지
- 화면/영역: PC 입출고 > 품목 전환 > 왼쪽 소스 품목 목록
- 증상: 소스 품목을 선택하면 왼쪽 목록이 선택 요약 카드처럼 바뀌어, 기존 검색 결과 목록이 사라진다.
- 사용자 불편: 선택이 완료된 것인지 검색 결과가 사라진 것인지 헷갈리고, 바로 다른 소스 품목으로 바꿔 고르기 어렵다.
- 확정 정책: 선택했다고 해서 목록이 사라지면 안 된다. 입출고 탭의 품목 리스트처럼 선택된 행에 체크 표시와 선택 강조만 붙인다.
- 구현 방향: 소스 선택 상태에서도 소스 검색창과 결과 목록을 유지한다. 선택된 item id와 일치하는 행에는 체크 아이콘, 선택 tint, `선택됨` 보조 표기를 적용한다.
- 수용 기준: 소스 품목 선택 후에도 왼쪽 검색창과 결과 목록이 그대로 보인다. 선택된 소스 품목 행만 체크/강조되고, 다른 행을 누르면 선택이 바뀐다.
- 브라우저 확인 시나리오: `/mes?tab=warehouse`에서 소스 품목을 선택한다. 왼쪽 목록이 사라지지 않고 선택된 행에 체크 표시가 보이는지 확인한 뒤, 다른 소스 품목을 선택해 상태가 바뀌는지 본다.
- 우선순위: 높음

### 6. 대상 품목 선택 후에도 대상 목록 유지
- 화면/영역: PC 입출고 > 품목 전환 > 오른쪽 대상 품목 목록
- 증상: 대상 품목을 선택하면 오른쪽 목록이 선택 요약 카드처럼 바뀌어, 기존 대상 후보 목록이 사라진다.
- 사용자 불편: 대상 후보끼리 비교하거나 바로 다른 대상 품목으로 바꿔 고르기 어렵다. 소스 목록과 대상 목록의 선택 문법도 달라져 보인다.
- 확정 정책: 대상 품목도 선택 후 목록을 유지한다. 입출고 탭 품목 리스트처럼 선택된 행에 체크 표시와 선택 강조만 붙인다.
- 구현 방향: 대상 선택 상태에서도 대상 검색창과 후보 목록을 유지한다. 선택된 대상 item id와 일치하는 행에는 체크 아이콘, 선택 tint, `선택됨` 보조 표기를 적용한다. 소스와 대상 목록의 선택 문법을 동일하게 맞춘다.
- 수용 기준: 대상 품목 선택 후에도 오른쪽 검색창과 대상 후보 목록이 그대로 보인다. 선택된 대상 품목 행만 체크/강조되고, 다른 후보를 누르면 선택이 바뀐다.
- 브라우저 확인 시나리오: 소스 품목을 선택해 대상 후보를 띄운 뒤 대상 품목을 선택한다. 오른쪽 목록이 사라지지 않고 선택된 행만 체크 표시되는지 확인한다.
- 우선순위: 높음

### 7. 단계 버튼 위쪽 여백을 아래 여백 수준으로 줄이기
- 화면/영역: PC 입출고 > 품목 전환 > 상단 헤더 아래 단계 버튼 묶음과 첫 본문 카드 사이 여백
- 증상: 단계 버튼 묶음 위쪽 여백이 아래쪽 여백보다 커 보여, 화면 상단에 빈 공간이 남는다. 사용자가 비교한 다른 화면의 탭/단계 버튼은 위아래 여백이 더 균형 있게 보인다.
- 사용자 불편: 실제 작업 영역이 아래로 밀리고, 상단 진행 버튼이 화면 안에서 떠 있는 것처럼 보여 밀도가 낮아진다.
- 확정 정책: 단계 버튼 묶음 위쪽 여백을 아래쪽 여백과 비슷한 수준으로 줄인다. 하단 본문 카드와의 간격은 현재처럼 답답하지 않게 유지하되, 위쪽 불필요한 여백을 우선 줄인다.
- 구현 방향: 품목 전환 화면의 상단 container padding/margin을 확인하고, 헤더 아래 `ic-conversion-progress` 또는 그 부모 wrapper의 top spacing을 줄인다. 다른 입출고 화면의 단계 탭 간격과 맞춰 공통 CSS를 건드릴 필요가 있는지 먼저 확인하되, 영향 범위가 넓으면 품목 전환 화면에만 좁게 적용한다.
- 수용 기준: 품목 전환 화면에서 단계 버튼 위쪽 여백이 아래쪽 여백과 시각적으로 비슷하게 보인다. 본문 카드 상단과 단계 버튼 하단은 붙어 보이지 않고, 단계 버튼 자체 높이와 내부 정렬은 유지된다.
- 브라우저 확인 시나리오: `/mes?tab=warehouse` 품목 전환 화면과 사용자가 비교한 다른 요청/작성 화면의 단계 버튼 영역을 나란히 확인한다. 품목 전환 화면의 위쪽 여백이 과하게 크지 않고, 아래 여백과 균형이 맞는지 본다.
- 우선순위: 중간

### 8. 하단 주요 CTA 버튼 색감과 크기를 기존 버튼 디자인과 맞추기
- 화면/영역: PC 입출고 > 품목 전환 > 하단 action bar의 `다음`, `실행 단계로`, `전환 실행` 버튼
- 증상: 품목 전환 하단 주요 버튼의 파란색 색감과 질감이 기존 화면에서 쓰던 버튼 디자인과 달라 보이고, 최종 실행 단계의 `전환 실행` 버튼은 차이 확인 단계의 `실행 단계로` 버튼과 크기까지 달라진다.
- 사용자 불편: 같은 DEXCOWIN MES 안에서 같은 “다음/실행” 성격의 버튼인데 다른 컴포넌트처럼 보여 화면 완성도가 떨어진다. 단계가 바뀔 때 버튼 높이/폭이 흔들리면 사용자는 같은 진행 CTA인지 다른 성격의 버튼인지 헷갈린다.
- 확정 정책: 품목 전환의 primary/secondary action 버튼은 기존 입출고/출하 화면에서 쓰던 공통 버튼 색감과 크기 문법을 맞춘다. `다음`, `실행 단계로`, `전환 실행`은 같은 primary 문법을 공유하고, 단계가 바뀌어도 버튼 높이와 기본 폭이 일관되어야 한다.
- 구현 방향: 품목 전환 전용 `icb icb-primary` 스타일이 기존 공통 버튼 색상/hover/focus/radius/shadow/height/padding/min-width와 다른지 확인하고 맞춘다. 차이 확인 단계와 최종 실행 단계의 하단 action bar wrapper가 다른 flex/grid 규칙을 쓰면서 버튼 크기가 달라지는지도 함께 확인한다.
- 수용 기준: `다음`, `실행 단계로`, `전환 실행` 버튼이 기존 MES 주요 CTA와 같은 색감과 상태 표현으로 보인다. 차이 확인 단계에서 최종 실행 단계로 이동해도 primary 버튼의 높이, 기본 폭, 내부 여백이 달라지지 않는다. hover/focus/disabled 상태도 과하게 튀지 않는다.
- 브라우저 확인 시나리오: 품목 전환 2단계, 3단계, 4단계로 이동하며 하단 primary 버튼을 확인한다. 기존 입출고 작업 화면의 주요 버튼과 색감이 어긋나지 않는지 비교하고, 차이 확인 단계의 `실행 단계로`와 최종 실행 단계의 `전환 실행` 버튼 크기가 같은지 본다.
- 우선순위: 중간

### 9. `소스·대상 다시 선택` 하단 버튼 제거
- 화면/영역: PC 입출고 > 품목 전환 > 차이 확인 단계 하단 action bar
- 증상: 차이 확인 단계 하단에 `소스·대상 다시 선택` 버튼이 보인다.
- 사용자 불편: 이미 상단 단계 네비게이션에서 소스·대상 선택 단계로 돌아갈 수 있는데, 하단에도 같은 성격의 버튼이 있어 액션 영역이 복잡해진다.
- 확정 정책: `소스·대상 다시 선택` 하단 버튼은 제거한다. 이전 단계로 돌아가는 동작은 상단 네비게이션으로 처리한다.
- 구현 방향: 차이 확인 단계 하단 action bar에서 해당 secondary 버튼 렌더링을 제거한다. 상단 단계 네비게이션이 실제로 소스·대상 선택 단계로 이동 가능한지 함께 확인한다.
- 수용 기준: 차이 확인 단계 하단에는 다음 진행에 필요한 핵심 CTA만 남는다. 소스/대상을 바꾸고 싶을 때는 상단 `소스·대상 선택` 단계 버튼으로 이동할 수 있다.
- 브라우저 확인 시나리오: 소스/대상 선택 후 차이 확인 단계로 이동한다. 하단에 `소스·대상 다시 선택` 버튼이 없고, 상단 단계 네비게이션으로 이전 단계 이동이 가능한지 본다.
- 우선순위: 중간

### 10. 최종 실행 단계에서 BOM 차이 내용을 함께 보여주기
- 화면/영역: PC 입출고 > 품목 전환 > 최종 실행 단계 본문과 하단 `차이 다시 보기` 버튼
- 증상: 최종 실행 단계 본문이 큰 빈 카드처럼 보이고, 내부에는 최종 실행 요약 몇 줄만 있어 정보가 휑하다. 사용자가 실제로 확인해야 할 BOM 차이는 `차이 다시 보기` 버튼으로 이전 단계에 돌아가야 볼 수 있다.
- 사용자 불편: 실행 직전 단계인데 무엇이 빠지고 무엇이 추가되는지 한 화면에서 확인하기 어렵다. `차이 다시 보기` 버튼을 눌러 왕복해야 해서 최종 확인 흐름이 끊긴다.
- 확정 정책: `차이 다시 보기` 버튼은 제거하고, 최종 실행 단계 안에서 차이 확인 내용을 한 번 더 보여준다. 최종 실행 화면은 “실행 요약 + 소스 BOM에서 빠지는 구성 + 대상 BOM 때문에 필요한 구성”을 함께 보여주는 확인 화면으로 만든다.
- 구현 방향: 차이 확인 단계에서 사용하는 차이 결과 렌더링을 최종 실행 단계에도 재사용하거나, 더 압축된 최종 확인용 차이 요약 컴포넌트로 추출한다. 상단에는 최종 실행 요약을 두고, 아래에는 회수 입고/추가 차감 목록을 2열 또는 섹션으로 표시한다. 하단 secondary `차이 다시 보기` 버튼은 제거한다.
- 수용 기준: 최종 실행 단계에서 사용자는 뒤로 가지 않고도 소스 차감/대상 입고/회수 입고/추가 차감 내용을 확인할 수 있다. 카드 내부가 과하게 비어 보이지 않고, `전환 실행` CTA만 자연스럽게 남는다.
- 브라우저 확인 시나리오: 차이 확인 단계에서 실행 단계로 이동한다. 최종 실행 단계 본문에 직전 차이 확인 단계의 핵심 목록이 보이는지 확인하고, 하단에 `차이 다시 보기` 버튼이 없는지 본다.
- 우선순위: 높음

## 구현 후 추가 발견

### 11. 품목 전환 전체의 장식 화살표 제거 및 관계 라벨 명확화
- 화면/영역: PC 입출고 > 품목 전환 > 상단 단계 진행 요약, 차이 확인, 최종 실행의 `ItemConversionStepChrome`, `PreviewSummary`, `PreviewDifferencePanels` 영역과 제거 대상인 완료 화면
- 증상: 단계 진행 요약과 소스·대상 품목명 사이의 `→`, BOM 차이 섹션 제목과 각 품목 행 앞의 `←`가 반복 노출된다. 현재 완료 화면에도 같은 화살표가 있다.
- 사용자 불편: 화살표가 이동 방향이나 재고 변동 의미를 명확히 전달하지 못해, 소스/대상과 BOM 구성 차이를 읽는 흐름을 방해한다.
- 확정 정책: 품목 전환 전체에서 장식용 `←`/`→` 표시는 모두 제거한다. 소스·대상 관계는 화살표 대신 `소스 품목`, `대상 품목` 라벨로 명시하며, BOM 차이 행은 품목명만 읽히게 한다. 완료 화면은 15번에서 제거하므로 해당 화면의 화살표도 함께 사라진다.
- 구현 방향: 단계 진행 요약과 `PreviewSummary`에서 소스/대상 품목명과 코드를 라벨 기반 정보로 재배치한다. `PreviewDifferencePanels`의 양쪽 섹션 제목 및 차이 행에서 화살표 문자열을 제거한다. 차이 확인과 최종 실행이 재사용하는 정보 표현은 동일하게 반영하고, 제거되는 `ItemConversionCompleteView` 관련 화살표 문자열도 함께 정리한다.
- 수용 기준: 품목 전환의 단계 진행 요약, 차이 확인, 최종 실행에서 `←`/`→` 장식이 보이지 않는다. 별도 전환 완료 화면도 노출되지 않는다. 소스 품목과 대상 품목, 빠지는 구성과 추가되는 구성을 라벨만 보고 구분할 수 있다.
- 브라우저 확인 시나리오: `/mes?tab=warehouse`에서 BOM 구성 전환 품목을 선택해 단계 진행 요약, 차이 확인, 최종 실행까지 이동한다. 각 화면의 요약과 좌·우 BOM 차이 목록에 화살표가 없는지 확인하고, 실행 성공 뒤 완료 화면이 뜨지 않는지 확인한다.
- 우선순위: 높음

### 12. 메모 라벨과 사유 입력칸을 한 줄로 정렬
- 화면/영역: PC 입출고 > 품목 전환 > 차이 확인 하단 action bar의 `메모 (필수)`와 사유 입력칸
- 증상: 메모 라벨이 입력칸 위 별도 줄에 있어, 넓은 PC 화면에서도 입력 영역의 세로 높이를 불필요하게 차지한다.
- 사용자 불편: 하단 action bar의 정보 밀도가 낮고, 메모가 입력칸의 조건인지 별도 안내인지 한눈에 연결되지 않는다.
- 확정 정책: `메모 (필수)` 또는 `메모 (선택)` 라벨은 사유 입력칸의 왼쪽 같은 줄에 둔다. 입력칸은 남은 폭을 사용하고, 필수/선택 상태와 placeholder 문구는 유지한다.
- 구현 방향: 차이 확인 단계의 메모 label을 세로 `grid` 구조에서 가로 `flex` 또는 입력 그룹 구조로 바꾼다. 라벨은 줄바꿈되지 않는 고정 폭으로, 입력칸은 `min-w-0 flex-1`로 두어 CTA 폭을 침범하지 않게 한다.
- 수용 기준: PC 차이 확인 화면에서 메모 라벨과 사유 입력칸이 한 줄에 보인다. 필수 메모와 선택 메모 모두 라벨·placeholder·유효성 동작이 유지되고, 실행 CTA와 겹치지 않는다.
- 브라우저 확인 시나리오: BOM 구성 전환과 SPEC 전환 각각에서 차이 확인 단계로 이동한다. `메모 (필수)` 또는 `메모 (선택)`이 해당 입력칸 왼쪽에 같은 줄로 보이고, 긴 사유를 입력해도 하단 CTA가 밀리지 않는지 확인한다.
- 우선순위: 중간

### 13. 차이 확인 진행 CTA 문구를 `다음`으로 통일
- 화면/영역: PC 입출고 > 품목 전환 > 소스·대상 선택 단계와 차이 확인 단계의 하단 primary CTA
- 증상: 소스·대상 선택 단계에서는 `다음`인데, 같은 진행 흐름의 차이 확인 단계에서는 `실행 단계로`로 표시된다.
- 사용자 불편: 단계 이동 버튼의 의미가 바뀐 것처럼 보여, 사용자가 같은 순차 진행 동작인지 즉시 파악하기 어렵다.
- 확정 정책: 최종 실행 전까지의 순차 진행 CTA는 모두 `다음`으로 통일한다. 실제 실행을 발생시키는 마지막 단계의 CTA만 `전환 실행`으로 유지한다.
- 구현 방향: 차이 확인 단계의 `item-conversion-execute-next-button` 기본 문구를 `다음`으로 변경한다. 비동기 처리 중 표기와 disabled 조건은 현재 동작을 유지한다.
- 수용 기준: 소스·대상 선택과 차이 확인 단계의 하단 진행 버튼이 모두 `다음`으로 보인다. 최종 실행 단계만 `전환 실행`으로 남고, 단계 이동 및 실행 조건은 바뀌지 않는다.
- 브라우저 확인 시나리오: 소스·대상을 선택한 뒤 차이 확인으로 이동한다. 첫 단계와 차이 확인 단계의 primary CTA가 모두 `다음`인지 확인하고, 다음을 눌러 최종 단계에서만 `전환 실행`이 보이는지 확인한다.
- 우선순위: 중간

### 14. 최종 실행 확인을 BOM 차이 뒤의 하단 확인 바로 재구성
- 화면/영역: PC 입출고 > 품목 전환 > 최종 실행 단계의 `최종 실행 확인` 영역과 `전환 실행` CTA 직전
- 증상: 현재 `최종 실행 확인`은 상단 요약과 BOM 차이 목록 사이에 일반 카드처럼 배치되어, 최종 확인 정보가 실제 확인 흐름의 마지막에 오지 않는다.
- 사용자 불편: 사용자가 BOM 차이를 모두 읽은 뒤 무엇을 최종 실행하는지 다시 확인해야 하는데, 확인 내용이 위쪽에 있어 시선이 왕복한다. 요약 카드와 같은 모양이라 실행 직전의 중요한 확인 영역으로도 구분되지 않는다.
- 확정 정책: `최종 실행 확인`은 BOM 차이 목록 뒤, 하단 `전환 실행` CTA 바로 위에 둔다. 일반 요약 카드와 다른 하단 확인 바로 구성하고, 소스 차감·대상 입고·메모를 한눈에 확인하게 한다.
- 구현 방향: 최종 실행 단계에서 현재 상단 `icf` 카드 렌더링을 제거하고, `PreviewDifferencePanels` 다음에 별도 확인 바를 둔다. 확인 바는 상단 구분선과 강조 제목, 소스 차감·대상 입고·메모의 압축 정보로 구성하며 `전환 실행` 버튼과 인접하게 배치한다.
- 수용 기준: 최종 실행 화면에서 BOM 차이 목록을 읽은 뒤 화면 맨 아래에서 `최종 실행 확인`과 `전환 실행`을 순서대로 확인할 수 있다. 최종 확인 영역은 상단 요약 카드와 시각적으로 구분되고, 빈 공간이나 중복 카드처럼 보이지 않는다.
- 브라우저 확인 시나리오: BOM 구성 전환으로 최종 실행 단계까지 이동한다. 상단에는 전환 요약과 BOM 차이만 보이고, BOM 차이 목록 뒤 최하단에 다른 모양의 `최종 실행 확인` 바와 `전환 실행` 버튼이 이어지는지 확인한다.
- 우선순위: 높음

### 15. 전환 실행 전 확인 팝업 후 일반 입출고 작성 화면으로 복귀
- 화면/영역: PC 입출고 > 품목 전환 > 최종 실행 단계의 `전환 실행` 버튼, 실행 성공 뒤 `품목 전환 완료` 화면
- 증상: `전환 실행`을 누르면 별도 확인 없이 API가 바로 실행되고, 성공하면 품목 전환에만 존재하는 완료 화면과 `새 품목 전환`·`입출고 이력`·`입출고로 돌아가기` 버튼이 표시된다.
- 사용자 불편: 재고를 실제로 바꾸는 실행에 마지막 확인 기회가 없고, 완료 뒤 화면이 일반 입출고 흐름과 달라 작업자가 다음 작업으로 자연스럽게 이어가지 못한다.
- 확정 정책: `전환 실행`을 누르면 실행할지 묻는 확인 팝업을 먼저 띄운다. 취소하면 최종 실행 단계에 그대로 남고, 승인해 실행이 성공하면 별도 완료 화면 없이 일반 입출고 작성의 작업 유형 선택 첫 화면으로 바로 돌아간다.
- 구현 방향: 기존 공통 확인 모달 문법을 사용해 소스 품목·대상 품목·수량을 보여주는 실행 확인 팝업을 추가한다. 승인 시에만 `api.executeItemConversion`을 호출하고, 성공 콜백은 완료 화면 상태로 전환하지 말고 품목 전환 history/state를 정리한 뒤 `closeItemConversion`과 같은 일반 입출고 작성 초기화 경로를 사용한다. `ItemConversionCompleteView`, 관련 view state, 완료 화면 전용 버튼과 테스트를 제거·갱신한다.
- 수용 기준: `전환 실행` 클릭만으로는 재고 전환 API가 호출되지 않고, 확인 팝업 승인 뒤에만 실행된다. 취소하면 현재 최종 실행 내용이 유지된다. 성공하면 `품목 전환 완료` 화면 없이 일반 입출고의 작업 유형 선택 화면으로 돌아가며, 별도 완료 화면 전용 버튼은 보이지 않는다.
- 브라우저 확인 시나리오: 최종 실행 단계에서 `전환 실행`을 누른다. 확인 팝업에서 취소해 최종 실행 화면이 유지되는지 확인하고, 다시 승인해 실행한다. 성공 후 `품목 전환 완료` 화면이 아니라 일반 입출고 작성 첫 화면으로 돌아오는지 확인한다.
- 우선순위: 높음

### 16. PF 선택 영역의 바깥 테두리 제거 및 본문 확장
- 화면/영역: PC 출하 > 출하 요청 작성 > 1단계 PF 검색·후보 목록을 감싼 바깥 테두리 상자
- 증상: PF 검색, 출하 수량, 후보 목록을 다시 한 번 감싼 바깥 테두리와 여백이 있어 실제 목록 영역이 좁아 보인다.
- 사용자 불편: 이미 검색창과 PF 후보 목록이 자체 경계를 갖고 있는데 바깥 상자가 중복돼, 작업에 필요한 목록 높이와 폭이 줄어든다.
- 확정 정책: PF 선택 1단계의 바깥 장식 테두리 상자는 제거한다. 그 상자가 차지하던 padding·border 공간은 안쪽 PF 검색과 후보 목록 작업 영역에 돌려준다.
- 구현 방향: 요청 작성 위저드의 1단계 외곽 wrapper에서 테두리·radius·padding만 제거하거나 최소화하고, 내부 `WorkStep`/PF 후보 목록이 부모의 가용 폭·높이를 채우도록 flex 및 overflow 규칙을 조정한다. 검색, 후보 목록 스크롤, 하단 action bar의 동작은 유지한다.
- 수용 기준: PF 검색과 후보 목록 바깥에 중복 테두리 상자가 보이지 않는다. 후보 목록의 폭 또는 높이가 기존보다 넓어지고, 스크롤·선택·다음 버튼 동작은 변하지 않는다.
- 브라우저 확인 시나리오: `/mes?tab=shipping&shippingView=requestWork`에서 출하 요청 작성을 연다. PF 검색과 후보 목록을 확인해 중복 바깥 테두리가 없고, 후보 목록이 늘어난 작업 영역을 쓰는지 확인한다.
- 우선순위: 중간

### 17. 출하 요청 1~5단계 버튼을 진행 바 폭에 맞춰 균등 배치
- 화면/영역: PC 출하 > 출하 요청 작성 > 상단 `1. 기준 PF 선택`부터 `5. 저장 및 전환`까지의 단계 버튼
- 증상: 단계 버튼 묶음이 왼쪽 시작점과 진행 바 폭을 충분히 쓰지 못해, 버튼 사이와 좌측에 불필요한 여백이 남는다.
- 사용자 불편: 단계가 다섯 개인 고정 진행 흐름인데도 상단 진행 바가 꽉 찬 작업 제어처럼 보이지 않아 시선이 분산된다.
- 확정 정책: 다섯 단계 버튼은 진행 바의 왼쪽부터 오른쪽까지 빈 여백 없이 균등 분할한다. 버튼 사이 간격은 경계가 구분되는 최소값만 유지한다.
- 구현 방향: 단계 버튼 컨테이너의 grid column과 gap, 부모 padding/margin을 점검해 `repeat(5, minmax(0, 1fr))` 기준으로 전체 폭을 사용하게 한다. 버튼 높이·색상·disabled/active 상태 문법은 유지한다.
- 수용 기준: 1~5단계 버튼이 상단 진행 바의 전체 폭을 균등하게 채우고, 왼쪽에 눈에 띄는 빈 여백이 없다. 긴 단계명도 잘리지 않으며 단계 클릭 동작은 유지된다.
- 브라우저 확인 시나리오: 출하 요청 작성 화면의 상단 진행 바를 확인한다. 첫 단계가 진행 바 왼쪽 경계에서 시작하고 다섯 버튼이 끝까지 균등 배치되는지 확인한다.
- 우선순위: 중간

### 18. 출하 수량을 하단 action bar로 이동하고 초기 안내 문구 제거
- 화면/영역: PC 출하 > 출하 요청 작성 > 1단계 상단 `출하 수량` 입력과 하단 `기준 PF를 먼저 선택하세요.` 상태 문구
- 증상: 출하 수량 입력은 PF 검색과 후보 목록 사이 상단에 있고, 하단 action bar에는 기준 PF를 선택하라는 안내 문구만 따로 표시된다.
- 사용자 불편: PF를 고른 뒤 수량을 정하는 핵심 조작이 후보 목록과 분리돼 있으며, 하단의 넓은 action bar 공간은 단순 안내 문구에만 쓰인다.
- 확정 정책: 출하 수량 입력을 1단계 하단 action bar의 상태 문구 자리에 둔다. `기준 PF를 먼저 선택하세요.` 문구는 제거하며, PF 미선택 상태에서도 수량 입력은 보이되 다음 진행 버튼의 비활성 조건은 유지한다.
- 구현 방향: 1단계 본문에서 `출하 수량` 필드 렌더링을 제거하고, action bar의 중앙 영역에 라벨과 수량 input을 한 줄로 배치한다. 기존 quantity ref, 정수 검증, 수량 변경 시 BOM/요청 상태 갱신은 그대로 연결한다. 다른 단계의 수량 안내 문구는 이번 범위에서 바꾸지 않는다.
- 수용 기준: 출하 요청 1단계에서 PF 검색과 후보 목록이 넓게 보이고, 하단 action bar에 `출하 수량` 입력이 있다. `기준 PF를 먼저 선택하세요.` 문구는 보이지 않으며, PF와 유효 수량이 모두 갖춰질 때만 다음 버튼이 활성화된다.
- 브라우저 확인 시나리오: 출하 요청 작성 1단계에서 PF를 선택하기 전과 후를 확인한다. 하단에 출하 수량을 입력할 수 있고 초기 안내 문구는 없는지, 수량을 바꾸고 PF를 선택하면 다음 버튼이 활성화되는지 확인한다.
- 우선순위: 높음

- 증상: `출하 요청 생성`, `출하 준비 중 전환` 등 기존 이벤트가 깨진 한글 문자열로 표시된다.
- 사용자 불편: 출하 요청의 상태 변경 이력을 읽을 수 없어, 언제 어떤 단계로 전환됐는지 확인하기 어렵다.
- 확정 정책: 알려진 출하 이벤트 타입은 저장된 message 문자열 상태와 관계없이 `event_type`별 정상 한글 라벨을 우선 표시한다. 알 수 없는 이벤트 타입만 원문 message 또는 event type을 fallback으로 사용한다. 기존 DB 이벤트를 수정하거나 삭제하지 않는다.
- 구현 방향: `shippingEventMessage`의 깨진 문자열 감지 조건을 한글 존재 여부가 아니라 알려진 `SHIPPING_EVENT_LABEL` 매핑 우선 규칙으로 바꾼다. `REQUEST_CREATED`, `REQUEST_UPDATED`, `SENT_TO_PREP`, `PREPARED`, `PREPARE_CANCELLED`, `PICKED_UP` 각각에 대해 정상 라벨이 반환되는 테스트를 추가하고, 깨진 한글처럼 보이는 legacy message 입력도 회귀 케이스로 넣는다.
- 수용 기준: 출하 이력 상세의 이벤트 이력에서 알려진 이벤트 타입은 항상 정상 한글 라벨로 보인다. 깨진 기존 message가 있어도 `출하 요청 생성`, `출하 준비 중 전환`, `출하 준비 완료`, `픽업 완료 처리`처럼 읽을 수 있는 문구가 표시된다.
- 브라우저 확인 시나리오: `/mes?tab=shipping&shippingView=historyWork`에서 이벤트가 있는 출하 이력을 연다. 이벤트 이력의 모든 행이 깨지지 않은 한글로 보이는지, 날짜와 이벤트 순서는 유지되는지 확인한다.
- 우선순위: 높음
