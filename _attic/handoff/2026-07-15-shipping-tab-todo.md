- 출하 요청 작성의 품목코드는 화면 단계와 목록 종류가 달라도 공정 코드(PR·PA·AF·PF)를 색으로 구분한다. PA는 기존 기본 파랑을 유지한다.
- 적용 범위는 기준 PF 선택, BOM 구성 조정, BOM 매칭, 최종 확인의 출하 품목·BOM·동반 출하품이다.
- 기존 계층 규칙은 유지한다. PA 구성품에는 AF가, PF 구성품에는 PA가 포함된다.
- BOM에 추가한 품목과 최종 확인 목록 모두 표준 BOM 정렬 순서를 사용한다.
- 5단계에서 변경 구성품이 있으면 `BOM·동반 출하품`은 432px 고정 높이와 내부 스크롤을 유지하고, `변경된 구성품`은 행 수에 맞는 자연 높이를 사용한다.
- 신규 PA/PF 예상 코드는 조회 시점의 비예약 참고값으로 표시하며, 저장 시 변경될 수 있음을 함께 안내한다.

- 3·5단계의 신규 PA/PF에는 비예약 예상 MES 코드와 `예상 코드 · 저장 시 변경 가능` 안내를 표시한다.

- 최신 사용자 결정에 따라 5단계 `BOM·동반 출하품`만 432px 고정 높이를 유지하고, `변경된 구성품`은 행 수에 맞는 자연 높이를 사용한다.

### 1. 출하 요청 전체의 PR·PA·AF·PF 품목코드 색상 구분 — 완료
- 화면/영역: PC 출하 > 출하 요청 작성의 1단계 기준 PF 검색 결과, 2단계 PA·PF 구성품 편집 목록, 3단계 매칭 결과, 5단계 출하 품목 및 `BOM·동반 출하품` 요약 목록
- 증상: 동일한 품목코드가 화면마다 기본 보조 텍스트 색으로 표시된다. 특히 5단계에는 PR·PF·AF만 별도 색상 처리하는 `SummaryCode`가 있으나 PA는 기본값으로 떨어지고, 1·2·3단계의 동일한 코드 표시는 이 규칙을 사용하지 않는다.
- 사용자 불편: `3-PR-…`, `3-PA-…`, `3-AF-…`, `3-PF-…`의 공정 구간을 코드 전체를 읽기 전에는 빠르게 구분하기 어렵고, 같은 품목코드가 단계마다 서로 다른 강조 수준으로 보인다.
- 확정 정책: 공정 코드가 포함된 MES 품목코드는 표시 위치와 관계없이 가운데 공정 구간(PR·PA·AF·PF)을 색으로 구분한다. 품목명·수량·상태 배지의 기존 의미와 색은 바꾸지 않는다.
- 정책 확인 필요: 네 코드의 최종 색상표를 확정한다. 현재 5단계의 기존 규칙은 PR=보라, AF=초록, PF=파랑이고 PA는 기본 파랑으로 처리된다. PA까지 다른 코드와 구분되는 색을 지정한 뒤 전 화면에 같은 표를 사용한다.
- 구현 방향: `mes_code`를 직접 텍스트로 렌더하는 1·2·3·5단계의 코드 표시를 공통 코드 프레젠테이션으로 통일한다. `NN-공정코드-NNNN` 형식일 때 가운데 공정코드만 강조하고, 비표준 코드·코드 없음·자동 생성 예정 안내는 기존 보조 텍스트 문법을 유지한다. 확인된 주요 렌더 경로는 `DesktopShippingView.tsx`의 PF 검색 결과(1498줄), BOM 편집 목록(2211줄), BOM 매칭 결과(2507~2513줄), 최종 확인 목록(2627줄), 출하 품목 헤더(2648줄)이다.
- 수용 기준:
  - PR·PA·AF·PF 코드가 섞여 표시되어도 각 공정코드 구간이 확정된 색상표에 따라 서로 구분된다.
  - 1·2·3·5단계에서 같은 MES 품목코드는 같은 공정코드 색을 사용한다.
  - 5단계 `BOM·동반 출하품`의 PA 코드가 PF 기본색으로 보이지 않고, 확정된 PA 색으로 표시된다.
  - 코드의 앞 모델 번호와 뒤 일련번호, 품목명, 수량, `기본`·`추가됨`·`제외됨` 상태 색은 기존 표시를 유지한다.
  - 비표준 코드, 코드 없음, `품목코드는 저장/준비 완료 시 자동 생성 예정` 문구는 오류 없이 기존 보조 텍스트로 보인다.
- 브라우저 확인 시나리오:
  1. 1단계 PF 검색 결과에서 `3-PF-0001`의 PF 구간 색을 확인한다.
  2. 2단계 PA 구성품의 `3-AF-0002`, PF 구성품의 `3-PA-0002`와 PR 코드가 있는 행을 함께 확인한다.
  3. 3단계 매칭 결과의 PA·PF 코드와 5단계 출하 품목 코드가 같은 규칙으로 보이는지 확인한다.
  4. 5단계 `BOM·동반 출하품`에서 PR·PA·AF·PF 코드가 섞인 목록을 확인하고, 코드 외 정보와 상태 배지가 바뀌지 않았는지 확인한다.
- 우선순위: 중간

### 2. 추가 BOM 품목과 최종 확인 목록을 표준 BOM 순서로 정렬 — 완료
- 화면/영역: PC 출하 > 출하 요청 작성 > 2단계 `BOM 구성 조정`의 PA·PF 구성품 목록, 5단계 `BOM·동반 출하품`의 PA·PF 목록
- 증상: 2단계에서 새로 추가한 품목은 현재 목록의 마지막에 붙는다. 5단계도 드래프트 배열의 현재 순서를 그대로 사용해, 기준 BOM과 추가·제외 항목이 표준 BOM 순서와 다르게 보일 수 있다.
- 사용자 불편: BOM 편집 직후 추가한 품목을 찾기 어렵고, 2단계에서 확인한 구성 순서와 최종 확인의 순서가 달라질 수 있다.
- 확정 정책: PA 구성품에는 AF가, PF 구성품에는 PA가 포함되는 기존 계층을 유지한다. 각 구성품 목록은 기존 BOM 표준 정렬을 따른다. 새로 추가한 항목도 같은 정렬 기준으로 재배치되어 해당 코드 순서상 첫 항목이면 목록 맨 위에 표시된다.
- 구현 방향: 프런트 드래프트에 새 행을 단순 append하지 않고, 백엔드 `bom_child_item_ordering()`의 표준 순서(부서 → 공정 단계 F·A·R → 일련번호 → MES 코드)를 프런트 표시·저장 순서의 단일 기준으로 사용한다. 2단계의 PA·PF 목록과 5단계 최종 확인 목록은 같은 정렬 함수를 거친 데이터를 렌더한다. 제외된 항목도 원래 BOM 상대 순서를 보존한다.
- 수용 기준:
  - AF 항목을 PA 구성품에 추가하면 표준 BOM 순서상 위치(예: AF가 선행이면 맨 위)에 즉시 표시된다.
  - PF 구성품에 PA 항목을 추가해도 같은 기준으로 재정렬된다.
  - 2단계의 포함·제외·추가 행 순서와 5단계 최종 확인 목록의 순서가 일치한다.
  - 동일 정렬 키일 때는 품목코드와 품목 ID까지 사용해 새로고침·재진입 후에도 순서가 흔들리지 않는다.
  - 수량, 추가됨·기본·제외됨 상태, 수정 기능은 기존과 동일하게 유지된다.
- 브라우저 확인 시나리오:
  1. 2단계 PA 구성품에 AF 코드 품목을 추가하고 즉시 목록 맨 위 또는 표준 순서 위치로 이동하는지 확인한다.
  2. PA·PF 양쪽에 여러 코드 종류를 추가·제외한 뒤 목록 순서가 표준 BOM 순서인지 확인한다.
  3. 5단계로 이동해 같은 구성품이 2단계와 동일 순서로 표시되는지 확인한다.
  4. 요청을 저장 후 다시 열어도 순서가 유지되는지 확인한다.
- 우선순위: 중간

### 3. 최종 확인의 새 PF 배치과 구성품 제목을 실제 품목명으로 표시 — 완료
- 화면/영역: PC 출하 > 출하 요청 작성 > 5단계 `BOM·동반 출하품`의 PA·PF 두 목록과 자동 생성 예정 품목 행
- 증상: 새 대상 PF가 자동 생성될 상황에서 `품목코드는 저장/준비 완료 시 자동 생성 예정` 행이 PA 구성품의 맨 위에 표시될 수 있다. 또한 두 목록 제목은 고정된 `PA 구성품`·`PF 구성품`이라 현재 어떤 PA/PF 품목의 구성인지 제목만으로 식별하기 어렵다.
- 사용자 불편: 최종 확인에서 새로 생성될 출하 PF가 어느 구성 목록에 속하는지 오해할 수 있고, 실제 품목명을 확인하려면 목록 행을 다시 찾아야 한다.
- 확정 정책: PA 구성품에는 AF, PF 구성품에는 PA가 포함되는 기존 BOM 계층은 변경하지 않는다. 자동 생성될 최종 PF는 PF 구성품 목록의 첫 행에서 표시한다. 두 구성품 목록의 고정 제목 대신 각 목록이 나타내는 실제 품목명을 표시한다.
- 확정 정책: AF 목록의 제목에는 PA 품목명을, PA 목록의 제목에는 PF 품목명을 표시한다. 새 PA와 새 PF가 동시에 생성되는 경우에도 이 부모-자식 관계와 제목 규칙을 유지한다.
- 구현 방향: 최종 확인의 새 항목을 `new PA`와 `new PF`로 구분해 렌더하며, 새 PF는 PF 목록 맨 위에 둔다. AF 목록은 PA 품목명, PA 목록은 PF 품목명과 생성 예정 상태를 제목으로 표시하고, 기존 정적 명칭은 보조 정보로만 남기거나 제거한다. 현재 `FinalRequirementReview`는 `newPaName` 행만 PA 그룹으로 보내므로, 생성 유형과 그룹을 분리한다.
- 수용 기준:
  - 새 PF가 필요한 요청의 자동 생성 예정 행은 PF 구성품 목록 맨 위에 표시된다.
  - 새 PF 행이 PA 구성품 목록에 잘못 나타나지 않는다.
  - AF 목록 제목에는 PA 품목명, PA 목록 제목에는 PF 품목명이 표시되어 어떤 상위 품목을 기준으로 한 구성인지 알 수 있다.
  - PA 구성품의 AF, PF 구성품의 PA 포함 구조와 기존 BOM 행·수량은 바뀌지 않는다.
  - 새 PA와 새 PF가 동시에 필요한 요청에서도 두 생성 예정 항목의 이름·그룹·순서가 구분된다.
- 브라우저 확인 시나리오:
  1. 기준 PF를 다른 출하 대상용으로 변경해 새 PA·PF가 필요한 요청을 만든다.
  2. 5단계에서 새 PF 이름과 자동 생성 안내가 PF 구성품의 첫 행인지 확인한다.
  3. AF 목록에는 PA 품목명, PA 목록에는 PF 품목명이 제목으로 표시되는지 확인한다.
  4. PA 구성품의 AF와 PF 구성품의 PA가 계속 포함되고, 각 목록의 순서가 2단계와 일치하는지 확인한다.
- 우선순위: 높음

### 4. 최종 확인의 BOM·변경 구성품 카드 높이와 스크롤 복구 — 완료
- 화면/영역: PC 출하 > 출하 요청 작성 > 5단계 `저장 및 전환`의 `BOM·동반 출하품`, `변경된 구성품`, 해당 본문 스크롤 영역
- 최종 결정: 현재 BOM 카드의 424px 실측 높이를 기준으로 `BOM·동반 출하품`은 432px 고정 높이를 사용한다. `변경된 구성품`은 후속 결정에 따라 자연 높이를 사용한다.
- 사용자 재확정: 변경된 구성품이 하나라도 있으면 `BOM·동반 출하품` 박스 높이는 내용량과 무관하게 항상 432px 고정으로 유지한다.
- 구현 방향: `shipping-final-summary`는 출하 품목 헤더, 432px 고정 BOM 카드, 자연 높이 변경 카드의 행 구조를 사용한다. BOM 목록과 외곽 본문은 각각 세로 스크롤하고, 하단 action bar는 고정한다.
- 수용 기준:
  - BOM 변경이 있을 때 `BOM·동반 출하품` 카드의 외곽 높이가 432px로 유지된다.
  - BOM 카드의 목록은 내부 스크롤되고, 변경 카드는 행 수에 맞는 자연 높이를 사용한다.
  - 출하 품목 헤더와 두 카드를 합친 높이가 본문보다 크면 5단계 본문에서 세로 스크롤할 수 있다.
  - 본문 스크롤 중에도 하단 `이전`·`출하 요청` action bar는 화면 하단 위치와 동작을 유지한다.
  - BOM 변경이 없는 경우에는 기존처럼 `BOM·동반 출하품`이 남는 높이를 사용하고, 빈 변경 구성품 카드를 만들지 않는다.
- 브라우저 확인 시나리오:
  1. 변경 구성품이 3개 이상인 요청을 5단계까지 진행한다.
  2. `BOM·동반 출하품` 카드가 432px인지, `변경된 구성품` 카드가 행 수에 맞는 자연 높이인지 확인한다.
  3. BOM 카드에 충분한 행을 만들어 내부 목록이 스크롤되고, 변경 카드가 바깥 본문 스크롤을 사용하는지 확인한다.
  4. 5단계 본문을 스크롤해 두 카드 전체를 확인하고, 하단 action bar가 고정되는지 확인한다.
  5. BOM 변경 없는 요청에서도 본문 높이·스크롤과 출하 요청 동작이 정상인지 확인한다.
- 우선순위: 높음

### 5. BOM 구성 조정의 두 구성품 제목에 기준 PF 품명 표시 — 완료
- 화면/영역: PC 출하 > 출하 요청 작성 > 2단계 `BOM 구성 조정`의 `PA 구성품 N개 포함`, `PF 구성품 N개 포함` 제목
- 수집 상태: 구현 완료
- 증상: 두 제목이 고정된 `PA 구성품`·`PF 구성품`과 개수만 표시한다. 현재 어느 기준 PF를 대상으로 BOM을 조정하는지 제목만으로 알 수 없다.
- 사용자 결정: 두 구성품 제목 모두 1단계에서 선택한 기준 PF의 실제 품목명을 표시한다. PA/PF 구성품 구분과 포함 개수 표시는 기존처럼 유지한다.
- 기존 항목과의 관계: 완료된 3번은 5단계 최종 확인 목록의 실제 품목명 제목 표시를 다룬다. 이 항목은 2단계 BOM 구성 조정 제목에만 적용한다.
- 구현 방향: 2단계의 두 `BomEditor`에 현재 기준 PF 품목명을 전달해, 제목에 품목명과 기존 구성품 구분·개수를 함께 표시한다. 기준 PF를 선택하지 않은 빈 상태와 PA/PF BOM 행의 계층·정렬·편집 동작은 변경하지 않는다.
- 수용 기준:
  - 1단계에서 기준 PF를 선택하면 PA 구성품과 PF 구성품 두 제목 모두 그 PF의 실제 품목명을 표시한다.
  - 두 제목에서 PA/PF 구성품 구분과 포함 개수는 계속 식별할 수 있다.
  - 기준 PF를 다른 품목으로 바꾸면 두 제목의 품명도 새 기준 PF로 즉시 갱신된다.
  - 기준 PF 미선택 상태의 안내와 BOM 구성품 행, 검색·추가·제외 동작은 기존과 동일하다.
- 브라우저 확인 시나리오:
  1. 1단계에서 기준 PF를 선택하고 2단계로 이동한다.
  2. PA 구성품과 PF 구성품 두 제목에 같은 기준 PF 품명이 표시되는지, PA/PF 구분 및 포함 개수가 유지되는지 확인한다.
  3. 이전 단계에서 다른 기준 PF로 변경한 뒤 재진입해 두 제목이 함께 갱신되는지 확인한다.
- 우선순위: 중간

### 6. 자동 생성 예정 품목의 예상 MES 코드 표시 — 완료
- 화면/영역: PC 출하 > 출하 요청 작성 > 3단계 `BOM 매칭`의 `새 PA 생성 예정`·`새 PF 생성 예정` 카드, 5단계 `저장 및 전환`의 자동 생성 예정 품목 행 및 출하 품목 헤더 안내
- 수집 상태: 구현 완료
- 증상: 새 PF/PA가 필요한 경우 `품목코드는 저장/준비 완료 시 자동 생성 예정`이라는 안내만 표시되어, 어떤 공정 코드와 일련번호의 품목이 생성될지 최종 확인 단계에서 판단할 수 없다.
- 사용자 요청: 3단계와 5단계의 자동 생성 예정 안내 모두에 예상 MES 품목코드를 `NN-공정코드-NNNN 예정` 형식으로 표시한다.
- 확정 정책: 저장 전 예상 코드는 기준 PF의 모델 기호와 PA/PF별 다음 일련번호로 산출하는 비예약 참고값이다. 다른 품목이 먼저 생성되면 실제 저장 코드가 달라질 수 있으므로 `예상 코드 · 저장 시 변경 가능`을 함께 표시한다.
- 기존 항목과의 관계: 완료된 1번은 기존 MES 코드의 공정 구간 색상 표시를 다룬다. 이 항목은 아직 생성되지 않은 품목의 예상 코드 표시를 추가한다. 완료된 3번의 자동 생성 예정 행 위치·제목 정책은 유지한다.
- 구현 방향: 자동 생성 대상의 공정 코드와 코드 생성 규칙을 이용해 예상 MES 코드를 산출하고, 3단계 새 PA/PF 생성 예고 카드 및 5단계 기존 자동 생성 안내와 함께 렌더한다. 실제 저장 과정에서 코드가 달라질 수 있다면 그 사실을 화면 문구로 명확히 알린다.
- 수용 기준:
  - 3단계 새 PA/PF 생성 예고 카드와 5단계 자동 생성 예정 PA/PF 행에서 예상 MES 코드와 `예정` 상태를 확인할 수 있다.
  - 예상 코드의 공정 구간은 생성 대상(PA 또는 PF)과 일치한다.
  - 새 PA와 새 PF가 동시에 필요한 경우 각 행의 예상 코드가 구분되어 표시된다.
  - 코드가 아직 확정되지 않은 사실과 저장 시 변경 가능 여부는 확정 정책에 따라 오해 없이 안내된다.
  - 기존 생성 예정 행의 그룹·정렬·품목명 표시와 저장 후 실제 품목코드 생성 동작은 변경하지 않는다.
- 브라우저 확인 시나리오:
  1. 새 PA 또는 PF가 필요한 출하 요청을 만들고 3단계와 5단계로 각각 이동한다.
  2. 3단계 새 PA/PF 생성 예고 카드와 5단계 자동 생성 예정 행에 대상 공정과 일치하는 예상 코드가 `예정` 상태로 표시되는지 확인한다.
  3. 새 PA와 PF가 모두 필요한 요청에서 두 예상 코드가 각각의 카드·행·그룹에 표시되는지 확인한다.
  4. 저장 후 실제 생성된 코드가 화면 정책(예약 코드 또는 참고값)에 맞는지 확인한다.
- 우선순위: 중간

### 7. 변경된 구성품 카드 높이를 행 수에 맞게 조정 — 완료
- 화면/영역: PC 출하 > 출하 요청 작성 > 5단계 `저장 및 전환`의 `변경된 구성품` 카드
- 수집 상태: 구현 완료
- 증상: 변경 구성품이 4개여서 두 줄만 표시되어도 카드가 고정 높이를 차지해, 마지막 행 아래에 불필요한 빈 공간이 생긴다.
- 사용자 결정: `변경된 구성품` 카드는 고정 높이를 사용하지 않고, 표시할 변경 행 수에 맞게 유동 높이로 표시한다.
- 기존 항목과의 관계: 완료된 4번의 `BOM·동반 출하품` 432px 고정 높이 정책은 유지한다. 이 항목은 같은 화면의 `변경된 구성품` 카드에만 적용한다.
- 구현 방향: 변경 구성품 카드의 432px 고정 높이 및 내부 고정 스크롤을 제거하고, 카드 본문이 변경 행의 실제 줄 수만큼만 높이를 갖게 한다. 5단계 바깥 본문 스크롤과 하단 action bar 고정 동작은 유지한다.
- 수용 기준:
  - 변경 구성품이 4개로 두 줄이면 카드 높이가 두 줄과 제목·여백에 필요한 만큼만 표시되고, 하단에 불필요한 빈 공간이 남지 않는다.
  - 변경 구성품 행 수가 늘거나 줄면 카드 높이도 그에 맞게 함께 변한다.
  - `BOM·동반 출하품` 카드는 변경 구성품 유무와 관계없이 기존 432px 고정 높이를 유지한다.
  - 변경 구성품이 많아져 5단계 본문보다 길어지면 바깥 본문에서 스크롤할 수 있으며, 하단 action bar는 고정된다.
  - 변경 구성품이 없는 경우 기존처럼 해당 카드는 표시하지 않는다.
- 브라우저 확인 시나리오:
  1. 변경 구성품이 4개인 요청을 5단계까지 진행한다.
  2. 변경 구성품 카드가 두 행 높이만 사용하고 마지막 행 아래에 큰 빈 공간이 없는지 확인한다.
  3. 변경 행 수를 추가·제외해 카드 높이가 함께 변하는지 확인한다.
  4. 같은 화면에서 `BOM·동반 출하품` 카드가 계속 432px 고정인지, 하단 action bar가 고정인지 확인한다.
- 우선순위: 중간

### 8. BOM 매칭의 변경 구성 요약 카드 제거 — 완료
- 화면/영역: PC 출하 > 출하 요청 작성 > 3단계 `BOM 매칭` 상단의 `N개 변경 구성`·`구성 변경 내용을 확인했습니다.` 카드
- 수집 상태: 구현 완료
- 증상: 상단 요약 카드는 변경 개수와 일반 안내만 보여주며, 바로 아래에 변경 품목의 실제 내용이 표시되는 `변경된 구성품` 목록과 정보가 중복된다.
- 사용자 결정: 3단계 상단의 변경 구성 요약 카드를 제거한다. 하단 `변경된 구성품` 목록을 변경 내용 확인의 단일 표시 영역으로 사용한다.
- 구현 방향: 상단 그리드에서 변경 구성 요약 카드와 관련 문구를 제거하고, 새 PA/PF 생성 예고 카드 및 출하 수량 영역의 기존 동작은 유지한다. 변경 품목 데이터와 하단 목록 렌더링은 변경하지 않는다.
- 수용 기준:
  - 3단계에 `N개 변경 구성`·`구성 변경 내용을 확인했습니다.` 요약 카드가 표시되지 않는다.
  - 변경된 구성품이 있으면 하단 `변경된 구성품` 목록에서 품목·코드·수량·제외 상태를 계속 확인할 수 있다.
  - 새 PA/PF 생성 예고 카드, 출하 수량 변경, 다음 단계 진행 동작은 기존과 동일하다.
  - 변경 구성품이 없는 요청에서도 3단계 레이아웃과 다음 단계 진행이 정상이다.
- 브라우저 확인 시나리오:
  1. 변경 구성품이 있는 요청으로 3단계에 진입한다.
  2. 상단 변경 구성 요약 카드가 없고, 하단 목록에 변경 상세가 계속 표시되는지 확인한다.
  3. 새 PA/PF 생성 예고와 출하 수량 변경 후 다음 단계 이동이 정상인지 확인한다.
- 우선순위: 중간
