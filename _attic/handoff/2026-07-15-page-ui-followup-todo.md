### 2. 출하 요청 작성의 중첩 외곽 프레임 제거 및 작업 표면 통합
- 화면/영역: PC 출하 > 출하 요청 작성 > 1~5단계 본문, 특히 1단계 PF 선택과 2~5단계 단계별 본문 프레임
- 증상: 출하 루트는 이미 둥근 외곽 카드인데, 요청 작성은 `Panel`과 2~5단계 `shipping-wizard-content-frame`의 테두리가 다시 겹친다. 1단계와 2~5단계의 프레임 처리도 서로 달라 화면이 한 겹 더 감싸진 것처럼 보인다.
- 사용자 불편: 목록·입력 영역에 쓸 수 있는 폭과 높이가 줄고, 다른 탭보다 작업 화면이 답답하게 보인다.
- 확정 정책: 출하 요청 작성 본문은 바깥 출하 루트 카드만 외곽 표면으로 사용한다. 단계별 본문은 그 크기를 그대로 채우며, 추가 외곽 테두리·별도 배경 카드로 감싸지 않는다.
- 구현 방향: 요청 작성 `Panel`과 단계별 content frame의 역할을 정리해 한 겹만 남긴다. 내부 입력·목록·요약 카드의 필요한 테두리는 유지한다. 배경은 순백색 값을 추가하지 않고 바깥 루트와 같은 `LEGACY_COLORS.s1` 표면 토큰을 사용한다.
- 허용 기준:
  - 1~5단계 모두 바깥 `rounded-[28px]` 출하 루트 카드 안에서 동일한 작업 표면으로 보인다.
  - 단계 탭, 입력 필드, PF 목록, 하단 이전/다음 또는 출하 요청 버튼의 기존 동작과 스크롤 범위가 유지된다.
  - 단계 본문 주위에 내용과 무관한 두 번째 큰 테두리 상자가 보이지 않는다.
  - 라이트·다크 테마에서 토큰 기반 대비가 유지된다.
- 브라우저 확인 시나리오:
  1. 출하 > 출하 요청 작성으로 진입해 1단계 PF 목록의 전체 폭·높이와 테두리 수를 확인한다.
  2. 2~5단계로 각각 이동해 동일한 한 겹의 작업 표면을 사용하는지 확인한다.
  3. 긴 PF 목록 및 많은 BOM 항목에서 목록 스크롤과 하단 동작 버튼이 겹치지 않는지 확인한다.

### 3. 최종 확인 BOM을 PA·PF 품목명 그룹으로 재구성
- 화면/영역: PC 출하 > 출하 요청 작성 > 5단계 `저장 및 전환` > `BOM·동반 출하품`
- 증상: `34-PR-0050`처럼 PR 코드인 행에도 PA 배지가 붙어, 품목 코드의 분류와 PA/PF 배지의 BOM 단계가 충돌하는 것처럼 보인다. 모든 행에 반복된 PA/PF 배지는 사용자가 보고 싶은 품목 구조보다 내부 단계 정보를 먼저 강조한다.
- 사용자 불편: 코드가 가리키는 품목과 PA/PF 배지의 의미를 별도로 해석해야 하며, 최종 출하 구성에서 PA 품목과 PF 품목의 관계를 한눈에 파악하기 어렵다.
- 확정 정책: PA 품목명 그룹 박스와 PF 품목명 그룹 박스를 분리하고, 각 그룹 안에서 해당 품목명·품목 코드·총 수량을 보여준다. 행 단위 PA/PF 배지는 제거한다. PR 등 실제 품목 코드는 그대로 표시한다.
- 정책 확인 필요: 동반 출하품을 PA/PF 중 어느 그룹에 포함할지, 또는 별도 `동반 출하품` 그룹으로 둘지 결정한다.
- 구현 방향: `FinalRequirementReview`의 단일 `rows` 렌더링을 PA/PF 그룹 단위로 구성한다. 코드와 수량의 표기 순서는 유지하되, 그룹 제목과 행의 정보 밀도를 기존 최종 확인 카드 문법에 맞춘다.
- 허용 기준:
  - PA와 PF가 각각 별도 품목명 그룹으로 구분되어 보인다.
  - `34-PR-0050` 등 실제 품목 코드는 변형되지 않으며 PA/PF 배지와 나란히 반복 표시되지 않는다.
  - 각 품목에서 품목명, 품목 코드, 총 수량을 확인할 수 있다.
  - 동반 출하품은 확정된 배치 정책에 맞고, BOM이 없거나 한 그룹이 비어도 빈 영역이 어색하게 남지 않는다.
- 브라우저 확인 시나리오:
  1. PA·PF 품목과 PR 코드가 함께 있는 출하 요청을 5단계까지 진행한다.
  2. PA/PF 그룹, 각 품목명·코드·총 수량, PR 코드 표기가 명확한지 확인한다.
  3. 동반 출하품이 있는 요청과 없는 요청을 각각 확인해 확정된 배치와 빈 상태를 검증한다.
  4. 항목이 많은 요청에서도 그룹 내부 스크롤과 기존 최종 확인 영역 높이 정책이 유지되는지 확인한다.
- 우선순위: 높음

### 4. 준비 체크 구성품 목록의 내부 스크롤화 및 완료 동작 고정
- 화면/영역: PC 출하 > 준비 체크 > PA 구성품, PF 구성품, 카톤·동반 출하품 목록과 하단 `준비 완료`/`픽업 완료` 동작 영역
- 증상: 준비 체크의 세 목록 패널이 항목 수만큼 계속 높아져 화면 전체가 스크롤된다. PA·PF·카톤/동반 출하품 박스가 제각각 길어지고, 아래 `준비 완료` 버튼을 보려면 목록을 끝까지 내려야 한다.
- 사용자 불편: 준비 완료 전에 필요한 구성품을 점검하면서 마지막 동작 버튼을 즉시 볼 수 없고, PA·PF 목록의 정보 밀도와 화면 균형이 무너진다.
- 확정 정책: 준비 체크는 바깥 작업 표면 높이에 맞춘다. PA·PF·카톤/동반 출하품의 세 목록 패널은 같은 작업 행 안에서 자연스럽게 제한된 높이를 사용하며, 넘치는 품목만 각 패널 내부에서 세로 스크롤한다. 하단 완료 동작 영역은 목록 아래에 고정해 스크롤하지 않아도 항상 보이게 한다.
- 구현 방향: `PrepSection`을 헤더·요약 지표·목록 행·동작 영역의 세로 flex 구조로 유지하고, 목록 행만 `min-h-0 flex-1`로 남긴다. `PrepRequirementList`와 `CompanionPrepList`는 패널과 목록에 높이 전파를 보장하고, 기존 `SHIPPING_SCROLL_LIST_CLASS`의 `overflow-y-auto`가 해당 목록 안에서만 작동하도록 한다. 바깥 `shipping-root-panel`에 새 스크롤을 만들지 않는다.
- 허용 기준:
  - PA·PF·카톤/동반 출하품 세 박스가 같은 행의 제한된 높이 안에 표시된다.
  - PA나 PF 항목이 많아도 각 박스의 목록만 세로 스크롤되며, 다른 두 박스와 하단 버튼 위치는 유지된다.
  - `준비 완료` 상태에서는 준비 완료 버튼, `준비 완료` 후에는 취소·픽업 완료 버튼이 화면 스크롤 없이 보인다.
  - 항목이 적을 때는 불필요하게 큰 빈 스크롤 영역이 생기지 않고, 빈 카톤/동반 출하품도 기존 빈 상태 문구를 유지한다.
  - 부족 재고 배지와 품목 코드·1대 기준·총 필요 수량 정보가 목록 스크롤 안에서도 누락되지 않는다.
- 브라우저 확인 시나리오:
  1. PA 구성품이 많은 준비 중 요청을 연다.
  2. PA 목록만 스크롤하면서 PF·카톤/동반 출하품 박스와 하단 준비 완료 버튼이 고정되어 보이는지 확인한다.
  3. PF 구성품이 많은 요청과 동반 출하품이 있는 요청에서도 각 목록이 독립적으로 스크롤되는지 확인한다.
  4. 준비 완료 후 같은 화면에서 `준비 완료 취소`와 `픽업 완료` 버튼이 스크롤 없이 보이는지 확인한다.
  5. 1920px 데스크톱 폭과 더 낮은 높이에서 항목·버튼·스크롤바가 겹치지 않는지 확인한다.
- 우선순위: 높음

### 5. 요청 상세 구성품 행의 코드 분류 강조와 수량 위치 분리
- 화면/영역: PC 출하 > 출하 요청 상세 > PA 구성품·PF 구성품의 각 품목 행
- 증상: 각 행의 보조 정보가 `34-PR-0051 · 4 EA`처럼 품목 코드와 수량을 한 줄에 이어 표시한다. PR·AF 같은 품목 코드 분류가 묻히고, 수량도 코드 바로 뒤에 붙어 있어 오른쪽의 여백을 활용하지 못한다.
- 사용자 불편: 구성품을 빠르게 훑을 때 코드의 분류(PR·AF 등)를 먼저 식별하기 어렵고, 수량이 품목 코드의 일부처럼 읽힌다.
- 확정 정책: PA·PF 구성품 행 모두에 같은 문법을 사용한다. 품목 코드는 분류 토큰(PR·AF·PA 등)을 시각적으로 강조해 표시하고, 수량과 단위는 카드 오른쪽의 고정 정렬 영역으로 분리한다.
- 구현 방향: `LineSummary`의 단일 `meta` 문자열을 품목 코드와 수량 데이터로 분리해 `SummaryList`에 전달한다. 하이픈으로 구분된 품목 코드의 가운데 분류 토큰은 굵게 강조하고 나머지 코드는 보조 색으로 유지한다. 수량은 오른쪽 정렬된 `tabular-nums` 텍스트 또는 기존 토큰 기반의 작은 컨테이너로 표시한다. 코드 형식을 해석할 수 없는 경우에는 전체 코드를 강조해 안전하게 표시한다.
- 허용 기준:
  - `34-PR-0051`에서는 `PR`, `3-AF-0018`에서는 `AF`가 코드 안에서 즉시 구분된다.
  - PA 구성품과 PF 구성품의 모든 행에서 수량·단위가 품목 코드 오른쪽의 동일한 위치에 정렬된다.
  - 품목명은 첫 줄, 코드와 수량은 둘째 줄의 좌·우 영역으로 분리되어 긴 코드·긴 품목명에서도 겹치거나 잘리지 않는다.
  - 카톤·동반 출하품을 포함해 같은 `SummaryList`를 쓰는 목록은 동일한 문법을 적용하며, 빈 상태와 기존 목록 스크롤은 유지된다.
  - 코드·수량의 표현 변경은 출하 요청 데이터나 수량 계산에 영향을 주지 않는다.
- 브라우저 확인 시나리오:

  4. 긴 품목명·긴 코드·두 자리 이상 수량에서 줄바꿈, 겹침, 가로 스크롤이 없는지 확인한다.
