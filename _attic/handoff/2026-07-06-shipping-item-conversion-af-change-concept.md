# 품목 전환 / 구성 변경 구상 메모

작성일: 2026-07-06

이 문서는 구현 계획이 아니라, 출하 탭의 `구성품 변경` 논의에서 시작해 `품목 전환`이라는 더 넓은 재고 작업으로 확장된 도메인 구상을 정리한 handoff 메모다. 현재 결론은 `품목 전환` 기능을 구현하는 방향이다. 다만 바로 구현으로 들어가기 전에, 여기의 확정 사항과 미확정 질문을 기준으로 범위, BOM 검증 기준, 입출고 이력 정책을 정리해야 한다.

## 1. 이 기능이 필요하다고 느낀 이유

현장에서는 출하 준비나 일반 작업 중 이미 만들어둔 PA, AF, AA 같은 조립품을 그대로 쓰지 못하고 다른 품목으로 바꿔야 하는 상황이 생긴다.

예를 들면 다음과 같다.

- 기존 PA에서 일부 구성품을 빼거나 추가해 다른 PA로 써야 한다.
- AF의 물리 구성은 같지만 펌웨어, 로고, 국가/출고처 세팅 때문에 다른 AF 품목으로 잡아야 한다.
- AF나 AA 단계에서도 일부 하위 구성품을 바꿔 다른 품목으로 인정해야 하는 일이 생길 수 있다.
- 출하 준비 중 final PA/PF가 부족해서 이미 준비된 다른 조립품을 원하는 품목으로 맞춰야 할 수 있다.

현재 기능만으로도 원칙적으로는 처리할 수 있다. 기존 조립품을 분해해서 하위 자재를 정상 재고로 되돌리고, 필요한 자재를 다시 투입해 원하는 품목을 생산 입고하면 된다.

하지만 이 방식은 작업 단계가 많고, 실제 현장에서 전부 해체하지 않은 경우에도 시스템상으로는 모두 풀었다가 다시 만든 것처럼 기록해야 한다. 그래서 실수와 재고 불일치 위험이 생기고, 작업 이력이 실제 현장 감각보다 과하게 복잡해질 수 있다.

이 문제를 줄이기 위해 처음에는 `구성품 변경` 기능을 생각했다. 그러나 논의가 진행되면서 이 기능이 PA에만 국한되지 않고 AF, AA 등 여러 조립 계층에서 발생할 수 있다는 점이 드러났다.

## 2. 지금 헷갈리는 이유

현재 논의가 어려운 이유는 서로 다른 성격의 업무가 하나의 이름 아래 섞여 있기 때문이다.

1. 분해 후 재생산
   - 정상 조립품을 실제로 풀고 다시 만드는 작업이다.
   - 분해, 자재 회수, 추가 투입, 새 품목 생산으로 처리한다.
   - 추적 정확성은 가장 높지만 작업 단계가 많다.

2. 차이만 반영하는 구성 변경
   - 실제로 전부 풀지는 않고, 바뀌는 구성품만 차감하거나 회수한다.
   - 현장 작업에 가깝고 빠르지만, 시스템이 실제 분해 과정을 일부 생략한다.

3. 사양 변경
   - 물리 BOM은 같고 펌웨어, 로고, 국가 세팅, 라벨 같은 품목 정체성만 바뀐다.
   - 재고 흐름은 소스 품목 차감과 대상 품목 입고만으로 충분할 수 있다.

4. 출하 준비 보정
   - 출하하려는 품목이 부족해서 기존 재고를 출하 대상 품목으로 맞추는 상황이다.
   - 업무 목적은 출하지만, 실제 재고 작업은 입출고/생산 쪽 성격이다.

이 네 가지는 현장에서는 모두 "품목을 바꾼다"처럼 보일 수 있다. 하지만 재고 원장 관점에서는 서로 다른 사건들이다. 그래서 구현 방향은 품목 전환으로 잡되, 어떤 케이스를 품목 전환이 처리하고 어떤 케이스를 기존 분해/생산 흐름으로 보낼지 정해야 한다.

## 3. 출하 탭과의 관계

처음에는 출하 준비 중 final PA/PF가 부족할 때 쓰는 보조 기능으로 `구성품 변경`을 생각했다. 그래서 출하 탭 안에 두는 구상이 자연스러워 보였다.

하지만 기능이 PA뿐 아니라 AF, AA 등 여러 조립품 전환으로 넓어지면 출하 탭에 두기에는 어색하다. 이 기능의 본질은 출하가 아니라, 이미 잡힌 재고를 다른 품목 재고로 바꾸는 공식 재고 작업이다.

따라서 현재 방향은 다음과 같다.

- 출하 탭은 출하 요청, 출하 준비, 준비 취소, 픽업 완료, 출하 이력에 집중한다.
- `품목 전환`은 입출고 탭 또는 재고 작업 영역에 두는 것이 더 자연스럽다.
- 출하 준비 화면에서는 필요할 때 `품목 전환으로 이동` 같은 연결만 제공할 수 있다.
- 출하 탭이 `품목 전환` 기능을 소유하지는 않는다.

## 4. 현재 출하 탭 흐름 요약

출하 탭은 단순 출고 화면이 아니라, 출하 요청부터 픽업 완료까지 재고 흐름을 단계별로 관리하는 작업 공간이다.

1. 출하 요청
   - 기준 PF를 선택한다.
   - PA/PF BOM을 확인하거나 변경한다.
   - 동반 출하품을 지정한다.
   - 출하 수량과 메모를 입력한다.
   - 같은 BOM의 기존 PA/PF가 있으면 재사용하고, 없으면 새 PA/PF 품목 정의를 만든다.
   - 이 단계에서는 재고가 움직이지 않는다.

2. 출하 준비 중
   - 요청이 실제 준비 작업 대상으로 넘어온 상태다.
   - 화면에서 최종 PA, 최종 PF, PA 구성품, PF 구성품, 동반 출하품을 확인한다.
   - 준비 완료 전에는 최종 PA 재고가 충분해야 한다.

3. 출하 준비 완료
   - 최종 PA를 출하 수량만큼 차감한다.
   - 최종 PF를 출하 수량만큼 입고한다.
   - 동반 출하품은 실제 차감하지 않고 출하 예약으로 묶어 가용에서 제외한다.

4. 준비 완료 취소
   - 준비 완료 때 생긴 PA 차감/PF 입고 로그를 역반영한다.
   - 동반 출하품 예약을 해제한다.
   - 출하 요청과 final PA/PF 정의는 유지한다.

5. 픽업 완료
   - 최종 PF를 실제 출하 차감한다.
   - 준비 완료 때 예약해둔 동반 출하품도 실제 출하 차감한다.
   - 요청은 출하 이력으로 이동한다.

## 5. 품목 전환이라는 상위 개념

`구성품 변경`이라는 이름은 PA에는 맞지만 AF, AA 같은 다른 조립 계층까지 담기에는 좁다. 사용자가 실제로 하려는 일은 "이미 있는 재고를 다른 품목 재고로 전환하는 것"이다.

따라서 상위 개념은 `품목 전환`이 더 자연스럽다.

품목 전환은 다음처럼 정의할 수 있다.

- 작업자가 소스 품목을 선택한다.
- 작업자가 대상 품목을 선택한다.
- 전환 수량 Q를 입력한다.
- 시스템이 두 품목의 BOM 관계를 비교한다.
- 전환 가능 여부와 필요한 재고 변동을 계산한다.
- 실행하면 소스 품목, 대상 품목, 추가 차감품, 회수 입고품에 대한 입출고 내역이 남는다.

다만 `품목 전환`을 너무 자유롭게 허용하면 재작업, 생산, 수량 조정을 우회하는 위험한 기능이 될 수 있다. 그래서 품목 계층, BOM 비교, 승인/메모 정책이 반드시 필요하다.

## 6. PA 구성품 변경 정의

PA 구성품 변경은 비교적 명확하다.

- 작업자가 소스 PA를 선택한다.
- 작업자가 대상 PA를 선택한다.
- 변경 수량 Q를 입력한다.
- 소스 PA의 직계 BOM과 대상 PA의 직계 BOM을 비교한다.
- 소스 PA 재고는 Q만큼 차감한다.
- 대상 PA 재고는 Q만큼 입고한다.
- 대상 PA에 더 필요한 구성품은 차감한다.
- 소스 PA에는 있었지만 대상 PA에서는 빠지거나 줄어든 구성품은 회수 입고한다.

예시:

```text
소스 PA 1개 = A 1개 + B 1개
대상 PA 1개 = A 1개 + C 2개
수량 3개 변경

소스 PA -3
대상 PA +3
C -6
B +3
A는 차이 없음
```

PA 구성품 변경은 처음 구상에서는 PA 직계 BOM 차이를 기준으로 처리했다. 그러나 품목 전환 개념이 AF, AA까지 넓어진다면 계층별로 직계 BOM만 볼지, 전체 BOM 전개를 볼지 다시 판단해야 한다.

## 7. AF/AA 변경으로 확장될 때의 문제

AF 변경은 처음에는 "펌웨어, 로고, 국가/출고처 세팅값만 바뀌는 작업"으로 논의했다. 이 경우 실물 구성품은 동일하고, 재고 원장은 단순 품목 전환이면 된다.

```text
소스 AF -Q
대상 AF +Q
추가 자재 차감 없음
회수 자재 입고 없음
```

하지만 논의 중에 AF도 PA처럼 실제 구성품을 일부 바꾸고 다른 AF로 재조립하는 경우가 있을 수 있다는 점이 나왔다. 더 나아가 AA에서도 같은 일이 생길 수 있다.

이렇게 되면 `AF 변경`이라는 별도 기능명도 좁다. 특정 계층 전용 기능이 아니라, BOM을 가진 조립품 전반에 적용될 수 있는 `품목 전환` 정책으로 봐야 한다.

가능한 판별 방식은 다음과 같다.

1. BOM 완전 동일
   - 실제 구성품은 같고 품목 정체성만 바뀌는 경우다.
   - 예: 펌웨어, 로고, 국가 세팅, 라벨 차이.
   - 재고 흐름은 소스 품목 차감 + 대상 품목 입고뿐이다.
   - 입출고 내역 보조 의미는 `사양 전환`이 자연스럽다.

2. BOM 차이 있음
   - 실제 구성품이 일부 바뀌는 경우다.
   - 소스 품목 차감, 대상 품목 입고, 추가 구성품 차감, 빠지는 구성품 회수 입고가 필요하다.
   - 입출고 내역 보조 의미는 `구성 전환`이 자연스럽다.

3. 전환 불가
   - 품목 계층이나 종류가 너무 다르다.
   - BOM 비교 기준상 같은 작업으로 볼 수 없다.
   - 실제로는 수량 조정, 재작업, 불량 처리, 생산 작업으로 보내야 하는 경우다.

## 8. BOM 동일성 기준 논점

품목 전환에서 가장 중요한 미확정 질문은 "BOM이 같다는 것을 어디까지 비교할 것인가"다.

후보는 다음과 같다.

1. 직계 BOM만 비교
   - 소스 품목 바로 아래 구성품과 대상 품목 바로 아래 구성품만 비교한다.
   - 구현과 설명은 단순하다.
   - 하지만 AA 아래, AR 아래, TR까지 내려가면 실제 구성 차이가 있을 수 있다.

2. 전체 BOM 트리 비교
   - 대상 품목 아래 전체 전개 결과가 같아야 단순 사양 변경으로 인정한다.
   - 물리 구성 동일성을 가장 안전하게 보장한다.
   - 다만 하위 BOM 품목 구조가 다르지만 최종 실물 구성은 같은 경우를 어떻게 볼지 추가 판단이 필요하다.

3. 최종 하위 자재 기준 비교
   - 중간 AA, AF 품목이 달라도 최종 하위 자재 구성이 같으면 동일하다고 본다.
   - 현장 실물 기준에는 가까울 수 있지만, 중간 품목의 의미가 흐려질 수 있다.

사용자 발언 기준으로는 "AA부터 TR까지"라는 표현이 나왔기 때문에, 단순 같은 모델 제한만으로는 부족하다. 품목 전환에는 최소한 BOM 비교가 필요하다.

## 9. 이 기능을 구현하기로 본 이유

가장 근본적인 질문은 `품목 전환` 기능을 정말 만들어야 하는가였다.

이 기능이 없어도 정석 절차로 처리할 수는 있다.

1. 기존 조립품을 분해한다.
2. 하위 자재를 정상 재고로 되돌린다.
3. 필요한 자재를 추가 투입한다.
4. 원하는 품목을 다시 생산 입고한다.

이 방식은 재고 설명력이 가장 좋다. 실제로 무엇을 풀었고 무엇을 다시 넣었는지 명확히 남는다.

하지만 지금 프로그램 기준으로는 이 분해 후 재생산이 사실상 유일한 처리 방법이고, 이는 현장 흐름과 너무 다르다. 실제로 전부 해체하지 않는 정상 조립품까지 시스템상으로 모두 풀었다가 다시 입고 잡는 것은 작업자 관점에서도 부자연스럽고, 이력도 불필요하게 길어진다.

따라서 `품목 전환`은 현장에서 이미 완성/반완성된 정상 물건을 실제 작업 수준에 맞게 다른 품목으로 인정하기 위한 공식 재고 작업으로 구현하는 방향이 맞다.

구현 판단 기준은 다음과 같다.

- 기능 자체는 구현한다.
- 다만 모든 변경을 무제한 허용하지 않는다.
- 실제 물건을 거의 안 뜯고 라벨/펌웨어만 바꾸는 일은 `사양 전환`으로 처리할 수 있다.
- BOM 차이가 있는 변경은 `구성 전환`으로 처리하되, BOM 비교, 권한, 메모, 이력 설계가 필요하다.
- 실제로 전부 분해하는 작업은 여전히 분해 후 재생산 흐름으로 처리할 수 있다.

현재 결론은 "품목 전환을 구현하지 말지 검토한다"가 아니라, "품목 전환은 필요하므로 구현하되, 기존 분해/생산 흐름을 대체하는 만능 기능이 되지 않게 범위와 검증 기준을 정한다"이다.

## 10. 지금까지 잠정 확정한 내용

- `구성품 변경`이라는 이름은 PA 전용 느낌이 강하다.
- AF, AA까지 고려하면 상위 이름은 `품목 전환`이 더 자연스럽다.
- 품목 전환은 구현하는 방향으로 결론을 잡았다.
- 품목 전환은 출하 탭보다 입출고 탭에 두는 것이 자연스럽다.
- 출하 탭은 품목 전환을 필요로 할 수 있지만, 기능을 소유하는 위치는 아니다.
- PA 구성품 변경은 기존 구상처럼 BOM 차이 기반으로 설명할 수 있다.
- AF 변경은 새 품목 생성 기능이 아니다.
- AF/AA 변경은 특정 계층 전용 기능보다 품목 전환의 한 케이스로 보는 편이 맞다.
- 같은 모델 제한만으로는 부족하다.
- BOM 동일성/차이 처리 정책이 필요하다.
- 이 기능은 재작업/생산/수량 조정을 우회할 수 있으므로 제한과 이력이 중요하다.

## 11. 아직 확정되지 않은 질문

다음 구현 계획을 세우기 전에 아래 질문을 먼저 결정해야 한다.

- 1차 구현 범위는 PA만 둘 것인가, AF/AA까지 포함할 것인가?
- 기능명은 `품목 전환`으로 통일할 것인가?
- 화면은 입출고 탭에 둘 것인가?
- BOM이 완전히 동일한 경우만 단순 전환으로 허용할 것인가?
- BOM이 다른 경우도 구성 차이 차감/회수로 자동 처리할 것인가?
- BOM 비교 범위는 직계 BOM인가, 전체 BOM 트리인가, 최종 하위 자재 기준인가?
- BOM이 다를 때 사용자가 직접 "구성 전환으로 처리"를 승인해야 하는가?
- 메모는 필수로 둘 것인가?
- 승인 권한이 필요한 작업으로 둘 것인가?
- 입출고 내역에는 `품목 전환` 하나로 남길 것인가, `사양 전환` / `구성 전환`을 구분할 것인가?
- 출하 준비 화면에서는 품목 전환으로 이동하는 연결만 둘 것인가?

## 12. 다음 세션 권장 진행

다음 세션에서는 아래 순서로 구현 계획을 구체화하는 편이 좋다.

1. 실제 현장에서 품목 전환이 필요한 대표 케이스를 PA, AF, AA 각각 1~2개씩 정리한다.
2. 각 케이스가 분해 후 재생산, 구성 전환, 사양 전환, 출하 준비 보정 중 어디에 속하는지 나눈다.
3. 품목 전환 기능이 처리할 1차 범위를 정한다.
4. 기존 분해/생산 흐름으로 보내야 할 예외 케이스를 정한다.
5. BOM 비교 기준을 확정한다.
6. 입출고 내역 표시명을 확정한다.
7. 그 다음에 구현 계획을 작성한다.

이 문서의 결론은 "품목 전환을 구현한다"이다. 이유는 현재 프로그램 기준으로는 모든 케이스를 분해 후 재생산으로 처리해야 하는데, 이는 현장 흐름과 너무 다르기 때문이다. 다만 이 기능은 여러 업무가 섞인 강한 재고 작업이므로, 구현 전 범위, BOM 검증 정책, 권한, 이력 표시 기준을 확정해야 한다.
