# DEXCOWIN MES UX TODO 구현 핸드오프

작성일: 2026-07-07  
갱신일: 2026-07-08  
상태: 구현 전 확정 TODO / 다음 세션 구현용 핸드오프  
범위: PC 화면 UX. 모바일 전용 최적화, DB schema, backend API 변경은 제외.

## 목적

이 문서는 사용자가 화면을 보며 짚은 UX 문제를 다음 구현 세션이 바로 처리할 수 있도록 정리한 실행 직전 TODO 문서다. 단순한 문제 목록이 아니라, 각 항목마다 증상, 확정 정책, 구현 방향, 수용 기준, 브라우저 확인 시나리오를 포함한다.

## 전체 현황

- 총 TODO: 29개

- 입출고: 13개
- 출하: 6개

## 확정 결정 요약

- 입출고 첫 화면에서는 작업 유형 선택 전 progress row를 숨긴다.
- 품목 전환은 기존 입출고 5단계 위저드에 얹지 않고 독립 4단계로 진행한다.
- 품목 전환 4단계는 `방식 선택 -> 소스·대상 선택 -> 차이 확인 -> 실행`으로 고정한다.
- 구성 전환 차이 확인은 소스 BOM과 대상 BOM을 좌우로 분리한다.
- BOM/품목 전환 펼침 디자인은 BOM 스타일로 통일한다.
- 입출고내역 묶음 펼침은 한 번에 하나만 열린다.
- 자동 승인 거래는 승인 줄을 숨긴다.
- 메모 카드는 작업자 직접 입력 메모 전용으로 두고, 시스템 기록은 별도 영역으로 분리한다.
- 우측 상세 구성 라인을 클릭하면 본문 목록에서 해당 묶음을 펼치고 클릭 라인을 하이라이트한다.
- 입출고 상단 `요청 작성 / 작성 중 / 내 요청`은 browser history에 쌓는다.
- 4단계 품목 확인에도 저장 버튼을 둔다.
- 최종 확인 BOM row는 카드 전체 클릭으로 펼친다.
- 출하 3단계 BOM 매칭은 `출하 가능 여부`를 최상단 결론으로 보여주고, 그 아래 PA/PF 재사용 여부와 변경 구성품을 보여준다.
- 출하 5단계 최종 확인 첫 줄은 `출하 품목 + 수량`이다.
- 긴 품목명은 2줄 clamp + tooltip으로 처리한다.
- 출하 요청 메모 단계는 남는 높이를 메모 입력 칸이 자연스럽게 채운다.
- 위험 모달은 흰 배경, 붉은 아이콘, 붉은 위험 버튼으로 통일한다.
- 우측 사이드 패널 X 버튼은 입출고내역 탭의 위치와 붉은 색을 기준으로 통일한다.
- 우측 사이드 패널 제목과 X 버튼은 같은 줄에 배치한다.
- 좌측 사이드바는 탭 이동 중에도 마우스가 위에 있으면 확장 상태를 유지한다.

## 공통 TODO

### 1. 공통 / 이탈 확인 모달 / danger 모달 시각 문법 통일

- 증상: 출하 요청 작성 이탈 모달처럼 위험 동작을 묻는 모달에서 아이콘과 위험 버튼이 황토색 계열이고, 카드 배경도 노르스름하게 보인다.
- 사용자 불편: `나가기`, `삭제`, `저장 안 하고 나가기` 같은 위험 동작이 화면마다 다르게 보이면 위험도를 빠르게 판단하기 어렵다.
- 확정 정책: 위험 버튼과 위험 아이콘만 붉은색으로 표시하고, 모달 카드 배경은 흰색으로 통일한다.
- 구현 방향: 공통 confirm/dirty guard 모달의 danger variant를 정리한다. danger action은 붉은 아이콘, 붉은 주요 버튼, 흰 카드 배경을 사용한다. 보조 버튼은 기존 중립 톤을 유지한다.
- 수용 기준: 출하 요청 작성 이탈 모달에서 `나가기` 버튼과 경고 아이콘이 붉은색이고, 모달 본문 배경은 흰색이다.
- 브라우저 확인 시나리오: `출하 -> 출하 요청 작성`에서 작업 중 뒤로가기 또는 탭 이동을 시도해 이탈 모달을 확인한다.
- 우선순위: 높음

### 2. 공통 / 좌측 사이드바 / 탭 이동 중 hover 확장 상태 유지

- 증상: 좌측 사이드바에 마우스를 올려 확장된 상태에서 탭 버튼을 클릭하면, 탭 이동 순간 사이드바가 잠깐 접혔다가 다시 펼쳐진다.
- 사용자 불편: 사용자는 마우스를 계속 사이드바 위에 두고 있는데 UI가 접혔다 다시 펼쳐져 화면이 튄 것처럼 느껴진다.
- 확정 정책: 탭 이동 중에도 마우스가 사이드바 위에 있으면 확장 상태를 유지한다.
- 구현 방향: 탭 클릭 시 hover/collapse 상태를 강제로 초기화하는 흐름을 제거하거나, route/tab 전환 중에도 pointer hover 상태를 보존한다. 탭 전환 후 hover 재감지로 다시 확장되는 구조가 아니라, 처음부터 접히지 않는 구조로 만든다.
- 수용 기준: 사이드바가 확장된 상태에서 다른 탭을 클릭해도 접힘 애니메이션이 끼어들지 않는다.
- 브라우저 확인 시나리오: 사이드바에 마우스를 올린 채 `입출고 -> 입출고내역 -> 출하 -> 불량` 순서로 탭을 클릭한다.
- 우선순위: 중간

## 입출고내역 TODO

### 3. 입출고내역 / 우측 상세 패널 / 상세 제목과 닫기 버튼을 같은 줄에 배치

- 증상: 우측 상세 패널에서 제목과 X 버튼이 서로 다른 줄처럼 보이거나, X 버튼 왼쪽 공간이 비어 보인다. 대시보드 패널은 X 버튼이 모서리에 너무 붙어 있는 느낌도 있다.
- 사용자 불편: 패널 헤더가 느슨해 보이고, 제목과 닫기 동작이 같은 헤더 묶음으로 읽히지 않는다.
- 확정 정책: 모든 우측 사이드 패널은 제목과 X 버튼을 같은 줄에 배치한다. X 버튼 위치와 붉은 색은 입출고내역 탭을 기준으로 통일한다.
- 구현 방향: 공통 패널 헤더가 있다면 그 컴포넌트로 통일하고, 없다면 탭별 패널 헤더 구조를 같은 flex row 문법으로 맞춘다. 제목은 남은 폭을 사용하고 2줄 clamp를 허용한다. X 버튼은 danger 색상, 적당한 안쪽 여백, 동일한 hit area를 가진다.
- 수용 기준: 입출고내역, 대시보드 등 우측 패널에서 제목과 X 버튼이 같은 줄에 있으며, X 버튼은 붉은 톤이다.
- 브라우저 확인 시나리오: 대시보드 품목 상세 패널과 입출고내역 상세 패널을 각각 열어 헤더를 비교한다.
- 우선순위: 높음

### 4. 입출고내역 / 목록 테이블 / 좁아진 화면에서도 상태·처리 컬럼이 잘리지 않게 재배치

- 증상: 우측 상세 패널이 열리거나 사이드바가 확장되면 입출고내역 목록의 마지막 `상태 · 처리` 컬럼이 잘린다.
- 사용자 불편: 좁아지는 것은 자연스럽지만 마지막 판단 컬럼이 잘리면 요청자, 승인자, 메모 같은 처리 정보를 확인하기 어렵다.
- 확정 정책: 폭이 좁아지면 품목명이 먼저 줄바꿈되어야 하며, 상태·처리 컬럼은 잘리지 않아야 한다.
- 구현 방향: PC compact 상태에서 대상/품목명 컬럼을 2줄 wrap 또는 clamp 가능하게 만들고, 품목코드/흐름/수량·재고/상태·처리 컬럼에는 최소 폭을 보장한다. 테이블 전체 overflow가 필요하면 마지막 컬럼이 보존되도록 grid/table width 정책을 조정한다.
- 수용 기준: 우측 상세 패널이 열린 상태에서도 `상태 · 처리` 텍스트와 버튼이 화면 밖으로 잘리지 않는다.
- 브라우저 확인 시나리오: 입출고내역에서 상세 패널을 열고 사이드바를 확장한 뒤 목록 마지막 컬럼을 확인한다.
- 우선순위: 높음

### 5. 입출고내역 / 우측 상세 구성 라인 클릭 / 본문 목록의 해당 묶음 펼침과 하이라이트

- 증상: 우측 상세 패널의 `구성 라인`에서 BOM 하위 품목을 클릭하면 본문 목록에서 해당 입출고 내역 하나만 보여주는 방식으로 반응한다.
- 사용자 불편: 하위 품목이 전체 작업 묶음 안에서 어떤 위치인지 보려는 상황인데, 목록이 필터처럼 바뀌면 상위/하위 맥락을 잃는다.
- 확정 정책: 우측 상세 구성 라인 클릭 시 본문 목록에서 해당 묶음을 펼치고, 클릭한 하위 라인을 하이라이트한다.
- 구현 방향: 상세 패널의 구성 라인 클릭 이벤트를 목록의 `select group + expand group + focus child line` 동작으로 연결한다. 목록 필터는 적용하지 않는다. 필요하면 해당 row로 scrollIntoView를 수행한다.
- 수용 기준: 우측 상세에서 하위 BOM 라인을 클릭하면 본문 목록의 상위 거래가 펼쳐지고, 클릭한 라인이 시각적으로 강조된다.
- 브라우저 확인 시나리오: 생산 BOM 상세 패널에서 하위 구성 라인을 클릭하고 본문 목록 위치와 하이라이트를 확인한다.
- 우선순위: 높음

### 6. 입출고내역 / 묶음 전개 row / 생산 BOM과 품목 전환 펼침 디자인 통일

- 증상: 생산 BOM을 펼쳤을 때와 품목 전환을 펼쳤을 때 하위 row 배경, 들여쓰기, 라벨, 수량 표현이 다르게 보인다.
- 사용자 불편: 둘 다 상위 거래 아래에 하위 영향을 보여주는 구조인데 디자인이 다르면 서로 다른 종류의 화면처럼 느껴져 해석이 늦어진다.
- 확정 정책: BOM 스타일로 통일한다.
- 구현 방향: 생산 BOM의 상위/하위 row 문법을 기준으로 품목 전환 펼침 row를 맞춘다. 품목 전환 고유 라벨은 `소스 차감`, `대상 입고`, `회수 입고`, `추가 차감`처럼 유지하되, 행 높이, 배경, 들여쓰기, 코드 위치, 수량 위치는 BOM과 맞춘다.
- 수용 기준: 생산 BOM과 품목 전환을 각각 펼쳤을 때 하위 row의 구조가 같은 화면 문법으로 보인다.
- 브라우저 확인 시나리오: 입출고내역에서 생산 row와 품목 전환 row를 번갈아 펼쳐 비교한다.
- 우선순위: 높음

### 7. 입출고내역 / 묶음 전개 상태 / 한 번에 하나만 펼침

- 증상: 생산 row는 다른 행을 클릭하면 기존 펼침이 접히지만, 품목 전환 row는 다른 행을 클릭해도 펼침이 남는다.
- 사용자 불편: 같은 목록 안에서 펼침 규칙이 다르면 어떤 row가 현재 선택된 거래인지 흐려진다.
- 확정 정책: 한 번에 하나의 묶음만 열린다.
- 구현 방향: 생산, 출하/출고 묶음, 품목 전환 묶음 모두 같은 expanded group state를 공유하게 한다. 다른 상위 row 선택 시 기존 펼침을 닫고 새 row만 연다. 같은 row 재클릭 시 닫는다.
- 수용 기준: 품목 전환 row를 펼친 뒤 생산 row를 클릭하면 품목 전환 row는 자동으로 접힌다.
- 브라우저 확인 시나리오: 입출고내역에서 품목 전환 row 펼침 -> 생산 row 클릭 -> 기존 품목 전환 row 접힘을 확인한다.
- 우선순위: 중간

### 8. 입출고내역 / 품목 전환 row 상태·처리 / 요청자와 승인자 표시 누락 보정

- 증상: 품목 전환 row의 `상태 · 처리` 영역에 요청자와 승인자 정보가 비어 있거나 다른 거래보다 약하게 보인다.
- 사용자 불편: 품목 전환은 재고를 직접 바꾸는 작업이므로 누가 실행했는지 추적이 중요하다.
- 확정 정책: 요청자는 표시한다. 승인 절차가 없는 자동 처리라면 승인 줄은 숨긴다.
- 구현 방향: 품목 전환 transaction log의 actor/requester 값을 기존 입출고내역 row presentation으로 연결한다. 승인자가 실제로 없고 자동 처리인 경우 승인 라인을 렌더링하지 않는다.
- 수용 기준: 품목 전환 row에는 요청자가 보이고, 승인 절차가 없는 경우 `승인 -` 같은 placeholder가 보이지 않는다.
- 브라우저 확인 시나리오: 입출고내역에서 품목 전환 row를 찾아 상태·처리 컬럼을 확인한다.
- 우선순위: 높음

### 9. 입출고내역 / 자동 승인 거래 표시 / 승인자·메모 placeholder 정리

- 증상: 생산 같은 자동 처리 거래에서 요청자와 승인자가 같은 이름으로 보이거나, 승인/메모 줄에 `-`가 반복 표시된다.
- 사용자 불편: 승인 절차가 없는데 승인자가 있는 것처럼 보이면 실제 승인 흐름을 오해할 수 있다.
- 확정 정책: 자동 승인 거래는 승인 줄을 숨긴다. 메모가 없으면 메모 줄 자체를 숨긴다.
- 구현 방향: 자동 처리/승인 불필요 거래를 판별해 승인자 라인을 렌더링하지 않는다. 메모 chip이나 메모 영역은 실제 메모가 있을 때만 표시한다.
- 수용 기준: 자동 처리 거래에서 `승인 자동`, `승인 -`, 메모 `-`가 기본 노출되지 않는다.

- 확정 정책: 상단 반영 요약 row 전체를 제거한다.
- 구현 방향: 4단계 품목 확인에서 해당 요약 row를 렌더링하지 않는다. 필요한 수량 정보는 각 품목 카드와 하단 CTA에서만 보여준다.
- 수용 기준: 4단계 상단에 반영 라인/총 수량/체크 해제 안내/오른쪽 숫자가 보이지 않는다.
- 브라우저 확인 시나리오: 입출고 4단계 진입 후 품목 카드 위에 요약 row가 없는지 확인한다.
- 우선순위: 높음

### 23. 입출고 / 최종 확인 헤더 / `총 N` 수량 라벨 제거

- 증상: 5단계 헤더에 `입고 · BOM · 반영 2건 · 총 20`처럼 `총 N`이 표시된다.
- 사용자 불편: 총 수량이 상위 입고 수량인지, 하위 차감 합계인지, 반영 라인 총합인지 즉시 알기 어렵다.
- 확정 정책: `총 N` 라벨을 제거한다.
- 구현 방향: 최종 확인 헤더는 작업 유형과 반영 건수까지만 보여준다. 수량은 각 품목 카드의 상위/하위 row에서 확인하게 한다.
- 수용 기준: 5단계 헤더에 `총 N`이 노출되지 않는다.
- 브라우저 확인 시나리오: 입출고 5단계에서 헤더 문구를 확인한다.
- 우선순위: 높음
