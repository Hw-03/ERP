# 품목 전환 실제 품목 예시

작성일: 2026-07-06

이 문서는 `품목 전환` 기능을 실제 DEXCOWIN MES 품목코드와 품명으로 설명하기 위한 예시 문서다. 구현 계획이 아니라, 사용자가 화면과 입출고 내역을 이해할 수 있도록 업무 흐름을 구체화한 기준 메모다.

예시는 현재 `C:\ERP\backend\mes.db`의 실제 품목, BOM, 재고를 읽어 작성했다. 문서의 전환 수량 Q는 설명을 위한 예시 수량이며, 실제 DB에는 어떤 재고 변경도 하지 않았다.

## 1. 품목 전환의 기본 정의

품목 전환은 이미 재고로 잡혀 있는 조립품을 다른 품목 재고로 바꾸는 공식 입출고 작업이다.

기존 프로그램 기준으로는 이런 상황을 처리하려면 보통 다음처럼 해야 한다.

1. 기존 조립품을 분해 출고한다.
2. 하위 자재를 정상 재고로 회수한다.
3. 필요한 구성품을 다시 차감한다.
4. 원하는 새 품목을 다시 생산 입고한다.

하지만 현장에서는 실제로 모든 물건을 완전히 해체하지 않는 경우가 많다. 펌웨어, 로고, 세팅, 라벨, 일부 구성품만 바뀌는 경우도 있고, 이미 만들어진 PA/AF/AA를 출하처나 요청 사양에 맞춰 다른 품목으로 인정해야 하는 경우도 있다.

그래서 이 기능은 `분해 + 생산`을 강제로 나누기보다, 실무자가 실제로 한 일을 `품목 전환`이라는 하나의 작업으로 남긴다.

## 2. 전환 방식

### 2.1 사양 전환

BOM 전체 전개 결과가 동일한 경우다. 즉 실제 구성품은 같고, 품목의 사양명, 출고처, 국가, 펌웨어, 로고, 표시 설정 같은 정체성만 바뀐다고 본다.

재고 흐름은 단순하다.

```text
소스 품목 -Q
대상 품목 +Q
구성품 차감 없음
구성품 회수 없음
```

입출고 내역에서는 다음처럼 읽히면 된다.

```text
품목 전환 / 사양 전환 / 소스 품목 → 대상 품목 / Q EA
```

### 2.2 구성 전환

BOM 전체 전개 결과가 다른 경우다. 소스 품목과 대상 품목의 하위 구성품에 차이가 있으므로, 소스 품목을 줄이고 대상 품목을 늘리는 것만으로는 재고가 맞지 않는다.

재고 흐름은 다음처럼 남는다.

```text
소스 품목 -Q
대상 품목 +Q
대상에 더 필요한 구성품 -N
소스에서 빠지는 구성품 +N
```

입출고 내역에서는 대표 row는 하나로 보이되, 펼치면 구성품 차이가 보이면 된다.

```text
품목 전환 / 구성 전환 / 소스 품목 → 대상 품목 / Q EA
  - 소스 품목 사용
  - 대상 품목 입고
  - 추가 구성품 차감
  - 빠지는 구성품 회수
```

## 3. PA 예시

PA는 가방 포장 완료 상태의 상위 조립품이다. 출하 준비와 가장 직접적으로 연결되는 단계지만, 품목 전환 기능 자체는 출하 탭이 아니라 입출고 탭에 두는 것이 자연스럽다.

### 3.1 PA 사양 전환 예시

두 품목은 DB 기준으로 전체 BOM 전개 결과가 동일하다. 즉 구성품 재고를 따로 차감하거나 회수하지 않고, 소스 PA 재고를 대상 PA 재고로 옮기는 사양 전환으로 볼 수 있다.

| 구분 | 품목코드 | 품명 | 현재 총재고 | 창고 | 부서 |
|---|---|---|---:|---:|---:|
| 소스 PA | `7-PA-0022` | DX-7020_70kV, 2mA_USA_Dexcowin Global_가방 포장 완료 | 400 EA | 200 EA | 200 EA |
| 대상 PA | `7-PA-0023` | DX-7020_70kV, 2mA_USA_Vector_가방 포장 완료 | 400 EA | 200 EA | 200 EA |

예시 작업 수량은 5 EA다.

작업자가 화면에서 다음처럼 선택한다.

```text
전환 방식: 사양 전환
소스 품목: 7-PA-0022 / DX-7020_70kV, 2mA_USA_Dexcowin Global_가방 포장 완료
대상 품목: 7-PA-0023 / DX-7020_70kV, 2mA_USA_Vector_가방 포장 완료
전환 수량: 5 EA
```

시스템은 BOM 전체 전개 결과를 비교한다.

```text
BOM 차이: 없음
처리 방식: 사양 전환 가능
```

실행 후 재고는 다음처럼 변한다.

| 품목 | 변경 전 | 변경 | 변경 후 |
|---|---:|---:|---:|
| `7-PA-0022` DX-7020_70kV, 2mA_USA_Dexcowin Global_가방 포장 완료 | 400 EA | -5 EA | 395 EA |
| `7-PA-0023` DX-7020_70kV, 2mA_USA_Vector_가방 포장 완료 | 400 EA | +5 EA | 405 EA |

입출고 내역 대표 row 예시는 다음과 같다.

```text
품목 전환
대상: DX-7020_70kV, 2mA_USA_Vector_가방 포장 완료
품목코드: 7-PA-0023
흐름: 사양 전환
수량: 5 EA
보조: 7-PA-0022 → 7-PA-0023
```

펼침 row는 단순하다.

```text
소스 PA 사용    7-PA-0022  -5 EA
대상 PA 입고    7-PA-0023  +5 EA
```

### 3.2 PA 구성 전환 예시

두 품목은 같은 DX3000 65kV 1.7mA 계열이지만, DB 기준으로 전체 BOM 전개 결과에 차이가 있다. 따라서 구성 전환으로 처리해야 한다.

| 구분 | 품목코드 | 품명 | 현재 총재고 | 창고 | 부서 |

## 4. AF 예시

AF는 PA보다 한 단계 아래의 완성/반완성 조립품이다. 펌웨어, 국가, 출고처, 로고, 에이밍 방식 같은 차이가 품목명에 반영되는 경우가 많다.

### 4.1 AF 사양 전환 예시

두 품목은 DB 기준으로 전체 BOM 전개 결과가 동일하다. 품명상 출고처 또는 사양명만 다른 것으로 볼 수 있으므로 사양 전환 예시로 적합하다.

| 구분 | 품목코드 | 품명 | 현재 총재고 | 창고 | 부서 |
|---|---|---|---:|---:|---:|
| 소스 AF | `7-AF-0030` | COCOON 70KV 2mA / 18cm Gray [덴탈 / 트리거 에이밍라이트] | 400 EA | 200 EA | 200 EA |
| 대상 AF | `7-AF-0031` | COCOON 70KV 2mA / 18cm Gray [Dentimax / 트리거 에이밍라이트] | 399 EA | 199 EA | 200 EA |

예시 작업 수량은 10 EA다.

```text
전환 방식: 사양 전환
소스 품목: 7-AF-0030 / COCOON 70KV 2mA / 18cm Gray [덴탈 / 트리거 에이밍라이트]
대상 품목: 7-AF-0031 / COCOON 70KV 2mA / 18cm Gray [Dentimax / 트리거 에이밍라이트]
전환 수량: 10 EA
```

BOM 비교 결과는 다음과 같다.

```text
BOM 차이: 없음
처리 방식: 사양 전환 가능
```

실행 후 재고는 다음처럼 변한다.

| 품목 | 변경 전 | 변경 | 변경 후 |
|---|---:|---:|---:|
| `7-AF-0030` COCOON 70KV 2mA / 18cm Gray [덴탈 / 트리거 에이밍라이트] | 400 EA | -10 EA | 390 EA |
| `7-AF-0031` COCOON 70KV 2mA / 18cm Gray [Dentimax / 트리거 에이밍라이트] | 399 EA | +10 EA | 409 EA |

입출고 내역 펼침 row는 다음 정도면 충분하다.

```text
소스 AF 사용    7-AF-0030  -10 EA
대상 AF 입고    7-AF-0031  +10 EA
```

### 4.2 AF 구성 전환 예시

두 품목은 같은 COCOON 70KV 2mA / 18cm Gray 계열이지만, 트리거 에이밍라이트 사양과 스위치 에이밍라이트 사양의 차이가 있다. DB 기준으로 전체 BOM 차이는 `7-AR-0258` 1개다.

| 구분 | 품목코드 | 품명 | 현재 총재고 | 창고 | 부서 |
|---|---|---|---:|---:|---:|
| 소스 AF | `7-AF-0030` | COCOON 70KV 2mA / 18cm Gray [덴탈 / 트리거 에이밍라이트] | 400 EA | 200 EA | 200 EA |
| 대상 AF | `7-AF-0033` | COCOON 70KV 2mA / 18cm Gray [동물 / 스위치 에이밍라이트] | 400 EA | 200 EA | 200 EA |

예시 작업 수량은 8 EA다.

```text
전환 방식: 구성 전환
소스 품목: 7-AF-0030 / COCOON 70KV 2mA / 18cm Gray [덴탈 / 트리거 에이밍라이트]
대상 품목: 7-AF-0033 / COCOON 70KV 2mA / 18cm Gray [동물 / 스위치 에이밍라이트]
전환 수량: 8 EA
```

BOM 차이는 다음과 같다.

| 처리 | 품목코드 | 품명 | 1 EA 차이 | 8 EA 반영 |
|---|---|---|---:|---:|
| 추가 차감 | `7-AR-0258` | COCOON 원형 락커스위치 2핀 (B) (YHM025-02R) 100mm [에이밍하네스 3번] | -1 | -8 |

`7-AR-0258`의 현재 총재고는 400 EA이므로, 예시 수량 8 EA는 재고 부족 없이 전환 가능하다.

실행 후 대표 재고는 다음처럼 변한다.

| 품목 | 변경 전 | 변경 | 변경 후 |
|---|---:|---:|---:|
| `7-AF-0030` COCOON 70KV 2mA / 18cm Gray [덴탈 / 트리거 에이밍라이트] | 400 EA | -8 EA | 392 EA |
| `7-AF-0033` COCOON 70KV 2mA / 18cm Gray [동물 / 스위치 에이밍라이트] | 400 EA | +8 EA | 408 EA |
| `7-AR-0258` COCOON 원형 락커스위치 2핀 (B) (YHM025-02R) 100mm [에이밍하네스 3번] | 400 EA | -8 EA | 392 EA |

입출고 내역 펼침 row는 다음처럼 보이면 된다.

```text
소스 AF 사용       7-AF-0030  -8 EA
대상 AF 입고       7-AF-0033  +8 EA
추가 구성품 차감   7-AR-0258  -8 EA
```

이 예시는 AF도 단순 이름 변경만 있는 것이 아니라, 하위 구성품 차이가 있는 구성 전환이 될 수 있음을 보여준다.

## 5. AA 예시

AA는 AF보다 더 하위의 조립품이다. 핸들, 바디, 콘, 컬리메이터 같은 실제 조립 단위가 많다. 이 단계에서도 품목 전환이 생길 수 있다.

### 5.1 AA 사양 전환 예시

두 품목은 DB 기준으로 전체 BOM 전개 결과가 동일하다. 품명에는 `[리모콘 포트 적용]` 차이가 있지만, 현재 등록된 BOM만 놓고 보면 추가 차감/회수 없이 사양 전환으로 처리할 수 있는 예시다.

| 구분 | 품목코드 | 품명 | 현재 총재고 | 창고 | 부서 |
|---|---|---|---:|---:|---:|
| 소스 AA | `7-AA-0054` | COCOON HANDLE ASS'Y (S) | 400 EA | 200 EA | 200 EA |
| 대상 AA | `7-AA-0055` | COCOON HANDLE ASS'Y (S) [리모콘 포트 적용] | 400 EA | 200 EA | 200 EA |

예시 작업 수량은 12 EA다.

```text
전환 방식: 사양 전환
소스 품목: 7-AA-0054 / COCOON HANDLE ASS'Y (S)
대상 품목: 7-AA-0055 / COCOON HANDLE ASS'Y (S) [리모콘 포트 적용]
전환 수량: 12 EA
```

BOM 비교 결과는 다음과 같다.

```text
BOM 차이: 없음
처리 방식: 사양 전환 가능
```

실행 후 재고는 다음처럼 변한다.

| 품목 | 변경 전 | 변경 | 변경 후 |
|---|---:|---:|---:|
| `7-AA-0054` COCOON HANDLE ASS'Y (S) | 400 EA | -12 EA | 388 EA |
| `7-AA-0055` COCOON HANDLE ASS'Y (S) [리모콘 포트 적용] | 400 EA | +12 EA | 412 EA |

입출고 내역 펼침 row는 다음처럼 충분하다.

```text
소스 AA 사용    7-AA-0054  -12 EA
대상 AA 입고    7-AA-0055  +12 EA
```

### 5.2 AA 구성 전환 예시

두 품목은 같은 COCOON TOP BODY 계열이지만, 로고가 있는 TOP BODY와 로고 없는 TOP BODY의 차이가 있다. 이 경우 DB 기준 BOM 차이가 명확하다.

| 구분 | 품목코드 | 품명 | 현재 총재고 | 창고 | 부서 |
|---|---|---|---:|---:|---:|
| 소스 AA | `7-AA-0048` | COCOON TOP BODY ASS'Y (S) | 400 EA | 200 EA | 200 EA |
| 대상 AA | `7-AA-0049` | COCOON TOB BODY ASS'Y (S) [COCOON 로고 없음] | 400 EA | 200 EA | 200 EA |

예시 작업 수량은 20 EA다.

```text
전환 방식: 구성 전환
소스 품목: 7-AA-0048 / COCOON TOP BODY ASS'Y (S)
대상 품목: 7-AA-0049 / COCOON TOB BODY ASS'Y (S) [COCOON 로고 없음]
전환 수량: 20 EA
```

BOM 차이는 다음과 같다.

| 처리 | 품목코드 | 품명 | 1 EA 차이 | 20 EA 반영 |
|---|---|---|---:|---:|
| 회수 입고 | `7-AR-0189` | COCOON TOP BODY (S) | +1 | +20 |
| 추가 차감 | `7-AR-0190` | COCOON TOP BODY (S) [COCOON 로고 없음] | -1 | -20 |

구성품 현재 재고는 다음과 같다.

| 품목코드 | 품명 | 현재 총재고 |
|---|---|---:|
| `7-AR-0189` | COCOON TOP BODY (S) | 400 EA |
| `7-AR-0190` | COCOON TOP BODY (S) [COCOON 로고 없음] | 400 EA |

실행 후 재고는 다음처럼 변한다.

| 품목 | 변경 전 | 변경 | 변경 후 |
|---|---:|---:|---:|
| `7-AA-0048` COCOON TOP BODY ASS'Y (S) | 400 EA | -20 EA | 380 EA |
| `7-AA-0049` COCOON TOB BODY ASS'Y (S) [COCOON 로고 없음] | 400 EA | +20 EA | 420 EA |
| `7-AR-0189` COCOON TOP BODY (S) | 400 EA | +20 EA | 420 EA |
| `7-AR-0190` COCOON TOP BODY (S) [COCOON 로고 없음] | 400 EA | -20 EA | 380 EA |

입출고 내역 펼침 row는 다음처럼 보이면 된다.

```text
소스 AA 사용       7-AA-0048  -20 EA
대상 AA 입고       7-AA-0049  +20 EA
회수 구성품 입고   7-AR-0189  +20 EA
추가 구성품 차감   7-AR-0190  -20 EA
```

이 예시는 AA 단계에서도 품목 전환이 충분히 독립 기능으로 필요하다는 근거가 된다.

## 6. 화면 흐름 제안

`입출고 > 요청 작성 > 품목 전환`을 누르면 바로 품목을 고르는 화면으로 들어가기보다, 먼저 전환 방식을 고르는 화면이 있는 편이 좋다.

```text
품목 전환
  ├─ 사양 전환
  └─ 구성 전환
```

다만 사용자가 전환 방식을 잘못 고를 수 있으므로, 최종 판정은 BOM 비교 결과가 해야 한다.

### 6.1 사양 전환을 선택한 경우

사용자가 사양 전환을 선택하고 소스/대상 품목을 고른다.

시스템은 전체 BOM을 비교한다.

- BOM이 같으면 실행 가능
- BOM이 다르면 실행 차단

차단 메시지는 이런 식이 좋다.

```text
BOM 차이가 있어 사양 전환으로 처리할 수 없습니다.
구성 전환으로 다시 진행하세요.
```

### 6.2 구성 전환을 선택한 경우

사용자가 구성 전환을 선택하고 소스/대상 품목을 고른다.

시스템은 전체 BOM을 비교한다.

- BOM이 다르면 차이 구성품을 보여주고 실행 가능
- BOM이 같으면 구성품 차이가 없다고 안내

이때는 막기보다 다음처럼 안내하는 것이 좋다.

```text
BOM 차이가 없습니다.
사양 전환으로 처리해도 됩니다.
```

사용자가 그래도 구성 전환으로 진행하려고 하면, 결과적으로 구성품 차감/회수는 없이 소스/대상 품목만 변동된다. 다만 이 경우 입출고 내역상으로는 `사양 전환`이 더 정확하다.

## 8. 정책 메모

현재 예시 기준으로 보면, 품목 전환은 PA 전용 기능으로 제한하면 안 된다. AF와 AA에서도 실제 품목 전환이 충분히 발생할 수 있다.

정책은 다음 방향이 적합하다.

- 기능명은 `품목 전환`으로 통일한다.
- 첫 화면에서 `사양 전환`과 `구성 전환`을 고르게 한다.
- 최종 판정은 사용자의 선택이 아니라 BOM 비교 결과가 한다.
- BOM 비교 기준은 직계 BOM이 아니라 전체 전개 BOM을 기본으로 한다.
- 전체 BOM 차이를 화면에 그대로 다 펼치지 말고, 차이가 나는 품목만 요약해서 보여준다.
- 소스 품목 재고 부족은 실행 차단한다.
- 추가 차감 구성품 재고 부족도 실행 차단한다.
- 회수 입고 품목은 재고 부족 검사를 하지 않는다.
- 메모는 구성 전환에서는 필수에 가깝고, 사양 전환에서는 선택으로 둘 수 있다.

이 정책을 따르면 품목 전환은 생산/분해/재작업을 우회하는 위험한 만능 기능이 아니라, 현장에서 실제로 일어나는 “이미 만들어진 재고의 정체성 또는 구성 변경”을 추적 가능한 입출고 작업으로 남기는 기능이 된다.

## 9. 1차 구현 확정 정책

2026-07-06 논의 기준으로, 1차 구현은 아래 제한을 적용한다.

### 9.1 대상 품목 범위

1차 품목 전환은 `PA`, `AF`, `AA` 품목만 허용한다.

이유는 다음과 같다.

- 품목 전환은 이미 조립되어 재고로 잡힌 품목을 다른 조립품으로 인정하는 작업이다.
- `PA`, `AF`, `AA`는 현장에서 실제 전환 사례가 생기는 조립 단계다.
- `PR`, `AR`, `TR` 같은 하위 자재까지 허용하면 일반 수량조정, 입출고, 생산 작업과 경계가 흐려질 수 있다.

따라서 1차 화면에서는 소스/대상 품목 검색 결과를 `PA`, `AF`, `AA`로 제한한다.

### 9.2 같은 단계끼리만 전환

소스 품목과 대상 품목은 같은 `process_type_code`끼리만 전환할 수 있다.

허용 예시는 다음과 같다.

```text
PA → PA 허용
AF → AF 허용
AA → AA 허용
```

차단 예시는 다음과 같다.

```text
PA → AF 차단
AF → AA 차단
AA → PA 차단
```

이유는 품목 단계가 바뀌면 단순 전환이 아니라 생산, 분해, 조립 단계 변경에 가까워지기 때문이다. 품목 전환은 같은 단계 안에서 품목 정체성 또는 구성 차이를 조정하는 기능으로 제한한다.

### 9.3 BOM 없는 품목은 전환 차단

소스 품목 또는 대상 품목 중 하나라도 BOM이 없으면 품목 전환을 막는다.

이유는 품목 전환의 핵심이 BOM 비교이기 때문이다.

```text
BOM 있음 + BOM 있음 → 비교 후 전환 가능
BOM 없음 + BOM 있음 → 차단
BOM 있음 + BOM 없음 → 차단
BOM 없음 + BOM 없음 → 차단
```

BOM 없는 품목의 재고를 바꾸고 싶다면 품목 전환이 아니라 일반 입출고, 수량 조정, 생산/분해 작업으로 처리해야 한다.

### 9.4 정상 재고만 전환

품목 전환은 정상 재고만 대상으로 한다.

불량 재고는 품목 전환 대상이 아니다. 불량 재고는 불량 처리, 불량 해제, 폐기, 재작업 흐름으로 처리한다.

이유는 다음과 같다.

- 품목 전환은 “정상 조립품을 다른 정상 조립품으로 인정”하는 작업이다.
- 불량품을 다른 품목으로 바꾸면 불량 원인, 폐기, 회수, 재작업 이력이 흐려질 수 있다.
- 불량 재고는 품질/책임 추적이 더 중요하므로 별도 흐름으로 분리해야 한다.

### 9.5 메모 정책

사양 전환은 메모를 선택 입력으로 둔다.

예시:

```text
펌웨어 변경
로고 설정 변경
출고처 표기 변경
국가 세팅 변경
```

구성 전환은 메모 필수를 권장한다.

예시:

```text
Vector 출고용으로 전환
로고 없는 TOP BODY 적용
스위치 에이밍 사양으로 변경
출하 요청 사양에 맞춰 구성 전환
```

1차 구현에서는 최소한 구성 전환 메모를 필수로 두는 방향이 안전하다. 사양 전환은 선택 입력으로 시작하되, 현장 이력 품질이 부족하면 후속으로 필수 전환을 검토한다.

### 9.6 1차 구현 기준 문장

1차 품목 전환은 다음 문장으로 정의한다.

```text
정상 재고에 있는 PA/AF/AA 품목을 같은 단계의 다른 PA/AF/AA 품목으로 전환한다.
소스와 대상은 모두 BOM이 있어야 하며, 전체 BOM 비교 결과에 따라 사양 전환 또는 구성 전환으로 처리한다.
불량 재고와 BOM 없는 품목은 품목 전환 대상에서 제외한다.
```

## 10. 품목 전환 UI 설계 추천

품목 전환 UI는 작업자가 실수 없이 고르고, 시스템이 BOM 차이를 판정해주고, 실행 전 재고 영향이 한눈에 보이는 흐름이어야 한다.

추천 흐름은 5단계다.

```text
1. 전환 방식 선택
2. 소스/대상/수량 선택
3. BOM 비교
4. 실행 확인
5. 완료
```

핵심은 전체 BOM을 전부 보여주는 것이 아니라, 전환으로 인해 달라지는 재고 영향만 보여주는 것이다. 작업자가 알고 싶은 것은 “전체 구조가 어떻게 생겼는가”보다 “이 작업을 실행하면 어떤 품목이 줄고 어떤 품목이 늘어나는가”이기 때문이다.

### 10.1 전환 방식 선택 화면

입출고 요청 작성 화면에서 `품목 전환` 카드를 누르면 바로 품목 검색 화면으로 들어가지 않는다. 먼저 전환 방식을 선택한다.

화면 제목은 `품목 전환 방식 선택`이 적합하다.

카드는 2개로 둔다.

#### 사양 전환 카드

```text
사양 전환
BOM은 같고 펌웨어·로고·출고처·국가 세팅만 다른 품목으로 전환합니다.
예: AF 국내용 → AF 수출용
```

사양 전환은 재고 영향이 단순하다.

```text
소스 품목 -Q
대상 품목 +Q
구성품 차감 없음
구성품 회수 없음
```

디자인 톤은 기존 입출고 화면의 파란/시안 계열을 사용한다. 비교적 가벼운 전환이라는 인상을 주되, 작업 자체는 재고를 바꾸는 일이므로 너무 장식적으로 만들지 않는다.

#### 구성 전환 카드

```text
구성 전환
BOM 차이를 반영해 추가 구성품을 차감하고 빠지는 구성품을 회수합니다.
예: PA 기본형 → PA Vector 사양
```

구성 전환은 재고 영향이 크다.

```text
소스 품목 -Q
대상 품목 +Q
추가 구성품 -N
빠지는 구성품 +N
```

디자인 톤은 기존 입출고 화면과 같은 카드 문법을 사용하되, `BOM 차이 반영` 같은 보조 배지를 붙여 사양 전환보다 더 무거운 작업임을 드러낸다.

두 카드 아래에는 안내문을 둔다.

```text
선택한 방식과 BOM 비교 결과가 다르면 시스템이 실행 전에 안내합니다.
```

사용자가 방식을 고르더라도 최종 판정은 BOM 비교 결과가 해야 한다.

### 10.2 소스/대상/수량 선택 화면

이 화면은 좌우 비교 구조가 가장 직관적이다.

```text
소스 품목                    대상 품목
차감될 품목                  전환 후 생길 품목
[검색/선택]                  [검색/선택]
```

중앙에는 시각적으로 `소스 → 대상` 흐름을 보여주는 화살표를 둔다.

전환 수량 입력은 좌우 품목 선택 영역 아래 또는 상단 중앙에 크게 둔다.

```text
전환 수량: [ 10 ] EA
```

수량은 소스 정상 재고보다 클 수 없다. 초과하면 즉시 차단한다.

```text
소스 정상 재고 8 EA보다 큰 수량은 전환할 수 없습니다.
```

품목 검색 결과는 1차 구현 기준에 맞춰 `PA`, `AF`, `AA`만 보여준다.

소스 품목을 선택하면 대상 품목 후보는 같은 `process_type_code`로 제한한다.

```text
소스 PA 선택 → 대상 PA만 검색
소스 AF 선택 → 대상 AF만 검색
소스 AA 선택 → 대상 AA만 검색
```

품목 검색 row는 다음 정보를 보여준다.

```text
7-AF-0030
COCOON 70KV 2mA / 18cm Gray [덴탈 / 트리거 에이밍라이트]
총 400 EA · 창고 200 EA · 부서 200 EA · BOM 등록
```

BOM 없는 품목은 아예 숨기기보다 비활성 row로 보여주는 편이 친절하다.

```text
BOM 없음 - 품목 전환 불가
```

이렇게 하면 사용자가 “왜 검색했는데 안 되는지”를 이해할 수 있다.

### 10.3 BOM 비교 결과 화면

소스 품목, 대상 품목, 전환 수량을 선택하면 `차이 확인` 버튼을 누른다. 시스템은 전체 BOM을 비교하고 결과를 세 가지 중 하나로 보여준다.

#### 사양 전환 가능

BOM 차이가 없는 경우다.

상단 요약:

```text
BOM 차이 없음
사양 전환으로 처리됩니다.
```

상세는 2줄이면 충분하다.

```text
소스 품목 사용  7-AF-0030  -10 EA
대상 품목 입고  7-AF-0031  +10 EA
```

이 상태에서는 구성품 차이 표를 보여줄 필요가 없다. 오히려 화면을 복잡하게 만든다.

#### 구성 전환 가능

BOM 차이가 있고, 추가 차감 품목의 재고도 충분한 경우다.

상단 요약:

```text
BOM 차이 있음
구성 전환으로 처리됩니다.
추가 차감 1품목 · 회수 입고 1품목
```

차이 표 컬럼은 다음 구성이 좋다.

```text
구분 / 품목코드 / 품명 / 1개 기준 차이 / 전환 수량 반영 / 현재 재고 / 처리 후
```

예시:

```text
추가 차감 | 7-AR-0190 | COCOON TOP BODY (S) [COCOON 로고 없음] | -1 | -20 | 400 | 380
회수 입고 | 7-AR-0189 | COCOON TOP BODY (S) | +1 | +20 | 400 | 420
```

`추가 차감`은 붉은 계열 배지, `회수 입고`는 초록/시안 계열 배지로 구분한다. 다만 표 전체에 강한 배경 줄무늬를 쓰면 난잡해지므로, 작은 배지와 수량 색상 정도로만 구분한다.

#### 전환 불가

전환 불가 사유는 명확히 보여줘야 한다.

대표 사유:

- 소스 정상 재고 부족
- 추가 차감 구성품 재고 부족
- 소스 또는 대상 BOM 없음
- 서로 다른 단계 품목 선택
- 불량 재고만 있음

상단 메시지 예시:

```text
전환할 수 없습니다.
7-AR-0190 재고가 12 EA 부족합니다.
```

전환 불가 상태에서는 실행 버튼을 비활성화한다.

### 10.4 실행 확인 화면

실행 전 확인은 모달보다 오른쪽 확인 패널 또는 하단 고정 확인 영역이 적합하다. 작업자가 BOM 비교표를 보면서 동시에 최종 실행 내용을 확인해야 하기 때문이다.

확인 영역은 네 덩어리로 구성한다.

```text
소스 품목
7-AA-0048 COCOON TOP BODY ASS'Y (S)
-20 EA

대상 품목
7-AA-0049 COCOON TOB BODY ASS'Y (S) [COCOON 로고 없음]
+20 EA

추가 차감
1품목 · 총 20 EA

회수 입고
1품목 · 총 20 EA
```

사양 전환의 경우:

```text
소스 품목 -10 EA
대상 품목 +10 EA
BOM 차이 없음
```

메모 입력 정책:

- 사양 전환: 메모 선택
- 구성 전환: 메모 필수

구성 전환에서 메모가 없으면 실행 버튼을 비활성화한다.

버튼 문구는 다음이 좋다.

```text
품목 전환 실행
```

확인 문구는 짧게 둔다.

```text
위 재고 변동으로 품목 전환을 실행합니다.
```

### 10.5 완료 화면

완료 화면은 화려할 필요가 없다. 작업자가 정상 처리 여부와 다음 이동 경로를 바로 알면 된다.

상단:

```text
품목 전환 완료
7-AA-0048 → 7-AA-0049
20 EA 전환됨
```

구성 전환 완료 요약:

```text
소스 품목 -20 EA
대상 품목 +20 EA
추가 차감 1품목 · 20 EA
회수 입고 1품목 · 20 EA
```

사양 전환 완료 요약:

```text
소스 품목 -10 EA
대상 품목 +10 EA
BOM 차이 없음
```

하단 버튼은 3개가 적합하다.

```text
새 품목 전환
입출고 내역에서 보기
입출고로 돌아가기
```

버튼 의미:

- `새 품목 전환`: 품목 전환 화면을 초기화하고 다시 시작한다.
- `입출고 내역에서 보기`: 방금 생성된 품목 전환 이력으로 이동한다.
- `입출고로 돌아가기`: 입출고 요청 작성 첫 화면으로 돌아간다.

### 10.6 UI 구현 원칙

품목 전환 UI는 기존 입출고/출하 화면과 같은 디자인 언어를 사용한다.

- 큰 카드 기반
- 좌우 비교 구조
- 단계 진행 표시
- 차이만 보여주는 요약표
- `lucide-react` 아이콘
- `LEGACY_COLORS` 또는 `var(--c-*)` 색상 토큰 사용
- 인라인 raw hex 사용 금지
- 버튼은 semantic `<button>` 사용
- 클릭 영역은 최소 44px 이상

특히 BOM 비교 화면은 전체 BOM tree를 보여주는 화면이 아니다. 차이 품목만 보여줘야 한다. 전체 BOM 구조를 모두 보여주면 생산/BOM 관리 화면처럼 복잡해져서, 현장 작업자가 품목 전환을 실행하기 어려워진다.

## 11. 남은 구현 작업과 우선순위

현재 남은 작업은 크게 세 덩어리로 본다.

1. 입출고 요청 작성 UX 개선
2. 품목 전환 v1 구현
3. 기존 임시 구현 정리와 입출고 내역 표시 마무리

### 11.1 입출고 요청 작성 UX 개선

가장 먼저 처리할 작업이다. 품목 전환 화면이 입출고 탭 안에 자연스럽게 들어가려면, 입출고 요청 작성 화면의 기본 문법이 먼저 정리되어야 한다.

현재 목표:

- 작업 유형 선택 후에는 불량/출하 탭처럼 전용 작업 화면에 들어온 느낌을 준다.
- 상단에는 5단계 진행 row만 남긴다.
- `작업 유형 선택`, `세부 작업과 부서 선택`, `창고 반출 품목 선택`, `품목 확인`, `최종 확인`을 한 줄에서 같은 형식으로 보여준다.
- 별도 제목, 숫자 배지, 중복 라벨은 제거한다.
- 완료된 단계는 얇은 summary row로 접어 보여준다.
- 품목 전환 화면에 들어갔다가 뒤로 돌아왔을 때 작업 유형 카드 높이가 작아지는 문제를 해결한다.

이 작업을 먼저 해야 하는 이유는, 품목 전환 v1을 기존 어색한 위저드 구조 위에 얹으면 화면이 다시 꼬일 가능성이 크기 때문이다.

### 11.2 품목 전환 v1

두 번째 큰 작업이다. 품목 전환은 단순히 기존 `구성품 변경`의 이름을 바꾸는 수준이 아니라, 사양 전환과 구성 전환을 구분하는 공식 입출고 작업으로 정리한다.

v1 범위:

- 사양 전환 / 구성 전환 방식 선택
- PA / AF / AA만 허용
- 같은 단계 품목끼리만 허용
- BOM 없는 품목 차단
- 정상 재고만 전환
- 전체 BOM 기준 비교
- 소스 품목 차감
- 대상 품목 입고
- 추가 구성품 차감
- 빠지는 구성품 회수 입고
- 실행 확인 화면
- 완료 화면
- 입출고 내역 이동

테스트 시나리오:

- 사양 전환 성공
- 구성 전환 성공
- 다른 단계 품목 차단
- BOM 없음 차단
- 소스 재고 부족 차단
- 추가 구성품 부족 차단
- 불량 재고 제외

### 11.3 기존 임시 구현 정리

현재 품목 전환은 이전 출하 탭의 `구성품 변경` 흐름에서 출발했기 때문에, 일부 API, 테스트, 문구, 화면 소속이 아직 임시 구현의 흔적을 가질 수 있다.

정리 대상:

- 출하 탭에 남은 구성품 변경 진입 경로 제거 확인
- 입출고 탭의 품목 전환 전용 흐름으로 대체
- 기존 `shipping component-change` 기반 API를 계속 쓸지, 새 `item-conversions` API로 분리할지 결정
- transaction log에 남길 작업명, reference, phase 정책 확정
- 입출고 내역 표시 정책 확정

입출고 내역 표시 방향:

- 대표 row 작업명: `품목 전환`
- 흐름: `사양 전환` 또는 `구성 전환`
- 펼침 row:
  - 소스 사용
  - 대상 입고
  - 추가 차감
  - 회수 입고

### 11.4 추천 진행 순서

구현 순서는 아래가 가장 안전하다.

1. 입출고 요청 작성 UX 개선을 먼저 마무리한다.
2. 품목 전환 v1의 backend/API/검증 정책을 구현한다.
3. 품목 전환 5-step UI를 구현한다.
4. 입출고 내역 표현을 정리한다.
5. 기존 출하 기반 임시 구현과 테스트를 정리한다.

결론적으로, 지금 바로 구현에 들어간다면 1번 `입출고 요청 작성 UX 개선`부터 끝내는 것이 맞다. 이 화면이 품목 전환 v1의 기반이 되기 때문이다.
