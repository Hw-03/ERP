## 입출고

### 1. 품목 전환 방식 선택 단계를 자동 판정으로 대체
- 화면/영역: PC 입출고 > 품목 전환 작업 흐름
- 증상: 현재 품목 전환은 `사양 전환`과 `구성 전환`을 작업자가 먼저 고르는 구조다.
- 사용자 불편: 작업자는 보통 "A 품목 재고를 B 품목으로 바꾼다"는 목적만 가지고 있으며, 전환 유형 판정은 시스템 내부 계산에 가깝다. 사용자가 전환 유형을 먼저 판단해야 하면 불필요한 인지 부담이 생긴다.
- 확정 정책: 전환 유형 선택 단계를 제거한다. 소스 품목, 대상 품목, 수량을 입력하면 시스템이 BOM 영향 여부로 전환 유형을 자동 판정한다. BOM 차감/회수/추가 투입이 있으면 `구성 전환`, BOM 영향이 없으면 `사양 전환`으로 처리한다. 수동 override 예외는 두지 않는다.
- 구현 방향: 기존 소스/대상 품목 선택 방식은 유지하되, 첫 단계의 `사양 전환 / 구성 전환` 카드 선택 UI를 없앤다. 차이 확인 단계 상단에서 "이 전환은 사양 전환으로 처리됩니다" 또는 "이 전환은 구성 전환으로 처리됩니다"처럼 시스템 판정 결과를 표시한다. 구성 전환인 경우에만 재고 영향, 추가 차감, 회수 입고, 사유/메모 필수 여부를 강하게 노출한다.
- 수용 기준: 작업자가 전환 유형을 직접 선택하지 않아도 품목 전환을 시작할 수 있다. 소스/대상/수량 입력 후 차이 확인 화면에서 자동 판정 결과가 명확히 보인다. BOM 차이가 있는 케이스는 구성 전환으로, BOM 차이가 없는 케이스는 사양 전환으로 처리된다.
- 브라우저 확인 시나리오: PC 입출고에서 품목 전환으로 진입한다. 전환 유형 선택 화면 없이 기존 품목 선택 화면으로 이동하는지 확인한다. BOM이 같은 품목 조합과 BOM 차이가 있는 품목 조합을 각각 선택해 차이 확인 화면의 판정 라벨과 재고 영향 표시가 올바른지 확인한다.
- 우선순위: 높음

### 2. 입출고 상단 작업 탭 카드의 상하 여백 균형 맞추기
- 화면/영역: PC 입출고 > 상단 `요청 작성 / 작성 중 / 내 요청` 탭 카드 영역
- 증상: 첨부 캡처 기준으로 탭 카드 위쪽 여백이 아래쪽 여백보다 크게 보여, 상단 영역이 아래로 밀린 듯한 인상을 준다.
- 사용자 불편: 같은 상단 탭 묶음 안에서 수직 여백이 맞지 않아 화면이 덜 정돈되어 보인다. 특히 카드 높이와 텍스트 정렬은 맞아도 바깥 위아래 공간이 달라 시각적 균형이 깨진다.
- 확정 정책: 아래 여백을 기준으로 위 여백을 맞춘다. 즉 아래쪽 공간을 늘리는 방식이 아니라, 위쪽 남는 공간을 줄여 상하 여백이 같은 밀도로 보이게 한다.
- 구현 방향: 탭 카드 자체의 높이/라운드/색상/활성 상태 표현은 유지하고, 탭 카드 묶음을 감싸는 컨테이너의 상단 padding 또는 margin만 조정한다. 주변 본문 영역과의 간격은 현재 아래 여백을 기준값으로 삼는다.
- 수용 기준: 캡처와 같은 PC 입출고 화면에서 탭 카드 묶음의 위쪽 외부 여백이 아래쪽 외부 여백과 시각적으로 동일하게 보인다. 탭 카드 내부 텍스트는 수직 중앙 정렬을 유지한다. `요청 작성`, `작성 중`, `내 요청` 카드 높이와 간격은 기존 디자인을 유지한다.
- 브라우저 확인 시나리오: PC 브라우저에서 `/mes?tab=warehouse`로 진입한다. 상단 탭 카드 묶음의 위쪽 빈 공간과 아래쪽 빈 공간을 비교한다. 활성/비활성 카드 모두 텍스트가 중앙에 있고, 위아래 바깥 여백이 아래 기준으로 정렬되어 보이는지 확인한다.
- 우선순위: 중간

### 3. 품목 전환 첫 화면을 자동 판정형 선택 화면으로 재구성
- 화면/영역: PC 입출고 > 품목 전환 > 현재 `사양 전환 / 구성 전환` 선택 화면
- 증상: 현재 화면은 진행바의 `방식 선택` 단계에서 좌우 대형 카드로 `사양 전환`과 `구성 전환`을 보여준다. 자동 판정 정책이 확정되면 이 두 선택지는 더 이상 사용자가 고를 대상이 아니다.
- 사용자 불편: 화면 대부분이 비어 있는 두 선택 카드로 채워져 있고, 사용자가 실제로 해야 하는 `소스 품목 / 대상 품목 / 수량` 선택으로 바로 이어지지 않는다. 자동 판정 정책과 맞지 않는 UI가 남으면 사용자는 "내가 전환 유형을 골라야 하나?"라고 다시 헷갈릴 수 있다.
- 확정 정책: 이 화면은 `사양 전환`과 `구성 전환`을 선택하는 화면이 아니라, 전환할 품목을 선택하면 시스템이 전환 유형을 자동 판정한다는 흐름을 드러내는 화면으로 재구성한다.
- 구현 방향: 디자인 스킬 기준으로 기존 DEXCOWIN MES의 밝은 산업형 UI, 토큰 기반 색상, 큰 카드, 부드러운 라운드, 파란 액센트, 명확한 진행 단계를 유지한다. 좌우 대형 빈 카드는 `소스 품목`과 `대상 품목` 선택 영역으로 전환하거나, 기존 품목 선택 UI가 더 적합하면 그 화면으로 곧장 진입시킨다. 진행바 문구는 `방식 선택` 대신 `품목 선택` 또는 `전환 대상 선택`처럼 사용자의 실제 행동을 나타내게 바꾼다. `사양/구성` 용어는 선택 카드가 아니라 차이 확인 단계의 자동 판정 라벨로만 사용한다.
- 수용 기준: 품목 전환 진입 직후 사용자는 전환 유형을 고르지 않고 바로 품목 선택 작업을 시작한다. 화면에 `사양 전환`과 `구성 전환`이 동등한 선택지처럼 크게 노출되지 않는다. 진행바와 화면 제목이 자동 판정 흐름을 반영한다. DEXCOWIN MES의 기존 PC 입출고 카드/진행바 디자인 언어와 어긋나지 않는다.
- 브라우저 확인 시나리오: PC 브라우저에서 `/mes?tab=warehouse`로 진입해 품목 전환을 선택한다. 첫 작업 화면에서 `사양 전환 / 구성 전환` 선택 카드가 사라졌는지 확인한다. 소스/대상 품목 선택 흐름이 바로 보이고, 차이 확인 단계에 들어갔을 때 시스템 판정 라벨로만 사양/구성 전환이 표시되는지 확인한다.
- 우선순위: 높음

### 4. 전환 수량 입력을 소스/대상 선택과 같은 핵심 위계로 올리기
- 화면/영역: PC 입출고 > 품목 전환 > 소스·대상 선택 단계 하단
- 증상: 첨부 캡처 기준으로 `전환 수량` 입력이 화면 왼쪽 아래에 작게 배치되어 잘 보이지 않는다. 소스 품목 목록과 대상 품목 영역에 비해 시각적 위계가 낮아, 수량 입력이 부가 정보처럼 보인다.
- 사용자 불편: 전환 수량은 재고 차감/입고와 BOM 차이 계산을 결정하는 핵심 조건인데, 현재 위치와 크기 때문에 작업자가 놓치기 쉽다. 특히 소스/대상 품목 선택 후 다음 단계로 넘어가려는 순간에야 작은 입력을 발견할 가능성이 있다.
- 확정 정책: `전환 수량`은 소스 품목, 대상 품목과 같은 단계의 필수 입력으로 취급한다. 화면 구석의 작은 보조 필드가 아니라, 품목 선택 흐름 안에서 자연스럽게 보이고 조작 가능한 위치에 둔다.
- 구현 방향: 기존 소스/대상 선택 방식은 유지하되, 수량 입력을 하단 왼쪽 고립 위치에서 빼내어 소스/대상 선택 요약 또는 두 품목 선택 영역 사이/아래의 명확한 입력 블록으로 재배치한다. 라벨은 `전환 수량`을 유지하되, 입력 크기와 주변 여백을 키워 핵심 액션으로 보이게 한다. 가능하면 선택된 소스 품목 재고와 연결해 "소스에서 n개를 대상 품목으로 전환"이라는 문장형 요약과 함께 보여준다.
- 수용 기준: 소스/대상 선택 화면을 처음 봤을 때 `전환 수량` 입력이 즉시 눈에 들어온다. 입력은 최소 44px 높이의 명확한 조작 영역을 가진다. 소스/대상 품목을 고른 뒤 수량이 현재 전환 조건의 일부라는 점이 화면 요약에서 드러난다. 다음 단계 버튼을 누르기 전에 수량 누락이나 기본값을 자연스럽게 확인할 수 있다.
- 브라우저 확인 시나리오: PC 브라우저에서 `/mes?tab=warehouse`로 진입해 품목 전환의 소스·대상 선택 단계로 이동한다. 소스 품목을 선택한 뒤 대상 품목 영역과 함께 `전환 수량`이 충분히 크게 보이는지 확인한다. 수량을 변경했을 때 화면의 전환 요약 또는 다음 단계 가능 여부가 자연스럽게 연결되어 보이는지 확인한다.
- 우선순위: 높음

### 5. 품목 전환의 품목 선택 방식을 기존 검색/선택 문법과 맞추기
- 화면/영역: PC 입출고 > 품목 전환 > 소스·대상 품목 선택 단계
- 증상: 현재 품목 전환은 왼쪽 `소스 품목`, 오른쪽 `대상 품목` 패널을 동시에 보여주고, 한쪽에서 품목을 선택하면 해당 패널의 검색 결과 영역이 비어 있는 큰 상태로 남는다. 첨부 캡처에서는 소스 품목 선택 후 왼쪽 패널이 선택 요약과 큰 빈 결과 영역으로 바뀌고, 오른쪽 패널에 대상 후보 목록이 남는다.

- 구현 방향: `소스 품목`과 `대상 품목`을 모두 대형 검색 결과 패널로 계속 유지하기보다, 선택 전에는 검색 결과 리스트를 보여주고 선택 후에는 선택 요약 카드와 `변경` 동작을 제공하는 문법을 우선 검토한다. 대상 품목은 소스 선택 이후 후보가 좁혀지는 기존 동작을 유지하되, 선택된 소스 쪽에는 빈 결과 목록 대신 선택 요약, 현재 재고, 전환 가능 조건, 변경 버튼을 보여준다. 소스와 대상이 모두 선택되면 두 품목의 관계와 전환 수량이 한 줄 또는 한 블록으로 요약되어야 한다.
- 수용 기준: 품목 전환 화면에서 품목 선택 방식이 기존 DEXCOWIN MES의 검색 결과 카드/선택 요약/변경 버튼 문법과 일관된다. 선택된 소스 품목 패널에 큰 빈 결과 영역이나 `검색 결과가 없습니다` 상태가 남지 않는다. 사용자는 현재 선택된 소스와 대상, 그리고 각 항목을 어떻게 변경하는지 즉시 이해할 수 있다.
- 브라우저 확인 시나리오: PC 브라우저에서 `/mes?tab=warehouse`로 진입해 품목 전환의 소스·대상 선택 단계로 이동한다. 소스 품목을 선택한다. 선택된 소스가 빈 검색 결과 패널이 아니라 명확한 선택 요약으로 표시되는지 확인한다. 대상 품목을 선택한 뒤 소스/대상/수량 요약과 변경 동작이 기존 화면 문법과 어울리는지 확인한다.

### 6. 품목 전환 내부 진행바와 본문 카드 사이의 상하 여백 균형 맞추기

- 확정 정책: 아까 상단 탭 카드에서 정한 것과 동일하게 아래 여백을 기준으로 위 여백을 맞춘다. 아래쪽 공간을 임의로 늘리기보다, 위쪽 남는 공간을 줄여 진행바 주변의 상하 외부 여백이 같은 밀도로 보이게 한다.
- 구현 방향: 진행바 카드 자체의 높이, 색상, 완료/활성/비활성 상태 표현은 유지한다. 진행바를 감싸는 품목 전환 작업 화면 컨테이너의 상단 padding 또는 margin을 조정하고, 아래 본문 카드와의 간격을 기준값으로 삼는다. 상단 탭 카드 여백 조정과 같은 spacing 규칙을 재사용해 화면마다 다른 임의 간격이 생기지 않게 한다.
- 수용 기준: 품목 전환 소스·대상 선택 화면에서 내부 진행바의 위쪽 빈 공간과 아래쪽 빈 공간이 시각적으로 균형 있게 보인다. 진행바는 아래 본문 카드와 자연스럽게 한 작업 묶음으로 읽힌다. 진행바 버튼의 높이, 간격, 텍스트 중앙 정렬은 기존 디자인을 유지한다.
- 브라우저 확인 시나리오: PC 브라우저에서 `/mes?tab=warehouse`로 진입해 품목 전환의 소스·대상 선택 단계로 이동한다. 내부 진행바 위쪽 여백과 아래 본문 카드까지의 여백을 비교한다. 상단 탭 카드 여백 조정과 같은 리듬으로 정렬되어 보이는지 확인한다.

### 7. BOM 매칭 단계의 우측 보조 카드들을 상단 요약으로 통합하고 변경 구성품 목록을 넓히기

- 사용자 불편: 실제로 사용자가 확인해야 하는 핵심은 변경 구성품인데, 보조 판단 카드들이 우측 너비를 차지해 변경 구성품 목록이 좁아진다. 변경 구성품이 많아지면 페이지 전체가 길어지는 대신 목록 내부에서 스크롤되어야 한다.

### 8. 출하 수량 카드의 수정 동선을 1단계 이동으로 통일
- 화면/영역: PC 출하 > 요청 작성 > BOM 매칭/최종 확인 상단 요약의 `출하 수량` 카드
- 증상: 현재 후속 단계에서 `출하 수량 10대`가 요약 카드로 표시된다. 사용자는 이 위치에서도 출하 수량을 바꿀 수 있게 할지, 아니면 수량을 바꾸기 위해 1단계로 돌아가게 할지 판단이 필요하다고 지적했다.
- 사용자 불편: 후속 단계에서 수량 오류를 발견했을 때 수정 동선이 불명확하다. 출하 수량은 BOM 수량, 변경 구성품 수량, 부족 수량을 모두 흔들기 때문에 후속 단계에서 직접 수정하면 숨은 재계산 범위가 커진다.
- 확정 정책: 후속 단계의 `출하 수량` 카드는 직접 입력 필드로 만들지 않는다. `수량 변경` 액션을 제공해 1단계로 돌아가 수정하게 한다. 현재 작성 내용은 보존하되, 수량 변경 후 필요한 BOM 매칭/요약/부족 계산은 새 수량 기준으로 다시 맞춘다.
- 구현 방향: 3단계 BOM 매칭과 5단계 최종 확인의 출하 수량 카드에 `수량 변경` 버튼 또는 링크를 둔다. 클릭하면 1단계로 이동하고 출하 수량 입력에 포커스를 준다. 기존 PF 선택, 구성 조정, 입력한 새 PA/PF 이름은 가능한 한 유지하되, 수량 의존 계산은 새 수량 기준으로 갱신한다.
- 수용 기준: 사용자는 후속 단계에서 출하 수량을 수정하려면 어디를 눌러야 하는지 즉시 알 수 있다. 후속 단계에서 수량 숫자만 직접 바꾸는 입력 UI는 없다. 수량 변경 후 BOM 구성 수량, 변경 구성품 수량, 부족 표시가 이전 수량 기준으로 남지 않는다.
- 브라우저 확인 시나리오: 출하 요청 작성 중 3단계 또는 5단계에서 출하 수량 요약을 확인한다. `수량 변경`을 눌러 1단계로 이동한다. 수량을 변경한 뒤 다시 후속 단계로 진행했을 때 모든 수량 요약이 새 출하 수량과 일치하는지 확인한다.
- 우선순위: 중간

### 9. 최종 확인 단계의 출하 품목을 새 PF 생성 예정 이름으로 표시

### 10. 최종 확인 단계에서 PA/PF BOM 전체와 동반 출하품을 함께 펼쳐 보여주기

- 증상: 현재 최종 확인 단계는 `PA 구성 12개 포함`, `PF 구성 6개 포함` 같은 요약 카드와 별도 카드들을 보여준다. 사용자는 준비 작업 화면처럼 PA/PF 구성 전체를 펼쳐서 보여주는 구조가 더 적합하다고 지적했다. 동반 출하품도 BOM과 함께 보여주되 색이나 라벨로 구분하는 방향을 제안했다.
- 사용자 불편: 마지막 단계에서 실제로 어떤 구성품과 동반 출하품이 요청에 포함되는지 전체를 한눈에 확인하기 어렵다. `PA 구성`, `PF 구성` 개수 요약만으로는 최종 확인으로 부족하고, `동반 출하품 없음` 같은 큰 빈 영역은 화면을 낭비한다.
- 확정 정책: 마지막 단계에서는 준비 작업 화면의 상세 구성 목록처럼 PA/PF BOM 전체를 펼쳐 보여준다. 동반 출하품은 같은 구성 확인 영역 안에 포함하되, BOM 구성품과 색/라벨을 다르게 하여 구분한다. 개수만 보여주는 PA/PF 요약 카드와 별도 새 PA/PF 예정 카드가 상단 요약에 통합되면 이 중복 카드들은 제거한다.
- 구현 방향: 최종 확인 영역을 `PA 구성`, `PF 구성`, `카톤·동반 출하품` 목록 중심으로 재구성한다. 각 목록은 내부 스크롤을 갖고, 준비 작업 화면의 `PrepRequirementList`와 유사한 카드 문법을 사용한다. 동반 출하품이 없으면 큰 빈 박스를 두지 말고, 상단 요약이나 작고 낮은 빈 상태로 처리한다. 변경된 구성품 요약은 전체 BOM 목록 안에서 `제외`, `추가`, `기본` 같은 라벨로 흡수하거나 별도 접힌 섹션으로 둔다.
- 수용 기준: 최종 확인 단계에서 사용자는 PA/PF 구성품 전체와 동반 출하품을 실제 품목명 기준으로 확인할 수 있다. PA/PF 개수 요약 카드만 보고 넘어가는 구조가 아니다. 동반 출하품이 없을 때도 큰 빈 카드가 화면 대부분을 차지하지 않는다.
- 브라우저 확인 시나리오: 출하 요청 작성 5단계로 이동한다. PA/PF 구성품 목록이 펼쳐져 보이는지 확인한다. 동반 출하품이 있는 케이스에서는 BOM과 함께 보이되 색/라벨로 구분되는지 확인한다. 동반 출하품이 없는 케이스에서는 큰 빈 영역이 사라졌는지 확인한다.
- 우선순위: 높음

### 11. 준비 작업 화면에서 부족한 구성품 라인을 목록 안에서 빨간색으로 표시
- 화면/영역: PC 출하 > 준비 작업 > PA/PF 구성품 목록과 재고 부족 표시
- 증상: 준비 작업 화면에는 상단에 재고 부족 경고가 있지만, 구성품 목록 안의 개별 품목이 부족한지 즉시 드러나지 않는다. 사용자는 구성품으로 들어가야 하는 것들 중 재고가 부족하면 해당 라인을 빨간색으로 표시하자고 제안했다.
- 사용자 불편: 작업자는 실제 준비 목록을 보면서 어떤 구성품이 부족한지 바로 찾아야 한다. 부족 정보가 별도 경고 박스에만 있으면 목록과 대조해야 해서 현장 작업 흐름이 느려진다.
- 확정 정책: PA/PF 구성품 및 동반 출하품 목록 안에서 재고 부족이 있는 라인은 빨간 계열로 표시한다. 부족 상태는 별도 경고 박스에도 남길 수 있지만, 목록 안의 라인 자체가 부족 상태를 가져야 한다.
- 구현 방향: `stock_shortages`의 item/phase 정보를 구성품 목록 라인과 매칭해 부족 라인에 빨간 border/background 또는 `부족` 배지를 적용한다. 부족 수량은 라인 하단 또는 우측에 작게 표시한다. 정상 라인과 부족 라인의 시각 차이는 DEXCOWIN MES의 기존 red tone/tint 규칙을 따른다.
- 수용 기준: 준비 작업 화면에서 부족한 구성품 라인은 목록 안에서 즉시 식별된다. 작업자는 별도 경고 박스를 읽지 않아도 어떤 품목이 부족한지 알 수 있다. 부족 라인 표시가 PA/PF 구성품과 동반 출하품 모두에 적용된다.
- 브라우저 확인 시나리오: 재고가 부족한 출하 준비 요청을 연다. PA/PF 구성품 목록에서 부족한 품목이 빨간색 또는 부족 배지로 표시되는지 확인한다. 정상 품목은 기존 색으로 유지되는지 확인한다.
- 우선순위: 높음

### 12. 준비 작업 재고 부족 수량을 `N개 부족`으로만 표시
- 화면/영역: PC 출하 > 준비 작업 > 재고 부족 경고 및 부족 라인 수량 표기
- 증상: 현재 부족 경고에 `필요 10 · 가용 0 · 부족 10` 같은 문구가 표시된다. 사용자는 이 말이 현장과는 잘 맞지 않는다고 지적했고, 부족한 수량만 알 수 있으면 된다고 결정했다.
- 사용자 불편: `필요`, `가용`, `부족`을 모두 나열하면 작업자가 실제로 필요한 정보보다 많은 숫자를 읽어야 한다. 준비 작업에서는 "몇 개가 모자란가"만 빠르게 알면 충분하다.
- 확정 정책: `필요/가용/부족` 3종 문구는 제거한다. 부족 라인에는 `3개 부족`, `10 EA 부족`처럼 부족한 수량만 짧게 표시한다.
- 구현 방향: `StockShortageNotice`와 구성품 부족 라인의 수량 표기를 `부족 {shortage_quantity}{unit}` 계열로 단순화한다. 요청 수량과 현재 재고 수량은 기본 화면에 노출하지 않는다. 부족 라인은 빨간 라인/배지로 표시해 시각 신호를 우선한다.
- 수용 기준: 준비 작업 화면에서 부족한 품목은 빨간 신호와 `N개 부족` 문구만으로 식별된다. `필요`, `가용`, `현재 재고` 같은 보조 수량 문구는 기본 화면에 보이지 않는다. 같은 부족 정보가 화면마다 다른 단어로 표시되지 않는다.
- 브라우저 확인 시나리오: 재고 부족이 있는 준비 작업을 연다. 부족 라인이 빨간 신호와 `N개 부족` 형식으로 표시되는지 확인한다. `필요 10 · 가용 0 · 부족 10` 같은 긴 문구가 사라졌는지 확인한다.
- 우선순위: 중간
