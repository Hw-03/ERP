# 시나리오: 재고 입출고

> [!summary] 요약
> 재고가 창고에 들어오고(입고), 부서로 나가고(이동), 소진되는(출고) 모든 흐름. 총 6가지 시나리오.

---

## 재고 3계층 복습

이 ERP의 모든 재고 작업은 다음 3개 수량을 동시에 관리한다:

| 위치 | 컬럼 | 예시 |
|------|------|------|
| 중앙 창고 | `warehouse_qty` | 100 |
| 부서별 (조립/고압/...) | `inventory_locations.production_quantity` | 조립팀 30, 고압팀 20 |
| 부서별 불량 | `inventory_locations.defective_quantity` | 조립팀 2 |

**불변식**: 모든 위치 수량의 합 = `inventory.total_quantity` (자동 동기화)

---

## 시나리오 1: 일반 입고 (공급업체→창고)

### 언제
- 외부 업체에서 자재가 납품됐을 때
- 구매팀이 발주한 물건이 도착했을 때

### 흐름

```
[UI] 창고 입출고 탭 → "입고" 버튼 → 품목 선택 + 수량 입력
    ↓
[api.ts] api.receiveInventory({item_id, quantity, note})
    ↓
[POST /api/inventory/receive]
    ↓
[Service] inventory.receive_confirmed()
    ├─ warehouse_qty += 수량
    ├─ _sync_total() 호출 (총수량 재계산)
    └─ transaction_logs 에 RECEIVE 기록
    ↓
[Response] 업데이트된 재고 정보 반환
```

### 예시
- 튜브 A타입 50개 입고
- 처리 전: 창고 100, 조립팀 30, 총 130
- 처리 후: 창고 150, 조립팀 30, 총 180
- `transaction_logs`: `RECEIVE, +50, item_id=1247`

---

## 시나리오 2: 창고 → 부서 이동

### 언제
- 조립팀이 생산용 자재 요청했을 때
- 특정 부서에 자재를 미리 배정할 때

### 흐름

```
[UI] 부서 입출고 탭 → 품목 선택 → 부서 선택 → "이관" 버튼
    ↓
[POST /api/inventory/transfer-to-production]
    ↓
[Service] inventory.transfer_to_production(item_id, dept, qty)
    ├─ warehouse_qty -= qty
    ├─ inventory_locations.production_quantity[dept] += qty
    ├─ _sync_total() (총수량 유지, 위치만 변경)
    └─ transaction_logs 에 TRANSFER_TO_PROD 기록
    ↓
```

### 예시
- 튜브 A 20개를 창고 → 조립팀 이동
- 처리 전: 창고 150, 조립팀 30
- 처리 후: 창고 130, 조립팀 50
- 총수량 180 그대로

### 실패 조건
- 창고 수량 부족 → 400 에러 "창고 재고 부족"
- pending으로 예약된 분은 이관 불가

## 시나리오 3: 부서 → 창고 반납

### 언제
- 부서에 보내뒀는데 안 써서 돌려보낼 때
- 잘못 배정해서 취소할 때

### 흐름

API: `POST /api/inventory/transfer-to-warehouse`

Service: `inventory.transfer_to_warehouse()`
- 부서 수량 감소
- 창고 수량 증가
- 총수량 유지

`transaction_logs`: `TRANSFER_TO_WH` 기록

---

## 시나리오 4: 부서 ↔ 부서 이동

### 언제
- 조립팀이 고압팀에 자재 넘겨줄 때

### 흐름

API: `POST /api/inventory/transfer-between-depts`

Service: `inventory.transfer_between_depts()`
- 출발 부서 수량 감소
- 도착 부서 수량 증가
- 창고 수량은 그대로
- 총수량 유지

`transaction_logs`: `TRANSFER_DEPT` 기록

---

## 시나리오 5: 불량 등록

### 언제
- 부서에서 쓰다가 불량 발견했을 때
- 작업 중 파손됐을 때

### 흐름

```
[UI] 품목 선택 → "불량 처리" → 부서 + 수량 입력
    ↓
[POST /api/inventory/mark-defective]
    ↓
[Service] inventory.mark_defective()
    ├─ production_quantity[dept] -= qty
    ├─ defective_quantity[dept] += qty
    ├─ (총수량은 그대로, 불량도 재고로 친다)
    └─ transaction_logs 에 MARK_DEFECTIVE 기록
    ↓
```

### 중요
- 불량 등록만으로는 재고가 사라지지 않음 (단지 상태만 바뀜)
- 실제로 버리려면 → 시나리오 6 (폐기 or 반품)

---

## 시나리오 6-A: 공급업체 반품 (불량 원자재)

### 언제
- 납품받은 원자재에 불량이 있어서 업체에 돌려보낼 때

### 흐름

API: `POST /api/inventory/return-to-supplier`

Service: 해당 수량만큼 재고 완전 차감
- `defective_quantity` 또는 `warehouse_qty` 에서 감소
- 총수량 감소
- `transaction_logs`: `SUPPLIER_RETURN`

---

## 시나리오 6-B: 폐기 (스크랩)

### 언제
- 불량 재고가 수리 불가로 판단돼서 버릴 때

### 흐름

API: `POST /api/scrap`

Service (`scrap.py`):
- `defective_quantity` 또는 `production_quantity` 감소
- 총수량 감소
- `scrap_logs` 에 기록
- `transaction_logs`: `SCRAP`

상세: `backend/app/routers/scrap.py`

---

## 시나리오 7: 단순 출고 (고객 배송)

### 언제
- 완제품(FG)을 고객에게 보낼 때

### 흐름

API: `POST /api/inventory/ship` 또는 `POST /api/inventory/ship-package` (패키지 단위)

- 완제품 재고 차감 (창고에서)
- `transaction_logs`: `SHIP`
- 패키지 사용 시 `ship_packages` 테이블에 기록

## 자주 발생하는 궁금증

**Q. 재고 조정(adjust)은 언제 쓰나?**
실제 재고가 시스템과 다를 때. 예: 창고 실사 결과 50개인데 시스템은 52개 → `POST /api/inventory/adjust` 로 -2 조정. `transaction_logs`: `ADJUST`. 자세한 건 `backend/app/routers/counts.py` 참고.

**Q. pending 이 뭐고 언제 걸리나?**
생산 배치(queue)에서 출고 예정된 수량. 배치 생성 즉시 `pending_quantity` 예약, 확정 시 실제 차감 + pending 해제. 취소 시 pending 만 해제.

**Q. 부서 수량을 직접 입력해서 고칠 수 없나?**
직접 수정 API는 없음. `adjust` + `transfer` 조합으로 조정. 직접 수정은 감사 추적 불가해 위험.

**Q. 불량 재고를 다시 정상으로 되돌릴 수 있나?**
현재 API 직접 미제공. 수동 `adjust`로 처리하거나 해당 라우터 확장 필요.

**Q. 창고 재고가 마이너스 될 수 있나?**
설계상 방지. 출고/이관 시 사전 검증해서 부족하면 에러 반환.
