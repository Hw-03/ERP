# 처음 읽는 사람을 위한 ERP Vault 안내

> [!summary] 역할
> 이 문서는 ERP 프로젝트를 처음 인수인계 받는 사람을 위한 **첫날 안내서**다.
> 코딩을 조금 아는 고등학생도 전체 흐름을 잡을 수 있도록, 읽는 순서와 헷갈리기 쉬운 개념을 먼저 정리한다.

---

## 이 Vault는 뭐예요?

이 폴더는 ERP 프로젝트의 실제 코드와 설명을 Obsidian에서 보기 좋게 정리한 **인수인계용 지식 창고**다.

원본 프로젝트는 `C:\ERP`에 있고, 이 Vault는 그 내용을 공부하기 쉽게 풀어쓴 버전이다. 실제 코드 파일도 들어 있지만, 대부분은 Obsidian에서 읽기 좋게 `.md` 문서로 한 번 감싼 형태다.

## 가장 중요한 한 문장

> 이 ERP는 공장 자재가 **어디에, 얼마나 있고, 언제 들어오고 나갔는지** 기록하는 내부 업무 시스템이다.

회계, 인사, 급여 시스템이 아니다. 이 프로젝트의 중심은 **품목, 재고, 입출고, 생산, 출하**다.

## 폴더 구조를 먼저 이해하기

| 위치 | 의미 |
|---|---|
| `backend/` | 서버 쪽 코드 설명. API, DB, 재고 처리 로직이 여기에 있다. |
| `frontend/` | 화면 쪽 코드 설명. 사용자가 보는 페이지와 버튼이 여기에 있다. |
| `docs/` | AI 인계, 진행 상황, 통합 리포트 같은 문서. |
| `scripts/` | 엑셀 자료를 읽거나 DB를 준비하는 보조 스크립트. |
| `data/` | 데이터 파일 설명. |
| `_vault/` | 원본 프로젝트에는 없는 Obsidian 전용 안내서, 대시보드, 템플릿. |
| `.obsidian/` | Obsidian 앱 설정. 보통 직접 볼 필요 없다. |

## 왜 파일 이름이 `.md.md`인가요?

예를 들어 원본 프로젝트에 `backend/app/main.py`라는 파일이 있으면, Vault에는 `backend/app/main.py.md`처럼 저장되어 있다.

이유는 간단하다. **원본 코드 파일을 Obsidian에서 설명 문서로 읽기 위해 Markdown 파일로 만든 것**이다.

| 원본 프로젝트 | Vault 문서 |
|---|---|
| `README.md` | `README.md.md` |
| `backend/app/main.py` | `backend/app/main.py.md` |
| `frontend/lib/api.ts` | `frontend/lib/api.ts.md` |
| `schema.sql` | `schema.sql.md` |

처음에는 이상해 보여도, “원본 파일 이름 + `.md`”라고 생각하면 된다.

## 30분 안에 전체 파악하기

처음 보는 날에는 코드를 바로 보지 말고 아래 순서대로 읽는다.

1. ERP Control Room — 전체 대시보드
2. ERP MOC — 프로젝트 전체 지도
3. 용어 사전 — 모르는 단어 확인
4. FAQ 전체 — 자주 막히는 질문 확인
5. 재고 입출고 시나리오 — 가장 기본 업무 흐름 이해
6. `docs/AI_HANDOVER.md.md` — 현재 UI와 개발 상태 확인

## 2시간 안에 구조 파악하기

30분 코스를 읽은 뒤, 아래 순서로 보면 화면과 서버가 어떻게 이어지는지 감이 잡힌다.

1. `frontend/frontend.md` — 프론트엔드 전체 설명
2. `frontend/app/legacy/legacy.md` — 현재 실제로 쓰는 화면
3. `frontend/lib/api.ts.md` — 화면이 서버에 요청을 보내는 통로
4. `backend/backend.md` — 백엔드 전체 설명
5. `backend/app/routers/routers.md` — API 주소 모음
6. `backend/app/models.py.md` — DB 테이블 구조

## 하루 안에 인수인계 받기

하루 동안 볼 수 있다면 업무 시나리오를 모두 읽는다.

| 순서 | 문서 | 왜 보나요? |
|---|---|---|
| 1 | 품목 등록 시나리오 | 새 자재가 시스템에 들어오는 시작점 |
| 2 | 재고 입출고 시나리오 | ERP에서 가장 자주 쓰는 기본 흐름 |
| 3 | 생산 배치 시나리오 | 자재가 생산에 쓰이는 방식 |
| 4 | 분해 반품 시나리오 | 생산 후 되돌리거나 분해하는 예외 흐름 |
| 5 | `docs/CODEX_PROGRESS.md.md` | 어디까지 개발됐는지 확인 |
| 6 | ERP System Map.canvas | 문서들을 그림처럼 연결해서 보기 |

## 난이도별 읽는 법

| 내 수준 | 먼저 볼 것 | 나중에 볼 것 |
|---|---|---|
| 코딩 거의 모름 | 용어 사전, FAQ 전체, 시나리오 문서 | `backend/`, `frontend/` 코드 노트 |
| HTML/JS 조금 앎 | `frontend/`, `frontend/app/legacy/` | `backend/app/routers/` |
| Python 조금 앎 | `backend/`, `backend/app/routers/` | `backend/app/services/`, `models.py` |
| DB 조금 앎 | `backend/app/models.py.md`, `schema.sql.md` | 마이그레이션/스크립트 문서 |

## 꼭 알아야 하는 단어 10개

| 단어 | 쉬운 뜻 |
|---|---|
| ERP | 회사 안의 자재, 재고, 생산 정보를 관리하는 시스템 |
| 품목 | 관리 대상이 되는 자재나 제품 하나 |
| 재고 | 지금 가지고 있는 수량 |
| 입고 | 자재가 들어오는 것 |
| 출고 | 자재가 나가는 것 |
| 이동 | 창고나 부서 사이에서 자재 위치가 바뀌는 것 |
| BOM | 제품 하나를 만들 때 필요한 부품 목록 |
| 백플러시 | 생산 완료 시 필요한 자재를 자동으로 차감하는 처리 |
| API | 화면과 서버가 대화하는 약속된 주소 |
| DB | 품목, 재고, 이력을 저장하는 데이터베이스 |

더 자세한 설명은 용어 사전에서 찾는다.

## 코드를 볼 때의 기본 흐름

화면에서 버튼을 누르면 대략 이런 순서로 일이 일어난다.

```text
사용자 화면
  -> frontend/lib/api.ts
  -> backend/app/routers
  -> backend/app/services
  -> backend/app/models.py
  -> SQLite DB
```

이 흐름만 기억하면 처음 보는 파일도 덜 무섭다.

## 헷갈리기 쉬운 것

> [!warning] `schema.sql`은 현재 운영 DB가 아니다
> 현재 실제 운영 기준은 `backend/app/models.py`다. `schema.sql`은 나중에 PostgreSQL로 이관할 때 참고하려고 만든 설계 문서다.

> [!warning] `_vault/`는 원본 프로젝트 폴더가 아니다
> `_vault/`에는 인수인계용 안내서, 대시보드, 템플릿이 들어 있다. 실제 코드 구조와 비교할 때는 `backend`, `frontend`, `docs`, `scripts`, `data`를 보면 된다.

> [!tip] Dataview가 없어도 괜찮다
> `ERP Control Room` 안의 `dataview` 블록은 자동 목록을 만들기 위한 플러그인 문법이다. 플러그인이 없어도 링크와 표는 그대로 읽을 수 있다.

## 처음 인수인계 받을 때 물어볼 질문

- 실제로 지금 운영 중인 화면 주소는 어디인가?
- 현재 DB 파일이나 서버는 어디에서 돌고 있는가?
- 엑셀 원본 파일은 누가, 언제 업데이트하는가?
- 재고 수량이 틀렸을 때 누가 수정 권한을 가지는가?
- 지금 가장 급한 미해결 문제는 무엇인가?
- 새 기능보다 더 중요한 안정화 작업이 있는가?

## 처음 수정하기 전에 지켜야 할 것

1. `_archive`, `_backup`, `frontend/_archive`는 건드리지 않는다.
2. 문서보다 실제 코드가 우선이다.
3. DB 관련 수정은 `models.py`, 라우터, 서비스 흐름을 같이 확인한다.
4. 화면 수정은 `frontend/app/legacy/`부터 확인한다.
5. 모르는 용어는 추측하지 말고 용어 사전에 추가한다.
