# 처음 읽는 사람을 위한 DEXCOWIN MES Vault 안내

> [!summary] 역할
> DEXCOWIN MES 프로젝트를 처음 인수인계 받는 사람이 **첫날** 읽는 안내서다.
> 1년 2개월차 비전공자 신입을 가정해 쓰여 있다. 어려운 단어는 [[erp/_vault/guides/용어사전]]에서 풀이를 본다.

> [!quote] 마음가짐
> "모르는 게 정상이다. 모르는 걸 모른 채로 두는 게 문제다."
> 첫 주는 코드를 고치지 말고 **읽기만** 한다. 한 화면을 누르면 어디로 흘러가는지 추적하는 데만 집중해라.

---

## 0. 이 시스템은 뭐예요?

> [!info] 한 문장 정의
> **DEXCOWIN MES**는 공장 자재가 어디에 얼마나 있고, 언제 어떻게 움직였는지를 기록하는 내부 운영 시스템이다.

- 공식 이름: **DEXCOWIN MES** (사내 문서·UI·발표 자료 모두 이 이름으로 통일)
- 옛 호칭: 초기엔 "ERP", "X-Ray ERP"로 불렸지만 현재는 사용하지 않는다. 단 폴더명 `erp/`, 일부 식별자(`erp_code`, `xray-erp`) 같은 **legacy 내부 식별자는 그대로 둔다** — 마음대로 바꾸면 마이그레이션·CI 가 와장창 깨진다.
- 대상 사용자: DEXCOWIN 내부 직원 (생산·창고·관리자)
- 주 화면: 데스크톱 셸 + 모바일 셸 (2026-05 모바일 4화면 1차 PASS)

> [!warning] 이름 규칙 (꼭 지킬 것)
> - 사용자 노출 텍스트: **DEXCOWIN MES**
> - 폴더·파일 경로: `erp/...` (그대로)
> - DB 컬럼·코드 식별자: `erp_code`, `xray-erp` 등 (그대로)

## 1. 이 Vault는 뭐예요?

`erp/` 프로젝트를 Obsidian에서 읽기 좋게 정리한 **문서 묶음**이다.

- 실제 코드는 `erp/backend/`, `erp/frontend/`, `erp/scripts/`, `erp/docs/`, `erp/data/`, `erp/docker/` 에 있다.
- `erp/vault/` 는 그 구조를 **설명하는 레이어**일 뿐, 코드를 돌릴 때는 아무 영향을 주지 않는다.
- Vault 자체는 `vault-sync` 브랜치에만 존재한다. `main` 브랜치엔 `vault/` 폴더가 없다.

> [!tip] 옵시디언으로 열어라
> 파일 트리만 봐도 헷갈리는데, Obsidian으로 `erp/vault/` 를 vault 루트로 잡고 열면 wiki-link·callout·mermaid 가 다 렌더링돼서 훨씬 잘 읽힌다.

## 2. 가장 중요한 한 문장

> [!quote] 시스템 본질
> 이 MES는 공장 자재가 **어디에 / 얼마나 / 언제 / 어떻게** 움직였는지를 기록하는 내부 운영 시스템이다.

이걸 잊으면 코드를 읽다가 길을 잃는다. "이 함수는 결국 재고를 움직이는 거다"라는 한 줄로 모든 라우터를 다시 해석해라.

## 3. 전체 흐름 한눈에

```mermaid
flowchart LR
  A[원자재 입고<br/>R 타입] --> B[창고 보관]
  B --> C[부서 작업<br/>A 타입 조립품]
  C --> D[검사·완성<br/>F 타입 완제품]
  D --> E[출고]
  C -. 불량 격리 .-> F[불량 처리 허브]
  F -. 정상 복귀 .-> C
  F -. 폐기/반품/분해 .-> G[(영구 기록)]
```

> [!info] 자재 흐름 4단계
> **자재(R) → 부서 작업(A) → 완제품(F) → 출고**
> 중간에 불량이 발생하면 별도 풀(격리)로 빠졌다가 **정상 복귀 / 폐기 / 공급처 반품 / 분해** 중 하나로 처리된다.

### 5.1 inventory 패키지 구조

> [!example] inventory 는 더 이상 단일 파일이 아니다
> 예전엔 `erp/backend/app/routers/inventory.py` 한 파일(807줄)이었지만, Phase 4 분할 이후 **패키지**가 됐다. `from app.routers import inventory` 라는 import 는 그대로 작동한다 (`__init__.py` 가 호환을 유지함).

```
erp/backend/app/routers/inventory/
├── __init__.py          # APIRouter 조립 + GET "" 목록만 직접 정의
├── _shared.py           # 응답 변환 헬퍼 (to_response_bulk 등)
├── query.py             # /summary, /locations/{item_id}
├── transactions.py      # /transactions, /transactions/export.{csv,xlsx}, PUT /transactions/{log_id}
├── transfer.py          # /transfer-to-production, /transfer-to-warehouse, /transfer-between-depts
├── receive.py           # /receive, /adjust
├── defective.py         # /mark-defective
├── supplier.py          # /return-to-supplier
└── weekly_report.py     # 주간보고 산출
```

> [!tip] inventory 안에서 길 찾기
> "이 화면이 어떤 핸들러를 부르나?" 가 궁금하면:
> 1. 프론트에서 호출하는 path 를 본다 (`/api/inventory/transfer-to-production` 같은 거)
> 2. `erp/backend/app/routers/inventory/` 안의 파일명을 보고 가장 가까운 서브 모듈을 연다
> 3. 라우터에서 `services/` 로 위임하는 패턴이 흔하다

## 8. 최근 3.5주 큰 변화 (2026-04-28 → 2026-05-21)

> [!info] 왜 이 절이 있나
> 옛 문서를 읽다 보면 "지금 코드랑 다른데?" 싶은 순간이 온다. 이 4건이 최근 굵직한 재설계라 문서와 코드가 어긋날 가능성이 높다.

- **불량 처리 흐름 재설계** — [[erp/docs/defect-handling-redesign.md]]
  격리·폐기·반품·분해 + 부서 계층(생산부 산하 6라인) + 결재 라우팅을 통째로 재설계.
- **items.item_code 통합** / `ErpCode` → `ItemCode` 도메인 rename
  품목코드 컬럼·도메인 용어를 단일화. 단 DB 컬럼 `erp_code` 는 legacy 식별자로 그대로 유지.
- **UtcDatetime 응답 스키마 확산 (F4b 완료)**
  모든 응답 시간 필드를 UtcDatetime 으로 통일해 화면 9시간 오차 문제를 근본 수정.
- **입출고 내역 죽은 거래 타입 5종 제거**
  실제로 호출되지 않던 거래 타입 라벨/분기를 제거해 내역 화면 정리.

> [!question] 옛 문서가 새 코드와 안 맞으면?
> **코드가 정답**이다. 문서는 따라잡지 못한 거다. CLAUDE.md 의 "If docs and live code disagree, trust the live code" 규칙 그대로.

## 9. 헷갈리기 쉬운 것

> [!warning] `legacy` 는 현재 실제 UI 다
> 이름이 legacy 라서 폐기된 것 같지만, 지금 사용자가 보는 **주 데스크톱 화면**이 `erp/frontend/app/legacy/` 다. 함부로 지우면 화면이 사라진다.

> [!warning] 예전 단일 파일을 찾지 마라
> - `erp/backend/app/routers/inventory.py` → 지금은 **패키지** (`inventory/` 폴더)
> - 옛 `AdminTab.tsx` 단일 거대 파일 → 섹션별 하위 컴포넌트로 분리
> - 옛 워크하우스 단일 위자드 → `_warehouse_v2/`, `_warehouse_sections/` 로 책임 분리
>
> 옛 import 경로가 검색돼도 지금은 그 파일이 없거나 형태가 바뀌었을 수 있다.

> [!warning] 죽은 라우터 이름들
> `queue`, `alerts`, `counts`, `loss`, `ship_packages` 는 코드에 없다. 옛 흔적일 뿐. (2026-05-20 정리)

> [!warning] 브랜치 정책
> - `main` — 코드만 (vault 없음)
> - `vault-sync` — vault 포함 (현재 브랜치)
> 합치지 말 것. `main` 에 vault 가 올라가면 코드 PR 리뷰가 진흙탕이 된다.

> [!warning] DB 는 함부로 만들지 마라
> "서버 한 번 띄워서 마이그레이션 자동으로 돌아갔으면 좋겠다" 같은 마음을 누르자.
> 스키마/시드/마이그레이션 작업은 명시적으로:
> ```bash
> cd erp/backend
> python bootstrap_db.py --all
> ```
> 그리고 영향을 사용자에게 먼저 보고하고 진행.

## 13. 처음 일주일을 위한 마음가짐

> [!quote] 신입에게
> "코드를 외울 필요 없다. 어디를 봐야 할지를 외워라."

- 첫 주는 **코드를 고치지 마라**. 화면 하나 누르고 → 호출 path 추적 → 라우터·서비스·모델까지 따라가 보는 훈련만.
- 모르는 단어는 [[erp/_vault/guides/용어사전]]에 다 있다. 없으면 알려달라고 해라.
- 옛 문서와 코드가 다르면 **코드가 정답**이다.
- 한 화면 동선이 끝까지 안 풀리면 [[erp/_vault/guides/위험지대_지도]] 를 봐라. 그 영역일 가능성이 높다.
- "이 코드 왜 이렇게 짰지?" 가 들면 [[erp/_vault/guides/바이브_코딩_컨텍스트]] 를 본다. AI 가 쓴 코드 특유의 흔적이 많다.

> [!tip] 둘러보기 순서 요약
> 1. 이 문서를 끝까지 본다
> 2. [[erp/_vault/guides/용어사전]] 으로 모르는 단어 5개만 찍어 본다
> 3. [[erp/_vault/dashboards/ERP_Control_Room]] 를 켜놓고 [[erp/_vault/guides/ERP_MOC]] 로 큰 그림을 잡는다
> 4. `start.bat` 한 번 실행해서 실제 화면을 본다 (코드 수정 X, **보기만**)
> 5. [[erp/_vault/guides/바이브_코딩_컨텍스트]] → [[erp/_vault/guides/AI_생성_코드_읽는_법]] → [[erp/_vault/guides/위험지대_지도]] 순으로 읽는다
> 6. [[erp/_vault/guides/첫주_체크리스트]] 를 따라 1주차를 보낸다
