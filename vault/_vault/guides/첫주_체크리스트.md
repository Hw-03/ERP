# ✅ 첫 주 체크리스트

> [!summary] 한 줄 결론
> 이 체크리스트를 끝까지 따라하면 **1주일 안에 첫 PR 을 안전하게 올릴 수 있다.**

#layer/meta #topic/onboarding #audience/junior

이 문서는 1년 2개월 IT 경력 비전공자 신입이 **DEXCOWIN MES** 코드에 자신감을 가질 때까지의 Day별 동선이다. 끝나면 두 가지가 가능해진다:

1. 코드 어디에 뭐 있는지 안다 (frontend → backend → DB 추적 가능)
2. 안전한 PR 1개를 만들 수 있다 (verify_local 통과 + 사용자 리뷰)

> [!tip] 진행 방식
> 체크박스를 옵시디언에서 직접 클릭해서 끄세요. 그날 분량이 안 끝나면 그냥 다음날로 넘기세요. 부끄러워하지 말고 막히면 사용자에게 바로 물어보세요.

---

## 🧰 시작 전 준비

오늘 (Day 0) 30분 안에 끝내는 환경 셋업.

### 필수 도구

- [ ] **VS Code** 또는 다른 IDE 설치
  - 추천 확장: Python, Pylance, ESLint, Prettier, GitLens
- [ ] **옵시디언** 설치 → `erp/vault/` 폴더를 vault 로 열기
  - 처음 열 때 "Trust this vault" 같은 거 묻거든 그냥 신뢰 누르세요
- [ ] **git** 설치 확인
  ```powershell
  git --version
  ```
- [ ] **Python 3.13+** 설치 확인
  ```powershell
  py --version
  ```
- [ ] **Node.js LTS** 설치 확인
  ```powershell
  node --version
  npm --version
  ```

> [!tip] 한 방에 설치하고 싶다면
> 그냥 `erp/start.bat` 한 번 실행하세요. Python / Node 가 없으면 winget 으로 자동 설치 옵션을 물어봅니다.

### 사용자에게 받아야 할 것

- [ ] `.env` 파일 (없으면 `.env.example` 참고해서 만들고, 채울 값은 사용자에게 물어보기)
- [ ] git 권한 (repo push 권한, branch 보호 룰)
- [ ] (선택) 옵시디언 vault 안에 본인 작업 폴더 만들 위치

> [!warning] 절대 하지 말 것
> - DB 파일 (`erp/backend/mes.db`) 을 임의로 삭제/덮어쓰기
> - `_attic/` 안의 파일을 마음대로 수정 (옛 `_archive/`, `_backup/`, `frontend/_archive/` 가 모두 이 안으로 통합됐음. `_attic/.obsidian/` 도 보호 영역)
> - main 브랜치에 직접 push

## 📅 Day 1 — Vault 산책

> [!summary] 목표
> "이 시스템이 왜 만들어졌고, 어떻게 굴러가는지" 큰 그림을 잡는다. 코드는 아직 안 만진다.

### Vault 읽기

- [ ] [[처음_읽는_사람]] 끝까지 읽기
- [ ] [[왜_이_시스템인가]] 끝까지 읽기 (왜 이걸 만들었나)
- [ ] [[바이브_코딩_컨텍스트]] 읽기 (이 시스템이 Claude Code / Codex 로 어떻게 만들어졌는지)
- [ ] [[용어사전]] 훑기 — 다음 용어들은 꼭 외울 것:
  - **ItemCode** (품목코드 포맷)
  - **FlowBadge** (입출고 화면의 흐름 표시)
  - **ADJUST 묶음** (재고 조정 트랜잭션)
  - **process_type_code** (TR/TA/TF/HR/HA/HF/...)
- [ ] [[ERP_Control_Room]] 대시보드 열어보기 (옵시디언 캔버스도 같이)

### 시스템 실제 띄우기

여기서부터는 실제로 손을 움직입니다.

- [ ] **백엔드 띄우기**:
  ```powershell
  cd backend
  python bootstrap_db.py --all
  python -m uvicorn app.main:app --reload
  ```
- [ ] **프론트엔드 띄우기** (새 터미널):
  ```bash
  cd frontend
  npm install
  npm run dev
  ```
- [ ] 브라우저에서 `http://localhost:3000` 열어서 첫 화면 확인
- [ ] (편하게 하고 싶으면) 그냥 루트에서 `start.bat` 한 번 실행해도 됨

### 화면 클릭해보기

- [ ] `/legacy` 경로의 4개 탭 다 눌러보기:
  - **대시보드** — 재고 요약, 생산 현황
  - **입출고** — 자재 받기/내기 (스캐너 흐름)
  - **입출고 내역** — 거래 로그
  - **관리자** — 품목/사용자/설정

> [!tip] 막히면
> "Day 1 의 X 단계에서 Y 라는 에러가 납니다" 라고 사용자에게 물어보세요. 화면 캡쳐가 있으면 더 좋습니다.

## 📅 Day 2 — 시나리오 따라가기

> [!summary] 목표
> 코드 어디에 뭐 있는지 **클릭으로** 알게 된다. 시나리오 노트를 따라가며 frontend → backend 경로를 실제로 열어본다.

### 시나리오 노트 따라하기

- [ ] [[시나리오_품목등록]] — 한 품목을 직접 만들어보기
  - 화면 클릭 → 어느 backend 라우터로 가는지 따라가기
- [ ] [[시나리오_재고입출고]] — 입출고 1회 시뮬레이션
  - 거래가 어디에 기록되는지 확인

### 실제 파일 열어보기

시나리오에 나온 코드 경로를 VS Code 에서 직접 열고, vault 노트에도 같은 게 있는지 wiki-link 로 점프:

- [ ] `erp/backend/app/routers/items.py` ↔ [[erp/backend/app/routers/items.py]]
- [ ] `erp/backend/app/routers/inventory.py` ↔ [[erp/backend/app/routers/inventory.py]]
- [ ] `erp/backend/app/models.py` ↔ [[erp/backend/app/models.py]]
- [ ] `erp/frontend/app/legacy/` ↔ [[erp/frontend/app/legacy/legacy]]
- [ ] `erp/frontend/lib/api.ts` 또는 `erp/frontend/lib/api/` 도메인 모듈들

> [!example] 추적 연습
> 화면에서 "품목 추가" 버튼 누름 → 어느 컴포넌트의 어느 함수가 호출됨? → 그게 어느 `lib/api/*.ts` 호출? → 그게 backend 어느 라우터? → 그게 어느 service / model 까지?
>
> 이 사슬을 한 번 끝까지 따라가면 Day 2 의 목표는 달성된 거예요.

## 📅 Day 3 — 추적 동선 익히기

> [!summary] 목표
> Day 2 에서 한 번 해본 추적을 **3가지 다른 방향에서** 반복한다. 손에 익을 때까지.

### 3대 추적 동선

- [ ] [[AI_생성_코드_읽는_법]] 끝까지 읽기
- [ ] **화면에서 출발**: 입출고 탭 한 화면을 골라서 끝까지 추적
  - frontend 컴포넌트 → `lib/api/*.ts` 호출 → backend 라우터 → service → `models.py`
- [ ] **API 에서 출발**: `GET /api/items` 한 엔드포인트를 골라서
  - 라우터 함수 → 어느 화면이 호출하는지 역추적
  - `erp/docs/openapi.json` 에서 스키마도 확인
- [ ] **DB 에서 출발**: `models.py` 의 한 테이블 (예: `Item`) 을 골라서
  - 어디서 INSERT / UPDATE 하는지 service 까지 역추적

### git log 도 읽어보기

- [ ] 한 파일의 변경 이력 따라가기:
  ```powershell
  git log -p backend/app/routers/items.py | Select-Object -First 100
  ```
- [ ] "이 함수는 왜 이렇게 됐지?" 싶을 때 `git blame` 으로 그 줄 커밋 메시지 보기

> [!tip] 추적이 막힐 때
> `Ctrl+Shift+F` (VS Code 전체 검색) 가 답입니다. 함수 이름 / API 경로 / 컴포넌트 이름 그대로 검색하면 호출처가 다 나옵니다.

## 📅 Day 4 — 위험지대 인식

> [!summary] 목표
> "어디를 건드리면 터지는지" 안다. 첫 PR 에서 위험지대를 피할 수 있게.

### 위험지대 학습

- [ ] [[위험지대_지도]] 끝까지 읽기. **7개 위험지대 위치 다 외우기**
- [ ] [[시나리오_분해반품]] 읽기 — 불량 처리 흐름 이해
- [ ] [[erp/docs/defect-handling-redesign.md]] 도 한 번 읽기 (불량 처리 재설계 배경)

### "묻고 손대라" 폴더 확인

다음 폴더들은 **사용자에게 묻기 전엔 절대 수정 금지**. 이전엔 루트에 `_archive/` `_backup/` `frontend/_archive/` 가 따로 있었지만, **지금은 모두 `erp/_attic/` 한 곳으로 통합됐다**:

- [ ] `erp/_attic/` 위치 확인 — 옛 `_archive`, `_backup`, `frontend/_archive` 가 모두 이 안에 통합됐음 (CLAUDE.md 의 옛 표기는 그대로 남아 있지만, 실제 폴더는 여기 다 모여 있다)
- [ ] `erp/_attic/_archive/` 안 둘러보기 — 옛 루트 `_archive/` 의 자리
- [ ] `erp/_attic/backend/_backup/` 안 둘러보기 — 옛 `backend/_backup/` 의 자리 (DB 백업 파일 포함)
- [ ] `erp/_attic/frontend/_archive/` 안 둘러보기 — 옛 `frontend/_archive/` 의 자리

> [!warning] 왜 묻고 손대라인가
> - `_attic/_archive/`, `_attic/backend/_backup/` : 운영 백업 / 이전 버전 DB 가 살아있음. 삭제하면 복구 불가.
> - `_attic/` 자체: 옛 작업 노트 + 참고 자료. 의도적으로 격리한 영역이라 "정리" 하면 안 됨 (`attic-quarantine-convention`).
> - `_attic/frontend/_archive/` : 옛 UI. 일부는 참고용으로 import 가 살아있을 수 있음.
> - `_attic/.obsidian/` : 옵시디언 플러그인/설정. 절대 손대지 말 것.

### 실수해도 되는 영역

반대로 다음은 비교적 안전합니다:

- `frontend/app/legacy/` 내부의 텍스트/색상 미세 조정
- vault 노트 (단, **vault-sync** 브랜치에서만)
- 명백한 오타

## 📅 Day 5 — 첫 PR

> [!summary] 목표
> **작고 안전한** 첫 PR 을 올린다. 크기보단 "내가 끝까지 했다" 가 중요.

### PR 후보 (셋 중 하나 골라)

- [ ] **A. 오타 / 라벨 / 색상 미세 조정** — frontend 한 화면의 작은 문구 수정
- [ ] **B. vault 노트 오류 수정** — 읽다가 발견한 오타나 깨진 wiki-link 고치기 (반드시 **vault-sync** 브랜치)
- [ ] **C. 죽은 import 제거** — 명백히 안 쓰이는 import 만. [[위험지대_지도]] 의 "묻고 손대라" 폴더(`_attic/` 통째)는 절대 X

> [!tip] 너무 큰 거 고르지 마세요
> 첫 PR 의 목표는 "흐름을 한 번 끝까지 돌아본다" 입니다. 영웅이 되는 건 다음 주부터.

### PR 만드는 순서

- [ ] **feature 브랜치 따기**:
  ```powershell
  git checkout -b fix/<your-short-name>
  ```
  (vault 수정이면 `git checkout vault-sync` 후 거기서 브랜치)
- [ ] **수정** — 한 가지만. 욕심내지 말 것.
- [ ] **로컬 검증** (commit 전 필수):
  ```powershell
  powershell -ExecutionPolicy Bypass -File .\scripts\dev\verify_local.ps1
  ```
- [ ] **commit** — 메시지 스타일은 `git log` 로 최근 커밋 보고 따라하기
  - 예: `fix: <짧은 설명>` / `docs: <짧은 설명>` / `chore: <짧은 설명>`
- [ ] **push**:
  ```powershell
  git push -u origin fix/<your-short-name>
  ```
- [ ] **PR 생성** — GitHub 웹에서 또는 `gh pr create`
- [ ] **사용자에게 리뷰 요청** — 슬랙/카톡/직접 알리기

> [!warning] verify_local 이 실패하면
> 절대 무시하고 push 하지 마세요. 에러 메시지 그대로 사용자에게 보여주고 같이 봅시다. (`--no-verify` 같은 것도 쓰지 마세요)

## 🎯 Week 1 끝났을 때 자기 평가

5개 다 ✅ 가 나오면 1주차 합격입니다.

- [ ] **흐름 설명**: frontend → backend → DB 흐름을 한 화면에 대해 끝까지 설명할 수 있다
- [ ] **위험지대 7개**: [[위험지대_지도]] 의 7개 영역 위치를 다 안다
- [ ] **verify 명령**: `verify_local.ps1` 을 막힘 없이 돌릴 수 있다
- [ ] **git 추적**: `git log` / `git diff` 로 "이게 왜 이렇게 됐는지" 추적할 수 있다
- [ ] **vault 네비게이션**: wiki-link 만으로 원하는 노트를 5초 안에 찾을 수 있다

> [!example] 자기 점검 질문 (사용자에게 답해보기)
> 1. 입출고 화면에서 "확인" 버튼을 누르면 어느 backend 함수가 호출되나요?
> 2. ItemCode 의 모델기호 `3` 은 무슨 모델인가요?
> 3. `_attic/` 안에서 옛 `_archive` 와 `_backup` 은 각각 어떻게 보존돼 있나? (힌트: `_attic/_archive/` vs `_attic/backend/_backup/`)
> 4. process_type_code 중 조립 F 타입은 `AF` 인가 `BF` 인가? 왜?
> 5. main 브랜치와 vault-sync 브랜치의 차이는?
>
> 4개 이상 답할 수 있으면 합격.
