# 시나리오: 새 품목 등록

> [!summary] 요약
> 사용자가 "신규 품목 추가" 버튼을 누른 순간부터, DB에 새 row가 생기고 화면에 반영되기까지 전체 흐름.

---

## 언제 이 시나리오가 발생하나

- 새 자재(원자재 또는 반제품)가 들어왔는데 기존 품목 마스터에 없을 때
- 새 제품 모델을 설계했을 때
- 엑셀에 있던 기존 품목을 DB로 이관할 때 (seed 스크립트 경로는 별개)

---

## 전체 흐름 한눈에

```
사용자 화면 클릭
    ↓
[Frontend] 입력 폼 채우기
    ↓
[api.ts] POST /api/items 호출
    ↓
[Router] routers/items.py 수신
    ↓
[Service] services/codes.py 가 ERP 코드 자동 생성
    ↓
[Model] models.py 의 Item 테이블에 INSERT
    ↓
[Inventory] 초기 재고 레코드(0수량) 생성
    ↓
[Response] 생성된 품목 정보 반환
    ↓
[Frontend] 목록 새로고침 → 새 품목 표시
```

---

## 단계별 상세

### 1. 화면에서 입력

위치: `frontend/app/legacy/_components/DesktopAdminView.tsx` 또는 품목 관리 페이지

사용자 입력:
- **이름** (예: "튜브_A타입_신형")
- **카테고리** (RM/TA/TF/.../FG 중 하나)
- **기호 슬롯** (예: 3번 슬롯 = ADX6000)
- **공정 타입** (예: TR, PA 등)
- **옵션** (선택사항, 예: BG, WM)
- **안전재고 수량** (품목 부족 경고 기준)
- **단위** (EA, KG, L 등)

### 2. API 호출

파일: `frontend/lib/api.ts` 의 `createItem()` 함수

```typescript
api.createItem({
  name: "튜브_A타입_신형",
  category: "TR",
  symbol_slot: 3,
  process_type: "TR",
  option_code: null,
  safety_stock: 10,
  unit: "EA"
})
```

내부적으로 `POST /api/items` 엔드포인트 호출.

### 3. Router에서 수신

파일: `backend/app/routers/items.py`

- Pydantic 스키마 `ItemCreate` 로 요청 본문 검증
- 필수 필드 누락 시 HTTP 422 반환

### 4. ERP 코드 자동 생성

파일: `backend/app/services/codes.py`

`generate_code(symbol_slot, process_type, option_code)` 호출.

내부 동작:
1. 같은 `symbol + process_type` 조합의 기존 최대 시리얼 조회
2. +1 해서 새 시리얼 결정 (예: 마지막이 0007이면 새로 0008)
3. `{기호}-{공정}-{시리얼:04d}[-{옵션}]` 형식으로 조립
4. 유효성 검증 (마스터 테이블에 등록된 기호/공정/옵션인지)

결과 예시: `3-TR-0008` 또는 `3-TR-0008-BG`

### 5. DB에 INSERT

파일: `backend/app/models.py`

`Item` 테이블에 새 row 삽입:

| 컬럼 | 예시 값 |
|------|---------|
| item_id | 1247 (자동 부여) |
| erp_code | "3-TR-0008" |
| name | "튜브_A타입_신형" |
| category | "TR" |
| symbol_slot | 3 |
| process_type | "TR" |
| safety_stock | 10 |
| unit | "EA" |

### 6. 재고 레코드 초기화

파일: `backend/app/services/inventory.py`

`get_or_create_inventory(item_id)` 호출:
- `inventory` 테이블에 수량 0인 row 생성
- 나중에 입고/출고 있을 때마다 이 row 를 업데이트

### 7. 응답 반환

생성된 품목의 전체 정보가 JSON으로 프론트에 돌아감:

```json
{
  "item_id": 1247,
  "erp_code": "3-TR-0008",
  "name": "튜브_A타입_신형",
  ...
}
```

### 8. 화면 반영

- 목록 리프레시 (`api.getItems()` 재호출)
- 토스트 메시지 "등록 완료"
- 새로 만든 품목이 목록 최상단 또는 해당 카테고리에 표시

## 실패할 수 있는 지점

| 단계 | 실패 상황 | 화면 반응 |
|------|---------|----------|
| 2 | 네트워크 오류 | "서버 연결 실패" 토스트 |
| 3 | 필수 필드 누락 | 폼 빨간 테두리 + 에러 |
| 4 | 없는 기호 슬롯 지정 | 422 에러 → "잘못된 기호 슬롯" |
| 4 | 없는 공정 코드 | 422 에러 → "잘못된 공정" |
| 4 | 중복된 ERP 코드 발생 (경합 상태) | 500 에러 → 재시도 필요 |
| 5 | DB 쓰기 실패 | 500 에러 → 관리자 문의 |

## 확인 방법

### 브라우저에서
1. 품목 목록 페이지 새로고침
2. 새로 만든 ERP 코드로 검색

### DB 직접 확인 (개발자)
```sql
SELECT * FROM items WHERE erp_code = '3-TR-0008';
SELECT * FROM inventory WHERE item_id = 1247;
```

### API 로그
백엔드 콘솔에 `POST /api/items - 201 Created` 출력 확인

## 자주 발생하는 궁금증

**Q. ERP 코드 수동으로 지정할 수 있나?**
기본적으로 자동 생성. 특수한 경우 코드 파싱 API(`POST /api/codes/parse`)로 검증 후 수동 입력 가능하지만 권장하지 않음.

**Q. 품목을 삭제하면 재고도 지워지나?**
현재 설계상 품목 삭제는 제한적. 이력 보존을 위해 `deleted_at` 플래그(soft delete) 방식 사용이 안전. 실제 구현은 `backend/app/routers/items.py` 확인 필요.

**Q. 엑셀로 대량 등록 가능한가?**
`scripts/` 폴더의 마이그레이션 스크립트로 일괄 시드 가능. 런타임에는 현재 한 건씩 API 호출이 표준.

**Q. 같은 이름의 품목 2개 만들 수 있나?**
이름은 유니크 제약 없음. ERP 코드가 실질적 고유 식별자. 같은 이름 중복 시 사용자가 혼동할 수 있으니 주의.
