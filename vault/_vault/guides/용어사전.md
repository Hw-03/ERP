# 📚 용어사전

> [!summary] 용도
> DEXCOWIN MES 코드/문서/UI 에서 마주칠 모든 전문용어를 비전공자가 이해할 수 있게 풀어 설명. 변경된 식별자(`ItemCode` 등)·새 라우터·새 트랜잭션 타입까지 모두 반영 (2026-05-21 기준).

> [!info] 핵심 변경 (2026-04-28 → 2026-05-21)
> - **ErpCode → ItemCode** (commit f1ff96c) · **UtcDatetime** 전체 응답 확산 (commit 4db421a)
> - 죽은 라우터 제거: queue / alerts / counts / loss / ship_packages
> - 시스템 공식명은 **DEXCOWIN MES**. ERP / X-Ray라 부르지 않는다.

## ㄱ

### 가용 수량 (available)

> [!info] 실제로 지금 쓸 수 있는 재고

```
available = warehouse_qty + Σ(InventoryLocation.quantity, status=PRODUCTION) − pending_quantity
```

격리(`DEFECTIVE`) 수량은 가용에서 빠진다. 큐 배치로 50개 예약돼 있으면 실제 가용도 50개 줄어듦.

**사용처**: `erp/backend/app/services/inventory.py`, `erp/backend/app/services/stock_math.py`

**관련 용어**: [[#Inventory 재고]], [[#예약 수량 pending_quantity]], [[#InventoryLocation 위치 재고]]

### 가상 테이블 (ORM 모델)

> [!info] DB 테이블을 파이썬 클래스로 표현한 것 — `erp/backend/app/models.py` 의 `Item` 클래스가 `items` 테이블과 연결

**관련 용어**: [[#ORM]], [[#Pydantic]]

### 공정 코드 (process_type_code)

> [!info] 품목이 어느 공정 단계인지 나타내는 2자리 코드 (총 18개)

예: `TR`(튜브 Raw), `PA`(완제품 Assembly). 첫 글자 = 공정 종류 (T튜브 / H고압 / V진공 / A조립 / P완제품), 둘째 글자 = 단계 (R 원자재 / A 조립). 품목 분류 단일 기준.

**사용처**: `erp/backend/app/models.py` (`ProcessType`), `erp/backend/app/utils/item_code.py`

**관련 용어**: [[#ItemCode]], [[#기호 product symbol]]

### 기호 (product symbol)

> [!info] 제품 모델을 나타내는 숫자 1자리

`3`=DX3000, `4`=ADX4000W, `6`=ADX6000FB, `7`=COCOON, `8`=SOLO. `product_symbols` 테이블 100개 슬롯. 공용 부품은 `346` 처럼 정렬 연결.

**사용처**: `erp/backend/app/models.py` (`ProductSymbol`)

**관련 용어**: [[#ItemCode]], [[#model_symbol]]

### 라우터 (router)

> [!info] API 엔드포인트를 기능별로 묶은 파일 (FastAPI). `erp/backend/app/routers/` 에 15개: `admin_audit / admin_audit_csv / bom / codes / departments / dept_adjustment / employees / io / items / models / production / settings / stock_requests / variance`

> [!warning] 죽은 라우터 제거됨
> 과거 `queue` / `alerts` / `counts` / `loss` / `ship_packages` 라우터는 모두 삭제됨 (commit 7f73550). 옛 문서에서 이 이름들을 보면 무시.

**관련 용어**: [[#엔드포인트 endpoint]], [[#StockRequest 결재 요청]]

### 라인 (line)

> [!info] 한 요청/배치 안에 들어가는 개별 자재/제품 한 줄

서비스 레이어에서 `LineInput` 컨테이너로 다룬다 (`item_id`, `quantity`, `from_bucket`, `from_department`, `to_bucket`, `to_department`). UI 에서 "조립부에서 게터 8개" 같은 한 줄이 라인이다.

**사용처**: `erp/backend/app/services/stock_requests.py:106`

**관련 용어**: [[#StockRequest 결재 요청]]

`erp/frontend/app/legacy/` 는 데스크탑 입출고탭, 대시보드, 내역 화면 등을 포함한 **현재 운영 중**인 UI 경로. "legacy 라서 만지면 안 되겠지" 라는 가정은 틀림. 실제 렌더 경로를 [[#import 경로 검증]] 후 작업하라는 CLAUDE.md 규칙이 여기에 적용된다.

**사용처**: `erp/frontend/app/legacy/_components/*`

**관련 용어**: [[#FlowBadge]], [[#모바일 wizard]], [[#_attic]]

**사용처**: `erp/backend/app/utils/item_code.py` (`slots_to_model_symbol`)

**관련 용어**: [[#ItemCode]], [[#기호 product symbol]]

## ㅂ

### 배치 (batch / io_batch)

> [!info] 한 번에 처리할 자재 묶음 (트랜잭션 그룹) — `io_batches` = 헤더, `TransactionLog.operation_batch_id` = FK. 같은 배치 트랜잭션은 입출고 내역에서 한 카드로 묶임

**사용처**: `erp/backend/app/models.py` (`TransactionLog.operation_batch_id`)

**관련 용어**: [[#ADJUST 묶음]], [[#FlowBadge]]

### 백플러시 (BACKFLUSH)

> [!info] 완제품 생산 확정 시 BOM 자재를 자동으로 빼는 처리 — "거꾸로 씻어내린다". 수동 자재 출고 불필요

**사용처**: `erp/backend/app/services/production.py`

**관련 용어**: [[#PRODUCE]], [[#BOM Bill of Materials]]

### bootstrap_db.py --all

> [!info] DB 초기화 / 시드 / 마이그레이션 명령

setup, schema change, migration, seed 작업은 모두 이 명령으로:

```bash
cd backend
python bootstrap_db.py --all
```

> [!warning] DB 변경 작업
> 서버 시작은 DB를 바꾸지 않는다. DB 가 바뀌는 건 오직 `bootstrap_db.py --all` 실행 시. 작업 전 영향도를 먼저 설명해야 한다 (CLAUDE.md 규칙).

**사용처**: `erp/backend/bootstrap_db.py`

**관련 용어**: [[#verify_local.ps1]]

### BOM (Bill of Materials, 자재명세서)

> [!info] 제품 1개 만들려면 어떤 자재가 몇 개 필요한지 기록한 표

예: "ADX6000 1대 = 튜브 1개 + 고압보드 2개 + 조립품 1개". 재귀 전개, 최대 깊이 10 (순환 방지).

**사용처**: `erp/backend/app/models.py` (`BOM`), `erp/backend/app/routers/bom.py`

**관련 용어**: [[#백플러시 BACKFLUSH]], [[#DISASSEMBLE]], [[#순환 BOM cyclic BOM]]

### 부서 (department)

> [!info] 자재/제품이 머무는 물리적 장소 — `DepartmentEnum` 10개: 조립 / 고압 / 진공 / 튜닝 / 튜브 / AS / 연구 / 영업 / 출하 / 기타

창고와는 별도로 관리. 향후 "생산부" 상위 부서가 추가되어 산하 6라인 결재를 묶을 예정 (`docs/defect-handling-redesign.md`).

**사용처**: `erp/backend/app/models.py` (`DepartmentEnum`, `Department`)

**관련 용어**: [[#warehouse_role]], [[#department_role]]

### 불량 (defective)

> [!info] 쓸 수 없다고 판단된 재고 — `InventoryLocation.status = DEFECTIVE` 행이 부서별 불량 풀. 격리 = 재작업 대기 풀 통합 (별도 "재작업중" 상태 없음)

**관련 용어**: [[#InventoryLocation 위치 재고]], [[#MARK_DEFECTIVE / SCRAP / SUPPLIER_RETURN / DISASSEMBLE]], [[#불량 처리 허브]]

### 불량 처리 허브

> [!info] 입출고탭 → 요청 작성 → Step 1 [불량] → 진입하는 별도 화면 (위자드 X, **대시보드 패턴**)

상단 KPI 카드 4개(격리 중 / 1년 이상 / 결재 대기 / 오늘 처리) + 액션 버튼 3개 + 부서별 접고 펼치는 목록. 대시보드 빨간 불량 수량 클릭으로도 진입.

**사용처**: `erp/docs/defect-handling-redesign.md` (설계 합의 완료, 구현 대기)

**관련 용어**: [[#MARK_DEFECTIVE / SCRAP / SUPPLIER_RETURN / DISASSEMBLE]]

## ㅅ

### 스키마 (schema)

> [!info] 두 가지 뜻 — (1) DB 스키마 = 테이블/컬럼 정의 (`erp/backend/app/models.py`), (2) API 스키마 = Pydantic 클래스 (`erp/backend/app/schemas.py`)

**관련 용어**: [[#Pydantic]], [[#UtcDatetime]]

### 순환 BOM (cyclic BOM)

> [!info] A↔B 가 서로 포함하는 잘못된 관계 — BOM 전개 시 최대 깊이 10 으로 무한루프 방지 (`erp/backend/app/routers/bom.py`)

### 시드 (seed)

> [!info] DB 에 초기 데이터를 집어넣는 스크립트 — [[#bootstrap_db.py --all]] 안에서 호출됨

### 시리얼 (serial_no)

> [!info] ItemCode 의 3번째 파트 — `process_type` 카테고리 내부에서 모델 무관 전역 유일

예: AR 카테고리에서 `3-AR-0335`, `4-AR-0336` 처럼 모델이 달라도 시리얼은 겹치지 않게 매김 (commit 188582d).

**사용처**: `erp/backend/app/utils/item_code.py` (`next_serial_no`)

**관련 용어**: [[#ItemCode]]

### SR-yyyymmdd-...

> [!info] StockRequest 의 `request_code` 형식

```
SR-{YYYYMMDD}-{HHMMSS}-{XXXXXXXX}
```

- 마지막 8자리: hex 랜덤 (32비트 엔트로피)
- 충돌 확률 약 1/4,294,967,296
- 충돌 시 라우터가 1회 재시도

예: `SR-20260520-052533-1A2B3C4D` (commit 21e98cd 참조: 5월 불량처리 승인 기록 1건의 코드 패턴).

**사용처**: `erp/backend/app/services/stock_requests.py:91-98` (`_generate_request_code`)

**관련 용어**: [[#StockRequest 결재 요청]], [[#request_code]]

### StockRequest (결재 요청)

> [!info] 입출고 결재 요청 — 창고 재고가 움직이는 작업은 승인 후에만 실재고 반영

**상태 흐름**: `draft → submitted → reserved → completed` (또는 `rejected` / `cancelled` / `failed_approval`)

**타입(`StockRequestTypeEnum`)**: `raw_receive` / `raw_ship` / `warehouse_to_dept` / `dept_to_warehouse` / `dept_internal` / `mark_defective_wh` / `mark_defective_prod` / `supplier_return` / `package_out` / `manual_adjustment`(낱개 IO)

**사용처**: `erp/backend/app/models.py`, `erp/backend/app/routers/stock_requests.py`

**관련 용어**: [[#SR-yyyymmdd-...]], [[#warehouse_role]], [[#department_role]]

## ㅇ

### 엔드포인트 (endpoint)

> [!info] API 경로 하나

예: `GET /api/items` 는 품목 목록을 돌려주는 엔드포인트. DEXCOWIN MES 에는 약 70개 엔드포인트가 있다 (queue/alerts/counts/loss/ship_packages 제거 후 기준).

**관련 용어**: [[#라우터 router]]

### 예약 수량 (pending_quantity)

> [!info] 큐 배치가 출고 예정으로 찜해둔 창고 수량

확정되면 실제 재고에서 차감되고 pending 은 0으로. 취소되면 pending 만 해제. DB-level constraint 로 `warehouse_qty >= pending_quantity` 보장.

**사용처**: `erp/backend/app/models.py` (`Inventory.pending_quantity`)

**관련 용어**: [[#가용 수량 available]]

### ItemCode

> [!info] 품목 고유 식별자 (구 ErpCode) — DB 컬럼 `items.item_code` (UNIQUE)

> [!quote] commit f1ff96c
> "chore: items.item_code 통합 + ErpCode → ItemCode 도메인 rename"

**4파트 구조**: `{model_symbol}-{process_type}-{serial:04d}[-{option}]`

| 예시 | 의미 |
|---|---|
| `346-AR-0001` | DX3000/ADX4000W/ADX6000 공용 AR 자재 1번 |
| `3-PA-0012-BG` | DX3000 전용 완제품, 옵션 BG |
| `34678-TR-0023` | 5개 모델 공용 튜브 |

> [!warning] legacy 식별자 보존
> 도메인 코드 안의 `erp_code` 변수/함수명은 일부 보존됨 (commit 214020e "도메인 erp_code 식별자는 보존"). 함부로 rename 금지.

**사용처**: `erp/backend/app/utils/item_code.py`, `erp/backend/app/models.py` (`Item.item_code`)

**관련 용어**: [[#model_symbol]], [[#공정 코드 process_type_code]], [[#시리얼 serial_no]]

### Item (품목)

> [!info] 관리 대상이 되는 자재/반제품/완제품

각 품목은 고유 `item_id` (UUID) + `item_code` ([[#ItemCode]]) 를 갖는다. `items` 테이블에 저장.

**사용처**: `erp/backend/app/models.py` (`Item`)

**관련 용어**: [[#ItemCode]], [[#Inventory 재고]]

### Inventory (재고)

> [!info] 한 품목의 전사 재고 요약

`items` 와 1:1. 컬럼:
- `quantity` — 총 보유 (창고 + 부서 합)
- `warehouse_qty` — 창고 보관량
- `pending_quantity` — 예약분 ([[#예약 수량 pending_quantity]])

부서별 분포는 [[#InventoryLocation 위치 재고]] 에서 본다.

**사용처**: `erp/backend/app/models.py` (`Inventory`)

### InventoryLocation (위치 재고)

> [!info] 부서 × 상태 단위 재고 분포

`(item_id, department, status)` 당 1행. `status` 는 `PRODUCTION` (정상) 또는 `DEFECTIVE` (격리).

```python
class LocationStatusEnum(str, enum.Enum):
    PRODUCTION = "PRODUCTION"
    DEFECTIVE = "DEFECTIVE"
```

> [!info] 핵심 설계
> 동일 부서 안에서 정상/불량 수량이 같은 행에 합쳐지지 않고, **상태별로 별도 행**으로 쌓인다. 격리는 즉시 / 정상 복귀도 즉시 (결재 X).

**사용처**: `erp/backend/app/models.py:211-245`

**관련 용어**: [[#MARK_DEFECTIVE]], [[#불량 처리 허브]], [[#재고 3계층]]

### 재고 3계층

> [!info] DEXCOWIN MES 의 핵심 설계

한 품목은 3가지 장소에 동시에 존재할 수 있다:

| 계층 | 컬럼 | 의미 |
|---|---|---|
| 창고 재고 | `Inventory.warehouse_qty` | 중앙 창고에 있는 수량 |
| 부서 정상 | `InventoryLocation.quantity (status=PRODUCTION)` | 각 부서 정상 풀 |
| 부서 불량 | `InventoryLocation.quantity (status=DEFECTIVE)` | 각 부서 격리 풀 |

`Inventory.quantity = warehouse_qty + Σ(InventoryLocation.quantity)` 항등식이 항상 성립해야 한다.

**사용처**: `erp/backend/app/services/stock_math.py`, `erp/backend/app/services/integrity.py`

### 트랜잭션 타입 한 장 표 (TransactionTypeEnum 11종)

> [!info] 현재 사용 중인 모든 거래 타입

| 타입 | 의미 | 결재 | 비고 |
|---|---|---|---|
| `RECEIVE` | 외부 → 창고 입고 | ○ | `raw_receive` 와 연결 |
| `PRODUCE` | 생산 (완제품 IN) | ○ | 확정 시 [[#백플러시 BACKFLUSH]] 자동 발생 |
| `SHIP` | 창고 → 외부 출고 | ○ | `raw_ship`, `package_out` |
| `ADJUST` | 수량 보정 | ○ | 낱개 IO + 부서 `correction` 매핑 |
| `BACKFLUSH` | BOM 자재 자동 차감 | (자동) | `PRODUCE` 확정 시 따라옴 |
| `DISASSEMBLE` | 분해 (PA/PF → 자재) | ○ | 본인 소멸 + 자식 입고/폐기 |
| `TRANSFER_TO_PROD` | 창고 → 부서 | ○ | `warehouse_to_dept` |
| `TRANSFER_TO_WH` | 부서 → 창고 | ○ | `dept_to_warehouse` |
| `TRANSFER_DEPT` | 부서 → 부서 | ○ | `dept_internal` |
| `MARK_DEFECTIVE` | 정상 ↔ 격리 | × (즉시) | `InventoryLocation.status` 토글 |
| `SUPPLIER_RETURN` | 공급처 반품 | ○ | R 정상/격리 모두 가능 |

> [!warning] 제거된 타입
> `LOSS` / `RETURN` / `RESERVE` / `RESERVE_RELEASE` / `SCRAP`(일부) — commit b792f7a 에서 죽은 거래 타입 5종 제거. 옛 문서에서 보이면 무시.

**사용처**: `erp/backend/app/models.py:59-75`

### MARK_DEFECTIVE / SCRAP / SUPPLIER_RETURN / DISASSEMBLE

> [!info] 불량 처리 4대 액션 트랜잭션

| 타입 | 의미 | 결재 | 격리 경유 |
|---|---|---|---|
| `MARK_DEFECTIVE` | 정상 → 불량 (격리). 역방향 = 정상 복귀 | × | — |
| `SCRAP` | 폐기 (재고 영구 소멸, 로그는 보존) | ○ | 거치거나 R정상에서 바로 |
| `SUPPLIER_RETURN` | 공급처 반품 (R 한정) | ○ | 거치거나 R정상에서 바로 |
| `DISASSEMBLE` | PA/PF 분해. 자식 = 분해 부서 정상 재고로 즉시 입고, 폐기 자식 = SCRAP | ○ | 격리만 가능 |

> [!info] 격리 = 재작업 대기 풀 통합
> 별도 "재작업중" 상태는 만들지 않음. 격리 풀에 1년씩 묵을 수 있음.

**사용처**: `erp/backend/app/models.py:69-70`, `erp/docs/defect-handling-redesign.md` § 7

**관련 용어**: [[#InventoryLocation 위치 재고]], [[#불량 처리 허브]]

### department_role

> [!info] 부서 결재 역할

값: `none` / `primary` / `deputy`. 부서 정/부 결재자. 낱개(manual/adjust_in/adjust_out) IO 작업 승인 권한. `warehouse_role` 와 별개.

> 라인(튜브/고압/진공/튜닝/조립/출하) 자체에는 별도 부서장이 없고, 산하 6라인 결재는 향후 "생산부" 상위 부서의 정/부가 통째로 처리 예정 (`docs/defect-handling-redesign.md` § 4).
