# 용어 사전

> [!summary] 용도
> 이 ERP에서 쓰이는 모든 전문용어를 비전공자가 이해할 수 있게 풀어 설명. 모르는 말이 나오면 여기서 먼저 찾아본다.

---

## ㄱ

**가용 수량 (available)**
실제로 지금 쓸 수 있는 재고. 계산식: `창고재고 + 부서별재고 - 예약분(pending)`. 창고에 100개 있어도 이미 배치로 50개 예약돼 있으면 실제 가용은 50개.

**가상 테이블 (ORM 모델)**
DB 테이블을 파이썬 클래스로 표현한 것. `models.py`에 정의된 `Item` 클래스가 실제 `items` 테이블과 연결된다. 개발자는 SQL 대신 파이썬 코드로 DB를 다룬다.

**공정 코드 (process type)**
품목이 어느 공정 단계인지 나타내는 2자리 코드. 예: `TR`(튜브 Raw), `PA`(완제품 Assembly). 첫 글자 = 공정종류(T튜브/H고압/V진공/A조립/P완제품), 둘째 글자 = 단계(R원자재/A조립).

**가용 수량 계산식**
이번 구조에서 자주 쓰는 기준식. `available = warehouse + production - pending` 으로 이해하면 된다. 총합 점검은 `stock_math.py` 와 `integrity.py` 를 함께 본다.

**기호 (product symbol)**
제품 모델을 나타내는 숫자. 예: 1=DX3000, 2=ADX4000W, 3=ADX6000. 100개 슬롯까지 등록 가능 (`product_symbols` 테이블).

---

## ㄴ

**내보내기 (export)**
DB 데이터를 엑셀/CSV 파일로 떨궈주는 기능. `/api/items/export` 같은 엔드포인트에서 제공.

---

## ㄷ

**대기열 (queue)**
아직 확정 안 된 생산/분해/반품 배치 목록. `OPEN` 상태로 만들어놓고 수량 조정 후 `CONFIRMED`로 확정한다. 확정 전에는 DB 재고에 실제 반영 안 됨(단, Pending 예약은 걸림).

---

## ㄹ

**라우터 (router)**
API 엔드포인트를 기능별로 묶은 파일. FastAPI에서 `@router.get("/items")` 같은 데코레이터로 경로를 선언한다. `backend/app/routers/` 폴더에 14개 있음.

**라인 (queue line)**
하나의 배치 안에 포함된 개별 자재/제품 항목. 배치 = 라인들의 묶음. 각 라인은 IN(들어옴)/OUT(나감)/SCRAP(폐기)/LOSS(손실) 방향을 가진다.

---

## ㅁ

**마스터 (master data)**
잘 바뀌지 않는 기준 데이터. 품목 마스터, 직원 마스터, 코드 마스터 등. 반대말은 거래(transaction) 데이터.

**모델 (SQLAlchemy Model)**
가상 테이블 참고.

---

## ㅂ

**배치 (batch)**
한 번에 처리할 자재 묶음. 생산 배치 하나는 "완제품 1개 만들려면 자재 A 3개, B 2개, C 5개가 나간다"를 한 단위로 처리한다. 상태: `OPEN` → `CONFIRMED` or `CANCELLED`.

**백플러시 (backflush)**
"거꾸로 씻어내린다"는 뜻. 완제품 생산이 확정되면, 그 BOM에 있는 모든 자재를 자동으로 재고에서 빼는 처리. 수동으로 자재별로 출고할 필요가 없어서 편함.

**배치 타입 3가지**
- `PRODUCE` — 만들기. 자재 OUT, 완제품 IN
- `DISASSEMBLE` — 분해. 완제품 OUT, 자재 IN (되돌리기)
- `RETURN` — 반품. 받은 거 돌려보내기

**부트스트랩 (bootstrap)**
서버 시작과 별도로 DB를 준비하거나 기준 데이터를 맞추는 초기화 작업. 이번 구조에선 `backend/bootstrap_db.py` 가 이 역할을 분명하게 맡는다.

**BOM (Bill of Materials, 자재명세서)**
제품 1개 만들려면 어떤 자재가 몇 개 필요한지 기록한 표. 예: "ADX6000 1대 = 튜브 1개 + 고압보드 2개 + 조립품 1개". 재귀적으로 전개된다(튜브 1개는 또 그 하위 자재로 분해 가능).

**부서 (department)**
자재/제품이 머무는 물리적 장소. 10개: 조립/고압/진공/튜닝/튜브/AS/연구/영업/출하/기타. 창고와 별도로 관리됨.

**불량 (defective)**
쓸 수 없다고 판단된 재고. 부서별로 별도 카운팅된다(`inventory_locations.defective_quantity`).

---

## ㅅ

**스키마 (schema)**
두 가지 뜻:
1. DB 구조 (테이블, 컬럼 정의) — 예: `schema.sql`
2. API 요청/응답 형식 (Pydantic) — 예: `schemas.py`의 `ItemCreate`, `ItemResponse`

**순환 BOM (cyclic BOM)**
A가 B를 포함하고 B가 다시 A를 포함하는 잘못된 관계. BOM 전개 시 최대 깊이 10으로 제한해서 무한루프 방지.

**시드 (seed)**
DB에 초기 데이터를 집어넣는 스크립트. `seed_bom.py`, `seed_employees.py` 등. 개발/테스트 용.

**시리얼 (serial)**
ERP 코드의 3번째 파트. 같은 `기호+공정` 조합 안에서 1부터 순차 부여. 0001, 0002, ...

---

## ㅇ

**엔드포인트 (endpoint)**
API 경로 하나. 예: `GET /api/items` 는 품목 목록을 돌려주는 엔드포인트. 이 ERP에는 약 70개 엔드포인트가 있다.

**예약 수량 (pending_quantity)**
배치가 `OPEN` 상태에서 출고 예정으로 찜해둔 수량. 확정되면 실제 재고에서 차감되고 pending은 0으로. 취소되면 pending만 해제.

**ERP 코드 (ERP Code)**
품목마다 부여되는 고유 식별자. 4파트 구조: `{기호}-{공정}-{시리얼}-{옵션}`. 예: `346-AR-0001`, `3-PA-0012-BG`. 자세한 건 `backend/app/utils/erp_code.py` 참고.

**모바일 wizard**
모바일 화면에서 입출고를 단계별로 진행하는 흐름. `frontend/app/legacy/_components/mobile/io/` 아래에서 본다.

---

## ㅈ

**재고 3계층**
이 ERP의 핵심 설계. 하나의 품목은 3가지 장소에 동시에 존재할 수 있다:
1. **창고 재고** (`warehouse_qty`) — 중앙 창고에 있는 수량
2. **부서 재고** (`production_quantity`) — 각 부서(조립/고압 등)에 나가있는 수량
3. **부서 불량** (`defective_quantity`) — 각 부서의 불량 수량

**거래 로그 (transaction_logs)**
모든 재고 변동을 기록하는 이력 테이블. 14종류 거래 타입 존재: RECEIVE, PRODUCE, SHIP, ADJUST, BACKFLUSH, SCRAP, LOSS, DISASSEMBLE, RETURN, RESERVE, RESERVE_RELEASE, TRANSFER_TO_PROD, TRANSFER_TO_WH, TRANSFER_DEPT, MARK_DEFECTIVE, SUPPLIER_RETURN.

**직급 (role)**
admin/manager/staff 3단계. admin은 모든 권한, manager는 일부 관리 권한, staff는 조회/기본 조작.

---

## ㅊ

**차이 (variance)**
배치 확정 시 "원래 예정된 BOM 수량 vs 실제 반영된 수량" 간 차이. `variance_logs`에 기록되어 나중에 추적 가능.

**출하 패키지 (ship package)**
출고 시 여러 제품을 한 묶음으로 관리하는 단위. 패키지에 아이템 추가 → 패키지 단위로 출고 처리.

---

## ㅌ

**트랜잭션 (transaction)**
DB 관점: 여러 SQL 작업을 한 덩어리로 묶어 "전부 성공 or 전부 실패"로 처리하는 단위. 배치 확정 시 사용 (중간에 실패하면 원상복구).

거래 관점: 재고 변동 한 건 (`transaction_logs`의 row 하나).

---

## ㅍ

**품목 (item)**
관리 대상이 되는 자재/반제품/완제품. 각각 고유 ERP 코드를 가진다. `items` 테이블에 저장.

**Pydantic**
파이썬에서 데이터 형식을 검증하는 라이브러리. FastAPI가 API 요청/응답을 검사할 때 사용. `schemas.py`가 Pydantic 클래스들의 모음.

## 영문 약어

| 약어 | 풀이 |
|------|------|
| API | Application Programming Interface. 프로그램끼리 대화하는 규약 |
| BOM | Bill of Materials. 자재명세서 |
| CRUD | Create/Read/Update/Delete. 생성/조회/수정/삭제 |
| DB | Database |
| ERP | Enterprise Resource Planning. 전사 자원 관리 |
| FK | Foreign Key. 외래키, 다른 테이블을 가리키는 값 |
| FG | Finished Good. 완제품 |
| JSON | JavaScript Object Notation. 데이터 교환 형식 |
| MES | Manufacturing Execution System. 제조 실행 시스템 |
| MOC | Map of Content. 지도 역할 문서 |
| ORM | Object-Relational Mapping. DB를 객체로 다루는 기법 |
| PK | Primary Key. 기본키, 행을 고유하게 식별 |
| RM | Raw Material. 원자재 |
| REST | Representational State Transfer. API 설계 방식 |
| SQL | Structured Query Language. DB 질의 언어 |
| UI | User Interface. 사용자 화면 |
| YAML | 사람이 읽기 쉬운 설정 파일 형식 (`.md` 상단 `---` 블록) |
