# 시나리오: 생산 배치 (PRODUCE)

> [!summary] 요약
> 완제품 1개를 만드는 과정에서 "어떤 자재가 자동으로 차감되고, 완제품 재고는 어떻게 늘어나는지" 전체 흐름. 이 ERP에서 가장 복잡하고 중요한 시나리오.

---

## 언제 이 시나리오가 발생하나

- 조립/생산팀이 특정 완제품(FG)을 만들기 시작할 때
- 하루 생산 계획을 시스템에 반영할 때
- 생산이 끝난 후 실제 투입량을 확정할 때

---

## 핵심 개념: 배치(batch)와 백플러시(backflush)

**배치** = "ADX6000 3대 만들기"라는 하나의 작업 단위.
- 이 배치 안에는 자동으로 "필요 자재 목록(OUT 라인들)"과 "생산 결과물(IN 라인)"이 들어간다.

**백플러시(backflush)** = "거꾸로 씻어내리기".
- 완제품이 생산 확정되면, 그에 필요한 자재들을 한꺼번에 재고에서 빼는 처리.
- 수동으로 자재마다 출고 안 해도 되니까 편함.

---

## 전체 흐름 한눈에

```
1. 배치 생성 (OPEN)
    └─ BOM 재귀 전개 → 하위 자재 OUT 라인 자동 생성
    └─ 각 OUT 라인의 수량만큼 pending_quantity 예약
    
2. 필요 시 수량 조정 / 라인 토글
    └─ 실제 투입량이 BOM과 다를 경우
    
3. 확정 (CONFIRMED) ⚡
    └─ 각 자재: warehouse_qty 감소 + pending 해제
    └─ 완제품: warehouse_qty 증가
    └─ transaction_logs 에 PRODUCE + BACKFLUSH 다수 기록
    └─ BOM 차이는 variance_logs 에 기록
    
   또는 취소 (CANCELLED)
    └─ pending 해제만, 실제 재고 변동 없음
```

---

## 단계별 상세

### 1. 배치 생성

**화면**: 생산 대기열 페이지 (`frontend/app/queue/`)

**API**: `POST /api/queue`

```json
{
  "batch_type": "PRODUCE",
  "parent_item_id": 1500,    // ADX6000 item_id
  "quantity": 3              // 3대 생산 예정
}
```

**내부 처리** (`services/queue.py` → `create_batch()`):

1. `queue_batches` 테이블에 새 batch 생성 (status=OPEN)
2. BOM 재귀 전개 (`services/bom.py` → `explode_bom()`)
   - ADX6000 → 튜브 1 + 고압보드 2 + 조립품 1
   - 튜브 1 → 튜브원자재 3
   - ... (깊이 10까지)
3. 전개 결과를 `queue_lines` 에 OUT 방향으로 추가
4. 완제품 자체는 IN 방향 라인으로 추가
5. 각 OUT 라인에 대해 해당 자재의 `pending_quantity += 수량` 예약

**예시 결과**:

| line | item | direction | quantity |
|------|------|-----------|----------|
| 1 | ADX6000 | IN | 3 |
| 2 | 튜브 | OUT | 3 |
| 3 | 고압보드 | OUT | 6 |
| 4 | 조립품 | OUT | 3 |
| 5 | 튜브원자재 | OUT | 9 |

### 2. 수량 조정 (선택)

**언제**: 실제 투입량이 BOM과 다를 때. 예: 고압보드 원래 6개 들어가야 하는데 실제 7개 썼을 때.

**API**:
- `PUT /api/queue/{id}/lines/{lineId}` — 수량 override
- `POST /api/queue/{id}/lines/{lineId}/toggle` — 포함/제외 전환
- `POST /api/queue/{id}/lines` — 라인 추가 (BOM에 없던 자재 투입)
- `DELETE /api/queue/{id}/lines/{lineId}` — 라인 제거

각 조정 시 해당 자재의 pending 재계산.

### 3-A. 확정 (CONFIRMED)

**API**: `POST /api/queue/{id}/confirm`

**내부 처리** (`services/queue.py` → `confirm_batch()`):

**트랜잭션 시작** (중간에 실패하면 전부 롤백)

1. 각 OUT 라인:
   - 자재의 `warehouse_qty -= line.quantity`
   - 자재의 `pending_quantity -= line.quantity` (예약 해제)
   - `transaction_logs` 에 `BACKFLUSH` 기록 (batch_id 참조)

2. 각 IN 라인 (완제품):
   - 완제품의 `warehouse_qty += line.quantity`
   - `transaction_logs` 에 `PRODUCE` 기록

3. BOM 원래 예정값과 실제 값 비교
   - 차이가 있으면 `variance_logs` 에 기록 (어떤 자재가 얼마나 덜/더 들었는지)

4. batch status = `CONFIRMED`, `confirmed_at` = now()

**트랜잭션 커밋**

### 3-B. 취소 (CANCELLED)

**API**: `POST /api/queue/{id}/cancel`

- 모든 OUT 라인의 pending 해제
- 실제 재고는 건드리지 않음
- batch status = `CANCELLED`

## 실제 예시

**상황**: ADX6000 (item_id=1500) 을 2대 생산.

**BOM**:
- ADX6000 1대 = 튜브 1 + 고압보드 2

**초기 재고**:
| 품목 | warehouse | pending |
|------|-----------|---------|
| ADX6000 | 5 | 0 |
| 튜브 | 10 | 0 |
| 고압보드 | 20 | 0 |

### Step 1: 배치 생성 후

| 품목 | warehouse | pending |
|------|-----------|---------|
| ADX6000 | 5 | 0 |
| 튜브 | 10 | 2 ⬆ |
| 고압보드 | 20 | 4 ⬆ |

(가용 수량은 튜브 8, 고압보드 16)

### Step 2: 확정 후

| 품목 | warehouse | pending |
|------|-----------|---------|
| ADX6000 | 7 ⬆ | 0 |
| 튜브 | 8 ⬇ | 0 |
| 고압보드 | 16 ⬇ | 0 |

**transaction_logs 에 추가된 행**:
1. `PRODUCE` +2 ADX6000 batch_id=123
2. `BACKFLUSH` -2 튜브 batch_id=123
3. `BACKFLUSH` -4 고압보드 batch_id=123

## 실패할 수 있는 지점

| 단계 | 실패 상황 | 대응 |
|------|---------|------|
| 1 | BOM이 등록 안 된 완제품 | 400 에러 "BOM 없음" |
| 1 | 순환 BOM | 최대 깊이 10으로 제한 |
| 1 | 가용 재고 부족 (pending 초과) | 경고 표시 (확정 불가 아님, OPEN은 가능) |
| 3-A | 확정 시점에 재고 부족 | 트랜잭션 롤백, 에러 |
| 3-A | DB 쓰기 실패 | 전체 롤백, 이전 상태 유지 |

## 확인 방법

### 배치 현황
- 화면: 생산 대기열 페이지
- API: `GET /api/queue?status=OPEN`

### 재고 반영 확인
- 화면: 재고 탭에서 해당 품목 찾기
- API: `GET /api/inventory/summary` 또는 `GET /api/items/{id}`

### 이력
- 화면: 이력 탭 → 타입 필터 `PRODUCE` / `BACKFLUSH`
- API: `GET /api/inventory/transactions?batch_id=123`

## 자주 발생하는 궁금증

**Q. OPEN 상태에서 재고 실제로 차감되나?**
아니다. `pending_quantity`만 걸림. 확정해야 실제 `warehouse_qty` 감소.

**Q. pending이 걸려있으면 다른 사람이 그 자재 쓸 수 없나?**
가용 수량(available = warehouse + production - pending)으로 계산하니, 다른 작업은 pending 뺀 나머지만 쓸 수 있다. 실질적으로 "먼저 예약한 사람이 우선" 방식.

**Q. 확정 전에 BOM이 바뀌면 어떻게 되나?**
이미 OPEN으로 만든 배치의 라인은 그대로 유지. 다시 만들어야 새 BOM 반영. 이 점이 혼란 원인이 될 수 있으니 BOM 변경 전후 타이밍 주의.

**Q. 자재가 부서 재고에 있는데 창고 재고에서만 차감되나?**
현재 설계상 기본 차감 위치는 `warehouse_qty`. 부서 재고 사용은 별도 이관 후 진행하는 구조. 상세는 `services/inventory.py` 의 `receive_confirmed()` / 역방향 로직 확인 필요.

**Q. variance_logs는 언제 봐야 하나?**
BOM과 실제 투입이 자주 다르면 → BOM이 현실과 안 맞다는 뜻. 주기적으로 variance 리포트 검토해서 BOM 업데이트.

**Q. 배치 여러 개 동시에 확정하면 충돌하나?**
같은 자재 사용 시 pending 기반으로 사전 예약했기 때문에 일반적으로 문제없음. 단, pending 예약 후 BOM이 바뀌었거나 재고 조정이 병행되면 variance 발생 가능.
