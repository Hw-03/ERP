# 시나리오: 분해(DISASSEMBLE) 및 반품(RETURN) 배치

> [!summary] 요약
> 생산을 되돌리거나(분해), 받은 물건을 돌려보내는(반품) 흐름. 둘 다 queue 배치로 처리되지만 방향과 결과가 다르다.

---

## 두 시나리오 비교

| 구분 | DISASSEMBLE (분해) | RETURN (반품) |
|------|------------------|--------------|
| 언제 | 이미 만든 완제품을 자재로 되돌릴 때 | 고객에게 판매했던 걸 돌려받거나, 작업 실패로 원상복구할 때 |
| 상위 품목 | OUT (완제품 재고 감소) | IN (반품 들어옴) |
| 하위 자재 | IN (자재 재고 증가) | IN + 누락분 LOSS |
| 대상 | BOM 직계 자식(direct_children)만 | BOM 직계 자식만 |

---

## 시나리오 A: 분해(DISASSEMBLE)

### 언제 발생하나
- 잘못 조립한 완제품을 해체할 때
- 시장에서 회수된 제품을 원자재로 돌려 재사용할 때

### 흐름

```
1. 배치 생성 (DISASSEMBLE, 완제품 N개)
    └─ 상위(완제품): OUT 라인 (재고 감소 예정)
    └─ 직계 자식 자재: IN 라인 (재고 증가 예정)
    └─ 상위의 warehouse_qty 만큼 pending 예약
    
2. 수량 조정 (필요 시)
    └─ 실제 분해 결과가 BOM과 다르면 override

3. 확정 (CONFIRMED)
    └─ 상위 warehouse_qty -= quantity
    └─ 각 자재 warehouse_qty += quantity
    └─ transaction_logs 에 DISASSEMBLE 기록
```

### 예시

**대상**: ADX6000 1대 (BOM: 튜브 1 + 고압보드 2)

**초기**:
| 품목 | warehouse |
|------|-----------|
| ADX6000 | 3 |
| 튜브 | 5 |
| 고압보드 | 10 |

**배치 생성 후 (OPEN)**:
- ADX6000 pending = 1
- 튜브/고압보드 재고 변동 없음 (IN이라 pending도 없음)

**확정 후**:
| 품목 | warehouse |
|------|-----------|
| ADX6000 | 2 ⬇ |
| 튜브 | 6 ⬆ |
| 고압보드 | 12 ⬆ |

**transaction_logs**:
1. `DISASSEMBLE` -1 ADX6000
2. `DISASSEMBLE` +1 튜브
3. `DISASSEMBLE` +2 고압보드

### 주의
- BOM 재귀 전개 안 함 (직계 자식만). 튜브 안의 원자재까지 자동 분해되지 않는다.
- 분해된 자재는 품질이 보장되지 않을 수 있으므로, 실제 운영에선 불량 처리 병행 고려.

## 시나리오 B: 반품(RETURN)

### 언제 발생하나
- 고객이 물건 돌려보냈을 때
- 작업 실패로 자재를 원래대로 되돌려야 할 때
- 분해와 비슷하지만, 상위 품목도 다시 재고로 받는 경우

### 흐름

```
1. 배치 생성 (RETURN, 완제품 N개)
    └─ 상위: IN 라인 (반품 들어옴)
    └─ 직계 자식: IN 라인 (분해돼서 같이 들어옴)
    └─ 누락된 자재가 있으면 LOSS 라인 자동 생성

2. 수량 조정 (필요 시)

3. 확정
    └─ 상위 warehouse_qty += quantity
    └─ 자식 자재 warehouse_qty += quantity (BOM 만큼)
    └─ LOSS 라인: loss_logs 에 기록
```

### 예시

**상황**: 고객이 ADX6000 2대 반품. 그런데 뜯어보니 고압보드 1개가 빠져있음(분실).

**초기**:
| 품목 | warehouse |
|------|-----------|
| ADX6000 | 0 |
| 튜브 | 0 |
| 고압보드 | 0 |

**배치 생성 후 (RETURN, 2대, 원본 BOM 기준)**:
- ADX6000 IN 2
- 튜브 IN 2 (2대 × 1개)
- 고압보드 IN 4 (2대 × 2개)

**조정**: 실제 들어온 고압보드는 3개뿐 → 해당 라인 수량 3으로 수정 + 누락 1개 LOSS 라인 추가

**확정 후**:
| 품목 | warehouse |
|------|-----------|
| ADX6000 | 2 ⬆ |
| 튜브 | 2 ⬆ |
| 고압보드 | 3 ⬆ (4가 아니라 3) |

**loss_logs**: 고압보드 1개 LOSS 기록

## 내부 로직 (services/queue.py)

두 시나리오 모두 `create_batch()` → `_seed_lines_from_bom()` 로 진입:

| batch_type | 상위 방향 | 자식 방향 | BOM 전개 |
|-----------|----------|----------|---------|
| PRODUCE | IN | OUT (재귀) | 깊이 10 |
| DISASSEMBLE | OUT | IN (직계) | 깊이 1 |
| RETURN | IN | IN + LOSS | 깊이 1 |

확정 시 `confirm_batch()` 는 각 라인의 방향에 따라:
- IN → `warehouse_qty += quantity`
- OUT → `warehouse_qty -= quantity`, pending 해제
- LOSS → `loss_logs` 기록 (재고 변동 없음)

## 실패할 수 있는 지점

| 단계 | 실패 상황 | 대응 |
|------|---------|------|
| 생성 | BOM 없는 품목 | 400 에러 |
| 생성 (DISASSEMBLE) | 분해할 완제품 재고 부족 | 경고 (확정 시 다시 체크) |
| 확정 | 확정 시점 재고 불충분 | 트랜잭션 롤백 |

## 확인 방법

### 배치 현황
- `GET /api/queue?status=CONFIRMED&batch_type=DISASSEMBLE`

### 재고 반영
- 화면 재고 탭에서 해당 품목

### 이력
- `GET /api/inventory/transactions?type=DISASSEMBLE`
- `GET /api/loss` (RETURN 시 누락분 확인)

## 자주 발생하는 궁금증

**Q. DISASSEMBLE이랑 RETURN 뭐가 다르지?**
- DISASSEMBLE: 상위 재고를 뜯어서 자재로 쪼개는 것 (상위 OUT, 자식 IN)
- RETURN: 이미 나갔던 것이 전부 돌아오는 것 (상위 IN, 자식 IN)

**Q. 분해할 때 원자재까지 다 분해되나?**
아니다. BOM 직계 자식만 1단계 전개. 필요하면 그 자재에 대해 또 DISASSEMBLE 배치 생성.

**Q. 반품 시 LOSS는 자동 계산되나?**
`create_batch` 단계에서는 BOM 기준으로 기본 IN 라인 생성. 실제 수량이 적으면 사용자가 토글/수정해서 LOSS 라인 만드는 구조. 자동 완전 처리는 아님.

**Q. 확정했다가 되돌릴 수 있나?**
직접 되돌리는 API 없음. 반대 방향 배치(예: PRODUCE 확정 → DISASSEMBLE로 되돌리기)로 수동 처리.

**Q. queue에 OPEN 상태로 오래 두면 문제있나?**
pending 예약이 계속 걸려있어 가용 재고 감소. 장기 미확정 배치는 정리 필요. cancel API로 해제.
