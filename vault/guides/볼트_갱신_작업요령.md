# 볼트 갱신 작업요령

> [!summary] 한 줄 요약
> `main`에 쌓인 코드 변경을 `vault-sync`로 가져온 뒤, 실제 원본 코드를 다시 읽고 Obsidian vault 노트를 최신화하는 반복 작업 루틴이다.

---

## 0. 이 문서가 필요한 순간

다음 상황이면 이 문서를 따라간다.

- `main`에서 기능 개발, UX 수정, 데이터 작업, 리팩터링이 며칠 이상 쌓였다.
- `vault-sync`가 `main`보다 뒤처져 있다.
- Obsidian 노트가 현재 코드 구조와 어긋나기 시작했다.
- 새 담당자에게 넘기기 전에 vault를 다시 읽기 좋게 만들고 싶다.

> [!warning] 핵심 원칙
> `main`은 항상 코드만 가진다. `vault-sync`만 같은 코드 + `vault/`를 가진다.

---

## 1. Vault 폴더 구조

```
vault/
├── ERP/                  ← 프로젝트 파일 구조 미러 노트 모음
│   ├── 📁_ERP.md         ← 루트 진입점
│   ├── backend/          ← backend/ 미러
│   ├── frontend/         ← frontend/ 미러
│   ├── docs/             ← docs/ 미러
│   ├── scripts/          ← scripts/ 미러
│   └── ...
└── guides/               ← 사람이 읽는 가이드 (이 파일 있는 곳)
    ├── 처음_읽는_사람.md
    ├── 전체_컨텍스트.md
    ├── 위험지대_지도.md
    ├── 볼트_갱신_작업요령.md  ← 이 파일
    └── 용어사전.md
```

이전 구조와 비교:
- `vault/ERP/` ← 이전: `vault/backend/`, `vault/frontend/` 등 루트 직접
- `vault/guides/` ← 이전: `vault/_vault/guides/`

## 2. 작업 모드 선택

### 권장: 별도 worktree 사용

실제 앱 데모나 개발 중인 `C:\ERP` 작업트리를 건드리고 싶지 않으면 별도 worktree가 가장 안전하다.

```powershell
cd C:\ERP
git fetch origin main vault-sync
git worktree add C:\ERP-vault-sync vault-sync
cd C:\ERP-vault-sync
```

장점:

- `C:\ERP`는 `main` 데모용으로 그대로 둘 수 있다.
- `vault-sync`에서 vault를 정리해도 앱 실행 작업트리와 섞이지 않는다.
- 브랜치 전환 중 `vault/` 삭제 충돌을 피하기 쉽다.

### 대안: 현재 작업트리에서 직접 작업

이미 `C:\ERP`가 `vault-sync`이고 앱 코드 변경이 없다면 그대로 작업해도 된다.

작업 전 확인:

```powershell
git status --short --branch
git branch --show-current
```

## 3. main → vault-sync 동기화

### 3.1 최신 상태 받기

```powershell
git fetch origin main vault-sync
git checkout vault-sync
git pull --ff-only origin vault-sync
```

### 3.2 머지 전 기준 저장

```powershell
git rev-parse HEAD > $env:TEMP\vault_sync_backup_ref.txt
git rev-parse origin/main > $env:TEMP\vault_sync_main_ref.txt
```

### 3.3 main 병합

```powershell
git merge origin/main --no-ff -m "Merge main into vault-sync (vault/ 보존)"
```

충돌 처리 원칙:

| 충돌 위치 | 선택 기준 |
|---|---|
| 앱 코드, docs, scripts, data | `origin/main` 쪽 채택 |
| `vault/` | `vault-sync` 쪽 유지 |
| `.obsidian/` | 개인 설정이면 보통 커밋 제외 |

### 3.4 코드 diff 확인

```powershell
git diff --exit-code origin/main HEAD -- ':!vault/'
```

정상 결과: 아무 출력 없음 + exit code 0. 즉, `main`과 `vault-sync`의 코드 부분이 완전히 같음.

## 4. main 변경분 파악

### 4.1 변경 커밋 보기

```powershell
$backup = Get-Content $env:TEMP\vault_sync_backup_ref.txt
git log --oneline --decorate $backup..origin/main
```

분류 기준:

| 분류 | vault 갱신 방향 |
|---|---|
| 구조 변경 (폴더 이동, 라우터 분리) | `ERP/` 허브 노트 우선 갱신 |
| 도메인 변경 (재고 계산, 승인 상태) | 시나리오/위험지대/용어사전 갱신 |
| UX 변경 (헤더, 입출고 화면) | 화면 노트와 전체_컨텍스트 갱신 |
| API 변경 (router/schema) | backend/frontend 양쪽 미러 노트 갱신 |

### 4.2 변경 파일 목록 보기

```powershell
$backup = Get-Content $env:TEMP\vault_sync_backup_ref.txt
git diff --name-status $backup..origin/main -- ':!vault/'
```

## 5. Vault 업데이트 규칙

### 5.1 파일 미러 노트

원본 파일의 vault 노트 경로:

```text
vault/ERP/<source_path>.md
```

예:

| 원본 | vault 노트 |
|---|---|
| `backend/app/models.py` | `vault/ERP/backend/app/models.py.md` |
| `frontend/lib/api.ts` | `vault/ERP/frontend/lib/api.ts.md` |
| `docs/OPERATIONS.md` | `vault/ERP/docs/OPERATIONS.md.md` |

### 5.2 폴더 허브 노트

폴더 구조가 바뀌면 `📁_*.md` 허브를 먼저 고친다.

예:

- `vault/ERP/backend/app/routers/📁_routers.md`
- `vault/ERP/frontend/app/legacy/_components/📁__components.md`
- `vault/ERP/frontend/lib/📁_lib.md`

### 5.3 수동 가이드 노트

도메인 흐름이 바뀌면 가이드도 갱신한다. 우선순위:

1. `guides/전체_컨텍스트.md`
2. `guides/처음_읽는_사람.md`
3. `guides/용어사전.md`
4. `guides/위험지대_지도.md`

### 5.4 삭제/이동된 원본

원본 파일이 삭제됐으면 `git mv` 를 사용해 미러 노트도 이동/삭제한다.

예:

```powershell
git mv vault/ERP/backend/app/routers/old_router.py.md vault/ERP/backend/app/routers/new_router.py.md
```

### 5.5 새 vault 파일 추가

새 vault 파일이 `.gitignore` 에 막히면 강제 추가가 필요하다.

```powershell
git add -f vault/guides/새_노트.md
```

## 6. 갱신 후 검증

### 6.1 코드 부분 diff

```powershell
git diff --exit-code origin/main HEAD -- ':!vault/'
```

이 명령이 비어 있어야 한다.

### 6.2 vault 파일 수

```powershell
$vaultCount = (git ls-tree -r HEAD --name-only -- vault | Measure-Object).Count
$vaultCount
```

0보다 커야 한다.

### 6.3 핵심 링크 확인

- `vault/ERP.md` — vault 루트 진입점
- `vault/guides/처음_읽는_사람.md`
- `vault/guides/전체_컨텍스트.md`
- `vault/guides/볼트_갱신_작업요령.md` (이 파일)
- `vault/guides/위험지대_지도.md`
- `vault/guides/용어사전.md`

## 7. 커밋/푸시 루틴

### 7.1 스테이징

의도한 vault 변경만 올린다.

```powershell
git add vault/guides/볼트_갱신_작업요령.md
git add vault/ERP/backend/app/models.py.md
# 새 파일이 .gitignore에 막히면
git add -f vault/guides/새_노트.md
```

### 7.2 커밋 메시지 예시

```text
docs(vault): refresh Obsidian handoff context
docs(vault): add vault update workflow
docs(vault): sync notes with latest main
```

### 7.3 push

```powershell
git push origin vault-sync
```

## 9. 실수 방지 체크리스트

작업 전:

- [ ] 현재 브랜치가 `vault-sync`인지 확인했다.
- [ ] `origin/main`, `origin/vault-sync`를 fetch했다.
- [ ] 머지 전 `vault-sync` 기준 SHA를 저장했다.

동기화 중:

- [ ] 코드 충돌은 `main` 쪽 기준으로 해결했다.
- [ ] `vault/` 충돌은 `vault-sync` 쪽 기준으로 보존했다.
- [ ] 머지 후 `vault/` 파일 수가 0보다 큰지 확인했다.
- [ ] `origin/main` 대비 코드 diff가 비어 있는지 확인했다.

노트 갱신 중:

- [ ] 원본 파일을 먼저 읽었다.
- [ ] 파일 미러 노트 경로가 `vault/ERP/` 아래인지 확인했다. (이전: `vault/backend/` 등 직접)
- [ ] 가이드 링크가 `[[guides/...]]` 또는 `[[ERP/...]]` 형식인지 확인했다.
- [ ] 폴더 구조가 바뀐 경우 허브 노트를 갱신했다.
- [ ] 삭제된 원본의 stale 미러 노트를 정리했다.
- [ ] `.obsidian` 개인 설정을 커밋에 섞지 않았다.

마무리:

- [ ] 새 vault 파일은 `git add -f`로 누락 없이 올렸다.
- [ ] `git diff --exit-code origin/main HEAD -- ':!vault/'`가 통과했다.
- [ ] 완료 보고에 머지 SHA, vault 파일 수, 코드 diff 결과, push 결과를 적었다.

---

## 10. 가장 중요한 결론

> [!quote]
> `vault-sync` 갱신은 단순 문서 작업이 아니다. "main의 실제 코드가 지금 어떤 시스템이 되었는지"를 다시 읽고, 다음 사람이 길을 잃지 않도록 지도와 표지판을 다시 세우는 작업이다.
