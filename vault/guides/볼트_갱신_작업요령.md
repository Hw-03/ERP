# 볼트 갱신 작업요령

main에서 코드 작업이 많이 쌓인 뒤 vault-sync를 다시 갱신할 때 쓰는 절차입니다.

## 원칙

- main에는 `vault/`를 넣지 않습니다.
- vault-sync만 `vault/`를 포함합니다.
- 갱신 전후로 vault 밖 코드 diff가 없어야 합니다.
- 기존 설명은 코드와 맞을 때만 살립니다.
- 캐시, 빌드 결과물, 의존성 내부 파일은 개별 설명을 만들지 않습니다.
- 코드가 main 보다 앞선 PR(예: Phase1 schemas 분리)은 병합된 뒤에만 vault에 반영합니다. 아직 main에 없는 구조를 미리 적지 않습니다.

## 작업 순서

1. `git fetch origin main vault-sync`
2. `git checkout vault-sync`
3. `git pull --ff-only origin vault-sync`
4. `git merge origin/main --no-ff -m "Merge main into vault-sync (vault/ 보존)"`
5. `vault/`가 유지됐는지 확인합니다.
6. 실제 코드를 읽고 `vault/ERP`, `vault/guides`를 갱신합니다.
7. `git diff --exit-code origin/main HEAD -- ':!vault/'`로 코드 diff가 없는지 확인합니다.
8. `git add -f vault/ERP vault/guides`만 staging합니다.

## 완료 보고

- 브랜치
- 기준 main 커밋
- vault-sync 커밋
- 변경한 vault 구조
- 보존한 기존 설명
- 삭제한 캐시/산출물 설명
- 검증 결과
- push 여부
