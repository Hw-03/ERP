# DEXCOWIN MES 전체 컨텍스트

> [!summary] 한 줄 요약
> DEXCOWIN MES를 처음 이어받는 사람이 바로 판단할 수 있도록 프로젝트의 현재 구조, 핵심 도메인, 위험 지점, 작업 순서를 한곳에 모은 컨텍스트다.

---

## 1. 프로젝트의 본질

DEXCOWIN MES는 공장 자재가 어디에, 얼마나, 어떤 상태로 있는지 추적하기 위해 만든 내부 운영 시스템이다.

핵심 질문은 네 가지다.

1. 어떤 품목이 있는가?
2. 그 품목은 창고, 생산부서, 불량 격리 중 어디에 있는가?
3. 누가 어떤 요청을 냈고, 누가 승인했는가?
4. 실제 재고 변화가 이력과 감사 로그에 남았는가?

공식 이름은 `DEXCOWIN MES`다. `ERP`, `X-Ray ERP`, `erp/`, `erp_code`, `xray-erp` 같은 이름은 과거 호칭이나 legacy 식별자로 남아 있다.

사용자 노출 문구와 문서에서는 `DEXCOWIN MES`를 쓰고, 폴더명과 DB 컬럼 같은 legacy 식별자는 명시 지시 없이 바꾸지 않는다.

---

## 2. 진실의 순서

정보가 충돌할 때는 다음 순서를 따른다.

1. 실제 코드: `backend/`, `frontend/`, `scripts/`
2. 작업 의도 문서: `plans/` 또는 관련 변경 계획
3. 공식 문서: `docs/`
4. 프로젝트 규칙: `CLAUDE.md`
5. Vault 노트: `vault/`
6. 변경 이력: `git log`

Vault는 큰 그림을 빨리 잡는 데 좋지만, 구현 판단은 반드시 원본 코드와 테스트로 되돌아가야 한다.

## 3. 브랜치 정책

| 브랜치 | 역할 |
|---|---|
| `main` | 코드만 유지. `vault/` 없음 |
| `vault-sync` | `main`과 같은 코드 + `vault/` 인수인계 문서 |

`main`에 `vault/`를 올리면 안 된다.

검증 기준:

```powershell
git diff --exit-code origin/main HEAD -- ':!vault/'
```

위 명령이 비어 있어야 정책에 맞다.

## 5. 핵심 도메인 모델

### 5.1 품목과 코드

품목 마스터는 MES의 출발점이다.

- `Item`: 품목의 기본 정보
- `ItemCode`: 기존 `ErpCode`에서 이름이 정리된 품목 코드 도메인
- 코드 형식: `{model_symbol}-{process_type}-{serial:04d}[-{option}]`
- 관련 코드: `backend/app/services/codes.py`, `backend/app/routers/codes.py`, `backend/app/routers/items.py`

### 5.2 재고 3-bucket

재고는 단순 총량이 아니라 위치와 상태를 함께 본다.

| 버킷 | 의미 |
|---|---|
| `warehouse` | 창고 재고 |
| `production` | 부서/생산 재고 |
| `defective` | 불량 격리 재고 |

핵심 불변식:

```text
Inventory.quantity == warehouse_qty + sum(InventoryLocation.quantity)
```

주요 코드:

- `backend/app/services/inventory.py`: 재고 변경의 핵심 진입점
- `backend/app/services/stock_math.py`: 사용 가능 수량, 승인 대기 수량 등 계산 기준
- `backend/app/services/integrity.py`: 재고 불변식 점검/복구
- `backend/app/routers/inventory/*`: 재고 조회, 입고, 이동, 불량, 공급처 반품, 이력

### 5.3 요청과 승인

실제 재고 이동은 많은 경우 요청과 승인으로 나뉜다.

- `StockRequest`: 작업자가 만든 입출고/이동 요청
- 요청 ID: `SR-yyyymmdd-...` 형태
- 상태 흐름: draft, submitted, reserved, rejected, cancelled, completed 등
- 창고 또는 부서 승인자가 PIN으로 승인/반려/취소한다.

주요 코드:

- `backend/app/routers/stock_requests.py`
- `backend/app/services/stock_requests.py`
- `frontend/lib/api/stock-requests.ts`

### 5.4 감사와 이력

재고가 바뀌면 이력이 남아야 한다.

- `TransactionLog`: 재고 변동의 감사 원장
- `AdminAuditLog`: 마스터/설정 변경 감사 로그
- 월별 CSV 미러: 외부 심사 대응용

주요 코드:

- `backend/app/services/audit.py`
- `backend/app/routers/inventory/transactions.py`
- `frontend/app/legacy/_components/DesktopHistoryView.tsx`

### 5.5 BOM과 생산

BOM은 상위 품목을 생산할 때 어떤 하위 자재가 얼마나 필요한지 정의한다.

주요 흐름:

1. 상위 품목 선택
2. BOM 전개
3. 하위 자재 사용량 계산
4. Backflush로 부품 차감
5. 완제품 재고 증가
6. 생산/차감 이력 기록

주요 코드:

- `backend/app/routers/bom.py`
- `backend/app/services/bom.py`
- `backend/app/routers/production.py`

### 5.6 불량 처리

격리(Quarantine) → 처리(scrap/return/disassemble) 2단계.

- 격리: `POST /defects/quarantine` — InventoryLocation을 DEFECTIVE 상태로
- 처리: StockRequest(mark_defective_wh/prod) 결재 워크플로
- 관련: [[ERP/backend/app/routers/defects.py]], [[ERP/frontend/app/legacy/_components/_defect_hub/📁__defect_hub]]

원칙:

- 격리는 즉시 가능 (결재 없음)
- 이후 처리는 정상 복귀, 폐기, 공급처 반품, 분해로 나뉜다.
- 폐기/반품/분해는 결재가 필요하다.
- 불량 버킷과 정상 재고를 섞으면 안 된다.
- 라인(튜브/고압/진공/튜닝/조립/출하) 자체에는 결재자가 없다. 생산부장(정/부) 또는 창고장(정/부)이 결재한다.

주요 코드:

- `backend/app/routers/defects.py`
- `backend/app/routers/inventory/defective.py`
- `backend/app/services/dept_adjustment.py`
- `docs/defect-handling-redesign.md`
- `frontend/app/legacy/_components/_defect_hub/`

## 9. 업무 흐름 시나리오 4가지

> [!example]- 재고 입출고 흐름
> **한 줄 요약**: 버튼 클릭 → API → 서비스 → DB → TransactionLog
>
> **두 갈래 경로**
> - 갈래 A (단건): `api/inventory/receive` → `routers/inventory/receive.py` → `services/inventory.py`
> - 갈래 B (위자드 묶음): `api/io/submit` → `routers/io.py` → `services/io.py` → `services/inventory.py` (내부 동일)
>
> **핵심 파일**
> - 입출고 패키지: `backend/app/routers/inventory/`
> - 입출고 v2: `backend/app/routers/io.py`
> - 재고 서비스: `backend/app/services/inventory.py`
> - 입출고 내역 화면: `frontend/app/legacy/_components/DesktopHistoryView.tsx`
>
> **주의사항**
> - `_sync_total()` 우회 = 불변식 깨짐
> - 수량 변동과 TransactionLog INSERT는 한 트랜잭션
> - 새 응답 스키마에는 `UtcDatetime` 쓸 것 (9시간 오차 방지)
> - 살아 있는 거래 타입 11종 안에서 해결 (`SCRAP`/`LOSS`/`RETURN`/`RESERVE`/`RESERVE_RELEASE` 제거됨)

> [!example]- 품목 등록 흐름
> **한 줄 요약**: 관리자 폼 → `POST /api/items` → ItemCode 4파트 자동 생성 → DB INSERT
>
> **ItemCode 4파트 구조**: `{model_symbol}-{process_type}-{serial:04d}[-{option}]`
>
> **코드 생성 순서**
> 1. `next_serial_no(model_symbol, process_type, db)` — 같은 카테고리 최대 serial +1
> 2. `make_item_code(symbol, process_type, serial, option)` — 문자열 조립
> 3. `validate_code(db, code)` — 마스터 테이블 존재 여부 검증
>
> **핵심 파일**
> - 라우터: `backend/app/routers/items.py`
> - 코드 유틸: `backend/app/utils/item_code.py`
> - 코드 서비스: `backend/app/services/codes.py`
>
> **주의사항**
> - PA / AA 계열은 단일 슬롯 심볼만 허용
> - serial_no 경합: `item_code` UNIQUE 제약이 최종 방어선
> - 등록 직후 `inventory` 레코드 0수량으로 자동 생성

> [!example]- 생산 배치 흐름
> **한 줄 요약**: `POST /api/production/receipt` → BOM 전개 → Backflush → PRODUCE 기록
>
> **Backflush**: 완제품 생산 확정 시 BOM 자재 자동 차감. `explode_bom()` 이 리프까지 전개.
>
> **PA·PF 계열 생산 순서**
> 1. `explode_bom(item_id, quantity)` — 리프 자재 목록 반환
> 2. `merge_requirements()` — 중복 자재 합산
> 3. `lock_inventories()` — FOR UPDATE 행 잠금
> 4. 사전 재고 검사 — 부족 시 422
> 5. `consume_warehouse()` per 자재 — BACKFLUSH 트랜잭션
> 6. `receive_confirmed()` — PRODUCE 트랜잭션
> 7. `db.commit()` — 한 번만
>
> **R 계열**: BOM 없음 → `adjust_in`/`adjust_out` 낱개 버튼만
>
> **핵심 파일**
> - 라우터: `backend/app/routers/production.py`
> - BOM 서비스: `backend/app/services/bom.py`
> - 재고 서비스: `backend/app/services/inventory.py`

> [!example]- 분해/반품 흐름
> **한 줄 요약**: 격리(MARK_DEFECTIVE) → 결재 → 폐기(SCRAP)/반품(SUPPLIER_RETURN)/분해(DISASSEMBLE)
>
> **2단계 리듬**: 격리는 즉시(결재 없음), 처리는 결재 필요
>
> **4가지 결말**
> - 정상 복귀: 즉시, 결재 없음, `MARK_DEFECTIVE` 역방향
> - 폐기: 결재 필요, `SCRAP`, 총수량 감소
> - 공급처 반품: 결재 필요, `SUPPLIER_RETURN`, R만 가능
> - 분해: 결재 필요, `DISASSEMBLE`, PA·PF만, 자식은 분해한 부서 정상 재고로 즉시 입고
>
> **R 계열 특이사항**: 격리 없이 정상에서 바로 폐기/반품 가능 (R 바로 처리)
>
> **핵심 파일**
> - 불량 처리 허브: `frontend/app/legacy/_components/_defect_hub/`
> - 불량 API: `backend/app/routers/defects.py`
> - 부서 조정 서비스: `backend/app/services/dept_adjustment.py`
> - 설계 합의 문서: `docs/defect-handling-redesign.md`
>
> **결재 권한**
> - 생산부장(정/부) OR 창고장(정/부)
> - 라인(튜브/고압/진공/튜닝/조립/출하) 자체에는 결재자 없음

## 11. 핵심 10문장

1. 이 시스템은 DEXCOWIN MES이고, 공장 자재의 위치/수량/상태/이력을 추적한다.
2. `main`은 코드만, `vault-sync`는 같은 코드와 `vault/`를 함께 가진다.
3. Vault는 인수인계 지도이지 실행 코드가 아니다.
4. 코드와 문서가 다르면 코드를 믿는다.
5. 실제 운영 프론트엔드는 `frontend/app/legacy/`다.
6. 재고는 warehouse, production, defective 3-bucket으로 봐야 한다.
7. 재고 수량 계산, 요청 상태 전이, 감사 로그는 최고 위험 지대다.
8. 불량 처리는 격리(즉시) → 처리(결재) 2단계다.
9. BOM/패키지 기능은 데이터 완성도와 함께 판단해야 한다.
10. 첫 수정 전에는 화면에서 DB까지 이어지는 호출 동선을 반드시 한 번 추적한다.
