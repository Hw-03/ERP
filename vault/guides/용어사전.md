# 용어사전

## DEXCOWIN MES

회사의 품목, 재고, 입출고, 승인, BOM, 불량 처리를 관리하는 시스템입니다. 예전에는 ERP라는 이름도 섞여 있었지만 사용자에게 보이는 공식 이름은 DEXCOWIN MES입니다.

## 품목 / Item

관리 대상이 되는 자재, 중간품, 완성품입니다. DB에서는 `Item` 모델이 기준입니다.

## ERP 코드 / erp_code

회사 품목을 식별하기 위한 코드입니다. 화면과 엑셀, DB를 연결하는 중요한 키입니다.

## 창고 재고 / 사용 가능 재고

아직 부서 작업으로 나가지 않은 재고입니다. 요청이 승인 대기 중이면 실제 보유 수량과 바로 사용할 수 있는 수량이 다를 수 있습니다.

## 승인 대기 수량

요청은 제출됐지만 아직 창고가 승인하지 않은 수량입니다. 예전 표현인 “점유중”보다 현장 업무에 맞는 표현입니다.

## StockRequest

입고, 출고, 부서 이동 같은 작업을 바로 처리하지 않고 요청과 승인 흐름으로 관리하기 위한 DB 모델입니다.

## TransactionLog

재고가 실제로 움직였다는 기록입니다. 나중에 왜 수량이 바뀌었는지 추적하는 감사 로그 성격이 있습니다.

## AdminAuditLog

관리자가 품목, 직원, 설정 같은 중요한 정보를 바꿨을 때 남기는 기록입니다.

## BOM

상위 품목을 만들기 위해 필요한 하위 품목 구성입니다. 나중에 묶음 출고나 생산 차감과 연결됩니다.

## Backflush

생산 완료를 등록할 때 BOM 기준으로 필요한 하위 자재를 자동 차감하는 방식입니다.

## PIN

직원 본인 확인용 번호입니다. 입출고 요청, 승인, 취소 같은 작업의 책임자 흐름과 연결됩니다.

## legacy

프론트 폴더명에는 legacy가 남아 있지만, 현재 실제 운영 화면이 이 경로에 있습니다. 과거 코드라는 뜻으로 오해하면 안 됩니다.

## vault-sync

코드와 Obsidian 설명을 함께 보관하는 브랜치입니다. main은 코드만 유지합니다.

## _attic

과거 자료 보관소입니다. 현재 운영 기준은 아니며, 참고용으로만 봅니다.

## 창고 지도

창고 평면도 위에 보관 구조(앵글)와 박스를 그려 "무엇이 어디에 얼마나 있는지"를 눈으로 보는 화면입니다. 박스에 담긴 품목 수량을 창고 재고와 대조해 실제와 맞는지 확인합니다. 모델: [[ERP/backend/app/models/warehouse.py]] (`WarehouseAngle`/`WarehouseBox`/`WarehouseBoxItem`).

## 앵글 / Angle

창고 평면도의 한 블록(랙) 하나입니다. 줄 수·층 수·자리 수로 칸이 나뉩니다. 관리 화면에서 추가·수정·삭제(구조 편집)합니다. 모델: `WarehouseAngle` ([[ERP/backend/app/models/warehouse.py]]).

## 박스 / Box

앵글 안의 한 자리(좌표: 앵글·줄·층·자리)에 놓인 박스 1개입니다. 크기는 대/중/소로 나뉘고, 빈 자리는 따로 기록하지 않고 박스가 놓일 때만 좌표로 표현됩니다. 모델: `WarehouseBox` ([[ERP/backend/app/models/warehouse.py]]).

## warehouse_qty

창고에만 있는(아직 부서로 나가지 않은) 재고 수량입니다. 창고 지도의 박스 수량은 이 `warehouse_qty`를 기준으로 대조합니다. 부서 보유분까지 더한 전체 보유 수량(`quantity`)과 혼동하면 안 됩니다. 위치: [[ERP/backend/app/models/inventory.py]].

## 인수인계 / HandoverDoc

튜브 담당자가 작성·제출하고, 고압/진공 담당자가 PIN으로 '인수 확인'을 하는 인수인계서입니다. 인수 확인하는 순간 품목 수량만큼 튜브→인수부서로 실제 재고가 이동합니다(생산 이동). 상태는 작성중(draft)·제출됨(submitted)·인수완료(received)입니다. 모델: `HandoverDoc`/`HandoverLine` ([[ERP/backend/app/models/handover.py]]).

## 결재 알림 / Notification

직원별로 쌓이는 알림입니다. 결재 요청 도착·승인·반려·인수인계 도착 같은 사건을 받는 사람에게 보여 줍니다. 화면은 30초마다 새로 확인(폴링)합니다. 모델: `Notification` ([[ERP/backend/app/models/notification.py]]).

## approval_rules / 결재 규칙

어떤 작업이 창고 결재·부서 결재·즉시 반영 중 무엇을 타는지 정하는 규칙 상수를 한곳에 모은 단일 원천입니다. 예전에는 같은 규칙이 여러 파일에 복제돼 있어 한쪽만 바뀌면 조용히 어긋났는데, 지금은 `services/approval_rules.py` 한 곳에서 관리하고 프론트와의 불일치는 테스트가 자동으로 잡습니다(ADR-0005). 위치: [[ERP/backend/app/services/approval_rules.py]].

## mes_code

품목을 식별하는 코드입니다. `모델기호-구분코드-일련번호`(예: 4자리 일련번호) 형태로, DB가 분해 필드 3종에서 자동 계산하는 값이라 직접 고쳐 쓸 수 없습니다. 전역에서 중복될 수 없으며, 소프트삭제된 코드도 영구 점유해 이력 추적성을 지킵니다. 예전의 `erp_code`를 대신하는 현재 기준 코드입니다. 위치: [[ERP/backend/app/models/item.py]].
