기존 방식의 문제점
왜 바꿔야 했을까요?
📋
담당자 없을 때 — 3개 파일 관리
물건을 이동할 때마다 창고 대장 작성 → 부서 재고 파일 수정 → 창고 재고 파일 수정. 세 단계 중 하나라도 빠지면 실재고와 기록이 맞지 않는 일이 빈번했습니다.
재고 불일치
🔁
담당자 있어도 — 이중 작업
창고 담당자가 생긴 후에도 창고 대장 작성과 부서 재고 파일 수정, 똑같은 내용을 두 곳에 입력해야 했습니다.
이중 업무
⚠️
불일치 시 임의 조작
실재고와 엑셀 재고가 안 맞으면, 원인을 파악하지 않고 입출고로 숫자를 맞추는 경우가 있었습니다. 이력이 남지 않아 데이터 신뢰성이 없어집니다.
신뢰성 제로
🔧
BOM — 하위 부품 수동 출고
완제품·공정완료품을 입고 처리할 때, 들어간 하위 부품을 엑셀에서 하나하나 찾아 수동으로 출고해야 했습니다. 누락이 생기기 쉬웠습니다.
누락 위험

현장 공감 유도

  • 재고 불일치 경험 있는지 청중 반응 확인
  • "이런 상황 많이 있으셨죠?" 한 마디 추가 가능

Q&A 대비

  • Q: 담당자 있으면 해결 안 되나? → 이중 작업·BOM 문제는 여전히 남음
  • Q: 엑셀이 더 편하지 않나? → 불일치·이력 없음 문제 강조
품목 분류
재고는 세 단계로 나눠 관리합니다.
R
원자재
Raw Material
외부에서 발주한 후 입고된 그 상태의 부품입니다. 아직 아무 가공도 하지 않은 상태입니다.
예: HR · VR · AR (고압·진공·조립 원자재)
A
중간공정
Assembly / Work-in-Process
ass'y 완제품이 되기 전에 미리 작업을 해둔 것입니다. 다음 공정에서 사용할 준비가 된 중간 결과물입니다.
예: HA · VA · AA (고압·진공·조립 중간공정)
F
공정완료
Finished Process
다음 공정으로 넘길 준비가 완전히 된 것입니다. 해당 부서에서의 작업이 끝난 상태입니다.
예: HF · VF · AF (고압·진공·조립 공정완료)

말할 한 줄

"코드 뒤 알파벳 하나로 이 부품이 어느 단계인지 바로 알 수 있습니다."

설명 포인트

  • 앞 알파벳 = 부서 (H:고압, V:진공, A:조립, T:튜브, N:튜닝, P:출하)
  • 뒤 알파벳 = 단계 (R:원자재, A:중간공정, F:공정완료)
  • 기존 공정 분류 체계를 그대로 반영한 것

Q&A 대비

  • Q: 코드는 누가 정했나? → 기존 공정 분류 체계 기반, 이미 정해진 것
  • Q: 직접 외워야 하나? → 시스템이 이름+코드 함께 표시함

말할 한 줄

"창고에서 물건 가져올 때 이제 대장 안 써도 됩니다. 앱에서 요청하면 권동환 사원이 확인하고 승인합니다."

현장 포인트

  • 앱 요청 → 창고 담당자 승인 → 수령 순서 강조
  • 승인 없이는 재고 변동 없음 = 무단 반출 방지

Q&A 대비

Q&A 대비

  • Q: BOM이 틀리면? → 관리자 페이지에서 수정 가능
  • Q: 낱개 승인 안 받으면? → 시스템에서 반려 처리됨
앞으로 관리 방식
이렇게 운영합니다.
1
창고 이동
창고 정 — 권동환 사원
창고 부 — 이필욱 부장
창고에서 물건을 가져가거나 반납할 때는 반드시 창고 정/부의 승인이 있어야 재고가 변동됩니다.
2
부서 낱개 입출고
각 부서 정 / 부 승인 필수
BOM을 통하지 않는 개별 품목 조정은 부서 정/부의 승인이 필요합니다. 임의 수량 조작을 막고 이력을 남깁니다.
3
불량 등록
발생 즉시 등록 — 승인 없음
불량이 생기면 바로 시스템에 등록합니다. 별도 승인 없이 재고에서 자동으로 차감됩니다.

말할 한 줄

"창고 이동은 권동환 사원, 부서 낱개 조정은 각 부서장, 불량은 바로 등록. 이 세 가지입니다."

현장 포인트

  • 부서 정/부 지정이 아직 안 된 부서는 발표 후 별도 안내
  • 불량 등록은 습관화가 핵심 — 발생 즉시 바로 할 것

  • Q: 불량 등록 실수하면? → 관리자 페이지에서 수정 가능
