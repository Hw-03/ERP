■ 이번 주 진행 사항 (5/11~5/14)

**1. 입출고 화면 v2 — 4단계 마법사 도입 (이번 주 핵심 작업)**
지난 주 보고드렸던 "입출고 화면·내역 흐름 재설계"에 본격적으로 착수하여,
실제 담당자 작업 동선 기준으로 작업유형(입고/출고/이동/조정) → 대상(부서·창고)
→ 품목·수량 → 확인의 4단계 마법사 구조로 재구성하였습니다.
- **작업유형 분기(sub_type)** — 입고·출고·이동·수량보정을 명확히 분리하여
  화면 어디에서 보아도 같은 버튼·태그가 일관되게 표시되도록 정비
- **수량 보정** 흐름을 입고·출고 방향으로 분리해 의도가 분명히 드러나도록 개편
- **권한 가드** — 작업유형별 접근 가능 부서·역할 사전 차단
- **부서 입출고 + BOM 가드** — 부서 재고를 BOM 기준으로 검증
- **위저드 시각 재디자인** — Step 2/3/4 카드와 카트 영역을 한눈에 들어오는 형태로 정비
- **BOM 묶음(번들) 헤더 통합 + 최종 확인 1단 화면** — 한 번에 들어가는 자재 묶음을
  간결하게 표시하고 마지막 확인을 1화면으로 압축
- **단계 카드 동적 높이 + 섹션 탭 재디자인** — 화면 비율에 맞춰 자동 정리

**2. 입출고 v2 안정성 — 중복 제출 방지(멱등성)**
네트워크 지연이나 더블 클릭으로 같은 입출고가 두 번 기록되는 문제를 막기 위해
요청 단위 식별자(client_request_id)를 도입하였습니다. 같은 요청은 서버에서 한 번만
반영되며, 충돌 시 사용자에게 적절한 메시지(409/503)가 표시됩니다.

**3. 부서 결재 + 직원 권한 분리**
부서별 결재 흐름과 직원 권한 체계를 분리하여, 작업 요청·승인·반려가 부서 기준으로
동작하도록 정비하였습니다. 결재 관련 백엔드 API 3건을 신설하고 명세도 함께 갱신하였습니다.
조립 부서의 직원 담당 모델을 별도로 등록하여, 입출고 화면에서 조립 그룹이 우선 정렬됩니다.

**4. 내 요청 카드 + 입출고 내역 화면 사람 친화화**
- "내 요청" 카드를 한눈에 진행 상태(대기/승인/반려/완료)가 보이도록 재설계
- 입출고 내역 상세 화면을 비개발자 기준으로 용어·표시를 다듬음
- 입출고 내역 **표 전면 개선** (5/14) — 가독성·정렬·필터 정비

**5. 재고 대시보드 품목 이미지 표시**
재고 대시보드 표에 품목 이미지 컬럼을 추가하여, 자재명·코드만으로는 구분이 어려운
품목을 시각적으로 바로 식별할 수 있도록 하였습니다. (Next/Image 최적화 적용)

**6. BOM 세팅 도구 — 실 데이터 입력 진행 (805건)**
지난 주 마무리한 BOM 세팅 도구로 실제 BOM 데이터를 입력하고 있습니다.
- 조립 + 진공 합본 698건 프리셋 임베드
- 고압 99건 추가 → 합본 796건 + DB 동기화
- 추가 튜닝(NF) 9건 → 현재 **805건** 진행
- BOM 부모 후보 자동 추출 스크립트 추가 (입출고 관리대장 2024-2026 원본 기반)

**7. 관리자 화면 전면 정비**
관리자 화면을 PIN 입력 → 8개 섹션 구조로 재정리하였습니다.
- BOM 관리 화면을 가로 분할 레이아웃으로 개편
- 대시보드 필터 정비 및 KPI 카드 컴팩트화

**8. 로그인 화면 단일 카드 재구성**
기존 3단계(부서 → 사번 → PIN) 로그인 화면을 한 화면에서 콤보박스 + PIN 입력으로
처리하도록 단순화하였습니다.

**9. 셀렉트(드롭다운) 디자인 통일 + 생산 가능 수량 정의 재정비**
- 화면마다 제각각이던 드롭다운을 공통 AppSelect로 통일
- 완제품을 만들 수 있는 수량의 정의를 명확히 분리:
  - **즉시 생산(immediate)** — 현재 부서 재고만으로 만들 수 있는 수량
  - **최대 생산(maximum)** — 창고 재고까지 옮겨오면 만들 수 있는 수량
- 화면에서는 status를 4분기(생산 가능/일부 가능/창고 보충 필요/불가)로 시각화

**10. 데이터 정비 — 폐기된 코드/컬럼 정리 (5/14)**
시스템에 남아 있던 더 이상 사용하지 않는 데이터·코드 잔재를 정리하여
앞으로 들어올 실 데이터와 충돌하지 않도록 하였습니다.
- 폐기된 `category` 컬럼 잔재 정리 → `process_type_code` 통일
- `legacy_model` 필드 완전 제거 (DB · API · UI 전 구간)
- 사용하지 않는 완제품 출하(ship) 작업유형 + ShipPackage 모델 전체 제거

**11. 시스템 안정화 / 개발 환경 정비**
- 운영 빌드와 개발 빌드의 디렉터리 분리(`.next-prod`) — 동시 실행 충돌 방지
- 개발 서버 재시작 헬퍼 추가, 임시 산출물 git 추적 제외
- 사이드바 메뉴 재클릭 시 페이지가 기본 상태로 리셋되도록 보정
- CI 자동 검증 실패 2건 즉시 수정 (테스트 라벨, API 명세 동기화)

화면은 제 자리 PC가 켜져 있을 때 http://192.168.0.63:3000 에서 확인하실 수 있습니다.
